# Brainstorming v8 — Pre-Implementation Design Gaps

Issues identified during a full design doc review + feasibility assessment. These need explicit decisions before implementation hits them. Items are ordered by how soon they'll block implementation (Phase 4 blockers first).

---

## 1. `shared struct` mutation mechanism — RESOLVED → Option (a) RefCell

**Blocks:** Phase 4 interpreter (shared struct execution), Phase 7 codegen

**Current state:** The design says `mut` fields on `shared struct` have "interior mutability" but never specifies the mechanism. The doc describes the semantics (RC-backed, assignment shares instead of moves) but not how mutation of `mut` fields actually works under the hood.

**Problem:** "Interior mutability" is not a mechanism — it's a property. The interpreter and codegen both need to know: when code writes `node.left = Some(child)` on a `shared struct`, what happens at runtime?

**Options:**

- **(a) `RefCell` semantics (runtime borrow checking).** Each `mut` field has a borrow flag. Taking `mut ref` sets the flag; a second concurrent `mut ref` panics at runtime. Safe by default; the `panics` effect propagates from mutation sites. Matches Rust's `RefCell<T>`.
  - Pro: Safe. Catches aliased mutation bugs at runtime.
  - Con: Every field mutation has a runtime borrow-check cost. Propagates `panics` effect from innocent-looking field writes.

- **(b) `UnsafeCell` semantics (compiler-trusted, no runtime check).** The compiler trusts that the effect system prevents aliased mutation. No runtime overhead. Undefined behavior if two tasks mutate simultaneously — but the effect system should prevent that via `writes` conflict detection.
  - Pro: Zero overhead. Relies on the effect system the language already has.
  - Con: If the effect system has a soundness gap (e.g., module-internal mutations with no resource — see item #2), this silently produces UB.

- **(c) Hybrid: `RefCell` in debug builds, `UnsafeCell` in release.** Runtime checks catch bugs during development; release builds trust the effect system.
  - Pro: Best of both — safe development, fast release.
  - Con: Debug/release behavior divergence. A bug that only manifests in release is the worst kind.

**Decision:** Option (a) — RefCell semantics everywhere, no caveats. The effect system tracks resource-level conflicts, not instance-level aliasing, so UnsafeCell (option b) is unsound in principle — not just "if the effect system has a gap." The runtime cost (single isize compare-and-set per field access) is unmeasurable in tree/graph workloads. Option (c) hybrid is rejected because debug/release behavior divergence produces bugs that only crash in production.

**Revisit trigger:** UnsafeCell optimization is tracked in design.md Deferred Items. Requires the effect system to be battle-tested in real-world usage AND formal analysis proving effect conflicts fully prevent aliased mutation before removing runtime checks.

## 2. `shared struct` module-internal effects — what resource? — RESOLVED → Option (c) Untracked

**Blocks:** Phase 4 interpreter (effect tracking for shared types), effect checker correctness

**Current state:** The design says "within the defining module, mutating `mut` fields does not require an effect annotation; the compiler tracks these mutations through private function effect inference." But effect inference needs a *resource name* to track. There is no specification of what resource module-internal `shared struct` mutations write to.

**Problem:** Two module-internal private functions that both mutate the same `shared struct` instance have a `writes+writes` conflict on *something* — but what? Without a resource, the effect checker can't detect the conflict, and the auto-concurrency system can't serialize them.

**Options:**

- **(a) Implicit per-type resource.** The compiler generates an implicit resource for each `shared struct` type. `shared struct TreeNode` implicitly declares `effect resource __TreeNode_internal;`. Module-internal mutations infer `writes(__TreeNode_internal)`. Never appears in user-written code; only visible in compiler diagnostics and `--output=json`.
  - Pro: Automatic. No annotation burden.
  - Con: Per-type, not per-instance — two unrelated `TreeNode` instances would serialize against each other. May over-serialize.

- **(b) Require explicit resource even for module-internal mutations.** The module author writes `effect resource TreeState;` and the compiler checks `writes(TreeState)` for any mutation.
  - Pro: Precise. Consistent with the rest of the effect system.
  - Con: Violates the promise that private functions need no annotations. Adds boilerplate for the most basic shared struct usage.

- **(c) Module-internal mutations are untracked.** The effect system does not track them; the `RefCell` runtime check (if chosen in item #1) catches aliased mutation at runtime instead.
  - Pro: Simple. No new machinery.
  - Con: Auto-concurrency cannot parallelize or serialize module-internal shared struct operations. Correctness depends entirely on runtime checks.

**Decision:** Option (c) — module-internal mutations are untracked by the effect system. RefCell (item 1) provides runtime safety for aliased mutation. Option (a) per-type resources are too coarse-grained to be useful — 1000 tree nodes would all serialize against each other, meaning zero concurrency for the primary shared struct use case. Option (b) violates the core design promise that private functions need no annotations. With RefCell catching actual bugs, the effect system handles everything else, and module-internal mutation is the narrow seam where runtime checks take over.

## 3. Closure calling through `ref` — RESOLVED → Option (a)

**Blocks:** Phase 4 interpreter (callback patterns, `for` loop over closures)

**Current state:** The design says `Vec[Fn(T) -> U]` holds closures and `vec[i]` always moves the closure out. But the most basic callback pattern — iterating over a collection of callbacks — borrows the collection via `for`, meaning each callback is accessed as `ref Fn(...)`.

**Problem:** Can you call a closure through a `ref`? The design doesn't address this. Without it:

```kara
let callbacks: Vec[Fn(i64) -> i64] = [|x| x + 1, |x| x * 2];
for f in callbacks {
    print(f(10));   // f is ref Fn(i64) -> i64 — is this legal?
}
```

This is the most common closure usage pattern and it doesn't work under the current spec.

**Options:**

- **(a) Calling through `ref` is always valid for repeatable closures.** A closure that captures only `Copy` or `ref` values can be called through `ref` — calling it doesn't consume anything. A once-callable closure (captures owned non-`Copy`) cannot be called through `ref` — the compiler rejects it.
  - Pro: Covers the common case. The distinction is already tracked by the ownership system.
  - Con: The error message for "can't call this closure through ref" needs to be clear about why.

- **(b) Calling through `ref` is always valid; once-callable closures simply can't exist in a `Vec`.** Storing a once-callable closure in a `Vec` is a compile error because the `Vec` can't guarantee the closure won't be called twice (via two `for` iterations over the same element, or `vec[i]` called twice).
  - Pro: Simpler rule. Once-callable closures are inherently owned values; putting them in collections doesn't make sense.
  - Con: Prevents some valid patterns (e.g., a work queue that pops and calls).

- **(c) Introduce `ref`-callable and `own`-callable closure distinctions in the type system.** `Fn` is ref-callable; `FnOnce` (or equivalent) is own-callable.
  - Pro: Precise.
  - Con: The whole point of the design was to avoid `FnOnce`.

**Decision:** Option (a) — calling through `ref` is valid for repeatable closures. The ownership system already tracks which closures are once-callable; this extends that knowledge to the call mechanism. Once-callable closures (capturing owned non-Copy values) cannot be called through `ref` — the compiler rejects it with a clear error explaining why.

## 4. `let _ =` and `#[must_use]` — RESOLVED → Option (b), design.md updated

**Blocks:** Phase 4 interpreter (warning behavior), Phase 5 diagnostics

**Current state:** The design says `let _ = connect(conn)?` still warns for `#[must_use]` types. Quote: "explicit discard does not suppress it."

**Problem:** `let _ =` is a universal convention meaning "I intentionally discard this" (Rust, Swift, Go, Python). Making it not suppress the warning means there is *no* way to intentionally discard a `#[must_use]` value without a diagnostic. This will frustrate every user who has used any of these languages.

**Options:**

- **(a) Current design: `let _ =` does not suppress.** The rationale is that `#[must_use]` types represent values that should always reach a terminal state (like `Connection[Connected]`). If you really want to discard, you must call a terminal method (e.g., `disconnect(conn)`).
  - Pro: Truly enforces the typestate protocol. No silent resource leaks.
  - Con: No escape hatch. Every `#[must_use]` type needs an explicit "discard" terminal method, even when the programmer genuinely wants to drop it (e.g., in tests).

- **(b) `let _ =` suppresses `#[must_use]`.** Standard convention. The programmer has explicitly acknowledged the discard.
  - Pro: Matches universal developer expectations. Still requires explicit syntax — accidental drops still warn.
  - Con: Weakens the typestate guarantee. A `Connection[Connected]` silently leaked via `let _ =`.

- **(c) `let _ =` suppresses at warning level; `#[must_use(strict)]` variant that survives `let _ =`.** Two tiers: normal `#[must_use]` follows convention (b); `#[must_use(strict)]` follows behavior (a) for genuinely critical types.
  - Pro: Best of both. Default behavior matches expectations; critical types get stronger enforcement.
  - Con: Adds complexity to the attribute system.

**Decision:** Option (b) — `let _ =` suppresses `#[must_use]`. design.md updated to match. `let _ =` is the universal "I intentionally discard this" idiom across Rust, Swift, Go, Python. Accidental drops (no binding) still warn. If typestate needs stricter enforcement, that's what `Drop` destructors and runtime checks are for.

## 5. `sequential {}` block scoping — RESOLVED → Option (a), design.md updated

**Blocks:** Phase 6 (auto-concurrency), but worth specifying now

**Current state:** The design says `sequential {}` is an expression whose value is the last statement. It's used to force source-order execution when effects don't capture the ordering requirement.

**Problem:** Can bindings inside `sequential {}` be used after the block? The design doesn't say.

```kara
sequential {
    let handle = init_hardware();
    configure_mode(handle);
}
use(handle);   // Is this valid? handle was declared inside the block.
```

**Options:**

- **(a) `sequential {}` is a normal block — bindings are scoped to it.** `handle` is not visible after the block. To extract values, use the block's expression value: `let handle = sequential { init_hardware() };`.
  - Pro: Consistent with all other blocks in the language. No new scoping rules.
  - Con: Extracting multiple values requires a tuple: `let (a, b) = sequential { ... (a, b) };`.

- **(b) `sequential {}` is transparent — bindings leak into the enclosing scope.** Like a label or annotation, not a true block.
  - Pro: More ergonomic for multi-step initialization.
  - Con: Unique scoping behavior unlike anything else in the language. Confusing.

**Decision:** Option (a) — normal block scoping. design.md updated with explicit scoping note. Consistent with all other blocks in the language.

## 6. `while let` effect re-evaluation — RESOLVED, design.md updated

**Blocks:** Phase 4 interpreter, effect checker

**Current state:** `while let Some(x) = iter.next() { ... }` is specified as a loop form. The design says nothing about whether the expression (`iter.next()`) is re-evaluated each iteration (it obviously must be), but more importantly, whether its effects are attributed once or per-iteration.

**Problem:** If `iter.next()` has effects (e.g., `reads(Network)` for a streaming iterator), are those effects:
- Attributed once to the enclosing function (the static view — "this function reads Network"), or
- Somehow counted per-iteration (no — effects are not quantified)?

**Likely answer:** Effects are a static property — `reads(Network)` appears once in the inferred set regardless of how many iterations occur. This is consistent with the rest of the design. But it should be stated explicitly, because a user might wonder whether a `while let` loop over a network stream has different effect semantics than a single read.

**Decision:** Added note to design.md Loops section: "Loop bodies contribute effects to the enclosing function's effect set exactly once — effects are a static property of the code, not counted per iteration."

## 7. `collect_all` dynamic fan-out — RESOLVED → Option (b), deferred to Phase 6

**Blocks:** Phase 6 (concurrency), but worth specifying now

**Current state:** `collect_all` takes up to 8 closure arguments (fixed-arity overloads). The design says "beyond 8, restructure as a collection with a loop."

**Problem:** What does "restructure as a collection with a loop" mean concretely? If I have `Vec[Fn() -> Result[T, E]]` closures, is there a `collect_all_vec` or equivalent? Or does the user write a manual `for` loop (which loses auto-parallelism since the compiler can't know the closures are independent)?

**Options:**

- **(a) Provide `collect_all_vec(closures: Vec[Fn() -> T with _]) -> Vec[T]`.** Dynamic fan-out with full auto-parallelism. The compiler knows all closures in the vec are independent (they're values, not aliased references).
  - Pro: Covers the dynamic fan-out case. Natural complement to fixed-arity `collect_all`.
  - Con: Homogeneous return type only.

- **(b) No dynamic fan-out in v1.** 8 branches is enough for the first release. Dynamic fan-out requires task spawning machinery that belongs in Phase 6.
  - Pro: Simpler. Avoids designing the task API prematurely.
  - Con: Users with >8 parallel fetches are stuck writing sequential loops.

**Decision:** Option (b) — no dynamic fan-out in v1. 8-branch limit is pragmatic. Dynamic fan-out (`collect_all_vec`) is a Phase 6 deliverable when task spawning machinery exists.

## 8. `dbg()` formatting — which trait? — RESOLVED → Option (b), add Debug trait

**Blocks:** Phase 4 interpreter (dbg implementation)

**Current state:** `dbg()` is described as a transparent debugging function that prints file, line, expression text, and value. The design mentions `Display` as the string formatting trait. It does not mention a separate `Debug` trait.

**Problem:** `Display` is for user-facing output (`"Alice"`, `"42"`). Debug output typically needs a different representation that shows structure (`User { name: "Alice", id: 42 }`). Rust uses `Debug` for `dbg!()` and `Display` for `format!()`. Does Kara have a `Debug` trait, and does `dbg()` use it?

**Options:**

- **(a) Single `Display` trait for everything.** `dbg()` uses `Display`. No separate `Debug` trait. Types that want structural output implement `Display` to show it.
  - Pro: Simpler — one formatting trait to learn.
  - Con: `Display` for `User` might be `"Alice"` (user-facing), but `dbg()` wants `User { name: "Alice", id: 42 }` (dev-facing). Can't have both from one trait.

- **(b) Add a `Debug` trait.** `dbg()` uses `Debug`. `f"..."` interpolation uses `Display`. `#[derive(Debug)]` generates structural representations.
  - Pro: Matches Rust. Separates user-facing from dev-facing formatting. `#[derive(Debug)]` is trivial to implement.
  - Con: Two formatting traits. Users must learn when to use which.

- **(c) `dbg()` uses `Display` but with a compiler-generated structural fallback.** If a type doesn't implement `Display`, the compiler generates a structural representation (field names + values) automatically. No separate trait.
  - Pro: One trait, but `dbg()` still works for types without `Display`.
  - Con: Magic fallback behavior. The compiler silently generates formatting for types that never opted in.

**Decision:** Option (b) — add a `Debug` trait. `dbg()` uses `Debug`; `f"..."` interpolation uses `Display`. `#[derive(Debug)]` generates structural representations. Every language that tried to unify user-facing and dev-facing formatting failed (Python `__str__`/`__repr__`, Rust `Display`/`Debug`, Haskell `Show`). The two-trait split is proven.

## 9. String `+` operator — RESOLVED → Option (a), hardcode for Phase 4

**Blocks:** Phase 4 interpreter (string operations)

**Current state:** The Strings section shows `"Hello, " + name` as valid syntax. But operator overloading is listed as a Deferred Item ("desugar `a + b` to `Add.add(a, b)` via standard traits").

**Problem:** Is `+` for strings a compiler-special-case hardcoded before operator overloading lands? Or is it expected to wait for the `Add` trait?

**Options:**

- **(a) Hardcode string `+` as a special case.** The compiler recognizes `String + String` (and `String + &str` / `&str + String`) without an `Add` trait. Replaced by the general mechanism when operator overloading lands.
  - Pro: Strings work in Phase 4 without implementing the full trait machinery for operators.
  - Con: Special case that must be removed later.

- **(b) Implement `Add` trait early, just for strings.** Define the `Add` trait and `impl Add for String` as the first operator overloading case. The interpreter recognizes `+` via trait dispatch.
  - Pro: No special cases. Tests the trait-based operator infrastructure early.
  - Con: More infrastructure to build for Phase 4.

- **(c) No string `+`. Use `f"..."` interpolation or `.concat()` instead.** Remove `+` from the string examples.
  - Pro: No special case. Interpolation is the idiomatic approach anyway.
  - Con: `a + b` for strings is so expected that its absence will surprise users.

**Decision:** Option (a) — hardcode `String + String` as a compiler special case for Phase 4. Replaced by `Add` trait dispatch when operator overloading lands in Phase 5+. String concatenation is too fundamental to wait for the full trait system.

## 10. NaN equality near FFI boundaries — RESOLVED → lint for Phase 5

**Blocks:** Nothing immediately. Worth noting for Phase 5 (tooling/diagnostics).

**Current state:** Kara defines `NaN == NaN` as `true`. The FFI interop note mentions the discrepancy with C/IEEE 754 but it's a single sentence buried in the Numeric Semantics section.

**Problem:** A function that passes floats to C via FFI and then checks equality on the C side will get different results. This is a subtle bug source. The design should specify whether the compiler warns about this.

**Decision:** Add a lint for Phase 5: when a `f64` value crosses an FFI boundary and is compared with `==`, emit a warning noting the NaN semantics difference. Suppress with `#[allow(ffi_float_eq)]`.

## 11. Effect polymorphism viral rule + default-permitted `allocates(Heap)` — RESOLVED → Option (a), design.md updated

**Blocks:** Effect checker correctness (current phase)

**Current state:** The viral rule says `with _` or `with E` must be declared at every public boundary. `allocates(Heap)` is default-permitted — public functions don't need to declare it.

**Problem:** Consider:

```kara
pub fn transform(data: Vec[i64], f: Fn(i64) -> i64 with _) -> Vec[i64] with _ {
    map(data, f)
}
```

If the caller passes a closure whose only effect is `allocates(Heap)`, does the viral rule still require `with _` on `transform`? The closure has *an* effect, but it's default-permitted.

Two interpretations:
- **(a) `with _` is always required when accepting an effectful closure, regardless of whether the effects are default-permitted.** The annotation means "this function is polymorphic over effects" — it's about the *mechanism*, not the specific effects.
- **(b) Default-permitted effects don't trigger the viral rule.** A closure with only `allocates(Heap)` is effectively pure from the annotation perspective.

**Decision:** Option (a) — `with _` is always required regardless of default-permitted status. design.md updated with explicit clarification on the viral rule. The annotation is about mechanism (effect polymorphism), not specific effect values.

## Summary of direct fixes already applied to design.md

These were unambiguous issues fixed without discussion:

1. **Broken table in Standard Data Structures** — The continuation table (String, Tuple, Option, Result, Pool, Handle) was missing its header row and appeared as raw pipe-delimited text. Added the header row.

2. **`Copy` + parameter mode inference interaction** — Added a clarifying note in Part 6 explaining that inference still runs on `Copy` types (parameters are inferred as `own`) but `own` on a `Copy` type triggers an implicit copy rather than a move, so RC fallback triggers are never reached for `Copy` types.

3. **Refinement type constraint language** — The "Allowed constraints" paragraph now explicitly states that arbitrary method calls (e.g., `self.is_ascii()`) are **not** allowed — the constraint language is restricted to predicates the compiler can evaluate without executing user code.
