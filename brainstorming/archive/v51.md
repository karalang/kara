# Brainstorming deferred_v1 — Effects and data-race prevention: resource-level vs data-level conflict checking

**Status:** DECIDED — 2026-04-25. Core design gap resolved across five analysis rounds (`v51_analysis_v1.md` through `v51_analysis_v5.md`). Design doc updated. See decisions below.

---

## Decisions

**Q1 — Scope of fix:** `par struct` / `par enum` keyword pair. `shared struct` retains its name for single-task RC semantics. `par struct` is the new concurrent shared type: always Arc, `mut` fields must be `Atomic[T]` or `Mutex[T]`, enforced at the definition site. This is the two-type split from `v51_analysis_v4.md`, implemented with the `par` keyword rather than `concurrent`.

**Q2 — `writes(Shared[T])` interaction:** Moot. C1 is definitively off the table. No `Shared[T]` resource family introduced. Effect system unchanged.

**Q3 — `lock` masking under C1:** Moot. C1 off the table. `lock` remains effect-transparent as designed.

**Q4 — Non-atomic field tearing:** Closed by construction. `par struct` forbids bare `mut` fields at the definition site — all `mut` fields are `Atomic[T]` or `Mutex[T]`. Tearing is structurally impossible.

**Q5 — Diagnostic story:** `karac explain` surfaces the `par struct` field constraint at the definition site (bare `mut` field → compile error with suggestion to use `Atomic[T]` or `Mutex[T]`). Simpler than the D-lite attribute story because the enforcement point is the definition, not the parallel region boundary. Declaration-site note on `shared struct` definition (when `mut` fields exist) reminds the programmer that cross-task access requires `par struct`.

**Q6 — Pitch language:** Updated. "compile-time concurrent mutation safety on `par struct` types" is now a literal property — enforced at the definition site, not aspirational. Precise pitch: *"effects track resource I/O conflicts across tasks; `par struct` enforces cross-task memory safety at the type definition."* Avoid: "prevents all concurrency bugs", "compile-time thread safety" (too broad). `shared struct` is the single-task reference-semantics type; `par struct` is the concurrent reference-semantics type.

**Keyword decisions:**
- `par struct` / `par enum` — concurrent shared types (reuses existing `par` keyword; maps to `par {}` block vocabulary)
- `shared struct` / `shared enum` — single-task RC types (name unchanged)
- `concurrent struct` — rejected (new vocabulary root not needed)
- `seq struct` — rejected (three-way ambiguity: plain `struct`, `seq struct`, and `par struct` all exist; `seq` doesn't distinguish the first two)
- `async` / `sync` as block keywords — rejected (wrong semantics signal; `async` implies colored functions, `Future`, `await`; `par {}` / `seq {}` are precise and carry no false baggage)

**Design doc changes made:**
- Lines 5281 and 5423: corrected false compile-time guarantee claims (runtime borrow flags ≠ compile-time rejection)
- Part 5b added: full `par struct` / `par enum` specification
- Arc-promotion pass (§ Rc vs Arc): updated Phase 2 to distinguish `par struct` (always Arc), `shared struct` (compile error if crossing parallel region with multi-task access), and other RC values (promote as before)
- Deferred Items: `Concurrent` auto-trait added as P1

**Original status:** OPEN — opened 2026-04-23 during a strategic pressure-test of Kāra's pitch against `docs/design.md`. The question is whether the effect system's current guarantees line up with what the pitch language ("compile-time concurrency conflict detection") implies, and whether the design should extend to close the gap or whether the framing should sharpen to match the design. Go-all-in scope per user direction: consider full design updates alongside framing-only fixes.

## Context

### What the effect system currently guarantees

The conflict matrix at `docs/design.md:3530–3556` defines conflicts over **resources**, not over **memory**:

| Combination | Same resource | Different resources |
|---|---|---|
| `reads` + `reads` | Safe | Safe |
| `reads` + `writes` | **Conflict** | Safe |
| `writes` + `writes` | **Conflict** | Safe |
| `sends` + `sends` | Safe | Safe |
| `sends` + `receives` | Safe | Safe |
| `receives` + `receives` | Safe | Safe |
| `allocates` + `allocates` | Safe | Safe |
| `panics` | N/A | Never conflicts |

"Resource" here is a named token — `Network`, `Fs`, `UserDB`, or a parameterized version like `Network[conn]`. **Not a memory region.** Two tasks that both declare `writes(UserState)` and mutate a `shared struct UserState` field race in the ordinary memory-safety sense unless something else (the RC layer, `Mutex[T]`, `Atomic[T]`) serializes them. The effect system treats them as a conflict and serializes *the effect-declared work* — but "serialize by effect" only actually serializes if both writes show up inside the auto-concurrency region the scheduler is looking at. A `spawn()`-based or `par {}`-based interleaving where two tasks both hold a `Rc<shared UserState>` still needs `Mutex` or atomics at the data level.

### Where the design already addresses this

The design is explicit about the two-layer split (`docs/design.md:338`):

> The `lock node { ... }` syntax ... is an **effect-transparent** code-generation construct: effects inside the lock block contribute to the enclosing function's inferred or declared effect set exactly as if they were written outside the block. `lock` does not scope, mask, or rename effects. Its role is purely runtime synchronization — the effect system and the mutex layer are complementary, with effects providing coarse compile-time serialization (per-resource) and `Mutex[T]` providing precise runtime serialization for intra-resource granularity that effects alone cannot express.

And `docs/design.md:340`:

> For any code that uses atomics or locks, the effect system tracks *what* is touched (via the resources reached by reads/writes to non-atomic state) and the mutex/atomic layer tracks *when* it is safe to touch it. ... The two layers do not overlap, and the language commits to never collapsing them into a single notion.

So the **design is internally consistent**: effects = resource-conflict, Mutex/Atomic = data-race. But:

### Why this still matters

1. **Pitch-expectation gap.** External framing that says "effects prevent concurrency bugs at compile time" is read by most listeners as "compile-time data-race freedom à la Rust's `Send`/`Sync`." The design does not deliver that — it delivers something related but coarser. When a user's mental model says *"I declared `writes(X)`, the compiler promised me conflict checking"*, and then they write a `shared struct` mutation that passes effect-check but panics at runtime on a borrow-flag conflict (or, worse, silently tears a non-atomic field), they will feel lied to.

2. **Parameterized resources are a partial answer.** `docs/design.md:5783, 5851–5853` and § Parameterized Resources already handle the common per-instance case (each TCP connection is its own resource). That's real and valuable. But it only helps when the mutated thing is a parameterized resource — which `shared struct` fields are not by default.

3. **Runtime borrow-tracking fires late.** The current `shared struct` mutable-field story (`docs/design.md:5419–5423`) uses runtime borrow flags that *panic* on conflict. That's memory-safe but user-hostile: the failure is dynamic, non-local, and discovered only on the one codepath that exercised the race. An effect-system extension would catch it statically.

4. **LLM code-generation stakes.** An LLM asked to "use the effects Kāra gives you" will produce code that declares effects and assumes effects cover the common concurrency pitfalls. If the coverage is narrower than the framing, the LLM produces subtly unsafe code with clean-looking signatures. This is the exact scenario our [LLM-era weighting](../memory-references) principle says to optimize against.

### Prior-art reference

- **Rust**: `Send` / `Sync` auto-traits plus borrow checker give compile-time data-race freedom. No explicit effect vocabulary.
- **Pony**: reference capabilities (`iso`, `val`, `ref`, `box`, `tag`, `trn`) classify how values can cross actor boundaries — direct data-level guarantees, no effect vocabulary.
- **Koka / Unison**: effect rows, but no data-race story (single-threaded or pure).
- **Nothing we know of bundles both** first-class effects *and* compile-time data-race freedom. Either-or is the industry norm.

## Open questions

### Q1 — What is the scope of the fix? Framing vs extension vs redesign.

- **A: Framing only (no design change).** Keep the two-layer contract. Retire "prevents concurrency bugs" from the pitch; replace with precise wording: *"Effects track which resources a function touches and serialize conflicting access to the same resource; data-race freedom on shared state is provided by `Mutex[T]`/`Atomic[T]`."* Write a dedicated "what effects do and don't do" section in the book. Ensure `karac explain` and diagnostics never suggest the stronger property.
  - **Pro:** zero spec churn. The design is sound as-is. Honest with users.
  - **Con:** weakens the unique-selling-proposition copy. "Effects track resources" is less headline-grabbing than "effects prevent concurrency bugs." May push the LLM-friendly story toward requiring users to remember two separate mechanisms.
- **B: Framing + guardrails (small design nudge).** Keep the two-layer contract, but add static help for the common sharp edge. Specifically: when a function declares `writes(R)` on a parameterized resource *and* captures a `shared struct` by reference, emit a compiler note at the signature: *"note: data-race freedom on captured `shared` values requires `Mutex[T]` or `Atomic[T]`; effects alone do not prevent interleaved writes to shared memory."* The note is suppressible with `#[allow(shared_without_lock)]`. Also: in `karac explain`, if a conflict cannot be proven false but the resources differ, surface the two-layer caveat in the trace.
  - **Pro:** cheap, preserves the design, closes the expectation gap at the exact moment the user would be misled.
  - **Con:** noise on benign cases (shared read-only). Requires a capture-analysis pass that knows `shared` values participated in the body.
- **C: Redesign — effects cover shared-state races too.** Extend the effect vocabulary so that writes to `shared` fields *are* an effect the system can statically reason about. Options C1–C3 below are three shapes.
  - **C1: Shared-mutation as a per-struct effect resource.** Declaring a `shared struct UserState` implicitly introduces a resource `Shared[UserState]`. Reads/writes of `mut` fields on any `UserState` value contribute `reads(Shared[UserState])` / `writes(Shared[UserState])`. Two tasks that both carry `writes(Shared[UserState])` conflict at effect-check. **Escape hatch:** `Mutex[UserState]` masks the effect, consistent with the `lock` block semantics already in `docs/design.md:338`.
  - **C2: Shared-mutation as a parameterized resource keyed by allocation identity.** Each `shared` value allocated contributes `Shared[v]`; effects carry the value as the resource parameter. Same serialization rules as other parameterized resources (`docs/design.md:5851–5853`). Catches cross-task races on specific instances, not on all instances of a type. Much finer grain than C1.
  - **C3: Capability-style marker types that travel with `shared` handles.** Lift Pony-style capabilities into the type system (`shared iso T`, `shared val T`, `shared ref T`). Send/Sync-equivalent checks at the type layer. Effects untouched. Larger surface change.
  - **Pro:** closes the gap at the language level. The pitch copy becomes honest without qualification. LLMs generating code get the same static coverage for shared state that they get for resource conflicts. Parity with Rust's Send/Sync promise.
  - **Con:** largest spec churn. Risks producing a parallel mechanism for what `Mutex` already handles. Risks making every `shared` value drag an effect through every calling function — effect-row explosion.
- **D: Type-system enforcement at the arc-promotion boundary — no new effects.** Rather than extending the effect vocabulary, repurpose the arc-promotion pass (`docs/design.md:5262–5283`) as a check site. The pass already identifies every `Rc` value whose live range overlaps a parallel region and promotes it to `Arc`. Extend it: when a promoted value is a `shared struct` with any `mut` field, verify that every write to those fields inside the parallel region is enclosed in a `lock` block. If not → compile error. No new effects, no new effect declarations, no `Shared[T]` resource family. The check is a structural property of the type and the parallel region, not of the effect row.

  **What `Mutex[T]` and `lock` are.** In Kāra, these are two halves of the same mechanism — not alternatives. `Mutex[T]` is a wrapper type that holds a value under mutual exclusion: `let node = Mutex(TreeNode { val: 1, ... })`. Once wrapped, you cannot read or write fields directly. The `lock` block is the only way to access the value — it acquires the mutex on entry and releases it on exit (including on early return or panic), with no guard value to forget: `lock node { node.val = 42 }`. You cannot write a valid `lock` block on a value that isn't wrapped in `Mutex`, so D's check — "every `mut` field write inside a parallel region must be inside a `lock` block" — implicitly requires the `Mutex` wrapping too. The two are enforced together.

  **What D specifically checks.** When the arc-promotion pass promotes an `Rc` to `Arc` (because the value's live range overlaps a parallel region), it additionally walks every write to a `mut` field of that `shared struct` inside the parallel region. If any such write is not syntactically enclosed in a `lock` block, the compiler emits a compile error — not a warning, not a runtime panic. The bad code cannot compile. The runtime borrow-flag panic (the current v1 fallback) cannot be reached because the compiler refuses to produce the binary.

  **Why this is sufficient without touching the effect system.** The arc-promotion pass already knows exactly which values cross parallel region boundaries — that's its primary job. D piggybacks the lock-block check onto information the pass already computes. The effect row stays clean: an accessor or helper that writes a `mut` field carries no new effect from D; the enforcement is structural (does this write have a `lock` ancestor in the parallel region?) rather than propagated through the call graph.

  - **Pro:** zero effect-system churn. No effect-row bloat — accessors and helpers stay pure if they are pure. Delivers a hard compile-time guarantee (not a warning): `mut` field writes in parallel regions without `lock` are rejected. Keeps effects and type/ownership concerns in separate lanes, matching the design's existing split. The arc-promotion pass already runs; this is an additional check on its output, not a new pass. B's warning is unnecessary alongside D — D is a hard error, so the runtime panic path is closed and there is nothing left to warn about.
  - **Con:** the enforcement site (arc-promotion boundary) is further from the declaration site than an effect annotation on the function signature — errors fire at the parallel region, not at the function that performs the write. May require the parallel-region check to see through call boundaries into called functions (interprocedural), which adds analysis complexity. Does not extend the pitch story to "effects cover data races" — a separate mechanism requires a separate sentence in the docs.

**Revised lean: D alone.** Rationale: D delivers a hard compile-time error — not a warning, not a runtime panic. Once D is in place, the runtime borrow-flag panic path is closed (the compiler refuses to produce the binary), so B's warning has nothing left to warn about and becomes redundant machinery. A alone understates a real capability gap. C1 is off the table because it reverses the design's existing decision (line 5439) that internal `mut` field writes don't require effect annotations — adding them virally through the internal call graph, not just at public boundaries. D is preferred over C1 because it delivers the same static guarantee without touching the effect vocabulary or reversing that decision. D keeps the two-layer contract (effects = resource I/O scheduling; type system = memory concurrency safety) architecturally clean. C2 is probably too fine-grained; C3 is a different flavor entirely.

### Q2 — If we do C1, how does `writes(Shared[UserState])` interact with existing `writes` declarations?

- **A: Additive, inferred-only on private fns, declared on public.** Same rule as resource effects today. A private fn that mutates a `shared UserState` field gets `writes(Shared[UserState])` in its inferred row; a public fn must declare it.
  - **Pro:** zero rule-learning. Same mental model.
  - **Con:** every public fn that touches a `shared` type declares the effect, propagating virally through the call graph. Public API surface grows.
- **B: Inferred everywhere, including on public fns; not part of the declared row.** Effect row split into "declared resource effects" (public boundary) and "inferred shared-access effects" (internal analysis only). Public callers see only the former.
  - **Pro:** no viral public-surface growth.
  - **Con:** asymmetry with the rest of the effect system. Hidden effects are harder to reason about and defeat the LLM-friendly "what you see is what you get" property.
- **C: Hybrid — declared row names the *type*, inferred row names the *instance*.** `writes(Shared[UserState])` on the declared row (type-level); `writes(Shared[user_value])` in `karac explain` (instance-level). Same pattern as the parameterized resource system already uses.
  - **Pro:** consistent with existing parameterized resource discipline.
  - **Con:** requires users to learn that `Shared[T]` is implicitly declared whenever their signature mentions a `shared` type — a new inference rule at the signature layer.

**Weak lean: C.** Rationale: it preserves the "declared row is the public contract" invariant while giving the conflict analyzer instance-grained data. Also matches the `karac explain` design where declared effects are type-level and inferred effects carry the finer information (`docs/design.md:5783, 5853`).

### Q3 — How do `Mutex[T]` / `Atomic[T]` mask the effect under C1?

Today, `lock node { ... }` is explicitly **not** effect-masking (`docs/design.md:338`). If C1 lands, we need to decide whether a `lock` on a mutex-wrapped `shared struct` drops `writes(Shared[...])` from the enclosing fn's effect row.

- **A: Yes, `lock` masks `writes(Shared[T])`.** Taking the mutex proves exclusive access at runtime; the conflict is already resolved by `Mutex`. The effect system drops the effect from the inferred row while the lock is held; the surrounding fn only carries the effect if there's an unlocked mutation somewhere.
  - **Pro:** correct and useful. The fn that locks before mutating gets a clean effect row.
  - **Con:** contradicts the current design's "effect-transparent lock" commitment (`docs/design.md:338`). Changes the semantics of `lock` itself — a broader change than C1.
- **B: No, `lock` does not mask.** The fn still carries `writes(Shared[T])`; it is the programmer's (or compiler's) job to show the effect is not conflicting via another mechanism.
  - **Pro:** preserves existing `lock` semantics.
  - **Con:** incompatible with C1 — effects would fire spurious conflicts on correctly-locked code, unless we add a separate "mutex-derived non-conflict" proof. Equivalent to reinventing Rust's `Sync` inside the effect system.
- **C: `lock` masks only when the enclosing type is `Mutex[T]` around a `shared struct`, not around arbitrary effect-emitting code.** The masking rule is scoped narrowly: `lock` over a `Mutex[Shared[T]]` drops `writes(Shared[T])`; `lock` over any other thing is effect-transparent as today.
  - **Pro:** preserves the existing guarantee for general code; adds the specific rule needed for C1.
  - **Con:** special-case machinery at the masking site. More rules to remember, more "why didn't my effect go away" diagnostics to write.

**Weak lean: A, with the acknowledgment that it is a real change to `lock` semantics.** Rationale: `lock` being effect-transparent today is defensible only because effects don't currently claim to cover data races. Once C1 lands, effect-transparency forces the programmer to mask the effect themselves, which is strictly worse than having `lock` do the natural thing. If we commit to C1, we should commit to A.

### Q4 — Non-atomic field tearing

Even with C1, writing to a `u64` field on a `shared struct` from two tasks without a lock and without `Atomic[T]` produces a torn read. If the effect system reports *conflict, serialize*, the programmer might reasonably assume atomicity where there is none.

- **A: C1 covers this automatically — conflict means "effects serialize or you declare both non-conflicting", and both branches are safe.** The scheduler either serializes (no tearing) or proves non-conflict (single-threaded access, no tearing).
  - **Pro:** no new work.
  - **Con:** only true inside the compiler's concurrency-visible scope. Manual `spawn` that hands a `shared` value to two threads and declares non-conflicting effects (`writes(Shared[A])` and `writes(Shared[B])` on disjoint types) can still tear a shared field that happens to appear in both paths transitively.
- **B: Field-level tearing is always a compile error on `mut` fields of `shared struct` unless the field is `Atomic[T]` or wrapped in `Mutex`.** Direct mutation of a non-atomic `u64 mut` on a `shared` value from any context that could see two threads is rejected.
  - **Pro:** plugs the hole completely.
  - **Con:** forces every `shared struct mut` field to be either atomic or locked. Heavy syntactic tax and design shift.
- **C: Rely on runtime borrow tracking (status quo, `docs/design.md:5419–5423`).** Keep the runtime flag; the C1 effect check is layered on top as a static compliment, not replacement.
  - **Pro:** no new rules.
  - **Con:** runtime panic is still the fallback. Loses the "no surprises" property.

**Weak lean: A if C1 lands.** Rationale: C1 already serializes conflicting `writes(Shared[T])`, and a careful programmer can't accidentally tear a field the effect system sees. Failures fall through to option C (runtime tracking) as the backstop. The performance cost of B is too high for the marginal safety win.

### Q5 — Diagnostic/explain story

Regardless of which path (B, C1, or combination), the diagnostic layer needs to surface the contract clearly. Current `karac explain` traces effect inference well (`docs/design.md:3606`). It does not explicitly surface "this signature does not protect against data races on shared state."

- **A: `karac explain --concurrency`** — a new view that annotates each captured/returned value with its data-race protection status (owned / borrowed / shared-unlocked / atomic / locked) alongside the effect row.
- **B: Inline notes at `shared` declaration sites** — when a `shared struct` is declared with any `mut` field, the compiler emits a once-per-struct note describing the mutation contract ("writes to `mut` fields are data-race-protected only when `locked` or `atomic`; declare via …").
- **C: Both A and B.**

**Weak lean: C.** Rationale: A is the tool for debugging a specific function; B is the tool for learning the contract at the moment it first matters. They address different audiences at different moments.

### Q6 — Pitch language regardless of outcome

Short list of phrasings to avoid and to prefer, for the book, the design doc introduction, and any external description:

- **Avoid:** "prevents concurrency bugs", "compile-time thread safety", "data-race-free by default"
- **Prefer:** "compile-time *resource* conflict detection", "effect-driven serialization of shared-resource access", "data-race freedom via `Mutex`/`Atomic` on top of effect-driven scheduling"
- **If C1 lands:** the prefer/avoid list rebalances — "compile-time data-race prevention on `shared` types" becomes a literal property.

## Risks / follow-up

- **Effect-row bloat under C1.** Every fn touching a `shared` type gains an effect. Mitigated by C in Q2 (type-level declared row) and by effect polymorphism (`docs/design.md:4183–4218`). Needs a real-code test — lift a few of the existing `examples/*.kara` programs into a hypothetical C1 world and see how noisy the signatures get.
- **Interaction with `with_provider`.** Providers (`docs/design.md:4581`) inject resources at runtime. If C1 treats `shared` types as resources, do providers also inject `shared` identities? Probably no — providers are for *external* resources, `shared` is for *internal* memory. Worth a sentence in the spec.
- **Interaction with RC fallback (Feature 4 Part 4).** RC fallback already promotes `Rc → Arc` when a value crosses a parallel region (`docs/design.md:5258–5274`). C1 conflict-checking and Arc promotion should line up — if a value gets promoted to `Arc`, it was visible to multiple tasks, which is also where `writes(Shared[T])` matters. Cross-check that the algorithms agree on what counts as a parallel region.
- **Kernel / embedded profiles.** Those profiles already forbid `allocates(Heap)`. Under C1, do they also forbid `shared struct`? Probably yes (already implicit — `shared` requires RC which requires heap). Confirm the profile table in `docs/design.md` still makes sense.
