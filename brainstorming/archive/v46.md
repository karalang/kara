# Brainstorming v46 — Effect-trait fusion (collapse `trait` + `effect resource`)

**Status:** ✓ RESOLVED — 2026-04-23. Adopted **F1a (status quo)** — keep `trait` + `effect resource` as two separate declarations. No spec, example, compiler, or runtime change; the only doc update is a one-sentence rationale added to `design.md § Provider-Rooted Resources` so readers learn *why* the two names are separate.

| Group | Option | Disposition |
|---|---|---|
| F1 — fuse? | F1a | Status quo. Trait = predicate; effect resource = tracked identity. Two names for two concepts. |
| F2 — non-resource traits | moot under F1a | `Eq` / `Ord` / `Iterator` / format / arith traits unchanged. |
| F3 — multi-resource case | moot under F1a | `effect resource UserDB: DatabaseProvider;` + `effect resource AuditDB: DatabaseProvider;` flows naturally. |
| F4 — orphan rules | unchanged | `impl Trait for Type` crate-locality rule preserved. |
| F5 — method effects | unchanged | Trait-method effect ceilings follow the existing rule at design.md:1971; resource identity drives the effect row at the call site for provider traits. |
| F6 — relation to D1a (v44) | unchanged | Separated `impl` blocks preserved; fusion and separated-blocks were orthogonal axes. |
| F7 — migration | none | Zero spec / example / compiler changes; one rationale sentence added to design.md. |

### Why F1a won

1. **Predicate vs identity is a load-bearing distinction.** A trait declares a contract (*does this type satisfy a predicate?*). An effect resource declares a tracked identity (*which specific instance do we route effects through?*). F1c's auto-synthesis would have created two *kinds* of resource identity for one trait — the auto-synthesized `DatabaseProvider` resource plus user-declared `UserDB` — a confusion tax the status quo doesn't pay. Under the LLM-era weighting (`feedback_design_weighting_llm_era.md`), collapsing predicate and identity into one name is exactly the conflation a generator mishandles subtly.
2. **The multi-resource case is the *typical* DI pattern, not the edge case.** UserDB/AuditDB is the canonical example in the spec (design.md:4540–4553) and in real code (`user_db_test.kara:13–19`). Test-clock vs system-clock, read-replica vs write-DB, multiple loggers — most real injections want N resources per trait. F1c would have shortened only the uncommon single-resource case while leaving the common multi-resource case paying the same declaration cost via F3a's "two forms." Net simplification was small.
3. **v46's own savings estimate was modest.** ~5 lines in `user_db_test.kara`, ~3 in `postgres_query_injected.kara`. A small prize for introducing a new top-level keyword and an auto-synthesized-resource semantic.
4. **F1c's example silently shifted F5.** Today's provider-trait methods (design.md:4535–4538) do not declare per-method verbs — `writes(UserDB)` arises from *invoking through the resource identity* at the call site, not from trait-method annotations. v46's F1c example at line 302 quietly migrated methods to `with writes(DatabaseProvider)` declarations on the `effect trait` — a semantic shift smuggled in, not "preserving today's F5c." Staying at F1a avoids needing to resolve that shift at all.
5. **Migration cost is zero.** Every existing Kāra program, example, and spec section stays valid. Compiler and runtime unchanged.

### F1d held as the fallback (not F1c)

If future evidence accumulates that F1a is too duplicative, the next move is **F1d (additive modifier): `trait X as effect resource { ... }`**, not F1c. F1d is opt-in per declaration, introduces no auto-synthesized resource, and keeps "trait = predicate" as the default mental model. It relocates the duplication rather than eliminating it, but without F1c's naming overload.

### What to watch (forward triggers for reopening)

Reopen fusion if any of these land:
- Three or more stdlib traits want effect-resource semantics (e.g., `Logger`, `Clock`, `RandomSource` as trait-bound rather than ambient). One case (`DatabaseProvider`) is not enough.
- Recurring reader feedback that the `trait X` + `effect resource R: X;` pair reads like duplication *at the point of use*, not just at the point of declaration.
- A concrete LLM-generation failure mode where a model reliably forgets the `effect resource` line after emitting `trait X { ... }`.

### Canonical-doc migration

- `docs/design.md § Provider-Rooted Resources (trait-based injection)` — added a one-sentence rationale explaining *why* `trait` and `effect resource` remain separate (predicate vs tracked identity, with the many-to-many relation flowing naturally).
- `docs/implementation_checklist.md` — no entry. F1a entails zero implementation work.
- `docs/deferred.md` — no entry. Nothing is deferred; the decision is status quo.
- Book — no entry. The book already presents `trait` and `effect resource` as separate concepts.
- Examples / compiler / runtime — no change.

### Related — not part of this resolution

Points raised in v46 that stay unsettled but weren't load-bearing to F1:
- **`Send + Sync` on `with_provider`.** Stays at today's shape (bound on the call site's concrete provider type).
- **`with_provider`'s generic bound phrasing `R: effect resource`.** Cosmetic; not reopened.
- **Ambient program-rooted resources.** Stay as today's `effect resource FileSystem;` (no trait bound). Unchanged.

## Original framing (for record)

**Framing.** `design_studies/findings.md` recommends closing v44 at **D1a (status quo: separated trait/impl blocks)** and opening a separate track on **D1d (effect-trait fusion)** — called out as "the genuinely interesting redesign v44 surfaced." `design_studies/parallel_fanout/findings.md` reinforced this: provider-rooted resources are how Kāra binds concurrency semantics cleanly, and the `trait` + `effect resource R: TraitName` pairing is where the language is most duplicative right now.

The concrete pattern in today's Kāra:

```kara
trait DatabaseProvider {                // (1) the behavioural contract
    fn insert(self, k: String, v: String) -> Result[(), String];
    fn get(self, k: String) -> Option[String];
}

effect resource UserDB: DatabaseProvider;   // (2) name the capability
effect resource AuditDB: DatabaseProvider;  // (3) another instance of the same contract

impl DatabaseProvider for InMemoryUserDB { ... }   // (4) a provider
```

Four concepts are in play (`struct`, `trait`, `impl`, `effect resource`). v44 D1d asked whether (1) and (2) can fuse — a trait *is* the contract surface for an effect resource, conforming types are valid providers. **That is this brainstorm.**

Out of scope here: D1 bundled-vs-separated (closed at D1a), D5 retroactive impls (kept as v44 leaned). This brainstorm is exclusively about *what a trait is* — not about *where impls live*.

**Ground rule (`memory/feedback_no_design_compromise.md`):** decide on Kāra's own merits. No existing language has first-class effect resources, so "no precedent" is the expected state, not a disqualifier. Conversely, "it would look like X in Rust/Kotlin" is a surfacing signal only.

## Decision lens — what we're optimizing for

Same weighting carried forward from v44/v45. Per `memory/feedback_design_weighting_llm_era.md`, **Kāra optimizes for an LLM-first authoring world.**

### 1. Compiler-verifiable robustness *(primary)*
Does fusion let the compiler catch more classes of error — e.g., "declared the trait but forgot the effect resource", "two resources with divergent conformance sets", "provider conforms to trait but can't be injected under the resource"?

### 2. LLM-friendliness *(primary)*
- **Fewer concepts** — every concept a model has to track costs context and increases the chance of a plausible-but-wrong program.
- **Local context completeness** — looking at a trait tells you whether it participates in effect tracking without cross-referencing `effect resource` declarations in a different file.
- **Unambiguous syntax** — one way to declare "this is a thing with behaviour that can be a provider."
- **Mechanical refactorability** — renaming a trait / moving it between modules must not silently break an `effect resource` declaration elsewhere.

### 3. Human ergonomics *(secondary)*
The `trait X` + `effect resource Y: X` pair is unusual enough that every onboarding reader asks "why two?" — a documented friction point. Carries weight but doesn't override robustness or LLM-friendliness.

## Context — the concrete duplication in today's language

**Where the pair appears in the spec (`docs/design.md`):**

- Line 4527–4535 — the canonical `DatabaseProvider` + `effect resource UserDB: DatabaseProvider` pair.
- Line 4545 — multi-bound form: `effect resource UserDB: DatabaseProvider + HealthCheckable`.
- Line 4553 — `with_provider` signature refers to `R.Provider` — the trait bound attached to the resource.
- Line 6946–6947 — embedded-domain resources (`effect resource Uart0;`) that do *not* carry a trait bound. Relevant below: fusion must have a story for trait-less resources too.

**Where it appears in real code:**

- `examples/tests/user_db_test.kara:13–19` — `trait DatabaseProvider { ... }` immediately followed by two `effect resource` declarations bound to it. The two-line ceremony is the full duplication surface.
- `design_studies/db_read/postgres_query_injected.kara:22–27` — the same shape for the injected DB study. The *direct*-use variant skips both; fusion is only a question in the injected form.

**What `effect resource R: Trait` contributes over `trait Trait` alone today:**

1. **Identity.** `R` is a distinct name the compiler can track in effect rows (`with reads(UserDB)`). Two resources can share a trait (`UserDB` and `AuditDB` both bound to `DatabaseProvider`) without collapsing in effect analysis.
2. **Provider rooting.** `with_provider[R](p, || body)` injects a concrete provider at the resource, not at the trait.
3. **Trait bound enforcement.** Anything passed to `with_provider[R]` must satisfy `R`'s trait bounds plus `Send + Sync`.
4. **Effect-row surface.** A function that uses `R.insert(...)` gets `writes(R)` in its effect set — the trait name alone never appears in effect rows.

Fusion must preserve all four, or deliberately change one with a reason stated.

## Context — what mainstream conventions look like

| Language | How a "behaviour contract" is declared | How an "injectable capability" is declared | Unified? |
|---|---|---|---|
| Rust | `trait` | Generic bound + user-built DI library (e.g., `dyn Trait` passed as parameter, `tokio::task_local!`) | No; DI is a library concern |
| Java | `interface` | DI framework (Spring `@Autowired`, Guice `@Inject`) | No; DI is a framework concern |
| Kotlin | `interface` | Framework (Koin, Hilt) or constructor injection | No |
| Scala | `trait` | Implicit parameters / `given` / Reader monad / ZIO environment | Partially — ZIO's `R` type parameter is a loose analogue |
| Swift | `protocol` | Environment objects (SwiftUI), constructor injection | No |
| Haskell | class | `ReaderT` / MTL / effect libraries (`polysemy`, `fused-effects`) | Partially — effect libraries unify the two, at the cost of a steep learning curve |
| Go | `interface` | Struct field of interface type (explicit) | No |

**Pattern:** Every mainstream language keeps "contract" and "injectable capability" strictly separate. The unification only appears in effect-system research languages (Koka, Eff, Frank) where effects are a first-class construct. Scala's ZIO comes closest in production, but its `ZIO[R, E, A]` type burns a type parameter on the environment and is widely described as the language's steepest learning curve.

**Kāra's angle:** Kāra already has first-class effect resources. Fusion would be unique among production-targeted languages — not because the idea is novel academically, but because no mainstream language has tried it outside a research/FP context. That's a genuine design opportunity *and* a genuine risk (no established precedent for what breaks).

## Scope of this brainstorm

**In scope:**
- Whether to fuse `trait` and `effect resource` — and if so, under what shape.
- How non-resource traits (`Eq`, `Ord`, `Hash`, `Display`, `Debug`, `Iterator`, `Clone`) fit — or don't.
- How trait-less resources (`effect resource Uart0;`, ambient `FileSystem` / `Network`) fit.
- Provider declaration syntax under fusion.
- Orphan / coherence rules under fusion.
- Migration path if fusion wins.

**Out of scope:**
- **D1 bundled-vs-separated** — closed at D1a per `design_studies/findings.md`.
- **D5 retroactive impls** — kept as v44 leaned; fusion should preserve this capability.
- **`with_provider` runtime implementation** — CR-24 follow-up slice, orthogonal.
- **Algebraic effect handlers** — explicitly rejected earlier (design.md:4524 "no algebraic effect handlers"); fusion does not reopen that choice.
- **Changing how `impl` blocks are spelled** — the *syntax* of conformance is a D1 question, not a D1d question.

## Group F — Fusion shape

### F1. The central question: fuse or not?  *(open)*

**Options:**

- **F1a. Leave as-is.** Keep `trait X { ... }` + `effect resource R: X;` as two declarations. Pay the duplication cost.
  - Pro: zero language change; every existing Kāra program and the spec continue to work unchanged.
  - Pro: the two concepts are genuinely different — a trait is a *predicate* (does this type satisfy the contract?), a resource is an *identity* (which instance do we mean?). Keeping them separate matches that distinction.
  - Pro: non-resource traits (`Eq`, `Ord`, `Hash`, `Display`, all iterator traits, all format traits) keep their natural shape. No tax for traits that don't participate in injection.
  - Con: the duplication is real and visible at every DI site — every reader asks "why two names?"
  - Con: findings.md and `parallel_fanout/findings.md` both flagged this as the most duplicative surface in the language.

- **F1b. Full fusion — every trait *is* an effect resource.** Collapse unconditionally. `trait DatabaseProvider { ... }` declares both the contract and an effect resource with the same name. `impl DatabaseProvider for InMemoryUserDB` declares a provider. Use in effect rows: `with reads(DatabaseProvider)`.
  - Pro: minimum concepts. One declaration, one name, one mental model.
  - Pro: completely eliminates the duplication.
  - Con: **forces every trait to claim resource semantics it doesn't have.** `Eq`, `Ord`, `Hash`, `Display`, `Debug`, `Iterator`, `FromIterator`, `Add`, `Sub`, `Mul`, `Div`, `Serialize`, `Deserialize` — all in the prelude, none of them resources in any useful sense. A `with reads(Eq)` annotation is absurd.
  - Con: **two resources sharing one trait (`UserDB` + `AuditDB` on `DatabaseProvider`) is lost.** Under F1b, there's only one resource per trait; to have two DBs you need two traits, even if they're identical.
  - Con: coherence rules explode — every `impl Eq for Foo` is now declaring a provider for an effect resource named `Eq`.
  - **Verdict: non-starter.** Breaks too much of the existing trait surface.

- **F1c. Two-tier: `pure trait` vs `effect trait`.** Keep `trait X { ... }` for predicates with no resource semantics (`Eq`, `Ord`, `Hash`, `Display`, `Iterator`). Introduce `effect trait X { ... }` (or equivalent keyword) — declaring this simultaneously creates a trait *and* an effect resource with the same name. Drop `effect resource R: X;` as a separate declaration form.
  ```kara
  effect trait DatabaseProvider {     // declares trait + resource in one go
      fn insert(self, k: String, v: String) -> Result[(), String];
      fn get(self, k: String) -> Option[String];
  }
  impl DatabaseProvider for InMemoryUserDB { ... }   // declares a provider
  ```
  - Pro: fuses the common case (one trait → one resource) into one declaration.
  - Pro: preserves non-resource traits unchanged. `Eq`, `Ord`, `Iterator` stay as `trait`, no tax.
  - Pro: `impl DatabaseProvider for Foo` is *unambiguously* a provider declaration — the effect-trait keyword signals it.
  - Pro: matches how `shared struct` already works in the language — an additive keyword that opts into richer semantics on top of the base concept.
  - Con: **loses the two-resources-one-trait case** (`UserDB` + `AuditDB` both bound to `DatabaseProvider`). Need an escape hatch — see F3 below.
  - Con: adds a third top-level declaration keyword alongside `trait` and `effect resource`. Strictly two keywords (`trait`, `effect trait`); the old `effect resource R: X;` form would be deprecated — so net zero keyword count, but migration cost is real.
  - Con: "effect" is overloaded in Kāra — it appears in `with <effects>`, `effect resource`, and now `effect trait`. A reader needs to learn three compound forms.

- **F1d. Additive modifier on existing traits.** Keep `trait X { ... }` as-is. When a trait should also serve as an effect resource, say so inline: `trait DatabaseProvider as effect resource { ... }` (or a post-declaration `effect resource from DatabaseProvider;` shortcut).
  ```kara
  trait DatabaseProvider as effect resource {
      fn insert(self, k: String, v: String) -> Result[(), String];
  }
  // auto-synthesizes an effect resource of the same name
  ```
  - Pro: adjacent to the existing `trait` keyword — a reader who knows `trait` sees one extra phrase.
  - Pro: every non-resource trait pays zero tax.
  - Pro: backwards-compatible — existing programs compile unchanged; fusion is opt-in per declaration.
  - Con: same limitation as F1c on two-resources-one-trait — still needs F3's escape hatch.
  - Con: syntactic noise. `as effect resource` in every DI trait header is more ceremony per declaration than F1c's `effect trait`, though the total surface across a program is similar.
  - Con: doesn't *simplify* the language surface; it relocates the duplication from two declarations to one declaration with a modifier.

- **F1e. Nominal-only fusion.** Keep the semantics of today's pair unchanged, but allow a single declaration as sugar:
  ```kara
  trait DatabaseProvider with effect resource UserDB { ... }
  ```
  desugars exactly to the current pair. Nothing deep changes; the code generated and the mental model are identical to F1a.
  - Pro: purely a writing convenience. Zero semantic shift.
  - Pro: fully preserves the two-resources-one-trait case (just write two sugared forms, or fall back to the explicit `effect resource` form).
  - Con: **doesn't fix the conceptual duplication** — readers still need to know both concepts. It just lets them write one line where they used to write two.
  - Con: a pure `karac fmt` concern, not a language-design concern. If this is the answer, it's the weakest possible case for changing anything.

**Provisional lean: F1c.** It does the most work with the least new machinery. Strikes the cleanest balance:
- Eliminates duplication for the common case (one trait → one resource).
- Pays zero tax for non-resource traits (huge prelude surface stays as-is).
- The `effect trait` keyword makes the resource semantics locally visible at the declaration site (LLM-friendly).
- The multi-resource case is the only cost, and F3 below shows it's cheap to solve.

**F1d is a serious alternative** — specifically because it's *additive* rather than a new top-level keyword, and because migration is per-declaration rather than wholesale. The weakness is that it doesn't reduce the language surface; it just relocates it.

**F1a stays viable** if the design verdict is "this duplication is the price of keeping predicates and identities separate, and that's the right call." The argument for F1a is philosophical (traits-are-predicates, resources-are-identities) rather than practical, but it's not dismissible.

### F2. Non-resource traits under fusion  *(depends on F1)*

Kāra's prelude (design.md:501) currently includes `Option`, `Result`, `Vec`, `Slice`, `Map`, `Set`, `String`, `StringSlice`, `Display`, `Debug`, `Eq`, `PartialEq`, `Ord`, `PartialOrd`, `Hash`, `Add`, `Sub`, `Mul`, `Div`. Eleven of these are traits; none of them have useful resource semantics.

**Under F1a:** these stay as `trait`. No change.
**Under F1b:** every one of them gets pulled into the effect-resource world. **Non-starter confirmed.**
**Under F1c:** they stay as `trait`. Only traits explicitly declared with `effect trait` gain resource semantics. The prelude keyword count increases by zero.
**Under F1d:** they stay as plain `trait`; no `as effect resource` modifier attached. Identical outcome to F1c for non-resource traits.

**Verdict:** F1c and F1d both handle this cleanly. The prelude is safe under either. F1b is the only option that runs into trouble here, and that trouble is conclusive.

### F3. Two resources sharing one contract  *(the hard case under F1c/F1d)*

Today: `effect resource UserDB: DatabaseProvider;` and `effect resource AuditDB: DatabaseProvider;` — two independent resources bound to the same trait. This pattern appears in the canonical example (`user_db_test.kara:18–19`).

Under F1c's `effect trait DatabaseProvider { ... }`, the trait and its same-named resource fuse. What happens when you want *two* resources?

**Options:**

- **F3a. Keep `effect resource R: X;` alongside `effect trait`.** An `effect trait` declares both a trait and a default resource of the same name; if you want additional resources bound to the same trait, you use the existing `effect resource` form. Two syntaxes for the two cases.
  - Pro: solves the problem with a mechanism that already exists.
  - Con: undoes half the simplification F1c was supposed to buy. A reader must still know both `effect trait X { ... }` and `effect resource R: X;` — the concept count isn't reduced, just the common case is cheaper.

- **F3b. Named-instance syntax.** `effect trait DatabaseProvider { ... }` creates a contract only; resources are named separately with a concise form.
  ```kara
  effect trait DatabaseProvider { ... }
  effect UserDB: DatabaseProvider;    // a resource of that contract
  effect AuditDB: DatabaseProvider;
  ```
  The `effect resource` keyword becomes just `effect`. Trait-less ambient resources (`effect Uart0;`) use the same keyword without a bound.
  - Pro: cuts "effect resource" to "effect" everywhere — one keyword instead of two. Reads more naturally in effect rows (`writes(UserDB)` ties back to `effect UserDB;`).
  - Pro: every resource — bound or unbound — uses the same declaration form.
  - Con: repurposing the `effect` keyword. Today `effect` appears inside `with` clauses only (inside type annotations). Making it a top-level declaration keyword is a non-trivial shift.
  - Con: "a trait whose name is also a resource" vs "a trait that is only a contract" now depends on whether a subsequent `effect NAME: Trait;` exists — a non-local property.

- **F3c. Multi-resource-per-effect-trait.** `effect trait DatabaseProvider { ... }` declares one trait and *no* automatic resource. You declare as many resources as you want, separately.
  ```kara
  effect trait DatabaseProvider { ... }
  effect resource UserDB: DatabaseProvider;
  effect resource AuditDB: DatabaseProvider;
  ```
  This is just F1a with the `effect` prefix on `trait` signaling "this trait carries resource semantics." The simplification is purely informational (the trait declaration tells you it's injection-shaped).
  - Pro: no semantic shift from today — purely an annotation.
  - Con: same objection as F1e: it relocates duplication, doesn't remove it.

**Provisional lean: F3a.** Accepts the hard truth that *some* programs want two resources of one contract, and keeps a mechanism for it. The common-case simplification from F1c is preserved (one trait, one resource, one line); only the uncommon case requires a second declaration. F3b is attractive but re-labelling `effect resource` as just `effect` is a non-trivial syntactic shift that warrants its own evaluation.

**Note:** whichever F3 option wins, the count of "resources-only, trait-less" declarations in realistic programs (`Uart0`, `Spi1`, ambient program-rooted resources) is small — maybe 10–20 across the whole stdlib. The multi-resource-per-trait case is also bounded. The total declaration count difference between options is <1% of a typical program. **This matters — it means F3 is close to a coin flip.**

### F4. Impl, retroactive impls, orphan rules  *(depends on F1)*

**Today** (design.md:1782): `impl Trait for Type` requires *either* the trait or the type to be defined in the current crate. Retroactive impls (implementing stdlib `Display` for your own type, or *your* trait for a third-party type) are supported.

**Under F1a:** unchanged.

**Under F1c:** `impl DatabaseProvider for ThirdPartyType` is legal iff your crate defines `DatabaseProvider` or `ThirdPartyType`. This is fine. The resource identity is tied to the trait's defining crate, which matches the orphan rule's existing locus.

**Retroactive-impl subtlety.** Today, if you impl someone else's trait for your type, you're just adding a conformance. Under F1c, impling someone else's `effect trait` for your type makes your type a *provider* for their resource. Is that a new capability? A new risk?

- **Capability:** you can now inject your type into their `with_provider[TheirResource](your_type, ...)` scope. That's what providers are for. No new risk.
- **Risk:** none surfaced yet. Providers always required a type implementing the trait; fusion doesn't change who can declare such impls.

**Verdict:** F4 is not a distinctive question. Fusion preserves the current orphan rule and retroactive-impl capability unchanged.

### F5. Effects declared by fused trait methods  *(depends on F1c/F1d)*

Under fusion, a method declared in an `effect trait` is both a trait method *and* the thing callers invoke via `R.method(...)` under an effect row. What are its effects?

Three interpretations:

- **F5a. Method effects stay declared per-trait.** The trait declares each method's effects exactly as today (design.md:1969). The effect row at the call site includes both the trait method's declared effects *and* the resource itself (`writes(R)`).
- **F5b. The resource subsumes the method's effects.** Calling `R.insert(...)` contributes only `writes(R)` (or the appropriate verb) — the per-method declared effects collapse into the resource's effect.
- **F5c. Per-method verb choice.** The `effect trait` method declares which verb it contributes (`reads(R)` / `writes(R)` / `sends(R)` / `receives(R)`).

Today's language (design.md:4532) implicitly follows F5c — the conformance-checking rule at design.md:1969 says "each trait's effect ceilings are declared on its own methods." Fusion should preserve this. **F5c is the provisional verdict.**

### F6. Relationship to D1a (separated `impl` blocks — from v44)  *(cross-cutting)*

v44 D1a closed with "struct, inherent impl, and trait impls stay as independent blocks." D1d (this doc) operates on a different axis — *what a trait is*, not *where impls live*. Consequently, D1d and D1a are orthogonal: all F1 options above preserve the separated-impl shape.

Concrete check against the canonical sample (`postgres_query_injected.kara`):

- Today (lines 22–27): `trait DatabaseProvider` + `effect resource UserDB: DatabaseProvider;` — two declarations.
- Under F1c: `effect trait DatabaseProvider { ... }` — one declaration replacing two. `impl DatabaseProvider for PostgresUserDB` at line 45 is unchanged.
- `providers { UserDB => ... }` at line 69 is unchanged — `UserDB` is the declared resource name, now the trait name too.

No D1a decision is disturbed. **Fusion and separated-blocks coexist cleanly.**

### F7. Migration surface  *(if fusion wins)*

Fusion is a syntactic *and* semantic change. What migrates:

- **Spec doc (`docs/design.md`):** lines 4522–4540, 4545, 6859–6946 (every `effect resource R: X;` example). Rewrite to show the fused form alongside the explicit form, or pick one canonical shape.
- **Examples (`examples/tests/user_db_test.kara`):** rewrite the trait + two resource declarations to use the fused form. ~5 lines saved.
- **Design studies (`design_studies/db_read/postgres_query_injected.kara`):** rewrite similarly. ~3 lines saved.
- **Compiler (`src/`):** parser must accept `effect trait` (or the chosen spelling); AST gains a variant; resolver / effect-checker route fused declarations through their existing `effect resource` pathways. Estimated: ~80–150 LOC, one CR.
- **Runtime:** no change; the fusion is a front-end concern.

**If F1d (additive modifier):** migration is opt-in. No existing program changes until its authors choose to rewrite. Spec grows an optional form; no examples need to change immediately.

**If F1c (new keyword):** the old form can be deprecated gracefully over a version cycle, or removed in a single pre-v1 breaking change. Given Kāra is pre-v1, a clean break is fair game.

**Verdict:** migration is bounded under either F1c or F1d. F1d is the lower-cost path if we're uncertain; F1c is the higher-payoff path if we're committed.

## Related questions (raised, not resolved here)

These surfaced while writing this brainstorm. Each should NOT hold up F1.

- **Does the `Send + Sync` bound on providers (design.md:4548) move when a trait becomes an `effect trait`?** Most likely it stays as a bound at the `with_provider` call site, not attached to the `effect trait` declaration — matches today's shape.
- **`with_provider[R]` signature under fusion.** The `R: effect resource` bound (design.md:4553) reads slightly oddly if `R` is now "an effect trait." Probably fine as-is; could be renamed to `R: effect` for cleanness, but that's T-1-shaped.
- **Do "ambient program-rooted resources" (`FileSystem`, `Network`, etc., per design.md:4596) fit under the fused keyword?** They have no trait bound. Under F3a they remain `effect resource FileSystem;` unchanged. Under F3b they'd be `effect FileSystem;` without a bound. Either works.
- **Documentation shift.** If F1c lands, the book's "traits and effect resources" chapter collapses into one chapter. If F1d lands, the chapter stays but gains an optional-modifier section. Minor.

## Provisional implication

**Lean (unresolved): adopt F1c + F3a + F5c (today's per-method effect declarations) + preserve F4 orphan rules unchanged.**

Concrete shape:

```kara
// Fused: trait + same-named resource in one declaration
effect trait DatabaseProvider {
    fn insert(self, k: String, v: String) -> Result[(), String] with writes(DatabaseProvider);
    fn get(self, k: String) -> Option[String] with reads(DatabaseProvider);
}

impl DatabaseProvider for InMemoryUserDB { ... }   // declares a provider

// Multi-resource case: still available via the existing declaration form
effect resource UserDB: DatabaseProvider;
effect resource AuditDB: DatabaseProvider;

// Non-resource traits: unchanged
trait Ord { ... }
trait Iterator { type Item; fn next(mut ref self) -> Option[Self.Item]; }
```

**What this means concretely:**

1. **design.md §Effects / §Traits / §Provider-Rooted Resources** — add the `effect trait X { ... }` form as the canonical one-trait-one-resource shape. Keep `trait X { ... }` + `effect resource R: X;` as the two-resources-one-trait fallback.
2. **Examples** — rewrite `user_db_test.kara` and `postgres_query_injected.kara` to use the fused form where only one resource per trait is needed.
3. **Implementation** — one CR, parser + AST + resolver changes. No runtime changes.

**What to watch:**

- First real stdlib trait that wants to be an `effect trait`. If `DatabaseProvider` is the only one and every other trait stays plain, fusion may be over-engineered for one case. If three or four stdlib traits want it (e.g., `Logger`, `Clock`, `RandomSource` as trait-bound rather than ambient), F1c is validated.
- Whether F3a's "two forms" feels like regression at the point of use. If every reader of `postgres_query_injected.kara` asks "why two forms?" the same way they asked "why `trait` + `effect resource`?", F1c hasn't solved the problem — it's only relocated it.
- The first programs that need *four or more* resources of one trait (e.g., `UserDB`, `AuditDB`, `ShardDB[i]`, `ReplicaDB`). At that point F3a's explicit form is the shape being reached for anyway.

**Open alternative:** F1d (additive `as effect resource` modifier). If the migration cost of F1c is perceived as too high, or if keeping `trait` as the single contract-declaring keyword is considered load-bearing, F1d reaches the same payoff at the cost of per-declaration syntactic noise. F1d should be held as a live alternative until F1c's migration is scoped concretely.

**Resolution path:** once F1 converges, migrate resolved bullets to `docs/design.md` (Effects + Traits sections) and `docs/implementation_checklist.md`; mark this doc RESOLVED and archive per `memory/feedback_brainstorming_close_for_good.md`.
