## 1. **Resolved** — Ambiguous phrasing: "observed by a caller" vs "a caller exists"

**Section:** Specification Layers (~line 112 vs 121)

**Problem:** Line 112 states parameter modes are guaranteed "once it has been observed by a caller" (passive voice, unclear boundary), while line 121 clarifies "Once a caller exists, the mode is guaranteed." These are subtly different — does "observed" mean type-checked against, or just called?

**Suggested fix:** Replace line 112's "observed by a caller" with explicit language matching line 121, e.g., "once a call from external code has been type-checked against the function's signature."

---

## 2. **Resolved** — Missing edge case: private functions with declared effects

**Section:** Specification Layers (~line 111-121)

**Problem:** Line 111 states the effect set of a **public** function is guaranteed. Lines 120-121 discuss private functions' **inferred** effect sets as reported behavior. But what if a private function has an *explicit* effect declaration? Is that guaranteed like public, or reported?

**Suggested fix:** Add: "For private functions with explicit effect declarations, the declared set is guaranteed (it creates an internal API contract). If inferred, it is reported."

---

## 3. **Resolved** — Terminology inconsistency: "collect effects" vs "unify effects"

**Section:** Type Inference (~lines 183, 196, 210)

**Problem:** Line 183 (table) says "collect effects into E", line 196 (lambda rule) says effects "unify" with E, and line 210 (effect variables section) defines "unify" as single-assignment with restrictions. Three different framings for the same operation.

**Suggested fix:** Standardize on "unify" and reference line 210's single-assignment definition from both line 183 and line 196.

---

## 4. **Resolved** — Incomplete table: closure synthesis mode missing parameter modes and effect set

**Section:** Type Inference (~line 183-184)

**Problem:** The table row for "Closure `|x: A| body` with no context" states the result is `Fn(A) -> B with ...` but doesn't specify parameter modes or what `...` resolves to. Per lines 2148-2152, closure parameter modes are inferred from body usage — this should be stated.

**Suggested fix:** Expand entry: "result `Fn(A₁_mode, ..., Aₙ_mode) -> B with E_inferred`, where each mode is inferred from body usage and `E_inferred` is the effect set inferred from body."

---

## 5. **Resolved** — Lambda rule incomplete for synthesis mode parameter modes

**Section:** Type Inference (~line 196)

**Problem:** The lambda rule states parameters must have explicit types or be inferable from first use, but doesn't state whether parameter modes are inferred or default to `own` in synthesis mode.

**Suggested fix:** Add: "each parameter's mode is inferred from its uses in body by the consume-predicate scan (Part 4, item 2)."

---

## 6. **Resolved** — Effect variable conflict scope unspecified

**Section:** Type Inference (~line 210)

**Problem:** "A second use of the same `E` that resolves to a different effect set is an effect-variable-conflict error" — but doesn't specify the scope. Is it per call, per function, or per compilation unit?

**Suggested fix:** State: "per function call — each call to a function with an effect variable `E` gets a fresh instance, so conflicts are local to that call."

---

## 7. **Resolved** — Ambiguous `??` operator semantics

**Section:** Error Handling (~line 246)

**Problem:** "`??` provides a default: `expr ?? fallback`" — doesn't specify whether it applies to `Option` only, `Result` only, or both. Also doesn't state the type requirement for `fallback` or whether it short-circuits.

**Suggested fix:** Clarify: "`??` short-circuits to `fallback` if `expr` is `None` or `Err(_)`. `fallback` must unify with the inner `T` type."

**Section:** Module System (~line 352)

**Problem:** The `pub` row says "End users + all project files + same directory" — "same directory" is redundant since "all project files" already encompasses it.

**Suggested fix:** Simplify to "End users + all project files."

## 9. **Resolved** — Missing effect annotations in collection method signatures

**Section:** Standard Data Structures / Collection Core Methods (~lines 652-709)

**Problem:** The preamble states "All methods that grow the collection have the `allocates(Heap)` effect" but no method signature in the tables actually shows `with allocates(Heap)`. Methods like `push()`, `insert()`, `extend()`, `filled()`, `from_fn()` are all missing their effects.

**Suggested fix:** Either (a) add `with allocates(Heap)` to all growing methods' signatures, or (b) add a note: "Effect annotations omitted for brevity; all methods that grow or reallocate carry `allocates(Heap)`."

## 11. **Resolved** — Missing `Deserialize` and `Deserializer` trait definitions

**Section:** Derive / Serialization (~lines 1097-1116)

**Problem:** The `Serialize` trait and `Serializer` methods are shown in code blocks, but `Deserialize` and `Deserializer` are only mentioned in prose. An implementer cannot know the exact signatures.

**Suggested fix:** Add trait definition blocks for both: `trait Deserialize { fn deserialize(d: mut ref Deserializer) -> Result[Self, DeserializeError]; }` and show `Deserializer` methods (e.g., `deserialize_bool`, `deserialize_i64`, `deserialize_string`, etc.).

## 12. **Resolved** — Ambiguity: `filter` semantics with non-copyable items

**Section:** Iterator Adaptors (~line 1010-1040)

**Problem:** `filter`'s signature returns `impl Iterator[Item = Self.Item]` (yielding by value). For non-copyable types, it's unclear whether `filter` moves elements out of the underlying collection or creates a filtered view.

**Suggested fix:** Clarify whether `filter` moves or borrows. If it moves, explain partial-consumption semantics. If it's a view, update the return type to reflect borrowed items.

## 13. **Resolved** — Iterator consumption: `IntoIterator` trait undefined

**Section:** Loops / Iterator Traits (~lines 972, 995)

**Problem:** Line 995 says "`for` never consumes" and suggests `.into_iter()` to consume, but the `IntoIterator` trait is never defined in the Iterator Traits section. The trait, its return type, and its effect-system interaction are missing.

**Suggested fix:** Define `IntoIterator` in the Iterator Traits section with its relationship to `Iterator`, return type, and how it interacts with ownership.

## 14. **Resolved** — Unclear scope of "loop effects are static" with chained adaptors

**Section:** Loops (~line 997)

**Problem:** Loop effects are "static" and counted once, but the adaptor section describes effect-polymorphic closures. For a chain like `.filter(f1).map(f2)`, is the effect the union of all closure effects counted once at the loop level?

**Suggested fix:** Add an example showing effect composition in a chained iterator loop, clarifying the union-once rule.

## 15. **Resolved** — Refinement type narrowing at call sites underspecified for cross-refinement

**Section:** Refinement Types (~lines 1590-1627)

**Problem:** The spec covers `T -> R` (narrowing) and `R -> T` (widening), but not `R1 -> R2` (cross-refinement coercion, e.g., `Positive` where `NonZero` expected). The example at line 1627 suggests rejection, but no explicit rule is stated.

**Suggested fix:** Add a call-site example showing cross-refinement coercion being rejected and requiring explicit `try_from`.

---

## 16. **Resolved** — Contract purity rule: `{panics}` set notation ambiguous

**Section:** Contracts (~line 1761)

**Problem:** The effect constraint for contracts is `{panics}`, but earlier sections use effects with resources (e.g., `reads(X)`). It's unclear whether `{panics}` means only the bare `panics` effect or includes resource-scoped variants.

**Suggested fix:** State explicitly: "`panics` as a bare effect (not `panics(X)` with a resource) is the only effect allowed in contracts."

## 17. **Resolved** — Contract static discharge algorithm underspecified

**Section:** Contracts (~line 1830)

**Problem:** "When a contract predicate is expressible within the refinement type system's expressiveness... the compiler discharges it statically." No algorithm or heuristic is given for determining when this applies.

**Suggested fix:** Provide a formal rule, e.g., "if parameter has type `T` where `T = base where P`, and `requires Q`, and `P` syntactically implies `Q`, then discharge statically."

## 39. **Resolved** — RC "live range" definition imprecise

**Section:** RC Dataflow Specification (~line 3955)

## 42. **Resolved** — `sequential {}` block return value: statement vs expression ambiguity

**Section:** Auto-Concurrency (~lines 4262-4273)

**Problem:** The doc says the block value is "the value of the last statement" but then shows extracting a value from a bare expression. In Rust-family languages, the distinction between a trailing statement (`;`) and a trailing expression (no `;`) is significant.

**Suggested fix:** Clarify: "If the final element is a statement (trailing `;`), the block evaluates to `()`. If it is a bare expression (no `;`), the block evaluates to that expression's value."

## 44. **Resolved** — "Structural equality" for RC'd values undefined

## 45. **Resolved** — `collect_all` failure/cancellation semantics underspecified

**Section:** Concurrency Semantics (~lines 4441-4451)

**Problem:** `collect_all` has heterogeneous error types, but the doc doesn't specify: (a) its effect signature, (b) whether all branches run to completion or can short-circuit, (c) how cancellation interacts (does it return `Err(Cancelled)` for cancelled branches?).

**Suggested fix:** Add: "All `collect_all` branches run to completion regardless of failure. The return tuple preserves each branch's result in source order, including `Err(Cancelled)` for cancelled branches."
