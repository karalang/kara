# Brainstorming v30 — Struct literal type name: required or inferable?

**Status: RESOLVED 2026-04-18 — deferred post-MVP.** Captured as a P2 (open / speculative) entry in `docs/deferred.md § Struct Literal Type Prefix in Check Mode`. Current lean ~55/45 toward (b) (make the prefix elidable in check mode) on consistency grounds, but not strong enough to act on today. Must decide before `docs/book/src/` introduces struct literals to external readers (ch1/ch2 tutorials). Original analysis preserved below for future reference.

---

Triggered while reading `examples/word_count.kara:17-20`: `count_words` returns `-> WordCount` and its tail expression is

```kara
WordCount {
    total: words.len(),
    unique: unique_words.len(),
}
```

The `WordCount` prefix is redundant given the return type annotation. Question: can (should) it be elided when a target type is in scope?

---

## Background

Current spec:

- `docs/syntax.md §5.15`: `STRUCT_LITERAL = IDENT "{" FIELD_INIT { "," FIELD_INIT } [ "," ] [ ".." EXPR ] "}"` — the leading `IDENT` is grammatically required.
- `docs/design.md:272`: "Struct literal. In check mode against a known struct type, each field expression checks against its declared field type. In synthesis mode, the struct name must appear in the literal; the result is the nominal type."

So the typechecker *already* distinguishes check-mode from synthesis-mode. Today the grammar rejects prefix-less literals before the typechecker sees them; the type-inference machinery would already be sufficient to resolve them if the grammar allowed.

Check-mode positions where a target type is known:

- Return position of a function with declared return type: `fn f() -> T { ... T-valued tail ... }`.
- `let x: T = ...`.
- Argument of a call: `f(literal)` where the parameter type is `T`.
- `if`/`match` branches in check mode.
- Struct-literal field value: `Outer { inner: { ... } }`.

Synthesis positions (no target) remain: top-level `let x = ...` without annotation, `return` from a function whose return type is itself being synthesized (none in current Kāra — return type is always declared), method-chain intermediate results.

---

## Framings

- **(a) Status quo.** Struct name is always required. `WordCount { ... }` at every construction site.
- **(b) Prefix optional in check mode.** `{ total: ..., unique: ... }` is legal when a unique target struct type is known; synthesis sites still require the prefix. Error if target is ambiguous (e.g., generic metavariable, `Any`-like position — though Kāra has no `Any`).
- **(c) Separate "anonymous record" syntax.** Some other brace form (e.g., `.{ ... }` à la Zig, or `@{ ... }`) signals "elide the name, use context." Keeps `Foo { ... }` unambiguously nominal.

---

## Arguments for (b)

- **Removes redundant writes.** The return-type annotation and the literal prefix state the same fact. DRY.
- **Aligns with the field-order spec.** `docs/design.md:2886` already mandates source-declaration order for struct literals, meaning the brace-list is already tightly bound to one specific struct's shape — not a free-form record.
- **Helps refactors.** Renaming a struct touches fewer call sites. (Minor; IDE rename handles this.)
- **Mainstream precedent.** C# target-typed `new()` (2020), Java `var` + records, Swift `.init(...)` on a typed target. The "semantics-Rust, syntax-mainstream" tiebreaker (per `MEMORY.md`) leans toward eliding when the target is known.

## Arguments for (a)

- **Local readability.** A reader looking at a struct literal knows the type without tracing outward to the return signature, let-binding, or enclosing call. For long functions or deeply nested literals this matters. **This is the only argument for (a) that is not undercut by other Kāra rules.**
- **Consistent with Rust.** Rust rejects bare `{ ... }` as a struct literal even when the target is known. Widely accepted by Rust users. *Weak* — the "Rust without being Rust" principle says syntax-mainstream breaks ties away from Rust when semantics aren't at stake.
- **Error-message quality.** `{ total: 42, foo: "x" }` in a position where two in-scope structs have overlapping fields requires a new class of diagnostics ("ambiguous target; candidates: `WordCount`, `PageStats`"). Solvable, not a blocker.
- **Macro / generated code.** Code that doesn't know its enclosing context must still write the prefix. Making it optional creates a stylistic fork. *Weak* — stylistic forks already exist (e.g., explicit vs. inferred generics).

## Arguments for (c)

- **Keeps nominal-by-default.** `Foo { ... }` stays unambiguous; `.{ ... }` is opt-in context-dependent.
- **Parser doesn't need target-awareness for the positive path.** `.{` is a distinct lexeme; `{` alone stays a block.
- **But:** introduces a second syntax for the same concept. Cost paid forever; benefit rare.

---

## Consistency check against existing Kāra rules

Kāra already uses check-mode inference in several places. The table shows whether each existing rule points toward (a) keeping the struct prefix mandatory, or (b) making it optional:

| Existing Kāra rule | Direction |
|---|---|
| Generic type arguments inferred from call-site context (`Vec.filled(5, 0) → Vec[i64]`, `docs/design.md:281`) | **(b)** |
| Integer/float literal type inferred from target type in check mode (`docs/design.md:264`) | **(b)** |
| Closure parameter types inferred from context or first use; grammar accepts, typechecker errors if unresolved (`docs/design.md:271`) | **(b)** — identical shape to what (b) would introduce |
| Enum-variant construction (`Some(x)`) requires the variant name | Neutral — variant name is load-bearing (selects the discriminant); struct name is metadata |
| Pattern match `Point { x, y }` requires the prefix | Neutral — patterns disambiguate against variants; expression-position literals don't |
| `[T]` vs `<T>` (unambiguous-by-grammar principle) | Mildly **(a)**, but solvable the same way closure params are: grammar accepts `{ ... }`, typechecker rejects in synthesis position |
| "Semantics-Rust, syntax-mainstream" tiebreaker (`MEMORY.md`) | **(b)** — C# target-typed `new()`, Swift `.init(...)` on typed target, Java record target-typing |

**Reading of the table:** requiring the struct prefix in check mode is the odd-one-out rule. Every other check-mode position in Kāra elides redundant annotations. The strongest argument for (a) is not consistency — it's the local-readability argument above.

## Other interactions

- **Field-init shorthand** (`Point { x, y }`) composes cleanly with (b): `{ x, y }` in check-mode against `Point`.
- **Struct update** (`..existing`): under (b), `{ name: "Bob", ..user }` could infer the struct type from `user` *or* from the target context. Worth spelling out which wins if they disagree (probably: both must agree, else compile error).
- **Nested generics.** In `let v: Vec[Foo] = [...literals...]` the target element type is `Foo`, so each literal gets prefix elision. No new rule needed.
- **Identifier case classes (CN-1..CN-7).** Irrelevant — the decision is about whether the Type identifier is required, not about its form.

---

## Not an argument

- **Parse ambiguity with blocks.** `{ x: expr, y: expr }` cannot be confused with a block because `IDENT ":"` at the head is not valid statement syntax (labels use a different form, if present). This is a solvable grammar problem, not a blocker.
- **Type inference complexity.** The typechecker already has check-mode infrastructure per `docs/design.md:260`. No new inference algorithm required.

---

## Current lean

**~55/45 toward (b), weakly held.** Consistency with existing check-mode inference rules is the deciding factor — the struct-literal prefix is the only redundant annotation Kāra currently mandates in a check-mode position. (c) is not worth a second syntax.

The only strong argument for (a) is local readability, which matters more in long functions than in short ones. That argument isn't strong enough to override the consistency story, but it's strong enough that the decision isn't urgent — can revisit once real Kāra code exists and we can count how often the prefix is redundant vs. load-bearing for readability.

---

## Resolution path

- **Proposal:** park in `docs/future_decisions.md` as a pre-1.0 ergonomic candidate (like v29 before the spike), not actively pursued until we have a larger corpus of Kāra code to evaluate redundancy empirically.
- **If pursued:** spike on a branch — pick a representative set of `.kara` examples, strip struct-literal prefixes where check-mode allows it, read the result. Convert the 55/45 into evidence.
- **Backstop:** must decide before `docs/book/src/` introduces struct literals to external readers in any tutorial, because the syntax shown in ch1 becomes load-bearing for muscle memory.

---

## Out of scope

- Tuple-struct construction (`Point(1.0, 2.0)`) — separate question; function-call syntax has no prefix to elide.
- Enum-variant construction (`Some(x)`, `IoError::InvalidUtf8`) — separate; variant name is load-bearing for discriminant selection.
- Anonymous records / structural types — Kāra is nominal; out of scope unless the language design changes.
