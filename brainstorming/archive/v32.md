# Brainstorming v32 — Call-site mutation markers and signature-level ownership annotations

**Status: RESOLVED 2026-04-18.** Q1 → **1A** (call-site `mut` required for fresh bindings passed to `mut ref T` / `mut Slice[T]` params; `ref` at call sites rejected). Q2 → **2A** (signatures always name the mode explicitly — one unified rule across free functions, methods, trait declarations, and receivers: default (owned) is bare, borrows/mutation are written). Sub-clarification: `own self` is dropped as a form; bare `self` is the owned/consuming receiver. Fresh-vs-forwarded rule is **rooted at the argument's place-expression base binding**, not the expression's own type — field projections and index expressions through a `mut ref` root forward unmarked. Full analysis, cases, and action items preserved below.

---

Two related but separable design questions need resolution before the book's ch12-ownership chapter stabilizes and before the typechecker pins call-site rules.

## Problem statement

Kāra infers parameter ownership modes (`own` / `ref` / `mut ref`) from a function body's syntactic use of each parameter (`docs/design.md` Part 1 — Parameter Mode Inference). Signatures today still carry explicit mode annotations: `fn sort_in_place(xs: mut Slice[T])`, `fn fmt(ref self, f: mut ref Formatter)`. Inference runs for checking purposes; it also underwrites the stability guarantee (`docs/design.md:114`) that inferred modes are part of the public contract.

Two open questions:

**Q1.** At the *call site*, when a caller passes a fresh variable binding to a function whose parameter mode is `mut ref T` (or whose type is `mut Slice[T]`), should the caller write a `mut` marker?

```kara
let mut v = [3, 1, 4, 1, 5];
sort_in_place(mut v);    // (A) marker required
sort_in_place(v);        // (B) marker optional / not required
```

**Q2.** At the *signature site*, when the mode can be determined from the body, should the signature elide the keyword?

```kara
fn process(data: ref Vec[i64]) -> Summary { data.iter().sum() }   // (A) explicit — today
fn process(data: Vec[i64]) -> Summary     { data.iter().sum() }   // (B) inferred — mode derived from body
```

The questions are related (both involve how mutation and borrowing are surfaced syntactically) but separable (a signature-explicit rule can pair with either call-site rule, and vice versa).

## Triggering evidence

- `examples/slice_basics.kara:47` calls `double_each(nums)` without a marker — but `double_each` takes `mut Slice[i64]` and `docs/design.md:2303` shows the canonical form as `sort_in_place(mut v)`. The typechecker currently accepts both; the example and the spec disagree.
- `docs/book/src/ch12-ownership.md` introduces receiver modes with explicit `mut ref self` throughout. If Q2 resolves toward inference, that whole section needs rewriting. If toward explicit, it stays as-is.
- `examples/leetcode/coin_change.kara:76, 83` have stray `ref` markers at call sites (`coin_change_memo(amount, ref coins)`). The spec has no rule permitting `ref` at call sites. These need to resolve one way or the other.

## Constraints from existing design

Four facts constrain the answer space:

**C1. Trait declarations must carry explicit modes.** A trait method has no body — inference has no input. `design.md` already uses explicit `mut ref self` in 50+ stdlib trait signatures (Vec / Map / Set / Deque method tables at lines 809–884; Iterator at 1156; Serialize / Deserialize at 1285–1308; Subscript at 2745; Display at 348; log.Backend at 3202). This is structural, not stylistic — inference cannot derive a signature from nothing.

**C2. Borrow-return source parameters must carry explicit `ref`.** `docs/design.md` Part 3:

```kara
fn longer(a: ref String, b: ref String) -> ref String {
    if a.len() > b.len() { a } else { b }
}
```

The tail-position bare identifiers `a` / `b` classify as *consume* under the use-predicate table (`design.md:4503`). Without explicit `ref`, inference picks `own` — contradicting the `ref String` return. The Part 3 opt-in design depends on the explicit annotation.

**C3. Method calls and field assignment never carry mutation markers.** `v.push(x)` mutates `v` silently. `s.field = 5` mutates `s` silently. `v[i] = x` desugars to `IndexMut` silently. Any rule that requires a marker on method calls would be a major surface break.

**C4. The stability guarantee (`design.md:114`) treats inferred public modes as part of the API.** A body edit that shifts `ref` → `mut ref` is a breaking change for callers. Today the explicit annotation in the signature pins the mode; if signatures elide the annotation, the stability contract needs a different enforcement mechanism (a diagnostic on mode shift, or a manifest of previous modes).

## Question 1 — Call-site `mut` marker

### Options

- **1A. Required for fresh bindings to mutating params.** `sort_in_place(mut v)`. Method receivers and field assignments still unmarked (C3). Forwarded mut-refs (arguments that are already `mut ref T` / `mut Slice[T]` in the current scope) unmarked — they're already a borrow, the original caller announced the mutation.
- **1B. Never required.** `sort_in_place(v)`. Signature / inference determines the mode; marker carries zero information the compiler needs.
- **1C. Required for all mutating calls, including method calls.** `v.mut.push(x)`. Breaks C3.

1C is off the table on C3. The real choice is 1A vs. 1B.

### The case for 1A

- **Scannability.** The single most-asked question in a codebase is "where does this variable get mutated?" `f(mut v)` answers it at the call site; `f(v)` forces the reader to jump to `f`'s signature (or body, if the mode is inferred). Scales with codebase size.
- **Fresh / forwarded asymmetry has a reason.** A fresh binding crossing a mutation boundary is news — the caller's variable may change because of this call. A forwarded mut-ref is already a borrow — the mutation surface was announced at the root caller. Marking only fresh bindings aligns ceremony with information content.
- **Locks inference visibly for the caller.** If a private function's body edit shifts an inferred mode from `ref` to `mut ref`, every call site currently written `f(v)` becomes a compile error under 1A (now needs `f(mut v)`). Callers acknowledge the change explicitly — no silent drift.
- **Rust `&mut x` precedent.** Familiar to newcomers coming from Rust. Three characters; modest cost.

### The case for 1B

- **Consistency with methods and fields.** `v.push(x)`, `s.field = 5`, `v[i] = x` all mutate without markers (C3). Requiring a marker only on free-function calls is an asymmetry readers must internalize.
- **`let mut v` already declared the binding as mutable.** A second marker at the call site is belt-and-suspenders.
- **"Semantics-Rust, syntax-mainstream" tiebreaker** (memory — `project_design_principle`). Python, Java, Kotlin, Swift, Go, JS, C# do not mark mutation at call sites. Rust is the outlier.
- **Redundant with signature.** If Q2 lands at explicit signatures (2A), the mode is already visible to the reader without the call-site marker.

### Weighing

The scannability argument is concrete and scales. The consistency argument against it (method-call / field-assignment asymmetry) is real but can be reframed: methods and field assignment are mutation *through an established binding or borrow* — the mutation surface is on the receiver / base object. A free-function call with a fresh binding is *handing a variable to external code* — categorically different.

The fresh / forwarded distinction operationalizes this: method calls are "forwarded" by definition (the receiver is the base), so no marker. Free-function calls with fresh bindings are "initiating" mutation of a variable, so marker. Field assignment is "in-scope mutation" — no external call, no marker. The three cases are not actually in the same category once you name the semantic distinction.

**Lean: 1A with the fresh / forwarded rule.**

### Fresh-vs-forwarded rule, stated precisely

The classifier is the **root of the argument place expression**, not the argument's own type. This is the key refinement — classifying on the expression's own type would require marking every field-through-a-`mut-ref` access (`update_map(mut s.m)` inside `fn helper(s: mut ref S)`), defeating the rule's purpose. Rooting at the place expression's base binding keeps the "mutation boundary is at the signature" story coherent: once the function has a `mut ref` parameter, mutations *through* that parameter are in-scope, not cross-boundary.

- **Fresh.** The argument is a place expression whose root is a local owned binding (`let v = ...` or `let mut v = ...`), or a non-place expression (temporary, literal, function return, block). `let mut v = [...]; f(v)` → fresh → `f(mut v)`. `f(mut make_vec())` → fresh (temp) → marker required.
- **Forwarded.** The argument is a place expression whose root is a `mut ref T` / `mut Slice[T]` binding in the current scope. Field projections, tuple indices, index expressions, and deref chains rooted at a mut-ref binding all forward. `fn helper(s: mut ref S) { other(s.m); update(s.inner.list[0]) }` — no markers; the mutation surface was announced at `helper`'s signature.
- **Method receivers.** Never marked. `v.push(x)` is `push(mut ref v, x)` semantically, but the `v.` syntax signals method dispatch and the marker is suppressed by convention (C3).
- **UFCS form.** `User.validate(v)` is a free-function call syntactically. First-argument marker rule applies. If `validate` takes `mut ref self`, write `User.validate(mut v)` (assuming `v` is fresh; if `v` is already a `mut ref User`, write `User.validate(v)` — forwarded). This creates a surface asymmetry with `v.validate()` (same call, no marker) — accepted as the documented consequence of the two syntactic forms. The mismatch diagnostic on `User.validate(v)` where `v` is fresh should suggest the method-call form (`v.validate(...)`) as an alternative.
- **Nested mut-ref returns are forwarded, not re-marked.** `other(wrap(mut v))` — inner `mut v` required (fresh); outer call unmarked because `wrap`'s return is already a `mut ref`. Writing `other(mut wrap(mut v))` is a common mistake; the diagnostic should say *"this argument is already a mut-ref; remove `mut`."*
- **Closures / method references capturing mutably.** The mutation boundary is the capture point, not the call of the closure. `let c = || v.push(5); c()` — no marker on `c()` even though calling `c` mutates `v`. This is an inherent consequence of capture semantics and applies under any call-site rule; not a 1A-specific hole. Spec should acknowledge it once.
- **`ref` at call sites is never legal.** Immutable borrows pass unmarked. The keyword `ref` in argument position is a parse error with a diagnostic pointing at the dropped form. Rationale: the signature declares `ref`, the call site doesn't need to repeat it, and allowing it creates a second spelling for the same call.

## Question 2 — Explicit vs. inferred modes in signatures

### Options

- **2A. Always explicit.** Today's state. `fn f(x: ref T)`, `fn f(x: mut ref T)`, `fn f(ref self)`, etc. Inference still runs internally for checking; it's just not surfaced at the signature.
- **2B. Always inferred.** Signatures elide mode keywords; the compiler derives the mode from the body.
- **2C. Mixed.** Explicit in positions where inference cannot reach (trait declarations, borrow-return sources); inferred elsewhere.

### 2B is ruled out by C1 + C2

Trait declarations have no body (C1). Borrow-return source parameters need explicit `ref` to satisfy source pinning (C2). 2B cannot be universal. The real choice is 2A vs. 2C.

### The case for 2A (always explicit)

- **Symmetric with the structural constraints.** Traits are explicit (C1). Borrow-return sources are explicit (C2). If those positions are explicit, making *all* positions explicit gives one rule: "signatures name the mode." Zero exceptions to teach.
- **Public API stability is trivial.** The author chooses the mode; body edits that would imply a different mode become typechecker errors against the declared signature. C4's stability guarantee is preserved by construction.
- **No stdlib churn.** 50+ trait signatures, Part 3 examples, and ch12 receiver-mode examples all stand as-is.
- **Call-site marker (1A) plus explicit signatures is self-consistent.** The reader sees the mode in two places: the signature and the call site. One reinforces the other.
- **Zero magic.** The reader never needs to open the body to understand the interface. `karac explain` becomes nice-to-have rather than load-bearing.

### The case for 2C (inferred where possible)

- **"Don't hand-write what inference can determine"** matches Kāra's design direction on parameter modes.
- **Free-function signatures become terser.** `fn process(data: Vec[i64]) -> Summary` reads closer to mainstream languages than `fn process(data: ref Vec[i64]) -> Summary`.
- **Mode-shift diagnostics (for C4) are tractable.** The compiler can maintain a manifest of previously-inferred public modes and fire a diagnostic on shift.

### Against 2C

- **Teaching story fragments.** Readers must learn: inferred for bodyful free functions and non-self method params; explicit for trait declarations; explicit for borrow-return source params; explicit for self receivers (if kept, per the stdlib pattern). Four positions, three rules. 2A is one rule.
- **Public API stability needs new infrastructure.** Mode-shift diagnostics require cross-version metadata — not trivial, and the failure mode is silent until the diagnostic ships.
- **Marginal ergonomic win.** `ref` in a signature is three characters. The stdlib already uses it extensively; rewriting for terseness is not obviously worth the churn.
- **The "mainstream syntax" argument cuts less than it looks.** Mainstream languages don't have borrow semantics at all — they have garbage collection or value types. Kāra has borrows; making them invisible doesn't make Kāra read like Python, it makes Kāra read like Rust-with-hidden-borrows. The actual comparable languages (Rust, Swift with `inout`, C# with `ref`) all mark borrows explicitly in signatures.

### Weighing

2A wins on teachability (one rule), stability (no new infrastructure), and stdlib impact (zero churn). 2C wins on terseness (three characters per signature) and directional alignment with "inference as default."

The teachability and stability arguments are concrete and durable. The terseness argument is cosmetic. The directional-alignment argument has a counterpoint: Kāra's inference is a *checking* aid (derive what the body implies; verify it matches the declaration), not a *signature derivation* mechanism. The same rule applies to effect inference — private functions infer; public functions declare. Signatures are declared.

**Lean: 2A. Always explicit.**

### Sub-clarification — one unified rule across all signature positions

2A applies identically to free functions, methods, trait declarations, and receivers. The rule is:

- **Default (owned)**: bare. `x: T`, bare `self`.
- **Borrow / mutation**: always written. `ref T`, `mut ref T`, `mut Slice[T]`, `ref self`, `mut ref self`.

`own self` is dropped as a form. Bare `self` already means owned (consuming) receiver — writing `own` on it violates the default-is-bare rule and is the only place in the signature grammar where a default mode is spelled out. Today's spec uses both spellings (`fn map(self, ...)` in Iterator; `fn transfer(own self, ...)` in contracts; prose at `design.md:2519` says "if `validate` takes `own self`"). They're synonyms; canonicalize on bare `self`.

Three receiver modes, three spellings: `self`, `ref self`, `mut ref self`. Matches non-self params exactly.

## How Q1 and Q2 interact

Under the leans (1A + 2A):

- Signature declares `ref` / `mut ref` / `own`. Reader sees the mode directly.
- Call site announces mutation for fresh bindings passed to `mut ref T` / `mut Slice[T]` params. Reader sees mutation at the boundary.
- Method calls and field assignment stay unmarked — their mutation surfaces are in-scope, not cross-boundary.
- UFCS form follows free-function rules — explicit marker on fresh mutating args.
- Stability is preserved by the signature, not the marker; the marker is belt-and-suspenders for readers, not for the compiler.

The two rules are orthogonal in principle but mutually reinforcing in practice.

## Recommendation

- **Q1: 1A** — call-site `mut` required for fresh bindings passed to `mut ref T` / `mut Slice[T]` params, with the fresh / forwarded rule. `ref` at call sites rejected.
- **Q2: 2A** — signatures always name the mode explicitly. One unified rule across free functions, methods, trait declarations, and receivers: default (owned) is bare, borrows are written. `own self` is dropped; bare `self` is the owned/consuming receiver.

### Typechecker

- Accept `f(x)` for owned and borrowed-immutable params.
- Require `mut x` for fresh bindings passed to `mut ref T` / `mut Slice[T]` params. Structured diagnostic on omission: "function `f` takes `x: mut ref T`; call with fresh binding `v` requires `f(mut v)` to permit the mutation."
- Reject `ref x` at call sites: "Kāra does not write `ref` at call sites — the parameter's mode is declared on `f`'s signature. Remove the keyword."
- Reject `mut x` on non-mutating calls: "function `f` takes `x: ref T` (immutable borrow); `mut` marker is not legal here. Remove `mut`."
- Implement fresh-vs-forwarded by rooting at the argument's place-expression base: walk the argument; if it is a place expression, find its root binding; if the root binding is `mut ref T` / `mut Slice[T]`, the argument is forwarded (accept unmarked); otherwise (owned root, or a non-place expression like a temporary / literal / call return), it is fresh (require the marker when the parameter is mutating). Classification is on the root, not on the argument's own type — field projections and index expressions through a mut-ref root all forward.
- Reject double-marking on nested mut-ref returns: `other(mut wrap(mut v))` — the outer `mut` is wrong because `wrap`'s return is already a mut-ref. Diagnostic: *"`wrap(...)` already returns a mut-ref; drop the outer `mut`."*

## Out of scope

- **Attribute / suppression syntax** (`#[allow(...)]`). No lint under the recommendation; no need.
- **Mode-shift diagnostic across versions.** Not required under 2A — signatures are explicit, so a body edit that implies a different mode fails at the declaration, not silently. Revisit only if a future change moves toward 2C.
- **`karac explain` UX.** Useful tooling but independent of this decision. Under 2A, explain is a convenience, not load-bearing.
- **Method-receiver marker.** Pinned as "never carries a marker" per C3. Reopening would require a separate brainstorming.
- **Higher-order / closure parameter modes in signatures.** Closures follow the same inference rule as fns; explicit mode on closure parameters (`|x: ref T| ...`) remains legal but rare. Not in scope here.
