# Brainstorming v5 — Mainstream Language Sanity Check

A survey of Kāra against mainstream languages to catch any obvious gaps that
slipped past the research-language reviews. These languages are Kāra's prior art
or close relatives; the goal here is verification, not ideation — finding things
that are standard in the field but absent from the spec.

Languages surveyed: **Rust, Go, Swift, Kotlin, TypeScript, C++**.

Kāra already acknowledges most of these in its Prior Art table. The items below
are things that *slipped through* despite those languages being known references.

---

## Priority 1 — Likely Gaps (spec holes, not deliberate omissions)

### 1.1 `if let` / `let...else` Patterns ✓ RESOLVED

**Source:** Rust, Swift, Kotlin.

The spec defines `while let Pattern = expr { ... }` as one of the four loop
forms but **never mentions `if let`**. These are complementary: `while let`
repeats while a pattern matches; `if let` executes once when a pattern matches.
Without `if let`, every single-pattern match requires a full `match` block.

```kara
// Without if let — verbose for a single branch:
match user.find(id) {
    Some(u) => process(u),
    None => (),
}

// With if let — direct:
if let Some(u) = user.find(id) {
    process(u);
}

// With else — handles both branches without match:
if let Some(u) = user.find(id) {
    process(u);
} else {
    log("not found");
}
```

**`let...else` (the "early exit" form, Rust 1.65+, Swift `guard let`).**
`let...else` is the pattern for destructuring a value at the top of a function
and returning early if it doesn't match — the most common usage of `if let` in
systems code:

```kara
// let...else: binds on success, executes the else block (must diverge) on failure
let Ok(config) = load_config(path) else {
    return Err(ConfigError.Missing);
};
// config is in scope here as the unwrapped value

// vs. the match equivalent:
let config = match load_config(path) {
    Ok(c) => c,
    Err(_) => return Err(ConfigError.Missing),
};
```

The `else` block in `let...else` must **diverge** — return, break, continue, or
call a diverging function (`unreachable()`, `panic()`). The binding from the
pattern is only in scope after the `let...else` statement, not inside the `else`
block. This is the same rule as Swift's `guard let`.

**`if let` chains.** Multiple `if let` patterns in a single condition:

```kara
if let Some(user) = find_user(id)
   && let Some(addr) = user.address
   && addr.country == "US"
{
    process_us_order(user, addr);
}
```

This is less urgent than the basic `if let` form — add to a follow-up decision.

**Resolved:**
- `let...else` is a separate statement (not a variant of `if let`). Binding scopes after the statement; `else` must diverge.
- `if let` and `let...else` are in scope for v1 — added to design.md.
- `if let` chains — deferred. Added to future_decisions.md §7.

### 1.2 `Copy` Semantics for Primitive Types ✓ RESOLVED

**Source:** Rust, C++, every systems language.

The spec defines immutability (`let` vs `let mut`), owned returns by default,
and parameter mode inference — but **never states whether primitive types are
implicitly copyable**. This is a genuine hole: what happens when you pass an
`i64` to a function that (via parameter mode inference) takes ownership?

```kara
fn double(x: i64) -> i64 { x * 2 }

let n = 42;
let result = double(n);
print(n);      // ERROR or OK? Spec is silent.
```

In Rust, `i64` is `Copy` — passing it to `double` implicitly copies the value;
`n` is still valid after. Without a `Copy` equivalent in Kāra, parameter mode
inference would infer `own i64`, consuming `n`. The programmer would then need
to write `double(ref n)` or `double(n.clone())` for every integer pass — clearly
not the intent.

**The implicit answer:** Kāra must have some notion of trivially-copyable types.
The spec *implies* this (it says things like `let x: i64 = 42` and uses integers
freely in examples without any clone syntax), but never states it explicitly.

**Three design options:**

| Option | Description | Precedent |
|---|---|---|
| **`Copy` trait** | Types implementing `Copy` are silently cloned on move. `#[derive(Copy)]` for user types. | Rust |
| **`Trivial` / implicit clone** | A narrower version — only applies to types with no heap allocations. Not a trait; inferred structurally. | C++ trivially copyable |
| **No separation — use `Clone` everywhere** | No silent copies; programmer always writes `.clone()`. | Austral (strict) |

**Recommendation:** Introduce a `Copy` trait (or equivalent marker). Primitive
types (`i64`, `f64`, `bool`, `char`, all fixed-width integers/floats) implement
`Copy` automatically. User-defined structs can `#[derive(Copy)]` when all fields
are `Copy`. `Copy` implies `Clone`. Types with heap allocations (`String`,
`Vec[T]`) are not `Copy`.

Interaction with parameter mode inference: if a function takes a `Copy` type
parameter as `own`, the compiler implicitly inserts a copy at the call site
rather than consuming the caller's binding. No programmer annotation needed —
the Copy semantics are transparent.

```kara
// With Copy semantics:
let n = 42;
double(n);     // copies n (i64 is Copy)
print(n);      // OK — n still valid

// Without Copy semantics:
let s = "hello".to_string();
process(s);    // moves s (String is not Copy)
print(s);      // ERROR: s moved
```

**Interaction with `#[derive]`.** `Copy` should be derivable on structs/enums
where all fields are `Copy`. The compiler rejects `#[derive(Copy)]` if any field
is not `Copy`.

**Interaction with `distinct type`.** A `distinct type UserId = i64` where `i64`
is `Copy` — is `UserId` automatically `Copy`? Recommend: yes, if the underlying
type is `Copy` and the `distinct type` doesn't `#[derive(Clone)]` manually to
prevent copies. (The programmer can always choose to not implement `Copy` on a
distinct type to prevent accidental copying — e.g., a `SessionToken` should not
be `Copy` even if its underlying type is `u64`.)

**Resolved:** Added to design.md §Feature 4 Part 6. Decisions: name is `Copy`; implies `Clone` (must derive both); `#[derive(Copy)]` only — no manual impl; `Array[T, N]: Copy` when `T: Copy` (compiler built-in); `distinct type` does NOT inherit `Copy` — must opt in explicitly.

### 1.3 Once-Callable Closures (`FnOnce` Equivalent) ✓ RESOLVED

**Source:** Rust's `FnOnce`/`FnMut`/`Fn` distinction.

Kāra's spec defines one closure type: `Fn(T) -> U`. Rust has three, because
closures interact with ownership:

| Rust | Meaning |
|---|---|
| `Fn(T) -> U` | Can be called any number of times; captures by reference or copy |
| `FnMut(T) -> U` | Can be called any number of times; may mutate captured state |
| `FnOnce(T) -> U` | Can be called exactly once; may consume captured values |

**Why this matters for Kāra.** Consider a closure that consumes a captured value:

```kara
fn make_sender(conn: Connection) -> Fn(Message) -> () {
    |msg| conn.send(msg)   // conn is captured and consumed on first call
}

let send = make_sender(conn);
send(msg1);   // conn consumed here
send(msg2);   // ERROR: conn already moved — but how does the type system know?
```

Without a `FnOnce` distinction, the type system cannot express "this closure may
only be called once" — the compiler either rejects the closure definition
(because it moves a captured value) or silently accepts double-use (unsound).

**Kāra's current closure spec** says: "Closures carry the effects of operations
they perform when called, not of captured variables. Capturing a variable is an
ownership operation (handled by tiered ownership), not an effect." This
acknowledges that capture has ownership implications, but doesn't specify how
one-shot closures are typed.

**Design options:**

**Option A — Three closure traits (Rust model).** `Fn`, `FnMut`, `FnOnce` as
separate traits where `Fn: FnMut: FnOnce`. Most expressive; most complex.

**Option B — Single `Fn` trait with ownership marker.** `Fn[once](T) -> U` for
once-callable closures. Preserves the existing `Fn(T) -> U` for the common case.

**Option C — Infer from usage.** The compiler infers whether a closure is
once-callable from its body. At the type level, `Fn(T) -> U` always means
"callable at least once" — the number of calls is tracked by the ownership
system, not the type system. If a closure captures a non-`Copy` owned value
and moves it out, the compiler prevents it from being called twice via
ownership, not via a distinct type.

**Recommendation: Option C — infer from usage, track via ownership.** This
keeps `Fn(T) -> U` as the single closure type and uses Kāra's existing ownership
machinery to prevent double-calls of closures that consume their captures. The
type of a once-callable closure is `Fn(T) -> U` but the *closure value itself*
is `own` — it can be called exactly once because calling it moves it (consuming
the captured value). This is how Kāra's ownership naturally models this without
introducing a second closure trait.

The compiler needs to understand that calling a closure moves it when the closure
type carries owned captures. This is an inference rule, not a type-level change.

**Where this interacts with the spec:** The spec says `Fn(T) -> U` is the closure
type used in function signatures. For `once`-style closures, the parameter would
be declared as:

```kara
fn run_once(f: Fn(Data) -> Result) {
    f(data)   // moves f — closure consumed here
}
```

The compiler would prevent `run_once` from calling `f` twice. No new syntax at
the type level, but the spec needs to explicitly state this semantic.

**Resolved:** Option C adopted. Added to design.md in the closures section. Single `Fn(T) -> U` type; once-callable behaviour enforced by ownership (calling moves the closure). `Vec[Fn(T) -> U]` works uniformly. UFCS `user.validate` is once-callable when `validate` takes `own self`.

## Priority 2 — Worth Adopting (clear value, design needed)

### 2.1 Default Parameter Values ✓ RESOLVED

**Source:** Swift, Kotlin, Python, TypeScript, C++.

Default values for function parameters allow callers to omit trailing arguments.
They're particularly valuable for configuration-heavy functions and optional
flags:

```kara
fn create_server(
    host: String,
    port: u16 = 8080,
    max_connections: i64 = 1000,
    timeout_ms: i64 = 5000,
) -> Server

// Full specification:
create_server("0.0.0.0", 443, 500, 10000)

// Using defaults for everything after host:
create_server("0.0.0.0")

// Mixing — requires named arguments (v5 §2.2):
create_server("0.0.0.0", max_connections: 100)
```

Default parameters compose naturally with named/labeled arguments from
v5 §2.2 — defaults make unlabeled trailing arguments optional;
labels allow specifying non-trailing arguments without filling in defaults for
the ones before them.

**Constraint:** Default values must be pure constant expressions — no function
calls with side effects, no `writes(...)` effects. Acceptable defaults: literals
(`42`, `"localhost"`, `true`), arithmetic on literals (`60 * 1000`), enum
variants (`Direction.North`), and calls to `const` functions. The compiler
evaluates defaults at compile time.

**Default on `Option[T]` parameters.** A very common pattern is
`option_arg: Option[T] = None`:

```kara
fn find(
    collection: ref Vec[T],
    target: ref T,
    start: Option[i64] = None,
) -> Option[i64]
```

This replaces the Rust pattern of `Option` parameters with separate builder
types or multiple function overloads.

**Interaction with effect polymorphism.** Default values that are closures must
declare effects explicitly:

```kara
fn process(
    data: Data,
    logger: Fn(String) -> () with _ = |_| (),   // no-op default
) -> Result
```

**Resolved:** Added to design.md. Per-call evaluation; no cross-parameter references; trailing only; label required to skip a default. `#[derive]` not relevant — trait impls, not constructors.

### 2.2 `where` Clauses for Complex Generic Bounds ✓ RESOLVED

**Source:** Rust, Swift, Kotlin (where), Haskell.

Kāra uses `[T: Bound]` syntax for type parameter bounds, inline with the
generic list. This is clean for short bounds (`[T: Ord]`, `[K: Hash + Eq, V]`)
but becomes unwieldy for functions with many constrained parameters or complex
multi-type constraints:

```kara
// Today: all constraints inline — reads poorly when bounds are long
fn merge_sorted[T: Ord + Clone + Display, U: From[T] + Into[String]](
    left: ref Vec[T],
    right: ref Vec[T],
    transform: Fn(T) -> U with _,
) -> Vec[U]

// With where clause: bounds on a separate line, much easier to read
fn merge_sorted[T, U, with E](
    left: ref Vec[T],
    right: ref Vec[T],
    transform: Fn(T) -> U with E,
) -> Vec[U] with E
where
    T: Ord + Clone + Display,
    U: From[T] + Into[String],
{ ... }
```

`where` clauses also unlock constraints that can't be expressed inline — bounds
on *associated type relationships* between two type parameters:

```kara
// Only expressible with where — not inline
fn zip[A, B, C](a: A, b: B) -> C
where
    A: Iterator,
    B: Iterator,
    C: Iterator,
    // Constraints between parameters that don't have a single "owner":
    (A.Item, B.Item): Into[C.Item],
{ ... }
```

(Note: the `A.Item` form assumes associated types, from v6 §2.2.
`where` clauses are more necessary once associated types exist.)

**Syntax.** After the return type and before the function body, after effects and
contracts:

```kara
fn binary_search[T](haystack: ref Vec[T], needle: ref T) -> Option[i64]
    reads(SearchIndex)
    requires haystack.is_sorted()
where T: Ord
{ ... }
```

Order: `fn` declaration → effect annotations → `requires`/`ensures` → `where` →
body. This matches the existing declaration ordering from the contracts section.

**For `impl` blocks:**

```kara
impl[T] Display for Vec[T]
where T: Display
{
    fn fmt(ref self) -> String { ... }
}
```

`where` on `impl` blocks is also needed for the conditional impl feature from
v6 §1.2 — it's a cleaner way to express those bounds.

**Resolved:** Added to design.md. Mixing inline and `where` allowed. `where` on structs/enums allowed (not deferred). Declaration order: effects → requires/ensures → where → body. `where` covers type bounds only; effect variable bounds are Future Decisions §1. Disambiguated from refinement type `where`.

### 2.3 Destructuring in Function Parameters ✓ RESOLVED

**Source:** Rust, Haskell, Erlang, TypeScript.

The spec supports destructuring in `let` bindings, `match` arms, and `for`
loops, but **never mentions destructuring in function parameter position**. This
is a natural extension of the same pattern system:

```kara
// Struct destructuring in parameter position
fn distance(Point { x: x1, y: y1 }: Point, Point { x: x2, y: y2 }: Point) -> f64 {
    ((x2 - x1).pow(2) + (y2 - y1).pow(2)).sqrt()
}

// Tuple destructuring — common for callbacks and iterators
fn sum_pairs(pairs: Vec[(i64, i64)]) -> i64 {
    pairs.iter().map(|(a, b)| a + b).fold(0, |acc, x| acc + x)
}

// Actually that's closure parameter destructuring — also useful:
let total = pairs.map(|(a, b)| a + b);
```

The most common use case is **closure parameter destructuring** — `|(a, b)|`
instead of `|pair| { let (a, b) = pair; ... }`. This is already implied by the
spec's for-loop desugaring (`for (key, value) in map`) which inherently uses
tuple destructuring.

**Interaction with parameter mode inference.** Destructuring in parameter position
extracts fields — the mode (own/ref/mut ref) is inferred per-binding, not per
parameter. If a struct field is only read in the function body, its binding is
`ref`; if consumed, `own`.

**Resolved:** Added to design.md. Full irrefutable patterns allowed (not just tuples); `_` valid; no effect annotation conflict; mode inference is per-binding. Closure parameter destructuring is the primary use case.

## Priority 3 — Add to Deferred Items

### 3.1 Associated Types in Traits ✓ Resolved

**Source:** Rust (`type Item`), Swift (`associatedtype`), Haskell.

Kāra's traits use generic type parameters: `trait Iterator[T]`. Rust and Swift
also support **associated types** — named output types that are determined by the
implementing type, not by the caller:

```kara
// Generic parameter on the trait (Kāra's current model):
trait Iterator[T] {
    fn next(mut ref self) -> Option[T]
}
// A type can implement Iterator[i64] AND Iterator[String] simultaneously.

// Associated type (Rust model):
trait Iterator {
    type Item;
    fn next(mut ref self) -> Option[Self.Item]
}
// A type has exactly one Item — implementing Iterator twice for different
// item types is a compile error.
```

**When associated types matter.** Generic parameters are flexible but require
callers to specify the type at each use site. Associated types encode a
one-to-one relationship — a collection has one element type, a parser has one
output type. Associated types also allow constraints like
`fn process[I: Iterator](iter: I) -> Vec[I.Item]` without a separate `T`
parameter at the call site.

**Why Kāra's current model may be sufficient.** For `Iterator[T]`, the generic
parameter model works well — callers know `T`. The limitations appear when you
need to talk about the associated type *in terms of the implementing type* without
naming it at every call site. As the standard library grows, this will become
necessary.

**Deferred until standard library phase.** The current spec works; this becomes
a limitation when `Iterable`, `From`, `Into`, and `Index` traits are defined.
At that point, associated types become necessary for clean function signatures.

### 3.2 Orphan Rules for `impl` Blocks ✓ Resolved

**Source:** Rust's coherence / orphan rules; Haskell's instance overlap rules.

The spec defines `impl Trait for Type` but **never states whether a third-party
crate can implement a foreign trait for a foreign type**. This is the "orphan
problem" — without a rule, two crates could both implement `Display for Vec[i64]`
and the compiler would have no way to resolve the conflict.

**The standard rule (Rust's "orphan rule"):** You may only write `impl Trait for Type`
if your crate defines *either* `Trait` *or* `Type` (or both). Implementing a
foreign trait for a foreign type is a compile error ("orphan impl").

```kara
// In your crate — OK: you define the trait
pub trait Printable { fn print(ref self); }
impl Printable for Vec[i64] { ... }     // OK: your trait, foreign type

// In your crate — OK: you define the type
struct MyPoint { x: f64, y: f64 }
impl Display for MyPoint { ... }        // OK: foreign trait, your type

// In your crate — ERROR: foreign trait, foreign type (orphan)
impl Display for Vec[i64] { ... }       // ERROR
```

**Without this rule**, the compiler cannot guarantee that trait impls are
coherent — two crates could provide conflicting impls and the behavior would
depend on which was imported first.

**This needs to be in the spec** before the module system or package manager
are finalized — it's a fundamental constraint on how third-party libraries can
interact.

### 3.3 Numeric Conversion Traits (`From`/`Into`) ✓ Resolved

**Source:** Rust. Already partially referenced in the spec.

The spec mentions `From[OtherError]` for error propagation via `?` but defers
it to Phase 8. **The gap:** `From`/`Into` are not just for error types — they
are the standard conversion interface for all user-defined type conversions.
Without them, converting between user-defined numeric types, wrapping foreign
types, or chaining conversions requires bespoke methods.

**The design is already implied** by the spec's `?` operator behavior:
`From[OtherError] for MyError` must be a trait with a specific signature. The
trait needs to be defined before Phase 8 so that type conversion patterns can be
built consistently.

```kara
trait From[T] {
    fn from(value: T) -> Self
}

// Into is derived from From automatically (blanket impl):
// impl[T, U: From[T]] Into[U] for T
trait Into[T] {
    fn into(self) -> T
}
```

The blanket impl (`impl[T, U: From[T]] Into[U] for T`) requires conditional
`impl` blocks from v6 §1.2 — this is another dependency on that
feature.

**Why mention it here:** The spec says "see Phase 8" but the `?` operator
already *requires* `From` to be defined semantically. The implementation can
be deferred but the design shape should be settled now.
