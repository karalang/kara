### Why D1a won (summary from `design_studies/findings.md`)

Four concrete bricks (domain type, DB read, parallel fanout, event stream) exercised the separated form under realistic programs. Only one (brick #1) surfaced any D1 tension, and that tension was bounded — not catastrophic, and paid for by typed errors + mechanical refactor + retroactive-impl capability. The bundled form would save ~5–10 lines per type in the common case but loses the retroactive-impl ergonomics that mainstream languages fake awkwardly or lack. Java `record` (post-16) and Python decorators (`@total_ordering`) close much of the ceremony gap that earlier Rust-vs-Java comparisons assumed, weakening the "bundled is shorter" argument on its own merits.

### Why D1d was split out rather than closed

Brick #3 (parallel fanout) made a case v44 didn't fully capture: the `trait X { ... }` + `effect resource R: X;` pairing in current Kāra is the most duplicative surface in the language. Fusion is about *what a trait is*, not *where impls live* — orthogonal to D1. It deserves its own evaluation and lives at `v46.md`.

## Original framing (for record)

**Framing.** A non-Rust-background reader (the project owner) hit Kāra's current `struct` / `trait` / `impl Trait for Struct` triangle and found it counterintuitive. The pain points: methods live in a separate block from the struct that owns them; two flavors of `impl` (inherent + trait) look similar but mean different things; the link between a struct and the trait it satisfies is invisible at the struct's definition site.

The "Rust without being Rust" design principle (`memory/project_design_principle.md`) says: when semantics permit, prefer mainstream syntax over Rust syntax. This brainstorm asks whether trait conformance is one of those places — and if so, what shape the alternative should take.

This is a **language-design** brainstorm, not an implementation one. No code in `src/` changes until this resolves.

**Ground rule (`memory/feedback_no_design_compromise.md`):** decide each item on Kāra's own merits. "Rust does X" / "Java does X" / "Kotlin does X" are surfacing signals, not deciding signals. The fact that mainstream developers find Rust's split confusing is *evidence about cognitive load*, which IS a Kāra-merits factor — but the resolution should defend itself on Kāra grounds, not "everyone else does it that way."

## Decision lens — what we're optimizing for

Per `memory/feedback_design_weighting_llm_era.md`, **Kāra optimizes for an LLM-first authoring world.** Most code is written by models; over time, humans may rarely read it at all. Existing languages (including Rust, Java, Kotlin) were designed when human authors were the only authors — Kāra is being designed when that's no longer true. This lens runs through every D/V/T/S decision below, in priority order:

### 1. Compiler-verifiable robustness *(primary)*
Does the choice catch more errors at compile time? Are there fewer ambiguous parses, fewer plausible-but-wrong programs that typecheck? Every error caught at compile time is one a model doesn't have to debug at runtime.

### 2. LLM-friendliness *(primary)*
Concretely, five sub-properties:

- **Local context completeness** — a generator can produce correct code with a small window; doesn't need the whole codebase loaded. Self-contained editing units beat sprawling cross-file constructs.
- **Unambiguous syntax** — fewer "did you mean X or Y" parses; fewer plausible-but-wrong generations that look right but compile to wrong behavior. Explicit markers (`override`, `mut`, `ref`) carry signal that disambiguates intent.
- **Compiler-pinpointable errors** — diagnostics that localize the exact problem so a model can self-correct without guessing. Vague "something somewhere is wrong" errors waste tokens and cause cascading bad fixes.
- **Composability without surprises** — features compose by clear rules; no hidden interactions, no "this feature only works with that one in the right order."
- **Mechanical refactorability** — tooling can rewrite code without semantic understanding (rename, move impls between files, add/remove conformance, extract module). Predictable structure makes both `karac fmt` and LLM-driven edits safer.

### 3. Human ergonomics *(secondary, but real)*
Still matters. Humans review, debug, audit, and onboard. A feature that's actively painful for humans is a problem even in an LLM-first world. But ergonomics is a tiebreaker, not the deciding signal — and a feature that's hard for humans but strong on (1) and (2) should stay.

### What this lens does NOT mean
- **Not "verbose is better."** LLMs handle terse syntax fine; the goal is *unambiguous*, not *long*.
- **Not "copy what existing LLM-favored languages do."** The optimization target is shifting fast; mainstream language design is still primarily human-optimized.
- **Not "humans are an afterthought."** Humans review, debug, and audit. Ergonomics affects them.
- **Not loyalty to any existing language.** Hybrid forms, novel forms, reinvented forms are all on the table when the conventional ones all carry meaningful tradeoffs.

Where a leaning shifts under this lens (from a human-only weighting), the shift is called out in the relevant section.

## Terminology note — "Type" in this doc

Throughout this brainstorm, **"Type"** in `impl Trait for Type` is the broader category — anything nameable that can have methods attached. In Kāra that covers:

- **Structs** (the common case in the examples below).
- **Enums** (sum types).
- **Primitive types** (`impl MyTrait for u64`, `impl MyTrait for String`).
- **Type aliases**.
- **Tuples** (`impl MyTrait for (i32, String)`).
- **Generic instantiations** (`impl MyTrait for Vec[T]`).

This breadth is load-bearing for D1: a bundled-only form (D1b) has no definition site to bundle into for primitives, tuples, or aliases — so it either bans those impls outright or needs a separated-form escape hatch, which is just D1c with extra steps.

## Context — what's currently designed

Per `docs/design.md`, Kāra inherits Rust's shape:

```kara
struct InMemoryUserDB { rows: Map[String, String] }

impl InMemoryUserDB {
    fn new() -> InMemoryUserDB { InMemoryUserDB { rows: Map.new() } }
}

impl DatabaseProvider for InMemoryUserDB {
    fn insert(mut self, key: String, value: String) -> Result[(), String] { ... }
    fn get(self, key: String) -> Option[String] { ... }
}
```

- `struct` declares fields only.
- `trait` declares method signatures (and optionally default bodies).
- `impl Type { ... }` adds **inherent** methods to a type.
- `impl Trait for Type { ... }` declares the type **satisfies** a trait and provides bodies.
- A type may have multiple `impl` blocks scattered across files (subject to coherence).
- Retroactive impls are allowed: `impl MyTrait for Vec[T] { ... }` for traits/types you didn't define.

## Context — what mainstream conventions look like

| Language | Conformance declaration | Where bodies live |
|---|---|---|
| Java/C# | `class Foo implements Bar` | inside the class body |
| Kotlin | `class Foo : Bar` (`override` on each method) | inside the class body |
| Swift | `struct Foo: Bar { ... }` OR `extension Foo: Bar { ... }` | inside type *or* extension block |
| TypeScript | `class Foo implements Bar` (structural) | inside the class body |
| Python | `class Foo(Bar)` or duck-typed | inside the class body |
| Go | implicit (no declaration) | on the type as methods |
| Rust | `impl Bar for Foo { ... }` (separate block) | in the impl block |

**Pattern:** every mainstream language except Rust and Go bundles the conformance declaration with the type. Swift uniquely allows both (extensions for the retroactive case).

## Scope of this brainstorm

In scope:
- The **shape** of how a type declares it satisfies a trait.
- The **place** trait method bodies live (with the struct vs. in a separate block).
- **Retroactive impls** — keep, restrict, or drop?
- **Default methods + overriding** — how the chosen shape handles them.
- **Generic impls** — `impl[T] Trait for Vec[T]` works in either shape; sketch both.
- **Visibility / discoverability tradeoffs** — what does a reader see at a glance?

Out of scope:
- Trait objects (`dyn Trait`) at runtime — orthogonal to the definition shape.
- The effect/visibility checker's internal phase logic — a syntactic shape change is mostly desugaring; checker semantics don't move.
- Method-call syntax (`value.method()` vs. `Type.method(value)`) — already settled in design.md.
- The orphan rule itself — only relevant if D5 keeps retroactive impls.

## Group D — Definition shape

### D1. The central question: how does a type declare it satisfies a trait?  *(open)*

**Options:**

- **D1a. Keep separated (current Rust-style).** Struct, inherent impl, and trait impls are independent blocks.
  - Pro: clean separation of concerns; trait impls can live in different files; supports retroactive impls naturally; matches Rust semantics 1:1 (less invention).
  - Con: the struct's definition site doesn't reveal which traits it satisfies — you have to search the file/codebase to discover behavior; extra ceremony for the common case ("define struct + immediately implement one or two traits"); two shapes of `impl` to learn.

- **D1b. Bundled only (Java/Kotlin-style).** Trait conformance is declared in the struct header; method bodies live inside the struct block. No separate `impl` blocks.
  - Pro: one place to look for everything a type does; matches universal mainstream intuition; eliminates the "two impl flavors" confusion.
  - Con: large structs implementing many traits become unwieldy; loses retroactive-impl capability entirely (or requires a new escape hatch); breaks the "data and behavior are separable" mental model some find valuable; **has no answer for non-struct/enum Types** (primitives, tuples, aliases) — see the Terminology note above. Either those impls are banned or D1b silently re-introduces D1c.

- **D1c. Hybrid (Swift-style).** Allow both forms. Bundled is the conventional path; separate blocks remain for organization or retroactive impls on external types.
  - Pro: ergonomic common case; preserves all current capability; gradual learning curve (start bundled, learn separate blocks when needed).
  - Con: two ways to do the same thing means style debates and inconsistency; tooling (`karac fmt`) needs an opinion; learners eventually meet both anyway.

- **D1d. Effect-trait fusion (out-of-the-box, novel).** Collapse `trait` and `effect resource` into one concept. A `trait` *is* the contract surface for an effect resource; conforming types are valid providers. Today's pair:
  ```kara
  trait DatabaseProvider { fn insert(...); fn get(...); }
  effect resource UserDB: DatabaseProvider;
  ```
  becomes one declaration:
  ```kara
  effect UserDB { fn insert(...); fn get(...); }
  impl UserDB for InMemoryUserDB { ... }   // declares a provider
  ```
  Pure value traits (`Ord`, `Hash`) without resource semantics need a different shape — perhaps a separate `pure trait` or stay as today's traits; this needs design.
  - Pro: collapses four concepts (struct, trait, impl, effect resource) into three by unifying two; eliminates the duplication seen in `examples/tests/user_db_test.kara`; LLM-friendlier (fewer concepts, less cross-referencing); leverages Kāra's first-class effects, no existing language has this.
  - Con: large semantic shift — affects design.md § Effects, § Traits, § Impls; needs a story for non-resource traits; risk of forcing every trait to declare resource semantics it doesn't have.
  - Compatible with D1a/D1b/D1c orthogonally — fusion is about *what* a trait is; the bundled-vs-separated question is about *where* impls live.

**Leaning.** D1c. The bundled form is right for the 80% case (define a type and immediately make it satisfy a trait or two). The separate form earns its keep for retroactive impls and for organizing large types — which D1b would force into awkward workarounds. The cost (two forms) is real but bounded; `karac fmt` can pick a default and document the migration.

**Lens-shift note.** Under the LLM-friendly lens, this leaning weakens. Separated form has real LLM advantages: each impl block is a self-contained editing unit (local context completeness), mechanical refactoring is easier (move conformance between files without touching the struct), and large bundled structs increase context-window pressure for any edit. The hybrid form likely still wins, but the *default* may want to flip — separated as canonical, bundled as the syntactic shortcut for the trivial 1-trait case. D1d (effect-trait fusion) deserves serious consideration before settling D1.

### D2. Bundled-form syntax shape.  *(open — only matters if D1b or D1c)*

**Options for the conformance clause:**

- **D2a. `struct Foo implements A, B { ... }`** — Java/TypeScript-flavored. Reads English-like.
- **D2b. `struct Foo : A, B { ... }`** — Kotlin/Swift/C++-flavored. Compact but `:` is overloaded with type ascription.
- **D2c. `struct Foo with A, B { ... }`** — novel; `with` reads naturally and avoids `:` collision.
- **D2d. `struct Foo extends A, B { ... }`** — wrong word for traits (extends implies inheritance which Kāra doesn't have).

**Leaning.** D2a. `implements` is the most semantically precise English word ("Foo implements the Bar contract"). Cost: longer keyword. Benefit: zero ambiguity for newcomers, and the Java/TypeScript demographic is large.

### D3. Marking which methods belong to which trait inside a bundled body.  *(open — only matters if D1b or D1c)*

A struct that implements three traits with two methods each has six trait-methods + N inherent methods inside one block. How does a reader (or the compiler's error messages) know which method came from which trait?

**Options:**

- **D3a. No marker; signature uniqueness disambiguates.** As long as `(name, arity, types)` uniquely identifies the trait method, the compiler matches automatically. (Java's approach.)
  - Pro: minimal syntax noise.
  - Con: ambiguous if two traits declare the same method name with the same signature; reader has to cross-reference traits to know origin.

- **D3b. `override` keyword on each trait method.** Kotlin-style: every method satisfying a trait gets `override fn ...`.
  - Pro: visibly distinguishes inherent from trait-satisfying methods; catches typos (declaring `instert` when trait has `insert`); makes default-override visible.
  - Con: more tokens per method.

- **D3c. Group trait methods inside an inner block.** `impl A { fn ... } impl B { fn ... }` *inside* the struct body.
  - Pro: visually groups; explicit.
  - Con: re-introduces the "two-block" ceremony D1b was trying to remove.

**Leaning.** D3b. Kotlin's `override` is mild ceremony for real reader benefit (and a valuable typo-catcher). Plays well with default methods (D4): `override` is *required* when overriding a default, signalling intent.

### D4. Default methods and overrides.  *(open)*

Traits can have method bodies (defaults) the conforming type may override.

**Options:**

- **D4a. Implicit override** (matches D3a). Defining a method with the trait's signature inside a conforming type silently replaces the default.
- **D4b. `override` required** (matches D3b). Explicit `override fn ...` to replace a default; without `override`, the compiler reports a name collision.
- **D4c. `default` keyword on traits + explicit override.** Trait declarations mark default-bodied methods with `default`; conforming types must `override` to replace.

**Leaning.** D4b paired with D3b. Single `override` keyword carries the dual meaning "this is a trait method AND I'm choosing to provide my own body." Aligns with Kotlin and avoids both Java's silent-override pitfall and C#'s `virtual`/`override` distinction.

### D5. Retroactive impls — keep, restrict, or drop?  *(open)*

Today: `impl MyTrait for SomeoneElsesType` is allowed (subject to coherence). Powerful for ergonomics (extending external library types) but increases language surface.

**Options:**

- **D5a. Keep, unchanged.** Retroactive impls remain; D1c's "separate block" form is their canonical home.
- **D5b. Keep but restrict to the same module/crate.** Drops the orphan-rule complexity; you can only impl your own traits OR your own types, not arbitrary external pairs.
- **D5c. Drop entirely.** Conformance must be declared at the type's definition. Mainstream-pure but removes a major Rust strength.

**Leaning.** D5a. Retroactive impls are one of the genuinely-good ideas Kāra inherits from Rust — they enable "extension method" patterns mainstream languages either lack or fake awkwardly (C# extension methods, Kotlin extension functions). The orphan rule is annoying for compiler implementers, not for users, and Kāra's effect/ownership system already pays similar coherence costs.

### D6. Generic impls in the bundled form.  *(open — only matters if D1b or D1c)*

```kara
// Separated:
impl[T: Ord] Sortable for Vec[T] { ... }
```

How does the bundled form express this when the impl is generic over the conforming *type's* parameters vs. the *trait's*?

**Options:**

- **D6a. Bundled form for the common case only; generic impls always use separated form.** Simple rule: parameterized conformance means separate block.
- **D6b. Bundled form supports parameters in the type header.** `struct Vec[T] implements Sortable where T: Ord { ... }` — feasible but starts to look heavy.

**Leaning.** D6a. The bundled form's value proposition is "the easy case is easy"; once you're parameterizing impls, you've left the easy case and the separated form's clarity wins.

## Group V — Visibility & discoverability

### V1. Where does a reader look to find what a struct can do?  *(open)*

**Current state (D1a):** depends. Some methods in `impl Foo {}`, others scattered across `impl Trait1 for Foo {}`, `impl Trait2 for Foo {}` blocks possibly across files.

**Bundled (D1b):** all in one place.

**Hybrid (D1c):** mostly bundled, sometimes split. Tooling (`karac doc`, IDE go-to-definition) can re-aggregate.

This isn't a separate decision — it's a tradeoff input to D1.

### V2. Effect annotations on trait methods.  *(open)*

In the bundled form, each method's effect annotation lives next to its body. In the separated form, you can compare a `impl Trait for Foo` block's effect annotations against the trait's declared effects in one glance. Does the bundled form make effect-conformance harder to audit?

**Leaning.** Probably not — `karac explain` already exists to show effect inference results; reading a single struct's bundled methods to verify effect conformance is no worse than reading a separate impl block. Doesn't move the needle on D1.

## Group T — Tooling implications

### T1. `karac fmt` — does it normalize between forms?  *(open — only matters if D1c)*

**Options:**

- **T1a. `fmt` is shape-preserving.** Whichever form you wrote, `fmt` keeps it.
- **T1b. `fmt` rewrites separated-form-with-no-retroactive-need into bundled.** Forces canonical form.

**Leaning.** T1a. `fmt` should not silently restructure code. If the project wants a single canonical shape, that's a `clippy`-style lint, not a `fmt` rule.

### T2. Migration of existing examples (`examples/*.kara`).  *(open)*

If D1c lands, do we mass-rewrite existing example files to use the bundled form?

**Leaning.** Selectively. Rewrite examples that are clearly "define + immediately impl" (e.g., the `user_db_test.kara` we just wrote). Leave examples that demonstrate retroactive impls or multi-block organization as-is.
