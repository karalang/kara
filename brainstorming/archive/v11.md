# Brainstorming v11 — External Design Review Delta

Issues raised by an external reviewer of `design.md` after a from-scratch comparison ("what would a best-for-today language look like, and how does Kāra measure up?"). The reviewer concluded Kāra is "already aiming at the right mountain" and framed the gaps as **semantic unification** work rather than missing features. This doc triages the reviewer's 14 proposed patches into the same category/risk structure as v10 so each can be accepted, amended, or rejected on its merits.

**Alignment with current design:** ~85-90% overlap. The reviewer's "what I would build from scratch" list matches Kāra on structs/ADTs/traits, Result/Option + tracked panics, fixed-verb effects with resources, inferred ownership tiers, structured concurrency from effect conflicts, layout-as-first-class, profiles, and AI-first tooling. Gaps concentrate in contracts/refinement types (~50%), arena/region expressibility, and the unification items below. The reviewer rejected inheritance, implicit exceptions, hidden allocation, async-first concurrency, macro-driven design, weak numerics, silent coercion, invisible globals, and stringly-typed metaprogramming — Kāra rejects all of these.

Organized into four categories: high-risk items where a change would touch core semantics and ripple through multiple phases, moderate-risk items that improve spec completeness without reshaping the core, lower-risk framing/philosophy items that clarify existing stances, and a follow-on category of mechanical interaction gaps surfaced *during triage* of the external review — things the reviewer didn't raise but that emerged when stress-testing the effect/ownership system at its hard edges.

## Category A: High-Risk Semantic Core

### 1. ~~Execution effects — distinguishing `blocks` / `suspends` from resource conflict~~ ✅ RESOLVED

**Decision: Option (a), with the third proposed verb (`requires_thread`) demoted to a built-in resource (`Thread.pinned`) instead of a verb.**

The verb count grows from six to **eight**: six resource verbs (drive conflict analysis) plus two execution verbs (drive scheduler placement). `blocks` and `suspends` take no resource parameter, are inferred for private functions and declared for public functions exactly like resource verbs, and are ignored by conflict analysis but consulted by the scheduler when choosing where to place a task.

`requires_thread` does *not* become a third execution verb. Thread pinning is an *identity* constraint, not a *placement* constraint, and the resource/conflict model handles identity cleanly. Kāra exposes a built-in resource `Thread.pinned`; declaring `writes(Thread.pinned)` means "this task claims its OS thread for its lifetime." Two such tasks conflict on the resource via the existing distinctness-graph machinery — no new mechanism needed. Adding a third verb would have duplicated mechanism that already exists.

**Rationale:**
- (a) chosen over (d) on the cost-of-retrofit argument: execution effects are signature-visible. Adding them post-release means every public function in the ecosystem has an incomplete effect declaration. The 1:10:100:∞ cost ratio applies — a day of spec work now vs. an ecosystem-wide migration later.
- (b) modeling `blocks` as `writes(Thread)` was rejected: thread occupancy is not a read/write relationship on a named resource, and diagnostics would lie ("writes to Thread"). Resource verbs answer *conflict*; execution verbs answer *placement*. Conflating the two muddies the model. Note that `writes(Thread.pinned)` for *pinning* is fine — pinning genuinely is a write-exclusive ownership claim, and the resource model fits identity questions even though it doesn't fit placement questions.
- (c) profile-only was rejected: profiles gate *what you can call*, not *how the scheduler places work* within a profile. Auto-concurrency depends on per-call-site information.
- (d) defer was rejected for the same reason as the (a)-vs-(d) point: signature-visible effects can't be retrofitted cheaply.

**FFI default:** `extern "C"` functions default to `blocks`. The programmer overrides with `@noblock` for pure-CPU C helpers (math routines, `strlen`, etc.). A false `blocks` produces conservative scheduling (recoverable performance loss); a false *no-blocks* produces silent thread starvation invisible to the effect system. Erring toward `blocks` fails in the safe direction.

**Inference leaves:**
- `blocks` — stdlib annotations on parking primitives (`std.time.sleep`, `std.fs.read_sync`, `std.sync.Mutex.lock`, blocking channel ops), plus FFI boundaries (default `blocks`, override `@noblock`). No syntactic trigger in user code.
- `suspends` — the `.await` operator (language primitive), plus stdlib/user functions that transitively contain it. FFI is *not* a leaf for `suspends` — C cannot suspend a Kāra task.

Both verbs propagate via the same transitive-closure algorithm used for resource effects, including the SCC fixed-point iteration for mutual recursion.

**Diagnostics:** `karac explain f` traces execution-effect inference exactly like resource-effect inference — every edge in the reasoning chain is surfaced. A function inferred `blocks` because it calls `g` which calls `std.fs.read_sync` gets the full path printed. Always-on, not opt-in.

**Note on higher-order interaction:** closure types must carry `blocks`/`suspends` in their type alongside resource effects, otherwise a function that takes a blocking closure would have its declared effects diverge from its actual scheduling behavior. This is the same problem as items 15 (effect polymorphism) and 16 (dyn dispatch × effects) below — execution verbs do not introduce a new instance of the problem, but they raise its urgency. Resolution lives in those items.

**Changes applied to design.md:**
- Feature 2 intro: verb count updated from six to eight (six resource + two execution); verb list expanded to show both categories.
- New subsection **Execution Effects (`blocks` / `suspends`)** added between conflict-rules section and effect-inference section. Covers verbs, scheduler-placement semantics, why they are a separate category from resource verbs, why `requires_thread` is modeled as `writes(Thread.pinned)` instead, inference rules, FFI default, and diagnostic exposure.
- Cross-ref bullet on "any expression with a declared or inferred effect" updated to mention both resource and execution verbs.
- Transparent-effects bullet: "six built-in verbs" → "eight built-in verbs," with "scheduling" added to the list of decisions they drive.

**Original problem and options (kept for context):**

**Problem:** The current six verbs (`reads`, `writes`, `sends`, `receives`, `allocates`, `panics`) answer "can these conflict?" but not "where should this run?" A function that calls `sleep`, does a synchronous FFI call, or waits on a mutex currently has no way to mark itself as thread-blocking in the effect system. Auto-concurrency can prove two tasks are conflict-free and schedule them on the same worker — where one silently stalls the other.

**Reviewer's proposal:** Add execution effects alongside resource effects:
- `blocks` — may block a thread
- `suspends` — may cooperatively yield
- `requires_thread` — must run on a dedicated thread

These influence *placement and scheduling*, not conflict detection.

**Why it matters for Kāra specifically:** Auto-concurrency is a headline feature. If the effect system can prove safety but can't express "don't put this on the async worker pool," the scheduler either has to be conservative everywhere (losing parallelism) or accept latency bugs that can't be diagnosed from the effect graph.

**Options:**

- **(a) Accept as proposed — add `blocks` / `suspends` as built-in verbs.** New top-level effect category, on par with the resource verbs. Inferred for private fns, declared on public fns like any other effect.
  - Pro: Clean. Scheduler has the information it needs. Matches how real runtimes actually care about "blocking vs cooperative."
  - Con: Grows the built-in verb list from 6 to 8+. Every public API in the ecosystem now has to think about blocking in its signature. Risk of verb proliferation ("why not `networks`, `sleeps`, `forks`…").

- **(b) Model blocking as a resource effect on a built-in `Thread` resource.** E.g. `writes(Thread.current)` means "may monopolize this thread." Reuses the existing resource/conflict machinery instead of adding a new effect class.
  - Pro: No new category. One uniform model.
  - Con: Shoehorning. Thread occupancy isn't really a read/write relationship. Diagnostics would talk about "writes to Thread" which is confusing.

- **(c) Profile-level only, not per-function.** Keep the 6 verbs unchanged. Declare blocking constraints at the profile layer (`no-blocking` profile rejects calls to stdlib fns known to block). Relies on stdlib annotations rather than effect inference.
  - Pro: Minimal language change. The problem is largely "don't call `sleep` in async contexts" which profiles already handle.
  - Con: Doesn't help the auto-concurrency scheduler within a single profile. No per-call-site reasoning. Third-party FFI wrappers have no way to declare blocking.

- **(d) Defer until auto-concurrency is operational and a concrete bug forces the decision.** Park it in `future_decisions.md`. Ship without it; revisit after the first real scheduling pain.
  - Pro: Avoids committing before the implementation teaches us what's actually needed.
  - Con: This is a signature-visible effect. Adding it later means breaking every public function's effect declaration. The whole point of declared effects is that they're part of the API contract — retrofitting `blocks` is an ecosystem-wide migration.

**Recommendation to consider:** (a) or (d), not (b) or (c). The category-boundary argument matters — pretending `blocks` is a resource muddles the model, and pushing it to profiles leaves the scheduler blind. The real question is whether to pay the complexity now or eat a migration later.

### 2. ~~Resource identity and alias formalization~~ ✅ RESOLVED

**Decision: Option (a) — add the tri-state principle as a preamble before the static-distinctness algorithm, with a per-fork-point clarification beyond what the reviewer proposed.**

The mechanism already exists in `design.md` (static distinctness rules, runtime distinctness graph, three layers of aliasing defense — all resolved in v10 item 14). What was missing was an explicit statement of the *principle* the algorithm implements. The new preamble states:

1. **Proven disjoint** — no conflict, free to parallelize.
2. **Proven identical** — full conflict, serialized.
3. **Unproven** — conservatively assumed to alias; runtime-partitioned or serialized.

Case 3 is the conservative default: *absence of proof* is never treated as *proof of absence*. That rule is what makes data-race-freedom a hard guarantee.

**Beyond the reviewer's exact wording:** the classification is **per-fork-point, not global**. The same pair `(UserDB[a], UserDB[b])` can be in case 1 at one call site (both literals) and case 3 at another (both computed). "Proven disjoint" is a property of *this use of this pair in this scope*, not a global property of the names. Stating this explicitly prevents readers from forming the wrong mental model.

**Diagnostic guarantee:** `karac explain` enumerates every case-3 (unproven) pair at every fork point with the reason it could not be classified, plus whether the fallback is runtime partitioning or serialization. Case 1 and case 2 are summarized (counts only), not enumerated — listing every proven edge would bury the cases that actually need attention. This is the half-step between the reviewer's minimal "surface uncertainty" ask and option (c)'s "surface everything" overreach.

**Rationale:**
- (a) chosen because the reviewer's readability complaint is legitimate: the algorithm describes *how*, not *what*, and a future reader cannot verify an algorithm change preserves intent if the intent was never written down. A paragraph of prose is nearly free and directly closes the gap.
- (b) no-change was rejected: handing readers a mechanism without a stated goal ages badly. The v10 resolution is correct as an algorithm but incomplete as a spec.
- (c) enumerate every edge was rejected: trivially decidable edges are noise. The interesting edges are the unproven ones — already what `karac explain` reports. Strengthening further would bloat the concurrency report and train readers to ignore it.

**Changes applied to design.md:**
- New subsection **The alias tri-state — principle before algorithm** added to the Parameterized Resources section, inserted before the "Static distinctness" rules. Covers the three cases, the conservative-default rule, the per-fork-point clarification, and the diagnostic guarantee. Cross-links forward to the rest of the section as "the mechanism that implements this principle."

**Original problem and options (kept for context):**

**Problem:** The current design discusses parameterized resources, provider-rooted resources, and resource namespacing (resolved in v10 item 14 via the distinctness-graph partition), but the *alias rules themselves* aren't stated as a top-level tri-state. A reader following the effect system chapter has to derive "when do two `R` values conflict?" from scattered examples.

- Proven disjoint → no conflict
- Proven identical → full conflict
- Otherwise → conservatively assumed to alias
- Compiler must surface alias uncertainty in diagnostics

**Relation to v10 item 14:** v10's distinctness-graph is the *algorithm* that answers "which partition does this value belong to at this fork?" The reviewer's patch is the *principle* that algorithm implements. They're compatible — the patch is a statement of intent, the v10 resolution is the mechanism.

**Options:**

- **(a) Add the tri-state rule as a preamble to the v10 distinctness-graph section.** Three bullets before the algorithm: disjoint / identical / assumed-alias. Plus a one-line requirement that `karac explain` must report which case applied and why.
  - Pro: Cheap. Makes the chapter readable top-down instead of requiring readers to reverse-engineer the principle from the algorithm.
  - Con: Some overlap with existing prose. Risk of the preamble drifting out of sync with the algorithm.

- **(b) No change — the v10 resolution is sufficient.** The algorithm implies the principle; readers who work through the examples arrive at the same conclusion.
  - Pro: Avoids duplication.
  - Con: Hands the reader a mechanism without a stated goal. Harder to verify the algorithm matches the intent later.

- **(c) Strengthen further — require *every* conflict-or-no-conflict edge to be emitted in the concurrency report, not just uncertain ones.** A more aggressive version of the patch.
  - Pro: Maximum explainability.
  - Con: Report bloat. Most edges are trivially decidable and noise to include.

### 3. ~~Parallel failure semantics — one unified chapter~~ ✅ RESOLVED

**Resolution (2026-04-11):** Option (a) — wrote a unified `### Parallel Failure and Cleanup` subsection in `docs/design.md` under Feature 5, immediately after Concurrency Semantics, and closed two of the three sub-gaps inline. Findings during drafting:

- **The reviewer's claim that "Err does NOT cancel siblings" was a misread of the current `design.md`.** The existing prose ("Cancel sibling tasks on first error" + the `collect_all` framing as "when you need every result") already commits to symmetric fail-fast — Err and panic both cancel, `collect_all` is the opt-in gather form. The unified chapter makes the symmetry explicit rather than flipping anything. A branch that would cancel on panic also cancels on Err; the single exception is that panic dominates Err in the scope's return value (`Err(Panic)` always wins over a sibling `Err`).
- **Err-vs-panic asymmetry sub-gap: closed as symmetric.** Rationale in the chapter: Kāra does not distinguish "expected failure" from "bug" for the purpose of cancellation, because in both cases a sibling whose work is invalidated by a peer failure is wasted work. The distinction between Err and panic is preserved where it matters (panic unwinds and produces `Err(Panic)` at the scope boundary; `errdefer(e)` is skipped on panic; double-panic aborts) but is irrelevant to whether peers get cancelled.
- **Cancellation-as-effect sub-gap: closed as "no".** Cancellation is a control-flow outcome of the enclosing structured-concurrency scope, not a resource interaction. Making it an effect would widen every `-> Result` function's caller's effect set transitively and un-actionably. Documented in the chapter as an explicit non-rule so future readers don't re-raise it.
- **Backpressure / fairness sub-gap: deferred.** Broken out as item 21 of this brainstorm. These are scheduler-layer (runtime) concerns — they belong with the concurrency-model-three-layers work in item 8, not with language-level failure semantics. The chapter documents that they are out of scope and points forward.

Also trimmed the now-duplicated panic-sequence bullet in Concurrency Semantics to a short pointer at the new chapter, and added a cross-reference bullet in the `defer`/`errdefer` rules list. The new chapter is now the single place to look for any parallel-failure question; `errdefer(e)` binding semantics are tabulated in one place for the first time.

### 4. ~~Ownership inference — semantic reachability vs syntactic presence~~ ✅ RESOLVED

**Decision: Option (b) — keep the syntactic rule, respec it as a deliberate predictability choice rather than an implementation approximation.**

The syntactic-consume-presence rule stays exactly as implemented. What changes is the *framing* in `docs/design.md`: the rule is no longer described as "pessimistic" or as "implementable without control-flow analysis." It is described as a deliberate design choice, justified by three properties that a reachability-based rule would forfeit.

1. **Predictability.** Parameter mode is signature-visible and part of the public API. A rule that depends on control-flow reachability forces the programmer to run dataflow analysis in their head to predict their own function's mode. A syntactic rule can be checked by scanning the body for consume operations — a property no user has to learn, because it is the rule they would derive on their own.
2. **Safety asymmetry.** The rule is conservative in the safe direction. It may reject programs that would be sound under a more precise analysis, but it never accepts a program with a real use-after-move. Memory safety is preserved unconditionally; the cost is paid only in expressiveness and occasionally in an unnecessary RC fallback, never in correctness.
3. **Cross-version stability.** Because mode is part of the public contract, any change in the inference rule is a potential break for downstream callers. The syntactic rule is defined to be *exactly* as precise as a syntactic scan — no more, no less — so a function's mode cannot silently flip between compiler releases as the analyzer gets smarter. A reachability-based rule would turn toolchain upgrades into API-breaking events; the syntactic rule makes that impossible by construction.

**Dead branches are intentional.** The `if impossible_condition { consume(x) }` case, which the reviewer cites as a reason to move to reachability, is accepted as the deliberate outcome of the rule: dead branches that affect inferred signatures are a code smell worth surfacing, not a defect worth engineering around. The fix is to delete the dead branch, not to teach the compiler to ignore it.

**Option (a) rejected** on the cross-version stability argument: "semantic reachability with a first-compiler syntactic approximation" has the same weakness the reviewer was trying to fix — signatures that change invisibly as the compiler improves, except now the cause is a toolchain bump rather than a source edit, which is arguably worse because nothing in the programmer's code changed.

**Option (c) rejected:** an `@mode(ref)` override would let users lie about what their function body does in order to get a nicer signature. That is the opposite of "mode is inferred from what the body does." No such annotation is added.

**Option (d) rejected** on the cost-of-retrofit argument: the rule is signature-visible and cannot be relaxed later without breaking callers.

**Changes applied to design.md:**
- **Feature 4, Part 1 (Parameter Mode Inference):** the "deliberate pessimistic/syntactic analysis" paragraph replaced with a four-paragraph justification: (1) the inference rule stated as deliberate and predictability-motivated, (2) the conservative-in-the-safe-direction guarantee, (3) explicit statement that dead branches count as consuming and are a code smell, (4) the cross-version stability guarantee.
- The three case entries (`if` / loop / match guard) are unchanged except that the first explicitly notes the rule is not a limitation to be lifted in later versions.
- The hand-wavy "the programmer can override explicitly" phrase is removed — no such override exists in the language, and option (c) is rejected above.

---

**Original problem and options (kept for context):**

**Problem:** The current design specifies ownership inference as pessimistic: if a `consume` appears in *any* branch, the parameter is inferred as `own`, regardless of whether that branch is actually reachable under realistic call patterns. v10 didn't touch this. The reviewer explicitly flags it as "baking implementation convenience into language semantics."

**Reviewer's proposal:** Restate the rule as "conservative and based on possible execution paths, not purely syntactic presence of operations." The first compiler can still implement a coarser approximation, but the *specification* should not be that approximation.

**Why it matters:** ownership mode is part of the public API contract. A change from `ref` to `own` is breaking. If the spec says "syntactic presence wins" and a user writes `if impossible_condition { consume(x) }`, the parameter is forever `own`, even though no execution path ever consumes it. A future compiler that does better reachability analysis cannot relax the mode without the spec lying about what it guarantees.

**The counter-argument the current design implicitly makes:** Predictability. "Scan the body for `consume`, if present → own" is trivially explainable. Semantic reachability requires users to understand control-flow analysis to predict their own function's signature. A user-predictable rule, even if coarse, is arguably better than a precise rule only the compiler understands.

**Options:**

- **(a) Accept the patch — respec as semantic reachability.** Mode is determined by whether a real execution path consumes the value. First compiler implements syntactic approximation with a known-imprecise marker in diagnostics.
  - Pro: Doesn't lock in convenience as law. Future compilers can improve without breaking the spec.
  - Con: Two-tier spec ("the rule vs what we do"). User predictability drops — you can't tell your own function's mode without asking `karac explain`.

- **(b) Keep syntactic rule but document it as a deliberate predictability choice, not an approximation.** Add spec prose: "Kāra uses syntactic consume-presence because mode must be predictable from the source without running dataflow analysis." Remove the implication that this is temporary.
  - Pro: Honest about the tradeoff. User can reason about mode from the source alone.
  - Con: Permanently forbids legitimate refinements. Dead branches leak into signatures.

- **(c) Middle ground — syntactic rule, but add an explicit `@mode(ref)` override to rescue false-positive-own cases.** Keeps the default predictable, gives an escape hatch for the dead-branch case.
  - Pro: Predictability + escape hatch.
  - Con: Yet another annotation. "Override inference" is a smell.

- **(d) Defer — not a real problem until someone hits it in practice.** Park in `future_decisions.md`.
  - Pro: Avoids premature commitment.
  - Con: It *is* a real problem; the reviewer identified it by reading the spec, not by running code. And it's a signature-visible rule, so retrofitting is breaking.

**This is the item most likely to trigger a real design change.** The other high-risk items add spec content; this one potentially *reverses* an existing decision.

## Category B: Moderate Spec-Completeness Gaps

### 5. ~~Ambient and nondeterministic effects — Clock, Random, Env as explicit resources~~ ✅ RESOLVED

**Decision: Option (a) with an ergonomic carve-out — blessed resources named at the language-spec level, private-function inference carries the burden, transparent-verb operations get a carve-out for observability, locale/scheduling/thread IDs punted.**

The principle is now stated at the language-spec level: every source of nondeterminism must enter through a named resource. The three blessed resources (`Clock`, `RandomSource`, `Env`) are the ones already present in the built-in resource table — no renaming. The new prose makes two things explicit that the table only implied: (i) that reading these resources is the *only* way nondeterminism can enter, and (ii) that these resources are part of the language contract, not stdlib convention that a rewrite could revoke.

**Why (a) over (b):** Option (b) — "add the principle but leave resource names to stdlib" — was rejected because the whole value of the patch is spec-level commitment to reproducibility. A principle with unnamed resources does not deliver that; stdlib could drift into `TimeProvider`/`ClockSource`/`SystemClock` and nothing in the language would prevent it. The v10 `main()` decision already set the precedent by naming `Env` at the language level. Consistency argues for naming `Clock` and `RandomSource` the same way.

**Why (a) over (c):** Option (c) — "reject, this is stdlib" — was rejected because deterministic replay testing is advertised as a Kāra language promise, not a library feature. The language has to know where nondeterminism can enter so that conflict analysis, provider injection, and the transparent-verb carve-out can treat those resources consistently. A stdlib decision is reversible; a reproducibility guarantee cannot be.

**The verbosity objection handled without punting.** The obvious objection to (a) is "`reads(Clock)` everywhere is a signature tax on every function that logs." Three design moves make it not feel like a tax:

1. **Private-function inference does the work.** Only public functions must *declare* nondeterminism effects. Private functions infer them from the call graph, the same way every other effect is inferred. The verbosity tax only hits signatures that cross the public boundary — which is exactly where it *should* hit, because a public function that reads the clock is externally nondeterministic and its callers deserve to know.
2. **Transparent-verb carve-out for observability.** A `traces`-declared transparent operation that reads `Clock`, `RandomSource`, or `Env` does not propagate those effects to its callers, mirroring the existing rule for `writes(Stderr)` inside transparent verbs. This is the most load-bearing concession — it means a logger that stamps timestamps on every line does not pollute every caller's signature. The carve-out is *strictly* for transparent-verb operations; a non-transparent function that reads the clock still propagates `reads(Clock)` normally. No back door for laundering clock reads into pure functions.
3. **No sub-resource split.** `Clock.wall` and `Clock.monotonic` stay as stdlib API surface, not effect-system resources. Two calls that both read `Clock` do not meaningfully conflict regardless of which sub-API they use, and splitting them would inflate the conflict graph without enabling any additional parallelism. Keeping `Clock` as one resource keeps the effect system simple and reuses existing machinery instead of inventing sub-resource semantics that nothing else needs.

**Out of scope for v1 (deliberately):** locale, scheduling nondeterminism, thread IDs, PID/UID. Locale is close to `Env` and can be folded in when the stdlib commits to a shape. Scheduling nondeterminism is already covered by the execution-verb work in item 1 (`blocks` / `suspends`) and the parallel-failure rules in item 3 — it doesn't need a dedicated resource. Thread/process identifiers are deferred until a concrete use case appears; adding them speculatively would invite a long tail of "what about X" with no principled stopping rule.

- New subsection **Nondeterminism as an Explicit Resource** added between [Built-in Primitive Resources](#built-in-primitive-resources) and [Effect Groups and Composition](#effect-groups-and-composition) under Feature 2. Covers the principle statement, the three blessed resources with stdlib examples, the inference-vs-declaration split, the transparent-verb carve-out (and its strictness), the deterministic-replay testing pattern with a code example, and the explicit v1 scope list (what is blessed / what is punted).
- TOC updated with the new subsection link.
- The existing Built-in Primitive Resources table is unchanged — the resources it already lists (`Clock`, `RandomSource`, `Env`) are exactly the ones the new subsection blesses. No new resources added, no renames.

**Original problem and options (kept for context):**

**Problem:** The current design models I/O as effects via resources, but nondeterminism *sources* (wall clock, RNG, process env, locale, scheduling nondeterminism) aren't called out explicitly. A reader might assume `std.time.now()` is pure, or that `env.var()` is side-effect-free. The v10 `main()` resolution already established that `env.args()` is `reads(Env)` — which is the right pattern, but it's buried in item 15.

**Reviewer's proposal:** Make it a first-class principle: "All nondeterministic inputs must enter through explicit resources" — `Clock.wall`, `Random.default`, `Env.process`. This ensures reproducibility and testability by construction.

**Relation to memory:** the `project_main_cli_design.md` memory already records `Env` as the resource for env vars and CLI args. This patch generalizes to Clock and Random.

**Options:**

- **(a) Add the principle + name the three resources in the effect system chapter.** Clock, Random, Env as blessed stdlib resources. All nondeterministic stdlib fns declare effects on these. Testing pattern: inject a fake Clock/Random via provider.
  - Pro: Aligns with existing Env pattern. Enables deterministic replay testing without runtime hooks.
  - Con: Forces `reads(Clock)` on every function that reads time. Verbose for pervasive clock access (logging, metrics, request IDs).

- **(b) Add the principle but leave the resource names to stdlib design.** Spec says "nondeterminism enters via resources"; stdlib picks the exact names later.

### 6. ~~Specification layers — guaranteed / reported / implementation-freedom~~ ✅ RESOLVED

### 7. ~~Contracts constraining effects (and effect polymorphism at higher-order boundaries)~~ ✅ RESOLVED

**Scope note:** This entry resolves both the original item 7 (contracts × effects) and the original item 15 (effect polymorphism at higher-order boundaries — relabeled as **7.b** below). They are bundled because the answer to 7 only makes sense once 7.b is settled: the "contracts can express effect constraints" question collapses immediately when the existing effect-polymorphism mechanism is recognized as the actual answer. Item 15's section header further down is now a redirect.

**Decision (7.a — contracts × effects): option (c). Contracts do not constrain effects.** Predicates of the form "this function does not allocate / panic / write to R" are not expressible as `requires` / `ensures` / `invariant`. The declared effect set on a public function is the canonical mechanism — omitting an effect from the declaration is the commitment that the function does not perform it, statically verified. There is no `forbids(verb)` predicate.

**Decision (7.b — effect polymorphism): the existing Effect Polymorphism section already implements the hybrid model (option (c) of the original item 15).** Public generics declare named or anonymous effect variables (`with E`, `with _`); private generics infer them. The viral annotation rule applies at every public boundary. Trait effects are handled via trait-level effect bounds (open-world model) for `dyn Trait`, and via `with E` / `with _` on trait methods for static dispatch. The only remaining gaps — confirming that variables can compose with fixed effects, and noting where effect-variable bounds are parked — are addressed as small additions to the existing section rather than a new chapter.

**Why this framing:**

1. **Declared effects already do what `forbids` would do.** A public function's declared effect set is an upper bound, statically verified against the body. Omitting `allocates` from the declaration *is* `forbids(allocates)`. A separate contract predicate would be a second spelling of the same constraint, with strictly weaker semantics — contracts strip in release, declared effects do not. This is a category error against the layer system from item 6: declared effects are *guaranteed semantics*, debug-stripped contracts can never reach that layer.

2. **The case for `forbids` only exists if you don't have effect polymorphism.** The closures-and-generics use case ("I want this private helper to forbid allocation") is real, but it is an effect-polymorphism question, not a contract question. Item 15 (now 7.b) shows the spec already has the hybrid model that handles it: the private function inherits the polymorphic effect state of its closures, the viral rule forces the public boundary to declare what effects flow through, and the declared effect set at that boundary is exactly the constraint. Bolting `forbids` onto contracts would compete with the proper mechanism instead of using it.

3. **Item 15 is mostly a documentation gap, not a design gap.** When the brainstorming was written, the open questions on effect polymorphism looked large: are variables first-class, can they compose, can they be bounded, how do they interact with traits, how does `dyn` work. Re-reading the existing Effect Polymorphism section showed that all five questions are answered in spec — variables are first-class with both named and anonymous forms, multiple variables compose via union (`with E1 E2`), traits use the same `with _` / `with E` machinery, `dyn Trait` uses trait-level effect bounds with the open-world model. The only items not pinned down were (i) variable + fixed-effect combination in the result clause, and (ii) the deferral status of bounds on effect variables. Both are added directly to the existing section without a new chapter.

4. **One mechanism, one job.** v10 already worked to make declared effects the single contract surface for public APIs. Adding a parallel `forbids` spelling splits that surface. Library authors would have to learn both, tools would have to reconcile them, and reviewers would have to remember which one is the authoritative one when the two disagree. The cleaner answer is that the surface is one — declared effects — and contracts handle value-level invariants only.

**Rejected alternatives:**

- **(7.a) option (a) — contracts grow `forbids(verb)` predicates.** A second spelling of an existing mechanism, in a strictly weaker layer (debug-stripped vs statically verified). Splits the contract surface for public APIs. Rejected.
- **(7.a) option (b) — defer until contracts themselves grow.** Contracts are already at gradual-verification Level 2.5 in the spec; this is not a "wait for contracts" question. Deferring would just mean the spec stays silent on the relationship and the question reappears in v12. Rejected in favor of stating the relationship now.
- **(7.b) option (a) — first-class effect variables required everywhere, including private functions.** Heavier than the existing inference/declaration split. Forces annotation in places where the compiler can already infer it. Rejected.
- **(7.b) option (b) — fully implicit, public effects depend on call sites.** Breaks the "public = declared = stable contract" rule that the rest of the spec relies on. Rejected.
- **(7.b) option (d) — closures cannot exceed enclosing fn effects.** Kills the dependency-injection pattern (passing a `writes(Log)` closure into a pure helper). Rejected.

**Changes applied:**

- `docs/design.md`, Contracts section: added "Contracts do not constrain effects" paragraph stating that effect-subset guarantees go through declared effects, with the layer rationale (declared effects are guaranteed semantics; contracts strip in release).
- `docs/design.md`, Effect Polymorphism section: added "Combining effect variables with fixed effects" subsection with `run_with_config` example (`with E reads(Config)`), and "Bounds on effect variables are deferred" subsection cross-linking to the existing deferred note under generic `where` clauses.
- No back-references from `docs/design.md` to this brainstorming file in either edit.
- No code changes — both resolutions are spec-only. The hybrid effect-polymorphism model is already implemented in the existing typechecker work; this brings the spec text in line with the implementation rather than the reverse.

### 15. ~~Effect polymorphism at higher-order boundaries~~ ✅ RESOLVED — see item 7.b

This item has been folded into item 7's resolution as **7.b**. The decision (hybrid: private generics infer effect flow, public generics declare named or anonymous effect variables, with the viral annotation rule applied at every public boundary) is recorded there, along with the `docs/design.md` changes that close the documentation gaps in the existing Effect Polymorphism section. **Problem:** When a generic function takes a closure parameter, what are the generic function's effects? Example:

```kara
fn map[T, U](xs: Vec[T], f: Fn(T) -> U) -> Vec[U] { ... }
```

If `f` internally does `writes(Log)`, does `map` inherit that effect? Clearly it must — otherwise calling `map(xs, |x| log_debug(x))` would silently drop the effect from the caller's declared signature. But how is this expressed in the type?

The standard answer is **effect variables**: `fn map[T, U, E](xs: Vec[T], f: Fn(T) -> U with E) with E`. The generic picks up whatever effects the closure has. `design.md` alludes to this but doesn't pin down:

- Are effect variables first-class in the generic system, or implicit and inferred?
- Can you bound them (`with E: subset(reads(Log) + writes(Log))`)?
- Can a generic function declare *additional* effects beyond the closure's (`with E + reads(Config)`)?
- What about multiple closures with different effect sets?
- How do effect variables interact with trait bounds?

**Why it matters:** Every iterator method, every callback API, every dependency-injection pattern hits this. `map`, `filter`, `fold`, `for_each`, `spawn`, `with_provider` — all take closures, all need to forward effects. A half-specified answer here means the stdlib can't be written without ad-hoc hacks, and user code that wraps closures breaks the effect story.

**Options:**

- **(a) First-class effect variables with explicit annotation and inference.** `with E` syntax in generics, inferred when omitted, can be bounded.
  - Pro: Expressive, principled. Matches Koka/Eff lineage.
  - Con: New syntax, significant typechecker work. Effect variance rules get hairy.

- **(b) Fully implicit — the compiler infers effect flow through closure parameters, no annotation needed.** Public generic functions get their effects computed by instantiation.
  - Pro: Clean for users. No new syntax.
  - Con: Public function effects are no longer a stable contract — they depend on callsite closures. Breaks the "public = declared" rule.

- **(c) Hybrid — private generics infer, public generics declare effect variables explicitly.** Matches the existing inference/declaration split.
  - Pro: Consistent with current philosophy.
  - Con: Still needs the effect-variable syntax.

- **(d) Forbid closures from performing effects not declared at the containing fn level.** Closures capture the outer fn's effect set and can't exceed it.
  - Pro: Trivial to implement. No polymorphism needed.
  - Con: Kills the dependency-injection pattern. Can't pass a `writes(Log)` closure into a pure helper.

**This is probably the largest single spec gap in v11.** It needs a real answer before the stdlib can finalize.

### 16. ~~Dynamic dispatch × effects~~ ✅ Resolved

**Resolution:** Option (a) — trait-level effect bounds — was already the direction sketched in the existing `dyn Trait` bullet list in `design.md`'s Effect Polymorphism section. This item resolves by *formalizing* that sketch into a properly specified subsection rather than reopening the model choice (options b and c are infeasible under Kāra's static, separate-compilation-friendly constraints; option d erases at the dyn boundary for the same reason as b).

**Changes applied to `design.md`** (replacing the 5-bullet sketch in Effect Polymorphism):

1. **Two levels of declaration.** A trait may declare effects at the *trait level* (the ceiling — what `dyn Trait` call sites see) and at the *per-method level* (the actual effects of each method, tighter than or equal to the trait-level bound). The trait-level bound must be the superset of the per-method union, so `dyn` calls safely cover any method the caller might invoke.
2. **Worked example.** `trait Storage with reads(Data), writes(Data)` with per-method narrowing (`load`/`exists` only read, `save` only writes), plus an impl-site error example (`NetworkBackedStore` adding `sends(Network)` to `load` is rejected against the trait's declared `reads(Data)`).
3. **Static dispatch narrows to per-method declarations.** A generic `fn f[T: Storage](s: ref T)` calling `s.load(key)` sees `reads(Data)` (the method declaration), not the trait-level union.
4. **`dyn Trait` call sites use the trait-level bound.** Worked example of `fn sync(store: ref dyn Storage, ...)` declaring `reads(Data), writes(Data)`. Clarification that if a `dyn` caller only invokes `store.load`, only `reads(Data)` is required — per-method declarations are load-bearing for subset callers.
5. **"Effect-opaque" reinterpreted concretely.** An unbound trait's `dyn` calls are now defined as contributing to an anonymous effect variable, so the only way to call them is from a `with _` context. This unifies the dyn-opacity story with the existing effect-polymorphism mechanism — no new "treated as all possible effects" semantics, just the viral annotation rule applied at the vtable. Library authors choose between bounded (precise tracking) and unbound (`with _` at every caller).
6. **Viral annotation extends to `dyn Trait`.** Calling a method on `dyn Trait` from a public function requires covering the relevant effects explicitly — per-method effects if only some methods are called, or the trait-level bound for the open set, or an effect variable. Nothing is implicitly inherited through the vtable.
7. **Phase note.** Implementation still Phase 8. Syntax is reserved from v1 so that trait definitions written today don't need rewriting. Until Phase 8, `dyn Trait` calls are parser-accepted but effect-checker-rejected unless the containing function declares `with _`.

**Interaction with item 15 (effect polymorphism).** Now unified: the unbound-trait case piggybacks on the existing `with _` mechanism instead of inventing a parallel "all effects" escape hatch. Item 15's resolution and item 16's resolution share one escape hatch for effect opacity.

**Problem:** Given `trait Printer { fn print(&self, s: String) }`, what are the effects of `print`? If one impl writes to stdout and another writes to a file, then:

```kara
fn use_printer(p: dyn Printer, msg: String) { p.print(msg) }
```

What effect does `use_printer` declare? The union of all possible impls? The intersection? Something declared on the trait itself?

**Why it matters:** Dynamic dispatch is how plugins, strategy patterns, and runtime polymorphism work. If effects aren't part of the trait contract, `dyn Trait` calls become effect-opaque and the whole verification story breaks at the vtable. If effects *are* part of the trait contract, then impls are constrained to not exceed what the trait declares — which is the sound answer but means traits need effect bounds.

**Options:**

- **(a) Traits declare effect bounds on methods; impls must fit within them.** `trait Printer { fn print(&self, s: String) with writes(IO) }` — any impl may do at most `writes(IO)`, no more.
  - Pro: Sound. Dynamic dispatch preserves effect verification.
  - Con: Forces trait authors to think about effects upfront. Over-approximation is common — the trait says `writes(IO)` but a specific impl only writes a file.

- **(b) Effects flow from impls through effect variables, erased at the dyn boundary.** Similar to how Rust erases generic type parameters at `dyn Trait`. The caller loses effect precision when going through `dyn`.
  - Pro: No upfront constraint on traits.
  - Con: "Effect-opaque dyn" means every `dyn Printer` call is treated as "all possible effects" — destroys the effect system for polymorphic code.

- **(c) Forbid effects on trait methods — traits must be pure.** Impls that need effects are wrapped in a concrete newtype.
  - Pro: Simple.
  - Con: Rules out 90% of real trait uses (loggers, storage backends, network clients).

- **(d) Associated effects — traits declare effect *variables* that impls bind.** `trait Printer { type E: Effect; fn print(&self, s: String) with Self.E }`.
  - Pro: Precise. Each impl keeps its actual effect set.
  - Con: Gets erased at `dyn` just like generic type params, so same problem as (b) when going through the vtable.

**Interacts with item 15.** The right answer probably ties them together.

### 18. Effect subtyping / variance rules ✅ RESOLVED

~~**Problem:** Can a function with declared effects `reads(Log)` be passed where `reads(Log) + writes(Log)` is expected? Intuitively yes — a function that does less is compatible with a slot that permits more. But is the rule written down?~~

~~- Is the effect set covariant in "adding effects is a widening"?~~
~~- Or contravariant in "a function that accepts fewer effects is more useful"?~~
~~- Higher-order: if `f: Fn(T) with reads(Log)` is passed into `g` expecting `Fn(T) with (reads(Log) + writes(Log))`, is that legal? (Yes, the closure just doesn't use its full allowance.)~~
~~- What about the reverse — passing a `writes(Log)` closure into a slot expecting `reads(Log)` only? (No.)~~

~~**Why it matters:** Without explicit variance rules, every function composition site has ad-hoc compatibility checks. This is the kind of thing that works by accident for simple cases and breaks in subtle ways at higher-order boundaries (item 15) or through dyn dispatch (item 16).~~

~~**Options:**~~

~~- **(a) State the subtyping rule explicitly: effect sets are ordered by subset; a fn with smaller effect set is a subtype of a fn with larger effect set.** One paragraph, covers all cases.~~
~~- **(b) Leave to the typechecker; document informally.** Works until it doesn't.~~

~~Almost certainly (a). The question is just where the rule gets written.~~

**Resolution:** Adopted **option (a)**. Added a new `#### Effect Set Subtyping` subsection at the top of `### Effect Polymorphism` in `docs/design.md`. The rule: effect sets are ordered by subset inclusion; a function type with a smaller effect set is a subtype of one with a larger set. At function-type boundaries (closure parameters, trait method slots, `dyn Trait` vtables, first-class function values) the standard variance rule applies — arguments contravariant, return types covariant, and effect sets **covariant in the function type** (a function that does less is usable where a function that may do more is expected). The reverse is a compile error, because admitting a larger-effect closure into a smaller-effect slot would let effects leak past the viral annotation rule.

The subsection includes a worked example (`run`, `reader`, `writer`, `pure`), explicitly notes that pure closures sit at the bottom of the lattice (which is why polymorphic `with _` accepts any closure including pure ones), and cross-references three places in the design that this single rule already licenses:

1. **Trait per-method narrowing of the trait-level ceiling** (item 16's `dyn Trait` model). A method declared `reads(Data)` under a trait ceiling of `reads(Data), writes(Data)` is an instance of effect-set subtyping.
2. **Effect-polymorphic `with _` / `with E`** accepting any closure — the polymorphic parameter sits at the top of the lattice and subtyping admits every concrete closure.
3. **`with _` public functions calling less-effectful private helpers** — the helper's inferred effect set is a subset of whatever `_` resolves to, legal by the same rule.

Also clarified that effect sets are not types: subtyping here is a relation on effect *sets*, not an OO-style coercion with runtime witnesses. The compiler simply accepts the function value at the slot when the set relationship holds, and `karac explain` reports `effect-subset-ok: {A} ⊆ {B}` when asked.

Option (b) rejected because "works by accident for simple cases and breaks in subtle ways at higher-order boundaries" is exactly what happens when a subtyping rule is informal — the typechecker accretes one-off allowances for `with _`, per-method narrowing, polymorphic resolution, etc., and any future feature that touches function types has to re-derive whether it's consistent with the others. Writing the rule down once turns those scattered allowances into instances of a single named principle.

### 19. Resource lifetime vs effect scope ✅ RESOLVED

~~**Problem:** A resource handle is created in scope A, passed into scope B, and scope B declares effects on that resource. When scope A ends, the handle is dropped. But what if scope B captured the handle in a closure that outlives A?~~

~~Concretely: provider-rooted resources (v10 item 7) create a resource tied to a `with_provider` scope. If a closure captures that resource and is returned from the `with_provider` block, what happens?~~

~~- Is it a compile error? (The effect declaration references a resource that no longer exists.)~~
~~- Does the resource's lifetime extend to the closure's lifetime? (Who owns it then?)~~
~~- Does the closure become unusable after the provider scope ends?~~

~~**Why it matters:** Providers are how v10 resolved bootstrapping. If the lifetime story is unclear, any pattern that captures a provider-bound resource into a longer-lived closure breaks.~~

~~**Options:**~~

~~- **(a) Resources tied to a provider scope cannot escape that scope.** Compile error if captured by an escaping closure.~~
  ~~- Pro: Simple. Matches intuition about scoped resources.~~
  ~~- Con: Limits closure-based APIs that want to capture a logger or config.~~

~~- **(b) Escaping closures extend the provider scope.** The provider stays alive as long as any closure holding its resources is alive.~~
  ~~- Pro: More flexible.~~
  ~~- Con: Providers have unbounded lifetime. Teardown semantics get murky.~~

~~- **(c) Explicit: closures that want to escape must mark themselves and the compiler errors if the captured resource is provider-bound.** Users opt into the restriction explicitly.~~

**Resolution:** Adopted **option (a), scoped specifically to provider-rooted resources**. Added a new "Provider-rooted resources cannot escape their provider scope" paragraph to the `### Provider-Rooted Resources` section in `docs/design.md`. The rule: a closure that captures a provider-rooted resource is bound to the `with_provider` block that rooted it and cannot be returned from, stored beyond, or handed out of that block via any escape path (return value, field assignment, channel send, spawned task that outlives the block). Compile error with a worked diagnostic example showing the captured resource, the provider block, and the specific escape path.

Three properties depend on the strict-scoping rule, all called out in the design:

1. **Test isolation is mechanical.** `with_provider(TestClock.new()) { ... }` must give a test a clean clock that disappears at block exit. A leaked closure from test 1 could otherwise contaminate test 2 through a shared scheduler — a bug no typechecker could catch after the fact.
2. **Teardown stays predictable.** Providers run cleanup (flush buffers, close sockets, release pool slots) at block exit via RAII. Option (b)'s "extend lifetime by refcount" would make teardown run at an arbitrary later point, not at scope exit.
3. **No ambient globals by accident.** An escaping closure holding a provider-bound resource is observationally identical to a module-level global over that resource — exactly the implicit-shared-state pattern Part 5's philosophy paragraph (item 11) ruled out.

**Escape hatches are restructure, not relax.** Users who need a "configured helper" pattern either (1) build the closure *outside* the provider block, taking the resource as a parameter, or (2) push the provider block up to the caller so the helper runs inside the caller's `with_provider` scope — the standard test-fixture pattern (`#[with_provider(...)]` on the test function, not on a helper it calls). Neither option relaxes the rule.

**Scope limit.** The restriction applies *only* to provider-rooted resources (those declared with `effect resource R: ProviderTrait` and injected via `with_provider` / `providers { }`). Ambient program-rooted resources — `FileSystem`, `Network`, `Heap`, `Clock` under the default system provider, `Env`, `Stdio`, `RandomSource` — have no lifetime constraint beyond program lifetime; closures capturing them travel freely. The distinction is deliberate: program-rooted resources exist for the whole program, whereas provider-rooted resources are scoped injections whose whole value is that they do *not* leak past their scope.

Options (b) and (c) rejected. Option (b) fails isolation and teardown simultaneously — teardown becomes a runtime refcount dance and test state leaks. Option (c) (explicit `#[escapes]` marker on closures) is just option (a) with extra ceremony: the compiler still has to reject when the captured resource is provider-bound, the marker gives the user no additional capability, and it adds a second way to reason about closure lifetime with no payoff.

~~**Problem:** Can two impls of the same trait (for different types) declare different effect sets? Presumably yes. But what about orphan impls / coherence rules?~~

~~- If crate A defines `trait Printer with writes(IO)` and crate B impls it for `MyType` with `writes(IO) + allocates`, does that work?~~
~~- Does the impl's effect set have to match the trait's declared bound (tying this to item 16)?~~
~~- If a trait method's effects are inferred (private trait, private impls), what does inference across crates even mean?~~

~~**Why it matters:** Coherence is the invariant "for any trait and type, there is at most one impl." Effects add a second dimension: "and its effect set is globally consistent." If crate B adds effects beyond what crate A's trait declared, downstream crates that call the trait method through generics get surprises.~~

~~**Options:**~~

~~- **(a) Impls must match trait-declared effect bounds exactly — no widening, no narrowing.** Strictest.~~
~~- **(b) Impls may narrow trait-declared effect bounds but not widen them.** Trait is upper bound.~~
~~- **(c) Trait declares lower bound, impls may add effects.** Inverted — but then generic callers must use the trait's declared upper bound, which doesn't exist, so generic calls get the union of all possible impls.~~

~~**(b) is almost certainly correct**, same rule as method signatures in subtyping. But it needs to be stated, and it depends on item 16's answer about whether traits declare effects at all.~~

**Resolution:** Adopted **option (b)**. Added a new `#### Trait Coherence and Effects` subsection in `docs/design.md`, placed immediately after the `dyn Trait` phase note and immediately before `### Closures and Effect Capture`. The rule: for any `(trait, type)` pair, the impl's per-method effects must form a subset of the corresponding trait method's declared effects. Impls narrow, never widen. This is the same rule as item 18's effect-set subtyping applied at the trait method slot.

The subsection covers four points the brainstorming item flagged as open:

1. **The coherence-layer statement of the narrowing rule**, with a worked example showing `Storage` impls for `LocalFileCache` (matches), `ReadOnlyView` (narrower `save` that panics — still valid), and `NetworkBackedStore` (rejected for widening `load` with `sends(Network)`).

2. **Orphan impls.** Kāra's orphan rule matches Rust's (downstream crates may impl foreign traits for local types), but downstream impls are bound by the foreign trait's per-method declarations. No downstream crate can raise the effect ceiling. This is what makes the open-world model sound under separate compilation — every generic call site in every upstream crate stays valid when new impls are published downstream.

3. **Private traits infer effects from local impls.** Private (module/crate-internal) traits need not declare a ceiling. The compiler computes each method's effect set as the union of per-method effects across all visible local impls, then propagates the inferred ceiling to call sites. There is no cross-crate dimension because private traits are not exported — `pub` is the gate that turns an inferred internal detail into a published commitment.

4. **Public traits with no declared ceiling are effect-opaque** — callers must propagate via `with _` (already specified under `dyn Trait`). This closes the "forgot to declare" loophole: the language never silently admits widening via an omitted ceiling. Library authors pick: declare a ceiling for precise tracking, or leave it open and accept `with _` propagation.

Option (a) (exact match) rejected because it forbids the reasonable case of an impl being *more* precise than the trait ceiling — `ReadOnlyView.save` panics instead of writing, so its effect set is narrower than `writes(Data)`, and an exact-match rule would either reject the impl or force it to widen back to the ceiling, both strictly worse outcomes. Option (c) (impls add effects) rejected because generic callers through `T: Trait` bounds have no ceiling to trust, so every generic call site would have to union over all possible impls — impossible under separate compilation and a soundness hazard even within one crate.

**Resolution:** Adopted **option (b)**. Item 8 is already resolved ("Three layers, named" at the top of Feature 5), so item 21 is no longer blocked. The work replaced the single "what is not covered" placeholder paragraph at the end of `Parallel Failure and Cleanup` with two new language-level rules and a new `#### Scheduler-layer concerns and minimum invariants` subsection in `docs/design.md`.

**New language-level rules** (added inline to `Parallel Failure and Cleanup`):

1. **Completion wins cancellation.** A sibling branch that has already passed its last compiler-inserted effect-boundary check when cancellation fires completes naturally and returns `Ok`. "Real work is never retroactively converted to `Cancelled`." Whether that `Ok` reaches the caller is still governed by the scope's source-order return precedence — in fail-fast `par` a source-earlier `Err` still wins and the completed `Ok` is not exposed to the scope's return value, but its `defer` blocks run and its effects on mutable state remain; in `collect_all` / `collect_all_vec` every branch's result is exposed including late-completing `Ok`s. This rule is what makes `collect_all`'s "every branch runs to completion" guarantee load-bearing — without it, a last-second retroactive cancellation would undermine it.

2. **Cancellation cascades into nested regions through the same cooperative mechanism.** When an outer scope cancels a child that contains its own inner `par` region, the outer cancellation becomes an effect-boundary observation inside the child, which triggers the child's own fail-fast path, which propagates into the inner region's siblings. No special cross-scope machinery — the existing rule composes through any depth of nesting. Worst-case latency is the sum of effect-boundary distances along the nesting path; programs needing hard preemption latency bounds must use platform-specific mechanisms outside the standard model.

**New `#### Scheduler-layer concerns and minimum invariants` subsection** — commits the language to the minimum cross-scheduler invariants and names the scheduler-specific questions that belong in the runtime chapter:

- **Minimum invariants (every conforming scheduler must satisfy):** (1) no lost work — a completed `Ok` is never silently discarded; (2) cancellation is eventually observed at the next effect-boundary check, bounded by static effect-boundary distance; (3) cancellation cascades through the cooperative mechanism without scheduler interference; (4) termination — every `par` region eventually joins, modulo the acknowledged pure-tight-loop exception; (5) scheduler deadlock-freedom — the scheduler itself may not introduce deadlocks or starvation that wedge progress.
- **Sub-questions deferred to per-profile runtime chapters:** backpressure under overload, fairness between unrelated parallel regions, and scheduler-specific cancellation-propagation optimizations beyond the cooperative baseline. Each is acknowledged as a real question with observable consequences, tractable within the minimum-invariant envelope, and with multiple reasonable answers depending on deployment target (server / embedded / GPU).

The closing note: programs written against the language-level rules remain valid across every future runtime; programs that depend on a specific scheduler's behavior must declare that dependency explicitly in their build profile.

Option (a) rejected: committing to a specific scheduler (work-stealing + FIFO + one chosen cancellation policy) would lock out profiles the language already promises to support — item 8's three-layer framing exists precisely to keep this split open. Option (c) rejected: pure-defer leaves real programs unable to distinguish bugs from legal variation; the minimum-invariants approach gives portable guarantees without over-committing.
