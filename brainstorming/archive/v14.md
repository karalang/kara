## Tier 1: High Impact

### 1. No labeled loops or labeled breaks

**Location:** syntax.md §5.7 (Loop Expressions).

**Problem.** The grammar defines `break` and `continue` with no label mechanism:

```
BREAK_EXPR    = "break" [ EXPR ]
CONTINUE_EXPR = "continue"
```

Nested loops are common in algorithms (matrix operations, graph traversal, search with early exit). Without labels, breaking out of an outer loop requires a boolean flag:

```kara
// Current: flag-based workaround
let mut found = false;
for row in matrix.iter() {
    for val in row.iter() {
        if val == target {
            found = true;
            break;      // only breaks inner loop
        }
    }
    if found { break; } // manual outer break
}
```

Every mainstream systems language has a solution: Rust (`'label: loop { break 'label }`), Go (`Label: for { break Label }`), Java (`label: for { break label }`), C (`goto`).

**Options:**

**(a) Rust-style tick labels: `'label: for ... { break 'label; }`**

```kara
'outer: for row in matrix.iter() {
    for val in row.iter() {
        if val == target {
            break 'outer;
        }
    }
}
```

Pros: Familiar to Rust users. Visually distinct (the `'` prefix). Works on `loop`, `while`, `for`.
Cons: The `'label` syntax is unusual outside Rust (lifetime-like sigil). Kāra doesn't have lifetimes, so `'` is overloaded with no other use.

**(b) Bare label with colon: `outer: for ... { break outer; }`**

```kara
outer: for row in matrix.iter() {
    for val in row.iter() {
        if val == target {
            break outer;
        }
    }
}
```

Pros: Familiar to Go/Java/C users. No new sigil. Reads naturally ("break outer").
Cons: `label:` could theoretically conflict with named-argument syntax or struct literals, but since labels only appear before loop expressions and struct literals require a type name, the parser can disambiguate.

**(c) Leave as-is — no labels**

Require flag variables or refactoring into helper functions that return early. Keeps the language smaller.

Cons: Real friction in algorithmic code. Every other systems language solved this decades ago.

### 2. `where` keyword overloaded for two unrelated concepts

**Location:** syntax.md §3.1 (Functions — WHERE_CLAUSE), §3.12 (Refinement Types).

**Problem.** `where` means two entirely different things depending on context:

```kara
// Meaning 1: Generic type constraints (like Rust)
fn sort[T](list: Vec[T]) -> Vec[T]
    where T: Ord + Display { ... }

// Meaning 2: Refinement type predicates (like Liquid Haskell)
type Percentage = f64 where self >= 0.0 && self <= 100.0
```

A programmer reading `where` in a type alias has to context-switch from "trait bounds" to "value predicates over self." These are fundamentally different: one constrains type parameters, the other constrains runtime values.

**Options:**

**(a) Rename the refinement predicate keyword — `where` for generics only, `satisfying` or `such_that` for refinements**

```kara
type Percentage = f64 satisfying self >= 0.0 && self <= 100.0
type NonEmpty[T] = Vec[T] satisfying self.len() > 0
```

Pros: Each keyword has exactly one meaning. `satisfying` reads naturally ("a float satisfying this condition"). No ambiguity.
Cons: Adds a keyword. `satisfying` is long (10 chars).

**(b) Use `if` for refinement predicates**

```kara
type Percentage = f64 if self >= 0.0 && self <= 100.0
```

Pros: Short. Reads naturally ("f64 if it's between 0 and 100"). No new keyword — `if` is already reserved.
Cons: `if` already means conditional expression; using it for type-level constraints is a conceptual stretch. Could confuse the parser in complex type alias contexts.

**(c) Keep `where` for both — context disambiguates**

The parser already knows which meaning applies: `where` after a type alias `= TYPE` is a refinement predicate; `where` after a generic parameter list is a trait bound. Programmers learn the distinction.

Pros: No new keywords. Consistent with SQL/math where `where` constrains in various contexts.
Cons: Same keyword, different semantics. Readability cost for newcomers.

### 3. `self` vs `own self` — unclear distinction

**Location:** syntax.md §3.1 (SELF_PARAM).

**Problem.** The grammar defines four self modes:

```
SELF_PARAM = "self" | "own self" | "ref self" | "mut ref self"
```

`ref self` and `mut ref self` are clear — borrowed and mutably borrowed. But what is the difference between `self` and `own self`? Both seem to mean "take ownership." The examples use bare `self` everywhere (e.g., `fn process(self, data: Data)` in trait definitions, `fn total_ratio(self)` in impl blocks). `own self` never appears in any example.

If `self` means "inferred mode" and `own self` means "explicitly owned," this parallels how regular parameters work (mode inferred from body). But trait method declarations use bare `self` too, and traits need explicit annotations. This creates ambiguity: is `fn next(self)` in a trait "inferred" or "owned by convention"?

**Options:**

**(a) Clarify: `self` = inferred, `own self` = explicit own. Document the distinction.**

The compiler infers `self`'s mode from the function body, just like regular parameters. `own self`, `ref self`, `mut ref self` are explicit overrides. In traits, bare `self` means "the impl chooses the mode" (like `with _` for effects).

This is consistent with the language's inference-first philosophy but needs clear documentation.

**(b) Remove `own self` — bare `self` always means owned, `ref self` for borrows**

Simplify to three modes: `self` (owned), `ref self` (borrowed), `mut ref self` (mutably borrowed). The "inferred" mode for self doesn't exist — self is always explicit because it determines the calling convention.

Pros: Simpler. Three modes, all explicit. No confusion.
Cons: Loses consistency with regular parameters (which have inferred modes). Can't override in trait declarations.

**(c) Make all self modes explicit — require `own self`, `ref self`, or `mut ref self`**

Ban bare `self`. Every method explicitly declares its self mode.

Pros: Zero ambiguity. Every method signature is self-documenting.
Cons: Verbose. `fn total_ratio(own self)` for a simple getter feels heavy. Breaks consistency with regular parameter inference.

## Tier 2: Medium Impact

### 4. Match arm comma after block looks unusual

**Location:** syntax.md §5.5 (Match Expressions).

**Problem.** Match arms are comma-separated. When an arm body is a block `{ ... }`, the trailing comma appears after the closing brace:

```kara
match result {
    Ok(value) => process(value),
    Err(e) => {
        log_error(e);
        return Err(e);
    },                    // ← comma after }
}
```

This `},` pattern looks unusual to programmers coming from most languages. Rust has the same convention, but many find it visually jarring. The comma after a closing brace feels like a punctuation stutter.

**Options:**

**(a) Make the comma optional after a block arm**

```kara
match result {
    Ok(value) => process(value),
    Err(e) => {
        log_error(e);
        return Err(e);
    }                     // no comma needed after block
}
```

The parser can determine arm boundaries: a `}` followed by another pattern (or the closing `}` of the match) unambiguously ends the arm. Expression arms still require commas.

Pros: Eliminates the visual stutter. Matches what most formatters would do anyway.
Cons: Two rules instead of one ("always comma" vs "comma after expr, optional after block"). Minor parser complexity.

**(b) Use semicolons instead of commas for match arm separation**

```kara
match result {
    Ok(value) => process(value);
    Err(e) => {
        log_error(e);
        return Err(e);
    }
}
```

Pros: Consistent with statement separation. `};` is less jarring than `},`.
Cons: Breaking change. Match arms are expressions, not statements — using `;` blurs that distinction. Rust and most ML-family languages use `,`.

**(c) Keep as-is — `},` is standard in Rust-derived syntax**

Kāra targets Rust programmers among others. The `},` pattern is well-established.

### 5. No step/stride syntax for numeric ranges

**Location:** syntax.md §5.13 (Range Expressions).

**Problem.** Ranges support `..` (exclusive) and `..=` (inclusive) but no step:

```kara
for i in 0..10 { ... }       // 0, 1, 2, ..., 9
for i in 0..=10 { ... }      // 0, 1, 2, ..., 10
```

Iterating with a step requires a method chain or a while loop:

```kara
// Method chain (if .step_by() exists on ranges)
for i in (0..10).step_by(2) { ... }

// Manual while
let mut i = 0;
while i < 10 {
    process(i);
    i = i + 2;
}
```

Stepping is extremely common: iterating even/odd indices, sampling every Nth element, countdown by decrement. The method chain requires parenthesizing the range, which is noisy.

**Options:**

**(a) Add a `step` keyword to `for` loops**

```kara
for i in 0..10 step 2 { ... }      // 0, 2, 4, 6, 8
for i in 10..=0 step -1 { ... }    // 10, 9, 8, ..., 0
```

Grammar: `FOR_EXPR = "for" PATTERN "in" EXPR [ "step" EXPR ] BLOCK`

Pros: Readable. No parentheses needed. Familiar from Python (`range(0, 10, 2)`) and many other languages.
Cons: Adds a contextual keyword. `step` after a range expression could theoretically be a variable name in non-for contexts, but the parser knows it follows `in EXPR` so disambiguation is straightforward.

**(b) Rely on `.step_by()` method on ranges (stdlib solution)**

```kara
for i in (0..10).step_by(2) { ... }
```

Pros: No language change. Standard library concern.
Cons: Requires parenthesizing the range. Verbose for a very common operation.

**(c) Add `by` keyword to range expressions themselves**

```kara
for i in 0..10 by 2 { ... }
let indices = 0..100 by 5;       // works outside for loops too
```

Grammar: `RANGE = EXPR ".." EXPR [ "by" EXPR ] | EXPR "..=" EXPR [ "by" EXPR ]`

Pros: Composable — ranges with steps work in any context, not just `for`. Short keyword.
Cons: Adds a keyword. `by` is a common variable name in some domains.

### 6. Pipe `?` precedence contradicts the operator table

**Location:** syntax.md §5.1 (Operator Precedence), §5.17 (Pipe Expressions).

**Problem.** The precedence table ranks `|>` at precedence 1 (loosest) and `?` at precedence 13 (second-tightest). By the table, `raw |> parse?` should parse as `raw |> (parse?)`, applying `?` to `parse` (a function) before piping.

But §5.17 explicitly states: "`|>` binds tighter than `?`: `raw |> parse?` means `(raw |> parse)?`"

This is a direct contradiction. The note overrides the table with a special rule for pipe-`?` interaction. The actual behavior makes sense (you want `?` on the pipe result, not the function), but the spec contradicts itself.

**Options:**

**(a) Fix the precedence table — move `?` below `|>`, or add a footnote**

Adjust the table to reflect reality. Either move `?` to precedence 0 (loosest) with a note that it applies per-stage in pipes, or add a footnote: "Within a pipe chain, `?` applies to the preceding pipe stage, not the preceding primary expression."

**(b) Change the syntax — use `?` before `|>` instead of after the function**

```kara
raw |> parse |>? validate |>? save
// or
raw |> parse? |> validate? |> save    // keep current, just fix the spec
```

This is really about fixing the spec, not the syntax. The current syntax reads naturally.

**(c) Clarify as a "postfix exception" — `?` in pipe position always binds to the full stage**

Document that `?` is a postfix operator that, within a pipe chain, applies to the entire pipe stage (the expression formed by piping the left value into the right function). This is analogous to how method chains work: `foo.bar().baz()?` applies `?` to the result of the whole chain, not just `baz`.

## Tier 3: Low Impact

### 7. `format()` vs `f"..."` — dual mechanism unclear

**Location:** syntax.md §1.3 (Literals — INTERP_STR), §12 (Design Decisions — #10).

**Problem.** The syntax spec defines `f"..."` string interpolation as a language feature. The design decisions table mentions `format()` "may be added later." But example code in `examples/word_count.kara` already uses `format()`:

```kara
fn format_report(filename: ref String, count: ref WordCount) -> String {
    format("{}: {} total words, {} unique", filename, count.total, count.unique)
}
```

Meanwhile all other examples use `f"..."`:

```kara
println(f"get(1) = {cache.get(1)}");
```

Are both in the language? Is `format()` a stdlib function or a builtin? When should a programmer use which? The coexistence of positional `{}` placeholders (`format()`) and embedded expressions (`f"..."`) is potentially confusing.

**Options:**

**(a) Clarify: `f"..."` is the language feature, `format()` is stdlib for dynamic format strings**

`f"..."` is always preferred for literal format strings. `format()` exists only for cases where the format string is a runtime value (e.g., from a config file or user input). Document this clearly.

**(b) Remove `format()` — `f"..."` is the only mechanism**

Dynamic format strings are rare enough to handle with explicit string concatenation or a builder pattern. One mechanism, zero confusion.

**(c) Keep both, undocumented — resolve when stdlib is designed**

The examples are aspirational. Defer the `format()` decision to stdlib design phase.

### 8. `fn main() -> Result[(), Error]` — verbose success type

**Location:** syntax.md §10 (Entry Point).

**Problem.** When `main()` returns a `Result`, the success type is always `()` (required by the spec). Writing `Result[(), Error]` is verbose and the `()` carries no information — it's always unit:

```kara
fn main() -> Result[(), Error] {
    let content = read_file(args[1])?;
    print(content);
    Ok(())
}
```

The `()` in `Result[(), Error]` and the `Ok(())` at the end are boilerplate.

**Options:**

**(a) Allow `Result[Error]` as shorthand for `Result[(), Error]` in main**

```kara
fn main() -> Result[Error] {
    let content = read_file(args[1])?;
    print(content);
    Ok(())
}
```

Pros: Less visual noise. The `()` success type is implied.
Cons: `Result[Error]` looks like a single-parameter generic, conflicting with the two-parameter `Result[T, E]` everywhere else. Could confuse: "is this `Result[(), Error]` or `Result[Error, ???]`?"

**(b) Allow implicit `Ok(())` — last expression doesn't need to be `Ok(())`**

```kara
fn main() -> Result[(), Error] {
    let content = read_file(args[1])?;
    print(content);
    // implicit Ok(()) — the function completed without error
}
```

Pros: Eliminates the `Ok(())` boilerplate. The success case is obvious: you reached the end without `?` propagating.
Cons: Inconsistent with non-main functions. Magic behavior. A reader might wonder "where's the return value?"

**(c) Keep as-is — explicit is better than implicit**

`Result[(), Error]` and `Ok(())` are clear and consistent with how Result works everywhere else. Two extra tokens is not a real burden.

### 9. `.await` and `par { }` absent from syntax.md

**Location:** design.md (Concurrency sections) vs syntax.md (no coverage).

**Problem.** design.md defines `.await` as a postfix operator for cooperative yield points and `par { }` as a block for explicit parallel regions. Neither appears in syntax.md's grammar, precedence table, expression list, or keyword list.

This is a documentation gap rather than a syntax-naturalness issue, but it affects anyone reading the spec to understand the full language surface.

**Options:**

**(a) Add both to syntax.md with grammar rules and examples**

Add `.await` to the postfix operators (§5.1 precedence, §5.2 primary expressions) and `par` to the block expressions (alongside `sequential`). Include in keyword list (§1.1).

**(b) Track as deferred — add to syntax.md when implementation begins**

If `.await` and `par { }` are Phase 5+ features, keep them in design.md only until implementation starts.

**(c) Add grammar stubs to syntax.md now, with "see design.md for semantics"**

Reserve the syntax and keywords now to prevent conflicts, but point to design.md for full semantics.
