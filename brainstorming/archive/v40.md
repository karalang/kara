# Brainstorming v40 — Language-level security primitives

**Status:** ✓ RESOLVED — 2026-04-20. All three items landed in canonical docs. This doc is archived.

**Resolution summary.** S1 → **A1** (type wrapper only, `#[sensitive]` attribute rejected); S2 → **B4** (defer entirely, no prelude reservation); S3 → **C2** (warning, default-on, suppressible). Landings:

- **S1 — `Secret[T]`:** `docs/design.md § Secret Type (`Secret[T]`)` (new top-level section between Performance Diagnostics and Unsafe Escape Hatch).
- **S2 — `Untrusted[T]`:** `docs/deferred.md § Taint Tracking (`Untrusted[T]` / `Validated[T]`)` (new P2 entry with re-evaluation triggers and non-breaking guarantee).
- **S3 — `// Safety:` lint:** `docs/design.md § Unsafe Escape Hatch § `undocumented_unsafe` lint` (new subsection).

**Deviation from the doc's initial leans:** the doc leaned A3 (ship both wrapper and attribute) and B3 (reserve the `Untrusted` prelude name). The resolution walked both back — A3 → A1 because A2 and A3 allow the convention-only failure mode (users downgrade to the attribute in "hard" cases and the audit-trail guarantee erodes), and B3 → B4 because a reserved-but-empty prelude symbol is a worse signal than absence. S3's C2 lean was confirmed as-is. Rationale is in the 2026-04-20 conversation turn following the v40 draft.

**Ground rule (from `feedback_no_design_compromise.md`):** decide each item on Kāra's own merits. "Rust does X / Swift does Y / OWASP recommends Z" is a *surfacing* signal, not a *deciding* signal. Security principles inform the design space; they do not override Kāra's shape.

**Scope.** Language-level (compiler-enforced or type-system-enforced) security primitives only. Stdlib-only concerns (CSPRNG default, TLS choices, argon2 for passwords) and runtime-only concerns (ASLR, stack canaries) are out of scope — they belong to stdlib and codegen threads respectively.

## Already covered in `design.md` (no action needed)

These classes of attack are already addressed by language design and are listed here so the brainstorming doesn't re-surface them:

- **Memory safety.** Ownership + bounds checks; `unsafe` is an opt-in escape hatch (`:6714`).
- **Integer overflow.** Always traps; silent wrap requires explicit `wrapping_*` methods (`:1139`). Kills a whole class of C/C++ CVEs (heartbleed-style length miscalculations, SUID integer-overflow-to-stack-overflow chains).
- **`iN::MIN / -1` and division-by-zero.** Separate, distinct traps — not silently UB (`:1151`).
- **Hash DoS.** `Map`/`Set` ship a per-process-seeded, non-stable default hasher; cryptographic and stable-digest cases go through `crypto` or `hash::stable` (`:3244`).
- **Pattern exhaustiveness.** No silent fall-through in `match`.
- **Capability-based side-effect gating.** The effect system *is* a capability system: `reads(FileSystem)`, `sends(Network)`, `reads(Env)` etc. are transitively declared, and profile-level `#[no_effect(...)]` denies a capability across the whole project (`:6314`). A library that doesn't declare `sends(Network)` *cannot* open a socket transitively.
- **Safe-by-default deserialization.** Derive-based `Serialize`/`Deserialize` walks types declared at compile time; no runtime reflection, no gadget chains (`:1534`).
- **`#[repr(packed)]` misalignment.** Taking a reference to a misaligned field requires `unsafe` (`:3355`).
- **Format-string attacks.** Kāra strings interpolate via typed trait dispatch (`{}` → `Display`, `{:?}` → `Debug`); there is no `printf`-style format-as-data path, so the whole class doesn't exist.

The three open items below are the gaps that remain after this baseline.

## Group A — Secret / sensitive data type

### S1. `Secret[T]` (or `#[sensitive]`) compiler-enforced secret wrapper  *(✓ RESOLVED — 2026-04-20 → A1)*

**Resolution.** Ship `Secret[T]` only (A1). The `#[sensitive]` field attribute (A2) is rejected for v1. Landed in `docs/design.md § Secret Type (`Secret[T]`)`. Key decisions:

- **Module:** `std.secret` (not in the prelude). Each file using secrets writes `use std.secret.{Secret};` — the import line is itself a review signal.
- **Non-impl traits:** `Debug`, `Display`, `Serialize`, `Deserialize`, `PartialEq`, `Eq`, `PartialOrd`, `Ord`, `Hash`, `Deref`, `Borrow`, `AsRef`, `Copy`. Each is a compile error with a diagnostic pointing to the section.
- **Access:** `.expose() -> ref T`, `.expose_mut() -> mut ref T`. `.into_inner()` is *not* provided — consuming extraction leaves an un-zeroized `T` that defeats the type's purpose; `.expose().clone()` is the explicit reviewable path.
- **Containing-type derives:** `#[derive(Debug)]` on a containing struct emits the field as `<redacted>`. `#[derive(Serialize)]` on a containing struct is a compile error — serialization requires an explicit decision (manual `Serialize` impl calling `.serialize_expose()`, or a non-secret mirror type).
- **Clone / Drop / Zeroize:** `Clone` where `T: Clone` (each clone is its own zeroize unit); `Drop` where `T: Zeroize` (overwrites backing bytes before field destructors); `Zeroize` trait with signature `fn zeroize(mut ref self)`, stdlib impls for `String`, `Vec[u8]`, `[u8; N]`, integer primitives.
- **Equality:** `ConstantTimeEq` (from `std.secret`) replaces `PartialEq`. `==` on `Secret[T]` is a compile error; the diagnostic points to `.ct_eq(...)`.

**Why A3 rejected.** A2 and A3 allow the convention-only failure mode — users downgrade to the attribute when the wrapper is inconvenient, and the audit-trail guarantee erodes. The "API compatibility with existing `String` fields" argument for A2 is a *feature* of A1 (forcing `.expose()` threading at each use site is the point, not a cost). Greenfield v1 code pays no retrofit tax.

**Why foundation-breaking.** The single largest source of real-world credential leaks in modern applications is trivially-derived observability: `log.info("user: {user:?}")` where `User` contains a session token, `println("{config:?}")` where `Config` contains an API key, `serde_json::to_string(&auth)` for debugging where `Auth` holds a refresh token. Once a derived `Debug` / `Display` / `Serialize` impl exists on a type that transitively contains a secret, every call site in the codebase is a potential leak vector, and the fix ("audit every log line, redact in each one") is the *least scalable* possible posture.

This is architecturally cheap to prevent *once* at the language level, and architecturally expensive to prevent by convention at every call site. The lever is small: a type or attribute that (a) the compiler forbids from appearing inside a derived `Debug`/`Display`/`Serialize`, (b) zeroes its backing memory on drop, and (c) provides constant-time equality. Adding this later does not *break* code, but it does leave every pre-existing log/debug/serialize call site as an unaudited leak — the retrofit value is lower than the greenfield value. Landing now means every `User` / `Config` / `Session` type designed during v1 can be written secret-aware from day one.

**Triggering evidence.** No tutorial example in `docs/book` currently writes a credential-bearing type, but the stdlib is starting to grow surfaces that would want this: `std.http` (v35 resolution) will have `Authorization` headers; `std.process` will have env-var reads; tests for provider injection (`design.md:2406`) want to stub `Token` types. Each will invent its own pattern if the language doesn't ship one.

### Design question

**Q1.** Does Kāra ship a first-class secret-bearing construct, and if so in what shape?

### Options

- **A1. Type wrapper: `Secret[T]` in `std.secret`.**
  ```kara
  let token: Secret[String] = Secret.new(user_input);
  log.info("logging in as {user.name}");         // fine
  log.info("token is {user.token:?}");           // COMPILE ERROR — Debug not impl for Secret[String]
  let inner: ref String = user.token.expose();   // explicit extraction; single point to grep for
  ```
  - Compiler enforcement: `Secret[T]` intentionally does **not** implement `Debug`, `Display`, `Serialize`, or `Hash`. It implements `ConstantTimeEq` but not `PartialEq` (== would be too easy to misuse as a non-constant-time check). It implements `Drop` with a zeroize pass. `.expose() -> ref T` is the only read path; `.expose_mut() -> mut ref T` for write.
  - Derive interaction: a struct containing `Secret[T]` *can* derive `Debug`/`Display` — the derived impl prints the field as `<redacted>` at its type, not by name. So `User { name: "alice", token: Secret[String] }` with `#[derive(Debug)]` prints `User { name: "alice", token: <redacted> }` without the programmer having to write a custom `Debug`.
  - Serialization: `Secret[T]` does not derive `Serialize` / `Deserialize`. Explicit `serialize_expose` / `deserialize_wrap` methods exist for the cases that genuinely need wire transit (TLS body, encrypted-at-rest). Rare and auditable.

- **A2. Attribute on the field: `#[sensitive]`.**
  ```kara
  #[derive(Debug, Serialize)]
  struct User {
      name: String,
      #[sensitive] token: String,
  }
  ```
  - Compiler enforcement: `#[sensitive]` participates in derive codegen. `Debug` prints `<redacted>`; `Serialize` skips the field (or `<redacted>`, configurable per-format); `Hash` participates normally (secrets are commonly keyed). Does *not* change the type — `user.token` is still `String`, assignable, passable as `ref String`.
  - No zeroize-on-drop without additional opt-in (attribute on the whole struct).
  - No constant-time equality — `==` on the field works as usual.

- **A3. Both.** Type wrapper as the primary mechanism (strong guarantees: redact, zeroize, constant-time); attribute as a lightweight fallback for fields that must stay a plain `String` for API compatibility.

- **A4. Neither — library convention.** Publish a `Secret[T]` crate with `no_debug`-style lints but no compile enforcement. Matches Rust's `secrecy` crate.

### The case for A1 (type wrapper)

- **Type-level guarantee.** If a field is `Secret[String]`, there is *no syntactic way* to log it or serialize it without writing `.expose()` — which is greppable, reviewable, and attention-drawing. The audit question "where does this token leak?" has a mechanical answer.
- **Composes with derives.** Containing types get safe-by-default `Debug`/`Display` without writing custom impls — the cost is one wrapper at the field, not a rewrite of every log/print call site.
- **Zeroize on drop is free.** `Drop` is a trait with a fixed method (v39 A1); one impl on `Secret[T]` covers every user of the wrapper.
- **Constant-time equality.** Token / HMAC / CSRF-token comparison is constant-time by default with no room for user error. Plain `==` on a `String` is variable-time — the attack is decades old and keeps finding victims (Keyczar 2009, xAuth 2011).
- **Public API signal.** `fn login(user: String, password: Secret[String])` communicates the trust level at the signature. Consumers of the library know immediately that `password` cannot be transcribed into a log line.

### The case for A2 (attribute)

- **Incremental adoption.** A large existing type with a new sensitive field doesn't force a wrapper retrofit at every use site. Annotate one field, done.
- **No type-system tax.** `user.token` stays a `String`. No `.expose()` threading, no trait-bound churn on generic code that currently takes `String`.
- **Derive is where the leak happens.** The vast majority of real leaks are derived `Debug`/`Serialize` impls emitting field values they shouldn't. Fixing derive codegen fixes the primary attack surface directly.

### Against A4 (library only)

- Kāra's stance on memory safety and effect tracking is "language-level is the right level for cross-cutting safety properties." Secrets are cross-cutting (every `log` / `dbg` / `Display` / `Serialize` is a potential sink). A library-level solution requires every derive macro, every logger, every format-specifier routine to *know about* the library. This is exactly the architecture-of-composition problem that the effect system avoids for side effects.
- Rust's `secrecy` crate exists and is widely used, but its failure mode is exactly this: it doesn't compose with user-written derives or third-party loggers unless they opt in. Language-level enforcement closes the loophole.

### Weighing

A1 gives strong guarantees at moderate syntactic cost; A2 gives medium guarantees at zero syntactic cost; A3 ships both at minor teaching complexity (two mechanisms, pick based on field ergonomics); A4 is status-quo-ish.

The key observation: A1 and A2 are orthogonal. A1 is about the *type* being self-describing ("this value is a secret, full stop, the language will enforce that"); A2 is about the *context* being self-describing ("in this struct, treat this field as sensitive when printing"). They solve overlapping but distinct problems and can coexist without conflict.

**Lean: A3 (both).** Primary mechanism is `Secret[T]` (the type wrapper — strong guarantees, signature-level trust signaling, composition with zeroize and constant-time). Secondary mechanism is `#[sensitive]` on fields (the attribute — ergonomic escape hatch for fields that must stay `String` for wire compatibility or third-party trait impls). Two mechanisms is a teaching cost, but they serve distinguishable use cases (wrapper = "this value's purpose is to carry a secret"; attribute = "this field happens to contain sensitive data in this type") and the cost is paid once in the security chapter of the book.

### Secondary questions (if A1 or A3 lands)

- **`Secret[T]` outside `std.secret` or prelude?** Prelude inclusion is probably correct — it's referenced as often as `Option` / `Result` once the pattern is normalized. Cost: one more prelude symbol.
- **`Secret[T]` Clone semantics.** Does `Secret[String]` clone produce a second `Secret`? Yes — `Clone` impl that clones inner and re-wraps. Zeroize happens per-instance; cloning a secret is explicit copying of the secret material and is fine (distinct from leak).
- **`Secret[T]` and `Copy`.** `Secret[T]` is never `Copy`, even if `T` is (`u64` auth tokens). Copy would mean silent bit-duplication, which defeats the audit trail. Explicit `.clone()` only.
- **`expose` naming.** Alternatives: `reveal`, `inner`, `unwrap_secret`, `extract`. `expose` communicates the attention it deserves; `unwrap_secret` pattern-matches to `Option.unwrap`'s "this better be safe" ethos. Pick at resolution.
- **Does `expose` show up in profile restrictions?** E.g., in an `app` profile, could a lint flag every `.expose()` in a log-adjacent function? Likely out of scope for v40 — revisit when `karac explain` catalogs attention points.
- **Zeroize on drop — `String` specifically.** `String` has a heap buffer; zeroizing means overwriting the heap region before `drop_in_place` releases it to the allocator. Allocator may still hand pages to another process via mmap; kernel-level defenses (mprotect, guard pages) are out of scope. The guarantee is "no plaintext in this process's heap after drop" — not "no plaintext on the machine ever."

## Group B — Untrusted / tainted input

### S2. `Untrusted[T]` taint marker  *(✓ RESOLVED — 2026-04-20 → B4)*

**Resolution.** Defer entirely (B4). No prelude reservation, no marker shipped in v1. Landed in `docs/deferred.md § Taint Tracking (`Untrusted[T]` / `Validated[T]`)` as a P2 entry with re-evaluation triggers and a non-breaking guarantee.

**Why B3 rejected.** Reserving the `Untrusted` prelude name without behavior is a worse signal than absence — users see the name, assume the language has an opinion, find the opinion empty, and lose trust in the prelude's precision. Kāra has namespaces, so squat-prevention (`user_crate::Untrusted` vs. `std.taint.Untrusted`) is cosmetic rather than operational.

**Key triggers for re-evaluation** (full set in `deferred.md`): enough v1 stdlib surfaces ship to stabilize the sink set; a credible Kāra-shaped proposal for validator composability emerges with worked examples; or a concrete injection-bug incident demonstrates existing defenses (effect-system capability gating, parameterized resources at sink boundaries, parse-before-use) are insufficient.

**Interim posture.** Injection prevention rests on effect-system capability gating (`reads(Network)` declares external data entering a function; `sends(Db)` declares a database sink) plus convention (parse-before-use, typed query builders at the library layer). This is documented as a valid end state, not a stopgap.

**Why foundation-breaking (with caveats).** SQL injection, shell injection, path traversal, SSRF-via-URL-concat, XSS, template injection — the entire family of injection bugs shares one structural shape: *data that originated from an external boundary flowed into a sink that interpreted it as code or as a structured address without an intervening validation step*. A type-level taint marker — `Untrusted[T]` at boundaries, `.validate()` step to strip it before sinks — catches this class at compile time.

The caveat: "what counts as a sink" is open-ended. SQL drivers are sinks. `Process.spawn` is a sink. File path join when the base is outside the intended directory is a sink. URL construction is a sink. An HTML template engine is a sink. Each needs to opt into "I require a `Validated[T]` here." The language-level primitive is just the marker and the validation step; the utility depends on the ecosystem adopting the pattern.

This is not foundation-breaking in the same sense as Group A — it's additive rather than retroactively-impossible. `Untrusted[T]` can ship in v1.1 without breaking v1 code, provided the taint primitives don't need to change shape. The call for v1 is whether to *reserve the surface* now (prelude name, stdlib module, `Validator` trait) so that v1 libraries can be written taint-aware, or defer the entire topic to post-v1.

### Design question

**Q2.** Does Kāra ship a taint-tracking primitive for v1, or reserve the surface and defer?

### Options

- **B1. Ship `Untrusted[T]` with minimal validator plumbing in v1.**
  ```kara
  fn parse_request_body(req: ref HttpRequest) -> Untrusted[String] { ... }

  fn insert_user(db: mut ref Db, name: Untrusted[String]) -> Result[(), Err] {
      let valid = name.validate(NameValidator)?;   // Validated[String]
      db.exec("INSERT INTO users VALUES (?)", [valid])
  }
  ```
  - `Untrusted[T]` has no methods except `.validate[V: Validator[T]](v: V) -> Result[Validated[T], V::Error]` and `.as_raw_untrusted() -> ref T` (explicit, attention-drawing escape for cases the language cannot model — logging the raw bytes, hashing for lookup, etc.).
  - `Validated[T]` wraps `T` and auto-derefs to `ref T` at read; can be `.into()`-converted back to owned `T` at the sink.
  - `Validator` is a user-defined trait; stdlib ships a handful (`NonEmpty`, `MaxLen[N]`, `MatchRegex`, `AsciiPrintable`, `PathSafe`).
  - Sinks are library-level — SQL driver, `Process.spawn`, `Path.join` opt into requiring `Validated[T]` instead of `T` in their API.

- **B2. Ship *only* the marker and defer validators.** `Untrusted[T]` exists, `.validate` is a method the user supplies via free-standing `fn validate_name(u: Untrusted[String]) -> Result[String, Err]`. No `Validator` trait. Simpler, but sinks can't bound-check against "this input came through some validator" — they just take `String` as today.

- **B3. Reserve the prelude name only.** `Untrusted` is a reserved symbol in v1 but implements nothing. v1.x or v1.1 fills in the body. Prevents user crates from squatting the name.

- **B4. Defer entirely.** No language-level taint in v1. Document as a post-v1 capability. Risk: v1 ecosystem evolves conventions that conflict with a later language-level primitive.

### The case for B1 (full ship)

- **Catches the real bug class.** Injection bugs are among the top 3 OWASP categories decade after decade. A working taint primitive in v1 means v1 stdlib surfaces (`std.http`, `std.process`, future SQL driver) can be written sink-typed from day one.
- **Rides the effect system's capability reasoning.** `reads(Network)` declares that a function has external input flowing into it; `Untrusted[T]` lets the *type* carry that provenance too. The two mechanisms are complementary — effects track *which capability produced the data*; taint tracks *whether it's been checked*.
- **Library-discoverable.** Sinks that take `Validated[T]` instead of `T` communicate their contract at the signature. No documentation archaeology.

### The case against B1

- **Sink-coverage gap.** For every sink that takes `Validated[T]`, there are ten that take `T`. Users routinely `.as_raw_untrusted()` to thread values through a non-aware API, and at that point the type-level guarantee dissolves. The value degrades gracefully but the *expectation* set by shipping it may not — users assume their code is safe because types compile.
- **API churn.** Every stdlib surface that takes external input needs to decide: does `std.fs.read(path: Path)` take `Validated[Path]` or `Path`? If the answer is `Path`, the marker is bypassed; if `Validated[Path]`, every caller that has a `Path` needs to `.validate()`. Pick one. The wrong pick is a daily friction.
- **Validator composability is non-trivial.** `MaxLen[10]` + `AsciiPrintable` as a single `Validator[String]` vs. two chained validators is a design question with several right answers. Under-designing here means users hack around it; over-designing blows v1 scope.
- **Scope creep risk.** A taint system done well involves flow-sensitive analysis (was this value *derived from* an untrusted value?), sinks-as-capabilities integration, and a mature validator library. v1 scope does not accommodate all of this.

### Weighing

B1 is the most ambitious and most valuable; B4 the simplest and least valuable. B3 is the "buy an option" move — reserve the name, prevent squatting, defer the work.

The scope concern in B1 is real. A partial taint system is worse than none if it creates false confidence. The API-churn concern is also real — forcing every `std.fs` function to take `Validated[Path]` is a tax on common code to prevent an uncommon case.

**Lean: B3 (reserve only).** Kāra ships v1 with `Untrusted` / `Validated` as reserved prelude names that don't yet implement anything — or implement the minimal `.validate()` method but with no stdlib `Validator` trait surface. Stdlib sinks in v1 take `T`, not `Validated[T]`. v1.1 or v1.2 introduces the full `Validator` trait, a stdlib validator library, and sink-by-sink adoption based on operational experience. This preserves the option without committing to an under-specified design.

This lean is weaker than A3 in Group A — open to being argued down to B1 if a credible scope proposal exists, or up to B4 if the reservation value is judged low.

### Secondary questions (if B1 or B2 lands)

- **Effect-system integration.** `reads(Network)` / `reads(Env)` / `reads(FileSystem)` all produce external data. Should these functions' *return types* be `Untrusted[T]` automatically? Too coercive — `read_config_file` returns structured config, not raw bytes, and by the time the user has called it the parsing has already happened. Better: a convention that *deserialization boundary* functions return `Untrusted[T]`, and stdlib deserializers expose this at their API.
- **Taint propagation.** Is `u.to_lowercase()` still `Untrusted[String]`? Yes — transformation preserves taint. `u.len()` returns `Untrusted[usize]` or `usize`? Likely `usize` — length is not injectable content. The rule needs a coherent story about what operations are *sanitizers* (strip taint) vs *transforms* (preserve taint) vs *derivations* (produce untainted derived data like length).
- **Generic containers.** `Vec[Untrusted[String]]`: does iterating yield `ref Untrusted[String]`? Yes, by construction. Does `Vec[Untrusted[String]].sort()` work? Yes — sort doesn't sink the contents. Does `Vec[Untrusted[String]].join(",")` produce `Untrusted[String]`? Yes — concatenation of tainted strings is tainted. Rule: any op whose output semantically carries content from the input carries taint.
- **Interaction with `Secret[T]`.** `Secret[Untrusted[String]]`? Legal composition, but confusing — one wrapper says "don't print this," the other says "validate this before sinking." The type system allows it; stylistically, `Secret[String]` is usually sufficient because secrets are produced by our code (token mint, etc.) rather than accepted from external boundaries. If external tokens are accepted (`Bearer` headers), `Untrusted[String]` validates format and length, then constructor wraps in `Secret`. The two stages are sequential.

## Group C — Unsafe audit trail

### S3. Mandatory `// Safety:` doc comment on `unsafe` blocks  *(✓ RESOLVED — 2026-04-20 → C2)*

**Resolution.** Ship C2 — warning, default-on, suppressible with `#[allow(undocumented_unsafe)]`. Landed in `docs/design.md § Unsafe Escape Hatch § `undocumented_unsafe` lint`. Key decisions:

- **Rule:** an `unsafe { ... }` block is preceded by a line-comment or doc-comment whose first non-whitespace content after `//` begins with `Safety:` (case-insensitive) and is followed by at least one non-whitespace word. Multiple `unsafe` blocks in the same function each require their own comment.
- **`unsafe fn`:** require a `# Safety` section in the doc comment — the same rule applied to the doc-comment grammar.
- **Suppression:** `#[allow(undocumented_unsafe)]` at block / function / module level. `#[deny(...)]` or project-wide `-D warnings` promotes to error.
- **Tutorial / stdlib posture:** ships with the lint at warn level and every `unsafe` block carrying a `Safety:` comment.

**Why C1 not chosen.** Kāra already promotes warnings to errors in CI via `-D warnings`, so C2 + `-D warnings` is operationally C1 for disciplined projects while preserving an escape hatch for research/exploratory code. C1's "no opt-out" stance is stricter than any other Kāra lint and would be inconsistent with the project's overall lint posture.

**Why foundation-breaking-adjacent.** Kāra's `unsafe` is used for MMIO registers, inline assembly, pointer arithmetic, unchecked transmute (`design.md:3385`, `:6714`, `:6776`). Every `unsafe` block is a trust-boundary assertion by the author: *"I have checked the invariant that the safe parts of the language can't check."* Making this assertion invisible — a bare `unsafe { ... }` block with no rationale — means the reviewer, the auditor, and future-you have to reconstruct the reasoning from the code. Rust has a lint for this (`clippy::undocumented_unsafe_blocks`) but it is off-by-default; Kāra starting with the lint *on* (or as a hard error) costs approximately nothing and establishes the norm from day one.

This is cheap: it's a lint rule, not a feature. Landing it now sets the tutorial expectation (every `unsafe` example in the book carries a `// Safety:` line) and prevents the "oh we'll add it later" drift that makes the retrofit painful.

### Design question

**Q3.** Does Kāra require (lint or hard-error) a `// Safety:` comment immediately preceding every `unsafe` block?

### Options

- **C1. Hard error.** `unsafe` block without a preceding `// Safety:` line is a compile error. Requires one non-empty line after `Safety:`.
- **C2. Warning (default-on).** Same check, but at warning level. `#[allow(undocumented_unsafe)]` suppresses. Matches Rust's opt-in clippy lint but on-by-default.
- **C3. Warning (default-off, opt-in via profile).** `app` profile turns it on, `embedded` leaves it off (MMIO has so much `unsafe` that the friction swamps the benefit). Or invert: `lib` on, others off.
- **C4. None.** Rely on code review to catch undocumented unsafe.

### The case for C1 (hard error)

- **Zero retrofit cost.** Every example in `design.md` currently lacks the `// Safety:` line. Fixing them during the v1 doc pass is a one-time edit. Shipping v1 with some examples documented and some not is worse than picking a policy.
- **Removes the negotiation.** "I'll add the comment later" becomes "I can't compile without it." Latency to the comment being written is the length of the edit, not the length of the review cycle.
- **Consistent with Kāra's stance on explicit declaration.** Effects are declared on public functions (`design.md:6314`); ownership modes are declared on signatures (v32 resolution); `unsafe` asserting a non-local invariant is in the same family. Hiding the reason is inconsistent.

### The case for C2 (warning)

- **Escape hatch for research / exploratory code.** `main.kara` one-shot with a single `unsafe` for perf experiment doesn't need a ceremony paragraph. A warning is visible; an error is friction.
- **Matches `#[allow(...)]` infrastructure.** Same mechanism as `#[allow(float_in_serialized_type)]` at `:1239`. Zero net complexity.

### The case for C3 (profile-gated)

- **Acknowledges asymmetry.** Embedded code has 10x the `unsafe` density of application code. Ceremony that's reasonable in an app is noise in a driver.

### Against C4

- Already argued above — losing the audit trail for zero benefit.

### Weighing

C1 and C2 are both reasonable. C3 over-engineers for v1 — profile-specific lint levels are a mechanism that doesn't yet exist for any other lint, and introducing it just for this one lint is out of proportion.

C1 vs. C2 is a friction-vs-strictness tradeoff. C2 is safer in the "we might be wrong about the right ceremony level" sense; C1 is stronger in the "we establish the norm by construction" sense. C2 with a clear `#[allow(...)]` story is almost as good as C1 operationally.

**Lean: C2 (warning, default-on, suppressible with `#[allow(undocumented_unsafe)]`).** Ship v1 with every `design.md` / tutorial example carrying a `// Safety:` line, and the lint enabled by default. Research code and quick experiments suppress with `#[allow(...)]` at the block or function level. Upgrade to C1 in a future major version if the warning-suppression rate is low enough to make hard-error tenable without ceremony revolts.

### Secondary questions (if C1 or C2 lands)

- **What counts as a `// Safety:` comment?** Structured choice: a line-comment or doc-comment *immediately preceding* the `unsafe { ... }` block, whose first non-whitespace content after `//` starts with `Safety:` (case-insensitive), with at least one non-empty word following. Permissive enough to not fight formatters; strict enough to prevent drive-by `// Safety: ok` deflection.
- **Unsafe functions vs unsafe blocks.** `unsafe fn` declarations require a `# Safety:` section in the doc comment (same rule, applied to the doc-comment grammar). Matches Rust's guideline; encoded as lint.
- **Block-level granularity.** One comment per block is sufficient; multiple unsafe blocks in the same function each need their own comment — different invariants.
