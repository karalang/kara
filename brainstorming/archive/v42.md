# Brainstorming v42 — `karac init` (project scaffolding)

**Status:** ✅ RESOLVED — 2026-04-21. All 11 items landed in canonical docs. This doc is archived.

**Resolution summary.** S1 → single subcommand `karac init [<name>]` with positional form (no `--new` alias, no `--name` override); S2 → abort on manifest/src collision with `--force` override, subdir form aborts if non-empty (no `--force`); T1 → `--bin` default, `--lib` opt-in, mutual exclusion preserved (workspaces handle mixed use cases); T2 → four files per template (`kara.toml`, `src/{main|lib}.kara`, `src/{main|lib}_test.kara`, `README.md`) plus `.gitignore` from E1; T3 → canonical manifest shape (`name`, `version = "0.1.0"`, `authors = []`, explicit `edition`, `[dependencies]`) with slice-2 parser follow-up patch (in same CR) expanding allow-list to parse the extra keys silently; T4 → classic hello world; T5 → `pub fn add` with `///` docstring, companion `lib_test.kara` tests it, `main_test.kara` ships a placeholder; V1 → strict `[a-z][a-z0-9_]*` + keyword rejection at scaffold time, abort with suggestion, no auto-translate; E1 → write `.gitignore` with single entry `/dist/`, don't run `git init`; E2 → README handled in T2, no LICENSE, no `.editorconfig`; R1 → standalone peer CR, lands immediately after CR-24 slice 2 commit and before slice 3; R2 → no breaking-change risk (template choices don't propagate).

## Context — what exists today

- CR-24 slice 2 landed `src/manifest.rs` with `Manifest { name, edition, warnings }`. `parse_manifest` accepts `[package]` with `name` (required, non-empty string) and `edition` (optional, validated against `KNOWN_EDITIONS = ["2026"]`). Every other `[package]` key emits a **soft warning** ("ignored" wording) — see `tests/manifest.rs:171` `unknown_package_keys_soft_warn_but_do_not_fail`. Non-`[package]` sections (`[dependencies]`, `[workspace]`, `[build]`, …) are silently ignored per `tests/manifest.rs:138`.
- `discover_project_root` walks up from CWD looking for `kara.toml`; `E0227 NotInsideKaraProject` fires when project-mode subcommands run outside a project.
- `karac build` (no-file form) is wired but the project-mode path is a stub pending slices 3+ (file walker). Single-file `karac run file.kara` remains unchanged — the escape hatch for book examples and one-off scripts.
- The CLI has no `init` / `new` subcommand. The help text (`src/cli.rs:253`) lists `run`, `check`, `build`, `query`, `fmt`, `help`, `version`.

## Context — what the design already commits to

From `docs/design.md § Module System` and § Package System (consolidated from v41 resolutions):

- Project layout: `myproject/{kara.toml, src/{main.kara|lib.kara, …}, examples/}`. `main.kara` and `lib.kara` are mutually exclusive; items in either are hoisted to the crate root.
- `main()` is the executable entry point — never `pub`, takes no parameters, returns `()` / `Result[(), E: Display]` / `ExitCode`. CLI args via `env.args()` (stdlib); `reads(Env)` propagates by effect inference.
- Edition is language-edition for forward compatibility. Currently only `"2026"` is recognized.
- v1 manifest parse surface: `[package].name` required, `[package].edition` optional-defaulted; everything else ignored. Canonical manifest example in `design.md:556-573` includes `version`, `authors`, `[dependencies]`, `[dev-dependencies]`, `[build]`, `[workspace]` — all of which are valid-but-noop until later CRs.
- Build artifact directory: `dist/<target>/` per `design.md:6250`. Not `target/` — that's a cargo-ism.
- No lockfile in v1 (the package-manager CR will introduce `kara.lock`; until then there's nothing to check in or ignore).
- Prelude auto-injects `Option`, `Result`, `Vec`, `println`, etc. — so a starter `println("Hello, world!")` needs no explicit import.

## Scope of this brainstorm

Only the **scaffolding command** — what `karac init` writes, how it's invoked, what validation applies, what ecosystem affordances (git, editor config) ship with it. Out of scope: the package-manager deliverable (registry, dependency resolution, `kara.lock` format, `karac add`), which has its own deferred track.

**Also deferred (and logged):** workspace-aware scaffolding (`karac init --workspace`). Workspace resolver/builder don't exist in v1, so scaffolding a root that `karac build` wouldn't honor would teach the wrong mental model. See `docs/deferred.md § P1 § Workspace Scaffolding` for the committed shape. V1 users hand-write the `[workspace]` root and run `karac init <name>` inside each member subdir.

## Group S — Subcommand surface

### S1. Subcommand shape  *(✅ RESOLVED — S1d: `karac init [<name>]` positional only)*

**Resolution.** Single subcommand: `karac init [<name>]`. Bare `karac init` scaffolds into CWD. `karac init myproj` creates `./myproj/` and scaffolds there. Matches `git init [directory]`'s long-standing idiom.

- **No `--new` alias.** Two ways to spell the same thing is a UX tax — docs split, users wonder if there's a subtle difference. Positional is unambiguous.
- **No `--name` override.** Directory name IS the package name. If the dir basename isn't a valid Kāra package identifier (see V1), error out: `` directory name `my-project` is not a valid Kāra package name; rename the directory or pass a valid name ``. No escape hatch that decouples dir name from package name — simpler mental model.

**Future-compatible flags** (none collide with the positional): `--bin` / `--lib` (T1), `--force` (S2). Others (`--edition`, `--quiet`) deferred until needed.

**Rejected alternatives.**
- **S1a.** Two subcommands (`init` + `new <name>`, cargo-style). Extra surface, extra command to document for zero semantic gain over the positional form.
- **S1b.** `init` only, no subdir creation. User has to `mkdir && cd` first — annoying for the common fresh-project path.
- **S1c.** `new` only, always creates subdir. Awkward for in-CWD case (CI / container setups where dir already exists).
- **S1b-variant with `--new <name>` flag.** Flag reads like "init, but actually new" — small cognitive mismatch. Positional is cleaner.

### S2. Behavior when target directory already contains files  *(✅ RESOLVED — S2b: abort on manifest/src collision, `--force` override)*

**Resolution.** Default: abort on collision with `kara.toml` or `src/main.kara` / `src/lib.kara`; tolerate other pre-existing files (README, LICENSE, `.git/`, etc.) — only scaffold what's missing. `--force` flag overrides the abort for the "regenerate, I know what I'm doing" case.

**Sub-decision — `karac init <name>` into existing directory.** If `./<name>/` already exists and is non-empty, abort. **No `--force` for the subdir path** — "create this directory" is clearer intent than "scaffold into existing dir"; if the user wants in-place scaffolding they should `cd <name> && karac init` explicitly. Empty existing `./<name>/` is fine.

**Rejected alternatives.**
- **S2c.** Abort on any non-empty CWD. Strictest; breaks the "I just `git clone`d a template and want to add Kāra" case.

## Group T — Template shape

### T1. Bin vs lib default  *(✅ RESOLVED — T1a: `--bin` default, `--lib` opt-in; mutual exclusion preserved)*

**Resolution.** `--bin` is the default; `--lib` is the opt-in flag. The flags are mutually exclusive — a Kāra package is either a binary or a library, never both, per `design.md:424`. Mixed bin+lib use cases (library + bundled CLI) are served by **workspaces** (multi-package repos), not by mixed packages.

Both templates scaffold their entry file under `src/`:
- `karac init --bin myproj` → `./myproj/{kara.toml, src/main.kara}`
- `karac init --lib myproj` → `./myproj/{kara.toml, src/lib.kara}`

There is no `bin/` or `lib/` directory at the project root — `src/` holds everything, and the entry-file name distinguishes the two.

**Why the mutual-exclusion rule holds** (reaffirmed — design.md:424 is explicit, not accidental):
1. **Cleaner hoisting.** Items in `main.kara` OR `lib.kara` hoist to the crate root; allowing both creates unnecessary collision surface in a single root namespace.
2. **Simpler package-identity model.** A `kara.toml` describes one deliverable unit.
3. **Workspaces cover the legitimate mixed use case.** A library shipping a CLI demo = workspace with two member packages (one lib, one bin) — the shape mainstream ecosystems converged on for serious projects.
4. **Divergence from cargo is principled.** "semantics-Rust, syntax-mainstream" doesn't mandate cargo parity where Kāra's stricter rule is clearer.

**Rejected alternatives.**
- **T1b.** `--lib` default. Reverses cargo; no evidence Kāra's audience skews library-first; breaks the "hello world" first-run path.
- **T1c.** No default — one of `--bin` / `--lib` must be explicit. Principled but unfriendly; extra keystroke on every init.

### T2. What files are written  *(✅ RESOLVED — T2b + T2e + title-only README: four files per template)*

**Resolution.** Both templates scaffold four files:

- **`--bin`:** `kara.toml`, `src/main.kara`, `src/main_test.kara`, `README.md`
- **`--lib`:** `kara.toml`, `src/lib.kara`, `src/lib_test.kara`, `README.md`

**Why starter test files ship now** (T2b + T2e over the minimum-viable T2a/T2d):

1. `_test.kara` colocation is the committed Kāra convention; the scaffolded file *is* that convention made visible on turn 0.
2. Nothing ships to end users before v1, and `karac test` will land before v1. A few months of dormant starter tests during CR-24 dogfooding is fine.
3. Avoids a deferred action item ("update init template when `karac test` lands"). Brainstorming-closes-for-good means no such migrations linger.

**Why README ships — title-only** (T2c over both "never" and "defer"):

1. **Forcing function for best-practice docs.** Every scaffolded project having a README in the working tree is a standing nudge — users open their editor, see it, fill it. Scaffolders that emit nothing about docs implicitly deprioritize docs.
2. **Title-only avoids template drift.** Content is `# {{project_name}}` — single line, interpolated from the package name. No opinion about sections, no `TODO:` placeholder that ages badly, nothing to maintain.
3. **Not empty.** An empty README feels like a scaffolder bug; some editors don't even show empty files clearly. Title-only reads as "canvas, start writing."
4. **Decided, not deferred.** If we weren't going to ever ship README, we'd say so; "we might add it later" creates template drift. Ship now, lock in.

**Rejected alternatives.**
- **Empty README.md:** reads as an error state, editors may hide it.
- **Title + one-line description (`# myproj\n\nA Kāra project.`):** either shipped-trivial (bad) or replaced-immediately (no net gain).
- **Title + HTML comment hint:** extra thing for users to delete; Markdown comments are confusing for beginners.

**README content (template):**

```markdown
# {{project_name}}
```

**Starter test content.**

- `src/lib_test.kara` (`--lib`) tests the starter `pub fn add` from T5 — see T5's resolution for the body.
- `src/main_test.kara` (`--bin`): trivial placeholder — `main()` isn't `pub`, so tests can't exercise it directly. The placeholder proves the test file is wired up and ships under the same naming convention as `lib_test.kara`; users replace it day one.

  ```kara
  test "placeholder" {
      assert_eq(1 + 1, 2);
  }
  ```

  Rejected alternatives for `main_test.kara`: empty-with-comment (demotes the "starter-test-is-real" intent); refactoring `main.kara` to extract a testable helper (bloats T4's clean hello-world form, couples test file to main file's internal shape).

### T3. `kara.toml` template keys  *(✅ RESOLVED — T3d: ship canonical shape, expand slice-2 allow-list in a follow-up patch)*

**Resolution.** Template writes the full canonical `[package]` shape up front. Slice 2's parser gets a follow-up patch that **expands its known-keys allow-list** so the additional keys parse silently (no semantic behavior in v1) until the package-manager CR picks them up. After that patch, the template is canonical and stays canonical — zero future template migrations, zero lingering "come back and update" bookkeeping.

**Template (both `--bin` and `--lib`):**

```toml
[package]
name = "{{project_name}}"
version = "0.1.0"
authors = []
edition = "2026"

[dependencies]
```

**Rationale for the specific keys included:**

| Key | Included? | Why |
|---|---|---|
| `name` | Yes | Required. Interpolated from scaffold argument / CWD basename. |
| `version` | Yes | Canonical per design.md. `"0.1.0"` is the universal pre-1.0 convention. |
| `authors` | Yes — **empty array** | No `git init` ships with this scaffolder (see E1), so reading `git config user.name` would be inconsistent. Empty array teaches the shape; users fill in. |
| `edition` | Yes — **explicit, not defaulted** | User sees the concept day one and isn't silently upgraded if `DEFAULT_EDITION` changes in a later release. |
| `[dependencies]` | Yes — **empty section** | Telegraphs where dependencies go; zero-cost. |
| `[dev-dependencies]` | No | Premature. Test files ship (T2) but manifest-level dev-deps have no v1 meaning and would just be noise. |
| `[build]` | No | No v1 meaning. Add when package-manager CR needs it. |
| `[workspace]` | No | Single-package init — only workspace roots have this section (deferred per P1). |

**Slice-2 follow-up patch required (part of the same CR as `karac init`):**

The parser's `[package]` known-keys set grows from `{name, edition}` to `{name, version, authors, edition}`. The top-level known-sections set grows from `{package, dependencies, dev-dependencies, build, workspace}` (already tolerated as "ignored") to the same set — the ignoring behavior stays; we just need `version` and `authors` to *not* emit the "ignored" soft warning. Unit test update: change `unknown_package_keys_soft_warn_but_do_not_fail` to cover a different unknown key (e.g., `homepage`), preserving the coverage for genuinely-unknown keys while making the canonical keys silent.

**Rejected alternatives.**
- **T3a alone.** Writes canonical shape, but without the parser patch every fresh project warns on first build — unacceptable UX.
- **T3b.** Writes only `{name, edition}`. Users copying `design.md:556-573` examples into their manifests get warned — suggests compiler is broken.
- **T3c.** T3b + explanatory comments. Teaches without committing the parser, but carries two deferred action items (populate keys + remove comments when package manager lands). Fails the "nothing deferred" test.

### T4. `main.kara` starter body  *(✅ RESOLVED — T4a: classic hello world)*

**Resolution.** Classic hello world, no commentary:

```kara
fn main() {
    println("Hello, world!")
}
```

**Why T4a over fancier alternatives:**
- Universal scaffolder norm — proves the toolchain works without teaching anything premature.
- No opinion about which advanced feature to showcase first.
- Ages well. Effect demos, `Result`-returning mains, `env.args()` — all would tie the starter to specific features that can churn. Hello world is stable across every version.
- Project title + guidance live in the README (T2), not the code file.

**Rejected alternatives.**
- **T4b.** T4a + comment pointing at the book. Small value; README covers guidance; book path may churn during CR-24.
- **T4c.** `env.args()` / `reads(Env)` demo. Concept-dense for turn 0; `env.args()` isn't implemented in the interpreter yet — starter would fail `karac run` under the interpreter path.
- **T4d.** `fn main() -> Result[(), Error]` teaching `?` from day one. Similarly concept-dense; premature for a hello-world slot.

### T5. `lib.kara` starter body  *(✅ RESOLVED — T5a: one `pub fn` with docstring)*

**Resolution.** Single `pub fn add` with a docstring:

```kara
/// Adds two integers.
pub fn add(a: i64, b: i64) -> i64 {
    a + b
}
```

**Why T5a:**
- Teaches three Kāra conventions on turn 0 that are harder to discover without seeing: `pub` visibility, `///` docstring syntax, and the `fn name(a: T, b: T) -> T` declaration shape.
- Gives the companion `src/lib_test.kara` (per T2) something concrete to test — otherwise the starter test has no subject.
- Cross-ecosystem default: cargo does this with `add(left: u64, right: u64)`, and the "add two things" example is stable enough that it ages perfectly.
- Concrete enough to be runnable/visible; small enough to not overfit to any specific domain.

**Companion `src/lib_test.kara` content (shipped by T2):**

```kara
test "add sums two integers" {
    assert_eq(add(2, 3), 5);
}
```

(Exact syntax may adjust to match `karac test`'s final shape when that CR lands — but the shape-correct starter is committed now per T2's "no deferred template updates" rule.)

**Rejected alternatives.**
- **T5b.** Empty file with a comment. Users see an empty-ish file and wonder if scaffolding worked; test file has nothing to test.
- **T5c.** T5a + pub struct demonstrating ownership. Starts to feel like a tutorial; struct choice is opinionated.

## Group V — Validation rules

### V1. Project name validation  *(✅ RESOLVED — V1a: strict at scaffold time, suggest fix, never silently translate)*

**Resolution.** Enforce full Kāra identifier rules at `karac init` time — dir name (if no positional arg) OR positional arg must match `[a-z][a-z0-9_]*` AND not be a reserved keyword. Invalid name → abort before any file is written, with a clear suggestion.

**Reject list:**
- Empty string (also caught by the manifest parser's `InvalidPackageName`, but we catch it earlier).
- Starts with digit or non-letter.
- Contains characters other than `[a-z0-9_]` — no hyphens, no uppercase, no dots, no unicode (unicode identifier rules punted to a later CR).
- Reserved keywords (`fn`, `let`, `mut`, `pub`, `if`, …) and reserved-but-unreleased like `mod`.

**Diagnostic shape:**

```
error: project name `my-app` is not a valid Kāra package name
  note: package names must match [a-z][a-z0-9_]* (snake_case, no hyphens)
  help: did you mean `my_app`?
```

**Consistency with S1 and sub-decision on the subdir form.**
- Bare `karac init` → validate basename of CWD.
- `karac init <name>` → validate `<name>`. If invalid, reject with the diagnostic above; do **not** silently sanitize (`my-app` → create `./my_app/` as the dirname). The mismatch between user-typed command and created dir would be worse UX than the reject-and-retype loop.
- Invariant preserved end-to-end: **dir name === package name === root module name**, always, across both init forms.

**Rejected alternatives.**
- **V1b.** Permissive + auto-translate (`my-app` → `my_app`). Cargo does it, but cargo has to — Rust crate names and Rust module identifiers diverge historically. Kāra has no such divergence; splitting "package name" from "module identifier" would add a translation layer forever for zero Kāra-side benefit.
- **V1c.** Accept anything, let the parser's `InvalidPackageName` fire on next build. User-hostile — error happens one step later than it should, after scaffolder already wrote files.

## Group E — Ecosystem hooks

### E1. `git init` + `.gitignore`  *(✅ RESOLVED — E1b: write `.gitignore` only, no `git init`)*

**Resolution.** Write a single `.gitignore` file with one entry: `/dist/`. Do **not** run `git init`. Scaffolder stays VCS-neutral — users pick git / hg / jj / pijul / nothing on their own terms.

**`.gitignore` template:**

```
/dist/
```

**Why no `git init`:**
- Introduces a dependency on a specific VCS tool into `karac`'s runtime requirements (and test environment).
- Forces a VCS choice on users who already picked one (or want none).
- Users who want git run it themselves in one command.

**Why `.gitignore` is still worth writing even without running `git init`:**
- Pure text, zero external dependency.
- Prevents `dist/` from polluting the user's first commit when they eventually `git init`.
- Users coming from cargo/npm expect it.

**Sub-decision — skip `.gitignore` if it already exists.** Yes. Same collision rule as S2 — never overwrite pre-existing files. If an existing `.gitignore` is missing `/dist/`, that's the user's call.

**Note on `kara.lock`:** not in the gitignore today because the lockfile doesn't exist in v1. When the package-manager CR lands, that CR updates the template (lockfile committed for bins, ignored for libs per `design.md:588`). One cohesive change at package-manager time, not a deferred action now.

**Rejected alternatives.**
- **E1a.** Run `git init` by default. Presumptuous; locks us to git as the VCS.
- **E1c.** Ship neither `.gitignore` nor run `git init`. First commit includes `dist/` — annoying.

### E2. README / LICENSE / editor configs  *(✅ RESOLVED — README in T2; no LICENSE, no `.editorconfig`)*

**Resolution.** README is handled in T2 — title-only `README.md` ships with both `--bin` and `--lib` templates as a forcing function for docs. The other two candidates:

- **LICENSE: don't ship.** License choice is genuinely opinionated (MIT / Apache-2.0 / GPL / proprietary) and scaffolding any specific one presumes audience intent the scaffolder can't know. Cargo doesn't ship one; GitHub's template picker handles it at repo-create time; users who care add one in seconds.
- **`.editorconfig`: don't ship.** Low-risk but low-value and opinionated about indent width / charset. If Kāra ever commits to a canonical indent, that belongs in `karac fmt`'s defaults, not propagated to every user project. One more deleted-by-users file.

## Group R — Implementation rollout

### R1. Where does `karac init` sit in the CR-24 slice ordering?  *(✅ RESOLVED — R1b: standalone peer CR, lands immediately after slice 2, before slice 3)*

**Resolution.** `karac init` ships as its **own standalone CR**, not as a slice inside CR-24. It lands on the schedule: slice 2 commit → `karac init` CR → CR-24 slice 3. Scaffolding is a sibling concern to the multi-file module system, not a link in its chain.

**Sequencing (pinned):**

1. **Commit slice 2 as-is** (uncommitted from prior session — `src/manifest.rs`, `tests/manifest.rs`, CLI + `lib.rs` + `Cargo.toml` edits).
2. **`karac init` CR.** Self-contained, includes three pieces in one patch set:
   - The scaffolder itself: `karac init [<name>] [--bin | --lib] [--force]`, templates per T3/T4/T5, V1a name validation, S2b collision rules, E1b `.gitignore`.
   - **T3d's slice-2 parser follow-up:** expand `[package]` known-keys allow-list from `{name, edition}` to `{name, version, authors, edition}`; retarget the `unknown_package_keys_soft_warn_but_do_not_fail` test onto a genuinely-unknown key (e.g., `homepage`).
   - CLI help text + integration tests (scratch-dir pattern from `tests/manifest.rs`).
3. **CR-24 slice 3** (file walker) and downstream slices resume.

**Why R1b over R1a (slice 2.5 inside CR-24):** Schedule-equivalent, but R1b keeps CR-24 scoped to its original charter (multi-file module system) and gives `karac init` its own rollback surface if anything goes sideways.

**Why R1b over R1c (defer until after all CR-24 slices):** Loses dogfooding benefit for the whole module-system track. Slices 3-9 all benefit from real scaffolded projects.

**Why the T3d patch lives inside the `karac init` CR, not a slice-2 addendum:** The allow-list expansion has no reason to exist without a scaffolder emitting manifests that use those keys. Coupling it to the CR that needs it avoids orphan patches.

### R2. Breaking-change surface  *(✅ RESOLVED — R2a: none. Template choices are one-time; don't propagate retroactively)*

**Resolution.** Zero breaking-change risk. `karac init` has no existing users (it doesn't exist), and template choices bake in per-project at scaffold time — they don't propagate to already-scaffolded projects. As long as templates stay valid under the v1 language spec, every scaffolded project compiles forever regardless of later template evolution.

**Audit of baked-in decisions (all non-breaking):**

- **Manifest template (T3d).** Scaffolded projects keep their v1 manifest shape. When the package-manager CR lands, old projects don't auto-migrate; release notes document any new sections to add. Existing templates compile fine because slice 2's parser (post-T3d allow-list expansion) tolerates all those keys.
- **Starter bodies (T4a / T5a).** Static. Changes to `println` semantics or `fn main()` return-type conventions don't rewrite scaffolded code.
- **`.gitignore` (E1b).** If `dist/` is renamed or supplemented later, old projects won't pick up the change. User edits their gitignore once.
- **Test file syntax (T2 sub-decision).** Placeholder `test "..." { ... }` shape matches the committed design. If `karac test`'s final syntax drifts, users fix a trivial placeholder line — but they'd be rewriting the placeholder for real content anyway.
