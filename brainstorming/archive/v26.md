# Brainstorming v26 — Generic `Slice[T]`: lift forward to MVP or keep deferred?

**Status: RESOLVED 2026-04-18.** All five questions landed on their leaning options. Summary at the top; original analysis below for future reference.

## Resolutions

- **Q1 → (b) Lift `Slice[T]` to MVP.** First-class primitive, shape `{ ptr: *const T, len: i64 }`, lives in the prelude. Range indexing ships alongside.
- **Q2 → (b)+(c) Auto-coerce at call boundaries, plus `.as_slice()` escape hatch.** `ref Vec[T]` / `ref Array[T, N]` / `Slice[T]` coerce to `Slice[T]` at call sites. Coercion does *not* fire on let-bindings or plain assignments — use `.as_slice()` / `.as_slice_mut()` in expression position where coercion can't reach.
- **Q3 → (a) `StringSlice` and `Slice[T]` are parallel-but-distinct.** Not `Slice[u8]`, not `Slice[char]`. The UTF-8 invariant lives in the `StringSlice` type, not in the element kind.
- **Q4 → (c) Single `Slice[T]` with a `mut` modifier.** `mut Slice[T]` uses the same modifier machinery as `mut ref T`. `ref mut Vec[T]` / `ref mut Array[T, N]` coerce to `mut Slice[T]` at call boundaries. One type name, one coercion rule, mutability threaded as a modifier.
- **Q5 → (a) Both implicitly `Copy`.** `Slice[T]` and `StringSlice` are unconditionally `Copy` regardless of element type — the slice header is a borrow handle, not a value-carrying type. `mut Slice[T]` is *not* `Copy` (uniqueness via the aliasing rule that governs `mut ref T`). `StringSlice`'s Copy status pinned explicitly in design.md.

## Original analysis (for reference)

Triggered by a gap-audit against the Rust book TOC. Rust book ch4 ("Understanding Ownership") introduces slices — `&[T]` and `&str` — as a core P0 concept. Kāra has `StringSlice` (design.md:2143, 2155-2174) as a zero-copy string view, but the general `Slice[T]` type is deferred: it appears once in design.md (line 2556) as the return type of `impl Index[Range[i64]]` with `type Output = Slice[T]`, "deferred to Phase 8 alongside the full stdlib."

The asymmetry is real. `StringSlice` ships in MVP because it's mentioned in the stdlib prelude (design.md:386). `Slice[T]` for every other sequence type does not. Meanwhile, Kāra's ownership story uses `ref Vec[T]` and `ref Array[T, N]` as borrows — but those are container-specific: a function written against `ref Vec[i64]` cannot be called with a `ref Array[i64, 5]`, and size-polymorphic `ref Array[T, N]` leaks the length into the signature.

## Discoveries from the survey

1. **`StringSlice` exists as a first-class type with its own section.** Lines 2143, 2155-2174 specify: a pointer + offset + length view, borrow rules via `ref T`, returned from parsing/splitting functions, convertible via `.to_string()`. Self-contained and orthogonal to `ref String`.
2. **`Slice[T]` appears once, without definition.** Design.md:2556 mentions it as the return type of range indexing. No fields specified, no methods, no relation to `ref Vec[T]` / `ref Array[T, N]`.
3. **No generic "view into contiguous memory" abstraction in MVP.** A stdlib function that operates on a sequence must choose between:
   - `ref Vec[T]` — forces caller to own a `Vec`, doesn't accept `Array`
   - `ref Array[T, N]` — size-polymorphic; every caller monomorphizes a distinct function
   - An `Iterable[T]` trait bound — works but loses O(1) indexed access
4. **Range indexing depends on `Slice[T]`.** Design.md:2556 says `collection[a..b]` returns `Slice[T]`. If `Slice[T]` doesn't exist in MVP, range indexing can't ship in MVP either. This is already implied by the "deferred to Phase 8" note, but it means a central Rust-style idiom is unavailable until Phase 8.
5. **`StringSlice` is consumed in stdlib signatures.** `impl From[StringSlice] for String` (design.md:1711) and `Vec[StringSlice]` return types in parsing examples (design.md:2165) both rely on `StringSlice` being a usable type, not a deferred one. The pattern is established; extending it to `Slice[T]` is a natural generalization.

## Q1. Does MVP need a generic `Slice[T]`?

Three options:

- **(a) Keep deferred to Phase 8 (status quo).** `Slice[T]` stays absent from MVP. Users write functions over `ref Vec[T]`, `ref Array[T, N]` with size generic, or `Iterable[T]` bound. Range indexing is unavailable. Book chapters that would teach slices teach `ref Vec[T]` instead.
- **(b) Lift `Slice[T]` to MVP as a first-class primitive.** Same shape as `StringSlice` but parameterized: `{ ptr: *const T, len: i64 }`. Supports indexed access, length, iteration. Lives in the prelude. Range indexing ships in MVP. `ref Vec[T]` / `ref Array[T, N]` coerce to `Slice[T]` at call boundaries.
- **(c) Unify ownership-tier views with a slice-like coercion.** No new type; instead, `ref Vec[T]` and `ref Array[T, N]` both coerce to a shared "borrowed sequence" type at call boundaries, similar to how Rust's `&Vec<T>` derefs to `&[T]`. Compiler-driven, not a user-level type. Range indexing returns this coercion target.

Tradeoffs:

- (a) minimizes MVP surface area. Cost: users writing sequence-consuming functions have to pick between overly-restrictive (`ref Vec`) and overly-generic (`Iterable`) signatures. Range indexing is missing. Book ch4 has to describe `StringSlice` as a special case without a general rule.
- (b) is the cleanest — one type, mirrors `StringSlice` exactly, enables range indexing. Cost: compiler has to lower `ref Vec[T]` and `ref Array[T, N]` to `Slice[T]` at call sites (automatic coercion), and stdlib gains a type. Not a huge lift if `StringSlice` already exists; most machinery generalizes.
- (c) avoids a new surface type but introduces a compiler coercion that users can't name. Hard to teach — "it's a `ref Vec` until it crosses a function boundary, then it's a something-else." Breaks the principle that every type in a signature is user-visible.

**Lean:** (b). `StringSlice` sets the precedent; `Slice[T]` is the obvious generalization. The incremental cost is small: one generic type, one coercion rule, and range indexing ships along with it. Keeping it deferred means Rust-ch4-level idioms are unreachable until Phase 8, which is at odds with a book chapter on ownership that needs to teach borrowed sequences.

## Q2. What's the relationship between `Slice[T]` and `ref Vec[T]` / `ref Array[T, N]`?

If Q1 lands on (b), the coercion rules need to be spelled out.

- **(a) Explicit construction only.** Users write `slice.from_vec(v)` or `v.as_slice()` to produce a `Slice[T]`. No implicit coercion; `ref Vec[T]` and `Slice[T]` are distinct types with distinct call sites.
- **(b) Auto-coerce at call boundaries.** A function taking `ref Slice[T]` accepts `ref Vec[T]`, `ref Array[T, N]`, and `Slice[T]` itself. Mirrors Rust's deref coercion.
- **(c) Auto-coerce plus a `.slice()` method.** (b) plus a named fallback for expression-position use (e.g., inside a closure).

**Lean:** (b)+(c). Coercion at call boundaries keeps signatures readable; a named method handles expression-position cases where coercion can't reach (analogous to the prefix-literal escape hatch from v25).

---

## Q3. Subset relationship with `StringSlice`

Is `StringSlice` just `Slice[u8]` with string-specific methods, or a distinct type?

- **(a) Distinct — status quo.** `StringSlice` keeps its own name. Rationale: strings are not byte sequences in Kāra (UTF-8 invariant, `char_at`, `bytes()` explicit), so `StringSlice` carries semantic constraints `Slice[u8]` doesn't.
- **(b) Alias — `StringSlice = Slice[u8]` with a newtype wrapper for the UTF-8 invariant.** Fewer concepts, but complicates the UTF-8 guarantee.
- **(c) `StringSlice` is `Slice[char]`.** Technically a `char` slice doesn't work because `String` is UTF-8, not a sequence of `char` — characters aren't fixed-width.

**Lean:** (a). `StringSlice` already has its own section and semantics; making it a `Slice[u8]` alias would force the UTF-8 invariant out of the type system. Keep them parallel-but-distinct: same shape, independent identity.

## Q4. Mutable slice variant — how do in-place operations over a borrowed range work?

Rust distinguishes `&[T]` (read-only view) from `&mut [T]` (mutable view). In-place `sort`, `fill`, `reverse`, `iter_mut` operate on the mutable variant. Kāra needs a parallel mechanism or a clear alternative.

Options:

- **(a) `Slice[T]` is read-only; mutable operations require the backing container directly.** Callers wanting in-place sort on a sub-range pass `ref mut Vec[T]` or `ref mut Array[T, N]` and use index-based APIs. No `mut Slice[T]`. Cost: no way to write a function like `fn sort_range[T: Ord](xs: mut Slice[T])` that works over both Vec and Array.
- **(b) Add `SliceMut[T]` as a second type.** Parallel to Rust's `&[T]` / `&mut [T]` split. `ref Vec[T]` coerces to `Slice[T]`; `ref mut Vec[T]` coerces to `SliceMut[T]`. Two types, two coercion paths.
- **(c) Single `Slice[T]` with a mutability marker in the type.** `Slice[T]` and `mut Slice[T]` — the `mut` modifier carries through. One type, one coercion path, mutability tracked like other Kāra ownership modifiers.

Tradeoffs:

- (a) keeps surface area small but forces stdlib to ship `Vec` and `Array` versions of every mutable sequence operation, and rules out third-party sequence containers from benefiting.
- (b) matches Rust's model exactly. Cost: two types for one concept; the coercion table doubles.
- (c) leans on Kāra's existing `mut` as a modifier, keeping one type name and one coercion rule. Matches the `ref T` / `mut ref T` distinction already established in Kāra's ownership story.

**Lean:** (c). A single `Slice[T]` with a `mut` modifier fits Kāra's existing conventions (parameter mode inference treats `ref` and `mut ref` as one mechanism with a modifier, not two separate types). Worth verifying against Part 1–3 of Tiered Ownership to confirm the modifier can thread through the type system cleanly.

## Q5. Copy semantics — is `Slice[T]` (and `StringSlice`) implicitly `Copy`?

A slice is a small aggregate (pointer + offset + length for `StringSlice`; pointer + length for `Slice[T]`). Trivially copyable at the machine level. But Kāra's `Copy` is opt-in and affects move semantics, so this must be pinned explicitly.

Options:

- **(a) Both implicitly `Copy`.** A slice passed to a function by value doesn't invalidate the caller's binding. Matches Rust's `&[T]` and `&str` (which are `Copy`). Ergonomic — users can freely duplicate slices.
- **(b) Neither `Copy`.** Slices move on assignment, like any other Kāra value. Cost: every function call that takes a `Slice[T]` by value consumes it; to reuse, callers must pass `ref Slice[T]` or call `.clone()`. Rust-style ergonomics lost.
- **(c) `Copy` when `T` is `Copy`, not otherwise.** Conditional on the element type. Matches how `Array[T, N]: Copy` would work if it existed.

Tradeoffs:

- (a) is the ergonomic default and matches StringSlice's likely intended behavior (nothing in design.md rules it out). A slice is semantically a borrow-handle, not a value-carrying type.
- (b) is Kāra-conservative but fights the whole point of a slice. A function taking `Slice[T]` by value and one taking `ref Slice[T]` are effectively identical in semantics; forcing `ref` everywhere adds noise without safety gain.
- (c) is subtle — element `Copy`-ness is about owned element semantics, but a slice doesn't own its elements. The slice header itself is always trivially copyable regardless of `T`.

**Lean:** (a). Both `Slice[T]` and `StringSlice` are implicitly `Copy`. Gap: design.md doesn't currently state `StringSlice`'s Copy status explicitly — this resolution is the pass that fixes both.
