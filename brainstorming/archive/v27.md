# Brainstorming v27 — Stdin and basic I/O API surface

**Status: RESOLVED 2026-04-18.** All five questions landed on their leaning options; Q3 kept (c) after the reconsideration flag was examined. Summary at the top; original analysis below for future reference.

## Resolutions

- **Q1 → (b) Partial spec.** design.md gains a § Standard I/O Surface section with names, module paths, signatures, and effect contracts. No implementation bodies — that's Phase 8. Mirrors the existing `print`/`println` precedent.
- **Q2 → (a) Minimum set only.** Seven functions ship in the spec: `io.read_line`, `io.read_to_string`, `io.flush`, `io.flush_stderr`, `env.args`, `env.var`, `fs.read_to_string`, `fs.write`. File handles, streaming, network, and process spawning deferred to Phase 8 detail.
- **Q3 → (c) Hybrid kept.** `IoError` covers stdin + filesystem; `env.VarError` stays separate for its two tight lookup variants (`NotPresent`, `NotUnicode`). `impl From[VarError] for IoError` is provided so CLI tools that mix env and file reads can collapse to one error type at `?` sites. The reconsideration flag around `InvalidUtf8` / `NotUnicode` overlap was examined and declined — one overlapping variant is not enough reason to collapse two otherwise-disjoint error domains.
- **Q4 → (a) `Result[String, IoError]`.** Invalid UTF-8 surfaces as `IoError.InvalidUtf8`. Preserves `String`'s UTF-8 invariant. A bytes-level `read_line_bytes` variant is deferred to Phase 8.
- **Q5 → (b) Split `Stdio` into three resources.** `Stdin` (reads), `Stdout` (writes), `Stderr` (writes). Sequential-order guarantee holds — the split enables parallelism only in explicit `par { }` or provably-independent call-graph regions, never across plain sequential statements.

## Applied changes

- `docs/design.md` new § Standard I/O Surface — module paths, function signatures, error types, UTF-8 policy, buffering stance, guessing-game and CLI-tool examples.

- `docs/implementation_checklist.md` — Phase 8 item covering the minimum I/O surface (error types, stdin functions, flush, env, fs, and a book-chapter acceptance criterion).

## Original analysis (for reference)

Triggered by a gap-audit against the Rust book TOC. Rust ch2 ("Guessing Game") leans on `io.stdin().read_line(&mut buf)`; ch12 ("I/O Project") leans on `std.env.args()`, `std.fs.read_to_string`, and `std.process.exit`; throughout the book, `File.open`, `BufReader`, and related APIs are table stakes. Kāra's design.md declares the relevant *effects* (`reads(Stdio)` line 3129, `reads(FileSystem)` line 3128, `reads(Env)` for args from memory), and lists `print` / `println` / `eprintln` (line 388-391) as stdlib functions — but there are no stdin, file, or env signatures in the spec. The book's getting-started tutorials can't exist without them.

---

## Discoveries from the survey

1. **Output is specified; input is not.** `print(value)` / `println(value)` / `eprintln(value)` are specified as stdlib functions with declared effects (design.md:388-391). No equivalent for reading a line from stdin, reading a file, getting env vars, or reading command-line args.
2. **Effects exist; API shapes don't.** `reads(Stdio)`, `reads(FileSystem)`, `reads(Env)` are defined as built-in resources (design.md:3124-3131 and memory `project_main_cli_design`). But there's no function signature that would *produce* those effects — only the effect taxonomy.
3. **`env.args()` referenced but not specified.** Design.md:3122 mentions `env.args()` as the source of the `reads(Env)` effect, and the memory entry `project_main_cli_design` says "CLI args via env.args() + reads(Env) inferred." But no signature, return type, or error handling is spelled out.
4. **File I/O referenced in concurrency examples.** Design.md:4798 shows `let local_data = fs.read_sync(path)` inside a `par` block — that's a stdlib method whose signature, module path, and error behavior are undefined in the spec.
5. **`main()` return type and exit path specified, but the stdin path for reading is not.** Design.md:3086-3122 specifies `main()` signatures and error display. Nothing comparable for stdin.
6. **Book chapter gap.** There is no ch01-hello-kara equivalent of ch2's Guessing Game. Without stdin APIs, the tutorial chain that Rust's book uses as its on-ramp can't be replicated.

The upshot: Kāra's MVP has verbs (effects) without nouns (APIs) for I/O. The compiler can check `reads(Stdio)` in an effect set, but users can't write code that produces one for stdin — because no function that reads stdin exists in the spec.

## Q1. Is stdlib I/O surface a design-spec concern or a Phase 8 implementation concern?

Three framings:

- **(a) Pure Phase 8 / stdlib concern.** Spec says nothing about I/O function shapes; Phase 8 defines them when the stdlib is built. Rationale: the compiler doesn't need to know about `stdin.read_line`; effects are the contract.
- **(b) Partial spec: names, module paths, and error contracts, but not method bodies.** Design.md gains a "Standard I/O Surface" section: module `io` has `read_line`, `read_to_string`, etc. with declared signatures and effect annotations. No implementation specified; that's Phase 8. Rationale: users and book authors need names to write against; signatures are API contracts.
- **(c) Full spec — treat I/O as an MVP-critical surface.** All stdin, file, env, and process APIs are specified in design.md with exact signatures, including error types. Rationale: the book leans on these in every chapter.

Tradeoffs:

- (a) keeps design.md lean but leaves the book with nothing to document in its tutorial chapters. Users writing `read_line` today don't know whether the function exists, what module it lives in, or what it returns.
- (b) specifies just enough for the book and the compiler's effect-inference tests to work. Implementations stay in Phase 8. This is what the `print`/`println`/`eprintln` entries already do (name + effect, no body).
- (c) over-invests in stdlib detail at the spec level. Every future stdlib bikeshed ends up in design.md. Unnecessary.

**Lean:** (b). Match the existing `print`/`println` precedent — name, module path, signature, effect contract, one line of description. No implementation detail. Lives in a new "Standard I/O Surface" subsection of § Standard Data Structures.

## Q2. Scope — what's the minimum viable API surface?

If Q1(b) wins, what's in the initial set?

Mapping Rust book needs to Kāra equivalents:

| Rust usage | Book chapter | Kāra candidate |
|---|---|---|
| `io.stdin().read_line(&mut buf)` | ch2 guessing game | `io.read_line() -> Result[String, IoError]` |
| `io.stdin().read_to_string(&mut buf)` | pipe usage (`cat f \| tool`) | `io.read_to_string() -> Result[String, IoError]` |
| `std.env.args()` | ch12 CLI tool | `env.args() -> Vec[String]` |
| `std.env.var("KEY")` | ch12, throughout | `env.var(name: ref String) -> Result[String, VarError]` |
| `std.fs.read_to_string(path)` | ch12 | `fs.read_to_string(path: ref String) -> Result[String, IoError]` |
| `std.fs.write(path, content)` | ch12 | `fs.write(path: ref String, content: ref String) -> Result[(), IoError]` |
| `File.open(path)` + `BufReader` | ch20 | `fs.open(path: ref String) -> Result[File, IoError]` |
| `std.process.exit(code)` | throughout | **already covered** — `main` returns `ExitCode` |
| `Stdin.lock()` / buffered stdin | ch12 | Phase 8 detail |

Minimum for the book's getting-started + CLI tutorial chapters:

- `io.read_line`, `io.read_to_string` (stdin slurp for pipe usage)
- `env.args`, `env.var`
- `fs.read_to_string`, `fs.write`

Deferrable to Phase 8 detail:

- `File` type + `BufReader` + streaming reads
- `io.stdin()` as a handle type (vs. one-shot `read_line` / `read_to_string`)
- Network I/O (design.md mentions `Network` as an effect but no API)
- Process spawning

**Buffering stance.** Stdin and stdout are line-buffered by default; file I/O is block-buffered. Explicit `io.flush()` available for `Stdout` / `Stderr`. The buffered/unbuffered distinction as a type-level concept (Rust's `BufRead` trait) is deferred to Phase 8 — MVP users get default buffering and `flush()` only.

Options:

- **(a) Minimum set only.** Spec the seven functions above, defer everything else. Book tutorials work; advanced I/O waits for Phase 8.
- **(b) Minimum set + File handle.** Spec `File` as a type with `open`, `read`, `write`, `close`. Required for any example that streams instead of slurps.
- **(c) Minimum set + stdin as a handle.** Spec `io.stdin()` returning a `Stdin` handle with `read_line` / `lines()` / `read_to_string` methods. Matches Rust's shape exactly.

**Lean:** (a). The Rust-book-equivalent tutorials all work with slurp-style APIs (`read_to_string`, `read_line`). File handles and streaming can be Phase 8. Keep the spec surface minimal.

## Q3. Error types — `IoError` / `VarError` / unified?

Rust's `std.io.Error` is a single omnibus type with a kind enum. Kāra's error style (design.md § Error Handling) favors per-domain error types that compose via `?` and `From`.

Options:

- **(a) Single `io.Error`.** One type, kind field (`NotFound`, `PermissionDenied`, `InvalidUtf8`, `EnvNotPresent`, `EnvNotUnicode`, etc.). Rust-style. Simplest for users.
- **(b) Per-domain types.** `fs.Error`, `env.VarError`, `io.StdioError`, each with domain-specific variants. Composes via `?` if a function returns a sum type or if `From` impls are provided.
- **(c) Hybrid.** One `IoError` covers filesystem + stdin; `env.VarError` stays separate (it's not an I/O error, it's a lookup miss).

**Lean:** (c), with a flag. (c) matches idiomatic Kāra (small, focused error types) without fragmenting trivially-related FS and stdio errors. `env.VarError` has two variants (`NotPresent`, `NotUnicode`) which is reason to keep it separate.

**Reconsideration flag:** `VarError.NotUnicode` is semantically identical to `IoError.InvalidUtf8` — same underlying condition (invalid UTF-8 bytes), different source. The "env is not I/O" framing is principled but the overlap weakens (c). If we find ourselves writing `From[VarError] for IoError` to flatten the error story for callers that do both env reads and file reads, (a) with a unified kind enum may be the right call. Worth re-examining before resolution.

## Q4. UTF-8 and stdin — what does `read_line` return?

`io.read_line` reads bytes from stdin. Rust returns the number of bytes read and mutates a buffer; Kāra's style favors return values (`Result[String, IoError]`). But stdin bytes might not be valid UTF-8.

Options:

- **(a) `Result[String, IoError]`.** Invalid UTF-8 surfaces as an `IoError.InvalidUtf8` variant. Matches Kāra's UTF-8-only `String` invariant (design.md:913).
- **(b) `Result[Vec[u8], IoError]` + explicit conversion.** Return raw bytes, let the caller validate. More flexible; worse ergonomics for the book's tutorials.
- **(c) Two functions.** `io.read_line() -> Result[String, IoError]` (validates UTF-8) and `io.read_line_bytes() -> Result[Vec[u8], IoError]` (raw).

**Lean:** (a) for now, with (c) available later if needed. The tutorial use case is string input; bytes-level input is Phase 8.

## Q5. Does stdin's effect belong to `Stdio` or a finer-grained resource?

`reads(Stdio)` is listed in design.md:3129 as the resource for "stdin/stdout/stderr." That means `read_line` and `println` both hit the same resource — which implies the compiler would serialize a program that reads from stdin while writing to stdout.

Options:

- **(a) Keep `Stdio` as the single resource.** Simpler. Serialization of stdin-read with stdout-write is usually what users want anyway (interleaved output is confusing).
- **(b) Split into `Stdin`, `Stdout`, `Stderr` as three distinct resources.** Allows parallelism between input and output. Matches the finer-grained resource model Kāra uses elsewhere.
- **(c) Parameterize on direction.** `reads(Stdio[In])` / `writes(Stdio[Out])` / `writes(Stdio[Err])` — uses parameterized-resource machinery (design.md § Parameterized Resources).

**Lean:** (b). Stdin reads and stdout writes truly don't conflict. Three-way split costs nothing at the effect-checker level and enables cleaner auto-concurrency. Downside: slightly more noise in effect signatures, but common functions can use effect groups to collapse. Confirm.

**Sequential-order guarantee.** The split does *not* let auto-concurrency reorder `print("prompt: "); let line = read_line()?;` into a racing `par`-style execution. Kāra preserves source order within a sequential block regardless of resource independence (design.md § Auto-Concurrency). The `Stdin`/`Stdout` split enables parallelism only in explicit `par { }` scopes or across provably-independent call-graph regions — never across plain sequential statements. This is the same rule that already applies to `reads(DbA); reads(DbB);` with distinct resources.
