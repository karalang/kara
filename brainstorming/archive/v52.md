# Brainstorming deferred_v3 — RC flavor (Rc vs Arc) user control and visibility

**Status:** OPEN — opened 2026-04-23 during a strategic pressure-test of Kāra's pitch against `docs/design.md`. The question is whether users should have a way to influence which RC flavor (non-atomic `Rc` vs atomic `Arc`) the compiler selects, in addition to the already-present `karac explain` visibility.

## Context

### Current design

Per `docs/design.md:5258–5274` (Phase 2 algorithm):

1. Phase 1 inserts tentative `Rc` at fallback sites (non-atomic).
2. Phase 2 walks the program graph; if any live range of an `Rc`-wrapped value crosses a parallel region (spawn, par block, task boundary), the tentative `Rc` is promoted to `Arc`.
3. Conservative default: when the promotion decision is ambiguous, the compiler picks `Arc`. (`docs/design.md:5271`)

Per `docs/design.md:124`: `karac explain` surfaces the representation for a given binding — one of `owned (stack)`, `ref (borrow)`, `shared (Rc)`, or `shared (Arc)`. The programmer can inspect but cannot direct.

Per `docs/design.md:5175`: RC insertion emits a compiler note by default (`warn_above_0`). The note names the trigger (re-use-after-consume, closure capture, container store).

Escape hatches that already exist:
- `#[no_rc]` on a function or module: forbids RC insertion; the compiler errors instead of inserting RC. Used for GPU / embedded / hot-path code.
- `#[allow(rc_fallback)]`: suppresses the RC-insertion note.

### What's missing

There is no way to say:
- "Inside this function, I know the value is single-threaded — keep it as `Rc`, don't promote to `Arc`."
- "Inside this hot loop, I want `Arc` even though you can't prove it crosses a parallel region — I'm going to send this over a channel later and want to avoid a later promotion."
- "This type should always be `Rc`, never `Arc`" or the reverse.

The conservative `Arc` default (design.md:5271) is the one that most matters performance-wise — atomic refcount updates are real cost in tight loops. Users whose profile shows the atomic RC as the hot spot have no lever other than rewriting the code to avoid RC (e.g., via `#[no_rc]` plus explicit lifetime management).

### Where this interacts with the broader pitch

1. **"Don't think about memory discipline" principle.** Kāra's pitch is that the compiler handles RC; users don't reach for `Rc<RefCell<T>>`. Adding user-facing knobs for RC flavor reintroduces the thing we just removed. Needs justification.
2. **But: performance-critical code exists.** Game loops, audio callbacks, embedded ISRs. For these paths, a wrong-flavor `Arc` is a real performance bug. The `#[no_rc]` escape hatch forbids RC entirely; there's no middle ground "you can use RC, but only the cheap one."
3. **Kāra's automatic RC is a differentiator only if it works right.** If users routinely find `Arc` where they wanted `Rc` and have no lever, the pitch hollows out.
4. **Relationship to effects.** The Phase 2 algorithm's "crosses a parallel region" check is effect-adjacent. If the effect system ever extends (see `brainstorming_deferred_v1.md`) with a clearer notion of shared-state effects, the Phase 2 promotion decision might become more precise — possibly closing this gap without explicit user control.

## Prior-art reference

- **Swift:** always atomic RC (ARC). No user control. Accepted performance cost.
- **Rust:** explicit choice at the type — `Rc<T>` vs `Arc<T>`. Not automatic; the whole point is programmer picks.
- **Hylo:** no RC; move-by-default. Not applicable.
- **ObjC / ObjC++:** atomic RC everywhere. Same as Swift.
- **Python:** thread-safe RC via GIL, transparent to programmer.

Kāra's design already picked an unusual middle — *automatic* RC with *flavor inferred*. Most languages are "explicit + programmer picks flavor" or "automatic + single flavor." Kāra's "automatic + inferred flavor" creates a knob that didn't exist elsewhere, and therefore also the question of whether the knob should be turnable.

## Open questions

### Q1 — Should users be able to influence the flavor at all?

- **A: No. Trust the compiler; if the default is wrong, the fix is compiler improvement, not a knob.** The Phase 2 algorithm is conservative; when profiling shows a real performance problem, we refine the algorithm, not add user overrides.
  - **Pro:** preserves "don't think about memory discipline." No surface growth.
  - **Con:** leaves performance-critical code without a lever. Requires every corner-case to be baked into the algorithm.
- **B: Yes, but only as a *hint*, not an override.** `#[prefer_rc]` / `#[prefer_arc]` attributes tell the Phase 2 pass which way to lean in ambiguous cases. When the compiler has proof either way, the hint is ignored (with a note).
  - **Pro:** preserves correctness (no unsafe Rc where Arc is required). Gives a performance knob. Fails safe.
  - **Con:** one more attribute to know about. Easy to misuse ("I added `#[prefer_rc]` and nothing changed" when the compiler proved Arc was needed).
- **C: Yes, as an assertion. `#[rc_only]` / `#[arc_only]` force the flavor. If the compiler determines the forced flavor is unsafe (e.g., `rc_only` but value crosses a parallel region), it errors.**
  - **Pro:** explicit performance contract. The programmer says "I claim this is single-threaded" and the compiler verifies.
  - **Con:** same shape as `#[no_rc]` — a safety assertion that can fail. User must understand when the compiler would reject.
- **D: Yes, as full override. User can force a flavor and the compiler respects it, trusting the programmer.**
  - **Pro:** maximum control.
  - **Con:** reintroduces `unsafe`-flavored trust. Forcing `Rc` on a cross-thread value is a memory-safety bug. Hard no.

**Weak lean: B+C combination.** Rationale: `#[prefer_*]` as hint (B) handles the "compiler's conservative default is wrong for my code" case without any safety risk. `#[rc_only]`/`#[arc_only]` as safety-checked assertion (C) handles the "I want a performance guarantee enforced by the compiler" case. D is unsafe. A is insufficient — the design already has `#[no_rc]` as a "forbid RC entirely" escape hatch, and `#[prefer_rc]` is the same category of affordance one step milder.

### Q2 — Granularity: per-binding, per-function, per-type, per-module?

- **A: Per binding.** `let x: Shared[T] = ...` with a local attribute. Most precise.
- **B: Per function.** `#[prefer_rc] fn hot_loop() { ... }` — all RC decisions inside this fn lean Rc.
- **C: Per type.** `#[prefer_rc] shared struct Foo { ... }` — all `Foo` values lean Rc.
- **D: Per module or project.** A default leaning for an entire package. e.g., embedded projects might set `prefer_rc = true`.

**Weak lean: B + D.** Rationale: per-binding (A) is too granular for automatic RC — users don't name the RC promotion in source. Per-type (C) is too coarse and risks changing semantics depending on where the type is used. Per-function covers the hot-path use case; per-module covers the embedded / GPU profile use case where a whole codebase has a consistent "don't pay for atomics" or "do pay for atomics" preference.

### Q3 — Visibility: is the current `karac explain` surface sufficient?

Currently `karac explain` reports the representation (`Rc` vs `Arc`) for a binding (`docs/design.md:124`). This is pull — the user has to ask.

- **A: Current pull model is sufficient.** Users who care about RC flavor run `karac explain`; users who don't, don't.
- **B: Push at RC-insertion sites.** The RC-insertion note (already emitted per `docs/design.md:5175`) should name the flavor chosen: *"note: inserted `Rc[T]` at line 37 (not promoted; value does not cross a parallel region)"* or *"note: inserted `Arc[T]` at line 37 (promoted: value crosses `spawn` at line 42)"*.
- **C: Push at parallel-region boundaries.** The `spawn` or `par {}` site emits a note listing which bindings got promoted to `Arc` because of it.
- **D: Dedicated `karac explain --rc`** — a concurrency-and-representation–focused view.

**Weak lean: B.** Rationale: the note is already being emitted (`warn_above_0`); extending it to name the flavor is a one-line change to the diagnostic and gives users the data at the exact point they would have questioned it. C is also good but slightly harder to thread through the codegen layer. D is nice-to-have but duplicative of `karac explain`'s existing representation field.

### Q4 — Interaction with `#[no_rc]`

`#[no_rc]` today says "no RC at all in this function; error instead of inserting." If we add `#[prefer_rc]` / `#[prefer_arc]`, what is the interaction?

- **A: Conflicting attributes are a compile error.** `#[no_rc]` and `#[prefer_rc]` on the same function = error.
  - **Pro:** obvious semantics.
  - **Con:** very obvious.
- **B: `#[no_rc]` always wins.** Nothing to document; no-RC is strictly stronger than prefer-*.
  - **Pro:** simple.
  - **Con:** a conflicting pair on the same function is probably a bug worth flagging.

**Weak lean: A.** Rationale: conflicts are almost always user error and should surface loudly.

### Q5 — Scope of this change in the roadmap

- **A: v1.** Land `#[prefer_rc]` / `#[prefer_arc]` in the same release as the RC fallback itself. Small surface, complements `#[no_rc]`.
- **B: Post-v1.** Ship v1 without user knobs, see whether the default is actually the complaint it looks like in theory. Revisit after real use.

**Weak lean: B.** Rationale: Kāra's pitch for automatic RC is "trust the compiler." Adding knobs in v1 is a signal of low confidence in the compiler's decisions. Deferring until real-world profiling identifies the pain is the right order. The memory note "prefer fixing small issues over deferring" applies to design gaps that block work; this isn't blocking anything, and "add knob when needed" is cheaper than "remove knob because it wasn't needed."

This is a **weak lean toward defer**, but it is the one question in this doc where defer is the right answer — the design is complete enough for v1, and the knob is speculative.

## Decisions needed

1. **Q1** — add user-influence affordances (B+C) or trust the compiler entirely (A)?
2. **Q2** — granularity if Q1 ≠ A.
3. **Q3** — improve RC-insertion diagnostic to name flavor (independent of Q1–Q2).
4. **Q4** — conflict rule with `#[no_rc]`.
5. **Q5** — v1 or post-v1.

## Risks / follow-up

- **Speculative knobs are worse than no knobs.** If real users never hit the problem, the knob is noise. Defer (Q5=B) is the safe bet.
- **`Arc` in hot loops is a known pitfall.** Real-world Rust projects routinely profile their way to `Rc` for single-threaded hot paths. Kāra will hit the same profile. The question is whether we bake the lever in preemptively or reactively.
- **Phase 2 algorithm may improve in the wild.** If we improve the "crosses a parallel region" analysis to be less conservative, the need for user knobs shrinks. Investing compiler effort may make the user-facing knob unnecessary.
- **Relation to `brainstorming_deferred_v1.md`.** If the deferred effects-coverage work lands (richer effect-based notion of sharing, e.g. `writes(Shared[T])`), Phase 2's promotion decision could be driven by the declared/inferred effect row instead of heuristic cross-region analysis. That would make the knob either obsolete or much more narrowly scoped.
- **Interaction with profiles.** The embedded / kernel profiles forbid `allocates(Heap)` and RC is heap-allocated, so these profiles already never see RC — the knob is moot there. The GPU profile is similar. Where the knob matters is the default profile's hot paths.
