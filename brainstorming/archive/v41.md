## Context — what exists today

- Parser accepts `mod name;` and `use path.seg.Item;` as top-level items (`src/parser.rs:1025`, `src/ast.rs:311`).
- Resolver registers `mod` as a `SymbolKind::Module` and `use` as a `SymbolKind::Import`, but neither creates a nested scope nor navigates to anything (`src/resolver.rs:933-955`).
- The CLI compiles **one file** at a time (`src/main.rs`, `src/cli.rs:981` `cmd_run(filename: &str, ...)`).
- There is no `kara.toml` parser, no project-root discovery, no file-tree walker.
- Stdlib types (`Option`, `Result`, `Vec`, `F32`, `F64`, `Ordering`) are injected directly into the typechecker's symbol table as "builtins" rather than imported from any prelude module.
- `docs/design.md` § Module System (`:420`) is clear: "no `mod` declarations needed; directory structure defines the module tree." The parser-level `mod name;` is design-obsolete.

## Context — what the design already commits to

From `docs/design.md § Module System` and § Package System:

- Each `.kara` file under `src/` is its own module. Path = directory path from project root, dots for separators (`src/db/connection.kara` → `db.connection`).
- Entry points: `main.kara` for executables, `lib.kara` for libraries.
- Three-level visibility: `pub` / default / `private`. `private` = same directory only.
- `use path.Item;` imports. `pub use` re-exports.
- Fully-qualified resource paths (`myapp.db.UserDB` ≠ `otherlib.db.UserDB`).
- Circular module dependencies are a compile error.
- Prelude: `Option`, `Result`, `Vec`, `Slice`, `String`, `StringSlice`, `Map`, `Set`, primitives, `Some`/`None`/`Ok`/`Err`, `print`/`println`/`eprintln`, compiler builtins.
- Conditional compilation via file suffixes: `poller_linux.kara`, `poller_macos.kara`, `poller_windows.kara`.
- `tests/` and `examples/` are separate compilation roots.
- `kara.toml` manifest (spec in § Package System).

## Group M — Module system keywords

### M0. `import path.Item;` vs `use path.Item;`  *(✓ RESOLVED — 2026-04-20 → M0a with brace-grouping for multi-item)*

**Resolution.** Adopt `import path.Item;` (M0a) with brace-grouping for multi-item imports folded in from day one (not deferred). Single-item takes no braces; multi-item uses `path.{A, B, C}`. Each item may carry an `as Alias` rename. `pub import` for re-exports.

**Canonical forms (landing in `docs/design.md § Module System`):**

```kara
import db.connection.Connection;                                 // single item
import db.connection.Connection as Conn;                         // single item + rename
import db.connection.{Connection, Pool};                          // multi-item
import db.connection.{Connection as Conn, Pool as P};            // multi-item + rename on each
pub import db.connection.Connection;                             // re-export (single)
pub import db.connection.{Connection, Pool};                     // re-export (multi)
import db.connection;                                             // binds `connection` as the module
```

**Rule for what gets bound.** The last segment of each imported path is introduced into the current scope. `import a.b.c;` binds `c`. `import a.b.{c, d};` binds `c` and `d`. Aliases override: `import a.b.c as C;` binds `C`. Rule is uniform across items and modules — no Python-style `import x.y` footgun.

**Wildcard imports (`import path.*;`) and nested grouping (`import a.{b.{c, d}, e};`):** not in v1. Decide later if/when concrete pain motivates them (see M3 below).

**Deferred sub-decisions (all addressed in follow-up items):** what happens to the reserved `use` keyword (see sub-decision below); relative paths `self.` / `super.` (see I1); grammar for line-wrapping long brace-lists (formatter's call — natural container is `{...}`).

**Sub-decision — reserved `use` keyword.** The lexer currently has `Token::Use`. With `import` adopted: **release `use`.** No Kāra feature on the roadmap needs it. If a future feature wants `use`, reserve it then. The lexer drops the keyword; `use` becomes a plain identifier.

**Implementation touch points:**
- Lexer: `Token::Use` → `Token::Import`; `use` becomes an ordinary identifier.
- Parser: `parse_use_decl` → `parse_import_decl`. Brace-grouped form is parsed as a new AST shape.
- AST: `UseDecl` → `ImportDecl { span, is_pub, path: Vec<String>, items: Vec<ImportItem> }` where `ImportItem { name: String, alias: Option<String> }`. Single-item case: `items` has one entry whose `name` is the last path segment. Multi-item case: `items` is the brace-list.
- Resolver: `collect_use` → `collect_import` — creates one symbol per `ImportItem`, keyed by `alias.unwrap_or(name)`.
- Formatter: arm that emits brace-grouped form with one item per line on wrap.
- Book chapter 13: swap prose examples from `use` to `import`.
- design.md § Module System: lock the forms above into the canonical doc.

**Rationale (consolidated from the resolution turn).**

Three shapes were compared: M0a (`import path.{items};` — Rust/Scala), M0c (`from path import items;` — Python), M0e (`import items from path;` — TypeScript/ESM). M0c rejected for awkward `pub from path import items` reading order and column-0 `from` hurting scannability. M0e rejected on three grounds: (1) single-item cases carry a brace-wrapper visual overhead; (2) path column floats to right-of-line, hurting scanning across an imports block; (3) no natural container for line-wrapping long imports. M0a wins on uniform last-segment binding rule (items and modules treated identically — no Python footgun), clean `pub import` composition, and clear formatter story for long wraps.

The "syntax-mainstream" design principle tipped away from `use` (Rust/C++ outlier) toward `import` (used by Python, Java, JS, TS, Go, Swift, Kotlin, Dart, Scala, Elm). Frequency-of-multi-item-imports was re-examined: multi-item is the dominant pattern in Rust, TypeScript, and Python alike — Java's pain (no grouping, `*`-vs-lint wars) confirms the ergonomic cost of missing the option. Grouping is therefore in v1, not deferred.

### M0-archive. (keyword choice rationale preserved above; skip to M1)  *(✓ RESOLVED)*

<summary>Full option analysis from the resolution discussion (reference only)</summary>

**Why it matters.** The keyword chosen for the import statement is the single most-seen piece of module-system syntax in the language — every `.kara` file has a cluster of them at the top. Getting it right matters for the "semantics-Rust, syntax-mainstream" tiebreaker (`project_design_principle.md`): Rust's `use` is an outlier. Python, Java, JavaScript, TypeScript, Go, Swift, Kotlin, Dart, Scala, and Elm all spell this `import`. C# uses `using`. Only Rust and C++ spell it `use`.

**Observation.** Kāra already spells generics `[T]` (mainstream) rather than `<T>` (Rust), method calls and field access uniformly with `.` (mainstream) rather than `::`/`.` split (Rust), and property access `obj.field` without paren-style getters. The import keyword is the same call: Rust's lexical choice versus the broader mainstream. The design principle explicitly makes this a tiebreaker.

**Options:**

- **M0a. `import path.Item;`** — mainstream. Also ergonomically consistent with the module model (design.md already describes the action as "imports," never "uses"): `docs/design.md:446` "End users import from the package root," `docs/design.md:469` "automatically imported into every source file."
  ```kara
  import db.connection.Connection;
  pub import auth.token.Token;
  ```
- **M0b. `use path.Item;`** — status quo; matches current parser + `Token::Use`.
- **M0c. `from path import Item;`** — Python-shaped, two-keyword. Less common outside Python. Allows natural multi-item grouping (`from db.connection import Connection, Pool`) without committing to brace-list syntax.
- **M0d. `using path.Item;`** — C# style. Reads as a preposition; awkward on its own line.

**Lean: M0a.** The design principle is explicit that syntax should default to mainstream when semantics are preserved — and here semantics are identical across all four options. `import` carries the right connotation (the statement brings a name into the current scope), is the most-common spelling across production languages, and removes a small Rust-ism that pays no semantic tax. M0c's two-keyword form is seductive for grouping but awkward for single-item (`from db.connection import Connection` is verbose for the common case); grouping can be added later via brace-list if needed (see M3, née M2). M0d reads poorly. M0b loses the tiebreaker.

**Implementation note.** Swap affects lexer (`Token::Use` → `Token::Import`, add `Token::Use` to reserved-keyword list if we want to future-proof), parser (`parse_use_decl` → `parse_import_decl`), AST (`UseDecl` → `ImportDecl`), resolver, formatter, book chapter 13, any design.md references. The `pub` prefix pattern carries through unchanged: `pub import db.connection.Connection;` (re-export).

**Sub-decision — reserve `use`.** If we adopt `import`, do we keep `use` as a reserved keyword so future uses (no pun) don't collide, or release it? **Lean: release it.** No Kāra feature on the roadmap needs `use`; reserving pre-emptively just costs users a plausible identifier. If a future feature wants `use`, reserve it then.

**Interaction with other items.**
- M2/M3 (grouping / glob) — orthogonal; works the same for `import`.
- I1 (absolute vs relative path resolution) — orthogonal.
- I2 (`pub use` re-exports) — becomes `pub import`.
- I3 (prelude auto-injection) — compiler-injected `import std.prelude.*;` at module top (or never written at all; no user-visible difference).

### M1. What happens to the parser-level `mod name;` construct?  *(✓ RESOLVED — 2026-04-20 → M1b)*

**Resolution.** Keep `mod` reserved as a keyword; reject `mod name;` at parse time with a directive-style diagnostic pointing the user at the directory-tree rule.

**Diagnostic wording.** `` `mod` declarations are not used in Kāra — module structure is derived from the directory tree. Each `.kara` file is its own module; put this file in the appropriate directory to define its module path. See `docs/design.md § Module System`. ``

**Implementation touch points:**
- Lexer: `Token::Mod` stays.
- Parser: `parse_mod_decl` is replaced by a small error-emitting stub at the `Token::Mod` branch in `parse_item` (`src/parser.rs:130`). The `Token::Semicolon` is consumed for recovery so parsing continues.
- AST: `Item::ModDecl(ModDecl)` and `struct ModDecl` removed. Formatter arm removed. Resolver's `collect_mod` removed; `SymbolKind::Module` also removed (nothing else uses it).
- No tests or examples exercise `mod name;` today (grep-confirmed), so no fixture migration.

**Why not M1a (full removal):** leaving the token reserved costs ~5 LoC and preserves the escape hatch if a future feature ever wants an inline module construct (conditional groups, sub-file modules). M1c (silent accept + no-op) was dismissed as sloppy.

### M2. Brace-grouping for multi-item imports  *(✓ RESOLVED — 2026-04-20 → folded into M0a)*

**Resolution.** Brace-grouping (`import path.{a, b, c};`) is in v1, folded into M0. Each item may carry an `as Alias` rename. See M0 for the canonical forms.

### M3. Wildcard (`import path.*;`) and nested grouping (`import a.{b.{c}};`)  *(🟡 OPEN — lean defer)*

**Why it matters.** Beyond the basic brace-list, Rust's `use` surface supports deeper patterns: wildcard imports (`use foo::*`), nested groups (`use a::{b::c, d}`), `self` in groups (`use a::{self, b}`), and glob-in-group (`use a::{b::*, c}`). Each is ergonomic sugar for common cases but each is also a grammar / review / IDE cost.

**Options:**

- **M3a. All of them in v1.** Match Rust's full expressiveness.
- **M3b. None of them in v1.** Ship the basic brace-list (M0) and nothing more. Add deeper patterns when concrete pain motivates each.
- **M3c. Wildcard only.** Gloss over nested grouping but allow `import db.connection.*;` for the common "I want everything this module exports" case.

**Lean: M3b.** Wildcards are audit-hostile (a reader can't tell from the import line what names come into scope), and nested grouping is a readability win *only* when the programmer would otherwise write many parallel imports — which hasn't yet been measured for Kāra since nobody's written a large Kāra codebase. M0's basic brace-list covers the common multi-item case. Open a follow-up brainstorm when real codebases hit the limit.

## Group P — Project structure & CLI

### P1. Do we require `kara.toml` to compile?  *(✓ RESOLVED — 2026-04-20 → P1a)*

**Resolution.** `kara.toml` is required for multi-file / project-mode builds. Single-file mode (`karac run file.kara`) remains the escape hatch for toy programs, book chapters, and inline examples.

**Project-root discovery rule.** `karac build` (and any other project-mode subcommand) walks up from CWD looking for `kara.toml`. First match wins. If no `kara.toml` is found before the filesystem root, emit: `` "not inside a Kāra project; add a `kara.toml` or run a single file with `karac run file.kara`" `` with exit code 1.

**Minimum `kara.toml` parse surface for CR-24.**

```toml
[package]
name = "myproject"        # required — used as the root module name prefix
edition = "2026"          # optional — validated if present, otherwise defaulted
```

Every other key is **ignored (not rejected)** during CR-24 — so a user's `[dependencies]`, `[workspace]`, `[build]`, etc. keys don't break the build but also don't do anything. Unknown keys at `[package]` emit a soft warning; invalid TOML is a hard error.

**Implementation touch points:**
- Add `toml` crate dep (Cargo.toml) — use a small, pure-Rust TOML parser.
- New `src/manifest.rs` module with `struct Manifest { name: String, edition: String }` and a `load_from_root(path) -> Result<Manifest>` entry point.
- New `karac build` CLI subcommand; existing `karac run file.kara` semantics unchanged.
- CLI's project-mode path calls `discover_project_root()` → `Manifest::load_from_root()` → proceeds to file walker (P2 / P3 territory).

**Rationale.** The manifest is the canonical project-identity signal per `docs/design.md § Package System`. Making it required for multi-file work aligns with Cargo, gives a stable place to stash edition / target / name. P1b (implicit via `src/main.kara` existing) allows two project-identity signals that can disagree, which is a recipe for confusion. P1c (always required) would break `karac run single_file.kara` and the book's early chapters. P1a threads the needle.

### P2. Module name for `main.kara` / `lib.kara` / nested `mod.kara`?  *(✓ RESOLVED — 2026-04-20 → P2b, root-hoist entry files, no `mod.kara`)*

**Resolution.** Each module path has exactly one canonical file. `mod.kara` is not recognized. Entry files (`main.kara`, `lib.kara`) hoist their items to the crate root.

**Module-path rules (landing in `docs/design.md § Module System`):**

| File path | Module path | Notes |
|---|---|---|
| `src/main.kara` | *(crate root)* | items appear at the root, callable without prefix |
| `src/lib.kara` | *(crate root)* | items appear at the root, callable without prefix |
| `src/greet.kara` | `greet` | |
| `src/db.kara` | `db` | items at `db.Item` |
| `src/db/connection.kara` | `db.connection` | items at `db.connection.Item` |
| `src/db/mod.kara` | — | **parse-time error:** `` "`mod.kara` is not a recognized module file; to hoist items into the `db` namespace, put them in `src/db.kara`" `` |

**Entry-file exclusivity.** `main.kara` and `lib.kara` cannot coexist in the same package. If both are present, emit: `` "a Kāra package cannot contain both `src/main.kara` and `src/lib.kara`; a package is either an executable or a library" `` (matching Cargo's behaviour). Neither is required if the package only exposes nested modules under `src/`, though the linker will fail if you try to `karac build` an executable without `main.kara`.

**Sub-module visibility rule.** A file like `src/db.kara` and its sibling directory `src/db/` are the same namespace. `src/db.kara` declares items that live at `db.*`; `src/db/connection.kara` declares items that live at `db.connection.*`. Re-exports in `src/db.kara` (`pub import db.connection.Connection;`) are the mechanism for hoisting items from nested files into the parent namespace — replacing everything `mod.kara` would have offered.

**Implementation touch points:**
- File walker (P3) maps `PathBuf` → `ModulePath` using the rules above.
- Entry-file detection happens once at walker construction.
- `src/db/mod.kara` check: walker rejects the filename specifically, with the diagnostic above.
- Both-entry-files check: walker rejects up-front.

**Rationale.** One-way-to-do-it for `db.kara` vs `db/mod.kara` avoids a Rust-era footgun — newcomers repeatedly ask "which do I use?" for no real benefit. Root-hoisting entry files matches every crate-concept language and makes `fn start()` in `main.kara` callable as `start`, not `main.start`, which is the overwhelmingly common need.

### P3. Do `tests/` and `examples/` compile as separate roots?  *(✓ RESOLVED — 2026-04-20 → no separate `tests/` root; colocation per v1 resolution)*

**Resolution (2026-04-20).** Kāra does not have a separate `tests/` compilation root. Unit tests are **colocated** as `_test.kara` files in the same directory as the code they test, per the v1 resolution (`v1.md:2469`). The `tests/db_test.kara` example in `design.md:513-519` is a stale leftover contradicting the v1 decision and needs cleanup.

**Test discovery and build rules:**
- `karac build` — ignores any `_test.kara` file under `src/` (excluded from the module walk).
- `karac test` — includes `_test.kara` files in the walk; they are treated as ordinary modules in the *same namespace* as their sibling non-`_test` files, with access to private items of that module (because "same directory = same module" per `design.md:2426`).
- The `_test.kara` prelude (additive imports like `Arbitrary`, `assert`, `assert_eq`) is injected at parse time *only* under `karac test`.

**`examples/`.** Still in scope per `design.md:515`, but deferred from CR-24. Each `examples/*.kara` is its own runnable program with `fn main()`. Will need its own follow-up CR for discovery / `karac run --example NAME`. Not a compilation-root concern in the multi-file sense — each example is a self-contained single-file program.

**CR-24 scope implication.** Only `src/` is walked. No cross-root dependency threading. Test infrastructure (the `_test.kara` additive prelude, `#[test]` attribute resolution, test runner output format, `karac test` subcommand) is its own follow-up CR — not required to unblock CR-18 or the core multi-file story. Existing Rust integration tests under `tests/*.rs` continue to exercise the compiler via its API, unchanged.

**design.md cleanup checklist** (done at the "land resolutions" step):
- `design.md:513-519` — remove the `tests/` entry from the project-layout example; update the "separate compilation roots" sentence to mention only `examples/`.
- Verify no other section references `tests/*.kara` as a distinct compilation root.

**Rationale.** The v1 resolution chose colocation on well-thought-out grounds (Go/Rust/Swift/Kotlin all colocate; separate-dir testing "breaks down at scale"). The stale `tests/` example in design.md predates that resolution. Removing it both simplifies CR-24 and fixes a real design inconsistency.

## Group C — Compilation model

### C1. AST shape for a multi-module project  *(✓ RESOLVED — 2026-04-20 → C1a)*

**Resolution.** Adopt `ProgramTree { modules: Vec<Module>, root_module: ModuleId }` with first-class `Module` structs. Each phase visits modules in dependency order via a shared `ModuleGraph`.

**Canonical shapes (landing in `src/ast.rs` + new `src/module.rs`):**

```rust
pub struct ProgramTree {
    pub modules: Vec<Module>,
    pub root: ModuleId,
    pub graph: ModuleGraph,
}

pub struct Module {
    pub id: ModuleId,
    pub path: ModulePath,       // Vec<String> — e.g. ["db", "connection"]
    pub file: PathBuf,          // original source path for diagnostics
    pub items: Vec<Item>,       // top-level items parsed from this file
    pub imports: Vec<ImportDecl>,
    pub is_test_file: bool,     // _test.kara flag
}

pub type ModuleId = usize;
pub type ModulePath = Vec<String>;   // interning deferred; Vec is fine for v1

pub struct ModuleGraph {
    edges: Vec<(ModuleId, ModuleId)>,    // importer → importee
    by_path: HashMap<ModulePath, ModuleId>,
}
```

**Pipeline shape.** Each phase (resolver, typechecker, effectchecker, ownership, interpreter) takes `&ProgramTree` and visits modules in topological order per the graph. Cross-module symbol lookup routes through `ModuleGraph::lookup(path, from: ModuleId)` which also performs visibility checking (see C2).

**Sub-decision — `ModulePath` representation.** Plain `Vec<String>` for v1. Interning (a `ModulePathId` newtype backed by a global interner) is a straightforward later optimization if profiling shows lookup or clone hotspots.

**Why C1a over the alternatives:**
- **C1b (flatten + prefix-qualify)** hides module structure from diagnostics. "this error is in `db.connection.foo`" loses the `src/db/connection.kara:42:9` path unless we attach file metadata to every symbol, which is uglier than just keeping modules first-class.
- **C1c (shared global table threaded through each phase)** couples ordering tightly and makes it easy to introduce order-dependent bugs as new passes are added. C1a's structural graph gives circular-dependency detection for free (Tarjan's SCC on edges) rather than as a runtime scan.
- **Forward compat.** A proper module graph gives `karac explain`, future LSP work, and incremental compilation a clean structure to query. C1b and C1c would each need retrofitting to support those.

**Implementation note — existing phases.** Each phase's current `&Program` entry point becomes `&Module` (for per-module work) with a `&ProgramTree` or `&ModuleGraph` passed alongside for cross-module lookups. The single-file `karac run file.kara` escape hatch wraps the file in a trivial one-module `ProgramTree`.

### C2. When does cross-module visibility fire?  *(✓ RESOLVED — 2026-04-20 → C2a)*

**Resolution.** Resolver enforces three-level visibility (`pub` / default / `private`) at every cross-module name lookup. Single-pass integration with name resolution — no separate visibility phase.

**Visibility check algorithm.** Given a name lookup for item `X` defined in module `D` from requesting module `R`:

| `X`'s visibility | Access allowed when |
|---|---|
| `pub` | always |
| *(default)* | `R` is any module in the same package as `D` (always true under single-package compilation; becomes meaningful when external packages ship) |
| `private` | `R.path` and `D.path` share the same directory path (parent prefix equal, modulo entry-file hoisting) |

**Same-directory rule for `private`.** Two modules are in the same directory when their `ModulePath`s share a common parent — i.e., `path[..path.len()-1]` is equal for both. Entry files (`main.kara` / `lib.kara` with root-hoisted items) count as residing in `src/` (root directory) for this check. A `_test.kara` file shares its directory with the module it tests (colocation per P3).

**Diagnostic codes:**
- **`E0221 PrivateTypeInPublicSignature`** — reused from CR-18 for signature leaks. The deferred field-access half of CR-18 *re-activates* once C2a lands: a `pub fn` that returns a `struct` whose non-`pub` field is accessed from outside the defining module is flagged the same way. Explicitly noted in the CR-18 reopening criterion.
- **`E0222 PrivateItemAccess`** — new code, emitted when a direct name lookup resolves to an item the caller cannot see. Message form: `` "cannot access `<name>` from module `<R.path>`: item is `<visibility>` to module `<D.path>`" `` with a suggestion pointing at `pub` (if the user owns `D`) or at a path-level re-export (if they don't).

**Why not C2b or C2c.** C2b (separate pass) duplicates the name-walk the resolver already does — every resolved reference gets re-walked and re-checked, with no upside. C2c (split between resolver and typechecker) puts the phase boundary at the worst place: `private` visibility depends on paths that the resolver already has in hand, and `pub` checks depend on the same walker. Splitting creates two callers of the visibility logic that have to stay in sync.

**CR-18 interaction.** CR-18's signature-side pass (`check_signature_visibility` in `src/typechecker.rs`) already flags private-types-in-public-signatures *intra-module*. Under C2a, the same check fires cross-module: a `pub fn` in module `A` whose return type references a non-`pub` type in module `B` is flagged. CR-18's deferred field-access half becomes a direct beneficiary of C2a — the checklist entry can move from `[~]` to `[x]` in the same CR-24 commit that lands C2a.

### C3. Effect inference across modules  *(✓ RESOLVED — 2026-04-20 → C3a, per-module fixed-point + cross-module table lookup)*

**Resolution.** Per-module effect inference. Each module's fixed-point runs over every function in the module regardless of visibility. Cross-module callers read the callee module's completed effect table. Modules processed in topological order per the `ModuleGraph`.

**Per-function effect treatment:**

| Function's visibility | Effect treatment |
|---|---|
| `pub fn` | Effects **declared** and verified against inferred (today's behavior, extended to multi-module) |
| *(default)* | Effects inferred. Cross-module (same-package) callers read the inferred set. |
| `private` | Effects inferred. Same-directory callers read the inferred set. |

**No `pub fn` requirement.** A module can have zero `pub fn`s and be entirely default-or-private — that's fine. Effect inference just produces an effect table consumed by same-package callers. The "public declarations are a contract" rule only applies when `pub fn` exists, because `pub` is the external-package boundary.

**Module processing order.** Topological sort over the `ModuleGraph`; callees processed before callers so each function's effect lookup hits a completed table. Circular module dependencies are a compile error (per D1 below), so the DAG is always topo-sortable.

**Relationship to auto-concurrency (Phase 5, not yet implemented).** The module system has **no direct effect on parallelization decisions**. Effect inference builds the effect table; auto-concurrency's conflict analysis uses resource-level conflicts (two calls to `writes(FileA)` conflict regardless of module origin; two calls to `writes(FileA)` and `writes(FileB)` can parallelize regardless of module origin). C3a only governs *how the effect table gets built* in the multi-module case; it doesn't move concurrency levers.

**Compatibility with Phase 3 SCC work.** The Phase 3 SCC-based effect inference pass (checklist line 25-27) becomes per-module under C3a. Within each module, SCC decomposition runs normally; cross-module calls resolve through completed tables and don't participate in the per-module SCC. `with E` polymorphism remains unaffected — public functions with `with E` are declared at the public surface and specialized at each caller, which works per-module. **Confirmed not blocked.**

**Why not C3b.** Whole-project fixed-point would infer effects for `pub fn` that then contradict the authored declaration (the declaration is the verified-against result). Strictly more work with no extra signal. C3a's per-module scoping is forward-compatible with incremental compilation, C3b is not.

## Group I — Imports, re-exports, and prelude

### I1. How does `import path.Item` resolve?  *(✓ RESOLVED — 2026-04-20 → I1a, absolute-only)*

**Resolution.** All import paths are absolute from the crate root. `import db.connection.Connection;` always starts at `src/db/connection.kara` regardless of which file the import appears in. No `self.` / `super.` / relative forms in v1.

**Resolution algorithm.**
1. Split the path by `.` into segments.
2. Start at the crate-root `ModuleId`.
3. For each segment except the last, look up the child module by name in the current `Module`'s sub-modules. Miss → `E0224 UnknownModule`.
4. For the last segment: look up an item by that name in the final module's top-level items. Miss → `E0225 UnknownItemInModule`.
5. If the item is found: check visibility per C2a's rules. Violation → `E0222 PrivateItemAccess`.
6. Bind the alias (`as X`) or last-segment name in the importing module's scope.

**Path separator — dots.** Confirmed: `db.connection.Connection` uses dots, consistent with expression-level paths (method / field access uniform syntax per design.md).

**Sibling-in-same-directory.** Uses absolute paths too. `src/db/connection.kara` writing `import db.pool.Pool;` to reach its sibling is the canonical form — no shorter form is provided. The verbosity is bounded at shallow depths, which Kāra's architecture should encourage.

**Non-breaking extension path.** If real codebases emerge with deeply nested hierarchies where absolute-path verbosity is painful, `self.` / `super.` can be added later as a non-breaking extension — absolute-only code keeps working unchanged. The reverse (ship relative-as-default, walk it back) is a breaking change, so I1a preserves optionality.

**Why not I1b / I1c.** Relative imports are a proven Rust/TypeScript friction point: (1) they break grep-based refactoring (TypeScript's `./foo` means different things in different files; absolute paths are one-to-one); (2) they make file moves fragile (relative `./sibling` silently points to the wrong thing after a reorg); (3) they introduce a style choice (when to use relative vs. absolute) that has no community consensus a decade into Rust. Kāra's directory model (one `src/` root, directory = namespace, short paths) doesn't pay for relative imports with enough ergonomic benefit to justify the cost.

### I2. How does `pub import path.Item` re-export?  *(✓ RESOLVED — 2026-04-20 → I2a + I2b, transparent re-export with optional rename)*

**Resolution.** `pub import db.connection.Connection;` (and `pub import ... as Conn;`) creates a transparent re-export. External consumers writing `import mylib.Connection;` land on *the same item* as `db.connection.Connection` — one canonical definition, two reachable names. Internal code keeps using `db.connection.Connection`.

**Identity semantics.** Re-exported items have exactly one canonical definition. `mylib.Connection` and `mylib.db.connection.Connection` are the **same type**, not two distinct types that share a shape. This matters for:
- Method resolution (methods defined on `Connection` are reachable via either name).
- Trait coherence (an `impl Foo for Connection` is one impl, findable via either name).
- Diagnostics (error messages pick the *shortest reachable* path by default — usually the re-exported name — but `karac explain` can show the canonical path).

**Visibility rule.** `pub import` requires the original item to be at least project-visible (default or `pub`) from the re-exporter's module. `pub import`ing a `private` item from outside its directory is `E0222 PrivateItemAccess`. Re-exporting always grants `pub`, overriding the original's external-boundary visibility — that's the whole point of the mechanism.

**Rename form.** `pub import db.connection.Connection as Conn;` is supported. Each `as` rename in a multi-item brace list works independently: `pub import db.connection.{Connection as Conn, Pool};` re-exports both, one renamed, one not.

**Cycle check.** Re-exports create edges in the `ModuleGraph`. If `a.kara` has `pub import b.X;` and `b.kara` has `pub import a.Y;`, that's a module-level cycle — `E0223 CircularModuleDependency` fires at graph construction (per D1 below). A pure type-level cycle (`a.X` refers to a type defined by re-export chain back to `a`) doesn't create a new cycle class; the module-level cycle check covers it.

**Glob re-exports (`pub import db.connection.*;`)** — deferred with M3 (wildcard imports). Revisit with M3's future work.

**Implementation note — symbol table.** Each module's symbol table stores `ImportedName { local_name: String, canonical_id: ItemId, is_pub: bool }`. Cross-module lookup resolves through `canonical_id` to the single canonical item. No renaming or duplication at the item level — only at the name-binding level.

### I3. Where does the prelude live?  *(✓ RESOLVED — 2026-04-21 → I3c, design committed; implementation phased)*

**Resolution.** Long-term design is I3c: stdlib types and traits live as real Kāra source; compiler builtins stay magic but expose stdlib-source declarations via a `#[compiler_builtin]` attribute. Implementation is phased across two CRs — CR-24 lands only the infrastructure, a dedicated follow-up CR materializes the stdlib source.

**Long-term prelude architecture (landing in `docs/design.md § Module System › Prelude`):**

Four categories, handled differently:

| Category | Examples | Treatment |
|---|---|---|
| **Primitive types** | `i8`..`i64`, `u8`..`u64`, `f32`, `f64`, `bool`, `char`, `!` | Lexer-level keywords. No import needed. No change. |
| **Stdlib types and traits** | `Option`, `Result`, `Vec`, `Slice`, `Map`, `Set`, `String`, `StringSlice`, `Display`, `Debug`, `Eq`, `PartialEq`, `Ord`, `PartialOrd`, `Hash`, `Add`, … | Real Kāra source in `runtime/stdlib/*.kara`. Baked into the `karac` binary at build time. Auto-imported via compiler-injected `import std.prelude.*;` at the top of every user module. |
| **Stdlib functions** | `print`, `println`, `eprintln` | Real Kāra source. Declare effects (`writes(Stdout)` etc.). Call down to runtime intrinsics for actual I/O. |
| **Compiler builtins** | `todo()`, `unreachable()`, `dbg(…)`, `assert(…)`, `assert_eq(…)`, `collect_all(…)` | Implementation stays compiler-side. Signatures + docstrings declared in stdlib source under `#[compiler_builtin]` so `karac doc` and IDE features see them. |

**Benefits of I3c as the long-term design:**

1. **Source-driven documentation.** Every stdlib method's docstring lives next to the definition; `karac doc` consumes it like any user type.
2. **IDE parity.** Go-to-definition on `Option` takes you to a readable Kāra file. LSP can index stdlib the same way it indexes user code.
3. **User reasoning parity.** Users can read the stdlib to understand what it does — enormously valuable for debugging, learning, and contributing.
4. **Maintenance.** Adding `Option.map_or_else` is a PR to `stdlib/option.kara`, not a typechecker change.
5. **Dogfooding.** If authoring `Option` in Kāra is painful, the compiler team feels it first — warts get found early.
6. **Effect integration is free.** `println` carries `writes(Stdout)` on its own signature; no special-case in the effect checker.
7. **Generics uniform.** `Vec[T]` is an ordinary generic struct — no typechecker carve-out for "the builtin Vec."

**Sub-decision — stdlib source location.** `runtime/stdlib/*.kara` under the `karac` repo. Baked into the compiler binary via a build-time `include_str!`-style mechanism so the compiler is self-contained. Users don't need to install stdlib separately; `karac` ships complete.

**Sub-decision — `#[compiler_builtin]` attribute shape.** `#[compiler_builtin]` marks a stdlib declaration whose implementation is provided by the compiler. The body (if any) is ignored or used only for `karac doc` examples. Attribute parsing falls out of existing attribute work. Intrinsic dispatch is a new mechanism — the typechecker maps `#[compiler_builtin] fn todo() -> !` to the internal `todo` lowering.

**Sub-decision — user override / shadowing.** Users can shadow prelude names by defining their own (`struct Option { ... }` in user code). The compiler allows this (it's just name resolution) but may emit a lint warning. No hard prohibition — the escape hatch has legitimate uses (wrapping a prelude type with extra semantics) and the cost of forbidding it is higher than the cost of the occasional misuse.

**Implementation phasing:**

- **CR-24 (this brainstorm's core deliverable):** lands the module graph, file walker, `import` resolution, cross-module visibility, re-exports, and the prelude auto-injection *mechanism*. The `register_builtin_types` shim **stays** — CR-24 does not write stdlib source. At end of CR-24, `std.prelude` is a placeholder that the shim populates.
- **Follow-up CR — "Materialize stdlib prelude as real Kāra source":** write `runtime/stdlib/{option,result,vec,map,set,slice,string,…}.kara`, implement the `#[compiler_builtin]` attribute + intrinsic dispatch, bake stdlib into the binary, retire `register_builtin_types`.
- **CR-24 and the stdlib CR are orthogonal in ordering.** The multi-file user-code infrastructure works without real stdlib source (shim still available). Once the stdlib CR lands, it plugs into the already-built module graph as an implicit root dependency.

**Why not I3a or I3b for the long term:**
- **I3a** collapses all prelude categories into "real module items," but compiler builtins genuinely can't be expressed in pure Kāra (`todo` has return type `!`, `dbg` captures source location, `assert` may be release-elided). Forcing everything into the stdlib-source mould fails for this category.
- **I3b** forces the compiler team to maintain a parallel universe of stdlib definitions in Rust — hand-written, hand-documented, hand-updated. This doesn't scale and has been walked back by every language that tried it (early Rust pre-1.0). Users lose reasoning parity and IDE support for the most-used types in the language.

**I3c threads the needle:** ordinary stdlib items live as ordinary Kāra code; genuine compiler magic stays magic but is documented alongside.

## Group F — Platform-conditional compilation

### F1. How does target detection work for file suffixes?  *(✓ RESOLVED — 2026-04-21 → F1a with permissive-walker forward-compat reservation)*

**Resolution.** Four OS-level suffixes for v1: `_linux`, `_macos`, `_windows`, `_wasm`. Target selected by `kara.toml [build].target` → `--target` CLI flag → host OS detected at `karac` build time (falls through in that order). The walker is **permissive**: only exact matches of the four known suffix tokens trigger platform filtering; any other trailing-underscore pattern is treated as part of the ordinary module name.

**Resolution algorithm (walker).**
1. For each `.kara` file under `src/`, split the stem at the last underscore: `stem = base + "_" + tail`.
2. If `tail` is one of `linux` / `macos` / `windows` / `wasm`:
   - The file is a platform file for that OS. Module name is `base`. Include in the build iff `tail` matches the current target OS; otherwise skip.
3. Otherwise, the file is an ordinary module file. Module name is the full stem (e.g., `poller_linux_x86_64.kara` → module `poller_linux_x86_64`, compiled unconditionally).

**Sub-decision — collision rule.** When both `poller_linux.kara` and `poller.kara` exist and target is Linux: **only the platform-suffixed file compiles** on Linux; `poller.kara` is skipped. On non-Linux platforms, `poller.kara` is the implicit fallback and compiles. Both files share the same module path (`poller`). Same rule applies to every platform.

**Sub-decision — symbol-level conflict.** If a shared fallback and a platform file *both* define the same top-level symbol (unlikely under the "only one compiles per target" rule, but possible in mixed file layouts), emit `E0226 ConflictingPlatformModule` — "platform file `poller_linux.kara` overrides shared `poller.kara` on Linux, but both declare `poll`."

**Sub-decision — test compilation target.** **Host for v1.** Cross-compilation tests are post-v1.

**Sub-decision — target flag sourcing order.** `--target` CLI flag overrides `kara.toml [build].target`, which overrides host default. `karac` detects host target at its own build time (baked in); no runtime OS detection.

**Forward-compat for post-v1 arch-level support.** F1a's permissive-walker rule preserves two independent extension paths, both of which are natural updates rather than breaking patches:

- **Path 1: `#[cfg]` attributes (item-level).** Purely additive. Lands as a post-v1 feature for fine-grained conditional compilation (`#[cfg(target_arch = "x86_64")]` on items, blocks, expressions). Zero interaction with suffix resolution — users who want arch-split write `#[cfg]` inside a `_linux.kara` file.
- **Path 2: Compound suffixes (module-level).** Extension of F1a's mechanism. `poller_linux_x86_64.kara`, `poller_macos_aarch64.kara`. Most-specific-match wins. Existing v1 `_linux.kara` files remain valid as the lower-specificity fallback. Introduction method is edition-gated or opt-in (`kara.toml [build].compound_suffixes = true`), so v1 files like `foo_linux_x86_64.kara` (module name ending in `_linux_x86_64`) don't suddenly change interpretation.

Both paths are natural updates; neither requires v1 users to rewrite code. The one commitment today that keeps them open: **the walker treats unknown trailing-underscore patterns as ordinary module names**, not errors. This reserves the pattern-space without claiming it.

### F2. Is the v1 file-suffix mechanism's minimum surface enough for stdlib?  *(✓ RESOLVED — 2026-04-21 → F2a, suffixes-only for stdlib v1)*

**Resolution.** V1 stdlib uses OS-level suffixes only — no `#[cfg]` in v1. Document pain points as they emerge (in `docs/deferred.md § Fine-Grained Conditional Compilation`); let concrete stdlib use cases motivate the `#[cfg]` design rather than speculating now.

**Why.** The common stdlib cases (`std.fs`, `std.net`, `std.process`) are OS-level and map cleanly to one file per OS. Arch-level and feature-level differences are post-v1 concerns; design.md:496 already marks `#[cfg]` as a planned post-v1 addition in `deferred.md`. Premature `#[cfg]` surface risks committing to a design before the concrete requirements exist.

### F2. Is the v1 file-suffix mechanism's minimum surface enough for stdlib?  *(🟡 OPEN)*

**Why it matters.** Design.md `:496` calls the suffix mechanism "coarse" and notes `#[cfg]` as planned post-v1. The question: will the stdlib need `#[cfg]` before v1 ships, or can it get by with suffixes?

**Observation.** No stdlib exists today. The stdlib is itself a v1 deliverable and doesn't yet test the mechanism. Low-level modules (`std.fs`, `std.net`, `std.process`, `std.thread`) will want conditional compilation.

**Options:**

- **F2a. Suffixes only.** Stdlib writes one file per platform per low-level module. Fine for `std.fs::File`, more painful for cases where only one function varies.
- **F2b. Suffixes + a minimal `#[cfg(os = "...")]` on items.** Small surface: function-level `#[cfg]` that either keeps or drops an item at parse time based on target. Punts the fuller `#[cfg]` system to post-v1.

**Lean: F2a for v1 per design.** Prove the coarse mechanism works before adding a finer one. Stdlib authors will hit the limits; document them (`docs/deferred.md § Fine-Grained Conditional Compilation` already tracks this); let the concrete pain motivate the `#[cfg]` design rather than speculative v1 surface.

## Group D — Diagnostics & dependency graph

### D1. Circular module dependencies — when detected?  *(✓ RESOLVED — 2026-04-21 → D1a, at module graph construction)*

**Resolution.** Build the `ModuleGraph` during the walker / resolver's first pass by collecting every file's `import` targets as edges. Run Tarjan's SCC on the graph; any SCC of size > 1 is a cycle. Emit `E0223 CircularModuleDependency` before name resolution runs.

**Why at graph construction, not later.** Module topological order is needed *for the resolver itself* — `import db.connection.Connection` can't be resolved until `db.connection`'s items are known. Detecting cycles at graph construction is the earliest point at which the information is available, produces the cleanest diagnostic (no downstream noise from failed resolutions), and lets the typechecker / effect checker / ownership checker assume acyclicity as a structural invariant.

**Diagnostic shape:**
```
error[E0223]: circular module dependency detected
  --> src/a.kara:1:1
   = note: cycle: a → b → c → a
   = help: break the cycle by moving shared items into a separate module that both sides import
```

The `note:` lists every module in the cycle in import order. The `help:` suggests the standard fix (extract shared items to a lower-layer module). Spans on each edge reference the specific `import` line that creates it.

**Interaction with Phase 3 SCC work (checklist line 25-27).** The effect checker's own SCC pass operates on the *intra-module* call graph (functions calling functions inside one module). Module-level SCC (this pass) and function-level SCC (Phase 3) are separate graphs on separate entities. D1a runs once, early; Phase 3 runs per-module during effect inference. No conflict.

### D2. Missing module / missing item diagnostics  *(✓ RESOLVED — 2026-04-21 → two distinct error codes)*

**Resolution.** Two distinct diagnostic codes for the two miss cases. Both include Levenshtein-based "did you mean" suggestions.

- **`E0224 UnknownModule`** — the module path itself doesn't resolve. Example: `import db.connectoin.Connection;` (typo in `connection`). Suggestions drawn from the closest existing module paths.
  ```
  error[E0224]: cannot find module `db.connectoin`
    --> src/app.kara:3:1
    |
  3 | import db.connectoin.Connection;
    |        ^^^^^^^^^^^^^ no such module
    |
    = help: did you mean `db.connection`?
  ```

- **`E0225 UnknownItemInModule`** — the module exists, but the named item isn't in its public API. Example: `import db.connection.Connnection;` (typo in `Connection`). Suggestions drawn from the target module's exported items.
  ```
  error[E0225]: cannot find item `Connnection` in module `db.connection`
    --> src/app.kara:3:1
    |
  3 | import db.connection.Connnection;
    |                      ^^^^^^^^^^^ not found in `db.connection`
    |
    = help: did you mean `Connection`?
    = note: `db.connection` exports: Connection, Pool, Error, DEFAULT_PORT
  ```

**Suggestion algorithm.** Levenshtein distance ≤ 2 on the final path segment (or on the full path for E0224); return the best match if any. The `note:` listing of available exports fires only for E0225 and only when the module exposes ≤ 10 items (to avoid noise); longer exports list is replaced by `= note: see `karac doc db.connection` for the full export list`.

**Combination with E0222.** If the item *exists* in the target module but is not visible (private / default-visibility from a different directory), emit `E0222 PrivateItemAccess` per C2a, not E0225. The two error codes are mutually exclusive: E0225 means "no such name in the module's name table"; E0222 means "name exists but visibility forbids access."

*(D2 resolution is included above with D1's resolution block.)*
