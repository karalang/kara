## Background

The code review looked at `resolver.rs`, `typechecker.rs`, `effectchecker.rs`, `ownership.rs`, `concurrency.rs`, `interpreter.rs`, and `codegen.rs`. Two findings need design input:

1. **`is_shared` metadata stops at typechecker.** The parser sets `is_shared` on `StructDef`/`EnumDef` (ast.rs:98, 123). The typechecker propagates it into `StructInfo`/`EnumInfo`. Codegen branches on it (codegen.rs:652, 872, 1053). But `ownership.rs` and `effectchecker.rs` contain **zero references** to `is_shared`. `tests/ownership.rs` has **zero** tests exercising `shared struct`/`shared enum`.

2. **~25 `panic!` sites in `interpreter.rs`.** CLAUDE.md says "every compiler phase must emit structured diagnostics with source spans ‚Äî never just panic." The interpreter has a `ControlFlow.ExitUnwind` / `error_trace` mechanism for structured error traces ‚Äî but ~25 call sites still use `panic!` for a mix of reasons: user-triggerable runtime errors (division by zero, integer overflow, break/continue outside a loop) and compiler-invariant violations (malformed AST, unknown opcode). No consistent policy.

The semantics of shared types are already canonical in `docs/design.md` and `brainstorming/shared_types.md` ‚Äî `shared struct/enum` = reference semantics, RC-managed assignment, cycles require `weak` on a back-edge. The questions below are about *how those semantics should shape ownership.rs and the interpreter's error model*, not about the semantics themselves.

## Q1. Ownership cycle detection for shared types

**Current behavior (ownership.rs:268-360):** `check_cycles` iterates all entries in `typecheck_result.struct_info` / `enum_info` uniformly. It calls `owned_type_name` (ownership.rs:1418) on each field, which returns `Some(name)` for every `Type.Named` regardless of `is_shared`. A cycle error is emitted with message `"ownership cycle detected: {} ‚Üí {}. Use 'weak' to break the cycle"` (ownership.rs:347).

**Observation:** The error message already mentions `weak` ‚Äî which is the right guidance for shared types but is misleading for non-shared types (where `Weak<T>` isn't the fix; the fix is restructuring). A regular `struct Foo { child: Foo }` is an ownership violation full stop, not a suggestion to use `weak`.

**Framings:**

- **(a) Split by `is_shared`: two error messages.** Shared-type cycle ‚Üí "shared type cycle detected: add `weak` to one back-edge". Non-shared cycle ‚Üí "ownership cycle detected: a type cannot directly contain itself; use `ref`, `Box[T]`, or make it `shared`". The cycle-detection algorithm stays the same; only the diagnostic branches on `is_shared`.
- **(b) Don't distinguish ‚Äî keep the current unified message.** The `weak` suggestion is fine for both. Non-shared users who hit this will learn they need indirection anyway. Keep the code simpler.
- **(c) Don't detect cycles for shared types at all.** Rationale: RC-managed cycles are a runtime leak, not a compile error. Let the user create the cycle and accept the leak, the same way Rust does for `Rc<RefCell<T>>` cycles.

**Tradeoffs:**

- (a) is the most accurate diagnostically ‚Äî the guidance is tailored to the type kind. Small branching cost.
- (b) is the laziest ‚Äî and arguably wrong for non-shared types, where `weak` isn't actually the right suggestion. The current message is misleading for half its audience.
- (c) matches Rust/Swift precedent but removes a safety net. KƒÅra's design philosophy is stricter than Rust about compile-time detection; letting shared cycles slip to runtime violates that.

**Lean:** (a). The language spec explicitly says `weak` is required on back-edges of shared cycles (shared_types.md table). Making the compiler *enforce* that ‚Äî not just suggest it ‚Äî is consistent with KƒÅra's "compile-time correctness" stance. For non-shared types, the diagnostic should steer toward indirection, not `weak`.

## Q2. Parameter mode inference & representation reporting for shared-typed bindings

**Current behavior (ownership.rs:185-244):** Param mode inference runs uniformly across all types. Final representation is built by consulting `rc_values` and `arc_values` maps (which are populated by cycle-detection RC-fallback and Arc-promotion). A shared-typed parameter that doesn't participate in a cycle is absent from `rc_values`, so its representation falls through to `match mode { Own => "owned (stack)", ... }` ‚Äî i.e., reported as `owned (stack)` even though the semantics at codegen are actually shared/RC.

**Observation:** This is a cosmetic misreporting, not a runtime correctness bug. Codegen consults `is_shared` directly (codegen.rs:652, 872, 1053) and does the right thing. But the ownership pass's output ‚Äî used by the compiler's `--explain` flag and by any downstream reasoning ‚Äî says `owned (stack)` for types that are semantically RC-managed.

Whether mode inference itself is "wrong" for shared params is subtler: the `Own`/`Ref`/`MutRef` modes are still meaningful for shared types (how does the function *reference* the RC handle ‚Äî by value, by borrow, by mut borrow?), just sitting on top of the shared semantics rather than replacing them.

**Framings:**

- **(a) Branch representation on `is_shared` only.** Leave mode inference untouched. In the representation-building loop (ownership.rs:214-229), check `is_shared_type(name)` before falling through to the mode match. Shared types always report as `shared (Rc)` or `shared (Arc)`. Small, surgical.
- **(b) Treat shared types as always-RC in `rc_values` from the start.** Pre-populate `rc_values` with every `shared struct/enum` binding at the start of ownership analysis. Then the existing representation code naturally reports them correctly. Deeper change; affects cycle detection (shared types already in `rc_values` at cycle-detection time) and Arc promotion.
- **(c) Introduce a distinct mode variant.** Add `OwnershipMode.Shared` alongside `Own`/`Ref`/`MutRef`. Inference produces it for shared-typed params. Forces every downstream consumer to handle the shared case explicitly.

**Tradeoffs:**

- (a) is minimal-scope, matches codegen's pattern (branch on `is_shared` where it matters), and avoids touching cycle detection. Correct for the reporting issue; leaves mode inference semantics untouched (probably correct ‚Äî see observation above).
- (b) unifies the RC handling but risks knock-on effects (e.g., Arc promotion currently triggers from `rc_values` membership; shared types now always in the set changes its meaning).
- (c) is the most principled but the biggest refactor ‚Äî every consumer of `OwnershipMode` needs updating. Overkill for what is, today, a reporting issue.

**Lean:** (a). The output-layer branch is the smallest change that fixes the visible bug; the semantics of `Own`/`Ref`/`MutRef` for shared-typed bindings are still meaningful (they describe how the RC handle is passed) and shouldn't be collapsed away. If downstream consumers turn out to need shared-vs-not distinctions for other reasons, revisit toward (c).

## Q3. Interpreter error-model policy

**Current behavior:** `interpreter.rs` uses `panic!` at ~25 sites. A non-exhaustive sampling:

| Site | Kind |
|---|---|
| Division by zero (line 2161) | User-triggerable runtime error |
| Integer overflow (line 2311) | User-triggerable runtime error |
| `break` outside a loop (line 487) | Compiler-invariant: earlier phases should reject |
| `'{}' is not a function` (line 499) | Compiler-invariant: typechecker should reject |
| Pattern match failure on ADT (various) | Compiler-invariant: exhaustiveness checker should reject |
| Unknown AST node (various) | Compiler-invariant: AST only emits known variants |

The interpreter does have `ControlFlow.ExitUnwind` and an error-trace mechanism ‚Äî so the infrastructure for structured runtime errors exists, it's just not used uniformly.

**Framings:**

- **(a) Split by triggerability.** User-triggerable runtime errors (division by zero, integer overflow, array-out-of-bounds, unwrap of None) ‚Üí structured error traces via `ControlFlow.ExitUnwind`. Compiler-invariant violations (unknown AST variant, type the checker should have caught) ‚Üí keep as `panic!` with `unreachable!` or a clear message. Rationale: invariant violations are compiler bugs; panicking produces a useful stack trace and signals "file a bug". User errors are program bugs; structured traces let the user debug their program.
- **(b) Everything to structured traces.** Even compiler-invariant panics become traces. Rationale: never panic, full stop. Cost: a compiler bug now produces a polite error message instead of a loud crash + stack trace, making debugging the compiler harder.
- **(c) Keep panics everywhere ‚Äî interpreter is dev-only.** The interpreter exists for Phase 4 bringup; real users will run via codegen. Rationale: the structured-diagnostic rule was written for the production compiler, and the interpreter is a testing tool. Cost: violates the literal wording of CLAUDE.md.

**Tradeoffs:**

- (a) respects the spirit of the CLAUDE.md rule (user-visible errors are structured) while keeping the compiler debuggable. Requires classifying each site ‚Äî a one-time audit, maybe 1-2 hours.
- (b) is absolutist and harder to debug during compiler development. Also paradoxical: if the typechecker is supposed to catch X, and the interpreter silently traces X, that suggests the typechecker's guarantee is weaker than advertised.
- (c) is pragmatic but puts the interpreter on a different diagnostic standard than the rest of the compiler. Inconsistency rots.

**Lean:** (a). Classify the ~25 sites during the fix pass:
- **Structured trace**: division by zero, integer overflow (in debug/checked mode), index out of bounds, `.unwrap()` of None, unreachable panic from `todo!()` in user code.
- **Keep as panic (with `unreachable!`)**: anything the resolver/typechecker/effectchecker/ownership checker should have caught earlier. Use `unreachable!("..., this should have been caught by X")` so the message signals the invariant violation.
- Edge case: arithmetic overflow ‚Äî if KƒÅra does release-mode wrapping like Rust, then this is an `unreachable!` in release and a trace in debug. Revisit alongside the debug/release split (likely a separate question).
