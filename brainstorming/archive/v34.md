# Brainstorming v34 — Interactive REPL / Jupyter kernel design questions

**Status: RESOLVED 2026-04-19.** All five questions resolved and folded back into `docs/design.md` (new section "Interactive Evaluation Model") and `docs/implementation_checklist.md § 5.4` / `docs/roadmap.md § Interactive Development`. See the resolution checklist at the bottom for the chosen options.

---

**Original framing (kept for history).** Kāra commits to shipping an interactive REPL and Jupyter kernel as a first-class, P0 deliverable (not a byproduct of the tree-walk interpreter). The interpreter already exists and is kept in sync with codegen semantics (see [implementation_checklist.md § Interpreter parity scope](../docs/implementation_checklist.md)); the open questions are about the *user-visible* semantics of cell-at-a-time execution and the notebook UX. Each question can be resolved independently.

Positioning: a first-class REPL is a differentiator Kāra can carry that Rust cannot match cheaply (`evcxr` recompiles per cell; JShell pays JVM startup and speaks Java verbosity). Decisions below should be evaluated against that positioning: make the common prototyping loop feel like Python, while preserving the language's ownership and effect guarantees.

## Q1. Cell scope and binding persistence

**The question.** What is the scope of a single notebook/REPL cell?

- **1A. One flat top-level scope.** Each cell extends the same top-level scope. `let x = 5` in cell 1 makes `x` visible in cell 2. Bindings accumulate; re-declaring `x` in cell 3 shadows (or errors — separate question). This matches Python/Jupyter mental model exactly and is what users will expect.
- **1B. Each cell a fresh function body, with an explicit `persist` keyword or `let pub` for across-cell carry.** Isolation by default; cross-cell sharing is opt-in. Matches Kāra's scoping discipline but adds ceremony that's alien to notebook users.
- **1C. Hybrid — 1A for `let` bindings, 1B for `fn` / `struct` / `impl` (those persist) and for anything effectful (effects don't leak across cells without a visible block).** Best-of-both ergonomically but hardest to explain.

**Lean.** Probably 1A with the caveat that *re-declaration* of the same name in a later cell is allowed and shadows — this matches Jupyter's habit of re-running modified cells. The static type system and ownership checker still apply: a value consumed in cell 3 is gone in cell 4. That's correct behavior, not a bug (see Q2 for UX softening).

**Open sub-questions.**
- Does a cell's enclosing scope act like the body of `fn main() { ... }` (allowing `?` via the question-mark operator's function-scoped contract), or like a top-level module (no `?`)? If the former, the "implicit main" wraps the entire cell history.
- Effect accumulation: cell 1 reads a file, cell 2 writes to stdout. Are effects computed per-cell or across the whole session? Per-cell is probably right but needs stating.
- Does re-defining a `struct` or `trait` in a later cell invalidate prior bindings that use it (Python does; Rust cannot)? If yes, the shadowing rule extends to types.

**Action item if resolved.** Document the cell model in a new `docs/design.md` § "Interactive Evaluation Model," parallel to the module scoping rules. Fold back into the book's REPL chapter.

## Q2. Ownership across cells — the `evcxr` paper-cut

**The question.** A user writes in cell 3:

```kara
let v = Vec.new();
v.push(1);
consume_vec(v);  // moves v
```

Then in cell 4:

```kara
print(v.len());
```

The compiler correctly rejects cell 4 (use-after-move). This is the single biggest friction point notebook users hit in `evcxr`, because Python has no equivalent concept — `v` "should just still be there." How should Kāra handle it?

- **2A. Strict — surface the error as-is.** Educational, honest, consistent with compiled behavior. Users learn ownership the hard way. Good for teaching, bad for casual prototyping.
- **2B. REPL-mode auto-clone.** In interactive mode only, when a binding is consumed, the REPL silently `.clone()`s it at the consume site and keeps the original live. Emits a `perf[auto-clone-in-repl]` note so the user sees what happened. Ergonomic, honest about the cost, preserves the "just works" feel. Divergence risk: code that runs in REPL may not compile as-is when saved to a `.kara` file.
- **2C. Diagnostic-only softening.** Keep strict semantics, but the REPL frontend intercepts use-after-move diagnostics and renders them with a notebook-aware hint: "cell 3 consumed `v` — to keep using it across cells, `let v = v.clone()` before consuming, or wrap the original in a `let mut snapshot = v.clone()` before calling `consume_vec`." Same compile error, better teaching moment.

**Lean.** 2C is the honest answer; 2B is the pragmatic one. A hybrid — 2C as default with a `#[repl_auto_clone]` session flag for users who opt in — lets users pick their mode. Auto-clone must never be silent: if it fires, it must leave a trace the user sees.

**Related concern — the "copy paste into a .kara file" divergence.** A core REPL promise should be: "what runs here runs in a `.kara` file." 2B breaks that promise unless paired with a "save session as file" export that inserts the auto-clones explicitly so the saved file compiles.

**Action item if resolved.** Document the REPL-vs-compiled divergence story in the REPL chapter of the book. If 2B ships, add a `karac repl --strict` flag that disables auto-clone for teaching contexts.

## Q3. Jupyter kernel scope and deliverables

**The question.** What does a "first-class Jupyter kernel" mean concretely?

**Minimum viable kernel (P0).**
- [ ] `jupyter_client` protocol compliance (ZMQ, shell/iopub/stdin/control channels).
- [ ] Cell execution returning `text/plain` for the last expression (Python-style display).
- [ ] Stderr routing for diagnostics with source spans (clickable in JupyterLab if possible).
- [ ] Interrupt handling (Ctrl+C cancels a running cell — cleanly for effectful code, via the existing cooperative-cancellation mechanism).
- [ ] Tab completion (minimum: identifiers in current scope + prelude).
- [ ] `%magic` for session introspection (`%effects`, `%ownership`, `%rc` — wrap `karac query`).

**Stretch (P1–P2).**
- [ ] Rich display: `text/html` for structs (reflection-based table rendering), `image/png` for plot outputs.
- [ ] Effect-aware execution indicator: a badge in cell output showing which resources the cell touched.
- [ ] Inline `karac explain` rendering for RC / ownership fallback notes.
- [ ] Widget protocol for interactive controls (IPython widgets equivalent) — deep and probably v2.

**Distribution.** `pip install karac-kernel` must work — even though Kāra isn't Python. Precedent: the `evcxr` kernel, the `IJava` kernel. This is non-trivial packaging work (kernelspec, pre-built binaries per platform, possibly a `conda-forge` recipe).

**Action item if resolved.** File separate checklist entries per bullet in the MVP set. Stretch items become their own Phase 8+ items when they're ready.

## Q4. Effect semantics in the REPL

**The question.** How do effects behave in an interactive session?

- Cell 1: `let f = File.open("data.txt")?` — declares `with reads(FileSystem)`.
- Cell 2: `for line in f.lines() { print(line) }` — extends the session's effect set with `writes(Stdout)`.

In compiled code, a function's effect set is the union of its body's effects. In the REPL, the "function" is the whole session. Do we:

- **4A. Compute effects per-cell, display them in the cell output footer.** Users see "this cell: reads(FileSystem), writes(Stdout)" as a status line. No single session-level effect set.
- **4B. Maintain a session-wide accumulated effect set, show it in a sidebar or with `%effects`.** Mirrors the compiled model more closely.
- **4C. Both — per-cell in the output, session-wide via `%effects`.** Most informative, slightly more work.

**Lean.** 4C. Effects are one of Kāra's differentiators; surfacing them in the REPL is exactly the kind of feedback the language promises but Python/Rust REPLs can't deliver. This is where the notebook story *earns its first-class status* — it shows users something they can't see anywhere else.

**Sub-question: effect-conflict detection.** If cell 2 writes to a file and cell 3 reads from the same file, does the session detect the conflict? In compiled code this would be a concurrency analysis concern. In the REPL, cells run sequentially by definition, so the conflict isn't a correctness bug — but showing the dependency in a timeline ("cell 3 depends on cell 2's write to `Files`") could be a real teaching tool.

**Action item if resolved.** Design the `%effects` magic output format; specify per-cell effect display in the kernel protocol layer.

## Q5. Book chapter positioning

**The question.** Where does the REPL/notebook story live in the book?

- Early (chapter 2 or 3, right after "Getting Started"): positions the REPL as the *primary* way to learn Kāra. Signals to Python-origin readers that Kāra welcomes their workflow. Risk: teaches ownership in a softened context, leaves readers surprised when they hit the strict mode in a compiled file.
- Middle (after types, before ownership): uses the REPL as a teaching aid for concrete examples. Pragmatic.
- Late (after ownership and effects): positions the REPL as advanced tooling. Defers the Python-onramp too long.

**Lean.** A dedicated chapter right after "Getting Started / Installation," showing the REPL and notebook alongside the first `.kara` file. Both paths shown side by side, same example, so the reader learns that REPL and file are two surfaces on the same language. Ownership and effects then get introduced in later chapters using REPL cells as the minimum-ceremony teaching surface.

**Action item if resolved.** Slot into the Phase 8 / book planning. Depends on REPL MVP shipping.

## Meta — sequencing and phase placement

The REPL binary (`karac repl`) is lower-cost than the Jupyter kernel — it can ship when the interpreter's stdlib surface is stable (Phase 4 → Phase 5 tooling). The Jupyter kernel should follow Phase 8 stdlib completeness so users don't hit "function not found" friction on common types. Both are P0 for the language's positioning; neither is P0 for the current active phase.

Possible ordering:
1. `karac repl` binary (simple line-based REPL over the interpreter) — attach to current Phase 5.x tooling work.
2. Resolve Q1 (cell scope), Q2 (ownership UX), Q4 (effect display) — brainstorming resolved before Jupyter kernel design starts.
3. Jupyter kernel MVP (Q3) — Phase 8 parallel track.
4. Book chapter (Q5) — follows kernel MVP.
5. Rich display, widgets, effect timeline — stretch, v1.5+.

Cell body acts like the body of an implicit `fn main() -> Result[Unit, Error] { ... }`, so `?` is legal. Folded into `docs/design.md § Interactive Evaluation Model > Cell Scope`.

Users who want 2B-style ergonomics opt in via `karac repl --auto-clone` / `%set auto-clone on`, which inserts `.clone()` at the consume site and emits a `perf[auto-clone-in-repl]` note — never silent. Folded into `docs/design.md § Interactive Evaluation Model > Ownership Across Cells`.
- [x] **Q3: Split delivery — `karac repl` + browser playground P0, Jupyter kernel P1-delivery / P0-priority.** P0 delivery: `karac repl` binary and a browser playground (`play.kara-lang.org`-style) — both optimize the frictionless-first-try adoption path.
