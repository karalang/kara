# Brainstorming v10 — Design Review Deep Dive

Issues identified during a comprehensive review of design.md after all v9 resolutions have been integrated. Organized into three categories: high-risk interactions between features that could force design changes, moderate-risk gaps that affect usability or performance, and lower-risk items worth discussing for completeness.

---

## Category A: High-Risk Feature Interactions

### 1. ~~RC insertion + auto-concurrency + `shared struct` — the three-way interaction~~ ✅ RESOLVED

**Decision: Option (d) — Two-phase: tentative Rc, then promote to Arc.**

Ownership analysis runs first and tentatively marks RC candidates as `Rc`. Auto-concurrency analysis runs second and identifies parallel regions. Any `Rc` value whose live range overlaps a parallel region is promoted to `Arc`. No fixed-point iteration is needed — Arc promotion never removes parallelism opportunities, so at most one promotion pass suffices.

**Rationale:**
- (a) is functionally equivalent but less explicit about the two-phase nature.
- (b) is too conservative — kills parallelism for common patterns like closure capture + later use.
- (c) always-Arc wastes atomic overhead on values that never cross task boundaries.
- (d) gives Rc when possible, Arc only when needed, with a clean non-iterative algorithm.

**Additional clarification:** "Ambiguous ownership" (the trigger for RC insertion) is now precisely defined in design.md as: "use-sites not ordered by control-flow dominance" — i.e., no single dominance chain exists among the value's use-sites.

**Changes applied to design.md:**
- Part 4 intro: replaced vague "ambiguous ownership" with the precise CFG dominance definition.
- Rc vs Arc section: renamed to "Rc vs Arc — Two-Phase Algorithm" and documented the decided two-phase pass ordering.

### 2. ~~Effect inference cost and incremental compilation~~ ✅ RESOLVED

**Decision: Options (b) + (a) — compiler-level caching with documented expectations.**

Effect inference cost is an implementation performance problem, not a language design problem. The compiler caches per-function effect summaries keyed by (function body hash, callee effect signatures); only functions whose key changes are re-inferred during incremental builds. The spec acknowledges that inference is per-module and recommends keeping modules focused.

**Rationale:**
- (c) sub-module blocks add language complexity to solve a problem that may not materialize in practice — most modules won't have 200+ private functions, and it contradicts the clean "directory = module" model.
- (d) optional annotations on private functions blur the elegant "private = inferred, public = declared" rule, creating style debates rather than enforcing a clear rule.
- (b) caching is a compiler implementation detail — effect summaries are pure functions of (body + callee effects), perfect for content-addressed caching with straightforward invalidation.
- (a) documenting expectations is always good practice regardless of caching.

**Changes applied to design.md:**
- Inference-is-transitive bullet: added compiler caching strategy and recommendation to keep modules focused.

### 3. ~~`Copy` types in conditional consume branches — interaction with RC fallback~~ ✅ RESOLVED

**Decision: Option (a) — accept different instantiations, different costs.**

This is how monomorphization works. The same generic code produces different runtime strategies per type — that's the whole point of the tiered ownership model (owned → ref → RC). The compiler already emits per-instantiation RC notes, which is the right granularity: the programmer sees "RC fallback used here" when they instantiate with a non-Copy type, exactly when they need to know.

**Rationale:**
- (c) requiring `T: Clone` is actively harmful — it forces deep copies when RC (cheap reference count bump) might be cheaper, and pollutes generic APIs with bounds that exist only to avoid a compiler mechanism.
- (b) definition-site notes are too noisy — most generic functions intentionally work across both Copy and non-Copy types. Warning on every such definition is spam.
- (a) is consistent with how generics work everywhere (Rust's `Vec<i32>` vs `Vec<String>` have very different performance profiles too). Per-instantiation RC notes are sufficient and already part of the design.

**No changes to design.md needed** — the existing spec already covers this correctly: Copy types get implicit copies, non-Copy types get RC fallback, and the compiler emits per-instantiation notes.

## Category B: Usability and Expressiveness Gaps

### 4. ~~Patterns that require named lifetimes in Rust — what happens in Kara?~~ ✅ RESOLVED

**Decision: All three sub-questions are covered by existing mechanisms.**

**(a) Iterator adapter chains — works as designed.** Each adapter struct wraps the previous iterator as a `ref` field. The compiler traces borrow sources transitively through field assignments to arbitrary depth. Chain depth (map → filter → take → zip) doesn't change the algorithm — it's just more levels of the same "trace ref field to its source" operation. Each adapter has exactly one `ref` field pointing to the inner iterator, so there's no ambiguity.

**(b) Self-referential structs — already deferred, index-based workaround is standard.** The spec explicitly lists self-referential structs as deferred. The workaround (index-based access with `(start, end)` pairs) is the same pattern used in real-world Rust — most Rust developers avoid self-referential structs too. Not a gap to resolve now.

**(c) Tuple return borrow precision — accept conservative tying.** `ref(a, b)` ties all returned borrows to all sources. This is overly conservative for `split_pair(a, b)` where each element borrows from a different parameter. Per-element annotation like `(ref(a) StringSlice, ref(b) StringSlice)` adds real syntax complexity for a rare case. **Workaround:** return a borrowed struct with named fields instead of a tuple — the compiler traces each field's source independently through field assignment, giving full precision without new syntax.

**No changes to design.md needed** — borrowed struct inference, transitive borrow tracking, and `ref(a, b)` return annotations already cover these cases. Self-referential structs remain correctly deferred.

**Options:**

- **(a) Validate inference against iterator chains; document limitations.** Test the transitive inference algorithm against realistic iterator adapter chains. Document which patterns work and which require RC fallback. Accept that some Rust zero-cost patterns pay an RC cost in Kara.
  - Pro: Pragmatic. Most code doesn't chain 5 iterator adapters.
  - Con: Performance-sensitive code may be forced into RC unnecessarily.

- **(b) Per-element borrow annotations on tuples.** Allow `(ref(a) T, ref(b) U)` to specify different borrow sources for different tuple elements. This restores per-lifetime precision without named lifetimes.
  - Pro: Solves the `split_pair` problem. Zero runtime cost.
  - Con: New syntax. Approaches the complexity of lifetime annotations for complex cases.

- **(c) Accept the conservatism.** Tuple returns with `ref(a, b)` tie all elements to all sources. If the caller needs finer control, return a borrowed struct (where the compiler traces per-field) or use RC.
  - Pro: Simple. No new syntax.
  - Con: Unnecessary scope restriction on callers in the `split_pair` case.

### 5. ~~`collect_all` limited to 8 branches — heterogeneous error handling at scale~~ ✅ RESOLVED

**Decision: Keep 8-branch limit for heterogeneous `collect_all`, add `collect_all_vec` for homogeneous cases. Variadic generics removes the limit later.**

- **`collect_all` (≤8, heterogeneous):** Stays as-is. 8 branches with different error types is sufficient — if you need more, chain multiple `collect_all` calls (trivial workaround). Having >8 heterogeneous branches is also a design smell suggesting the function should be decomposed.
- **`collect_all_vec` (unlimited, homogeneous):** New. Takes a `Vec[Fn() -> Result[T, E]]` (uniform error type), returns `Vec[Result[T, E]]`. Covers the common >8 case: form validation with many fields, health checks across many services, configuration validation — these almost always share an error type.
- **Variadic generics (future):** When comptime lands, a single generic `collect_all` replaces both the fixed overloads and `collect_all_vec`. No arity limit, full heterogeneous support. Clean long-term solution.

**design.md updated** — added `collect_all_vec` to the structured concurrency section.

### 6. ~~`i64` for all indices — negative index handling~~ ✅ RESOLVED

**Decision: Negative indices panic. `.last(n)` provides explicit end-relative access.**

- **Negative indices are out-of-bounds errors.** A negative `i64` used as an index panics at runtime, same as a positive out-of-bounds index. The `panics` effect applies. This is consistent with Kāra's "explicit intent" philosophy — integer overflow traps, and negative indexing traps for the same reason: `-1` might be a computation error, not intentional end-access. No Python-style wrap-around.
- **`.last(n)` for end-relative access.** `.last()` defaults to 0 (returns the last element). `.last(1)` returns second-to-last, `.last(2)` third-to-last, etc. Negative argument panics, out-of-bounds panics. Self-explanatory name, no new vocabulary needed.
  ```kara
  let v = [10, 20, 30, 40, 50];
  v.last()    // 50 — last element (default n=0)
  v.last(1)   // 40 — 1 before last
  v.last(2)   // 30 — 2 before last
  v[-1]       // panic: negative index
  ```

**design.md updated** — added negative index panic semantics and `.last(n)` method.

### 7. ~~Provider bootstrapping — how does `main()` set up providers?~~ ✅ RESOLVED

**Decision: Option (b) — `providers { }` block as syntactic sugar over nested `with_provider`, with evaluate-all-then-scope semantics. `with_provider` remains the primitive.**

`providers { }` flattens the callback pyramid for bootstrapping points (`main()`, test setup, server init). All provider expressions evaluate top-to-bottom **before** any provider scope is entered — fail-fast with nothing to unwind on error. The block desugars to nested `with_provider` calls (first-listed = outermost scope).

`with_provider` stays for single-provider injection, mid-function scoping, and dynamic/conditional swapping. Both forms exist because they serve different use sites — same relationship as `?` and `match` on `Result`.

**Rationale:**
- (a) nested-only forces a helper function with viral effect forwarding (`with _`), fragile to maintain as resources change.
- (c) attribute-based registration is implicit, contradicts "explicit intent" philosophy, and creates hidden initialization order dependencies.
- (b) is pure sugar over the existing primitive, adds no new semantics, and the evaluate-all-then-scope rule cleanly solves the error handling question (no partial scope cleanup).

**Provider interdependency:** Since all expressions evaluate before any scope starts, one provider cannot use another during construction. Shared setup values go in `let` bindings before the block.

**Changes applied to design.md:**
- Added `providers { }` Block section with desugaring rules, semantics table, and interdependency guidance.
- Added TOC entry.

### 8. ~~Error message philosophy and diagnostic quality~~ ✅ RESOLVED

**Decision: Narrow (a) — commit to five diagnostic-quality rules in the spec, as JSON schema additions, not a prose style guide.**

Error message *wording* is an implementation concern and belongs to the compiler team. But error *structure* — what fields exist, what provenance is tracked, how diagnostics are grouped — is design-relevant because it constrains how every compiler phase must record its work. Those structural rules belong in the spec.

Five rules added to the AI-First Compiler Interface section of design.md, as a new "Diagnostic Quality Commitments" subsection under "Structured Compiler Output":

1. **Channel separation — stdout = JSON signal, stderr = human noise.** The JSON diagnostic document contains only errors, warnings, performance notes, concurrency reports, and the `program_effects` summary. Build progress ("Compiling foo"), phase timings, and cache-hit messages go to stderr. AI agents read stdout and have nothing to filter past. Phase timings, when requested, go to a separate file (`--emit-timings=timings.json`), never mixed into the diagnostic stream.

2. **Root-cause grouping via `primary` / `consequence_of`.** Every diagnostic has an `id` and a `primary: true|false` flag. Consequence diagnostics carry `consequence_of: "<id>"` pointing at their root cause. AI agents fix primaries first.

3. **Cascade cap of 20 primary errors per file.** Once exceeded, a `truncated` marker is emitted and reporting stops *for that file* only. Consequence diagnostics don't count against the cap. **Applies only to errors** — warnings, performance notes, and concurrency reports are never capped, because they describe the state of a working program and agents want the complete set.

4. **`concept` field linking to spec anchors.** Every diagnostic carries `concept: "<spec-section-anchor>"` (e.g. `ownership/rc-fallback`). Resolves against `docs/design.md`'s TOC for authoritative context. Anchors, not URLs — part of the spec contract, stable across versions.

5. **`derivation` chain for inferred diagnostics.** Effect, ownership, type, RC-fallback, and parallelization diagnostics that result from transitive inference must include an ordered list of `{file, line, reason}` tuples showing the chain of reasoning. For performance notes and concurrency reports, derivation *is* the content — a bare "this allocates" is unactionable; the chain showing *why* is.

Per-category applicability:

| Commitment                  | Errors         | Warnings       | Perf notes     | Concurrency reports |
|-----------------------------|----------------|----------------|----------------|---------------------|
| `primary` / `consequence_of`| required       | optional       | N/A            | N/A                 |
| Cascade cap (20 / file)     | errors only    | no cap         | no cap         | no cap              |
| `concept` field             | required       | required       | required       | required            |
| `derivation` chain          | when inferred  | when inferred  | always         | always              |

**Rationale:**
- (b) defer-to-implementation risks shipping with poor errors and losing early adopters. More importantly, derivation chains aren't a "polish later" feature — they require every inference phase to record provenance as it walks, which is an architectural decision, not a UI decision.
- Pure (a) as a prose style guide would be scope creep. The narrow version — schema additions plus five rules — avoids that by mandating *what fields exist and what they mean*, not how to word messages.
- The five rules are complementary, not alternatives. Cap addresses cascade; primary/consequence addresses "fix-this-first"; concept addresses context-retrieval for AI agents; derivation addresses the "mysterious inferred result" problem that is the single biggest diagnostic-quality lever for an effect-typed language; channel separation addresses the "info logs before actual errors" problem traditional compilers have.

**Changes applied to design.md:**
- Added `### 1. Structured Compiler Output` → new subsection `#### Diagnostic Quality Commitments` with the five rules and per-category table, placed between the main JSON example and the "Error return traces" subsection.
- Updated the main JSON diagnostic example to show `id`, `primary`, `concept`, and `derivation` fields in context.
- Schema-uniformity cleanup in `karac query concurrency` example: renamed ad-hoc `why_parallel` and stage-level `reason` fields to `derivation` arrays with `{reason}` entries, matching the diagnostic envelope.
- Schema-uniformity cleanup in `serialized_pairs` example: renamed the categorical `reason` enum (`"data_dependency"` / `"effect_conflict"`) to `kind`, freeing the `reason` field name for prose use inside derivation chains.

### 9. ~~Module initialization and global state~~ ✅ RESOLVED

**Decision: Option (a), formalized — module-level `let` bindings are allowed, but their initializers must be compile-time constant expressions. No `const` keyword, no `static` keyword, no module initialization phase.**

Reading the design doc carefully revealed that module-level `let` bindings are **already assumed to exist** in the embedded section (`#[link_section(".vectors")] let INTERRUPT_VECTORS: ...`, `#[link_section(".dtcmram")] let mut dma_rx_buffer: ...`). Item 9 is not "should we add module-level bindings" — it is **"formalize the implicit rule already at work in those examples and decide what is allowed in the initializer."**

The rule: a module-level `let` or `let mut` binding's initializer must be a compile-time constant expression. Kāra has no runtime module-init phase — no eager init, no lazy init, no two-tier `const`/`static` scheme.

**Allowed initializers:**
- Literals, arithmetic/boolean/comparison on literals
- Enum variants with constant arguments
- Struct/tuple/array literals built from constant expressions
- Array repeat form `[0; N]`
- References to other module-level bindings
- FFI pointer literals for hardware addresses (embedded only)

**Forbidden initializers:**
- Any expression with a declared or inferred effect
- Function calls (until `const fn` / comptime lands — deferred)
- Closures
- Runtime heap allocation

**Visibility:** standard three-tier (`pub let`, default `let`, `private let`). `pub let` is a semver commitment on both the type and the value.

**`let mut` at module scope:** allowed, primarily for embedded use (memory-mapped regions, DMA buffers). In application code, prefer runtime-initialized values inside `main`.

**Runtime-initialized values pattern.** Values that need true runtime initialization (loaded config, DB pool, compiled regex) are constructed in `main` and passed via a context struct with methods:

```kara
impl AppContext {
    pub fn new(path: String) -> Result[AppContext, Error]
        reads(FileSystem) reads(Env) allocates(Heap)
    { ... }

    pub fn handle_request(ref self, req: Request) -> Result[Response, Error]
        reads(UserDB) writes(Cache)
    { ... }
}

fn main() -> Result[(), Error] reads(FileSystem) reads(Env) reads(UserDB) ... {
    let ctx = AppContext.new("app.toml")?
    run_server(ref ctx, 8080)
}
```

Method receivers (`ref self`) carry the context without polluting every signature; only the 1–3 functions at the very top pass `ref ctx` explicitly. Pure leaf utilities take neither `self` nor a context parameter.

**Rationale:**

- **Effect discipline.** Kāra's entire effect system rests on the rule that every effect has an attributable source. A `let CONFIG = load_config("app.toml")` at module scope would produce `reads(FileSystem)` with no function to attribute it to, no `?` path for errors, and no answer to "what runs first when two modules' initializers depend on each other." Compile-time initializers eliminate all of these questions by eliminating the runtime step.
- **Embedded cases already fit.** The existing `#[link_section(".vectors")]` / `[0; 512]` examples use only constant initializers — the rule formalizes what the embedded section already does, without adding a restriction.
- **No new keyword needed.** Introducing a `const` keyword in addition to `let` is redundant: the restriction is not keyword-based, it is "module-level initializers must be compile-time constant," which applies uniformly to `let` and `let mut`.
- **Option (b) / (c) / (d) rejected:** (b) eager init has initialization-order bugs and unattributed startup effects; (c) lazy init hides when effects run, conflicting with effect tracking; (d) two-tier `const`/`static` adds a keyword without solving the fundamental effect-attribution problem.
- **Pervasive-parameter fear mitigated by methods.** The `impl` blocks + `ref self` pattern means context-dependent code is written as methods, not as functions taking `ctx: ref AppContext` at every level. The spine of the call graph threads `ref ctx` in 1–3 places; everything below uses `self`.

**Loose end flagged:** design.md previously referenced "calls to `const` functions" in the default-parameter-values rule (line 1398), but `const fn` / comptime is listed as a deferred feature. The wording has been tightened to say "calls to `const fn` will be permitted once the comptime feature lands — not available in v1."

**Changes applied to design.md:**
- Added new `### Module-Level Bindings` subsection under Module System, placed between Module System and Struct Field Visibility. Contains the compile-time initializer rule, allowed/forbidden initializer forms, `let` vs `let mut` at module scope, visibility (including `pub let` semver commitment), rationale for rejecting runtime initialization, the full `AppContext` + `main` runtime-configuration pattern, and a note on request-scoped state vs program-scoped state.
- Added a cross-reference from the `#[link_section]` / `INTERRUPT_VECTORS` embedded example (Linker Control Attributes section) to the new Module-Level Bindings subsection, noting that the embedded examples already satisfy the compile-time initializer rule without special cases.
- Tightened the default-parameter-values rule to cross-reference Module-Level Bindings for the list of allowed forms and to clarify that `const fn` support is deferred.

## Category C: Lower-Risk Items

### 10. ~~`NaN == NaN` is `true` — FFI and serialization implications~~ ✅ RESOLVED

**Decision:** Kāra's `NaN == NaN` rule is a **language-internal contract**. Once a float leaves the language — through FFI or through a serialization format — IEEE 754 reasserts itself. Two boundaries, handled distinctly:

1. **FFI boundary** — already covered. The existing `ffi_float_semantics` warning fires on `extern` float crossings.

2. **Serialization boundary** — new, scoped lint: `float_in_serialized_type`. Fires when a type with `#[derive(Serialize)]` or `#[derive(Deserialize)]` contains an `f32` or `f64` field (directly or transitively). Warning, not error. Points at the float field, not the derive. Suppressible per-field/type/module with `#[allow(float_in_serialized_type)]`. Activates when the derives land (they're deferred post-v1); the lint is part of their definition.

**Key insight — why this lint, not a broader one:** A pure-Kāra JSON/MessagePack/Protobuf library is indistinguishable from any other Kāra code — no `extern`, no compiler-visible crossing. So the FFI warning alone can't catch it. But a blanket `HashMap[f64, _]` / `HashSet[f64]` lint would be far too noisy: float-keyed collections are legitimate in scientific computing, geometry indexing, histograms, and memoization caches where the collection never leaves the process. Tying the lint to the `#[derive(Serialize)]` / `#[derive(Deserialize)]` attributes isolates exactly the types that cross the boundary and leaves in-process float collections alone.

**Format-level NaN variance** (documented as background, not enforced by the compiler):
- **JSON** — no NaN encoding at all. Libraries emit `null`, the string `"NaN"`, or raise.
- **MessagePack** — carries IEEE bits; consumers follow IEEE semantics.
- **Protobuf** — carries IEEE bits; consumers follow IEEE semantics.

Round-trip of a `HashMap[f64, T]` with a NaN key through any of these produces a map whose NaN entry may no longer be reachable on the other end. This is inherent to the formats, not something Kāra can paper over.

**Guidance for NaN-tolerant code that serializes** (in order of preference):
- Use a non-float key: convert via `f64.to_bits` → `u64`, or wrap in a `distinct type` that canonicalizes NaN on construction.
- Normalize NaN to a sentinel value before insertion and strip it after deserialization.
- Accept the footgun explicitly with `#[allow(float_in_serialized_type)]` and document the format's NaN behavior in a comment at the allow site.

**Rejected alternatives:**
- **Do nothing (document-only).** Too easy to miss. The existing FFI warning sets a precedent that boundary crossings deserve compiler attention.
- **Flow-sensitive detection at call sites.** Expensive to implement, requires tracking serialization traits through arbitrary code paths, and creates false negatives when the boundary is dynamic. The derive-attached lint catches the same cases with trivial syntactic analysis.
- **Blanket `HashMap[f64, _]` lint.** Too noisy for legitimate local-use patterns (geometry, histograms, caches). Would generate suppression noise everywhere and train users to `#[allow]` by reflex, weakening the signal.

**Changes applied to design.md:**
- **Float Semantics section** — added "Boundary contract for NaN semantics" paragraph with the two-boundary model (FFI + serialization), the `float_in_serialized_type` lint specification, the explicit non-lint cases (`HashMap[f64, T]`, `HashSet[f64]`, `Vec[f64]` in ordinary code), and the NaN-tolerant serialization guidance.
- Cross-referenced deferred items (`Serialize` / `Deserialize` derives) so the lint's activation condition is explicit.

### 11. ~~No `FnOnce` — silent behavior change when captures change~~ ✅ RESOLVED

**Decision:** Accept the base design — no `FnOnce` type, no `once` keyword, no `move` keyword. Once-callable closures remain an ownership consequence, not a type distinction. But tighten the spec in two places the original brainstorming didn't cover, and add a diagnostic requirement that turns the "silent refactoring transition" into a self-contained error.

**Base rule (unchanged):** Calling a closure is a move when it captures owned non-`Copy` values and moves them out on call. The type is still `Fn(T) -> U`; enforcement comes from move checking.

**New: modular composition — where the error fires.** The brainstorming doc framed this as a local-binding problem, but the more interesting case is a closure passed to a function that calls it in a loop. The spec previously didn't say whether the error fires inside the callee or at the call site. It now does:

- **Call-multiplicity is part of parameter mode inference.** For private functions, the compiler infers whether each closure parameter is called ≤1 time or >1 time. Parameters called more than once carry a *multi-call* constraint.
- **Public `Fn(T) -> U` parameters are multi-callable by default.** No surface syntax for once-callable parameters — it's rare enough that the handful of genuine cases use a factory pattern (`Fn() -> Fn(T) -> U`) or take the owned value directly. This keeps the common case annotation-free and pushes the rare case to an explicit workaround.
- **The error fires at the call site, not inside the callee.** Passing a once-callable closure to a multi-call parameter is rejected where the closure is passed. The diagnostic names the specific owned capture making the closure once-callable and the specific call inside the callee that requires multi-call — so the caller can fix at their end without reading the callee's source.

**New: diagnostic requirement for closure re-use errors.** The brainstorming doc's concern about silent refactoring transitions is a diagnostic-quality problem, not a language-design one. The rule: when the ownership checker rejects a second call of a once-callable closure — locally *or* at a call-site mismatch — the diagnostic must:

1. Point at the specific owned capture whose move-out makes the closure once-callable.
2. Show the first call (which consumed the closure) and the second call (which was rejected).
3. Offer concrete fixes: capture by reference (`&name`), clone inside the closure body (`name.clone()`), or restructure to avoid the second call.

Example shape:

```
error: cannot call `f` more than once
  --> foo.kara:5:1
   |
 3 | let f = |x| { consume(name); x + 1 };
   |               ---- `name` is an owned String, moved out of the closure on call
 4 | f(1);
   | ---- `f` consumed here (name moved out)
 5 | f(2);
   | ^ cannot call — closure was already consumed
   |
   = help: capture `name` by reference: `|x| { consume(&name); ... }`
   = help: or clone inside the closure: `|x| { consume(name.clone()); ... }`
```

With this diagnostic, a refactor that adds a move-out to a closure body no longer produces a confusing "f already moved" — it produces an error that explains the flip and how to undo it.

**Rejected alternatives:**

- **(b) `once` keyword as documentation.** Opt-in annotation that the compiler weakly checks (warn if a closure marked `once` is actually multi-callable). Rejected: comments and binding names already do this job; a keyword that's only documentation is ceremony without force.
- **(c) Require `move` keyword on closures that capture owned values.** Rust-style explicit capture. Rejected: contradicts Kāra's inference-first philosophy; annotates the common case to disambiguate a rare one.
- **Type-level `FnOnce(T) -> U`.** Doubles the closure-type surface area to distinguish a narrow case. Rejected: the constraint (via parameter-mode inference) lives where it belongs — in the callee's analysis — not in a type split users have to reason about at every signature.
- **Compiler note on refactoring transitions.** Would require tracking previous state across compilations. Not feasible, and the diagnostic improvement above makes it unnecessary.

**Changes applied to design.md:**

- **"Closure ownership and once-callable closures" section (around line 1375)** — added two new paragraph blocks:
  - *"Modular composition — where the error fires"*: call-multiplicity as part of parameter mode inference (private), public `Fn(T) -> U` as multi-callable by default, factory pattern for the rare once-callable-parameter case, rule that the error fires at the call site.
  - *"Diagnostic requirement for closure re-use errors"*: the three-point requirement (identify the capture, show both calls, offer concrete fixes) plus explicit statement that the "silent transition" concern is a diagnostic-quality problem the language deliberately does not solve via type-level machinery.

### 12. ~~`sequential {}` usage — should the compiler track frequency?~~ ✅ RESOLVED

**Decision:** Accept option (a), deferred to when `--perf-report` tooling exists. Reject (b) lint threshold. Reject (c) "not a compiler concern."

**Rule:**

- Normal builds emit **no** warning, note, or lint on `sequential { }` usage regardless of frequency.
- `sequential { }` frequency is reframed as a **language-health metric** (feedback for Kāra designers about whether the effect system is expressive enough), not a **code-quality metric** (feedback for a project about whether its code is good).
- When `karac build --perf-report` eventually ships, `sequential { }` block count and locations will be one signal among several — alongside monomorphization sizes, unblocked-parallel-region counts, and other auto-parallelism diagnostics — surfaced only when the user opts in.
- No `--perf-report` flag is specified in design.md yet; it is mentioned as future tooling, not committed.

**Rationale:**

- **Why not a lint threshold (b).** Arbitrary. Some domains (hardware initialization, protocol state machines, explicit register sequences) legitimately need many sequential regions. A lint that fires on correct code gets silenced project-wide, which destroys the signal. Also: pre-v1 we have zero data on what "too many" even means.
- **Why not "not a compiler concern" (c).** The compiler already knows where every `sequential { }` is — pushing this to a third-party linter means re-parsing Kāra to recover information the compiler has for free. Kāra's whole pitch is that the effect system sees semantic dependencies, so *why* someone reached for `sequential { }` is feedback the language designers need, and the compiler is the only tool positioned to collect it losslessly.
- **Why perf-report instead of normal build output.** Language-health metrics belong in a tool users run occasionally and designers read regularly, not in the noisy feedback loop of normal compilation. A warning on every build would train users to silence it; a perf-report opt-in preserves the signal.

**Changes applied to design.md:**

- **"`sequential { }` block — forcing source-order execution" section (around line 2696)** — added one new paragraph immediately after the existing description:

### 13. ~~Compilation speed expectations~~ ✅ RESOLVED

**Decision:** Accept option (b) softened — architectural readiness for incrementality without committing to an actual incremental compilation framework, plus explicit deferral of numeric targets. Reject (a) set targets. Reject pure (c) defer everything.

**Rule — the four architectural principles:**

1. **Function-local analysis by default.** Type checking, effect inference (within an SCC), ownership analysis, and monomorphization requests all operate on function-sized units with explicit, named inputs. Whole-module traversals are a code smell and require justification. This preserves the option of function-keyed caching later.
2. **SCC as the natural cache unit for effect inference.** Effect inference already groups mutually-recursive private functions into strongly-connected components to reach the fixed point — that SCC is the natural invalidation unit, so the implementation preserves SCC boundaries rather than flattening them away.
3. **Named dependencies between phases.** Each analysis result explicitly records what it consumed (called function signatures, in-scope resource declarations, trait impls visible at the call site, generic instantiations requested). A future incremental layer can then invalidate correctly without re-deriving the dependency graph from scratch.
4. **No numeric speed targets before measurement.** Pre-v1 there is no baseline data; picking targets from a hat produces goals that are either trivially met or impossible. The tree-walk interpreter phase is explicitly not a performance target. First targets come from baseline measurements on non-trivial Kāra programs.

**Known-risky passes called out now:**

- **Monomorphization.** Rust's well-known pain point. Function-local and lazy — instantiate on demand, not eagerly.
- **Effect inference SCC fixed-point.** Fixed-point stays local to each SCC; lattice design bounds iteration count (finite lattice, monotonic transfer functions).
- **Ownership analysis with RC fallback detection.** Dataflow on each function's CFG — stays linear as long as it is function-local and does not reach across function boundaries.

**Rationale:**

- **Why not (a) numeric targets.** Pre-v1 we have zero baseline data on Kāra compilation cost. Picking "100k lines/sec type checking" or "<500ms incremental rebuild" without baseline data gives a target that is either trivially met or impossible — neither outcome teaches us anything, and an arbitrary number in the spec becomes a cargo-cult obligation later. The honest target is "fast enough the AI isn't waiting," and for small-to-medium programs on modern hardware a reasonably-written non-incremental compiler already hits this. Scale is where incrementality becomes mandatory, not prototype phase.
- **Why not pure (c) defer.** Dangerous because pass architecture decisions (whether each phase is function-local or whole-module) are hard to undo later. If we write type checking, effect inference, or ownership analysis as whole-module traversals now, we lock out function-keyed caching forever. The tree-walk interpreter phase does not need to be *fast*, but the pass *shape* committed in that phase constrains every later phase.
- **Why (b) softened instead of pure (b).** Full Salsa-style incremental infrastructure doubles compiler complexity. Building it now, for a prototype that is still a tree-walk interpreter, is premature. The four principles preserve the *option* of adding incrementality later without a pass-architecture rewrite, without paying the complexity cost now.

**What deliberately does NOT go in design.md:**

### 14. ~~Parameterized resources — the "syntactically distinct variable names" rule~~ ✅ RESOLVED

**Resolution:** Option (b) with partition-scaling. The compiler proves distinctness statically when it can (literals differing, algebraic distinctness like `id` vs `id+1`, disjoint scopes with no aliasing) and falls back to a runtime **distinctness graph** at each fork point: nodes are the parameterized-resource accesses about to be parallelized, edges connect pairs whose distinctness is unproven. The fork lowers to a partition-by-key — all accesses are grouped into equivalence classes at runtime, calls within a class are serialized, classes run in parallel. Cost scales with the number of *unproven* edges, not with N: a single `if k1 == k2` compare for two unproven accesses, pairwise compares for small N, a `HashMap[Key -> ClassId]` for large N (O(N) amortized). The existing auto-concurrency cost model serializes the whole group if partition cost exceeds expected speedup. Resource key types must implement `Eq` (pairwise path) or `Hash + Eq` (hash-partition path); non-hashable keys like closures are rejected at resource declaration. `karac explain` shows at each fork point which pairs were proven distinct, which are runtime-guarded, and whether partitioning is pairwise or hash-based, so the programmer can hoist checks when a hot loop needs it. This preserves Kāra's data-race-freedom guarantee — silent under-serialization is never accepted — while keeping runtime cost proportional to actual ambiguity. Spec updated in the "Parameterized Resources (opt-in finer granularity)" section.

**Context:** Parameterized resources (`UserDB[user_id]`) allow fine-grained effect partitioning. Two resource parameters are statically distinguishable iff they are different literals or "syntactically distinct variable names."

**Problem:** The "syntactically distinct variable names" rule is surprising:

```kara
fn update_both(id1: i64, id2: i64) {
    update_profile(id1);    // writes(UserDB[id1])
    update_settings(id2);   // writes(UserDB[id2])
    // id1 and id2 are syntactically distinct → parallelized
    // But the caller might pass the same value for both!
}

update_both(42, 42);  // Oops — same user, but the compiler parallelized
```

The spec acknowledges this: "a caller that passes the same runtime value for two syntactically distinct parameters may under-serialize, which is a known and accepted limitation." But this is a correctness issue, not just a performance issue — concurrent writes to the same user row could corrupt data.

**What needs deciding:**

- Is under-serialization acceptable for parameterized resources?
- Should the compiler insert a runtime equality check when two "distinct" parameters come from the same scope? (i.e., at the call site, if `id1 == id2`, serialize instead of parallelize)
- Should parameterized resources require a `#[allow(under_serialize)]` annotation to acknowledge the risk?

**Options:**

- **(a) Accept and document.** The spec already acknowledges this. Add a more prominent warning. The responsibility is on the programmer to ensure distinct parameters have distinct values.
  - Pro: Simple. No runtime cost.
  - Con: Silent correctness bug. "Accepted limitation" that leads to data corruption is a hard sell.

- **(b) Runtime equality check at fork points.** When the compiler parallelizes two calls with parameterized resources from syntactically distinct variables, it inserts `if id1 == id2 { serialize } else { parallelize }`. Runtime cost is one integer comparison per fork point.
  - Pro: Correct. Cheap. No false parallelization.
  - Con: Runtime check in a system advertised as compile-time. Requires the parameter type to implement `Eq`. May be impossible for complex parameter expressions.

- **(c) Conservative default with opt-in.** Don't distinguish variables by default — only literals and constants. Programmers opt in to variable-based distinction with an explicit annotation: `update_profile(id1) #[distinct_resource]`.
  - Pro: Safe by default. Opt-in risk.
  - Con: Reduces the value of parameterized resources significantly. Most real uses involve variables, not literals.

### 15. ~~`main()` return type and error handling~~ ✅ RESOLVED

**Resolution:** `main()` accepts exactly three return shapes: `()`, `Result[(), E: Display]`, or `ExitCode`. On `Ok(())` → exit 0; on `Err(e)` → runtime prints `Error: {e}\n` to stderr via `Display` and exits with literal code `1` (not platform `EXIT_FAILURE`, for byte-reproducibility). `ExitCode` is a stdlib newtype with `SUCCESS`, `FAILURE`, and `from(i32)`; returning it unwinds cleanly with `defer`/`errdefer` running normally. There is intentionally **no** `process.exit()` primitive — it would bypass cleanup, and Kāra's cleanup-is-guaranteed contract overrides the ergonomic. Programs that want both error propagation and custom exit codes write the match-and-eprintln pattern explicitly. The `E: Display` bound is checked directly at the `main()` signature (no `Termination` trait — ceremony for a single use site). Panics exit with code `101` (distinct from `Err`-exit-1, so shell pipelines can distinguish bugs from expected failures), printing `panicked: {message}` + source location; `defer`/`errdefer` behavior during unwind is scoped to item 17. `env.args()` continues to work via inferred `reads(Env)` — no parameter to `main()`, no explicit effect annotation. Backtraces gated on `KARA_BACKTRACE=1`. Spec updated in new "Entry Point (`main()`)" subsection under Effect Types.

---

**Original context:** The spec says `main()` is not `pub`, so effects are inferred. It shows `main()` returning `()` implicitly. The error handling section defines `Result[T, E]` with `?`.

**Problem:** What happens when `main()` returns an error?

```kara
fn main() -> Result[(), AppError] {
    let config = load_config()?;   // may fail
    run_server(config)?;           // may fail
    Ok(())
}
```

Questions:

1. What's the exit code? Does `Err` map to exit code 1? Can the programmer control the exit code?
2. What's printed? Does the runtime print the error? Using `Display`? `Debug`?
3. Can `main()` return types other than `()` and `Result`? (e.g., `i32` as an exit code)
4. If `main()` panics (via `unwrap()` or similar), is that different from returning `Err`?

**What needs deciding:**

- Allowed return types for `main()`.
- Error display behavior when `main()` returns `Err`.
- Exit code mapping.

**Options:**

- **(a) Rust-style.** `main()` can return `()` (exit 0) or `Result[(), E: Display]` (exit 0 on `Ok`, exit 1 on `Err` with error printed to stderr). Custom exit codes via `std.process.exit(code)`.
  - Pro: Proven. Covers common cases.
  - Con: `std.process.exit` bypasses destructors. Not great for a language with `defer`.

- **(b) Richer return type.** `main()` can return `()`, `Result[(), E: Display]`, or `ExitCode` (a newtype for `i32`). `ExitCode` is the clean way to control exit codes without bypassing cleanup.
  - Pro: Clean. Destructors run. Exit code is explicit.
  - Con: More types to learn.

- **(c) Infer from usage.** If `main()` uses `?`, the compiler infers it returns `Result`. If not, it returns `()`. Error display is always `Display` trait. Exit code is always 0 (Ok) or 1 (Err).
  - Pro: Zero ceremony for common cases.
  - Con: No custom exit codes without a separate mechanism.

### 16. ~~Operator overloading — deferred but design-adjacent~~ ✅ RESOLVED

**Resolution:** Operators are trait-dispatched from Day 1. The "deferred" piece is only *user-defined* implementations of operator traits; the trait infrastructure, stdlib impls, and desugaring rules all ship in v1. There is no parser/lexer-level operator table for built-in types to migrate away from — `a + b` is lowered to `Add.add(a, b)` uniformly during the lowering phase (after type checking). The v1 operator trait set covers arithmetic (`Add`/`Sub`/`Mul`/`Div`/`Rem`/`Neg`), comparison (`Eq`/`Ord`), bitwise (`BitAnd`/`BitOr`/`BitXor`/`Shl`/`Shr`/`Not`), and subscript (`Index`/`IndexMut`). All operator traits are **homogeneous** in v1 — `fn op(self, rhs: Self) -> Self` with no associated `Output` type and no heterogeneous `Rhs`. Consequence: no implicit numeric promotion; `i32 + i64` is a compile error, explicit `as` cast required. Mixed-type arithmetic (associated `Output`, heterogeneous `Rhs`) is a later additive extension for SIMD/decimal/duration. Compound assignment (`+=` etc.) desugars to `a = a + b` — no `AddAssign` in v1. `String + String` works via stdlib `impl Add for String` (carries `allocates(Heap)`). `Vec[T]` intentionally does NOT implement `Add` — `vec1 + vec2` is a compile error with a diagnostic pointing at `.concat()`/`.extend()`, because the concat-vs-elementwise ambiguity justifies forcing an explicit method. `Ord` is promoted from "Later" to v1 derivable (requires `Eq`, generates lexicographic field-order comparison). Distinct types can derive `Eq`/`Ord` (enabling `<` comparison) but NOT arithmetic traits — the whole point of `distinct UserId = u64` is to prevent `user1 + user2` from type-checking. Refinement types coerce to base type for arithmetic, matching the existing "arithmetic returns the base type" rule. User-defined `impl Add for MyType` is rejected in v1 with a clear diagnostic; lifting the restriction is a one-line resolver flag flip, zero backwards-compat burden. Diagnostics speak trait language (`"type Vec[T] does not implement trait Add"`), not operator language. `karac explain --concept=operators` surfaces the full desugaring table. Spec updated in new "Operator Traits" subsection after Subscript Trait. Roadmap updated with full v1 checklist. Derivable trait list updated (`Ord` moved to v1 set).

---

**Original context:** Operator overloading was listed as deferred, with the note "desugar `a + b` to `Add.add(a, b)` via standard traits." But several committed features implicitly depend on it:

- `Vec[T]` with `+` for concatenation (or is this a method?)
- `String` with `+` for concatenation (mentioned in the strings section)
- Refinement types with arithmetic (`p + 1` returns the base type)
- Distinct types with `#[derive(Ord)]` — how does `<` work without operator traits?

**Problem:** If operator overloading is deferred, how do these features work in the current implementation? Are operators hard-coded for built-in types? If so, the deferred operator overloading will need to be backwards-compatible with the hard-coded behavior.

**What needs deciding:**

- Are operators for built-in types hard-coded in the compiler or implemented via traits from the start?
- If hard-coded, what's the migration path when operator traits land?
- Should operator overloading be promoted from "deferred" to "Phase N" to avoid this debt?

### 17. ~~Interaction between `defer`/`errdefer` and auto-concurrency~~ ✅ RESOLVED

**Resolution:** Structured concurrency already handles the lifetime question — no new ownership rules are needed. All parallel tasks within a scope must join before the scope can exit, and `defer`/`errdefer` run on scope exit, so cleanup fires strictly after every sibling has joined. A parallel task can legally borrow a binding referenced by a later `defer`/`errdefer` block; the task's lifetime is a strict prefix of the cleanup's. The existing "no move capture" rule plus structured concurrency's join-before-exit guarantee is sufficient. Four clarifications added to the spec so the implicit becomes explicit: (1) an **implicit join barrier** runs before any cleanup block fires; (2) cleanup blocks are **never auto-parallelized among themselves** — strict LIFO execution preserves the close-child-before-parent ordering that `defer` exists for; (3) cleanup effects contribute to the enclosing function's inferred effect set (unchanged) but do **not** serialize the parallel region with itself, because they run after the join; (4) a parallel task may borrow a binding later referenced by cleanup.

Panic unwind semantics (promised by item 15 resolution): panics are an exceptional exit, so **both `defer` and `errdefer` run** during panic unwind, using the same ordering as on error return (`errdefer` first, then `defer`, each LIFO). One wrinkle: `errdefer(e) { ... }` blocks that bind an error value are **skipped on panic** — there is no `Err` value of the binding's type during a panic, and the alternatives (synthesize via `FromPanic` trait, or abort on panic-through-errdefer(e)) trade type-system complexity for marginal coverage. Programmers who need cleanup on every non-success exit use parameterless `errdefer` or plain `defer`. Cancellation flows through cleanup: when a panic in a parallel task triggers sibling cancellation, each cancelled sibling runs its own cleanup stack on the way out (because `Cancelled` is a real error value, `errdefer(e)` blocks fire with `e = Cancelled`). Double-panic — a cleanup block panicking while the stack is already unwinding — **aborts the process** with exit code `134`, a `"double panic"` message, and both panic locations on stderr. No recursive unwinding. This is the only place Kāra's cleanup-is-guaranteed contract breaks, and it breaks loudly and deterministically, because recovering from a cleanup-path panic is fundamentally impossible without losing information. Spec updates in the `defer`/`errdefer` rules list and the Concurrency Semantics panic unwind sequence.

---

**Original context:** `defer` runs at scope exit. `errdefer` runs only on error exit. Auto-concurrency may parallelize statements within a function body.

**Problem:** Consider:

```kara
fn process() -> Result[(), Error] {
    let conn = open()?;
    defer conn.close();

    let data = fetch(conn)?;      // reads(DB)
    let report = analyze(data);   // pure computation
    save(report)?;                // writes(ReportDB)
    Ok(())
}
```

Can `analyze(data)` and `save(report)` be parallelized? `save` depends on `report` (data dependency), so no. But what if:

```kara
fn process() -> Result[(), Error] {
    let conn = open()?;
    defer conn.close();

    let metrics = compute_metrics();   // pure
    let audit = log_access(conn)?;     // reads(AuditDB)
    save_metrics(metrics)?;            // writes(MetricsDB)
    Ok(())
}
```

`compute_metrics()` and `log_access(conn)` have no data dependency and different resources — the compiler might parallelize them. But `defer conn.close()` means `conn` must stay alive until scope exit. Does auto-parallelism interact correctly with `defer`? Specifically:

- If `log_access(conn)` runs in a parallel task, does it borrow `conn` across the task boundary?
- Does the `defer` scope extend to cover the parallel task's lifetime?

**What needs deciding:**

- Do `defer`/`errdefer` bindings create an implicit "must outlive all parallel tasks" constraint?
- Should the compiler refuse to parallelize code that crosses a `defer` binding's scope?
- Or does the structured concurrency model (all tasks join before scope exit) already handle this?

### 18. `dbg()` and transparent effects — interaction with effect-sensitive decisions ✅ RESOLVED

**Resolution (2026-04-11):** Interleaved transparent-effect output is an accepted non-goal. Transparent verbs exist precisely so they don't participate in scheduling; buffering output per-task and flushing in source order at join points would reintroduce the serialization the transparent designation exists to avoid. The spec now documents this explicitly in the transparent-effects section: transparent verbs carry a **no output-ordering guarantee** clause, and source order is preserved only within a single task. `dbg()` gets **per-line atomicity** (each call emits one complete line as a single `write(2)`; POSIX guarantees atomicity up to `PIPE_BUF` = 4096 bytes on Linux) — so output lines are never character-interleaved, only whole-line-reordered under parallelism. Lines larger than `PIPE_BUF` may tear; no runtime check. Snapshot-testing `dbg()` output under any `par { }` region is unsupported; convert to `eprintln` for deterministic stderr capture (`writes(Stderr)` conflicts with itself and forces sequential execution within the parallel region, which is the documented escape hatch). Structured logging libraries must *not* use transparent effects — they should declare a non-transparent `logs` verb so callers see the effect in their inferred signatures and the scheduler serializes concurrent writes. Transparent verbs are reserved for *diagnostic bailout only*: inline inspection (`dbg()`), ad-hoc counters, trace points read by eye during investigation. No `KARA_DBG_ORDERED` env flag, no special buffering mode — both would add semantic weight to "transparent" that defeats the purpose. Spec updates: "Effect Annotation Workflow" transparent-effects bullet (no-ordering guarantee + structured logging guidance) and `dbg()` bullet (per-line atomicity + snapshot-testing unsupported note) in docs/design.md.

**Context:** `dbg()` uses a transparent `debugs` effect. Transparent effects "never conflict with anything and never propagate to callers."

**Problem:** `dbg()` writes to stderr. If the compiler parallelizes two calls that both use `dbg()`, their stderr output may interleave. This is usually fine for debugging, but:

1. Snapshot tests that capture stderr would be non-deterministic.
2. Structured logging (if a library uses `dbg()`-style transparent effects for logging) would produce interleaved output.

The spec says transparent effects are "purely documentary." But they do have observable side effects (stderr writes). The "transparent" designation means the *effect system* ignores them — not that they have no effect on the program.

**What needs deciding:**

- Is interleaved transparent-effect output acceptable?
- Should transparent effects be serialized within a parallel group (i.e., transparent effects don't affect *scheduling*, but their output is buffered and flushed in source order)?
- Or is this explicitly a non-goal? ("If you need ordered output, use a real effect, not a transparent one.")

### 19. `weak` field access ergonomics ✅ RESOLVED

**Resolution (2026-04-11):** Accept the base ergonomics with explicit guidance. No new language surface. `weak` field access returns `Option[T]` — this stays, because the `None` case is a real possibility and surfacing it through the type system is the whole point of `weak`. The idiomatic tool for "structurally guaranteed valid" access is **`Option.expect(msg: str) -> T`** — a general stdlib method, not a `weak`-specific one. The `msg` argument documents the invariant at the use site (`"parent always exists on constructed tree"`), carries the `panics` effect, and surfaces in panic diagnostics. The real ergonomic unlock is **"upgrade once per scope, not per access"**: reading `node.parent` produces an `Option[TreeNode]` whose `Some` arm holds a strong reference valid for the local scope, so programmers should bind it once at the top of a traversal loop instead of chaining `.parent.expect(...).parent.expect(...)` across multiple hops. The `while let Some(p) = current.parent { current = p; }` form from the item 19 example is already optimal — one upgrade per iteration, no redundant work. **Rejected: `unsafe { .assume_valid() }`.** Kāra does not introduce an `unsafe` block for weak-ref assertions; an `assume_valid` that produces UB on a dead reference removes the property that makes `weak` worth having. If a reference's validity is statically guaranteed, it should be a strong reference — restructure to break the cycle, or use a `HashMap[NodeId, NodeData]` back-pointer registry (application pattern, not a language feature). **Rejected: `weak!` macro / `.assert_valid()` shorter spellings.** They're `.expect()` with a different name; a second way to spell the same thing is not worth the language surface area. **Diagnostic rule:** checker emits a suggestion-level diagnostic recommending `.expect("…")` when `.unwrap()` is called on the `Option` from a `weak` field access, with a machine-applicable fix — this follows the general `.unwrap()` rule on `Option`, not a weak-specific check. Spec update: new "`weak` field access ergonomics" subsection under the `weak` reference rules in docs/design.md, covering the four guidance bullets and the rejection rationale.

**Context:** Accessing a `weak` field yields `Option[T]`. The programmer must handle the `None` case via pattern matching or `?.`.

**Problem:** In tree/graph structures with parent pointers, almost every access to `parent` requires unwrapping:

```kara
shared struct TreeNode {
    val: i64,
    mut parent: weak TreeNode,
    mut children: Vec[TreeNode],
}

fn root(node: TreeNode) -> TreeNode {
    // Must handle None at every level
    let mut current = node;
    while let Some(p) = current.parent {
        current = p;
    }
    current
}
```

This is correct but verbose. In practice, many `weak` references are known to be valid during certain program phases (e.g., a parent pointer is always valid while the tree exists). The programmer has no way to express "this weak ref is guaranteed valid in this context" without `unwrap()` (which adds `panics`).

**What needs deciding:**

- Is this the intended ergonomics for `weak` fields?
- Should there be a `weak!` or `strong` assertion that's cheaper than `unwrap()` but still panics?
- Or should there be a way to temporarily "strengthen" a weak reference for a scope (like Rust's `Weak.upgrade()` already in the spec)?

**Options:**

- **(a) Accept the ergonomics.** `weak` references are intentionally uncertain. The `Option` unwrapping is the cost of memory safety. `?.` chains make it tolerable.
  - Pro: Safe. Consistent. No special cases.
  - Con: Verbose for patterns where validity is structurally guaranteed.

- **(b) `expect` method with message.** `node.parent.expect("parent always exists in constructed tree")` — panics with a message. Same as `unwrap()` but with better diagnostics. Still carries `panics` effect.
  - Pro: Better error messages. Same safety model.
  - Con: Still panics. Still carries the effect.

- **(c) `assume_valid` in `unsafe` block.** For performance-critical graph traversal, allow `unsafe { node.parent.assume_valid() }` that skips the liveness check. Undefined behavior if the referent is dead.
  - Pro: Zero-cost for hot paths. Explicit unsafe boundary.
  - Con: Unsafe escape hatch for a safe abstraction. Defeats the purpose of `weak`.
