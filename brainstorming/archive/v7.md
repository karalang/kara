# Brainstorming v7 — Open Design Questions

These items were identified during a full design doc review and require explicit decisions before the spec can be considered complete. Each item includes the current state, the problem, and options to consider.

---

## 1. `panics` vs `Heap` asymmetry — ✓ RESOLVED

**Decision:** Option (a) — documented the rationale in `design.md` (Conflict Rules section, after the conflict table).

**Current state:** `allocates(Heap)` is transparent by default — it does not propagate up the call chain and does not need to be declared by callers. `panics` always propagates and must be declared by every public function that may panic (directly or transitively).

**Problem:** The asymmetry is unexplained. Both are "side effects" of calling a function. A developer encountering this for the first time will reasonably ask: why does `Heap` get to be invisible but `panics` doesn't?

**Options:**
- **(a) Explain the asymmetry explicitly in the design doc.** Rationale: `Heap` is benign under normal conditions (every function can allocate; it's the default mode); `panics` is exceptional — a crashing program has fundamentally different behavior than a running one, so callers must opt in to the risk. Document this reasoning.
- **(b) Make `panics` transparent too.** Removes the asymmetry, but hides crash risk from callers. Likely a bad trade-off for a safety-focused language.
- **(c) Introduce a `#[no_panic]` proof annotation** that allows functions to opt out of `panics` propagation if they can prove termination (e.g., no `unwrap()`, no index access). Transparent by default for provably-safe functions.
- **(d) Make `Heap` non-transparent** — require `allocates(Heap)` declarations everywhere. Eliminates the asymmetry in the other direction. Very verbose; likely unacceptable ergonomically.

## 2. RC insertion criteria — formal algorithm — ✓ RESOLVED

**Decision:** Options (a)/(b)/(c) are not alternatives — they are three named triggers for the same fundamental rule. Rc vs Arc resolved via live range ∩ parallel region intersection. Formal algorithm added to `design.md` (Part 4: RC Fallback, "RC Insertion Algorithm" subsection).

**Current state:** The spec says "when the compiler can't determine ownership statically and `ref` isn't used, it falls back to RC." No formal definition of "can't determine."

**Problem:** An implementer needs a precise rule. Two specific sub-questions:
1. What is the exact condition that triggers RC insertion?
2. When does the compiler choose `Rc` vs `Arc`?

**Options for (1):**
- **(a) Any parameter that is consumed on at least one code path and also used (read or consumed) on another code path after the first use** → RC. This is conservative but deterministic.
- **(b) Any parameter that crosses a closure boundary where the closure outlives the function** → RC. Plus (a).
- **(c) Any parameter stored into a data structure whose lifetime the compiler can't track statically** → RC. Plus (a) and (b).

**Options for (2):**
- `Rc` whenever the value never crosses a task boundary (no effect that implies task-level parallelism touches it). `Arc` when effect analysis detects the value is used across concurrent tasks.
- The challenge: effect analysis is conservative. A value near a concurrent call might get `Arc` unnecessarily. Need a rule for how "close" is too close.

**Decision needed:** Write a formal flowchart for RC insertion. The spec should be precise enough that two independent implementers produce the same behavior.

## 3. Parameter mode inference edge cases — ✓ RESOLVED

**Decision:** Option (a) for `if` branches and loop bodies — pessimistic/syntactic; any consume makes the parameter `own`. Match guards: consuming inside a guard is a compile error (guards can fail; the value must remain available for subsequent arms). Documented in `design.md` (Part 1: Parameter Mode Inference, "Inference rule" section).

**Current state:** The rule is "if any code path consumes a parameter, it's `own`." Edge cases are not covered.

**Problem:** Three specific gaps:
1. A parameter consumed in only one branch of an `if` — is it `own` even when the consuming branch is never taken?
2. A parameter consumed conditionally in a loop body — same question.
3. A parameter consumed inside a `match` guard — guards don't affect exhaustiveness but do affect reachability.

**Options:**
- **(a) Pessimistic / syntactic analysis:** Any syntactic consume in any arm/branch/guard makes the parameter `own`, regardless of reachability. Simple to implement; predictable for the programmer. May produce unnecessary RC fallback.
- **(b) Flow-sensitive analysis:** If the compiler can prove a consuming branch is unreachable (e.g., dead code), it may infer `ref`. More precise but complex; requires control-flow reachability analysis before mode inference.
- **(c) Conservative-but-not-pessimistic:** Match guards do not count as consuming code paths (since guards can fail). `if` branches do. Loop bodies do (since the loop may execute). Middle ground.

**Recommended starting point:** Option (a) for the first release — simple, predictable, worst case is unnecessary RC which the programmer can override. Document the exact rule.

## 4. Effect group versioning — stable vs. open — ✓ RESOLVED

**Decision:** Option (b) — add a `stable` group modifier. Groups are open by default (option (a) behavior), which is correct for extensibility. `stable` is an opt-in promise by library authors that the group's effect set is fixed; the compiler enforces it by rejecting additions. Documented in `design.md` (Feature 2: Effects, versioning/compatibility table section).

**Current state:** All effect groups are "open contracts." Adding an effect to a published group is a minor version change. Callers using the group name automatically accept new effects without a code change.

**Problem:** This means a library can silently serialize previously-parallel operations in a downstream codebase by bumping a minor version. The downstream developer gets no warning — their code still compiles, but may be slower. There's also no way for a library to signal "this group will not grow" (a stability promise).

**Options:**
- **(a) No change.** The current design is a deliberate trade-off: group names are the extensibility mechanism, and callers who want stability should use concrete individual effects instead. Document this clearly.
- **(b) Add a `stable` group modifier.** `stable effect group Validation { reads(UserDB), writes(AuditLog) }` — the compiler rejects adding new effects to a `stable` group without a major version bump. Libraries opt in explicitly.
- **(c) Versioned effect groups.** `effect group Validation@1 { ... }`, `effect group Validation@2 { ... }`. Callers pin to a version. Complex; probably over-engineered.
- **(d) Emit a deprecation-style warning at the call site** when a group gains a new effect, so downstream callers are at least notified on their next compile.

**Decision needed:** Is option (a) good enough, or is the silent performance regression risk unacceptable for library authors?

## 5. `shared struct` + `pub mut` + concurrency safety — ✓ RESOLVED

**Decision:** Option (a) — `pub mut` fields on `shared struct` may only be written from outside the module via `pub fn` methods that declare `writes(...)` effects. Direct field writes (`obj.field = x`) are module-internal only. This ensures the effect system can detect all `writes`+`writes` conflicts at concurrency boundaries. Documented in `design.md` (Struct Field Visibility, `` `shared struct` fields `` section).

**Current state:** The spec claims "concurrency safety for `pub mut` fields on `shared struct` is enforced at the function level by the effect system: concurrent mutations to the same resource produce a `writes`+`writes` conflict."

**Problem:** This claim is harder to satisfy than it appears. The effect system prevents two *functions* from being parallelized if both write to the same resource. But:
- Any RC holder can mutate a `pub mut` field directly (not via a method), without going through an effect-annotated function.
- Two closures that each hold an RC to the same `shared struct` and each write to a `pub mut` field could be run concurrently without the compiler seeing a `writes(X)` conflict — because neither closure is annotated with effects.
- Field access is not effect-tracked; only function calls are.

**Options:**
- **(a) Require `pub mut` fields on `shared struct` to be accessed via methods only.** Direct field mutation from outside the module is disallowed; only module-internal code (which the compiler can track) can mutate directly. Externally, mutation goes through `pub fn` methods that declare effects. This is the cleanest solution.
- **(b) Make `pub mut weak field` and `pub mut field` on `shared struct` implicitly `Mutex`-wrapped.** Correct but adds overhead and removes the "effect system is sufficient" claim.
- **(c) Add a lint/warning for `pub mut` fields on `shared struct`** and note in the spec that the programmer is responsible for ensuring effect annotations cover all mutation paths.
- **(d) Accept the current model as-is** and clarify that `pub mut` fields are "unsafe-adjacent" — the programmer is asserting that all mutation happens through tracked functions.

**Decision needed:** Is option (a) (mutation via methods only from outside the module) acceptable ergonomically? It makes the struct behave more like a classic encapsulated type.

## 6. `dyn Trait` effect computation scope — ✓ RESOLVED

**Decision:** Option (b) — trait-level effect bounds. Trait definitions declare their effect surface with a `with` clause; all impls must have effects ⊆ that bound (compile error otherwise). Call sites use the declared bound. Traits with no `with` clause are effect-opaque at `dyn Trait` call sites. Documented in `design.md` (Effects section, `` `dyn Trait` `` paragraph).

**Current state:** The spec says effects on `dyn Trait` calls use "the worst-case union of all known impl effects." This section is explicitly deferred to Phase 8.

**Problem:** Two concrete questions that must be answered before implementation:
1. **Scope of "known":** Is it all impls in the current compilation unit? All impls the compiler has ever seen (across the whole program)? Only impls the programmer explicitly lists?
2. **Extensibility:** If a new impl is added later (e.g., in a downstream library) with different effects, does it break existing call sites?

**Options:**
- **(a) Closed-world assumption at link time.** The compiler collects all impls for the trait across the entire program and takes their union. Correct but requires whole-program analysis; breaks incremental compilation for dynamic dispatch.
- **(b) Open-world with declared bounds.** The programmer declares an effect bound at the trait level: `trait Handler with reads(UserDB)` — all impls must have effects ⊆ that bound. Call sites use the declared bound, not the actual impls. New impls that exceed the bound are a compile error.
- **(c) Conservative: `dyn Trait` always gets the "all effects" annotation** (equivalent to `with _`). Maximally safe; eliminates the need to track impl effects through dynamic dispatch. Call sites that use `dyn Trait` are effect-opaque.

**Recommended starting point:** Option (b) — trait-level effect bounds. This is consistent with how Rust's `Send`/`Sync` bounds work for trait objects. The programmer states the contract at definition time.

## 7. `collect_all` with heterogeneous error types — ✓ RESOLVED

**Decision:** Option (c) — `collect_all` returns a heterogeneous tuple of Results: `(Result[A, E1], Result[B, E2], Result[C, E3], ...)`. No error unification required. Since all branches run in parallel and complete before `collect_all` returns, all outcomes are already available — the full tuple preserves the complete error trail. Fail-fast behavior is derived by chaining `?` on each result. Documented in `design.md` (`collect_all` Type constraints section).

**Current state:** The spec says `collect_all` requires all branches to return `Result[_, E]` with the same error type `E`. No exception.

**Problem:** In practice, branches often produce different error types (e.g., `FetchError`, `ParseError`, `DbError`). The programmer must manually unify them. Compare with how `?` handles this — cross-type propagation is allowed via `impl From[OtherError] for MyError`.

**Options:**
- **(a) Keep the restriction.** Forces the programmer to think about error unification explicitly before calling `collect_all`. Simpler compiler implementation; cleaner type signatures.
- **(b) Allow `impl From` to resolve heterogeneous types automatically**, the same way `?` does. If `From[B] for A` and `From[C] for A` are in scope, `collect_all(|| -> Result[_, B], || -> Result[_, C])` infers `E = A`. Ergonomic but adds inference complexity.
- **(c) Allow heterogeneous types, return a heterogeneous tuple of Results.** `collect_all(|| -> Result[X, E1], || -> Result[Y, E2])` returns `(Result[X, E1], Result[Y, E2])`. No unification needed. The type is already a tuple; just let the error types differ too. May require `collect_all` to have a different signature or overload set.

**Decision needed:** Does option (c) work cleanly with the fixed-arity overload model already in the spec? If so, it may be the simplest and most expressive choice.

## 8. Forcing sequential execution for non-conflicting operations — ✓ RESOLVED

**Decision:** Gap 2 — option (b): add a `sequential { }` block. All statements inside execute in source order; auto-parallelism is suppressed. The block is an expression (value = last statement). Covers the `()` return case where artificial data dependencies would be noise. Gap 3 — option (a): `karac build --sequential` is a debug-only, global, compile-time flag; per-scope production needs are served by `sequential { }`. Documented in `design.md` (Concurrency Semantics — `sequential { }` block and `--sequential` entries).

**Current state:** Two analyses determine whether operations run sequentially or in parallel: data dependency (B uses A's return value) and effect conflict (writes+reads or writes+writes on the same resource). Source order is now documented as the tiebreaker when serialization is forced. Gap 1 (source-order guarantee) was fixed directly in `design.md`.

**Problem — two open questions remain:**

**Gap 2: No escape hatch for pure/non-conflicting operations that need ordering.**
If two operations have no data dependency and no conflicting effects, the compiler parallelizes them. But sometimes there is a semantic ordering requirement the effect system cannot see — a protocol step, a precondition that is not expressed as a resource, or an operation whose side effect is not tracked. The doc says this is "the programmer's responsibility to express via explicit data flow," but gives no mechanism.

The workaround is an artificial data dependency:
```
let token = step_a();
let result = step_b(token);   // B now depends on A's return
```
This works when `step_a` returns a meaningful value. When it returns `()`, the programmer has no clean option — threading a unit value through `step_b` is noise. There is no documented sequencing primitive.

**Gap 3: `karac build --sequential` scope is ambiguous.**
The flag is mentioned once under Concurrency Semantics as a debugging tool. It is not clear:
- Is it a global compile-time flag that disables all auto-parallelism, or a runtime switch?
- Is it available in production builds, or debug-only?
- Is there a per-scope equivalent (`#[sequential]` block or annotation) for cases where sequential execution is needed in production?

**Options for Gap 2:**
- **(a) Document the artificial data dependency workaround explicitly.** No new syntax. Accept that `let _ = step_a()` followed by `step_b()` is not sufficient (no dependency created) and that callers must thread a return value. If `step_a` returns `()`, this is a design smell — the operation should declare an effect that naturally serializes it.
- **(b) Add a `sequential { }` block.** Operations inside execute in source order, regardless of what the effect system says. Parallel execution is suppressed for the block. Simple; explicit; zero ambiguity. Cost: new syntax and a block the compiler must treat specially.
- **(c) Add a `seq()` builtin combinator.** `seq(step_a, step_b)` — runs step_a then step_b, returns step_b's result. Composes without introducing a block syntax. Similar to Haskell's `>>` for sequencing effectful actions. Works well with the pipe operator.
- **(d) Rely on effect declarations.** If two operations truly need sequencing, they share a resource — require the programmer to declare the shared resource explicitly via effect aliasing. No new language construct; disciplines the programmer to make the dependency visible. Enforces that "invisible" ordering requirements become visible in the effect system.

**Options for Gap 3:**
- **(a) `--sequential` is debug-only, global, compile-time flag.** Document this clearly. For per-scope sequential needs, option (b) or (c) above covers it.
- **(b) `--sequential` is also available in production, as a diagnostic / compatibility mode.** Useful for tracing correctness bugs where parallelism is suspected. Still global.
- **(c) Add a per-function or per-scope annotation** (`#[sequential]`) that disables auto-parallelism within that scope regardless of build mode. Complements the global flag.

**Decision needed:**
1. Is there a real use case for Gap 2 that options (a) or (d) don't cover? If so, pick (b) or (c).
2. Clarify the semantics and scope of `karac build --sequential`.

## 9. `with` keyword parsing disambiguation — RESOLVED

**Decision:** Option (a) — position-based disambiguation. No syntax changes needed.

**Resolution:** The four uses of `with` are never ambiguous because the parser always knows which production it is in before encountering the keyword:

1. **Generic effect variable** (`fn f[T, with E]`) — parser is inside `[...]` delimiters, parsing a generic parameter list. `with` after a comma is unambiguously an effect variable declaration.
2. **Function effect annotation** (`fn f() -> T with reads(DB)`) — parser has finished parsing the return type at the top level of a function signature. `with` here opens the effect annotation.
3. **Closure type annotation** (`Fn(T) -> U with reads(DB)`) — same as #2 but nested inside a `Fn` type production. Parser knows it is parsing a closure type.
4. **Effect group in annotation** (`with Validation writes(AuditLog)`) — not a separate `with` keyword. Group names are items in the space-separated effect list opened by `with` from cases #2/#3. No nested `with`; no disambiguation needed.

This is the same pattern used by many established languages: `for` in Rust (loops vs HRTB), `static` in C/C++ (linkage vs persistence vs class members), `in` in C# (foreach vs parameter modifier vs LINQ). Positional disambiguation is standard practice.

**Effect list syntax: space-separated, not comma-separated.** Effects after `with` are space-separated: `with reads(DB) writes(Log) panics`. This reads naturally — `with` opens a context and everything following is a peer item under that umbrella. Commas would make `with` appear to bind only to the first item. Style recommendation: use multiline formatting for 3+ effects:

```
pub fn process(order: Order) -> Result[Receipt, Error]
    with reads(UserDB, InventoryDB)
         writes(OrderDB)
         sends(PaymentGateway)
         allocates
```

**Practical impact is small:** only `pub` functions require effect annotations — private function effects are inferred. Most pub functions have 1-2 effects; functions with many effects should use effect groups.

## 10. Borrowed struct borrow tracking semantics — RESOLVED (option b: source pinning)

**Resolution:** Use source pinning — constructing a borrowed struct creates a borrow on each source variable, pinning it (cannot be moved, reassigned, or dropped) for the struct's lifetime. No lifetime annotations; no full lifetime region inference.

**How the three gaps are resolved:**

1. **Multiple `ref` fields from different sources:** Each source gets an independent borrow. All sources stay pinned until the borrowed struct's last use.
   ```
   let input = "source text"
   let config = load_config()
   let parser = Parser { source: ref input, config: ref config, position: 0 }
   // Both `input` and `config` are borrowed — neither can be moved or reassigned while `parser` is live.
   ```

2. **Returning a borrowed struct:** A function may return a borrowed struct only if all `ref` field sources come from the function's parameters (not locals). The caller's borrow then extends to cover the returned value — analogous to Rust's single-lifetime elision rule without explicit `'a` annotations.
   ```
   fn make_parser(source: ref String) -> Parser {
       Parser { source: source, position: 0 }  // OK — source is a parameter
   }

   fn bad_parser() -> Parser {
       let local = "text"
       Parser { source: ref local, position: 0 }  // ERROR — local would be dropped on return
   }
   ```

3. **Source moved/dropped while struct is live:** Compile error. The borrow prevents it — same semantics as Rust's "cannot move out of `x` because it is borrowed."
   ```
   let input = "source text"
   let parser = Parser { source: ref input, position: 0 }
   let moved = input   // ERROR — `input` is borrowed by `parser`
   ```

**Why not the other options:**
- **(a) Lifetime regions** — full lifetime inference is the most complex part of Rust's type system and the biggest contributor to its learning curve. Source pinning achieves the same safety for common patterns without the implementation complexity or annotation burden.
- **(c) Same-scope only** — too restrictive. Borrowed structs like `Parser` need to be returnable from factory functions; same-scope confinement makes them barely useful.

**Implementation:** The ownership checker tracks a `BorrowSet` per variable. When a borrowed struct is constructed, the checker adds a borrow entry for each `ref` field's source. Move/reassign/drop of a borrowed source triggers an error with a span pointing to the conflicting borrow. For returns, the checker verifies all `ref` sources trace back to parameters.

**Multi-source and transitive inference (brainstorming v9 §2):** The compiler fully infers the borrow set for multi-source and transitive borrowed struct returns — no `ref(a, b)` annotation is needed on borrowed struct returns. Source pinning guarantees every `ref` value is traceable, so inference always succeeds. When borrow complexity outgrows borrowed structs, `shared struct` (RC) is the escape hatch.

## 11. Transparent effects — conditional propagation contradiction — RESOLVED (option c: rename the mechanism)

**Resolution:** The spec was conflating two distinct mechanisms under the word "transparent." They are now separated:

| Mechanism | Example | In effect set? | Propagates to callers? | Profile can restrict? |
|---|---|---|---|---|
| **Transparent** | `debugs` | No | No | No — invisible by design |
| **Default-permitted** | `allocates(Heap)` | Yes | Yes (normally) | Yes — default profile permits; GPU/embedded profile rejects |

**Transparent effects** (e.g., `debugs`) are truly invisible: they never appear in a function's effect set, never propagate, and never conflict with anything. The `transparent effect` keyword retains this absolute meaning.

**Default-permitted effects** (e.g., `allocates(Heap)`) are real effects that the effect checker infers and propagates like any other effect. The only special treatment is that the *default project profile* includes a permit rule for them, so they do not require explicit declaration in function signatures. A GPU or embedded profile removes that permit rule, causing undeclared `allocates(Heap)` to be flagged as an error.

**How it works end-to-end:**
```
// Effect checker infers allocates(Heap) — it propagates normally
fn build_list() -> Vec[i32] {
    Vec.new()  // inferred: allocates(Heap)
}

fn caller() {
    let xs = build_list()  // inferred: allocates(Heap) — propagated from build_list
}

// Default profile: allocates(Heap) is permitted — no error
// GPU profile: allocates(Heap) is NOT permitted — error: "caller() has undeclared effect allocates(Heap)"
```

**Why not the other options:**
- **(a) Transparency is always-global** — if `allocates(Heap)` never propagates, the project profile has nothing to latch onto. You'd need a separate scan mechanism outside the effect system to detect heap allocation, duplicating what the effect checker already does.
- **(b) Transparency is context-sensitive** — couples the effect checker to the project profile during inference. Inference should be deterministic from source code alone; profiles should be a post-pass filter, not an input to inference.

**Spec update needed:** Replace "transparent by default" language for `allocates(Heap)` with "default-permitted." The `transparent effect` declaration keeps its absolute semantics. Project profiles gain a `permit` list for default-permitted effects.

## 12. `shared struct` mutation — internal vs external annotation rule — RESOLVED (option a)

**Resolution:** Split the ambiguous sentence in design.md into two explicit rules that distinguish internal from external mutation:

| Context | Rule |
|---------|------|
| **Internal** (within defining module) | Mutating `mut` fields does not require an effect annotation — the compiler tracks mutations through private function effect inference |

## 13. `collect_all` — effect inference at call site — RESOLVED (option a)

**Resolution:** The inferred effect of a `collect_all` call is the union of all branch closures' effects. This is not a special rule — it follows directly from the existing closure effect propagation semantics:

1. `collect_all` invokes all its closure arguments (it does not store them).
2. Invoked closures use `with _` semantics — their effects propagate to the caller (design.md line 1809).
3. Multiple closure arguments each carry their own effects; the compiler unions them at the call site (design.md line 1835).

**Example:** If `branch1` has `reads(DB)` and `branch2` has `writes(Log)`, the enclosing function's inferred effect set includes `reads(DB) writes(Log)`. For a public function, both effects must be declared.

**Why not (b):** Option (b) describes *stored* closures (`with writes(R)` form), where effects track to the point of invocation, not creation. `collect_all` invokes its closures immediately and waits for all to complete — this is the `with _` pattern, not the storage pattern.

**Why not (c):** Option (c) is equivalent to (a) but frames it as a `collect_all`-specific rule. No special rule is needed — the general closure invocation semantics already cover this case. Adding a redundant rule would imply that closures elsewhere might work differently, which they don't.

**Spec update:** No design.md change needed — the existing closure effect propagation rules (Closures and Effect Capture section) and the `collect_all` description already imply the correct behavior. Added an explicit effect note to the `collect_all` example for clarity.

## 14. `weak` references in pattern matching — RESOLVED (option a)

**Resolution:** Weak field access yields `Option[T]` where the `Some` variant already holds a strong reference (upgrade happens at the point of access). Pattern matching on weak fields requires no special rule — it follows directly from existing semantics:

1. Accessing a `weak` field yields `Option[T]` (design.md line 2211). This access IS the upgrade — the runtime checks liveness and returns `Some(strong_ref)` or `None`.
2. `Some(p)` destructures the `Option` normally. `p` is a strong reference, guaranteed live for the duration of the match arm.
3. If the referent was deallocated, the access yields `None`, and the `Some(p)` pattern does not match.

The programmer never sees a "raw weak" value. Weak references are a storage-level concept: they affect how the field is stored (weak RC pointer) and what happens when you read it (liveness check → `Option`). Once you have a `Some`, you have a strong reference.

**Example:**
```
shared struct TreeNode {
    val: i64,
    mut parent: weak TreeNode,
    mut left: Option[TreeNode],
}

if let TreeNode { parent: Some(p), .. } = node {
    // p is a strong TreeNode reference — RC was incremented when
    // node.parent was accessed and yielded Some(p)
    print(p.val);   // safe — p is guaranteed live
}
// p goes out of scope → RC decremented
```

**Why not (b):** `.upgrade()` and field access do the same thing — both yield `Option[T]` with a strong reference inside `Some`. Requiring an explicit `.upgrade()` before matching adds ceremony with no safety benefit. `.upgrade()` remains available for cases where the programmer wants to name the `Option` explicitly before matching.

**Why not (c):** If `p` inside the arm were still a weak reference, the `Some(p)` pattern would give a false sense of liveness. The programmer would need `?.` on every access of `p`, making the `Some` match pointless. This contradicts the spec's design: weak field access already performs the upgrade and returns `Option[T]` with a strong `Some` — there is no observable weak-reference-in-hand state.

**Spec update:** No design.md change needed. The existing rules — "Accessing a `weak` field yields `Option[T]`" and "destructuring increments RC; bindings are shared references" — already imply option (a). Added a clarifying note to the weak reference rules that field access is the upgrade point.

## 16. Parameterized resources — computed values and distinguishability — RESOLVED (already in spec + clarification)

**Resolution:** The existing spec (design.md line 1867) already handles cases 1 and 2. The distinguishability rule states that parameters are distinguishable iff both are "integer or string literals (or compile-time constants reducible to literals via const-folding)" with different values, or syntactically distinct variable names. This means:

1. **`UserDB[ids[0]]` vs `UserDB[ids[1]]`** — Both are computed expressions (array indexing), not literals or variable names. Already covered by the catch-all: conservatively same resource, serialized.
2. **`UserDB[1 + 1]` vs `UserDB[2]`** — The spec already says "compile-time constants reducible to literals via const-folding." `1 + 1` folds to `2`, so these are the same resource. Already specified.
3. **`UserDB[x]` in two branches of an `if`** — This was the only genuinely underspecified case. Resolution: effect analysis merges branches conservatively — the effect set of an `if`/`match` is the union of all arms. No path-sensitive reasoning. This is consistent with the spec's stated philosophy: "symbolic reasoning does not cross call-site boundaries" and conservative serialization is always the default.

**Spec update:** Added explicit examples for all three cases and a branch-merging rule to design.md.

## 17. Effect Variable Bounds Syntax — RESOLVED

**Conflict:** design.md said effect variable bounds "will use `where` syntax when implemented." future_decisions.md §1 showed bounds inline: `with E: no writes(_)`.

**Decision:** Inline syntax (Option A). Effect bounds are declared on the effect variable itself, not in the `where` clause:

```kara
fn safe_transform[T, with E: no writes(_)](data: T, f: Fn(T) -> T with E) -> T with E
```

**Rationale:** Inline bounds mirror how type bounds work (`[T: Ord]`), keeping the constraint next to the declaration. The `where` clause already serves two roles (type bounds and refinement predicates) — adding effect bounds would overload it further.

Both design.md and future_decisions.md §1 have been updated to agree.
