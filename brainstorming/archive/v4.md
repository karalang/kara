# Brainstorming v4 ‚Äî Open Design Questions

Issues surfaced during design review that require explicit decisions before the spec is authoritative.

---

## 1. Effect Inference on Mutual Recursion ‚Äî RESOLVED

**Resolution:** SCC-based fixed-point with effect-polymorphic SCC support. See `docs/design.md` ‚Äî "Mutual Recursion and SCCs" under Effect Inference and Boundaries.

Key decisions made:
- Tarjan's SCC decomposition + reverse topological order processing
- Within each SCC: empty-set initialization, monotone propagation, iterate to fixed point
- Public functions remain inference firewalls (mixed SCCs degenerate cleanly)
- Private SCCs parameterized by a named effect variable `[with E]` stay polymorphic until resolved at the external call site
- Compiler emits a note (not an error) for each mutual recursion group, suppressible with `#[allow(mutual_recursion_note)]`

Deferred (decided, non-breaking): effect variable bounds (`with E: no writes(X)`). See `docs/future_decisions.md`.

## 2. Implicit RC + Equality Semantics ‚Äî RESOLVED

**Resolution:** `==` always uses structural equality via `Eq`, regardless of ownership tier. RC is never observable. See `docs/design.md` ‚Äî "Equality and Mutation Semantics for Implicitly RC'd Values" under Part 4: RC Fallback.

Key decisions made:
- `==` uses `Eq` trait ‚Äî ownership tier never affects observable behavior
- `ref_eq()` is not available on implicitly RC'd values; only meaningful on `shared struct` where reference semantics are explicit
- RC and mutation are mutually exclusive ‚Äî the compiler only RC's when single ownership is impossible, which already prevents mutation through those references
- Copy-on-write was rejected: it would insert silent heap allocations, violating `allocates(Heap)` effect semantics

## 3. `collect_all` Arity and Type Design ‚úì RESOLVED

The doc describes `collect_all` as taking closures with different return types and returning "a tuple of Result values."

**Decisions:**
- **Variadic:** `collect_all` is a compiler builtin resolved via fixed-arity overloads. No general variadics needed in the type system; the compiler generates overloads for each arity internally.
- **Arity limit:** Maximum 8 branches. Documented explicitly ‚Äî beyond 8, use a collection with a loop.
- **Heterogeneous types:** Branches may return heterogeneous error types ‚Äî no unification is required. Success types may also differ freely. Return type is `(Result[A, E1], Result[B, E2], ...)` matching branch order, preserving each branch's own error type. **Updated:** Earlier resolution required a shared error type `E`; design.md relaxes this ‚Äî since all branches run to completion, the full tuple preserves the complete error trail.
- **Syntax:** Function call with closures (`|| expr`), not block form. The closure makes deferred execution explicit and is consistent with the rest of the language. Block form was rejected because it would require synthesizing anonymous struct types for named branches, adding significant complexity for no meaningful ergonomic gain.

## 4. Cross-Refinement `as` Cast and the `panics` Effect ‚úì RESOLVED

The spec says:

```
let nz: NonZero = p as NonZero;  // runtime check inserted (assert self != 0)
```

If the check fails, it panics. `panics` is a tracked effect.

**The question:** Does a cross-refinement `as` cast implicitly add `panics` to the enclosing function's inferred effect set? If yes, does that propagate to public function declarations (requiring explicit `panics` annotation)?

**Decision:** Yes ‚Äî `as` for cross-refinement propagates `panics`, same as `unwrap()`. The non-panicking path is `try_from`, which returns `Result` and carries no effect (consistent with how narrowing already works). Two forms:

- `NonZero.try_from(p)?` ‚Äî recoverable, no `panics`
- `p as NonZero` ‚Äî asserting, propagates `panics`

Future: once the static implication checker lands, `as` will require compile-time proof of implication, emit no runtime check, and carry no `panics` effect. Until then it is the "I know this holds" escape hatch, with the cost made visible through the effect system.

## 5. String Interpolation Format Specifiers ‚úì RESOLVED

**Resolution:** Display-only in v1. `{expr:spec}` syntax is reserved (compile error today). Implementation deferred to Phase 8 (Standard Library). See `docs/design.md` ‚Äî "String Interpolation" and `docs/implementation_checklist.md` ‚Äî Phase 8.

Key decisions:
- `{expr}` ‚Üí `Display.to_string()` only, always
- `{expr:spec}` is reserved syntax ‚Äî not claimed by anything else so the space is available when the feature lands
- Workaround until then: explicit method calls (`x.round_to(2)`, helper functions)
- Format specifier design (syntax, `Format` trait vs `Display`, `#[derive]` interaction) is an open sub-question for Phase 8

## 6. Effect Group Semver ‚Äî Is Adding an Effect Breaking for Callers? ‚úì RESOLVED

**Resolution:** Non-breaking (minor) is correct, but only under two explicit rules now in `docs/design.md` ‚Äî "Effect Semver Rules".

Key decisions made:
- **Group-name annotations are open-contract.** `with OrderProcessing` in a public function's declaration binds to the group as an abstraction. The annotation remains valid as the group grows ‚Äî no code change required on the caller's side.
- **Compiler-suggested fix diffs prefer group names over expanded individual effects.** This is what makes the "minor" ruling mechanically consistent: callers following the compiler's guidance get the open-contract annotation automatically.
- **Callers who expand group effects to individual annotations opt out.** Writing `with reads(UserDB), writes(OrderDB)` instead of `with Validation` is allowed but opts out of the group abstraction ‚Äî additions to the group are then breaking for those callers. Their choice.
- **Loss of auto-parallelism is accepted as minor.** Adding to a group may serialize previously parallel calls. The caller signed up for the group's full effect surface, including future growth.

The brainstorming concern (compiler flagging a missing-annotation error) only arises when callers expand groups to individual effects. The design intent ‚Äî and the compiler's suggestion behavior ‚Äî steers callers toward group-name annotations.

## 7. `String` Indexing Ergonomics ‚Äî RESOLVED

`s[i]` returns `Option[char]` at O(n) cost. The compiler warns when used inside loops. But the syntax `s[i]` looks like O(1) array access to anyone coming from any other language.

**The question:** Should `s[i]` on strings be removed in favor of an explicit `s.char_at(i)` method, making the O(n) cost impossible to miss? Or is the compiler warning sufficient?

**Decision:** Remove `s[i]` on `String` entirely ‚Äî it is a compile error. Use `s.char_at(i) -> Option[char]` instead.

*Rationale:* The compiler warning was itself an admission that the syntax was wrong. A warning can be ignored; a compile error cannot. UTF-8 is kept (UCS-4 fixed-width would buy only O(1) indexing at the cost of 4√ó memory for ASCII strings ‚Äî the one operation the design is removing). The explicit method name makes the O(n) cost impossible to miss. When repeated indexed access is genuinely needed, `let chars: Vec[char] = s.chars().collect()` gives O(1) access on a type where that cost is honest. G74 in BRAINSTORMING_V3 is superseded ‚Äî the warning mechanism is no longer needed.

## 8. Iterator Trait ‚Äî RESOLVED

The language has `for x in collection` syntax, but the trait a user type must implement to support it is not specified. Even if the full iterator standard library is deferred, the core trait signature must exist.

**Decisions:**
- Two-trait design using associated types: `Iterable` (the collection) and `Iterator` (the cursor):
  ```kara
  trait Iterable {
      type Item
      fn iter(ref self) -> impl Iterator[Item = Self.Item]
  }

  trait Iterator {
      type Item
      fn next(mut ref self) -> Option[Self.Item]
  }
  ```
  **Updated:** Earlier resolution used generic parameters (`Iterable[T]`, `Iterator[T]`). design.md now uses associated types, which encode the one-to-one relationship between a collection/cursor and its element type.
- `for x in collection` desugars to calling `.iter()` (borrowing) then `.next()` repeatedly. The collection is never consumed by a bare `for` loop.
- To consume, call an explicit method (e.g., `.into_iter()`) ‚Äî no silent moves.
- `for (key, value) in map` falls out naturally: the map's `Iterable` implementation sets `Item = (K, V)`, matched by existing tuple destructuring. No special trait machinery needed.
- These trait signatures must be locked in before the interpreter implements `for` loops, even if the full stdlib is deferred.

See `docs/design.md` ‚Äî "Loops" and "Iterator Traits" under the Type System section.

## 9. `main()` Signature and CLI Arguments ‚Äî RESOLVED

**Decisions:**
- `main()` returns either `()` or `Result[(), Error]` ‚Äî both valid. Simple scripts use `()`, programs using `?` propagation use `Result`.
- On `Result` exit with `Err`: runtime prints `Error: {message}` to stderr and exits with code 1. Format driven by `Display` impl on the error type.
- CLI args via `env.args()` stdlib function ‚Äî not a parameter to `main`, not a manually declared effect resource.
- `Env` effect resource covers both `env.args()` and `env.vars()` ‚Äî anything read from the process environment.
- `reads(Env)` propagates to `main()` through effect inference automatically. Since `main()` is not `pub`, the annotation is never written explicitly ‚Äî the compiler infers and reports it.

See `docs/syntax.md` ‚Äî "Entry Point" for the two valid `main()` signatures.

## 10. Panic Semantics ‚Äî Unwinding vs Abort ‚Äî RESOLVED

The doc says `panics` maps to "abort, unwinding" ‚Äî but these are two fundamentally different behaviors:

- **Abort**: process immediately terminates; no destructors run.
- **Unwinding**: stack unwinds; RAII destructors run (`drop` equivalents).

**The question:** Does KƒÅra support stack unwinding, or only abort? This matters for:
- Whether RAII resource handles (file descriptors, locks) are cleaned up on panic
- Whether `shared struct` RC decrements happen
- Whether `catch_panic` (like Rust's `std.panic.catch_unwind`) is a future feature

**Decision:**
- Default behavior is **stack unwinding** ‚Äî destructors run, RC decrements happen, RAII handles are cleaned up.
- Abort is available via build profile flag (`panic = "abort"`), used by `kernel`/`embedded` profiles where unwinding infrastructure is unavailable.
- Double-panics (panic while panicking) always abort.
- No `catch_panic` ‚Äî panics are unrecoverable by design. Task boundaries catching panics for structured concurrency cancellation is a runtime mechanism, not a user-facing API.

See `docs/design.md` ‚Äî effects section, `panics` paragraph.

## 11. `with_provider` Test Isolation ‚Äî RESOLVED

The doc notes that parallel tests sharing a mutable in-memory provider must use `Mutex` internally, or use `karac test --sequential`.

**The question:** Is `--sequential` sufficient, or does the language need a per-test provider scope ‚Äî where each test gets a fresh provider instance automatically?

This affects the ergonomics of writing tests for stateful providers (e.g., an in-memory database that should start empty for each test).

**Decision:** `--sequential` is insufficient as the primary answer. The problem is not parallelism but state isolation: sequential tests sharing the same provider instance still see each other's state. The fix must be per-test provider construction.

First-class `#[test]` + `with_provider` integration: when a `#[test]` function declares a provider binding, the test runner constructs a fresh instance per test and wraps the test body in `with_provider(Provider.new(), || { ... })` automatically. Proposed syntax:

```
#[test]
#[with_provider(UserDB, InMemoryUserDB.new)]
fn test_create_user() { ... }
```

`--sequential` is retained for tests that genuinely require serialization (shared ports, shared files), but is not the recommended mechanism for state isolation.

No new language feature is required ‚Äî this is a test runner convention layered on top of existing `with_provider` semantics.

## 12. `let` vs `let mut` ‚Äî Local Variable Mutability Not Formalized ‚Äî RESOLVED

design.md states an "immutability by default" principle and references `let` vs `let mut` as the mechanism (in the context of `shared struct` fields). However, the Variable Binding Rules section never formally defines whether `let` locals are immutable by default.

Every example in the repo (`two_sum`, `course_schedule`, `group_anagrams`, etc.) uses bare `let` for HashMap and Vec bindings that are immediately mutated ‚Äî suggesting the examples assume `let` is mutable. This contradicts the stated principle.

**Decision:** `let` is immutable by default. `let mut` is required to reassign a binding or call a method that takes `mut ref self`. This matches the stated "immutability by default" principle and is consistent with the `ref`/`mut ref` distinction already in the ownership system ‚Äî mutation is always opt-in, at every level.

Key rules:
- `let x = expr` ‚Äî immutable. Reassigning (`x = ...`) or calling `mut ref self` methods is a compile error.
- `let mut x = expr` ‚Äî mutable. Reassignment and mutating method calls are allowed.
- Shadowing (`let x = ...` again in the same scope) is always allowed regardless of `mut`; shadowing creates a new binding, it does not mutate the old one.
- `let mut` affects only the local binding; it has no effect on parameter mode inference. The ownership checker treats move/borrow independently.

The examples were written assuming Python semantics and have been audited and corrected ‚Äî all bindings that are reassigned or passed as `mut ref` now use `let mut`. See `docs/design.md` ‚Äî "Variable Binding Rules".

## 13. Inclusive Range Syntax (`..=`) Not Defined ‚Äî RESOLVED

design.md and all examples use exclusive ranges (`0..n`). Inclusive ranges (`0..=n`) are never defined.

Surfaced in the coin change bottom-up example: `for i in 1..=amount` and `for _ in 0..=amount`. This pattern comes up naturally in any loop that must include the endpoint ‚Äî DP tabulation, inclusive scans, `binary_search` bounds.

**Decision:** Both `..` (exclusive) and `..=` (inclusive) are supported. `lo..=hi` includes the endpoint; `lo..hi` does not. Prefer `..=` over `lo..(hi + 1)` to avoid off-by-one errors ‚Äî the `+1` workaround always works but forces the programmer to mentally adjust the bound. See `docs/design.md` ‚Äî "Loops / Range literals".

## 14. Integer Literal Type Suffixes Not Defined ‚Äî RESOLVED

design.md says integer literals default to `i64` and float literals to `f64`. No suffix syntax is defined.

Surfaced in coin change: `-1i64` used to force a type on an integer literal. Since `i64` is already the default, the suffix is redundant in that case ‚Äî but suffixes become necessary when you genuinely need a non-default type in an expression (e.g., `1i32` in a `Vec[i32]` context where inference doesn't reach).

**Decision:** Rust-style suffixes are supported: `1i8`, `1i16`, `1i32`, `1i64`, `1u8`, `1u16`, `1u32`, `1u64`, `1f32`, `1.0f64`. No suffix means `i64` for integer literals and `f64` for float literals ‚Äî the defaults are unchanged. Prefer binding annotation (`let x: i32 = 42`) when a binding is present; use a suffix when the literal appears in an expression context with no binding to annotate (e.g., `Vec.filled(n, 0i32)`). A suffix that matches the default type (e.g., `42i64`, `3.14f64`) is valid but the compiler warns ‚Äî suppressible with `#[allow(redundant_suffix)]`. See `docs/design.md` ‚Äî "Numeric Semantics / Literal suffixes".

## 15. `Vec` Initialization with a Fill Value ‚Äî RESOLVED

The standard way to pre-fill a Vec to a fixed size requires a push loop:

```
let table: Vec[i64] = Vec.new();
for _ in 0..n {
    table.push(sentinel);
}
```

This pattern comes up in every bottom-up DP, preallocated buffer, or fixed-size grid scenario. It is the KƒÅra equivalent of Rust's `vec![val; n]` or Python's `[val] * n`.

**Decision:** Two first-class constructors are added to the stdlib:

```kara
Vec.filled(n: i64, val: T) -> Vec[T] where T: Clone
Vec.from_fn(n: i64, f: Fn(i64) -> T) -> Vec[T]
```

`Vec.filled` replaces the push-loop for uniform initialization. `Vec.from_fn` covers index-dependent initialization (e.g., `Vec.from_fn(5, |i| i * 2)` ‚Üí `[0, 2, 4, 6, 8]`). `T: Clone` on `filled` allows non-`Copy` types. The binding can be `let` (not `let mut`) when no further mutation is needed after construction. See `docs/design.md` ‚Äî "Vec constructors".

## 16. Postfix Ternary and Implicit Truthiness ‚Äî RESOLVED

Python-style postfix ternary: `let v = 1 if k else 2`. Currently unsupported ‚Äî only the prefix block form exists (`if cond { a } else { b }`).

**Sub-questions:**
- Should the postfix form be added as syntactic sugar, or is the prefix form sufficient?
- Should `if` conditions support implicit truthiness (non-zero int, non-null, non-empty Option) rather than requiring an explicit `bool` expression?
- If added, what precedence level? Python's very-low-precedence approach can surprise users.

**Decision:** Both rejected.

- **Postfix ternary ‚Äî rejected.** The prefix block form `if cond { a } else { b }` already works as an expression. Postfix ternary adds a second syntax for the same thing, creates a parser ambiguity around precedence (Python's very-low-precedence approach surprises users repeatedly), and has no ergonomic advantage when the prefix form is an expression.
- **Implicit truthiness ‚Äî rejected.** `if` conditions must be `bool`. KƒÅra is explicit at every level (mutation, borrows, casts); implicit coercion of integers or `Option` to `bool` cuts against that. The meaning of `if x` is type-dependent and invisible at the call site. Explicit forms are short and unambiguous: `x != 0`, `x.is_some()`, `!items.is_empty()`.

See `docs/design.md` ‚Äî "Conditionals" and `docs/syntax.md` ‚Äî "5.4 If Expressions".
