# Brainstorming v25 — Sequence literals: default type, size inference, escape hatch

**Status: RESOLVED.** All questions answered; design.md § Collection Literals updated; ch09-collections.md rewritten. Implementation task queued in `docs/implementation_checklist.md` (Phase 4 — typechecker changes sequence literals from Array-default to Vec-default, adds prefix-literal form). This brainstorming doc is archived — do not consult again; refer to design.md and the implementation checklist instead.

## Resolutions

- **Q1 → Vec default.** Bare `[1, 2, 3]` synthesizes to `Vec[T]`. Annotation drives in check mode (`let x: Array[T, N] = [...]` works).
- **Q2 → replaced by prefix-literal form.** Size inference syntax `Array[T, _]` is unnecessary; the user-proposed `Array[1, 2, 3]` prefix form infers size from the contents without a placeholder.
- **Q3 → prefix-literal form (`TypeName[...]`) is the escape hatch.** Annotation covers binding sites; the prefix form covers expression-position cases (function args, returns). The form is generalized to all stdlib collections (`Vec[...]`, `Array[...]`, `Set[...]`, `Map[...]`), not Array-only.
- **Q4 → documented in design.md and ch09.** Typechecker change deferred to the implementation checklist.

Additional rules captured during resolution:
- Repeat form `[v; n]` works both bare and inside prefix (`Array[0; 256]`).
- Empty prefix-literal (`Array[]`, `Vec[]`) requires a type annotation.
- `Array[T, N]` uninitialized declaration (`let arr: Array[T, N];`) is valid per Kāra's definite-assignment rule; v1 DA tracks whole-value assignment only (per-slot fills need a pre-initialized array).

---

Triggered by a documentation inconsistency in `docs/book/src/ch09-collections.md`: every collection example uses inference (`Vec.new()`, `Map.new()`, tuple literals, map literals) except fixed-size `Array`, which is shown as `let data: Array[i64, 4] = [10, 40, 20, 30]`. Investigating what the spec actually mandates surfaced three gaps. A single-literal form (`[…]`) has to serve both `Vec[T]` and `Array[T, N]`, and the rules for picking between them are only half-written.

---

## Discoveries from the survey

1. **`[…]` default is not stated as a formal rule.** `docs/design.md` has precise default-type rules for scalar literals (lines 190, 207, 793: integer → `i64`, float → `f64`, string → `String`, bool → `bool`), but no analogous rule for `[…]`. The default is only implied by two example comments in the effect-system section:
   - Line 3490: `let fs = [f1, f2];  // inferred: Vec[Fn() with reads(X) writes(Y)]`
   - Line 3494: `let hs = [on_read, on_write];  // inferred: Vec[Fn() with reads(DB) writes(DB)]`
2. **Implementation disagrees with the implied rule.** `src/typechecker.rs:1825-1843` infers `ArrayLiteral` as `Type.Array { element, size: elements.len() }`. The bare `[1,2,3]` today becomes `Array[i64, 3]`, not `Vec[i64]`. The design-doc comment and the implementation contradict each other.
3. **No `Array` size-inference syntax exists.** Every `Array[T, N]` example in `design.md` spells `N` explicitly. No `Array[T, _]` placeholder is mentioned, nor is any rule that says "if the RHS literal has K elements, `N` can be omitted". The `N` is redundant when the initializer is a literal (the compiler can count), but the spec has no way to elide it.
4. **No documented escape hatch.** If the default is Vec, there's no spelled-out way to force `Array` from a bare literal other than a full `Array[T, N]` annotation. If the default is Array, same for Vec — no `Vec.from([…])` or similar appears in the Collections section. The module-level example on line 509 (`let mut SCRATCH: Array[u8, 256] = [0; 256]`) works only because of full annotation.
5. **Book chapter reflects the confusion.** `ch09-collections.md` shows `let names = ["Alice", "Bob", "Charlie"]` under the *Vec* heading (implying Vec default) and `let mut scores = [0, 0, 0, 0]` under the *Array* heading without annotation (implying Array default). Readers can't tell what the compiler does from either example.

## Q1. What should `[1, 2, 3]` default to?

Three options:

- **(a) Default to `Vec[T]`.** Matches the design-doc comment at line 3490. The common case — growable, heap-allocated, pushed into, returned from functions — gets the short syntax. `Array[T, N]` requires either a type annotation or a different constructor. Changes the current implementation.
- **(b) Default to `Array[T, N]`.** Matches current `src/typechecker.rs`. Matches Rust's `[1, 2, 3]` (which is `[i32; 3]`, a fixed-size array). `Vec[T]` requires `Vec.from([…])`, `vec[1, 2, 3]`, or similar. Changes the design doc.
- **(c) Context-dependent, no default.** The literal is polymorphic; if no context constrains it, emit an error demanding an annotation. Safer but more verbose. Would break any code today that relies on inference for `let x = [1,2,3]`.

Tradeoffs:

- (a) is what most modern systems languages do (Swift, Kotlin, Python think of `[…]` as list/dynamic). Matches the design-doc intent. Day-1 ergonomics: `let xs = [1,2,3]; xs.push(4)` just works. Cost: allocates, even for short temporaries where `Array` would be cheaper — but most uses of `[…]` in user code are genuinely sequences that grow, not fixed tables.
- (b) is what Rust does and matches current implementation. Cost: `xs.push(4)` on the default fails because Array has no `push`. Users then have to annotate every temp with `Vec[T]` or add `.to_vec()`. Day-1 ergonomics worse unless `[…]` is rarely the tool users reach for.
- (c) avoids the forced choice but adds friction at every use site. Probably wrong for a language that otherwise prioritizes inference.

**Lean:** (a). Growable-by-default matches the language's otherwise-inferred ergonomics, and (a) is what the design doc's one example already says. Array is the optimization; Vec is the norm. Confirm.

## Q2. Can `Array[T, N]` have its size inferred?

When the initializer is an array literal (`Array[i64, _] = [10, 40, 20, 30]`) or repeat form (`Array[u8, _] = [0; 256]`), the size is derivable. The spec currently requires the user to write it anyway.

Options:

- **(a) Require explicit `N`, always.** Current behavior in the examples. Redundant for literals but unambiguous.
- **(b) Allow `_` in the size slot.** `let data: Array[i64, _] = [10, 40, 20, 30]` — the compiler infers `N = 4`. Matches Rust's `let data: [i64; _] = [...];` (which Rust has as an unstable feature).
- **(c) Allow the size slot to be omitted entirely.** `let data: Array[i64] = [10, 40, 20, 30]` — the annotation disambiguates Array-vs-Vec, and the size comes from the literal. Shorter than (b).
- **(d) Allow both a bare `Array[T]` annotation as an Array escape hatch AND `Array[T, N]` where `N` is required to match.** This is (c) plus making `Array[T, N]` still mandatory when the user wants to enforce a specific size.

Tradeoffs:

- (a) is the least magical. Users who care about the size have to write it. Cost: stutter when the literal makes the size obvious.
- (b) is a precise escape hatch. Keeps `Array[T, N]` as the canonical form and only allows `_` when the size is derivable. Users reading the code see `_` and know "inferred from RHS".
- (c) is the shortest but collapses two concepts — "disambiguate from Vec" and "infer the size". The resulting `Array[T]` annotation looks like a generic type name that's missing a parameter.
- (d) combines clarity of (b) with shorthand of (c). Probably the fullest answer but has the most surface.

**Lean:** (b). `_` as "infer this" is already the convention for type holes in other languages and keeps `Array[T, N]` as the canonical form. Confirm.

## Q3. Escape hatch for the non-default

If Q1 resolves to `Vec` default (option a), users need a way to get an `Array` from a bare literal. Three candidates:

- **(a) Full annotation is the only escape hatch.** `let data: Array[i64, 4] = [10, 40, 20, 30]` or (per Q2) `Array[i64, _] = [...]`. No constructor needed.
- **(b) `Array.from([…])` constructor.** Mirrors `Vec.from` elsewhere in the stdlib. Usable in expression position (arguments, returns) without a binding annotation.
- **(c) Both.** Annotation is the common case; constructor is available for expression contexts where annotation can't reach.

And symmetrically, if Q1 resolves to `Array` default (option b), users need a way to get a `Vec` from a bare literal — same three options, applied to Vec.

**Lean:** (c), paired with Q1(a). Annotation covers 95% of cases; the constructor covers expression-context needs (`some_fn(Array.from([1,2,3]))`). The constructor already fits how `Vec.from_fn` and `Vec.filled` work in the Collections section. Confirm.

## Q4. What changes when we resolve this?

- **`docs/design.md` § Standard Data Structures** — add a precise rule for `[…]` default type, add the size-inference syntax (per Q2), add the escape-hatch constructor (per Q3).
- **`docs/book/src/ch09-collections.md`** — rewrite examples consistent with the resolved default; remove the inconsistency between Vec `let names = [...]` and Array `let mut scores = [...]` examples.
- **`src/typechecker.rs:1825-1843`** — `ExprKind.ArrayLiteral` currently produces `Type.Array` unconditionally. If Q1(a) wins, this needs to produce `Vec[T]` by default and be coerced to `Array[T, N]` only under a context type. If Q1(b) wins, leave as-is but document it.
- **`src/parser.rs`** — if Q2(b) allows `Array[T, _]`, the parser may need to accept `_` in type-argument position. Check current behavior.
- **Tests** — `tests/typechecker.rs` likely has tests locking in the current Array-default behavior. If Q1 flips, these change.
