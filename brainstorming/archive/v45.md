## Why S1b won

The decision lens from v45 (compiler-verifiable robustness + LLM-friendliness primary; human ergonomics secondary) points at S1b over S1c/S1d because:

- **Effect system earns its keep.** Kāra already carries `with _` ceilings (design.md:292, 2074, 2086). S1b is *using existing machinery*, not inventing a parallel hierarchy. S1c/S1d reintroduce the color problem the effect system was designed to dissolve.
- **One combinator library.** `map` / `filter` / `take` / `fold` work identically over `Vec`, `io.lines()`, or a Kafka consumer. No Iterator-vs-Stream style debates, no conversions at boundaries, no library fragmentation.
- **Mechanical refactorability.** Swapping a source from `Vec` to `io.lines()` changes the effect row at the public boundary and nothing else in the chain.
- **Distinct design payoff.** No mainstream language has tried first-class-effect-system-based unification of iteration. S1b is the option where Kāra's effect system produces a visibly different answer from Rust / Kotlin / JS / Python / Swift.

S1a (do nothing) was viable but locked out fluent combinators on effectful sources — the exact pain point findings.md Brick #4 flagged.

## Why S3c (not S3b)

`Option[Item]` + errors-in-`Item` is already what Rust's `std::io::Lines` and Kāra's Brick #4 prototype do, successfully. A three-way `StreamEvent` enum (S3b) would introduce a new termination shape just for effectful iteration — redundant given `Result` composes cleanly inside `Item`, and risks fragmenting combinator libraries (terminal adaptors would need to handle both `Option` and `StreamEvent` shapes).

## Why S4 deferred

Naming and exact signatures for `retry_on` / `skip_errors` / `take_until_error` depend on the first real use case hitting stdlib. Designing them speculatively now would lock in shape choices that the first real consumer might reject.

## Original framing (for record)

**Framing.** `design_studies/event_stream/findings.md` (Brick #4) compared stdin line-streaming across Java/Python/Rust/Kāra. The observation that prompted this doc:

> **Parity with Rust on the in-function shape.** Neither language has a special "stream" type; both use loop + read.

The project owner asked: streams are a common usage pattern (stdin, file lines, Kafka, SSE, log tailing, sensor data, HTTP chunked bodies). Should Kāra add a dedicated `Stream` type, or is the current `loop` + `io.read_line()` + effect-tagged signature enough?

This is a **language-design** brainstorm, not an implementation one. No code in `src/` changes until this resolves. `docs/design.md` lines 1486–1565 (Iterator traits + adaptors) are the current spec.

**Ground rule (`memory/feedback_no_design_compromise.md`):** decide on Kāra's own merits. "Rust has Stream" / "Kotlin has Flow" / "Java has Flow" are surfacing signals, not deciding signals. The fact that every major language ended up with *some* push-source abstraction is evidence of real ergonomic pressure — which IS a Kāra-merits factor — but the resolution defends on Kāra grounds.

## Decision lens — what we're optimizing for

Per `memory/feedback_design_weighting_llm_era.md`, **Kāra optimizes for an LLM-first authoring world.** Priorities carried forward from v44:

### 1. Compiler-verifiable robustness *(primary)*
Can the compiler catch "forgot to handle EOF", "blocked in an async context", "swallowed a fatal I/O error while iterating", "leaked a file handle on exit"? Every such error caught at compile time is one a model doesn't have to debug at runtime.

### 2. LLM-friendliness *(primary)*
- **Local context completeness** — a combinator chain is legible from the chain alone; no need to jump to trait definitions.
- **Unambiguous syntax** — one name for "thing I pull items from", not `Iterator` vs. `Stream` vs. `Flow` depending on sync/async.
- **Compiler-pinpointable errors** — effect mismatches (`suspends` expected, `blocks` supplied) must localize precisely.
- **Composability without surprises** — map/filter/take_while/fold compose identically regardless of source (in-memory vec, stdin, network).
- **Mechanical refactorability** — swapping source from "stdin lines" to "Kafka messages" should change type and effect row, not shape.

### 3. Human ergonomics *(secondary)*
The *color problem* (sync vs. async function duplication) is a documented human pain point in Rust, JS, and Python. Carries weight here but doesn't override robustness.

## Terminology note — "push" vs "pull"

Throughout: **pull-model** means the consumer calls `next()` / `read_line()` when ready for the next item. **Push-model** means items arrive regardless of whether the consumer is ready (e.g., signal handlers, reactive callbacks, backpressured channels).

In practice, most so-called "streams" in Rust / Kotlin Flow / fs2 / futures are *pull-model* — `poll_next()` / `collect()` still ask for the next item. Kāra's existing `Iterator.next()` is pull-model. Truly push-model sources (hardware interrupts, reactive push) are out of scope for this brainstorm — they need a fundamentally different abstraction (channels, actor inboxes) already covered by `sends`/`receives` effects and `effect resource Channel[T]`.

**This brainstorm is about pull-model iteration over async/blocking/remote sources** — the Rust `Stream` / Kotlin `Flow` / fs2 `Stream` equivalent, not the push/reactive world.

## Context — what's currently specified (design.md)

**Iterator surface (1486–1504):**
```kara
trait Iterable    { type Item; fn iter(ref self) -> impl Iterator[Item = Self.Item] }
trait Iterator    { type Item; fn next(mut ref self) -> Option[Self.Item] }
trait IntoIterator{ type Item; fn into_iter(self) -> impl Iterator[Item = Self.Item] }
```

**`next()` has no `with _` ceiling** (design.md:1498). Per the rule at design.md:2074 ("trait methods without `with _` require pure impls"), **implementations of `Iterator` must currently be effect-free**. That means:

- ✅ `Vec.iter()` — pure allocation-free iteration, fine.
- ❌ `StdinLines` that calls `io.read_line()` inside `next()` — **rejected today** because `next()` has no effect ceiling.
- ❌ `KafkaConsumer` yielding `receives(Kafka), suspends` — rejected for the same reason.

**Adaptors ARE effect-polymorphic** (design.md:1524):
> All closure-accepting adaptors are effect-polymorphic — the adaptor's effect set includes the closure's effects.

So `.filter(pred)` / `.map(f)` propagate the *closure's* effects, but not the *source iterator's* effects. This asymmetry is the core gap.

**Loop effects are static** (design.md:1518): `while let Some(x) = it.next()` contributes `it`'s effects to the enclosing function exactly once. This rule is already correct for effectful sources — the missing piece is just the trait signature change.

## Context — what mainstream conventions look like

| Language | Sync iteration | Async/blocking source | How the split is spelled |
|---|---|---|---|
| Rust | `Iterator` | `futures::Stream` | Two separate traits; `async` color; `tokio-stream` combinators shadow `Iterator` combinators |
| Java | `Iterator` / `Stream` | `Flow.Publisher` (reactive streams) | Two hierarchies; reactive lib for async |
| Kotlin | `Sequence` | `Flow` | Two abstractions; Flow uses `suspend` |
| Scala | `Iterator` | `fs2.Stream` / `Akka Source` | Community libs for async |
| Python | iterator protocol | async iterator (`async for`) | Two protocols (`__iter__` / `__aiter__`); `async` color |
| Swift | `Sequence` / `IteratorProtocol` | `AsyncSequence` | Two protocols; `async` color |
| Go | `for range` | channels + `<-` | Channels subsume the "async source" case; no second iterator type |
| JS/TS | `Iterator` / `Iterable` | `AsyncIterator` / `AsyncIterable` | Two protocols; `async` color |

**Pattern:** every mainstream language except Go ended up with *two* iteration abstractions — a sync one and an async one — because the underlying function-color split (sync `next()` vs. async `next()`) forces it. Go dodges the split by using channels for anything async, but that trades off composability (no fluent `.map().filter()` on channels).

**Kāra's angle:** the effect system already tracks `blocks` / `suspends` / `reads(X)` / `receives(Y)` at the signature level. A single `Iterator` trait with effect-polymorphic `next()` can *in principle* subsume all four quadrants (sync/blocking/async/remote) under one abstraction — something no mainstream language has tried because none of them have a first-class effect system to lean on.

## Scope of this brainstorm

In scope:
- Whether `Iterator.next()` should gain a `with _` effect ceiling.
- Whether to add a separate `Stream` type, or unify under `Iterator`.
- How combinators (`map`, `filter`, `take_while`, `fold`, etc.) propagate source effects.
- Error-strategy combinators for per-item failures (retry / skip / abort).
- EOF/termination shape (`Option.None` vs. a dedicated `StreamEvent` enum).

Out of scope:
- **Push-model reactive streams** — covered by channels + `sends`/`receives`.
- **Backpressure primitives** — live in `effect resource Channel[T]` land, not Iterator.
- **`async`/`await` syntax** — Kāra uses `suspends` at the signature level; no function-color markers.
- **Parallel iterators** (`par_iter`) — separate concurrency-model brainstorm, orthogonal to this question.
- **Broadcast / fan-out / fan-in** — structural patterns around iterators, not the iterator trait itself.

## Group S — Stream shape

### S1. The central question: one trait or two?  *(open)*

**Options:**

- **S1a. Leave as-is.** `Iterator.next()` stays pure. Effectful sources remain free functions (`io.read_line()`), consumed via `loop { match ... }`. Combinators (`map`, `filter`) apply only to pure-source iterators.
  - Pro: zero language change; findings.md says the `loop + match` shape is already competitive (≈20 lines, Kāra second-shortest after Python).
  - Pro: preserves the simplest possible Iterator semantics.
  - Con: **no fluent combinators on effectful sources** — you can't write `io.lines().filter(...).take(10)` because `io.lines()` would need to return something, and that something can't satisfy `Iterator`.
  - Con: users who reach for Iterator combinators on streams have to hand-roll the loop and re-implement map/filter inline — the exact pain point Java/Python/Rust solved with Stream/Flow/iterator.
  - Con: as soon as one real use case (Kafka consumer, SSE parser, log tailer) lands in stdlib, this answer becomes untenable and we're back here.

- **S1b. One trait, effect-polymorphic.** Add `with _` to `Iterator.next()`:
  ```kara
  trait Iterator {
      type Item
      fn next(mut ref self) -> Option[Self.Item] with _
  }
  ```
  A single `Iterator` trait covers sync/blocking/async/remote. Combinators propagate both closure effects and source effects.
  - Pro: **no color problem.** One name, one trait, one combinator library. Fundamentally different from Rust's Iterator/Stream split.
  - Pro: fits Kāra's existing effect system cleanly — `with _` is already the mechanism (design.md:292, 2074, 2086). This is using existing machinery, not inventing new.
  - Pro: findings.md's observation that "composability via effect-typed boundaries rather than generic iterator traits" becomes the *implementation strategy*, not just an observation.
  - Pro: matches Go's philosophical simplicity ("just one iteration concept") while preserving fluent combinators Go lacks.
  - Con: every adaptor signature gains a source-effect variable slot (small bloat in the trait docs, invisible at call sites).
  - Con: requires stating how source-effect and closure-effect compose in chained adaptors. Not novel — design.md:1524 already does this for closures; extending it to sources is additive.
  - Con: **no separate type signal for "this might block/suspend."** A caller reads `impl Iterator[String]` and has to look at the effect row to tell. Mitigated by Kāra's effect-display tools (`karac explain`, effect summaries in build output) already present for other signatures.

- **S1c. Add a separate `Stream` type.** Follow Rust/Kotlin/JS: `Iterator` stays pure, `Stream` is the effectful sibling.
  ```kara
  trait Stream {
      type Item
      fn next(mut ref self) -> Option[Self.Item] with _
  }
  ```
  - Pro: visual distinction at the type level ("this is a stream, expect effects").
  - Pro: matches Rust's shape precisely for developers porting Rust code (*note: per design principle this is a weak argument — surfacing signal only*).
  - Con: **reintroduces the color problem Kāra's effect system was supposed to dissolve.** Every combinator written twice; libraries pick a side; style debates about which to return.
  - Con: conversions (`Iterator → Stream`, `Stream → Iterator`) needed at every boundary; each conversion is a friction point.
  - Con: two trait hierarchies fragment the stdlib — which one gets `enumerate` / `zip` / `take` / `collect`?
  - Con: undercuts the distinctive "Kāra's effect system replaces the sync/async split" story that *is* the design principle's payoff here.

- **S1d. Hybrid — async Stream, sync Iterator.** Iterator stays pure; Stream is specifically for `suspends` sources; blocking sources (`blocks`, `reads(Stdin)`) stay on Iterator.
  - Pro: preserves the intuition "blocking is just iteration you happen to do slowly."
  - Con: three categories now (pure Iterator, blocking Iterator, suspending Stream) with fuzzy boundaries. Where does a Kafka consumer in a blocking mode live? Where does a file-reader that sometimes blocks and sometimes doesn't live? Answer: wherever someone arbitrarily draws the line, and that line moves.
  - Con: all of S1c's cons, plus a new "which effect puts you in which hierarchy" rule to remember.
  - Con: Kāra's effect lattice doesn't elsewhere distinguish `blocks` and `suspends` at the type level — they're peer execution verbs. Introducing a type-level split here would be inconsistent.

**Provisional lean: S1b.** It's the option that does the most work with the least new machinery, and it's the only option where Kāra actually *demonstrates* a design payoff from having a first-class effect system — every other option is reinventing something existing languages already have. S1a is viable if the payoff is considered too speculative; S1c/S1d are strictly worse than S1b in the LLM-era lens.

### S2. Effect composition in adaptor chains  *(depends on S1)*

If S1b: adaptors gain a source-effect variable. The signature for `map` currently reads (design.md:1528):

```kara
fn map[U, with E](self, f: Fn(Self.Item) -> U with E)
    -> impl Iterator[Item = U] with E
```

Under S1b it would become:

```kara
fn map[U, with E_src, with E_fn](self: impl Iterator[Item=Self.Item] with E_src,
                                 f: Fn(Self.Item) -> U with E_fn)
    -> impl Iterator[Item = U] with E_src ∪ E_fn
```

Where `E_src` is inherited from the source iterator's `next()` and `E_fn` from the closure. The result is union of both. Design.md:1524 already states the union rule for closure effects across chained adaptors; this extends the same rule across source boundaries.

**Alternative (simpler signature):** hide `E_src` as an implicit effect row on `self`. Users never write it; the compiler threads it through. Preferred — keeps combinator signatures at their current readability.

**Open sub-question:** does the loop-level static-effects rule (design.md:1518) need updating? Current wording says "a while-let loop over a network stream has the same effect signature as a single read." That already implicitly covers S1b — the source's effects flow through `next()` into the loop body's effect set, counted once. **Verdict:** no change needed; the rule was written in anticipation of effectful sources.

### S3. EOF / termination shape  *(depends on S1)*

Currently `Iterator.next()` returns `Option[Item]` — `None` signals end. For effectful sources, termination is richer:

- Fatal: connection dropped, file handle invalidated — not the same as "no more items right now."
- Transient: single-item parse error — should the stream end or skip?
- Graceful: EOF reached cleanly.

**Options:**
- **S3a. Keep `Option[Item]`.** Fatal/transient errors surface via the effect row (`panics`) or via `Result` in `Item` (i.e., `Iterator[Result[T, E]]`). Simple; matches current Rust `std::io::Lines` shape.
- **S3b. Three-way `StreamEvent[T, E]`.** `next()` returns `Continue(T)` / `End` / `Fatal(E)`. Richer, but introduces a new enum just for streaming.
- **S3c. Per-iterator choice.** Sources pick their own `Item` type — `Iterator[Result[String, IoError]]` for a fallible source, `Iterator[String]` for a clean one. No new language machinery.

**Provisional lean: S3c** (already the Rust convention, already what findings.md/Brick #4 showed working). Defer S3b unless a concrete use case demands it.

### S4. Error-strategy combinators  *(new, depends on S1b)*

If streams-as-iterators is the shape, the stdlib needs combinators that address per-item robustness — the exact property findings.md called out as distinct from sync iteration:

```kara
fn retry_on[E, P: Fn(ref E) -> bool](self, pred: P, max: i64)
    -> impl Iterator[Item = Self.Item]

fn skip_errors[T, E](self: impl Iterator[Item = Result[T, E]])
    -> impl Iterator[Item = T]

fn take_until_error[T, E](self: impl Iterator[Item = Result[T, E]])
    -> impl Iterator[Item = T]
```

Not novel — every stream library has equivalents. What's novel is that under S1b, these compose with map/filter/take without a second library.

**Scope note:** don't design the full combinator library in this brainstorm. List the category as in-scope and defer naming / exact signatures to implementation-phase design.

### S5. Effect row visibility at call sites  *(open, cross-cutting)*

If `impl Iterator[String] for StdinLines` inherits `reads(Stdin), blocks` via `next()`, a function consuming it must declare those effects (public) or inherit them (private). Two risks:

- **Risk:** users think "this is just an Iterator" and are surprised when `reads(Stdin)` shows up in their function's inferred effects.
- **Risk:** mechanical refactors (inline a loop that uses `stdin.lines()`) change the enclosing function's effect row, possibly breaking public signatures.

**Mitigations:**
- Naming convention for effectful Iterator-producing functions (`io.lines()`, `net.messages()`, `fs.walk_entries()`) — the module prefix signals the effect.
- `karac explain` already traces effect provenance; this use case is a natural fit.
- The `public_effects = "declared"` default already forces awareness at public boundaries.

**Provisional verdict:** no new mechanism needed. Existing effect-display infrastructure covers this.

## Related questions (raised, not resolved here)

These surfaced while writing this brainstorm. Each is a separate design question that should NOT hold up S1.

- **Should `std.json.Json` be prelude-adjacent?** From the conversation that led here. If Iterator-based stream parsing (`io.lines().map(|l| Json.deserialize[Event](l))`) becomes the canonical shape, the ergonomics of `Json` access become more visible.
- **Does `fs.read_lines(path)` return `Iterator<String>` or `Result<Iterator<String>, IoError>`?** The outer `Result` catches "file missing" at open time; the inner `next()` result handles per-line errors. Convention, not novelty.
- **Closure-over-iterator ownership.** If `let it = source.filter(|x| x.valid);` and `filter` borrows `source`, then lifetime and ownership of `source` vs. `it` needs a rule. Design.md:1516 already says `for` borrows; adaptors need the same treatment stated.

## Provisional implication

**Lean (unresolved): adopt S1b + S3c + defer S4 combinators to a later implementation pass.**

- S1b: `Iterator.next() with _` — one trait, effect-polymorphic. Demonstrates that Kāra's effect system replaces the sync/async split other languages live with. Minimal language change; uses existing `with _` machinery.
- S3c: per-iterator error-in-Item convention. Zero language change.
- S4: error-strategy combinators — acknowledge the category, defer naming/signatures.

**What this means concretely:**

1. **design.md:1498** — add `with _` to `Iterator.next()`.
2. **design.md:1524** — extend the effect-union rule to cover source effects in addition to closure effects.
3. **design.md:1518** — confirm the static-effects rule already covers this case (no change; just cross-reference).
4. **Stdlib plan** — `io.lines()`, `fs.read_lines(path)`, and future channel/consumer adaptors return `impl Iterator[Item] with <appropriate effects>`. No separate `Stream` module.

**What to watch:**

- First non-trivial Kāra program that chains `io.lines().filter(...).map(...).take(n)`. If the effect annotations read cleanly, S1b is validated. If users hit confusion about why their `main()`'s inferred effects include `reads(Stdin)`, revisit S5's mitigations.
- First effectful trait method implementation. S1b implies `next()` with `with _` is the template for all effect-polymorphic trait methods; make sure the teaching docs (`docs/book/`) present it clearly the first time this shape appears.
- Moment a push-model (reactive) source lands in stdlib. That is explicitly out of scope for this brainstorm — don't let it leak in and force reopening.

**Resolution path:** once S1 converges, migrate resolved bullets to `docs/design.md` (Iterator section) and `docs/implementation_checklist.md`; mark this doc RESOLVED and archive per `memory/feedback_brainstorming_close_for_good.md`.
