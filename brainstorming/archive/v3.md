## G61 — `karac query concurrency` Output Schema

**Problem:** `karac query concurrency <fn>` is mentioned in design.md (Compiler Query API section) as returning "parallelization decisions" as JSON, but the schema is unspecified. An LLM cannot verify or explain the compiler's concurrency decisions without knowing what fields to read.

**What the compiler knows and should expose:**

The compiler builds an internal dependency graph from two analyses:
1. **Data dependency edges** — call B uses return value of call A → must be sequential.
2. **Effect conflict edges** — call A and call B conflict on the same resource → must be sequential.

Everything else runs in parallel. The output schema should make both analyses visible so an LLM (or human) can independently verify each parallelization decision.

**Options:**

**Option A — Flat list of calls with status.**
Each call in the function body gets a `"parallel": true/false` flag and a `"reason"` string. Simple, but loses the grouping structure — doesn't show *which* calls run together.

```json
{
  "function": "load_dashboard",
  "calls": [
    {"call": "fetch_profile(user_id)", "parallel": true, "group": 0},
    {"call": "fetch_orders(user_id)",  "parallel": true, "group": 0},
    {"call": "fetch_notifs(user_id)",  "parallel": true, "group": 0},
    {"call": "build_dashboard(...)",   "parallel": false, "reason": "data_dependency", "depends_on": ["fetch_profile", "fetch_orders", "fetch_notifs"]}
  ]
}
```

**Option B — Parallel groups with dependency edges (graph shape).**
Output is a sequence of stages. Each stage is either a single call or a parallel group. Between stages, dependency edges are listed with their reason.

```json
{
  "function": "load_dashboard",
  "stages": [
    {
      "type": "parallel_group",
      "calls": [
        {"call": "fetch_profile(user_id)", "line": 2, "effects": ["reads(UserDB)"]},
        {"call": "fetch_orders(user_id)",  "line": 3, "effects": ["reads(OrderDB)"]},
        {"call": "fetch_notifs(user_id)",  "line": 4, "effects": ["reads(NotifDB)"]}
      ],
      "why_parallel": "no data dependencies between calls; no effect conflicts (different resources, all reads)"
    },
    {
      "type": "sync_point",
      "reason": "build_dashboard takes return values of all three calls"
    },
    {
      "type": "sequential",
      "call": "build_dashboard(profile, orders, notifications)",
      "line": 7,
      "effects": []
    }
  ],
  "serialized_pairs": []
}
```

For a function with serialization:
```json
{
  "function": "update_then_fetch",
  "stages": [
    {"type": "sequential", "call": "save_user(user)",       "line": 2, "effects": ["writes(UserDB)"]},
    {"type": "sequential", "call": "fetch_user(user.id)",   "line": 3, "effects": ["reads(UserDB)"]}
  ],
  "serialized_pairs": [
    {
      "a": "save_user(user)",
      "b": "fetch_user(user.id)",
      "reason": "effect_conflict",
      "conflict": "writes(UserDB) vs reads(UserDB)"
    }
  ]
}
```

**Option C — Option B + counterfactual field.**
Same as B, but each serialized pair also includes a `"would_parallelize_if"` field — what the programmer would need to declare (e.g., `independent` annotation or parameterized resources) to allow parallelization. This is the most LLM-useful form: not just what happened, but what the path to optimization looks like.

```json
"serialized_pairs": [
  {
    "a": "save_user(user)",
    "b": "fetch_user(user.id)",
    "reason": "effect_conflict",
    "conflict": "writes(UserDB) vs reads(UserDB)",
    "would_parallelize_if": "no path — read-after-write ordering is semantically required here"
  }
]
```

Versus a case that *could* be parallelized:
```json
"serialized_pairs": [
  {
    "a": "write_audit_log(event)",
    "b": "update_record(id, data)",
    "reason": "effect_conflict",
    "conflict": "writes(AuditDB) vs writes(UserDB)",
    "would_parallelize_if": "declare resources as independent: `independent AuditDB, UserDB;` — safe if audit and user tables are on separate connections"
  }
]
```

**Decision:** Option C.

Option B is the minimum useful schema — it exposes the full graph structure and makes every serialization reason explicit. Option C adds `"would_parallelize_if"` which turns the query output into an actionable optimization hint. This fits the AI-first design goal: an LLM can read the output, verify each decision, and suggest improvements without needing to re-analyze the source. The counterfactual field is nullable (`null` when the serialization is semantically required) so it adds no noise when there's nothing to suggest.

The `karac query concurrency` output is read-only analysis — it does not modify source. The `"would_parallelize_if"` field is a hint string, not a machine-applicable diff, because the decision to declare `independent` resources has semantic implications the programmer must verify. Effect fix diffs (in `karac build --output=json`) remain the place for mechanical, auto-applicable changes.

**design.md changes:**
- Expand the `karac query concurrency` entry in the Compiler Query API section with the Option C schema and examples.
- Add a note that `"serialized_pairs[].would_parallelize_if"` is `null` when the ordering is semantically required (read-after-write, data dependency forced sequencing).

## G62 — Testing Section Claims "Four" File Conventions but Lists Three

**Problem:** The Testing section opens with "Four file conventions, four commands. `karac build` ignores all of them:" but the table only has three rows: `_test.kara`, `_fuzz.kara`, `_bench.kara`. Either the count is wrong, or a fourth convention was dropped at some point and the intro was not updated.

**Options:**

**Option A — The count is simply wrong; update to "three".**
The three listed conventions cover the space cleanly (correctness tests, fuzz, benchmarks). No fourth convention is needed.

**Option B — A fourth convention exists and was accidentally omitted.**
Candidate: integration/end-to-end tests. Could be `_e2e.kara` or `_integration.kara` with a `karac test --integration` command. These run against real external services (databases, APIs) and are kept separate from unit tests. This is a real pattern (Rust has `tests/` vs `src/`), but Kāra already handles module-private access via same-directory `_test.kara` — a separate e2e convention could be useful.

**Decision:** Option A + extension. Fix the count to three. Add `#[test(requires = [...])]` attribute in `_test.kara` for tests that need live external services. `karac test` skips tests with unsatisfied `requires`; `karac test --all` runs everything. The `requires` list names effect resources — consistent with the effect system already in place. No fourth file convention needed: a file-level split forces a hard unit/e2e binary, while `requires` is finer-grained and composes with the effect system.

**design.md changes:** Fixed "Four" → "Three" in the intro sentence; added `#[test(requires = ...)]` documentation block after the conventions table.

## G64 — Borrowed Struct Field (`ref` in Struct) Has No Specified Mechanism

**Problem:** The Tiered Ownership section shows this example:

```
struct Parser {
    source: ref String,
    position: i64,
}
```

The doc says "No named lifetime parameters." But a struct field that is a borrow (`ref String` rather than an owned `String`) needs some mechanism to express its validity constraint — the `Parser` cannot outlive the `String` it borrows from. In Rust this requires `'a`. The doc does not explain how this works without lifetime parameters.

This may be an unresolved design gap, a holdover from an earlier draft, or evidence that `ref` in struct fields requires a special rule not yet written.

**Options:**

**Option A — Forbid `ref` in struct fields entirely.**
The example is a mistake. Structs own their data. If a parser needs a string view, it owns a `StringSlice` (which is a borrow-like type already defined) or an owned `String`. The `Parser` example could be rewritten as `source: StringSlice` (borrowed) or `source: String` (owned).

**Option B — Allow `ref` struct fields with implicit single-lifetime rule.**
A struct with any `ref` field is treated as having an implicit single lifetime — it cannot outlive the values it borrows. No named parameter, just a structural constraint the compiler enforces. The programmer writes `ref String`; the compiler treats the struct as "borrowed from its construction context." This is a weaker form of Rust lifetime elision.

**Option C — Allow `ref` struct fields and require explicit `ref(param)` syntax at construction.**
Similar to the `ref(a, b)` return annotation — construction syntax would bind the struct's borrow to a specific source. This is expressive but adds complexity and is close to Rust's approach.

**Decision:** Option B. A struct with any `ref` field is a *borrowed struct*. Its scope is bounded by the scope of every value its `ref` fields borrow from — the compiler infers this at construction and enforces it at every assignment and return site. No lifetime annotation is written by the programmer.

Returning a borrowed struct from a function is valid only if every borrowed source is a parameter; the return receives an implicit `ref(param)` constraint, exactly mirroring the existing `ref(a, b)` rule for borrow returns. This is a natural extension of the already-specified borrow-return mechanism — no new syntax, no new concept, just the same rule applied to structs.

The `Parser` example in design.md is correct as written and is now semantically backed by this rule.

## G66 — `split_by_variant` Interaction with `group` Blocks in Layout

**Problem:** Layout blocks support two mechanisms — `group { ... }` for struct field grouping and `split_by_variant` for enum variants. The spec shows them used separately but never specifies their interaction rules. Can you write:

```
layout shapes: Vec[Shape] {
    split_by_variant
    group some_field { ... }
}
```

It is also unspecified what happens when an enum has zero or one variant.

**Options:**

**Option A — Mutually exclusive.** A layout block may contain either `split_by_variant` or `group` blocks, not both. Mixing is a compile error. Rationale: `split_by_variant` reorganizes storage by variant boundary; `group` reorganizes by field proximity — the two axes conflict in a well-defined way.

**Option B — `split_by_variant` first, then `group` applies within each variant's fields.** `split_by_variant` determines the top-level segmentation; `group` blocks then apply inside each variant, controlling which fields within a variant are stored together. Rationale: composable and expressive, but adds complexity to the layout compiler.

**Option C — `split_by_variant` is the only layout directive allowed for enum types; `group` is struct-only.** Simpler mental model: enums use `split_by_variant`, structs use `group`. Zero-or-one variant enums: zero variants → no-op (empty layout); one variant → equivalent to a struct, `group` allowed.

**Decision:** Option C.

`split_by_variant` is enum-only; `group` is struct-only. The type being laid out determines which directive is valid — the compiler rejects mixing at semantic analysis time. This gives a clear, teachable mental model: enums use `split_by_variant`, structs use `group`.

Special cases: a single-variant enum is structurally equivalent to a struct, so `group` is allowed and `split_by_variant` is a no-op — the compiler accepts either. A zero-variant enum layout block is a no-op (uninhabited types have no instances).

Option B's composability appeal breaks down on heterogeneous variants: `group hot { center, radius }` at the top level of a multi-variant enum layout is ambiguous because `center` does not exist in every variant. Expressing per-variant field grouping would require variant-scoped syntax (`variant Circle { group hot { ... } }`), which is a new feature not covered by the current flat `LAYOUT_ITEM` grammar. Option C does not preclude adding that syntax later — it simply defers it rather than leaving the flat-syntax interaction undefined.

**design.md changes:** Expanded the Enums bullet in Layout Rules to state the type-directed rule, the single-variant and zero-variant special cases, and a note that per-variant field grouping is deferred.

## G67 — Provider Trait Enforcement Point and `with_provider` Signature

**Problem:** `design.md` states provider traits "require `Send + Sync`" and that `with_provider` wraps in `Arc` internally (added via G56). But neither the enforcement point nor the full signature of `with_provider` is specified. Two questions:

1. Is `Send + Sync` enforced at the `effect resource` declaration site, or at the `with_provider` call site?
2. Can you declare multiple provider trait bounds? `effect resource X: Trait1 + Trait2;`
3. What is the complete type signature of `with_provider`?

**Options:**

**Option A — Enforce at declaration; single trait bound only.** `effect resource UserDB: DatabaseProvider;` requires `DatabaseProvider: Send + Sync` or the declaration is a compile error. Multiple bounds disallowed — define a supertrait instead. `with_provider[R: effect resource, P: R.Provider](provider: P, f: Fn() -> T with R) -> T`.

**Option B — Enforce at call site; multiple bounds allowed.** The resource declaration is unconstrained; `with_provider` enforces `Send + Sync` on the concrete provider at the call site. `effect resource X: Trait1 + Trait2;` is valid, meaning any provider must implement both. Signature: `with_provider[R: effect resource, P: R.Provider + Send + Sync](provider: P, f: Fn() -> T with R) -> T`.

**Option C — Same as B but `Send + Sync` is implicit on all provider traits.** The compiler automatically adds `Send + Sync` bounds to any trait used as a provider — programmers never need to write them. A lint warns if a user manually writes `trait Foo: Send + Sync` for a provider trait (redundant).

**Decision:** Option B.

`Send + Sync` is enforced at the `with_provider` call site on the concrete provider type. Provider traits carry no implicit thread-safety bound — the constraint is visible in `with_provider`'s signature and reported there when violated. This matches how Rust handles thread-safety (e.g. `Arc<T>` requires `T: Send + Sync` at use, not at definition) and avoids over-constraining traits that may be used in single-threaded test contexts.

Multiple trait bounds are allowed: `effect resource UserDB: Trait1 + Trait2;` is valid. The `with_provider` call site then requires `P: Trait1 + Trait2 + Send + Sync`.

Option C's implicit bound injection was rejected: the compiler silently modifying a trait's effective bound produces confusing error messages (no visible bound at the error site) and penalizes programmers for being explicit via a redundancy lint.

Test providers using `Cell` or `Rc` internally cannot satisfy `Send + Sync` and therefore cannot be passed to `with_provider` directly. The workaround is `Mutex`-wrapping the mutable state, or running tests with `karac test --sequential` where the `Sync` bound is trivially satisfied.

**design.md changes:** Removed `Send + Sync` from the `DatabaseProvider` trait definition; added multiple-bounds example; added explicit `with_provider` signature; updated the concurrency semantics paragraph to describe call-site enforcement.

## G68 — Effect Polymorphism (`with _`) in Generic Trait Contexts

**Problem:** Effect polymorphism via `with _` is specified for closures and standalone functions. But when a trait method is declared `with _` and a generic function is bounded on that trait, three questions arise:

1. Does `fn run[T: Processor](p: T, ...) -> ...` automatically get `with _` or must the programmer declare it?
2. Can you write effect bounds on type parameters? `fn run[T: Processor + writes(DB)](...)` — is this valid syntax?
3. If two `impl Processor` blocks have different concrete effects, can the compiler verify callers at monomorphization time?

**Options:**

**Option A — Propagate automatically; no effect bounds on type params.** A generic function calling a `with _` method inherits `with _` automatically. Effect bounds on type parameters are not supported — if a caller needs to know the effect, it must take a concrete type. Monomorphization verifies each instantiation independently.

**Option B — Require explicit declaration; no effect bounds.** Same as A for effect bounds, but the programmer must write `with _` on the generic function explicitly. A compile error points out the missing annotation. This makes effect signatures self-documenting at every level.

**Option C — Allow effect bounds on type params.** `fn run[T: Processor + writes(DB)]` is valid and constrains which implementations can be passed. A `T: writes(DB)` bound means "the concrete type's methods write to DB." This is the most expressive but significantly complicates the type system.

**Decision:** Options B and C both adopted — B now, C in Phase 6.

**Option B — explicit `with _` required.** A generic function that calls a `with _` trait method must declare `with _` explicitly. The compiler rejects the function with a missing-annotation error if omitted. Auto-propagation (Option A) was rejected: silently making a function effect-polymorphic defeats the purpose of the effect system — every function in the call chain should declare what it might do. The `async` analogy applies: Rust's viral `async` annotation is intentional, not a limitation.

**Trait definitions do not change.** Trait methods continue to use `with _` as before. No new syntax is needed on trait definitions or impl blocks. The compiler already infers/verifies effects on every impl — that information is available for bound checking without requiring programmers to annotate impls separately.

**Option C — effect bounds on type params — design closed, implementation in Phase 6.** `fn run[T: Processor + writes(DB)](...)` is valid planned syntax. The bound is checked against the concrete impl's effects at monomorphization time (the compiler already has this information from effect inference). This enables precise auto-concurrency analysis: the compiler can determine at the generic function call site that any instantiation of `run` writes to DB without waiting for monomorphization to propagate that information. Trait definitions remain unchanged — bounds live at the generic function, not on the trait. Implementation is targeted for Phase 6 because that is when precise effect information through generic function boundaries is needed for concurrency analysis; it is not needed for phases 1–5.

Option C rejected as a day-1 feature: the implementation complexity is high (effect bounds interact with the trait solver and monomorphization pipeline) and there are no use cases that require it before Phase 6. Reserving the syntax space now means Phase 6 can add it as a clean extension with no migration cost.

**design.md changes:** Updated the Effect Polymorphism section to state that `with _` on a generic function is required (not auto-inferred); added a note that effect bounds on type params (`T: writes(DB)`) are planned syntax targeting Phase 6.

## G69 — Parameterized Resources: Defining "Statically Distinguishable"

**Problem:** The spec says the compiler serializes calls to `UserDB[user_id]` when user IDs "aren't statically known," treating them conservatively as the same resource. But "statically distinguishable" is never defined. Three ambiguous cases:

1. `const MY_ID: i64 = 42;` — is this distinguishable from a literal `42`? From a different const?
2. A function parameter passed through: `fn process(id: i64) { update_profile(id); update_settings(id); }` — same parameter, so same or different?
3. Expressions: `update_profile(id + 1); update_settings(id);` — are these conservatively the same?

**Options:**

**Option A — Literals and named consts only.** Two resource parameters are distinguishable iff both are integer/string literals or named compile-time constants, and their values are statically unequal. All other cases (function params, expressions, runtime values) are conservatively treated as the same resource.

**Option B — Symbolic identity.** Two parameters are distinguishable iff they are provably different symbols — e.g., two different named consts, or two different literal values. Two occurrences of the *same* variable (`id, id`) are treated as the same resource (conflict). Two occurrences of *different* variables (`id_a, id_b`) are treated as potentially different (no conflict). Expressions involving the same variable are treated conservatively as the same.

**Option C — SMT-assisted (future work).** Full symbolic reasoning via an integrated SMT solver. Deferred to Phase 4+ as an optimization; Option B is the initial implementation.

**Decision:** Option B — Symbolic identity, with const-folding applied before comparison.

Two resource parameters are distinguishable iff:
- Both are integer or string literals (or compile-time constants reducible to literals via const-folding), and their evaluated values differ; **or**
- They are references to syntactically distinct variable names at the same call site.

All other cases — expressions, computed values, or two occurrences of the same variable — are treated conservatively as the same resource and serialized. Symbolic reasoning does not cross call-site boundaries: a caller that passes the same runtime value for two syntactically distinct parameters may under-serialize; this is a known and accepted limitation.

**design.md changes:** Updated the Parameterized Resources section to specify the distinguishability rules and note the call-boundary limitation.

## G70 — `#[test(requires = [...])]` Resource Availability Infrastructure

**Problem:** G62 added `#[test(requires = [UserDB, PaymentAPI])]` to skip tests with unavailable external resources. But the infrastructure for determining availability is unspecified:

1. Are names in `requires` bare identifiers, module-qualified paths, or strings?
2. What is the interface `karac test` uses to probe availability — environment variables, config file, a registered health check?
3. `karac test --all` runs everything — does it fail or skip unavailable resources?

**Options:**

**Option A — Environment variable convention.** Each effect resource `UserDB` is considered available iff the environment variable `KARA_RESOURCE_USERDB` is set (non-empty). `karac test` skips; `karac test --all` fails on any skipped test. Names in `requires` are bare identifiers (module path stripped).

**Option B — `kara.toml` resource registry.** A `[test.resources]` table in `kara.toml` maps resource names to health-check commands. `karac test` runs each health check before running tests; a failing check marks the resource unavailable. More explicit, CI-friendly, but requires config.

**Option C — Option A for simple cases + Option B as override.** Default to env-var detection; `kara.toml` can override with an explicit health-check command. Zero-config by default, explicit when needed.

**Decision:** Option C — two-level probe, with the following specifics:

- **Names in `requires`** are module-qualified paths (`db.UserDB`, not bare `UserDB`). Bare names would cause collisions when multiple modules define resources with similar names; the compiler already holds canonical qualified names from effect declarations.
- **Env var derivation**: uppercase the qualified name, replace `.` with `_`, prefix `KARA_RESOURCE_`. So `db.UserDB` → `KARA_RESOURCE_DB_USERDB`. Set and non-empty means available.
- **Health-check override**: a `[test.resources]` table in `kara.toml` maps resource paths to shell commands. If present, the command takes precedence over the env var — resource is available iff the command exits 0. This handles the case where an env var is set but the service is actually down.
- **`karac test` (default)**: skip unavailable tests; print the missing resource name and which env var or config key to set.
- **`karac test --all`**: fail if any required resource is unavailable. The flag means "run everything" — silent skips defeat that intent.

**design.md changes:** Updated the Testing section to use module-qualified names in the example, specify the two-level availability probe, document the env var derivation rule, and show the `kara.toml` health-check syntax.

## G71 — Refinement Type Subtyping Rules

**Problem:** The spec shows implicit subtyping between refinement types:

```
type NonZero = i32 where self != 0
type Positive = i32 where self > 0
let p: Positive = ...;
let nz: NonZero = p;  // allowed
```

But the subtyping algorithm is unspecified. Questions:
1. Is `Positive <: NonZero` determined by logical implication of constraints (`self > 0` implies `self != 0`), and if so, what solver is used?
2. Does `NonEmpty[Positive]` coerce to `NonEmpty[i32]` (covariant in the type parameter)?
3. Can refinement types form arbitrary partial orders, or is the hierarchy always `refined <: base`?

**Options:**

**Option A — Only widening to the unrefined base is implicit; cross-refinement requires explicit cast.** `Positive` widens to `i32` for free. Widening from `Positive` to `NonZero` requires `p as NonZero` with a runtime check inserted. No SMT reasoning at compile time. Simple to implement.

**Option B — Syntactic implication only.** If constraint B is a syntactic sub-expression of constraint A (e.g., `self > 0` trivially implies `self != 0`), the compiler accepts the coercion without a runtime check. Otherwise, a check is inserted. No external solver.

**Option C — SMT-backed implication (future work).** Full logical implication via Z3 or similar. `Positive <: NonZero` is proven at compile time; the runtime check is elided. Deferred; Option A is the initial implementation.

**Decision:** Option A, with explicit cross-refinement rules and covariance specified.

Three subtyping cases:

- **Widening (refined → base):** Implicit and free. `Positive` widens to `i32` silently. No runtime check.
- **Narrowing (base → refined):** Explicit `try_from` always. Compiler elides the error path for provable literals; runtime check emitted for dynamic values.
- **Cross-refinement (refined A → refined B, same base):** NOT implicit. Requires explicit `as` cast. Compiler inserts a runtime check for the target constraint (panicking assert). The spec example `let nz: NonZero = p; // allowed` is a compile error — there is no implicit cross-refinement coercion. In a future phase, the compiler can elide the check when it statically proves implication (Option C territory).

Option B (syntactic implication) was rejected: "syntactic sub-expression" catches almost nothing useful — `self > 0` implying `self != 0` is arithmetic reasoning, not syntactic. Making B useful would require building most of an SMT solver anyway.

**Covariance.** `NonEmpty[Positive] <: NonEmpty[i32]` implicitly (base-widening covariance, free). `NonEmpty[Positive] <: NonEmpty[NonZero]` is not implicit — same rule as the scalar cross-refinement case.

**design.md changes:** Subtyping section rewritten as three numbered cases; cross-refinement rule and covariance rule added; incorrect implicit example removed.

## G72 — Transparent Effect Verb Scoping Across Modules

**Problem:** The spec introduces `transparent effect verb traces;` as a user-defined verb that does not propagate through call stacks. But module-level scoping rules are absent:

1. If declared in `module logging`, can `module app` use `traces(Console)` without re-declaring?
2. Is `transparent effect verb` a module-level item or a crate-level item?
3. Can a transparent verb appear inside an `effect group`? If so, does it remain non-propagating inside the group?

**Options:**

**Option A — Crate-level item; declared once.** `transparent effect verb` is declared at crate root (or re-exported). Any module in the crate can use it. Cannot appear inside an effect group — transparent verbs and grouped effects are orthogonal mechanisms.

**Option B — Module-level item with `pub` visibility.** Same scoping rules as `pub fn`. `pub transparent effect verb traces;` exports it. A transparent verb inside a group: the group propagates normally, but the transparent verb's own contribution does not further propagate beyond the direct caller. Composable but complex.

**Option C — Same as A, but transparent verbs can be `pub(crate)` or `pub`.** Scoping follows normal visibility rules; declaration is a module item. No group interaction — transparent verbs cannot be listed inside `effect group` blocks.

**Decision:** Option C, with one addition: placing a transparent verb inside an `effect group` block is a compile error (not merely "not allowed").

*Rationale:* Option A collapses into Option C once you allow re-exports — Option C just makes the model explicit and uniform. Option B is rejected because transparent verbs and effect groups are conceptually opposed: groups exist to propagate effects; transparent verbs suppress propagation. Allowing them to coexist inside a group creates a two-class group where some members propagate and some don't, which is harder to reason about for no meaningful gain.

*Specifics:*
1. `transparent effect verb` is a module-level item. Visibility follows the same rules as any other item (`pub`, `pub(crate)`, private).
2. The full module path is the canonical identity — `logging.traces` and `app.traces` are distinct verbs even if both are named `traces`.
3. Downstream modules use a transparent verb via `use logging.traces;` — no re-declaration needed.
4. A transparent verb listed inside an `effect group` block is a compile error with a clear message pointing to the `transparent` declaration.

## G73 — Closure Effect Inference When Captured Effects Vary

**Problem:** Closure effect annotations are specified for three forms (`Fn(T) -> U`, `Fn(T) -> U with _`, `Fn(T) -> U with writes(R)`), but edge cases are unspecified:

1. What type is inferred for `[f1, f2]` when `f1: Fn() with reads(X)` and `f2: Fn() with writes(Y)`? `Vec[Fn() with _]`? A union? A compile error?
2. Can effect-annotated closure types appear in struct fields? `callback: Fn(Event) with writes(DB)`?
3. What is the type of a closure that captures another `Fn() with _`?

**Options:**

**Option A — `with _` is the universal upper bound; heterogeneous closure collections use it.** `[f1, f2]` infers `Vec[Fn() with _]`. Struct fields with effect annotations are allowed, and the effect is tracked through the struct. A closure capturing `Fn() with _` is itself `Fn() with _`.

**Option B — Heterogeneous collections disallowed; struct fields allowed.** Storing closures with different concrete effects in the same collection is a type error — use a `Vec[Fn() with _]` explicitly if needed. Struct fields with specific effect annotations allowed.

**Option C — Same as A, but with a lint.** The inferred `Vec[Fn() with _]` is allowed but the compiler emits a note suggesting the programmer consider whether mixed-effect closures in a collection is intentional, since it silences effect tracking at the collection level.

**Decision:** Option B, with a concrete-union refinement.

*Rationale:* Options A and C both allow silent loss of effect tracking by automatically widening to `with _`. A lint (Option C) doesn't restore the discarded information. Kāra's philosophy is that `_` is an explicit escape hatch, not a fallback — requiring the programmer to write `Vec[Fn() with _]` costs one annotation and signals deliberate intent. Option B handles all three sub-problems cleanly.

*Specifics:*

1. **Heterogeneous collections.** The compiler first attempts to unify effects concretely: if the union of all element effects can be expressed without `_` (e.g., `reads(X)` + `reads(X) writes(Y)` → `reads(X) writes(Y)`), it does so silently. Only when the union cannot be expressed concretely does it become a compile error requiring explicit `Vec[Fn() with _]`. This preserves tracking wherever possible. Error message includes the exact annotation fix:
   ```
   error: closure collection has heterogeneous effects; annotate explicitly
     --> src/app.rs:14:18
      |
   14 |     let handlers = [on_read, on_write];
      |                    ^^^^^^^^^^^^^^^^^^ inferred element types have effects `reads(DB)` and `writes(DB)`
      |
      = help: if mixed effects are intentional, annotate: `Vec[Fn() with _]`
   ```
2. **Struct fields.** `callback: Fn(Event) with writes(DB)` is allowed. Calling `self.callback(event)` inside a function contributes `writes(DB)` to that function's inferred effects — the effect checker tracks invocation through stored closures.
3. **Closure capturing `Fn() with _`.** Is itself `Fn() with _`. Calling a wildcard closure can produce arbitrary effects, so the capturing closure inherits `_`.

*Open sub-question — CLOSED:* Closure types carrying multiple concrete effects (`Fn() with reads(X) writes(Y)`) are confirmed valid. Multiple effects are space-separated after `with`, matching function declaration syntax. Added to `docs/design.md` — "First-Class Functions" section.

## G75 — Layout Block: Fields Not Assigned to Any Group

**Problem:** A layout block may list `group` blocks that cover only some of a struct's fields. The spec does not specify what happens to unmentioned fields:

```
struct Entity { position: Vec3, velocity: Vec3, name: String, id: u64 }
layout entities: Vec[Entity] {
    group physics { position, velocity }
    // name and id not mentioned
}
```

**Options:**

**Option A — Unmentioned fields form an implicit trailing group.** They are stored together after all explicitly named groups. No warning.

**Option B — Unmentioned fields stay in declaration order, interleaved after all groups.** The layout is: SoA-grouped fields first, then AoS-ordered remaining fields. Warning emitted suggesting the programmer explicitly group or exclude them.

**Option C — Unmentioned fields are a compile error.** Every field must appear in exactly one group. If you want default ordering, write `group rest { ... }` explicitly. Forces explicit intent.

**Decision:** Option A, with a one-time warning.

*Rationale:* Kāra already has a well-defined default layout (AoS). An implicit trailing group for unmentioned fields is semantically clean — it means "these fields get the default treatment." Typos in group field names are already caught as unknown-field errors, so the implicit fallback is not a silent footgun. Option C's mandatory `group rest { ... }` creates real friction for the common case of a struct with many cold fields the programmer doesn't care to group.

*Specifics:*
1. **Implicit trailing group:** Fields not listed in any group are collected into an implicit trailing AoS group, stored after all explicitly named groups. No name is assigned; the compiler refers to them as "unassigned fields" in diagnostics.
2. **Warning:** The compiler emits a single warning per layout block listing the unassigned fields. The warning fires once per layout block, not once per field.
3. **Suppression:** `#[allow(layout_unassigned_fields)]` on the layout block (or enclosing module) suppresses the warning for intentional partial layouts.
4. **Duplicate fields:** A field listed in more than one group is a compile error — every field belongs to exactly one group.

## G76 — `kara.toml` `no_effects` Profile Syntax

**Problem:** `design.md` mentions project profiles can restrict effects:

```toml
no_effects = ["allocates(Heap)"]
```

But the syntax is unspecified beyond this one example:
1. Are the strings effect expressions (same grammar as function signatures) or a restricted form?
2. Can you restrict multiple verbs/resources? `no_effects = ["allocates(Heap)", "sends(Network)"]`?
3. What is the error if a violation is detected — compile error or a lint warning?
4. Can a specific file or module opt out via `#[allow(...)]`?

**Options:**

**Option A — Full effect expression strings; always a compile error; no per-file opt-out.** Values use the same grammar as effect annotations. Multiple entries allowed. Violations are compile errors — the profile is an enforced constraint, not advisory. No `#[allow]` override.

**Option B — Restricted to `verb(Resource)` pairs only; warning by default; upgradeable to error.** Only simple `verb(Resource)` pairs supported (no groups). `deny_effects` key for error-level enforcement, `warn_effects` for advisory. Per-module opt-out via `#[allow(effect_policy)]`.

**Option C — Same as A, but `#[allow(effect_policy)]` available with a mandatory comment explaining the exception.** Enforced by default; explicit opt-out with documented rationale allowed. Balances strictness with escape hatch.

**Decision:** Option A.

*Rationale:* Profiles are project-wide contracts, not advisories. A kernel profile that says "no heap allocation" means exactly that for the entire project — a per-file opt-out would let any contributor quietly poke holes in the invariant. The existing `unsafe` block is the correct escape hatch when a genuine exception is needed; it already carries the right signal ("I am knowingly violating a safety contract") and doesn't require a new opt-out mechanism. Full effect expression strings are the right syntax because they are the established grammar throughout the language — a restricted form would create two overlapping syntaxes with no benefit.

*Specifics:*
1. **Syntax:** `no_effects` values use the same effect expression grammar as function annotations. Both `verb(Resource)` (e.g., `"allocates(Heap)"`) and bare `verb` (e.g., `"panics"`) are valid. Multiple entries are allowed as a TOML array.
2. **Enforcement:** Violations are always compile errors, using the same structured diagnostics as per-function effect violations.
3. **Scope:** The profile applies to every function in the project — public and private — not just public ones. Effect checking is transitive; restricting only public functions would let violations hide inside private call chains.
4. **No opt-out attribute:** There is no `#[allow(effect_policy)]` or equivalent. The only escape hatch is `unsafe { ... }`, which already exists for intentional contract violations.

## G65 — `#[derive(Arbitrary)]` Used in Testing but Absent from Derive Section

**Problem:** The Testing section uses `#[derive(Arbitrary)]` as a first-class example for property tests:

```
#[derive(Arbitrary)]
struct Point { x: f64, y: f64 }
```

But the Derive section lists derivable traits as: `Eq`, `Hash`, `Display`, `Clone` (later: `Ord`, `Serialize`, `Deserialize`). `Arbitrary` is not mentioned. This creates an inconsistency — a programmer reading Derive would not know it exists.

**Options:**

**Option A — Add `Arbitrary` to the Derive section under "Testing".**
`Arbitrary` is a testing trait (random input generation for property tests). List it separately from the core derivable traits with a note that it is test-only (only available in `_test.kara` contexts, not production code).

**Option B — Add `Arbitrary` to the "Later" list alongside `Ord`, `Serialize`, `Deserialize`.**
Treat it as a standard library trait, not a compiler built-in. Property test framework provides it.

**Decision:** Option A. `Arbitrary` is added to the Derive section under a "Test-only derivable traits" heading, listed in a table alongside what it generates. A note states it is provided by the property-test framework (not the core standard library) and that the compiler enforces it cannot be derived in production code — only in `_test.kara` files. This keeps the Derive section organized by use context and makes `Arbitrary` discoverable without polluting the core trait list.
