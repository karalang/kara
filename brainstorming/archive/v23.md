# Brainstorming v23 — Effect declaration burden on public functions

**Status: RESOLVED (Q1, Q2, Q3). Trait-method policy moved to brainstorming v24 — our inline Q4 resolution contradicted the existing design § Trait Coherence and Effects, which had already thought through the open-world vs. closed-world split more carefully.**

The current spec (design.md § Effect Inference and Boundaries) requires every `pub fn` to declare its full effect set, verified against inference. The compiler suggests exact annotations on mismatch, transparent effects (`traces`, `debugs`) and default-permitted `allocates(Heap)` keep common noise out, and effect groups let wide signatures stay short.

Even so, for libraries with large public surfaces the annotation line can dominate signatures, and drift between inference and declaration becomes routine churn. This doc exists to reconsider the boundary between *inferred* and *declared* effects for public APIs.

---

## 1. Should public functions be required to declare effects at all?

**Question:** What's the policy for effects on `pub fn`?

- **(a) Mandatory declaration (status quo).** Every `pub fn` declares its effect set; compiler verifies. Maximum API stability — internal changes can never silently change the public effect surface. Maximum annotation burden. Matches the current spec.
- **(b) Inference by default, opt-in freezing.** `pub fn` effects are inferred like private functions and displayed in compiler output / `karac explain` / docs. Authors opt into stability per function (or per module) with `#[frozen_effects]` or a `with effects(...)` clause when they want the declaration to be load-bearing. Low-ceremony for apps and prototypes; serious libraries still get the stability guarantee when they ask for it.
- **(c) Inference with project-level policy.** `kara.toml` picks the default: `public_effects = "declared" | "inferred" | "inferred_warn"`. Libraries turn on `declared` for their `pub` API; application crates leave it `inferred`. Same mechanism as the existing `no_effects` project profile knob, so there's precedent.
- **(d) Tiered visibility.** `pub` = inferred + displayed; a new `pub(stable)` or `pub api` = declared + verified. Callers depending on stability opt into the stricter tier; quick public helpers stay unannotated. Adds a language-level concept.

**Tradeoffs:**

- (a) is the strictest contract and needs no new machinery, but has the highest friction — relevant for a language whose core pitch is effect-aware concurrency; if users start stripping `pub` to avoid the ceremony, the language loses.
- (b) trades some stability for ergonomics. The risk is effect drift: a library author refactors an internal helper, inference newly reports `writes(Disk)`, no one notices until a downstream consumer's build breaks. Compiler-displayed diffs in CI help but don't fully close the gap.
- (c) keeps the existing machinery and makes the policy a project-author call. Libraries pay the cost where stability matters; apps don't.
- (d) is the most expressive but adds a visibility axis for a problem that may not warrant one. Close cousin of Rust's `pub(crate)`.

**Resolved — (c), simplified.** One toml knob: `public_effects = "declared" | "inferred"`. Default `declared` (status quo — stability contract preserved for libraries). Flip to `inferred` for prototyping and applications. No per-function attribute override, no lib/bin default split, no `inferred_warn` middle ground. The per-function `#[frozen_effects]` attribute considered earlier was cut — it reads like an afterthought patch and adds a language-level concept for a policy that's better handled at the project level. Rationale: since most code is LLM-written, annotation burden is cheap; the real friction is prototyping, which is a per-project stance. Libraries keep `declared`; prototypes flip the knob once.

## 2. If we go with (b) or (c), how does the displayed effect set stay visible without a declaration?

**Question:** Where does the inferred effect surface live when it's not in the signature?

- **(a) Compiler output only.** `karac build` prints the program-level and per-public-fn effect summary. Consumers read it off the terminal or JSON output.
- **(b) Generated docs.** `karac doc` renders the inferred effect set next to each `pub fn` signature. Consumers read it there; CI diffs the rendered docs.
- **(c) Interface file (`.karai` or similar).** A compiler-emitted sidecar that records the effect surface in a stable format. Checked into the repo; effect drift shows up in a commit diff without the declaration being in the source.
- **(d) All three.** The interface file is the machine-readable source of truth; docs and compiler output render from it.

Only relevant if Q1 lands on (b)/(c)/(d).

**Resolved — (a) + (b).** Under `inferred` mode, `karac build` prints the per-public-fn inferred effect set (already does this for `main()`), and `karac doc` renders it alongside signatures. `.karai` sidecar (option c) is deferred — the build/doc surfaces are enough for the prototyping use case this mode targets. If a library later wants stable machine-readable diffs, it should switch to `declared`.

## 3. Interaction with `no_effects` project profile

**Question:** How does an inferred-public-effects policy interact with `no_effects = [...]` in `kara.toml`?

Today, `no_effects` removes the default permit for listed effects on public functions — which presupposes that those functions declare effects. If public effects become inferred, `no_effects` must instead flag *inferred* presence of the listed effects. Mechanically the same check (inferred set ∩ forbidden set), just on inferred instead of declared data.

Probably trivial, but worth confirming the check still fires at the right place and the diagnostic still points at the offending source line (the call that introduced the forbidden effect, not just "this function").

**Resolved — no change needed.** The `no_effects` check already runs on inferred effects for private functions; under `public_effects = "inferred"` it just runs on inferred effects for public functions too. Same machinery, same diagnostic shape. The diagnostic already points at the introducing call site for private fns; that behavior carries over.

## Q4: Trait method effect policy — moved to v24

Initially resolved here as "trait methods always declare, independent of visibility." That resolution was wrong: `docs/design.md` § Trait Coherence and Effects already specifies that **public traits declare** (open-world, must declare up front to stay sound under separate compilation) while **private traits infer from the union of visible impls** (closed-world, inference is safe). The existing open/closed-world justification is stronger than the uniformity argument we used, so the existing design wins.

The question that remains — how `public_effects = "inferred"` interacts with trait effect declarations, plus several underspecified edges (trait-level `with ...` shorthand, supertrait inheritance, impl-exceeds-trait diagnostic, default method body checking, associated-function rule) — is tracked in **brainstorming v24**.
