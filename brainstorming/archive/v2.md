# Kāra Brainstorming V2 — Gaps Identified from V8 Synthesis

**Origin:** Comparison of design.md against a "best of" synthesis (V8) drawn from seven independent language design brainstorms. This document captures gaps where V8 identified features that would strengthen Kāra's "language to rule them all" ambition.

**Format:** Each section follows the same structure as v1.md — explore options, weigh trade-offs, reach a decision. Resolved items graduate to design.md. Items marked DEFERRED stay here with rationale.

**Numbering:** Gaps are numbered G40+ to avoid collision with v1.md's G1–G39 numbering.

---

## G40 — Refinement Types (Level 2)

**Status: RESOLVED — graduated to design.md**

### The Problem

Kāra has no way to express value constraints in the type system. Division by zero, out-of-range errors, empty collection access — all detectable at compile time in principle — are only caught at runtime. design.md already calls Level 2 refinement types "achievable without an SMT solver and a valuable differentiator" but defers them entirely.

### What V8 Proposes

```
type NonZero = i32 where self != 0
type Positive = i32 where self > 0
type Percentage = f64 where self >= 0.0 && self <= 100.0
type NonEmpty[T] = Vec[T] where self.len() > 0
type ValidPort = u16 where self >= 1 && self <= 65535
```

At system boundaries, you validate into refined types:

```
fn parse_port(input: String) -> Result[ValidPort, ParseError] {
    let n = input.parse[u16]()?;
    ValidPort.try_from(n)  // Err if constraint fails
}
```

Inside the system, the type carries the proof:

```
fn divide(a: i32, b: NonZero) -> i32 {
    a / b  // can never be division by zero
}

fn head[T](list: NonEmpty[T]) -> T {
    list[0]  // can never be empty
}
```

### Design Questions

1. **Syntax.** Kāra already reserves `where` for this. Use `type Name = BaseType where constraint` or a different form?

2. **What constraints are allowed?** Options:
   - **Numeric comparisons only** (`>, <, >=, <=, ==, !=`) — simplest, covers 80% of use cases (NonZero, Positive, ValidPort, Percentage). No SMT solver needed — the compiler checks at construction sites.
   - **Numeric + length** — add `self.len()` for collections. Covers NonEmpty.
   - **Numeric + length + regex** — add string pattern matching. Covers ValidEmail-style types.
   - **Arbitrary boolean expressions** — most powerful but hardest to verify at compile time.

3. **When are constraints checked?**
   - **At construction** (runtime): `ValidPort.try_from(n)` returns `Result`. This is always needed at system boundaries.
   - **At compile time** (when provable): `let p: ValidPort = 80` — compiler knows 80 is in range, no runtime check. `let p: ValidPort = some_function()` — can't prove, requires explicit `try_from`.
   - **Propagation**: Once a value has a refined type, operations that preserve the constraint keep it. `p + 1` where `p: Positive` — is the result still `Positive`? Probably not worth tracking (too complex). Keep it simple: refined types are checked at construction, then carried as-is.

4. **Interaction with effects.** Constructing a refined type from untrusted input should probably carry an effect or at minimum a `Result`. The `try_from` pattern handles this naturally.

5. **Interaction with generics.** `fn process[T: Ord](list: NonEmpty[T]) -> T` should work. The refinement is on the container, not the element type.

### Recommendation

Start with **numeric comparisons + `len()`** on the `where` clause. No SMT solver. Runtime checks at construction boundaries, compile-time elision when provably satisfied. Refined types are subtypes of their base — `NonZero` is usable anywhere `i32` is expected (implicit widening to base type is always safe). Going from base to refined requires `try_from`.

### Prior Art

- **Ada/SPARK:** Subtypes with range constraints. `subtype Positive is Integer range 1 .. Integer'Last`. Proven in production for decades.
- **Liquid Haskell:** Full refinement types with SMT. Powerful but complex.
- **TypeScript (branded types):** Runtime-only, no compiler verification.
- **Koka:** Refinement types via effect handlers. Overkill for Kāra's model.

## G41 — Distinct Types (Newtypes)

**Status: RESOLVED — graduated to design.md**

### The Problem

Kāra has no zero-cost way to create semantically distinct types from the same underlying representation. `UserId` and `PostId` are both `u64`, but passing a `PostId` where a `UserId` is expected is a common, silent bug. You'd need a full wrapper struct today:

```
struct UserId { value: u64 }
struct PostId { value: u64 }
```

This works but is verbose — you need `.value` everywhere, manual trait impls, etc.

### What V8 Proposes

```
distinct type UserId = u64
distinct type PostId = u64
distinct type Meters = f64
distinct type Seconds = f64

fn get_user(id: UserId) -> User { ... }

let uid = UserId(42);
let pid = PostId(42);
get_user(pid);    // COMPILE ERROR: expected UserId, got PostId
```

Zero runtime cost. The underlying representation is identical to `u64`. The type system prevents mixing.

### Design Questions

1. **Keyword.** `distinct type` (V8), `newtype` (Haskell), or something else?

2. **Which operations carry through?** If `UserId = u64`, can you do `uid + 1`? Options:
   - **No operations carry through.** `distinct type` creates an opaque type. You must explicitly convert to/from the base type. Safest but most verbose.
   - **Read-only operations carry through.** Comparison (`==`, `<`), hashing, display — yes. Arithmetic — no. A `UserId + UserId` doesn't make semantic sense.
   - **All operations carry through.** Least safe — you could accidentally do `user_id + post_id`.
   - **Configurable via derive.** `#[derive(Eq, Hash, Display)]` on the distinct type declaration.

3. **Conversion.** How do you go between `UserId` and `u64`?
   - Constructor: `UserId(42)` — wrap
   - Method: `uid.raw()` or `uid.into()` — unwrap
   - `as` cast: `uid as u64` — explicit, grep-able

4. **Interaction with generics.** `fn find[T: Eq](items: Vec[T], target: T) -> Option[i64]` should work with `Vec[UserId]`.

5. **Interaction with refinement types (G40).** Can you combine them? `distinct type ValidPort = u16 where self >= 1 && self <= 65535` — distinct AND constrained. This would be powerful.

### Recommendation

`distinct type Name = BaseType` with **no operations carried through by default**. Use `#[derive(Eq, Hash, Display, Ord)]` to opt into specific behaviors. Constructor syntax `Name(value)` for wrapping, `.raw()` method for unwrapping. This is the safest and most explicit approach.

Allow combining with `where` constraints if G40 is implemented: `distinct type ValidPort = u16 where self >= 1 && self <= 65535`.

### Prior Art

- **Haskell:** `newtype`. Zero-cost, no operations carry through.
- **Rust:** No built-in newtype. Requires wrapper struct + manual impls.
- **Go:** `type UserId uint64`. Operations carry through (less safe).
- **Swift:** No built-in. Uses `struct` wrappers.
- **Ada:** `type Meters is new Float`. Operations carry through.

## G42 — Contracts (requires / ensures / invariant)

**Status: RESOLVED — graduated to design.md**

### The Problem

Kāra's effect system tracks WHAT a function does to the outside world, but not WHAT a function guarantees about its inputs and outputs. There's no way to express "this function requires a sorted input" or "this function guarantees the output is sorted" in a compiler-verifiable way.

Contracts are especially important for AI-generated code: the human writes the spec (contracts), the AI generates the implementation, the compiler verifies the implementation satisfies the spec.

### What V8 Proposes

```
fn binary_search[T: Ord](
    haystack: ref [T],
    needle: ref T,
) -> Option[usize]
    requires haystack.is_sorted()
    ensures |result| match result {
        Some(i) => haystack[i] == *needle,
        None => !haystack.contains(needle),
    }
{
    // implementation
}
```

Type invariants:

```
struct SortedVec[T: Ord] {
    data: Vec[T],

    invariant self.data.is_sorted()
}
```

### Design Questions

1. **Verification strategy.** Three options, not mutually exclusive:
   - **Runtime checks in debug builds** (like `assert`). Simplest. Zero cost in release. This alone is useful.
   - **Compile-time checks where provable** (simple constant folding / range analysis). Catches `binary_search(unsorted_literal, ...)`.
   - **SMT solver integration** (Z3). Full formal verification. Kāra's design.md already says "may never be built."

2. **Syntax.** V8 uses `requires`/`ensures` after the signature, before the body. This reads well and is unambiguous. But Kāra already has effect annotations in that position (`reads(X), writes(Y)`). Can they coexist?

   Possible syntax:
   ```
   fn binary_search[T: Ord](haystack: ref Vec[T], needle: ref T) -> Option[usize]
       reads(SearchIndex)
       requires haystack.is_sorted()
       ensures |result| result.is_some() implies haystack[result.unwrap()] == *needle
   { ... }
   ```

   Effects → requires → ensures. Clean ordering.

3. **What expressions are allowed in contracts?** Options:
   - **Pure expressions only.** No side effects. The compiler enforces purity (ties into the effect system — contracts must have no effects).
   - **Method calls on arguments.** `haystack.is_sorted()` — the method must be pure.
   - **Quantifiers.** `forall i in 0..result.len(): result[i] <= result[i+1]` — for full specification power. Complex.

4. **`old()` references.** For postconditions that reference the pre-call state: `ensures permutation(old(input), result)`. Requires the runtime to snapshot the input before the call (in debug builds).

5. **Interaction with effects.** Contracts should be pure — they can't perform IO or mutate state. The effect system naturally enforces this: contract expressions must have no effects.

6. **Interaction with refinement types (G40).** Refinement types are lightweight contracts on types. Full contracts are heavyweight contracts on functions. They compose: `fn f(x: Positive) -> Positive requires x < 100 ensures result > x`.

### Recommendation

Start with **runtime-checked `requires`/`ensures`/`invariant`** in debug builds, stripped in release. Pure expressions only (enforced by effect system). No SMT solver. This is immediately useful for:
- Documentation that doesn't drift from reality
- AI-generated code verification
- Debug-time bug detection

Defer `old()` references and quantifiers to later. Start simple:
- `requires`: boolean expression over parameters
- `ensures`: boolean expression over `result` and parameters
- `invariant`: boolean expression over `self` fields, checked after every pub method

### Prior Art

- **Ada/SPARK:** `Pre` and `Post` aspects. Runtime-checked or formally verified.
- **Eiffel:** Design by Contract pioneer. `require`/`ensure`/`invariant`.
- **D:** `in`/`out` contracts on functions.
- **Rust:** No built-in. `debug_assert!` is the closest.
- **Dafny:** Full formal verification. Too heavy for a systems language.

## G43 — Complexity Budgets in Contracts

**Status: DEFERRED — heap case covered by effect system; stack/complexity cases deferred pending G46 (Profiles)**

### The Problem

For embedded, kernel, and real-time code, it's not enough to know WHAT a function does — you need to know its resource consumption. "This function must not heap-allocate" and "this function must run in O(n log n)" are constraints the compiler should enforce.

### What V8 Proposes

```
fn sort(buf: mut ref [i32]) -> Result<(), SortErr>
    requires is_valid(buf)
    ensures is_sorted(buf)
    budget alloc <= 0           // must not heap-allocate
    budget worst_case <= O(n log n)
{
    // in-place heapsort — satisfies all constraints
}
```

### Design Questions

1. **`budget alloc <= 0` already exists in Kāra** — it's `#[no_effect(allocates(Heap))]` or `#[no_rc]`. The effect system handles this. Should `budget` be separate syntax or just alias effect constraints?

2. **Complexity bounds (`O(n log n)`)** are extremely hard to verify automatically. Even simple loop analysis is undecidable in the general case. Options:
   - **Documentation only** — `budget` is a comment the compiler doesn't check but includes in docs/diagnostics.
   - **Checked against manual annotation** — the programmer annotates complexity; the compiler doesn't verify but warns if the annotation changes.
   - **Simple case analysis** — count loop nesting, flag obvious violations (O(n^2) when O(n) is claimed). Very limited.

3. **Stack budget.** `budget stack <= 4096` — the compiler CAN verify this (LLVM can compute stack frame sizes). This is immediately useful for embedded/kernel.

### Recommendation

- **`budget alloc`**: Map to Kāra's existing `#[no_effect(allocates(Heap))]`. No new syntax needed — the effect system already handles this.
- **`budget stack`**: Implementable. The compiler knows stack frame sizes. Add as a contract annotation: `budget stack <= N`.
- **`budget complexity`**: Documentation-only annotation for now. No verification. Include in generated docs and performance reports.

This is a natural extension of G42 (contracts). Implement after contracts land.

## G44 — Property-Based Testing + Fuzz Testing

**Status: RESOLVED — graduated to design.md**

### The Problem

Kāra has `#[test]` with `assert()` — basic unit tests. For AI-generated code, this is insufficient. AI implementations need stronger verification: property-based tests that check invariants across random inputs, and fuzz tests that check for crashes on arbitrary input.

### What V8 Proposes

```
test property "sort preserves length" {
    forall (input: Vec[i32]) {
        let sorted = input.clone().sort();
        assert_eq(sorted.len(), input.len());
    }
}

test fuzz "parser doesn't crash" {
    forall (input: Vec[u8]) {
        let _ = parse(input);
    }
}

test snapshot "pretty printer output" {
    let ast = parse("fn foo() { 42 }");
    assert_snapshot(pretty_print(ast));
}

test bench "vector dot product" {
    bench(|| dot(a, b));
}
```

### Design Questions

1. **Property tests need a random generator.** Options:
   - **Built-in `Arbitrary` trait** — types that can generate random instances. `#[derive(Arbitrary)]` for user types (requires comptime or built-in derive).
   - **Stdlib generator** — `std.test.gen[T]()` returns a random instance.
   - **External dependency** — use a testing library. Against the "batteries included" philosophy.

2. **Shrinking.** When a property test fails, the framework should find the minimal failing input. This requires the generator to support shrinking. Built-in or optional?

3. **Fuzz tests** need different infrastructure than property tests. Property tests are fast (in-process random generation). Fuzz tests are slow (coverage-guided, often use libFuzzer or AFL). Should fuzz tests be a language feature or a toolchain feature?

4. **Snapshot tests** need a file system for storing snapshots. The test runner needs `--update-snapshots` support. Straightforward to implement but needs `writes(FileSystem)` — which makes them effectful tests.

5. **Benchmarks** need statistical analysis (min/max/mean/stddev, warmup, iteration count). Built into the test runner or separate command?

6. **Syntax.** V8 uses `test property "name" { forall ... }`. Kāra uses convention-based testing (`_test.kara` files + `test_` prefix). Options for extended test types:
   - `#[property] fn test_name(input: Vec[i32]) { ... }` — attribute marks the test kind
   - `test property "name" { forall ... }` — new syntax, more readable
   - Add `#[property]`, `#[fuzz]`, `#[bench]`, `#[snapshot]` attributes on `test_`-prefixed functions

### Recommendation

Use **attributes on test functions** to distinguish test kinds. The `test_` prefix identifies a function as a test; attributes like `#[property]`, `#[fuzz]`, `#[bench]` specify what *kind* of test:

```
// in sort_test.kara

fn test_sort_basic() {
    assert(sort([3, 1, 2]) == [1, 2, 3]);
}

#[property]
fn test_sort_preserves_length(input: Vec[i32]) {
    assert(sort(input.clone()).len() == input.len());
}

#[fuzz]
fn test_parser_doesnt_crash(input: Vec[u8]) {
    let _ = parse(input);
}

#[bench]
fn bench_sort(b: Bencher) {
    b.iter(|| sort(random_vec(10000)));
}
```

Property tests: built-in `Arbitrary` trait with `#[derive(Arbitrary)]`. Shrinking included. Fuzz tests: compile to libFuzzer-compatible harness via `karac test --fuzz`. Benchmarks: built into test runner via `karac test --bench`.

Defer snapshot testing — it's useful but lower priority than property and fuzz.

## G45 — Handles as First-Class Pattern

**Status: RESOLVED — graduated to design.md**

### The Problem

Kāra addresses self-referential structures and shared data with `shared struct` (RC-based) and `Arena<T>` (stdlib primitive). But many performance-critical systems use a third pattern: **typed handles** — integer indices into pools. This is how ECS game engines, OS kernels (file descriptors), web servers (connection IDs), and compilers (AST node arenas) work.

Kāra's design.md mentions `Arena<T>` + `ArenaRef<T>` but calls it "unergonomic." V8 makes handles first-class.

### What V8 Proposes

```
struct EntityHandle = handle(EntityPool)

let pool = EntityPool.new();
let h: EntityHandle = pool.insert(Entity { name: "player", health: 100 });

let entity = pool.get(h);   // returns Option[ref Entity]

// Handles can be stored in structs, copied, sent across threads
struct GameWorld {
    entities: EntityPool,
    player: EntityHandle,
    enemies: Vec[EntityHandle],
}

// Dangling handles return None, not UB
pool.remove(h);
let gone = pool.get(h);  // returns None
```

### Design Questions

1. **Language feature or stdlib type?** Options:
   - **Language feature** (`handle(Pool)` syntax): Compiler understands handles, can optimize, can prevent misuse (e.g., using a handle with the wrong pool).
   - **Stdlib type** (`Pool[T]` + `Handle[T]`): Simpler. Doesn't require language changes. Less compiler support for safety.

2. **Cross-pool safety.** The biggest danger with handles is using a handle from Pool A with Pool B. Options:
   - **Typed handles** (`EntityHandle` is distinct from `WeaponHandle`): Compile-time safe. Requires distinct types per pool.
   - **Generational indices** (handle stores a generation counter; pool validates): Runtime safe. Catches use-after-remove.
   - **Both**: Typed + generational. Maximum safety.

3. **Interaction with `shared struct`.** Should `shared struct` remain for truly shared data (trees, graphs) while handles are for pool-based data (ECS, arenas)? Or does one subsume the other?

4. **Interaction with effects.** Accessing a pool could be an effect (`reads(EntityPool)`, `writes(EntityPool)`). This would let the auto-concurrency system parallelize access to different pools.

### Recommendation

**Stdlib type, not language feature.** `Pool[T]` and `Handle[T]` in `std.collections`. Generational indices for runtime safety. Combined with `distinct type` (G41) for compile-time pool discrimination:

```
distinct type EntityHandle = Handle[Entity];
distinct type WeaponHandle = Handle[Weapon];
```

This gives you compile-time type safety (can't mix entity and weapon handles) + runtime generational safety (can't use a stale handle) without language changes.

`shared struct` remains for inherently shared data (trees, graphs). `Pool[T]` is for indexed-access patterns (ECS, arenas). They're complementary, not competing.

### Prior Art

- **Bevy ECS (Rust):** `Entity` is a generational index.
- **Slotmap (Rust crate):** `SlotMap<K, V>` with typed keys.
- **Flecs (C):** Entity IDs as u64 with generation.

## G46 — Profiles (Kernel / Engine / Server / Embedded)

**Status: RESOLVED — kernel and embedded graduated to design.md; engine and server deferred (underlying features not yet designed)**

### The Problem

Kāra targets everything from OS kernels to web servers, but has no way to express project-wide constraints. `#[no_rc]` is per-function. There's no way to say "this entire project must not heap-allocate" or "this project must not have panic paths."

### What V8 Proposes

Profiles constrain available features per target:

```
#[profile(kernel)]     // no stdlib, no panic paths, custom panic handler
#[profile(engine)]     // SIMD, arenas, hot-reload, SoA layout optimization
#[profile(server)]     // async I/O, structured concurrency, observability
#[profile(embedded)]   // compile-time memory budgets, interrupt-safe subsets
```

### Design Questions

1. **Are profiles a language feature or a build config?** Options:
   - **Language attribute** (`#[profile(kernel)]`): Checked at compile time. Can't accidentally use forbidden features.
   - **Build flag** (`karac build --profile kernel`): Simpler. Applied project-wide.
   - **Manifest config** (`kara.toml: profile = "kernel"`): Same as build flag, persisted.

2. **What does each profile restrict?**

   | Feature | Kernel | Engine | Server | Embedded |
   |---|---|---|---|---|
   | `std` library | No | Yes | Yes | No |
   | Heap allocation | Explicit only | Yes | Yes | Budget-limited |
   | RC fallback | No | Yes | Yes | No |
   | Panic paths | Custom handler | Yes | Graceful | No |
   | Concurrency | Manual | Auto | Auto | No |
   | Layout blocks | Yes | Yes | Optional | Yes |
   | SIMD | Manual | Auto | Optional | Manual |

3. **Custom profiles.** Should users be able to define their own profiles? Or is a fixed set sufficient?

4. **Interaction with effects.** Profiles are essentially project-wide effect restrictions. A kernel profile could be `#[no_effect(allocates(Heap), panics)]` applied globally. This maps naturally to Kāra's effect system.

### Recommendation

**Build config in `kara.toml`, not a language attribute.** Profiles are project-wide constraints, not per-file decisions. They map to sets of effect restrictions:

```toml
[profile]
name = "kernel"
no_effects = ["allocates(Heap)", "panics"]
no_std = true
panic_handler = "custom"
```

The compiler enforces these as project-wide effect budgets. This reuses the existing effect system rather than adding a parallel constraint mechanism.

Start with `kernel` and `embedded` profiles (most constrained). `engine` and `server` profiles are mostly about enabling features (SIMD, observability) which can be handled by feature flags.

Defer custom profiles until the fixed set proves limiting.

## G47 — Generics Syntax: `[T]` vs `<T>`

**Status: RESOLVED — switched to `[T]`. Unambiguous grammar, no turbofish, prior art in Go 1.18. All design.md examples updated.**

### The Problem

Kāra uses Rust's `<T>` syntax for generics, inheriting the "turbofish" problem: `<T>` is ambiguous with the less-than operator in parsing contexts. Rust solved this with the `.<T>` turbofish syntax, which is widely considered ugly.

V8 uses `[T]` — unambiguous because `[]` is not used as a comparison operator.

### Trade-offs

**`<T>` (current):**
- Pro: Familiar to Rust/C++/Java/C#/TypeScript developers
- Con: Parsing ambiguity (`foo<bar>(baz)` — is this `foo < bar > (baz)` or `foo.<bar>(baz)`?)
- Con: Turbofish (`.<T>`) is ugly
- Con: AI generates more parsing errors with `<T>`
- Con: design.md and all Kāra examples have since been updated to `[T]`

**`[T]` (V8):**
- Pro: Zero parsing ambiguity (LL(1) clean)
- Pro: AI generates correct generics more often
- Pro: Cleaner syntax (`HashMap[String, i64]` vs `HashMap<String, i64>`)
- Con: `[]` is also used for array indexing (`arr[0]`). Could confuse: `Vec[i32]` (generic) vs `vec[0]` (index). But these are syntactically distinguishable (type position vs expression position).
- Con: Unfamiliar to most developers
- Con: Would require rewriting ALL existing Kāra examples, parser, tests

### Recommendation

**Switch to `[T]`.** The parsing ambiguity and AI code generation concerns outweigh the familiarity benefit. `[T]` is unambiguous (LL(1) clean), eliminates turbofish, and AI models produce more correct generics with it. Pre-1.0 is the right time for this break. All design.md examples and the parser have been updated.

## G48 — String Story (Zero-Copy Views, SSO, Interning)

**Status: RESOLVED**

### Decision

Three string types:

- **`String`** — owned, heap-allocated. SSO (Small String Optimization) for strings ≤ 23 bytes: stored inline in the struct, no heap allocation. Programmer never sees the difference — it's an internal detail.
- **`StringSlice`** — borrowed view into a `String` (pointer + offset + length). Zero-copy. Follows the same borrow rules as `ref T` — cannot outlive its parent `String`. Returned with `ref(s)` annotation. Call `.to_string()` to get an owned copy when needed.
- **`InternedString`** — deduplicated string handle. `intern("x")` on the same string always returns the same handle. Equality is O(1) integer comparison. Opt-in. For symbol tables, config keys, compiler internals.

COW was considered and rejected: hidden memory retention (a substring keeps the entire parent buffer alive) is a systems-language footgun. Java dropped COW substrings in JDK 7 for exactly this reason. `StringSlice` gives zero-copy reads with explicit, visible semantics.

### Prior Art

- **Swift:** COW strings with value semantics. Works well in practice.
- **Rust:** `String` + `&str`. Maximum performance, maximum complexity.
- **Go:** Immutable strings with slicing (already COW-like).
- **Java:** Immutable `String` with `substring()` views (was COW, then changed to copying in JDK 7 — burned by hidden retention).

## G49 — Hot Reloading

**Status: RESOLVED — tracked in `docs/design.md` Deferred Items**

Decision: Option 1 (shared library reloading with `#[stable_abi]` boundaries). Deferred to Phase 9+ pending stable ABI and LLVM backend.

### The Problem

Game developers and server developers need to reload code without restarting the process. This conflicts with AOT compilation.

### Options

1. **Shared library reloading.** Compile changed modules to `.so`/`.dylib`, `dlopen` at runtime. Requires `#[stable_abi]` on reload boundaries. Kāra's layout blocks make this harder (SoA layouts may change).

2. **JIT for debug builds.** Debug builds use a JIT compiler with hot-reload support. Release builds are pure AOT. Two compilation paths.

3. **State serialization + restart.** Serialize game state, rebuild binary, deserialize. Slow but simple.

4. **Interpreter mode for debug.** Kāra already has a tree-walk interpreter (Phase 4). Use it for debug/development with hot-reload. AOT for release.

### Recommendation

**Defer.** This is a Phase 9+ concern. The interpreter (Phase 4) already enables a REPL-like workflow for development. Hot reloading for games/servers is important but requires the LLVM backend (Phase 7) and ABI stability to be settled first.

When implemented, option 1 (shared library reloading) with `#[stable_abi]` boundaries is the most practical approach. The effect system helps: reload boundaries are functions that cross module boundaries, and effect annotations ensure the reloaded code has compatible effects.

## G50 — Spec-First Programming

**Status: RESOLVED — tracked in `docs/design.md` Deferred Items**

Decision: Deferred to Phase 10+ pending working contracts (G42) and gradual verification (Level 3). Contracts should be designed with spec-first in mind as a future end state.

### The Problem

If AI writes 99% of code, what does the human write? The answer: specifications. The human declares WHAT via types + contracts + tests. The AI generates HOW.

### What V8 Proposes

```
spec fn sort[T: Ord](items: Vec[T]) -> Vec[T]
    ensures |result| {
        result.len() == items.len()
        and result.is_sorted()
        and result.is_permutation_of(items)
    }

// AI generates the implementation.
// Compiler verifies it satisfies the spec.
// Human never reads the implementation unless debugging.
```

### Why This Matters for Kāra

Kāra's motto is "the programmer declares *what* and *why*, the compiler decides *how* and *where*." Spec-first programming is the logical conclusion of this philosophy. The effect system already separates what a function does (effects) from how it does it (implementation). Contracts (G42) add what a function guarantees. Spec-first programming takes the final step: the implementation itself becomes an AI-generated artifact.

### Recommendation

**Defer until contracts (G42) are implemented and proven.** Spec-first programming is the most radical feature in V8, and it requires:
1. Working contracts (G42)
2. A verification strategy (at minimum runtime checks)
3. An AI integration story (how does the compiler invoke AI? Or is this a toolchain concern?)

## G51 — First-Class Functions

**Status: RESOLVED**

### Decision

Named functions are first-class values implicitly — no `.fn` syntax, no separate fn-pointer type. Function values unify with closures under `Fn(T) -> U`. Effects carry through on the type. Method references: `User.validate` is unbound (`self` is explicit first arg), `user.validate` is bound (captures `self` as a closure). Generic functions: type annotation drives inference; `sort[i64]` is the explicit specialization escape hatch.

### The Problem (original)

Kāra has first-class closures (`Fn(T) -> U`), but named functions are not explicitly first-class values. You can pass a closure anywhere, but can you pass a named function directly?

```
fn add(a: i64, b: i64) -> i64 { a + b }
fn sub(a: i64, b: i64) -> i64 { a - b }

// Should this work?
let f = add;
let ops: Vec[Fn(i64, i64) -> i64] = [add, sub];
let result = apply(add, 3, 4);
```

Or does the programmer need to wrap in a closure every time?

```
let f = |a, b| add(a, b);   // unnecessary ceremony
```

### What Most Languages Do

| Language | Named functions as values? | How |
|---|---|---|
| **Rust** | Yes | Named function coerces to `fn(T) -> U` pointer or `Fn` trait |
| **Swift** | Yes | Functions are closures |
| **Kotlin** | Yes | `.add` syntax for function references |
| **Go** | Yes | Functions are values natively |
| **JavaScript** | Yes | Functions are objects |
| **C** | Partially | Function pointers, no closures |
| **Java** | No (pre-8) | Method references (`Class.method`) added in Java 8 |

Every modern language treats functions as first-class. It's table stakes.

### Design Questions

1. **Syntax.** How do you reference a named function as a value?
   - **Implicit** (Rust/Go/JS): `let f = add` — the function name IS the value. Simple, but can be ambiguous if `add` is also a variable name (Kāra's scoping rules already handle this — functions and variables share the same namespace).
   - **Explicit reference** (Kotlin): `let f = .add` — unambiguous, but extra syntax.
   - **Implicit is better for Kāra.** Less ceremony, consistent with closures. If `|a, b| a + b` is a value, then `add` should be too — they're the same thing conceptually.

2. **Type of a function value.** A named function `fn add(a: i64, b: i64) -> i64` used as a value should have type `Fn(i64, i64) -> i64`. No separate `fn` pointer type — unify with the closure type. Simplifies the mental model.

3. **Effects.** A function with effects should carry them: `fn save(u: User) writes(UserDB)` used as a value has type `Fn(User) with writes(UserDB)`. This already works with Kāra's existing `Fn(T) -> U with _` and `Fn(T) -> U with E` syntax.

4. **Methods.** Can you reference methods as values?
   ```
   let f = User.validate;       // static method reference
   let g = user.validate;        // bound method (captures `user`)
   ```
   Bound methods are essentially closures that capture `self`. Static method references are just function values.

5. **Interaction with UFCS.** Since `user.validate()` and `User.validate(user)` are the same call, `User.validate` as a value should have type `Fn(User) -> Result[(), Error]` — the `self` parameter becomes an explicit first argument.

### Recommendation

**Named functions are first-class values, implicitly.** No special syntax needed:

```
fn add(a: i64, b: i64) -> i64 { a + b }

let f = add;                           // f: Fn(i64, i64) -> i64
let ops = [add, sub, mul];             // Vec[Fn(i64, i64) -> i64]
let result = apply(add, 3, 4);        // pass directly

// Methods work via UFCS
let validator = User.validate;        // Fn(User) -> Result[(), Error]
```

Function values unify with the closure type `Fn` — no separate `fn` pointer type. Effects carry through naturally. This is how Rust, Go, Swift, and JS all work. No reason to deviate.

### Prior Art

- **Rust:** Functions coerce to `Fn` traits. Zero-cost — function pointers are just addresses.
- **Swift:** Functions are closures. `let f = add` works.
- **Go:** Functions are values. `f := add` works.
- **Haskell:** Everything is a function. Functions are the primary value type.

## G52 — GPU Subset Constraints

**Status: RESOLVED**

`#[gpu]` is a constraint declaration ("this function uses only GPU-compatible features"), not a routing directive. Calling a `#[gpu]` function on CPU is fine; `gpu.dispatch(f, data)` is what actually sends work to GPU. Fixed-size arrays use `Array[T, N]`. `GpuSafe` trait is structural by default (compiler walks type tree), available as an explicit bound for generics. Compiler validates the full call graph from a `#[gpu]` root with chain-aware error messages. Data transfer made explicit via `reads(GpuBuffer)` / `writes(GpuBuffer)` effects.

### The Problem

design.md says Kāra compiles to GPU compute shaders, "enforced by effects (no `allocates(Heap)`, no `panics`, no recursion)." But this is incomplete. GPU hardware has fundamental constraints that go beyond effects — no dynamic memory, no recursion, no strings, no dynamic dispatch. There's no explicit spec for what Kāra features are allowed in GPU code, so the compiler can't give a clear error when you try to run incompatible code on GPU.

### What GPUs Can't Do

| Constraint | Reason |
|---|---|
| No heap allocation | GPU has no general-purpose allocator — memory is pre-allocated buffers |
| No recursion | GPU has no call stack (or extremely limited) |
| No panics / unwinding | No exception or abort mechanism |
| No `String` / dynamic strings | Requires heap allocation |
| No `Vec`, `HashMap`, `HashSet` | Requires heap allocation |
| No RC / `shared struct` | No reference counting infrastructure |
| No closures (capturing) | GPU can't capture from host memory |
| No dynamic dispatch (`dyn Trait`) | No vtables on GPU |
| No I/O effects | No filesystem, network, stdio |
| No `Option`/`Result` with heap types | `Option[i64]` is fine; `Option[String]` is not |

### What IS Allowed on GPU

| Feature | Status |
|---|---|
| Primitives (`i32`, `f64`, `bool`, etc.) | Yes |
| Fixed-size arrays (`Array[T, N]`) | Yes |
| Structs (with only GPU-compatible fields) | Yes |
| Enums (with only GPU-compatible fields) | Yes — maps to tagged unions |
| Pattern matching | Yes — compiles to branches |
| Loops (`for`, `while`, `loop`) | Yes (but no unbounded loops on some targets) |
| Generics (monomorphized) | Yes — resolved at compile time |
| Pure functions | Yes |
| Traits (static dispatch only) | Yes — monomorphized away |
| Math operations | Yes |
| Refinement types (G40) | Yes — constraint checked before GPU dispatch |
| Distinct types (G41) | Yes — zero-cost, erased at compile time |
| Layout blocks | Yes — maps naturally to GPU buffer layouts |

### Design Questions

1. **How does the user mark code as GPU-targeted?** Options:
   - **Per-function annotation:** `#[gpu] fn compute(data: ref Array[f64, 1024]) -> f64 { ... }`
   - **Per-module annotation:** `#[gpu] mod physics { ... }` — everything in the module must be GPU-compatible
   - **Inferred from usage:** If a function is passed to a GPU dispatch API (`gpu.dispatch(compute, data)`), the compiler checks compatibility at the call site
   - **Profile (G46):** `#[profile(gpu)]` on the module — ties into the profile system

2. **Error reporting.** When a function uses an incompatible feature, the error should be specific:
   ```
   error[E0801]: `String` is not GPU-compatible
     --> src/compute.kara:12:5
      |
   12 |     let name = "hello".to_string();
      |     ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
      |
      = note: GPU functions cannot use heap-allocated types
      = hint: use fixed-size `Array[u8, N]` for GPU-compatible text
   ```

3. **Transitivity.** If `#[gpu] fn foo()` calls `bar()`, must `bar` also be `#[gpu]`? Options:
   - **Explicit:** All called functions must be annotated `#[gpu]`. Verbose but clear.
   - **Inferred:** The compiler validates the full call graph from a `#[gpu]` root. If any function in the graph uses incompatible features, error at the root. Less annotation, but errors may point far from the root.
   - **Inferred with validation** (recommended): Compiler walks the call graph, errors point to the specific incompatible call with the full chain: `foo() → bar() → baz() uses String`.

4. **Shared code.** Many functions are valid on both CPU and GPU (pure math, struct operations). How to avoid duplicating code?
   - If a function has no incompatible features, it's automatically GPU-compatible — no annotation needed
   - The `#[gpu]` annotation is only needed at dispatch boundaries
   - This is similar to Rust's `const fn` — compatible functions work in both contexts without annotation

5. **Data transfer.** GPU code operates on data that must be transferred between CPU and GPU memory. This is where effects could help:
   ```
   effect resource GpuBuffer: GpuMemory;

   fn dispatch(kernel: Fn(ref Array[f64, _]) -> Array[f64, _], data: ref Array[f64, _])
       reads(GpuBuffer), writes(GpuBuffer)
   {
       // compiler handles: upload data → run kernel → download result
   }
   ```

6. **Interaction with layout blocks.** Layout groups map directly to GPU buffers — this is already in design.md. `group physics { position, velocity }` could map to a single GPU buffer for compute shaders that only touch physics data.

### Recommendation

**`#[gpu]` annotation at dispatch boundaries, inferred compatibility for called functions:**

```
// No annotation needed — pure math, automatically GPU-compatible
fn dot(a: ref Array[f64, 3], b: ref Array[f64, 3]) -> f64 {
    a[0]*b[0] + a[1]*b[1] + a[2]*b[2]
}

// GPU entry point — compiler validates the full call graph
#[gpu]
fn physics_step(positions: ref Array[Vec3, 1024], velocities: ref Array[Vec3, 1024], dt: f64) -> Array[Vec3, 1024] {
    let mut result = positions;
    for i in 0..1024 {
        result[i] = add(positions[i], scale(velocities[i], dt));
    }
    result
}

// CPU-side dispatch
fn run_simulation(world: mut ref World) writes(GpuBuffer) {
    let new_positions = gpu.dispatch(physics_step, world.positions, world.velocities, dt);
    world.positions = new_positions;
}
```

The compiler enforces GPU constraints as a **subset check** — not a new language, just a restricted subset of Kāra. Error messages explain exactly what's incompatible and why. Ties into the effect system for data transfer (`reads(GpuBuffer)`, `writes(GpuBuffer)`).

This approach avoids code duplication (pure functions work on both CPU and GPU), gives clear errors, and leverages existing Kāra features (effects, layout blocks, monomorphized generics).

### Prior Art

- **CUDA:** `__device__` / `__global__` annotations. Separate compilation. Restricted C++ subset.
- **Metal Shading Language:** Separate language (C++ subset). `kernel` functions for compute.
- **WGSL (WebGPU):** Separate language with explicit constraints.
- **Mojo:** `@parameter` for compile-time, GPU targets via MLIR.
- **Rust (rust-gpu):** Uses Rust's type system to enforce GPU constraints at compile time.
- **Futhark:** Pure functional GPU language — everything is GPU-compatible by construction.
