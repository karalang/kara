Concerns and Potential Issues
[ADDRESSED] NaN == NaN is a double-edged sword. While it eliminates bugs for beginners, it creates a semantic boundary at every FFI and serialization point. The float_in_serialized_type lint helps, but code that round-trips through JSON or MessagePack will silently break NaN-keyed maps. The design acknowledges this but the mitigation (a lint on a deferred feature) is weak for v1.

[ADDRESSED] No process.exit() is restrictive. The reasoning (bypass defer/errdefer) is sound, but ExitCode from main requires threading the exit decision all the way up the call stack. In practice, long-running servers that need to exit on signal handlers will find this painful. Consider whether a defer-respecting exit() is feasible.

[ADDRESSED] Implicit RC is invisible but not free. The "type is always T, never Rc[T]" philosophy means programmers can't see when they're paying refcount overhead. The warn_above_0 default note helps, but during rapid iteration AI agents may just suppress these. The refcount cost in hot loops could be a production surprise.

[ADDRESSED] Module-level let mut + embedded concerns. Module-level let mut with compile-time initializers is fine, but the document says concurrent mutation from ISR + main-loop uses Atomic[T] — which is deferred to Phase 10. This creates a gap for embedded use cases in v1.

[ADDRESSED] Refinement types: no flow-sensitive narrowing is a real ergonomic cost. if x > 0 { f(x) } where f: fn(Positive) requiring explicit Positive.try_from(x)? inside the branch will feel verbose in practice. The document correctly defers occurrence typing but should acknowledge this will be a top complaint.
Resolution: Added "Known ergonomic cost" note to design.md. Accepted trade-off — compiler verifies correctness from static types, not runtime values.

[ADDRESSED] collect_all limited to 8 branches. Fixed-arity overloads cap at 8. While collect_all_vec handles the homogeneous case, heterogeneous >8 requires chaining. This will feel artificial.
Resolution: Added chaining example and note that cap is due to no variadic generics; >8 heterogeneous is rare in practice.

[ADDRESSED] Three-level visibility (pub / default / private) may confuse. The default visibility being "all files in the project but not end users" is unusual. Most languages have default = private-to-module. Newcomers will likely expect default = private and accidentally expose internals within the project.
Resolution: Added "Why default is project-internal" note to design.md. Default = private-to-module would mean private-to-directory, forcing most functions to need an explicit keyword.

Minor Observations
[ADDRESSED] s[i] being a compile error on String is good (forces char_at(i) to surface O(n) cost), but will frustrate newcomers. Good diagnostic messaging is essential.
Resolution: Added newcomer-friendly diagnostic spec to design.md.

The pipe operator |> with _ placeholder and the restriction of at most one _ per stage is well-designed.

[ADDRESSED] Operator traits are homogeneous in v1 (no mixed-type arithmetic) — this is the right call for simplicity, but i32 + i64 being an error will be a frequent friction point given implicit widening exists elsewhere.
Resolution: Changed design — compiler now auto-inserts lossless widening at operator boundaries (same coercion as assignments). Only conversions in the implicit widening table are eligible.

[ADDRESSED] The sequential {} block naming is slightly confusing alongside par {}. Consider whether ordered {} or keeping just the seq {} abbreviation mentioned in the concurrency section would be clearer. (The doc uses both sequential and seq in different places.)
Resolution: Standardized to `seq {}` across design.md, glossary.md, roadmap.md.

Questions for the Designer
[ADDRESSED] Effect groups and semver: Adding an effect to a non-stable group is "minor" but may serialize previously-parallel code. Is this truly a minor change if it causes a 10x performance regression in a downstream hot path?
Resolution: Semver classification stays "minor" (open contract). Added two diagnostics: library-side note at publish time when a group expands, consumer-side note at upgrade time when parallelization changes. Library author can always manually bump major if they judge the impact warrants it.

[ADDRESSED] shared struct per-field borrow tracking: The doc says runtime borrow tracking is per-field, inserted by the compiler. What's the memory overhead? For a struct with 20 mut fields, that's 20 hidden borrow flags.
Resolution: Documented overhead — 1 byte per `mut` field per instance. Only `mut` fields carry flags. `karac explain` surfaces total overhead per type.

[ADDRESSED] Provider-rooted resource escape: The strict no-escape rule is well-motivated, but with_provider wrapping providers in Arc internally means every provider access goes through an Arc deref. Is this cost justified for single-threaded test scenarios?
Resolution: Documented why overhead is negligible — Arc deref is a plain pointer dereference; atomic ops only on clone/drop, not every access. Single-threaded test pays ~2 atomic operations total.

[ADDRESSED] blocks defaulting on all extern "C" functions: This is safe-by-default, but many C libraries are pure CPU code. Will @noblock become boilerplate on every FFI declaration in practice?
Resolution: Added block-level `@noblock` — apply to an entire `extern "C" { }` block so all declarations inside inherit it. Individual declarations can override with explicit `blocks`.
