# Brainstorming v35 — First-class data science: Tensor, shape types, numeric ergonomics

**Status: RESOLVED 2026-04-19.** All ten questions resolved and folded back into `docs/design.md` (new "Numerical Types" section, updated "Numeric Semantics," new "Script Mode" subsection), `docs/syntax.md` (shape literal grammar, Shape-kind generic param), `docs/implementation_checklist.md` (Phase 8 stdlib additions), and `docs/roadmap.md` (v1 vs v1.5 tier structure). See the resolution checklist at the bottom for the chosen options. A follow-up brainstorming on GPU call-site dispatch (per Q9) will open post-Phase-10.

---

**Original framing (kept for history).** Kāra's current scope positions it as a systems language with a REPL; this document captures the design decisions required to promote it to a first-class data-science and scripting language without forking the codebase or creating a second dialect. The framing thesis: **statically shape-typed, GPU-native, zero-copy numerical code with a notebook UX is a genuine unfilled niche.** Python is dynamically typed; Rust has no REPL; Julia compromises type safety; Mojo is proprietary. Kāra has a window.

This document covers ten design questions. The first four are **forcing functions** — decisions that must land before stdlib work calcifies around the wrong shape, because retrofitting them later would either break signatures across the whole numerical stdlib or fork the type system. The remaining six are sequencing / stdlib scope questions that can be deferred to their implementing phase as long as the forcing functions are resolved early.

Pairs with [brainstorming v34 (REPL / Jupyter)](./v34.md); the notebook UX is the delivery vehicle, this document is about what gets delivered.

## Forcing-function decisions (resolve early)

### Q1. Tensor as a first-class primitive — shape representation

**The question.** `Array[T, N]` is fixed-size 1D. `Vec[T]` is heap-allocated 1D. For data science we need N-dimensional numerical containers with shape information carried in the type. What is the canonical shape representation?

- **1A. `Tensor[T, Shape]` where Shape is a list of const-exprs or `?` for dynamic.**
  ```
  Tensor[f64, [3, 4, 5]]    // all dims known at compile time
  Tensor[f64, [3, 4, ?]]    // first two static, last dynamic
  Tensor[f64, [?, ?, ?]]    // rank 3, all dims dynamic
  ```
  Most expressive; lets the compiler elide bounds checks when shapes are fully static and do shape propagation when partially static. Requires generics to accept const-lists (extension of existing `Array[T, N]` compile-time N).

- **1B. `Tensor[T, Rank]` — rank-typed only, shapes are runtime.**
  ```
  Tensor[f64, 3]   // 3D, shape at runtime
  ```
  PyTorch's approach. Simpler type system; catches "rank mismatch" but not "dim mismatch." Most of the value of shape typing is in dim checking (`matmul`'s K dimension); this option leaves that on the floor.

- **1C. Untyped rank — `Tensor[T]` only, like NumPy.** No compile-time shape or rank info. Simple, but gives up the whole differentiator.

- **1D. Tiered — `Array1[T, N]`, `Array2[T, R, C]`, `Array3[T, ...]` for static shapes; `NArray[T]` for fully dynamic.** Cumbersome; doesn't scale past rank 4 or 5; breaks generic code over rank.

**Lean:** 1A. It's strictly more expressive than 1B, and the implementation cost is a natural extension of `Array[T, N]` — the compile-time const-list support is already partially there. The `?` escape hatch for dynamic dims preserves ergonomics when shapes are runtime-only (e.g., loaded from disk). Concretely: let type-level generics accept `[const_expr | ?]` as a Shape argument.

**Open sub-questions.**
- Does Shape have its own kind (a type-level list) or piggyback on tuple types? Leaning toward a distinct kind `Shape` so `[3, 4]` in type position parses as a shape, not a tuple.
- Shape element type: `i64` const-expr or a dedicated `Dim` type that can be `Static(i64)` or `Dynamic`? Probably a compile-time sum in the type system rather than two separate concepts.
- How does rank-polymorphism work? `fn reduce[T, ...S](t: Tensor[T, S]) -> T` needs variadic shape params. Rust handles this via traits on tuples; Kāra could do similar or use `...S` syntax.

**Action item if resolved.** Add `Tensor[T, Shape]` to design.md as a prelude type parallel to `Vec[T]` and `Array[T, N]`; specify the shape-argument grammar in syntax.md; add a new section on shape typing in the ownership/type-inference chapter.

### Q2. Shape type expressiveness — what can shapes contain?

**The question.** Given 1A, what expressions can appear in a shape?

- **2A. Literals only.** `Tensor[f64, [3, 4, 5]]` — concrete numbers. Useless for writing generic numerical code.
- **2B. Type-generic dims.** `fn sum[N](t: Tensor[f64, [N]]) -> f64` — each dim is a separate generic parameter. Works for simple cases; ugly for high rank.
- **2C. Type-generic dims + relations between params.** `fn matmul[M, K, N](a: Tensor[f64, [M, K]], b: Tensor[f64, [K, N]]) -> Tensor[f64, [M, N]]` — the compiler matches `K` on both sides at type-check time. This is where the language beats every mainstream alternative.
- **2D. Arithmetic on shape params.** `fn concat[A, B](x: Tensor[T, [A]], y: Tensor[T, [B]]) -> Tensor[T, [A + B]]` — shape expressions with arithmetic. Powerful; requires the const-evaluator to handle type-level math.
- **2E. Full dependent types.** Runtime values that flow into types. Dissertation territory.

**Lean:** 2C as the v1 commitment; 2D as a v1.5 target if the const-evaluator is up for it. 2E is not in scope.

Why 2C is the sweet spot: it handles matmul, conv, dot product, elementwise-with-same-shape, gather, scatter — essentially every operation where dim checking *catches real bugs* without requiring the type system to do algebra. 2D is a strict superset with more implementation cost; defer until a concrete use case forces it.

**Open sub-questions.**
- How are 2C shape params introduced syntactically? `fn matmul[M, K, N](...)` treats them like any generic param but they're *dim-valued*, not type-valued. A separate generic kind (`[M: Dim, K: Dim, N: Dim]`) is clearer but more ceremony. Probably: infer the kind from context (a param appearing only in shape position is a `Dim`).
- When a function is called with a fully-static shape, the generic params should be inferable from the call site (`matmul(a3x4, b4x5)` infers `M=3, K=4, N=5`). Standard generic inference; no new machinery.
- What about `?` propagation? Calling `matmul(a: Tensor[f64, [3, ?]], b: Tensor[f64, [?, 5]])` — does `K` become dynamic? Need a precise rule: `?` unifies with any concrete dim at the type level, degrading the result to a partially-dynamic shape.

**Action item if resolved.** Spec the shape-param kind rule and inference algorithm; add shape-error diagnostics to the typechecker (`expected Tensor[f64, [M, K]], got Tensor[f64, [M, K']]` with a pointer to where K was bound).

### Q3. Broadcasting semantics

**The question.** NumPy's broadcasting rules are load-bearing for numerical code (`arr + 1` adds a scalar to every element; `(3, 4) + (1, 4)` broadcasts rows). How should Kāra handle this?

- **3A. No broadcasting — shapes must match exactly.** Simple, catches bugs, hostile to numerical code. `arr + 1` requires `arr + Tensor.full(arr.shape, 1)`.
- **3B. NumPy-style implicit broadcasting with runtime shape check.** Familiar to Python users; gives up compile-time dim checking on broadcasted ops.
- **3C. Explicit broadcasting via methods.** `arr.broadcast_add(1)` or `arr + scalar(1)`. No implicit rules; compile-time checks preserved. Ergonomically heavy.
- **3D. Typed broadcasting — shape types encode compatibility.** `Tensor[T, [1, N]] + Tensor[T, [M, N]]` is statically allowed to broadcast; `[M, N] + [K, N]` where `M != K` is a compile error. Research-grade; requires the type system to understand broadcasting rules.

**Lean:** Hybrid of 3B and 3C — implicit broadcasting for **scalar + tensor** only (the 90% case), explicit methods for **tensor + tensor** broadcasts where a dim differs. So:
- `arr + 1` → works implicitly, scalar promotes to match.
- `arr + other_arr` → shapes must match exactly; compile-time error if not.
- `arr.broadcast_add(row_vec)` → explicit method for tensor-tensor broadcasting.

Rationale: scalar broadcasting is so universally expected that rejecting it would alienate every numerical user immediately, but the tricky NumPy rules (trailing-axis alignment, singleton-dim expansion) are exactly where runtime bugs breed. Making those explicit via methods preserves the shape-type safety guarantee without being hostile.

3D is the intellectually beautiful answer but gates the first data-science release on a research-grade type system. Revisit in v2.

**Action item if resolved.** Define the scalar-broadcast rule precisely (scalars are `Tensor[T, []]` or just `T`? — probably allow both and have the operator trait dispatch handle it). Spec `broadcast_add` / `broadcast_mul` etc. as stdlib methods.

### Q4. Narrow numeric promotion relaxation

**The question.** Design.md currently says "no implicit numeric promotion — `i32 + i64` requires explicit `as` cast." For numerical code this is paper-cut per expression. How much can we relax without opening the Python-style implicit-coercion can of worms?

- **4A. Status quo — all mixed-type numeric operations require `as`.** Correct, catches bugs, hostile to data science.
- **4B. Literal-involved promotion.** `arr + 1` works (the literal `1` promotes to the element type). `arr + other_var` still requires matching types. Narrow, safe, handles the 95% case.
- **4C. Safe widening.** Promotions that lose no information happen implicitly: `i32 + i64 → i64`, `i32 + f64 → f64`, `f32 + f64 → f64`. Narrowing still requires `as`. Rustaceans will hiss; Python users will be happier.
- **4D. Full Python/NumPy-style coercion.** Any numeric type mixes with any other. `i32 / i32 → f64` if it doesn't divide evenly. Gives up type safety in a way that bleeds through the whole language.

**Lean:** 4B for v1; consider 4C if real-world data-science code hits the literal-only rule as friction. 4D is off the table.

4B is a minimum-ceremony change: the typechecker already knows literals have a pseudo-type until bound; extending that to "a literal appearing in a binary op with a typed value takes that value's type" is a small coercion rule, not a typesystem overhaul. It preserves the invariant that *two typed expressions* always follow the same rules everywhere (no numerics dialect).

4C is more helpful but more invasive — it changes the meaning of `i32 + i64` silently, which has precedent footguns (Java's `int + long`, JavaScript's `5 + "5"`). Hold for now.

**Action item if resolved.** Amend design.md § Operator Traits: add the literal-promotion clause. Extend the lowering pass to inject a cast for literals in mixed-type binary operations. Update `tests/codegen.rs` and `tests/typechecker.rs` with explicit cases.

## Binding but lower-urgency decisions (resolve before stdlib work)

### Q5. Arrow-compatible memory layout commitment

**The question.** Should `Tensor` and the future `Column`/`DataFrame` types use Apache Arrow's memory layout as their canonical in-memory representation?

- **5A. Yes — Tensor/Column are Arrow-format in stdlib.** Zero-copy interop with pyarrow (Python onramp), Polars, DuckDB, Parquet, the whole modern data ecosystem. Commits stdlib binary layout but the layout is already the de-facto standard.
- **5B. Separate `ArrowTensor` for interop; native Tensor keeps Kāra's own layout.** Two types, conversion cost between them, second-class feel.
- **5C. Uncommitted — let libraries choose per-case.** Everyone reinvents and nothing interops.

**Lean:** 5A. Single decision, massive downstream dividend. Arrow's layout (null-bitmap + contiguous-buffer-per-column, Arrow IPC for cross-process) is well-designed, already the winning standard, and happens to align with what Kāra would build from first principles anyway (SoA friendly, null-bitmap is what we'd pick).

Strategic angle: a new language's stdlib choosing Arrow immediately signals to data-practitioner readers "these people know the ecosystem." It's a credibility decision as much as a technical one.

**Sub-decisions that fall out of 5A.**
- Null representation is bitmap-based (one bit per element, 0 = null, 1 = valid). See Q7.
- `Tensor` memory is contiguous per-dim-order (C-order or F-order — pick one, align with Arrow's convention, document).
- Dictionary-encoded columns, run-end encoding, and fixed-size-list all become natural stdlib types later.

**Action item if resolved.** Add § "Memory Layout Commitments" to design.md stating Arrow conformance for `Tensor`, `Column`, `DataFrame`, `String` (UTF-8, which Arrow already expects). Pin `Tensor`'s buffer layout in Phase 7.2-style codegen spec.

### Q6. Nullable array primitive — how is "missing" encoded?

**The question.** Numerical code needs fast nullable columns. `Vec[Option[f64]]` is disastrous (indirection per element, 50%+ memory overhead). What does stdlib offer?

- **6A. Bitmap-backed nullable column.** `Column[T]` stores `{ ptr data, ptr null_bitmap, len, capacity }`. One bit per element signals validity. Zero overhead per element; ~12% extra memory for the bitmap. Matches Arrow directly.
- **6B. NaN-only for floats, Option arrays for everything else.** Free for floats (NaN is a sentinel); expensive for integers and categorical data.
- **6C. Both — bitmap-backed `Column[T]` for general data; NaN permitted as a per-element convention for floats and handled by methods like `fillna`.** Programmer chooses per-case.

**Lean:** 6C. Arrow compatibility forces 6A in stdlib anyway; NaN is a convention users rely on in float data and we shouldn't reject it. The stdlib exposes both: `Column[f64]` has a bitmap and users can also write `f64::NAN` as a per-element sentinel if they want.

**Sub-questions.**
- Does `Tensor[T, Shape]` have a nullability option or is it dense-only? Leaning dense-only for Tensor (numerical computing assumes validity); nullability is a `Column` concern. Keeps the core numerical primitive fast.
- Arithmetic on nullable columns: null + anything = null (SQL semantics). Stdlib methods respect this.

**Action item if resolved.** Add `Column[T]` to stdlib plan (Phase 8+); spec null-propagation semantics; implement bitmap-aware arithmetic in the stdlib impl blocks.

### Q7. Script mode — files without `fn main`

**The question.** A one-off script should not need `fn main() { ... }` wrapping. Python lets you type `print(1 + 1)` and run the file. Can Kāra?

- **7A. Files without `fn main` run their top-level statements as an implicit main.** Ergonomic match with Python/Ruby. Existing files with `fn main` are unchanged.
- **7B. Explicit `#[script]` or `#!` shebang trigger.** Opt-in; preserves "every program has a main function" as a structural invariant.
- **7C. Don't support — keep `fn main` required.** Fine for compiled-only deployment; hostile to scripting.

**Lean:** 7A. Cheap to implement (the parser already accepts top-level items; extend to accept statements in a script context and synthesize a `main`). Users moving from Python notebooks hit this friction immediately; the fix is trivial. The structural invariant under the hood is unchanged — there's still a main function, the compiler just wrote it.

**Sub-question.** What effects does the synthesized main declare? Probably `pub fn main() { ... }` with inferred effects (treating it as private for effect-inference purposes). If script mode + REPL effect inference (brainstorming v34 Q4) are consistent, this aligns.

**Action item if resolved.** Parser + resolver change: accept statement-level items at file top; synthesize `fn main` wrapping them. One-line note in the book under "Running a single-file script."

### Q8. Lazy evaluation + pipeline fusion

**The question.** `arr.filter(f).map(g).sum()` in NumPy/Pandas eagerly allocates an intermediate array per step. Polars and modern data frameworks fuse the pipeline into one loop. Should Kāra's stdlib ship eager (like NumPy) or lazy (like Polars)?

- **8A. Eager — methods return materialized collections.** Simple, predictable, wasteful on long pipelines.
- **8B. Lazy — chained methods return an iterator/query that evaluates at `.collect()` / terminal ops.** Efficient, matches modern data ecosystem, can be library-level without language support. Requires discipline around "when does this actually run?" which intersects with the effect system.
- **8C. Both surfaces — eager `Vec`/`Tensor` methods, lazy `Stream[T]` / `LazyColumn[T]` with explicit `.collect()`.** Programmer picks per use case.

**Lean:** 8C. Lazy should not be the default (the "where does this run" question is confusing), but should be natively supported for exactly the pipelines that need it. Kāra's effect system is actually a superpower here: a `LazyColumn.filter(f).map(g)` pipeline inherits the effects of `f` and `g` at the type level, so the compiler knows the deferred computation's effect set without any extra machinery.

This is a v1.5 feature: language-level support is trivial (Kāra already has closures and iterators); the work is stdlib API design.

**Action item if resolved.** Phase 8+ stdlib item: `LazyColumn[T]` with `filter`, `map`, `fold`, `collect`. Document that eager methods are the default on `Column` and `Tensor`; lazy methods require explicit `.lazy()`.

### Q9. GPU as call-site backend (not just compile target)

**The question.** Phase 10 adds WASM and GPU compute shaders as compile targets. For data science, GPU should be callable *from within* a CPU program at the call site — `arr.on(gpu).map(f).collect()` — not a whole-program deployment mode. How is this surfaced?

- **9A. Method-level backend selection.** `.on(gpu)` returns a `GpuTensor[T, Shape]`; subsequent ops run on GPU; `.to_cpu()` moves back. Programmer controls per-call.
- **9B. Profile-based dispatch.** A compilation profile says "this function runs on GPU." Similar to Rust's `#[target_feature]`. Function-level, not call-level.
- **9C. Effect-based dispatch.** Add a `runs_on(GPU)` effect; the effect system routes ops. Clean integration with the rest of the language but conceptually novel.
- **9D. Defer — Phase 10 ships GPU as target-only; call-site ergonomics are a v2 concern.**

**Lean:** 9A with 9C as a longer-term evolution. 9A is the pattern users know (CuPy, PyTorch), ships incrementally, and doesn't require new language primitives — just stdlib types (`GpuTensor`) and codegen plumbing. 9C is beautiful but high-risk; don't design the GPU story around an unproven effect pattern.

Defer full design to a follow-up brainstorming once Phase 10 scoping concretizes.

**Action item if resolved.** No action yet — document as a pending design decision in design.md's GPU section; revisit when Phase 10 starts.

### Q10. Data-science stdlib scope — what ships in v1?

**The question.** How much of the data-ecosystem does stdlib cover vs. external libraries?

- **10A. Stdlib is minimal — just Tensor, Column, DataFrame, CSV reader, Parquet reader. Everything else is user-space.**
- **10B. Stdlib is medium — add regex, HTTP client, process spawn, JSON, basic stats (mean, stddev, percentile).**
- **10C. Stdlib is opinionated — ship a full Polars-equivalent DataFrame library, plotting primitives, ML basics.** Batteries-included like Python.

**Lean:** 10B for v1, with 10A as a fallback if Phase 8 is running long. The primitives for data science (Tensor, Column, Arrow I/O, stats, regex, HTTP) are small enough to ship as stdlib; heavier abstractions (query optimizer, plotting, ML) should emerge as community libraries once the primitives are stable. Don't let stdlib ambition delay shipping the language.

Scripting-critical additions (regex, HTTP, process spawn) should be Phase 8 non-negotiables — without them, "scripting" feels second-class even if Tensor is perfect.

**Action item if resolved.** Extend Phase 8 stdlib plan in implementation_checklist.md with: Tensor, Column, DataFrame (minimal), Arrow IPC, Parquet, CSV, JSON, Regex, HTTP client, Process, Stats. Sequence within Phase 8; don't block earlier work.

## Meta — acknowledged gaps

Positioning principle: **understate, over-deliver.** The defensible claim is "serious attempt at a language equally comfortable in a kernel, a server, a notebook, and a GPU shader" — not "one language to rule them all." Honesty about where the design deliberately stops is part of the positioning, not a weakness to hide.

The gaps below are not action items on this doc. They are diagnostic — an inventory so the team, contributors, and early users understand what Kāra will and won't be. Some are permanent scope decisions. Some are ecosystem problems that no language design can solve unilaterally. Some are deferred technical work with plausible future paths. A few are tooling gaps that will matter for adoption but are orthogonal to language semantics.

### Permanent non-goals (deliberate scope boundaries)

- **Logic programming / Prolog-class.** A different paradigm that would fracture the language. If you need unification and backtracking, Kāra is not your tool.
- **APL/K-style terse array syntax.** Cultural mismatch with Rust-influenced syntax. Shape-typed tensors are the numerical ergonomics target; array-language terseness is not.
- **Full dependent types.** Level 3-4 verification is deferred indefinitely per roadmap. Users needing proof-assistant-grade expressivity reach for Coq / Lean / Idris.
- **Async / await syntax.** Deliberately absent — the effect-based concurrency model is the intended replacement. Readers expecting `async fn` / `.await` will not find them; that is a positioning decision, not a gap.
- **Dynamic typing surface.** No gradual typing, no `Any`, no runtime type identity tricks. Kāra is statically typed end-to-end; Python users who want "just don't annotate" ergonomics have type inference as the compromise, not dynamic dispatch.
- **Inheritance / classical OOP.** `shared struct` gives reference semantics but not class hierarchies. Traits cover polymorphism. Programs expecting deep inheritance trees will feel the absence.

### Ecosystem gaps (not solvable by language design)

- **Frontend UI toolkit.** React / SwiftUI / Qt / Flutter-class. Limits web-frontend and desktop-app development. Solvable by community libraries once WASM (Phase 10) lands, but stdlib will not ship one. Breaking for frontend-centric teams.
- **Mobile platform SDK bindings.** iOS (Swift/ObjC) and Android (Kotlin/Java) native APIs. Platform-controlled surfaces; binding work scales with vendor releases and is perpetual. Breaking for mobile-first development.
- **Game engine.** Unity / Unreal / Bevy-class. The SoA layouts and GPU story are foundational, but an engine is years of domain-specific work beyond the language.
- **Pre-existing stdlib scale.** Python's stdlib is 30 years deep across `os`, `subprocess`, `datetime`, `collections`, `itertools`, `functools`, `socket`, `email`, and a hundred others. No new language matches that in v1 or v2. Kāra's v1 stdlib will feel thin; that is inherent, not a failure.
- **Scientific / ML library ecosystem.** sklearn, statsmodels, scipy, pytorch, jax. Years of research-grade implementations. Arrow interop (Q5) partially bridges this for data; algorithmic libraries must be regrown.
- **Pre-existing debugging / profiling tools.** perf, Instruments, gdb-with-Rust-pretty-printers, etc. — these wrap around language runtimes that already exist. New language = new tool-integration work.

### Deferred technical gaps (plausible future paths, not in v1)

- **Autodiff.** Breaking for ML research and gradient-based optimization. JAX-class autodiff is a research program of its own. Retrofit path: either via comptime (when deferred comptime ships) or via dedicated derivative-tracking types with trait-based operator lifting. Months-to-years of design work regardless of path. Data science that is *not* ML (analytics, stats, dashboards) does not require autodiff.
- **Supervision trees / Erlang-class fault tolerance.** Degraded, not breaking. Structured concurrency covers in-process needs; let-it-crash + supervisor patterns are a known complement that v1 does not attempt. Retrofit path: a stdlib actor-like layer atop the effect system; feasible but not scoped.
- **Lisp-class metaprogramming.** Comptime is deferred per roadmap. Programs that rely on heavy AST manipulation, DSL hosting, or aspect-oriented patterns will reach for external tooling. v1.5+ story if it ships at all.
- **Research-grade type system features.** Higher-kinded types, type families, GADTs, rank-N polymorphism. v1 focuses on pragmatic shape types (Q2); Haskell / Scala / Idris remain the choice for users who need HKT.
- **SMT-backed formal verification.** Level 2 (refinement types) and Level 2.5 (runtime-checked contracts) ship per Phase 9. Level 3-4 (Z3 integration, full invariant proofs) are deferred indefinitely.
- **Symbolic computation.** SymPy-class symbolic math. Library-level; possible atop comptime once comptime lands; not a stdlib goal.
- **Distributed systems primitives.** No distributed transactions, no consensus protocols, no actor messaging across nodes. Cluster-level systems would build *on* Kāra; Kāra does not build them in.
- **Hard real-time guarantees.** Soft real-time is accessible via the embedded profile and `allocates(Heap)` effect tracking, but hard real-time (worst-case latency proofs, deterministic GC absence, formal WCET analysis) is beyond v1.
- **Control-flow interop with Python / R / Ruby.** Arrow memory interop (Q5) gives zero-copy *data* exchange with Python; *calling* Python from Kāra or embedding Kāra in a Python process requires bridge work equivalent to PyO3. Breaking for teams with heavy existing Python codebases who want gradual migration.

### Tooling / maturity gaps (real near-term adoption blockers)

- **Package manager.** Deferred to Phase 7-8 per design.md. Breaking at scale for scripting adoption and any multi-file project: users cannot share, version, or install scripts without it. Must land before the ecosystem can form. Roughly cargo-equivalent scope; not small work.
- **Language server (LSP).** Roadmap marks this as v2+, post-self-hosting. Breaking for IDE-grade developer experience. Tree-sitter grammar + incremental-query architecture is a multi-quarter project even with the compiler as a library.
- **Debugger integration.** Not currently scoped. DWARF emission, DAP protocol implementation, Kāra-aware pretty-printers. Breaking for serious systems work. Realistic for Phase 10+ once LLVM debug-info emission is turned on.
- **Incremental compilation.** Large-codebase compile times. Deferred per the reactive-query-based compilation direction (roadmap Future section). Will bite once real projects grow beyond a few thousand lines.
- **Cross-compilation ergonomics.** Phase 10 lists WASM / GPU / FPGA as targets but does not detail the cross-toolchain UX (rustup-target-add-style workflow). A real ergonomics gap for deployment-heavy teams.
- **Profiler / flamegraph story.** Effect-annotated profiling is a potential differentiator (attribute time to effect resource), but no design work exists yet. Breaking for performance-critical adopters.
- **Stable ABI / binary-compatibility story.** Currently compiler+runtime are bundled (per design.md § Runtime Distribution). Independent library distribution (publish a `.a` / `.so` usable across Kāra versions) is explicitly out of scope for v1. Limits third-party library distribution.

### What this inventory tells us

Three patterns fall out:

1. **The interesting gaps are ecosystem and tooling, not language-level.** The language design genuinely covers most important computing paradigms. The biggest near-term obstacles to adoption — package manager, LSP, debugger, autodiff — are build-the-infrastructure projects rather than design-the-semantics projects. That is a good problem to have.
2. **The breaking adoption-blockers for v1 are autodiff (ML research), package manager (any serious scripting), and LSP (any serious developer team).** Prioritization should reflect that. Autodiff is the only one that is substantially a language-level design question; the other two are execution work.
3. **The permanent non-goals are few and deliberately chosen.** Not being Prolog or Haskell or APL is a positioning decision, not a limitation. Saying so out loud gives the language identity and prevents the trap of trying to be everything to everyone.

**Honest summary.** The design can plausibly cover 80%+ of important computing domains to a first-class standard. It will not cover 100%, and pretending otherwise would undermine the more defensible claim that Kāra is genuinely excellent at the 80% it addresses. The list above is the 20% — a mix of "permanently out of scope," "ecosystem-shaped," and "real work that takes time." Being transparent about this list is part of the "understate, over-deliver" posture: users know what they are signing up for, and the project is not perpetually judged against an impossible benchmark.

## Meta — positioning risk

The reason to do this now is not "to compete with Python in v1" — that's a losing framing, and the ecosystem scale gap is a decade of work. The reason is that **the language's bones need to be shaped for numerical code if it's ever going to carry that use case.** Adding shape types or Arrow layout or scalar broadcasting *later* means either a forked dialect or a stdlib API break. Adding them *now* costs a few months of design and preserves the option value forever.

If in 18 months the data-science bet isn't panning out, the shape-type infrastructure still pays for itself in systems code (safe FFI with C tensor libraries, image/audio processing, graphics). If it *is* panning out, Kāra has a genuine unoccupied position.

The moon is in reach if the bones are right.

## Resolution checklist

### Forcing functions (resolved)

- [x] **Q1: 1A `Tensor[T, Shape]` with `?` for dynamic dims.** Shape is a new generic-parameter kind (alongside type and const-expr kinds). Shape literal `[3, 4, ?]` is new grammar in type-argument position — `?` is reused (already lexed as an expression operator) as a dynamic-dim marker. No new keywords — `Tensor` is a prelude type, same bucket as `Vec`. Variadic rank polymorphism uses `...S` syntax. Folded into `design.md § Numerical Types > Tensor` and `syntax.md § 6.2 / § 6.4`.

- [x] **Q2: 2C type-generic dims with relations (`matmul[M, K, N]`) for v1.** Compiler unifies `K` on both sides at type-check time. Dim-kind inferred from context (no explicit `M: Dim` annotation ceremony). Standard generic inference at call sites. `?` unifies with any Dim param and degrades the result type's corresponding position to `?`. 2D arithmetic on shape params deferred to v1.5 pending const-evaluator readiness; 2E full dependent types out of scope. Folded into `design.md § Numerical Types > Tensor`.

- [x] **Q3: Hybrid 3B+3C — implicit scalar-tensor broadcasting, explicit methods for tensor-tensor.** `arr + 1.0` works; `arr + mismatched_tensor` compile errors; `arr.broadcast_add(row_vec)` is the explicit method form. Scalar is both `T` and `Tensor[T, []]` — operator trait handles both identically. 3D typed broadcasting revisited in v2. Folded into `design.md § Numerical Types > Broadcasting`.

- [x] **Q4: 4B literal-involved promotion, scoped to numeric binary operators.** A literal in a binary op with a typed operand takes that operand's type. Two typed expressions still require matching types (no silent widening between vars). Function calls and patterns keep strict literal-matches-declared-type. 4C safe widening revisited if real-world friction warrants. Folded into `design.md § Numeric Semantics > Literal-involved promotion`.

### Binding but lower-urgency (resolved)

- [x] **Q5: 5A Arrow conformance for Tensor, Column, DataFrame.** Buffer layout is specification-layer commitment — breaking change to modify. Null representation bitmap-based (1 bit per element). Tensor memory contiguous in C-order (row-major). UTF-8 strings (already Kāra commitment). Enables zero-copy interop with pyarrow, Polars, DuckDB, Parquet, Arrow IPC. GPU-compatibility note: Arrow layout maps cleanly to GPU device buffers (cuDF precedent). Folded into `design.md § Numerical Types > Memory Layout Commitments`.

- [x] **Q6: 6C bitmap-backed `Column[T]` + NaN convention for floats; dense-only `Tensor`.** Column[T] is the general nullable primitive (matches Arrow); NaN permitted as per-element float convention; fillna/dropna handle either. Tensor has no nullability — numerical computing assumes validity per element (also GPU-friendly: cuBLAS/BLAS kernels assume dense). Arithmetic on nullable columns uses SQL null-propagation (null + x = null). Folded into `design.md § Numerical Types > Column`.

- [x] **Q7: 7A files without `fn main` synthesize one.** Top-level statements form the body of an implicit `fn main() -> Result[Unit, Error]`; effects inferred. Items (fn, struct) hoist alongside statements; statements form main body in file order. Error if file has both top-level statements AND explicit fn main. Aligns with v34 REPL cell-as-main-body model — one conceptual story across file and REPL surfaces. Folded into `design.md § Entry Point > Script Mode` and `syntax.md § 2 Program Structure`.

- [x] **Q8: 8C lazy surfaces as v1.5 feature.** Eager default on Vec/Tensor/Column. Lazy opt-in via `.lazy()`. `Iterator[T]` is the language-level lazy primitive; `LazyColumn[T]` / `LazyTensor[T]` are stdlib specializations carrying schema/shape. Effect system composes — lazy pipeline inherits closure effects at type level. GPU-compatibility note: lazy is load-bearing for GPU kernel fusion (whole-pipeline visibility enables single-kernel-launch codegen); lazy API should be revisited alongside the GPU brainstorming post-Phase-10. Folded into `roadmap.md § Phase 8 > Lazy evaluation (v1.5)`.

- [x] **Q9: 9D for now, 9A as stated future direction.** v1: no GPU call-site support; Phase 10 ships GPU as compile-target only. Post-Phase-10: follow-up brainstorming opens to concretize 9A-style `GpuTensor` + `.on(gpu)` / `.to_cpu()` pattern (matches CuPy, PyTorch, JAX device semantics). 9C effect-based dispatch revisited in v2+ if effect-dispatch pattern has crystallized from other work. Numerical stdlib from Q1-Q6 composes with whatever GPU story eventually lands — trait-dispatched reduce / map / element-wise ops (Q10 annotation) keep the API surface open. Folded into `design.md § Numerical Types > GPU Backend (deferred)` and `roadmap.md § Phase 10 follow-up`.

- [x] **Q10: 10B medium stdlib scope for v1.** Ship-list: Tensor, Column, DataFrame, Arrow IPC, Parquet, CSV, regex, http (client only), process, json, basic stats (mean/stddev/percentile/min/max/sort/argsort). Fallback: regex + http are most at-risk if Phase 8 slips — pushable to v1.1; rest are cheap enough to keep. GPU-compatibility note: stats / reduce / map ops are trait-dispatched (`Reduce`, `ElementwiseMap`) rather than hardcoded to CPU types, so `GpuTensor` can implement the same surface later without API break. 10C opinionated stdlib (Polars-equivalent DataFrame, plotting, ML) off the table — community libraries. Folded into `implementation_checklist.md § Phase 8`.
