# v60 — Kāra v1 action list (from Rust 1.0 → 1.85 sweep)

Distilled action list. Per-release notes, edition deep-dives, RFC histories, and tier-justifications were stripped — only the "do this in design.md / parser / stdlib" verdicts remain.

Tiers: **S** = adopt at v1, no question · **A** = adopt at v1, design carefully · **B** = settle position at v1, defer impl if needed · **C** = lower priority but known-good · **E** = soundness/position decisions to pin in `design.md` · **Future** = reservations for when macros/editions land.

## Tier S — Adopt at v1, no question

1. **Const generics — full surface.** Spec `[T, const N: usize]` with integer/bool/char/enum const params and const expressions in bounds. Add to `design.md` generics section. Don't ship Rust's MVP path.

2. **Match/binding ergonomics gap-fill.** Audit Phase 2 parser + Phase 3 typechecker for: `let-else`, match ergonomics, field-init shorthand, `..=` patterns, subslice patterns. Confirm or add.

3. **`unsafe extern { ... }` as the only legal extern form.** Require `#[unsafe(...)]` wrapping for any attribute that affects soundness / layout / linking. Add to `design.md` FFI + attribute sections.

4. **`unsafe_op_in_unsafe_fn` rule.** `unsafe fn` declares a precondition only — body still requires inner `unsafe { }` blocks for unsafe operations. Pin in `design.md` unsafe section.

5. **No `static mut`.** Pin as deliberate omission in `design.md` with one-line rationale. Mutable globals via `Mutex[T]` / `Atomic[T]` / `OnceCell[T]` only.

6. **`#[track_caller]` for stdlib panic-emitters.** Add attribute to `design.md`; apply to all `unwrap` / `expect` / indexing / overflow / div-by-zero panic sites in stdlib.

7. **`#[non_exhaustive]` on evolvable stdlib types.** Add attribute; mandate on stdlib enums/structs where field/variant additions are expected post-v1.

8. **`#[must_use]` mandate.** Add attribute; apply to `Result`, `Option`, every iterator adapter, every guard type (mutex/lock/refcell), every builder return.

9. **Reserve future keywords.** At v1: `gen`, `become`, `do`, `final`, `override`, `priv`, `try`, `typeof`, `virtual`, `async`, `await`, `comptime`, `pure`, `move`, `box`. Lexer-level "reserved" diagnostic on use.

10. **Reserve single-letter prefix-string syntax.** `a"..."` through `z"..."` reserved at v1; "reserved prefix" diagnostic on unrecognized.

11. **Reserve `#`-guarded string syntax.** `#"..."#`, `##`, `###`, and `#`-prefixed forms reserved at v1.

12. **`MaybeUninit[T]` only.** Never offer `mem::uninitialized()`. If `mem::zeroed[T]` is offered, bound it `T: Pod`.

20. **Strict-provenance pointer APIs.** ✅ Decided 2026-05-02 — full spec landed in design.md § Pointer Provenance. Adopts Rust's strict-provenance model: pointers carry both an address and provenance, and the `as`-cast form silently conflates the two. The `ptr` namespace splits the operations: `addr(p) -> usize` (read address bits, provenance stays attached, safe), `with_addr(p, addr) -> *const T` and `with_addr_mut` (reseat an existing pointer's address — provenance preserved, safe; bounded by the source's allocation, so cannot escape), `expose(p) -> usize` and `expose_mut` (read address bits *and* publish provenance to the global "exposed" set for later round-tripping, safe), `from_exposed[T](addr) -> *const T` and `from_exposed_mut` (synthesise a pointer from an exposed address, *unsafe* — caller asserts the prior `expose` and that the pointee is still live). The four casts `*const T as usize` / `*mut T as usize` / `usize as *const T` / `usize as *mut T` are rejected at typecheck (`E_PTR_TO_INT_CAST_FORBIDDEN` / `E_INT_TO_PTR_CAST_FORBIDDEN`). Pointer→int rejection includes a suggested-fix span pointing at `ptr.addr(p)` (the safer default); int→pointer rejection has no auto-fix because the choice between `with_addr` and `from_exposed` depends on whether a base pointer is in scope. Raw-to-raw casts (`*const T as *mut T`) and integer-to-integer casts (`usize as i64`) remain unaffected — only the kind-crossing cast is forbidden. Updated the existing § Raw Pointer Construction cast-rejection paragraph (which had said "*const T as usize via expose is unaffected") to point at the new strict-provenance section. Updated `ptr` prelude module entry and syntax.md § 5.19 Cast Expressions to mention the four new APIs. Implementation tracked under Phase 5.2 in implementation_checklist.md (typechecker cast-rejection + stdlib `ptr` extensions; codegen lowering to LLVM `ptrtoint` / `inttoptr` with provenance-preserving metadata in Phase 7).

22. **`union` type for FFI.** ✅ Decided 2026-05-02 — full spec landed at design.md § FFI Unions. The v60 phrasing `extern "C" union Foo { ... }` was a Rust-leaning shorthand; Kāra spells the same shape `#[repr(C)] union Foo { ... }` so it composes with the existing `#[repr]` ABI table rather than introducing a parallel `extern`-prefixed item form. Design pin: `union` is a **separate kind** from `enum` (enum is the *tagged* variant type with discriminant tracking and per-variant exhaustiveness checking; union is the *untagged* FFI shape with byte reinterpretation), top-level item, must carry `#[repr(C)]` (or `#[repr(C, packed)]`) — bare `union Foo { ... }` is rejected (`E_UNION_REQUIRES_REPR`); every field type must be `Copy`/`Pod` (rejects `Drop` field types and removes the drop-ordering footgun); construction literal names exactly one field (multi-field and zero-field literals rejected); field reads, borrows, and pattern destructures require `unsafe { }`; field assignment is safe (Pod write); `ptr.const(u.field)` / `ptr.mut(u.field)` are safe (no reinterpretation); `impl Drop` rejected; `#[derive(...)]` rejected; `#[non_exhaustive]` rejected; no generics in v1 (additive later). Layout: `size = max(field_sizes)`, `align = max(field_aligns)`. Cross-references: design.md `#[repr]` table updated to mention union as a permitted target for `#[repr(C)]`; syntax.md adds `union` to the Declarations keyword block, adds `UNION_DEF` to the `ITEM` rule, and adds § 3.3a Unions (FFI). Implementation tracked under Phase 5.2 in implementation_checklist.md alongside C-string literals (item 18) and the `unsafe extern { }` block requirement (item 3) as the FFI-tightening cluster. Worked example in design.md uses `epoll_data` to show the canonical FFI-binding pattern: declare the union as a top-level type and reference it from `unsafe extern "C" { }` block signatures, mirroring the `#[repr(C)] struct` pattern.

24. **`extern_types` for opaque FFI.** ✅ Decided 2026-05-02 — full spec landed at design.md § Opaque Foreign Types: `extern "C" { type Foo; }`. The form is `type Foo;` *inside* an existing `unsafe extern "ABI" { ... }` block (not a top-level `extern "C" type Foo;`, which would mismatch the existing extern-block discipline established by item 3). Updated the § `unsafe extern { ... }` block requirement section to enumerate two legal item kinds inside the block: foreign function declarations and opaque foreign type declarations. Design pin: opaque types are *unsized* (`Sized = false`), exist only behind a pointer or reference; the dedicated kind (separate from `struct` and `union`) makes the no-known-layout contract honest, since a phantom-field struct would either lie about `size_of` or commit to a fictional layout that handcuffs the C library author. Permitted operations: `*const Foo` / `*mut Foo` / `ref Foo` / `mut ref Foo` everywhere a pointer/reference type is permitted; FFI signatures using the type behind a pointer/reference. Forbidden operations (each with a focused diagnostic): by-value bindings (`E_OPAQUE_TYPE_REQUIRES_INDIRECTION`), by-value parameters/returns, as a non-indirect struct/array field, field access (`E_OPAQUE_TYPE_NO_FIELDS`), method calls (`E_OPAQUE_TYPE_NO_METHODS`), `size_of` / `align_of` (`E_OPAQUE_TYPE_NO_KNOWN_SIZE`), `impl ... for Foo` whether inherent or trait (`E_OPAQUE_TYPE_NO_INHERENT_OR_TRAIT_IMPLS`), `#[derive(...)]`, generics (`E_OPAQUE_TYPE_GENERIC_FORBIDDEN`). The wrapper-type idiom (`distinct type SafeFile = *mut FILE; impl Drop for SafeFile { fn drop(...) }`) is the supported way to attach Kāra-side trait conformance and `Drop`-routed cleanup (worked example with `fopen`/`fclose` in design.md). Trust-not-verify continues to apply: the C side guarantees the type exists and that pointers Kāra receives are valid. Cross-version stability is the whole point — opaque types are layout-version-independent, while `#[repr(C)] struct` is the form when the C library publishes its layout. Implementation tracked under Phase 5.2 in implementation_checklist.md alongside items 3, 18, 22 as the FFI-tightening cluster.

25. **`offset_of(MyStruct, field)`.** ✅ Decided 2026-05-02 — picked option (a) `offset_of[T](field)`, extending Kāra's existing `size_of[T]()` / `align_of[T]()` pattern. Type lives in the generic-arg slot; field is a special-form positional argument (a dotted field-path identifier sequence, not a value expression). Rejected option (b) `offset_of(T, field)` (Rust-macro shape — both args positional and special) because it does not pattern-match the existing `*_of` family. Rejected option (c) `T.offset_of(field)` (type-static method form, parallel to `MaybeUninit[T].uninit()` / `Type.try_from()`) because the `*_of` queries are conceptually a single family that should share a shape, and `size_of` / `align_of` already use the bracket-generic form. Companion stdlib `ptr.container_of` / `ptr.container_of_mut` lands in the same item — the intrusive-DS use case is concentrated enough to warrant a helper, lowered via the strict-provenance APIs from item 20 (`field_ptr.with_addr(field_ptr.addr() - offset_of[T](field))`). 26. **`catch_panic` primitive.** ✅ Decided 2026-05-02 — full spec landed at design.md § Catching Panics: `catch_panic[T]`. The pre-v60 design forbade `catch_panic` ("panics are unrecoverable by design"); item 26 reverses that for two specific use cases: work-stealing schedulers (the runtime already catches panics internally at task boundaries; `catch_panic` is that mechanism surfaced as a user-facing API) and FFI boundaries (Kāra functions exposed to C must absorb panics before the unwind crosses into a non-Kāra frame; `extern "C-unwind"` calls into Kāra-side code symmetrically need an absorption point). The signature is a `#[compiler_builtin]` with closed-form effect arithmetic: `fn catch_panic[T, with E](f: Fn() -> T with E) -> Result[T, PanicInfo] with (E - panics)` — non-user-writable because Kāra has no source-level way to express "remove `panics` from an effect set". `PanicInfo` carries `message`, `location`, `kind` (`#[non_exhaustive]` enum: `Explicit`, `Unwrap`, `Indexing`, `IntegerOverflow`, `DivisionByZero`, `ContractViolation`, `Foreign`, `DropDuringUnwind`), `thread_name`, `backtrace`; `Send + Clone`. Caught: explicit panics, implicit panics (unwrap/index/overflow/div-zero), contract violations, transitive panics from any callee, `extern "C-unwind"` foreign exceptions (`PanicKind.Foreign`). NOT caught: `process.exit()` (propagates via the runtime "Exit" tag — the catch frame inspects the tag and re-raises), `abort()`, double-panics (always abort). Build-profile gate: `catch_panic` is unavailable under `panic = "abort"` (kernel/embedded/ISR profiles) — calls produce `error[E_CATCH_PANIC_REQUIRES_UNWIND]`; `extern "C-unwind"` is rejected at parse under the same profiles. Auto-concurrency interaction: a panic absorbed inside a single branch of a `par`/`collect_all` scope produces `Ok(Err(PanicInfo))` from that branch — peer cancellation is NOT triggered (per-branch isolation, exactly what custom schedulers need). Re-raise: no method on `PanicInfo`; the caller writes `panic!(f"{info.message}")` to propagate, deliberately keeping the new panic site visible in stack traces rather than smuggling the original location forward. Updated the existing § Panic semantics paragraph to remove the "User code cannot catch panics — there is no `catch_panic` facility; panics are unrecoverable by design" stance and forward-reference the new section. Updated the `process.exit()` paragraph to confirm uncatchable behavior via the runtime exit tag. Updated the `extern "C-unwind"` § First-release constraint paragraph (now § Absorbing the panic at the FFI boundary) to reference `catch_panic` as the supported absorption form rather than punting to a deferred entry. Updated the deferred-items table row for `extern "C-unwind"` exception absorption from "Phase 7+" / "deferred" to "Phase 7" / "Specced as catch_panic". Resolved the `catch_panic` / `process.exit()` interaction note in deferred.md — moved from "deferred" to "resolved" status. Implementation tracked under Phase 6 (typechecker / structured-concurrency interaction) and Phase 7 (LLVM `invoke` / landing-pad codegen, runtime panic-payload capture, "Exit" tag for `process.exit`) in implementation_checklist.md.

27. **Panic-across-FFI semantics.** ✅ Decided 2026-05-02 — full spec landed at design.md § Panic Semantics at the FFI Boundary, defined for all four cases (import/export × C/C-unwind ABIs): (1) `pub extern "C" fn` exports auto-abort on panic — codegen emits a separate trampoline symbol that catches the unwind at the boundary and routes to `std.process.abort()` with the panic message on stderr; this is *defined-abort*, not undefined behavior. (2) `pub extern "C-unwind" fn` exports propagate the panic across the boundary as a C++-exception-shaped unwind; C++ callers may catch, C callers see process termination at the next runtime call into a function with no exception handler. (3) `extern "C"` imports — existing rule unchanged: programmer declares `panics` if the C function may abort/exit/longjmp/install signal handlers; trusted at the FFI boundary. (4) `extern "C-unwind"` imports — existing rule unchanged: `panics` defaulted by the ABI; the exception surfaces to Kāra as `PanicKind.Foreign` (per item 26). Effect-system composability: `pub extern "C" fn` MAY declare `panics` (describes Kāra-side call path; trampoline is invisible to the effect system); `pub extern "C-unwind" fn` MUST declare `panics` if the body may panic (`E_EXTERN_C_UNWIND_REQUIRES_PANICS`) — silently absorbing would defeat the ABI's purpose. Implementation pattern: bare Kāra-callable symbol (standard mangling) + exported wrapper symbol with the C-facing name that wraps the call in a catch+abort landing pad; Kāra-internal calls use the bare symbol with no overhead. Build-profile interaction: under `panic = "abort"` (kernel/embedded/ISR profiles), the trampoline is elided (every panic is already abort) and `extern "C-unwind"` (both declarations and imports) is rejected at parse with `E_EXTERN_C_UNWIND_REQUIRES_UNWIND_PROFILE`. Surfacing recoverable errors to C from a `pub extern "C" fn` body: wrap in `catch_panic` and translate to a C-shaped error code (worked example in design.md); the auto-abort trampoline remains as the safety net for panics that escape the user-written catch (panic during the panic handler itself). Updated the existing "Definitions vs. imports" paragraph at § Linker Control Attributes to cross-reference the new section. Implementation tracked under Phase 7 in implementation_checklist.md alongside item 26 (depends on the same LLVM unwinding substrate). The four cases compose with the rest of the effect system without any special-case rules — the existing inference/propagation machinery handles them uniformly because the trampoline is a codegen-only artifact.

28. **Tail-expression temporary scope.** ✅ Decided 2026-05-02 — pinned in design.md § Drop ordering within a branch with the lock-in-tail worked example. The rule (Rust 2024 edition behavior): a temporary created inside a block's tail expression is scoped to the block, with a live range ending immediately after the tail expression finishes evaluating (including borrow uses chained off it); its drop is pushed onto the unified LIFO drop+defer stack *after* every `let` binding's drop, because the tail expression is later in program order than every `let`. At scope exit the LIFO drain therefore pops tail-expression temporaries before block locals — closes the lock-held-too-long footgun where a `MutexGuard` produced by `mu.lock()` in tail position would otherwise outlive locals declared before it. Move-out exception preserved: a tail-expression temporary that *is* the block's return value (rather than a value computed *from* one) escapes via the standard NLL move-out rule and is not dropped here. Statement-position temporaries (expression-statements ending in `;`) continue to drop at the `;` — unchanged. The rule was already implicit in Kāra's program-order-LIFO design (tail temp is pushed last, pops first), but pinning it explicitly makes the lock-eagerness guarantee a load-bearing language commitment rather than a happy accident of the implementation. Updated the existing terse mention in the cleanup-rules bullet at § Module System to call out the tail-temp-first behavior with a forward reference. Implementation tracked under Phase 4 / Phase 5.2 in implementation_checklist.md (verification of NLL output — no new IR or codegen, but explicit test coverage for nested blocks, lock-in-tail, move-out, and borrow-into-tail patterns).

29. **`if let` scrutinee temporary scope.** ✅ Decided 2026-05-02 — pinned in design.md § `if let` and `let...else` with worked lock-in-scrutinee examples for all three forms (`if let`, `let...else`, `while let`). The rule (Rust 2024 edition behavior): temporaries created inside the scrutinee expression of `if let PAT = SCRUT { … } else { … }`, `while let`, and `let PAT = SCRUT else { … };` are scoped to the matching arm — *not* to the surrounding statement. In the `then` arm of `if let`, scrutinee temporaries live through the entire body (because pattern-bound names may borrow into them) and drop at arm exit; in the `else` arm of `if let` / `else if let` / `let...else`, scrutinee temporaries have already been dropped *before* the arm body begins; in `while let`, per-iteration drop at body exit before the next iteration's scrutinee is evaluated. Closes the lock-held-during-else-branch footgun (and per-iteration lock-not-released footgun in `while let`). Symmetric with item 28's tail-expression-temporary rule: both fire eagerly at the first program point where the temporary is provably no longer needed, rather than being hoisted to the enclosing statement's scope. Pattern-bound names (`value` in `if let Some(value) = ...`) are NOT temporaries — they are local bindings of the arm and follow standard `let`-binding rules; the rule covers only anonymous intermediates of the scrutinee expression. Forward note: when `if let` chains land (deferred to Phase 5+), the rule applies per-link — each link's temporaries drop before the next link is evaluated (on success) or before the else arm runs (on failure). Implementation requires placing the scrutinee-temp drops on the *non-matching* control-flow edge in the lowering of `if let` / `let-else` / `while let` (rather than at the join point after both arms); tracked under Phase 4 / Phase 5.2 in implementation_checklist.md alongside item 28. Test plan covers lock-in-scrutinee for all three forms via instrumented destructors recording drop order, and pattern-bound-name lifetime holding borrows into scrutinee temps in the then-arm without dangling.

56. **Destructors are NOT soundness mechanisms.** Defers/locks/scopes enforced by the type system, never by `Drop` (re: leakpocalypse). ✅ Decided 2026-05-02 — Tier-E pin landed. New `**Destructors are NOT soundness mechanisms**` paragraph block added to design.md `### Part 8: Drop — User-Defined Destructors` (between the "Interaction with move semantics" paragraph and the chapter-end horizontal rule). **The principle**: a `Drop` impl that does not run must never be able to cause memory unsafety, UB, or violation of any type-system invariant. Drop is best-effort cleanup — if it runs, it releases the resource; if it doesn't, the resource leaks but the program remains sound. **Historical context** (the 2015 Rust "leakpocalypse"): `thread::scope` and other APIs originally derived soundness from "Drop will run", which broke in the presence of `mem::forget`, Rc cycles, and `process.exit`. The fix was to redesign the APIs so soundness lives in the type system. Kāra adopts the same lesson up front. **Comprehensive enforcement table** added to design.md enumerating every Kāra invariant a naive implementation might be tempted to back with Drop, alongside the type-system mechanism that actually enforces it: `TaskHandle[T]` join-before-scope-exit (`ScopeLocal` rejection at three positions, structured `par` / `TaskGroup` join barriers); `MutexGuard` lock-held lifetime (`lock { }` block grammar — the body's scope IS the held-lock region); borrowed-reference outlives-source (live-range analysis); cross-task type rejection (closed cross-task-unsafe enumeration at typecheck); refinement-type invariant (try_from boundary check at construction); provider-rooted resource non-escape (typecheck rejection of escape paths). In every row, Drop *also* runs cleanup, but the row's invariant is enforced by the type system. **Forward-commitment to `mem.forget`-style primitives**: if a future Kāra release ships a primitive that explicitly skips Drop, it will be a **safe** operation. Skipping Drop cannot violate type safety because none of the safety story rests on Drop firing. Knowing today that any future `forget` is safe is the load-bearing reason for the principle — it constrains every stdlib API design today against baking soundness into a destructor. **Implication for stdlib design**: every stdlib type with a `Drop` impl satisfies the rule by construction; the destructor runs *desirable* cleanup, not *required-for-soundness* cleanup. **Implication for `defer` / `errdefer`**: same rule — defer blocks are operational cleanup guarantees, not type-system soundness primitives. The "cleanup is guaranteed except in the two named exceptions" (double-panic and `panic = "abort"`) does not cascade into a soundness hazard precisely because no soundness invariant rests on cleanup firing. **The rule in one sentence**: *soundness lives in the type system; cleanup lives in the runtime.* The two layers cooperate (type system guarantees cleanup will run on every normal path; runtime executes it) but the type system's soundness story does not depend on the runtime's execution of cleanup. **Implementation tracked as a Phase 8 checklist** entry — not a compile-time enforcement (the principle is too semantic to mechanize) but a stdlib code-review checklist baked into PR templates: every new stdlib type with a `Drop` impl gets a reviewer-answered "if Drop is removed, what becomes unsafe?" with the required answer "nothing." Plus per-type "Drop-skip soundness" tests for shared/public stdlib types, a one-time audit of existing v1 stdlib types with `Drop` (`MutexGuard`, `RwLockReadGuard`, etc.), and template boilerplate doc-comments at every `impl Drop` site reinforcing the principle. **No syntax.md change** — pure documentation pin. **No new compiler diagnostic** — the related compiler-enforced rules (`Drop` body must be `panics`-free, `Drop`/`Copy` mutually exclusive, partial-move rejection on `Drop`-bearing types) are pinned elsewhere; this entry is the *meta-principle* underlying them.

57. **Variance declared explicitly per stdlib parametric type.** No default-inference for soundness-critical cases. ✅ Decided 2026-05-02 — adopted at v1 with **per-type declarations + stdlib audit + invariant default for user types** (option a of the decision matrix). **The principle**: Kāra has **no auto-inference of variance from structural use**. Every variance commitment is the result of an explicit declaration — either a position-based rule fixed at the language definition (function arguments, return types, reference targets) or a per-type marker on a stdlib parametric type's generic parameters. Inference can produce wrong variance in the presence of interior mutability, generic instantiation, or trait coherence corner cases; explicit declaration is the load-bearing answer. **Position-based variance — fixed by the language** (unchanged from prior spec): `mut ref T` invariant; `ref T` covariant; function return covariant; effect-set covariant; function argument contravariant. **Per-type variance — declared by the stdlib type** (new): `+T` covariant, `-T` contravariant, `=T` (or no marker) invariant. The default is **invariant**, the conservative choice. **Stdlib audit pinned in design.md**: `Vec[=T]`, `Slice[=T]`, `Array[=T, const N]`, `Map[=K, =V]`, `Set[=T]`, `VecDeque[=T]`, `SortedSet[=T]`, `Option[+T]`, `Result[+T, +E]`, `Iterator[+T]`, `Box[=T]`, `Rc[=T]`, `Arc[=T]`, `Sender[=T]`, `Receiver[=T]`, `Channel[=T]`, `Atomic[=T]`, `Mutex[=T]`, `RwLock[=T]`, `LazyLock[=T]`, `OnceLock[=T]`, `OnceCell[=T]`, `MaybeUninit[=T]`, `PhantomData[=T]`, `TaskHandle[+T]`, `AlreadySetError[+T]`. The covariant types unlock subtyping that the all-invariant rule blocks: `Iterator[Positive]` widens to `Iterator[i32]`, `Option[NonZero]` widens to `Option[i32]`, `Result[Positive, IoError]` widens to `Result[i32, IoError]`. **Soundness verification at the type-decl site**: `verify_variance` typechecker pass walks the type's structure and checks that each declared parameter appears only in positions consistent with its variance — `+T` requires covariant-only positions (return types, immutable-borrow targets, covariant slots of inner types); `-T` requires contravariant-only; `=T` is unrestricted. Violations produce `E_VARIANCE_DECLARATION_INCONSISTENT` with the offending position underlined and a fix-it suggesting the next-most-conservative declaration. **User types are invariant at v1** — the `+`/`-` markers in user code are rejected with `E_VARIANCE_USER_DECL_NOT_YET`; user-side variance is a P1 follow-up tracked in deferred.md (the v1 reservation ensures the surface lands later without source breaks). **Stdlib lint**: every parametric stdlib type must carry an explicit variance declaration on every parameter — invariance via `=T` or no-marker is fine; the lint rejects accidentally adding a parametric type without thinking about variance. **Refinement-base widening** (existing rule): orthogonal to nominal variance. `NonEmpty[Positive]` widens to `NonEmpty[i32]` via the refinement-base widening at the parameter slot AND the `Iterator[+T]` widening at the type-level (when applicable); both rules compose. `Vec[Positive]` does NOT widen to `Vec[i32]` because `Vec[=T]` is invariant — the refinement widening cannot promote a parameter through an invariant slot. **Subtyping rule at use sites**: when checking `Foo[Sub] <: Foo[Super]`, the variance declaration of `Foo`'s parameter governs the check. **`syntax.md §6.4`** updated: `GENERIC_PARAM` extended with optional leading `VARIANCE_MARKER` (`+` / `-` / `=`); the marker attaches to the parameter, not the bound. **Higher-kinded polymorphism and phantom variance markers** remain deferred (P2 — no committed design); v1 surface covers every realistic stdlib case. Implementation tracked as a Phase 5.2 entry — parser + AST extension, the verifier pass, the stdlib audit + lint, and the user-side rejection diagnostic. No P1 carve-out for the verifier; user-side declaration surface IS the P1 follow-up.

58. **`mut ref T` is invariant in `T`.** Pin explicitly. ✅ Decided 2026-05-02 — Tier-E pin landed. The rule was already in the position-based variance table at design.md `### Variance` (`Mutable reference target (mut ref T) | invariant`); v60 item 58 adds an **explicit prose pin** immediately after the table with the load-bearing-soundness rationale. **The rule**: `mut ref T` does not widen via subtyping in either direction. `mut ref Sub` is not a subtype of `mut ref Super` even when `Sub <: Super`, and `mut ref Super` is not a subtype of `mut ref Sub`. **The reasoning** (the classic invariance-of-mut-ref soundness lemma): a `mut ref Super` allows writing any `Super` value, including ones not valid as `Sub`; a `mut ref Sub` allows reading a `Sub` and treating the result as `Super` only via the standard widening on the read result. Allowing either direction of subtyping on `mut ref T` would let a write through one alias produce a value that another alias's read could not soundly classify — a write of a generic `Super` through `mut ref Super` followed by a read through `mut ref Sub` (assuming covariance) would witness a `Super`-typed value where the type system promises a `Sub`. **Refinement-base widening also rejected for `mut ref T`**: `mut ref Positive` does NOT widen to `mut ref i32` — the same rule applies. Users who need to write `i32` values into a `Positive` slot must use `try_from` at the construction boundary, not a `mut ref` widening. This means the existing refinement-base-widening rule (`Positive → i32`) is **value-level only**, not propagated through `mut ref`. **Why pin it explicitly** (rationale): so no future feature (variance-relaxation lint, subtyping-through-trait-coercion, distinct-type covariance) accidentally weakens it. The pin documents the soundness invariant at the place reviewers look first when considering variance changes. **No implementation work** — the rule is already enforced by the existing variance verifier (item 57); the pin is a Tier-E documentation-only addition. **No syntax.md change** — no grammar impact. **No implementation_checklist.md entry** — the verifier from item 57 already handles this; pinning the rule explicitly is a documentation-only commitment.

59. **Drop order rule pinned.** Locals in reverse order; tail-expression temporaries before locals (per #28). ✅ Decided 2026-05-02 — Tier-E pin landed. The substantive rules were already specced in design.md `#### Drop ordering within a branch` (the unified drop+defer stack with LIFO ordering, plus the explicit tail-temp-before-locals rule from item 28); v60 item 59 adds an **explicit "Drop order — pinned rules at a glance" callout** at the top of the section so reviewers see the canonical answer without reconstructing it from LIFO mechanics. **The two pinned rules**: (1) **Locals drop in reverse declaration order** — last-declared first. Applies uniformly inside every block, every `par {}` branch, every match arm, every `if`/`else` body, every closure body. Consistent with the existing struct-field drop-in-reverse-declaration-order rule. (2) **Tail-expression temporaries drop before block locals** — the tail expression is later in program order than every `let`, so its temporaries pop first under the LIFO drain. Closes the lock-held-too-long footgun documented at item 28 of v60. Statement-position temporaries (expression-statements ending in `;`) drop at the `;`, unchanged. Both rules fall out of the unified drop+defer stack ordered by program-order of introduction; the Tier-E callout makes them immediately visible. **Cross-references** to existing struct-field drop-order rule (item at design.md `### Part 8: Drop > Field drop order`) and to item 28 (tail-expression temporary scope) preserved. **No implementation work** — both rules are already enforced by the existing NLL drop-placement pass; the pin is documentation-only. **No syntax.md change** — no grammar impact. **No implementation_checklist.md entry** — verified that the existing drop-placement test coverage from item 28's resolution covers both rules.

60. **Temporary lifetime rule pinned.** Document explicitly — no informal "longest reasonable scope" intuition. ✅ Decided 2026-05-02 — Tier-E pin landed. New `### Temporary Lifetime Rules` section in design.md sits between `### \`if let\` and \`let...else\`` and `### Loops`, consolidating the per-position temporary-lifetime spec at the top level where readers find it without reconstructing it from individual feature sections. **The canonical rule**: a temporary's live range ends at the **end of the smallest enclosing eager-evaluation context** — explicitly, with no fall-back on informal "longest reasonable scope" intuition. **Position-specific table** enumerates twelve positions with their drop points and cross-references: statement-position (`;`), tail expression of a block (per item 28), scrutinee of if-let/while-let/let-else (per item 29), match-expression scrutinee (live across all arms; drops at match exit), match-arm guard (drops before arm body runs — load-bearing for guards that take borrows), function/method call argument (after call returns), operator argument, index expression, struct/tuple/array literal field, `return expr;` value, closure-body tail (recursive), loop-body block (per-iteration drop), `for` loop iterator value (lives across entire loop). **Binding-extension exception**: `let r = ref some_function();` extends the temporary to the binding's live range (the only case where a temporary outlives its immediate expression); binding-extension does NOT apply in every other position (function arguments, return values, tail expressions, scrutinees). **Composition with NLL**: position-specific rule sets the latest drop point (ceiling); NLL can shorten via last-use analysis (floor); the actual drop fires at the floor. NLL never extends past the ceiling — that would invalidate the lock-eagerness guarantee from item 28. **Composition with drop ordering** (item 59): orthogonal — this section determines *when* a temporary's live range ends; drop ordering determines *how* multiple expiring temporaries are sequenced relative to each other (LIFO over program-order). **No `'static` extension or hidden promotion** — unlike Rust, Kāra does not silently promote temporary string literals or constant expressions to `'static`. String literals are already `'static` because their type is `StringSlice`, not because of temporary promotion. Keeps the temporary-lifetime model fully explicit. **Why an explicit table** (rationale): Rust's pre-2024-edition rule was "temporaries live as long as the longest reasonable scope" — informal, intuition-driven, footgun-rich (the `mu.lock().get(k).cloned()` lock-eagerness footgun was the canonical case). Kāra commits to per-position rules so a reader (human or LLM) can look up "where does this temporary drop?" against a finite table rather than reasoning about implicit scope hoisting. **Cross-references** to items 28 (tail-expression scope), 29 (scrutinee scope), 59 (drop ordering) pinned. **No implementation work** — both the existing item-28 / item-29 rules and the broader position-specific rules are enforced by the existing NLL drop-placement pass. **No syntax.md change** — pure semantics. **No implementation_checklist.md entry** — verified that existing test coverage from items 28 / 29 covers the canonical-rule positions; the additional positions (match-arm guard, function-call argument, etc.) are handled implicitly by the existing NLL machinery and don't need separate test slices beyond what each feature's checklist entry already mandates.

61. **No auto-traits.** No `Send` / `Sync`-style auto-derivation. Document the position. ✅ Decided 2026-05-02 — Tier-E pin landed. New `### No Auto-Traits` section in design.md sits between `### Marker Traits` and `### Trait Aliases`, anchoring the eight existing in-text references that pointed at "item 61 — No Auto-Traits" before the section existed. **The principle**: every trait implementation in Kāra is **explicit user code** (`impl Foo for T { }`) or **explicit compiler code** (sealed compiler-known impls for a small fixed set). There is no compiler mechanism that implicitly implements a trait based on structural composition — nothing parallel to Rust's `Send` / `Sync` derivation. **Why no auto-traits** (rationale paragraph): auto-traits in Rust are the canonical case study in why "obvious" trait propagation is hazardous — accumulated soundness corrections, edge cases, surprising-propagation bugs over a decade. The fundamental issue is the long-tail type-property question ("is my random type Send?") whose answer depends on transitive containment through every field, generic parameter, and trait impl. Removing the long tail by making the answer **explicit at the type definition site** trades a small amount of declaration syntax for a large reduction in compile-time mystery. **The five alternative mechanisms** that replace what auto-traits would have served, each chosen at the type definition site, visible to the reader, verified by the compiler against a closed enumeration: (1) **Closed enumeration of rejected types** for cross-task safety (item 48 — `shared struct`, `Rc[T]`, `OnceCell[T]`, raw pointers, etc.); (2) **Keyword-declared type kind** for shared-vs-cross-task semantics (`shared struct` vs `par struct`); (3) **Explicit `marker trait`** for opt-in tagging (`marker trait Pod;`, `Eq`, `ScopeLocal`, etc. — item 55); (4) **`#[derive(Trait)]` opt-in** for derivable behavior traits (`Eq`, `PartialEq`, `Ord`, `Hash`, `Clone`, `Copy`, `Debug`, `Default`, `Display`, `Serialize`, `Deserialize`); (5) **Refinement-base widening** for type-narrowing relationships through covariant slots. **Mapping table** in the new section enumerates every Rust auto-trait pattern (Send, Sync, !Send, Sized, Unpin, UnwindSafe, Default-derived, user-defined markers) alongside its Kāra equivalent — every Rust auto-trait either has a closed-enumeration / keyword-declared / explicit-derive equivalent, or has no v1 surface (because the underlying feature isn't in v1). **Composition with `marker trait`**: marker traits are explicit-implementation only; the compiler's hand-emitted impls for sealed markers (`ScopeLocal`, future `Concurrent`) are NOT auto-derivation — the impls are written by compiler authors against a closed set of types. **Composition with stdlib variance** (item 57): trait impls explicit, type-level variance explicit — both commitments compose to cover the two main soundness-critical type properties at the definition site rather than at the use site. **Forward-commitment**: `mem.forget` is safe (per item 56) — the no-auto-traits rule is consistent with this; soundness lives in the type system, not in trait derivation. **Deferred `Concurrent` auto-trait entry — superseded**: the previously-deferred entry for `Concurrent` as an auto-trait in design.md's deferred-items table is **rewritten** to note it's superseded by item 48's closed-enumeration approach. If `Concurrent` ships at all, it lands as an explicit `marker trait` with `impl Concurrent for T { }` user/compiler code, not as an auto-trait. Whether the explicit marker is needed for generic bounds (`fn broadcast[T: Concurrent](...)`) is a Phase 10 decision; the structural cross-task-safe set already covers cross-task safety without `Concurrent` shipping at all. The "auto-trait" wording is preserved in the deferred entry for historical context only. **No syntax.md change** — pure principle pin. **No implementation_checklist.md entry** — the principle is enforced by the absence of auto-derivation infrastructure in the typechecker, plus the explicit checks committed in items 48 (cross-task-safe set) and 57 (variance verifier); no new check needed for item 61 itself.

62. **Reserve `expr_<year>` macro fragment specifier namespace.** Edition-versionable from day one. ✅ Decided 2026-05-02 — adopted at v1 with the **narrow `expr_<NNNN>`-only reservation** scope. New `### Reserved Fragment-Specifier Identifier Namespace` section in design.md sits in the family of reserved-syntax sections (between `### Reserved \`#\`-Guarded String Syntax` and the closing horizontal rule before `## Type System and Object Model`). **The reservation**: identifiers matching `expr_<NNNN>` where `NNNN` is a four-digit year in `2020`–`2099` reject at lex with `E_RESERVED_FRAGMENT_SPECIFIER_NAMESPACE`. The diagnostic offers two fix-its: `r#expr_<NNNN>` (raw-identifier escape per item 39, accepted by the exemption rule) or rename to a non-year-suffixed form. **Why the reservation exists**: future macros or comptime AST extensions may need edition-versioned syntax categories — names that distinguish "expression syntax as of edition 2026" from "expression syntax as of edition 2030." The naming convention `expr_<NNNN>` parallels Rust's `macro_rules!` fragment-specifier convention (`expr_2024`, `expr_2027` in proposals); it leaves the namespace open even though Kāra's current metaprogramming direction is `comptime fn` AST→AST rather than `macro_rules!`. Reserving at v1 prevents a source-breaking change post-v1 if Kāra adopts edition-versioned fragment specifiers (or edition-versioned AST node types using parallel naming). **Forward-compatibility contract**: when a future Kāra release introduces an `expr_<NNNN>` syntax category, no existing v1 program needs to change because no v1 program could use the name as an identifier — the lexer rejects it. Programs that needed the exact name today via `r#expr_<NNNN>` continue to work. **Narrow scope at v1**: only `expr_` prefix is reserved. Other plausible fragment-specifier prefixes (`pat`, `ty`, `stmt`, `block`, `tt`, `ident`, `lit`, `path`, `vis`, `meta`, `item`, `lifetime`) are **not** reserved at v1 — extending the reservation post-v1 to additional prefixes is non-breaking-additive (existing user identifiers using those prefixes continue to work). **Year range**: `2020`–`2099` inclusive — matches the realistic edition-year window for any Kāra release. Years outside this range (`expr_1999`, `expr_3000`) parse as ordinary identifiers. **No connection to literal year value**: the reservation is purely a namespace allocation. `expr_2026` is reserved today even though Kāra's current edition is `2026`; `expr_2099` is reserved even though no compiler will see it for decades. **`syntax.md §1.3` updated**: `IDENTIFIER` rule extended with an inline note naming the `E_RESERVED_FRAGMENT_SPECIFIER_NAMESPACE` reservation and the `r#expr_<NNNN>` workaround. Implementation tracked as a Phase 1 (Lexer) entry — small lex-time regex check after identifier tokenisation, reusing the existing raw-identifier exemption infrastructure. No P1 carve-out — the reservation ships at v1 to lock in the namespace before users can rely on it.

63. **`missing_fragment_specifier` as hard error from day one.** When macros land, definition-time rejection (not call-time). ✅ Decided 2026-05-02 — Tier "Future-proofing" pin committed at v1 with **forward-commitment** to definition-time validation when the metaprogramming surface ships. The rule lands at v1 as a principle pin in design.md `### Reserved Fragment-Specifier Identifier Namespace > Forward-commitment — definition-time fragment-specifier validation` and a companion pin in deferred.md `### Comptime — AST→AST comptime fn` between the three-forms-compose paragraph and the types-as-first-class-values subsection. **The principle**: when Kāra eventually ships a metaprogramming surface that uses fragment specifiers, every metavariable specifier MUST be validated at the **definition site**, not at the use site. A macro definition that omits or malforms a specifier produces a hard error at the macro's declaration, never a deferred error at every call site. **Why** (rationale): Rust pre-1.55 accepted macro definitions with missing specifiers and surfaced the error only at call time, with the error message pointing at the call site rather than the broken macro — confusing diagnostics that scattered the bug's evidence across every call. Kāra commits to definition-time rejection from day one. **Status under `comptime fn`** (the v1 deferred metaprogramming direction): the rule is already implicit — `comptime fn` parameters require type annotations per the existing fn-parameter rules (item 53). A `comptime fn my_macro(e)` (no type) is rejected at definition by the existing typechecker; this entry pins that the rule is intentional and applies to every future metaprogramming form. The diagnostic `E_MISSING_FRAGMENT_SPECIFIER` fires at the comptime fn's definition site, naming the missing fragment specifier and offering the typed-AST-shape fix-it (`Expr`, `Stmt`, `Type`, etc.). **Status under hypothetical `macro_rules!`** (not in v1, not committed for any future release): if such a system ever ships, the parser rejects `macro_rules! foo { ($x) => { ... } }` at the macro's declaration site with the same error code; the bug's evidence is local to the declaration, not scattered across calls. **No v1 implementation work** — the rule is enforced today by the existing fn-parameter type-annotation requirement; the forward-commitment pin documents the rule as load-bearing for all future metaprogramming surfaces. **No syntax.md change** — pure principle pin. **No implementation_checklist.md entry** — the existing fn-parameter rule from item 53 (and the broader "trait method parameters require names" + analogous free-fn rule) covers comptime fn parameters today; if macro_rules-style macros ever ship, the rule extends to them in their own implementation slice. **Cross-references**: design.md and deferred.md pins both reference each other; the design.md home is in the family of forward-looking metaprogramming reservations (alongside item 62's `expr_<NNNN>` namespace reservation), keeping the two metaprogramming-future-proofing items colocated.

64. **Edition migration as warning-promotion pipeline.** Declare in roadmap: post-v1 editions used to graduate warnings → deny → hard error. ✅ Decided 2026-05-02 — adopted at v1 with **three-stage promotion pipeline** spec'd in design.md and declared in roadmap.md as the post-v1 commitment. **Where the spec lives**: design.md `### Project Manifest > Edition semantics > Migration policy — warning-promotion pipeline` (new bullet) and roadmap.md `## Future: Editions and Migration Pipeline` (new section between Comptime and Language Server). **The three-stage promotion**: (1) **Warning stage** — upcoming behavior change reported as warn-by-default lint in the current edition; suppressible via `#[allow(<lint_name>)]`; lint name is permanent per the lint-namespace stability rule. Authors see the warning during normal `karac build` runs and have lead time. (2) **Deny stage** — lint promotes to deny-by-default at a chosen point during the current edition's lifetime, typically one minor compiler release after stage 1. Programs that haven't addressed the lint now require explicit `#[allow]` to compile. (3) **Hard-error stage** — at the next edition boundary, the lint disappears and the underlying behavior change becomes a hard error. Code under the new edition must conform. **Tooling support** (post-v1 deliverables in roadmap): `karac explain --edition <NEXT>` projects the full migration timeline; `karac fix --edition <NEXT>` applies mechanical migrations; `[lints]` table in `kara.toml` for per-lint policy override (`[lints] foo = "deny"` to escalate immediately, `[lints] bar = "allow"` to defer past the deny stage). **Concrete examples already in v1 design** that will be the first migrations the pipeline carries: private-function effect inference broadening (per § Specification Layers, Reported behavior — the canonical motivating case), future fragment-specifier reservations (per item 62 — extending to `pat_<NNNN>`, `ty_<NNNN>`, etc. post-v1), future keyword reservations (when an edition reserves a token like `async`, warn-stage flags every binding using the name; deny-stage forces `r#async`; edition-boundary hard error rejects the bare form). **Why three stages, not a single hard break** (rationale paragraph): warning stage gives ecosystem authors lead time before any compilation failure; deny stage forces awareness without forcing immediate migration (the `#[allow]` opt-out is the escape valve); edition-boundary hard error is the final stop. The pipeline matches Rust's edition-migration discipline (Rust 2021 → 2024 graduates patterns through warn → deny → error) and is the load-bearing answer to "how do post-v1 breaking changes ship without breaking everyone overnight?" **No v1 implementation work** — the pipeline tooling lands when a second edition is ready to ship; building it before then would be infrastructure without a customer. v1 commits to the discipline so the first post-v1 edition has a clear runway. **No syntax.md change** — the `[lints]` table is `kara.toml` configuration, not language grammar; the `#[allow(...)]` machinery already exists. **No implementation_checklist.md entry** — the migration tooling deliverables are tracked in roadmap.md `## Future: Editions and Migration Pipeline`. **Cross-references**: design.md migration-policy bullet references the lint-level attribute machinery (item 35), the lint-namespace stability rule, the `karac explain` / `karac fix` tooling, and the canonical examples (effect inference broadening, item 62 reservations, keyword reservations). **This is the final v60 item — all 64 items resolved.**
