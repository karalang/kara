## Item 1 — Critical-path identification

**What might be MISSING (per author's invitation).** A read of design.md surfaces two areas that look comprehensively covered but could legitimately go deeper for a no-compromise language:

- **Debugger / profiler interface.** ~~Language-level contract with debuggers (stack frame shape, scope-bound data observability, `par {}` and `suspends` reconstruction) is undefined.~~ **RESOLVED 2026-04-29 — spec a Debugger Contract section in design.md Part 7 (Tooling), v1 scope.** The bullet conflated three separable threads with different answers:
  1. **Codegen debug info (DWARF, source maps).** Phase 7 hygiene — required for any debugger to attach. LLVM does most of the work. Not a design question.
  2. **Runtime stack model under `par {}` and `suspends`.** The actual Kāra-specific contract. Hybrid model: worker stack as ground truth, side metadata for logical-stack reconstruction, gated on profile (`[profile.dev]` on, `[profile.release]` off). Contract specifies four elements — (a) static spawn-site IDs per `par {}` block, (b) parent-frame reference per worker, (c) await-chain pointer per suspended task, (d) `std.runtime::list_tasks()` for enumerating suspended tasks. Lands with Phase 8a auto-concurrency codegen — this is the moment the runtime first emits par/suspend code, so the contract must be in place before then or it gets locked in by accident.
  3. **Effect / RC introspection at runtime.** Useful but not load-bearing. Defer to post-v1.

  **Q1 — detached spawn?** Verified structured-only. `design.md:6951` defers unstructured `spawn` (P2); `design.md:6916` enforces it via `TaskHandle[T]: ScopeLocal`. All three concurrency mechanisms (auto-concurrency, `par {}`, `spawn()`/`TaskGroup`) guarantee parent frame outlives child. Contract has one shape: parent-ref always present; main task tagged "root" via sentinel.

  **Q2 — per-spawn vs. per-block metadata?** Per-block for `par {}` (O(par-blocks-in-source) ≪ O(workers); workers anonymous + interchangeable in Kāra's data-parallel model; debugger plugin combines block metadata + worker thread-local index). Per-task for `suspends` (tasks have distinct awaiters; no per-block aggregation recovers the await tree). Asymmetric because the primitives are asymmetric — `par` is bulk data-parallelism, `suspends` is structured task concurrency.

  **Tooling out of scope for v1.** gdb/lldb plugins, DAP integration, time-travel debugging, profiler GUI — all post-v1. The v1 deliverable is the *contract* and the runtime metadata emission, not the consumer tools.

  **Follow-up implementation work** ✅ shipped 2026-04-30: `design.md § AI-First Compiler Interface > 3. Debugger Contract` adjacent to `karac query`; cross-reference + runtime-metadata-emission entry added under `roadmap.md § Phase 8 > Auto-Concurrency Codegen`.
- **Crash report / diagnostic capture format.** ~~`kara_panic` wire format (effect-traced panics, RC-fallback notes, parallel-failure summaries) is unspecified.~~ **RESOLVED 2026-04-29 — spec a Crash Report Format section in design.md Part 7 (Tooling), v1 scope. Contract is language-level; handler implementation is stdlib.**

  **Open question split:** the bullet stacked "language contract or stdlib?" with "what goes in it?" — they're separable.

  **(a) Language contract for the wire format, stdlib for the handler.** Mirrors the Debugger Contract resolution. Production tooling (Sentry/Datadog/ELK, post-mortem analyzers, automated bug filers) parses panic output to dedupe and group — if format changes between minor versions because it's stdlib-controlled, every crash dashboard breaks. Same constraint as Rust's panic *semantics* (language) vs. handler (`std::panic`, overridable). Spec the format in design.md; implementation lives in `std.panic` / `std.runtime`.

  **(b) Unifying insight:** the crash report is a **serialized Debugger Contract snapshot at panic-time**, plus panic-specific fields. Same metadata sources (spawn IDs, parent-frame refs, await chain, effect summaries) — one emission point feeds both. Designing them as separate contracts would duplicate the metadata layer.

  **Two formats, both required:**
  - **Structured (JSON).** The load-bearing one for tooling. Tools key on this. JSON over msgpack: speed/size don't matter for write-once-read-rarely crash files (single-digit KB overhead is irrelevant); `jq`-debuggability matters a lot ("ssh into prod, find crash file" needs zero-friction inspection); universal tooling support on the fast ingestion path. Streaming runtime IPC (high-frequency `list_tasks`/`get_effects`) is a separate format decision under the Debugger Contract, not this one.
  - **Human-readable.** Pretty-printed for terminal. Free to iterate without breaking tools.

  **Required structured fields:** (1) panic site — file:line:col, demangled fn name, IP; (2) panic kind discriminant — `explicit` / `rc_fallback_borrow` / `par_fail_fast_cancel` / `div_by_zero` / `index_out_of_bounds` / `effect_boundary_violation` / `provider_missing` / etc; (3) message; (4) logical stack via Debugger Contract reconstruction (per-block for par, per-task for suspends), each frame carrying source location + effect summary; (5) provider stack — active `with_provider` bindings at panic time; (6) RC-fallback annotations — frames where the panic chain crosses an RC drop, with the source location where the compiler decided to RC; (7) parallel context — for panics inside `par`/`spawn`/`TaskGroup`, which branch is panicking, which siblings are still running, which got cancelled by fail-fast; (8) build metadata — kāra version, target, profile, commit hash.

  **Output discipline:** stderr gets a 5-10 line summary + crash file path. Crash file gets the full structured report. Default path `/tmp/kara-crash-{pid}-{timestamp}.json`, configurable via `KARA_CRASH_DIR`.

  **Edge cases the spec must address:**
  - **Panic-during-panic-report.** Report-writer itself panics (out-of-disk). Fall back to abort + minimal stderr line. Don't loop.
  - **Drop-time panic.** RC drop sequence runs during scope exit / unwind. Spec must commit that drop-panics get captured (today famously bad in C++).
  - **Concurrent panics.** Two tasks panic simultaneously. Each writes its own crash file; stderr lists both. Process exit code reflects fail-fast.
  - **WASM target.** No filesystem in the browser — payload serializes to a JS-side handler hook (`window.karac_crash` or similar) instead of file write. WASI is fine.
  - **GPU target.** GPU code can't write disk. Panic in GPU lowering becomes host-side panic at kernel-launch site with `gpu kernel failed` marker.
  - **Embedded `panics_off` profile.** Format doesn't apply — panics unreachable / abort. Spec scopes itself to "profiles where panics can occur."

  **Phase placement:** contract spec in design.md (now); implementation in Phase 8a stdlib `std.panic` / `std.runtime` co-developed with the Debugger Contract (shared metadata sources); WASM/GPU adaptations in Phase 10.

  **Follow-up implementation work** ✅ shipped 2026-04-30: `design.md § AI-First Compiler Interface > 4. Crash Report Format` adjacent to Debugger Contract; cross-reference + `std.panic` / `std.runtime` entries added under `roadmap.md § Phase 8 > Diagnostics`.

### P1 — Dependency resolution algorithm

**Question.** When `[dependencies]` becomes load-bearing, what algorithm resolves `name = "1.2"` constraints into concrete versions across the dependency graph?

**Options.**
- **Cargo / PubGrub.** Conservative semver resolution; picks the latest compatible version unless `Cargo.lock` pins it; conflict diagnostic shows the constraint chain. Considered state-of-the-art for clear diagnostics.
- **Go / MVS (Minimum Version Selection).** Pick the *minimum* version that satisfies every constraint. Surprises rare; no surprise upgrades; but you're often on older code than necessary.
- **npm / per-import installations.** Each dep gets its own copy of its deps — no unification. Result: enormous `node_modules`, but no version conflicts (because there are no shared versions).
- **Hex (Erlang/Elixir) / flat resolution with explicit conflict.** Single version per package across the graph; conflicts are user-resolved.

**Tradeoffs.**
- PubGrub: best diagnostics, most-upgraded code, occasional resolution failures users have to think about.
- MVS: no surprises, ages packages, simpler implementation.
- npm: no conflicts ever, fat builds, security-audit nightmare (one CVE in N transitive copies).
- Flat: clean mental model, requires user-driven conflict resolution.

**Default lean.** PubGrub-style. Diagnostics are the load-bearing concern (matches the rest of Kāra's posture); fat-builds are a non-starter for a systems language; MVS's age-by-default conflicts with "stay current."

**Resolution.** PubGrub. Locked verbatim with the lean. v1.1 implementation work tracked under `implementation_checklist.md § 5.5`.

### P2 — Registry — when, and what shape?

**Question.** Design.md says "default registry is TBD; v1 is git URL + local path only." What's the eventual shape, and what's the trigger to design it?

**Options for shape.**
- **Cargo / crates.io.** Centralized, immutable (no version overwrite), public, single source of truth; namespace squatting is real.
- **Go / proxy.golang.org.** Decentralized — package identity is the git URL itself; a proxy caches+serves them. No namespace problem (URLs are namespaces); harder to revoke.
- **npm.** Centralized + scoped (`@org/pkg`); private registries possible; multi-registry fetch.
- **Hex.pm.** Centralized + scoped; immutable.
- **No registry, ever.** Pure git-URL + path deps forever (Deno-style — npm-via-CDN-URL).

**Tradeoffs.**
- Centralized + immutable: best for reproducibility and supply-chain audit. Requires hosting infrastructure.
- Decentralized (Go-style): no infrastructure burden; tied to git availability.
- No registry: simpler; relies on third-party (GitHub) availability; no central audit point.

**Trigger.** The first time a Kāra package is widely depended on by external consumers. Probably post-v1 stdlib stabilization.

**Default lean.** Decentralized (Go-style proxy + git-URL identity) for first deployment. Add centralized registry later if/when adoption demands it. Namespace squatting on a centralized registry pre-v1 is a worse failure mode than "you have to write the git URL."

**Resolution.** Option A — Go-style decentralized + explicit proxy commitment. Package identity is the source git URL; a proxy at `proxy.kara-lang.org` (or equivalent) mirrors every git URL ever resolved through it, immutably. The proxy is *not* optional infrastructure — without it, decentralized identity cannot honor `kara.lock` reproducibility (a force-pushed branch, deleted repo, or transient network partition would break builds). Lockfile records both the original git URL and the proxy mirror reference + content hash. The pure-git-URL-no-proxy variant (option B) was rejected as honest-but-too-weak relative to P6's bit-exact reproducibility promise; the centralized-from-day-one variant (option C, Cargo/Hex-shaped) was rejected as too much infrastructure to commit pre-stable. Trigger to add centralization on top of A: when the project / foundation is large enough to host it, OR when namespace squatting becomes an issue (probably never if git-URL identity stays the canonical form).

### P3 — Lockfile mechanics

**Question.** `kara.lock` is committed in design.md. What does it actually contain, and when does it land?

**Sub-questions.**
- **Hash content of dep, or just version?** Cargo hashes; Go records both module sum and content hash. Hashing makes registry compromises detectable.
- **Per-target lock or unified?** A single `kara.lock` for all build targets (current Cargo posture) or one per target (more reproducibility, more files).
- **What does `karac update` do?** Bump everything to latest semver-compatible? Bump one named package? Bump within a constraint range?
- **Library vs. binary lockfile commitment?** Design.md says "checked into VCS for executables; omitted for libraries." This is the Cargo convention. Worth confirming.
- **Timing.** Does lockfile ship in v1, v1.1, or with the first real registry?

**Default lean.** Content-hash + version recorded; single lockfile across targets; `karac update [package]` form for surgical updates plus bare `karac update` for blanket; bin-yes / lib-no commitment matches design.md; ship lockfile in **v1.1** alongside first real `[dependencies]` resolution.

**Resolution.** Locked verbatim with the lean. Concrete fields recorded per package: name, exact version, source URL (proxy mirror or git URL), BLAKE3 content hash, resolver-output dependency tree. Hashing is BLAKE3 (already adopted across the toolchain).

### P4 — Build artifact cache

**Question.** Where do compiled artifacts live, and how is the cache keyed?

**Sub-questions.**
- **Per-project `dist/`** (already in `.gitignore`) vs. **global cache** (`~/.kara/cache/`-style) vs. **both** (Cargo's `target/` per project + `~/.cargo/registry/cache/` for downloaded deps).
- **Cache key.** What makes two builds reuse a cached artifact? (compiler version, dep version, edition, profile, feature flags, target triple).
- **Invalidation.** When does the cache get cleared? On compiler upgrade? Profile change? Manually only?
- **Sharing across projects.** Should two projects with the same dep version share the cached compiled object?

**Default lean.** Per-project `dist/` for project artifacts (already there); global `~/.kara/cache/` for downloaded + compiled dep packages keyed by `(compiler-version, package-version, edition, profile, target)`. Manual `karac clean` to clear; automatic invalidation on compiler upgrade.

**Resolution.** Locked verbatim with the lean. `karac clean` removes per-project `dist/`; `karac clean --global` removes the user-wide cache.

### P5 — `[features]` flags

**Question.** Cargo has feature flags for conditional compilation. Design.md does not currently. Are they in scope?

**Where they'd interact.**
- **Profiles** — `no_effects = ["allocates(Heap)"]` is feature-flag-shaped already.
- **Stdlib layers** — `core` / `alloc` / `std` is feature-flag-shaped.
- **Optional dependencies** — Cargo's `[features]` lets you pull in a dep conditionally.

**Options.**
- **Add `[features]` table.** Standard Cargo-style. Powerful but adds matrix explosion in CI ("does this work with feature X off?").
- **Reject feature flags.** Use profiles for everything; conditional dependencies via separate manifest files per target.
- **Profile-based features only.** A feature is just a profile entry; no separate `[features]` syntax.

**Tradeoffs.**
- Cargo-style: flexible; Cargo's biggest pain point in practice (feature-unification bugs across the dep graph).
- Reject: simpler; some users will need conditional code.
- Profile-based: unified mental model; may not cover all cases.

**Default lean.** Profile-based features unified with the existing profile system. Don't add a separate `[features]` axis. If a real use case shows up that profiles can't cover, revisit.

**Resolution.** Option A — strict profile-based, no per-dep features in v1, v2 RFC slot reserved. The cases that feature flags traditionally cover are addressed individually rather than via a unified feature axis: test-only deps via `[dev-dependencies]`; target-conditional source via `#[cfg(target_os = "...")]`; build-config-conditional behavior via profiles (release / dev / bench / `no_std` / user-defined); per-target dependencies via `[target.X.dependencies]` blocks; library-conditional surface via the multi-package decomposition pattern (`foo` / `foo-async` / `foo-blocking` — established in Tokio, serde). The constrained "feature axis without transitive unification" alternative (option C) was rejected for v1 because (i) it requires source-level `#[cfg(feature = "...")]` syntax + compiler conditional-compile pass + cross-build-API-diff diagnostics, all substantial v1 surface for an unproven payoff; (ii) the forcing function (Tokio-shaped libraries with 10+ feature flags) doesn't yet exist for Kāra; (iii) reversibility is asymmetric — adding C in v2 is non-breaking, removing it in v2 breaks every dependent library. The Cargo-shaped axis (option B) was rejected outright as Kāra's biggest pain point to consciously avoid. The trigger to open the v2 RFC: if the "ship multiple packages" pattern becomes widely lamented in real Kāra usage.

### P6 — Reproducibility guarantee

**Question.** Does Kāra promise bit-exact reproducible builds across machines?

**What it requires.**
- Lockfile (P3).
- Toolchain version pin (P10).
- Deterministic codegen (no embedded timestamps, no random symbol names, sorted output).
- Deterministic linking (already partly handled by LLVM, but not guaranteed).
- No host-environment leakage (paths, hostname, user, locale).

**Options.**
- **Promise and enforce.** Hash artifacts; CI verifies bit-equality. Highest engineering cost; biggest user trust signal.
- **Promise and best-effort.** Document goal; don't enforce. Cargo's posture today.
- **Don't promise.** Most languages.

**Default lean.** Promise reproducibility; enforce in CI for the compiler itself; document the user-side requirements (lockfile committed, toolchain pinned). Treat any non-determinism as a bug. Aligns with the no-compromise posture.

**Resolution.** Locked verbatim with the lean. The bit-exact promise holds given (a) committed `kara.lock`, (b) pinned `karac-toolchain.toml`, (c) same target triple. Build-twice-and-hash CI for the compiler itself is tracked as a v1.1 deliverable.

### P7 — Build tool surface beyond compile

**Question.** Cargo offers `cargo doc`, `cargo bench`, `cargo publish`, `cargo install`, `cargo update`, `cargo audit`, `cargo fix`. Which are `karac` subcommands? Which are deferred?

**Inventory.**
- `karac doc` — generate HTML docs from source. Real work; depends on rustdoc-class doc-comment processing.
- `karac bench` — run benchmarks. Real work; needs a bench harness + statistical reporting.
- `karac publish` — push package to a registry. Blocked on P2 registry decision.
- `karac install` — install a binary from a package. Useful even pre-registry (path/git deps).
- `karac update` — update lockfile (P3).
- `karac audit` — scan deps for known vulnerabilities. Blocked on P2 registry having vulnerability metadata.
- `karac fix` — apply auto-fixable diagnostics. Already partly there via the `--output=jsonl` machine-applicable fix diff path; needs a CLI wrapper.
- `karac clean` — clear build cache.

**Default lean.** v1 has `build / run / check / fmt / query / test / init`. Add `update` + `clean` + `install` in v1.1 once `[dependencies]` resolution lands. Defer `doc / bench / publish / audit` to v1.5+ as they each need their own infrastructure.

**Resolution.** Locked verbatim with the lean, plus `karac vendor` joins the v1.1 tranche (P9), and `karac fix` lands opportunistically (no tranche gate — the existing `--output=jsonl` machine-applicable diff path is the underlying mechanism; `karac fix` is the CLI wrapper that applies the diffs). Final tranche split: **v1** = `build / run / check / fmt / query / test / init / repl / doc`; **v1.1** = `update / clean / install / vendor`; **v1.5+** = `bench / publish / audit`.

### P8 — Cross-compilation ergonomics

**Question.** `[build].target` is in the manifest. Where does target-specific configuration live?

**Sub-questions.**
- **Per-target `[dependencies]`?** A dep that's only needed when targeting WASM. Cargo has `[target.'cfg(target_os = "linux")'.dependencies]`.
- **Per-target features?** Activate `feature_x` only when targeting embedded.
- **Per-target lockfile entries?** Different versions of the same dep across targets.
- **Build matrix.** A single `karac build` that produces N artifacts for N targets, vs. one invocation per target.

**Default lean.** Per-target `[dependencies]` and `[target.X.profile]` blocks. One `karac build --target X` invocation per target; build matrix tooling (CI-side cross-compilation) is user concern, not compiler concern.

**Resolution.** Locked verbatim with the lean. Per-target features are subsumed by the P5 resolution (no per-package feature axis); per-target lockfile entries are unnecessary (single lockfile across targets is the P3 commitment).

### P9 — Vendoring / offline builds

**Question.** Air-gapped, regulated, or pure-CI environments may need offline builds with all deps pre-fetched. Commit?

**Options.**
- **`karac vendor`** — copy all resolved deps into a `vendor/` directory; subsequent builds use that directory only.
- **`karac fetch --offline-only`** — download once, refuse subsequent network access.
- **Don't commit.** Users who need this build their own tooling.

**Default lean.** Ship `karac vendor` as part of the v1.1 dependency-resolution work. It's cheap once resolution is wired and the air-gapped audience is non-negligible (regulated industries, enterprise security).

**Resolution.** Locked verbatim with the lean. `karac vendor` copies all resolved deps into `vendor/` at the project root; `karac build --offline` reads only from `vendor/` and refuses any network access.

### P10 — Toolchain version pinning

**Question.** Beyond `edition = "2026"` (a language-level commitment), is there a per-project compiler-version pin?

**Options.**
- **`karac.toml` field.** `[toolchain] version = "0.42"` in the manifest.
- **Separate file.** `karac-toolchain.toml` (rustup-style).
- **None.** User pins via system package manager / nix / etc.

**Tradeoffs.**
- In-manifest: one fewer file. Conflates language edition with compiler version.
- Separate file: rustup ecosystem precedent. Allows toolchain managers to read it without parsing the full manifest.
- None: punts to user. Loses the reproducibility guarantee from P6.

**Default lean.** Separate file (`karac-toolchain.toml`), rustup-style. Decouples "what version of the language" (edition) from "what version of the compiler" (toolchain). Read by the eventual `karaup` toolchain manager; absent file means "use installed compiler."

**Resolution.** Locked, with the original framing tightened to separate two distinct concerns the lean had implicitly conflated:

- **Library MSRV** (a constraint published with the package, consumed by the resolver): `[package].kara-version = ">=0.42"` field in `kara.toml`. Participates in dependency resolution; a package whose `kara-version` exceeds the active toolchain version triggers a structured resolution failure.
- **Binary toolchain pin** (a developer-experience preference for a specific checkout, consumed by toolchain managers): optional `karac-toolchain.toml` at project (or workspace) root. Schema (v1): `version` (required), `targets` (optional list of cross-compile targets to pre-install). Channels (stable / beta / nightly), components (separable tools beyond `karac`), and install profiles are deferred — Kāra has no release-channel split yet; the schema is forward-extensible.

The split mirrors Rust's `rust-version` (in `Cargo.toml`) vs. `rust-toolchain.toml` separation. In-manifest pinning was rejected because it conflates "constraint I publish" (load-bearing for transitive resolution) with "developer-experience preference for this checkout" (not relevant to consumers of the library); separating into two files matches both purposes cleanly.

### P11 — REPL / script dependency scope

**Status:** Added 2026-04-29 from a side conversation about whether Kāra needs a Python-style venv. Conclusion was "no" (the Cargo model already gives the isolation venv provides), but the question of which deps are in scope when `karac repl` or `karac run script.kara` runs is real and not yet specified.

**Question.** When `karac repl` (or a standalone-script invocation like `karac run foo.kara`) runs, what dependencies are in scope? When inside a project directory? When outside any project?

**Sub-questions.**
- **Outside any project.** Stdlib only? Or read a user-global manifest at `~/.kara/repl.toml`?
- **Inside a project (cwd contains `kara.toml`).** Auto-load project deps? Or require explicit `--manifest` to opt in?
- **Standalone scripts.** Does `karac run foo.kara` discover a manifest by walking up from the file's directory? Or only from cwd? Or only when explicitly passed?
- **Override.** A `--manifest path/to/kara.toml` flag to force a specific manifest? A `--no-manifest` to force stdlib-only?
- **REPL meta-command for ad-hoc deps.** A `:dep name = "1.2"` that mutates an in-memory manifest for the session? (Mirrors `:provide` from Item 10.)

**Why this matters.** Python solved this with venv (badly — global PATH manipulation, stateful "activate" ritual, `which python` lying). Cargo doesn't have a REPL, so the question hasn't come up there. Kāra has both a REPL/Jupyter surface (P0 in the roadmap) and a manifest-driven build, so the question lands directly.

Getting it right means a Python-origin user can `karac repl` in any directory without a confusing "package not found" error from stale state, and a Kāra developer in their project directory gets project deps without ceremony. Getting it wrong means either the friction Python users came to avoid, or the surprise of "why is this dep in scope, I didn't ask for it?"

**Options.**
- **A. Implicit project loading + explicit override.** `karac repl` walks up from cwd looking for `kara.toml`; if found, project deps are in scope. `--no-manifest` forces stdlib-only. `--manifest path` overrides. Mirrors `cargo` and `go run` behavior.
- **B. Explicit only.** `karac repl` is always stdlib-only unless `--manifest` is passed. Simpler mental model; more typing for the common case.
- **C. Implicit + REPL meta-command for ad-hoc deps.** A on top, plus `:dep` for adding deps in-session without editing the manifest.

**Default lean.** A for v1 (covers 90% of use cases without ceremony), C as the v1.1 enhancement once the dependency-resolution work from P1–P3 lands. B is too friction-heavy for the interactive surface that's a P0 priority.

**Resolution.** A for v1, C for v1.1. Locked verbatim with the lean for the main question. M1–M4 resolved as follows:

**M1 — `karac run foo.kara` walks up from the script's own directory, not cwd (option M1c).** The two original options were both wrong: M1a (walk up from cwd, symmetric with `karac repl`) creates the surprise "same script, two cwds, different deps in scope"; M1b (always standalone) is too friction-heavy for the same reason A beat B on the main question. M1c uses the script's filesystem location as its stable identity: `karac run ~/project/scripts/foo.kara` finds `~/project/kara.toml` regardless of where it is invoked from; `karac run /tmp/foo.kara` from inside `~/project/` does *not* see project deps because the script lives outside the project tree. `--manifest` and `--no-manifest` overrides apply to both `karac repl` and `karac run`.

**M2 — `:dep` writes to in-memory session state only.** No persistence. Symmetric with `:provide` (Item 10) — both are session-scoped state mutations with no implicit persistence. Session ends, dep is gone. The session-resumption use case (long-lived REPL sessions surviving restarts) is unproven for Kāra's audience; if it becomes a felt need, it lands as an explicit `:save-session` / `:load-session` snapshot mechanism, not via implicit persistence of `:dep` state. The simpler default avoids session-ID management, gc, and "which session am I in" complexity.

**M3 — Jupyter parity is tiered.** v1.1: `:dep` works in Jupyter via the existing kernel meta-command channel — no Jupyter-specific work needed. v1.5+: optional `%dep` cell-magic alias as a comfort feature for users coming from Python (`%pip install` muscle memory); pure aliasing, identical semantics. Deferred indefinitely: manifest-editor widget (ipywidgets-shaped surface is over-scope; the line-magic covers the use case).

**M4 — `:dep` and `:provide` are independent state mutations.** Both stack into session state; neither invalidates the other. If `:dep` swaps out a package whose types were referenced by an active `:provide` registration, the user gets a normal type error when they next try to use that provider — no special cross-reference tracking. Diagnostic: standard "type X not found" with the location pointing at the `:provide` site. The non-conservative alternative ("`:dep` invalidates affected providers automatically") would require tracking cross-package provider references through session state, which is real engineering for an edge case (swapping out packages mid-session is rare and recoverable).

## Item 7 — Cumulative runtime cost story

**Status:** RESOLVED 2026-04-30 — both deliverables shipped. (a) **Cumulative Cost Surface** subsection added to `docs/design.md § Performance Diagnostics` enumerating every silent runtime cost the compiler may insert, with order-of-magnitude estimates and links back to the section that introduces each one. (b) **`karac query cost-summary`** static-count contract added to `docs/design.md § Compiler Query API` (alongside `effects` / `ownership` / `concurrency` / `layout`); v1 reports the compiler's own bookkeeping of how many cost-insertion sites it emitted; runtime attribution (sampling-profiler-based) is deferred to post-v1. Implementation tracked under `docs/implementation_checklist.md` Phase 5.3; first ground-truth measurement target tracked under `docs/roadmap.md` Phase 8.

**Observation.** The "no-compromise" framing leans heavily on the claim that the compiler picks the right representation for you. In practice, the compiler reaches for a runtime-cost fallback in many places, and each fallback is well-specified individually:

- RC fallback when single-source ownership can't be proved (Feature 4 Part 4).
- Arc-wrapping inside `with_provider` boundaries (Feature 2, Provider-Rooted Resources — ~10–40ns per access acknowledged).
- Per-field RefCell-style borrow flags on `shared struct` mut-fields (1 byte/field, runtime panic on conflict).
- Per-fork runtime partitioning when parameterized resource keys aren't literals.
- REPL `--auto-clone` mode inserting `.clone()` calls.

There is nowhere in design.md that adds these up and asks "what does a real program actually pay?" Each individual fallback reads as cheap; the sum across a non-trivial program is not analyzed.

**Why this matters.** "No compromise" is an axiomatic claim about cost. If a programmer writes idiomatic Kāra and the compiler silently chooses RC + Arc + borrow-flag + partition-guard, the programmer's recourse is "rewrite to satisfy the dominance/distinctness analyses" — which is approximately as hard as writing lifetimes in Rust, just with different vocabulary.

The audit mechanism (`karac explain`, `warn_above_0`) catches individual decisions. It doesn't tell you "across this binary, you're paying 8% to RC fallback, 3% to provider Arc, 5% to partition guards." That's the number that determines whether the no-compromise pitch survives contact with a real workload.

**Resolution.** Both. Appendix locks the design's honesty story; tooling makes the claim auditable per-program. Static counts are sufficient for v1 — runtime attribution is acknowledged as a post-v1 follow-up so the spec stays honest about what `karac query cost-summary` does and does not measure.

**F1 — top-level section vs. subsection of Performance Diagnostics.** Subsection. The cumulative-cost story is the *load-bearing case* for Performance Diagnostics' existence; promoting it to a sibling top-level section oversells against the rest of Part 7. Performance Diagnostics was previously thin (three tiers — inline notes, summary report, suppression); the Cumulative Cost Surface subsection balances the chapter without disrupting its structure.

**F2 — Phase 5 (tooling) vs. Phase 8 (with stdlib).** Split. The query tool counts *static compiler decisions* (RC ops, Arc-provider-wraps, borrow-flag fields, partition-guard sites, auto-clone insertions) — that data exists the moment the relevant passes do, most of which are now complete (Phase 6 + Phase 7 closed). Static-count form ships with the rest of `karac query` in Phase 5.3. *Runtime attribution* (what fraction of wall-clock time is paid at each site) requires a sampling profiler over an instrumented binary against a representative workload — that's Phase 8+ once stdlib + a real workload exist. The honest split is in the spec because the static answer is the one the compiler can give without external infrastructure.

**F3 — first ground-truth measurement target.** **Parallax-lite**. A stripped-down precursor to Demo 1 (Parallax — Auto-Concurrency API Gateway, see `docs/demo_ideas.md`) — same shape (HTTP server, providers for upstream services, fan-out + join), narrower surface (one upstream instead of four, single resource per endpoint). Parallax-lite is the first program in the codebase with non-trivial Provider-Rooted Resources + auto-concurrency + (likely) RC fallback in one place — the right shape to validate that the Cumulative Cost Surface table reflects what the programmer actually pays. Mend (Demo 2) is too synthetic — single-function transcripts. Discrepancies between expected and observed costs feed back as edits to the Cumulative Cost Surface table.

## Item 8 — Auto-concurrency cost model is unspecified

**Status:** RESOLVED 2026-04-30 — Option B: defer with an explicit "v1 status" disclaimer in Feature 5 + classification entry in Seed Classification + tracked v1.x specification work. The cost model was the most consequential gap in the second-pass audit, but the right v1 answer is **honesty, not premature specification**. Specifying numbers without empirical measurement risks locking in heuristics that turn out to be wrong; the right path is a degenerate interim model in v1, ground-truth measurement against Parallax-lite, then the full specification as a v1.x deliverable.

**Observation.** Feature 5 references "the cost model" multiple times — "the compiler may choose not to parallelize if the cost model estimates that scheduling overhead exceeds the speedup." The cost model itself is never specified. Concretely:

- Pure call inside a `for` loop body — does the compiler parallelize across iterations? On what basis?
- Two `reads` on different resources, but both are fast in-memory lookups — fork or fuse?
- Three `reads` where one is fast and two are slow — fork all three or only the slow pair?
- Loop body touches a parameterized resource; key distinctness is dynamic — fork speculatively or serialize?

Every real program lives or dies on these heuristics, and they're not in the spec.

**Why this matters.** The headline pitch is "write sequential code, get free parallelism speedup." The credibility of that claim is entirely the cost model's. Right now the parallelization story holds up only at the level of "two independent `reads` on different resources fork" — a claim the spec demonstrates but doesn't generalize. Without a specified cost model, a Phase 8 codegen implementer is making the call independently, and two compiler versions can disagree on which programs parallelize.

This also reaches the Reported Behavior layer (§ Specification Layers): if `karac explain` shows different parallelization decisions across versions for the same source, that's exactly the within-edition reported-behavior drift the spec is otherwise careful about.

**Resolution.** Option B for v1; Option A as a v1.x deliverable. The disclaimer ships now (cheap, honest, locks the v1 surface); the specification work is sequenced behind Parallax-lite measurement so the eventual numbers reflect real workloads. Option C (configuration knob) is wrong for the no-compromise posture — smart defaults are the deliverable, not "the user picks."

**G1 — Reported Behavior or Implementation Freedom.** **Reported.** `karac query concurrency` already exposes parallelization decisions with reasoning, and tools (CI dashboards, code-review bots) will key on them. Reported is the only honest classification: stable within a compiler release, may improve across releases (every change itemized in release notes — same discipline as inferred private-function effects). *Semantic independence* (whether two operations *may* parallelize) stays Guaranteed; only *will the compiler parallelize them* is reported. Added to design.md § Specification Layers > Seed Classification.

**G2 — Deterministic across runs?** **Yes — deterministic per (source, compiler version, target profile).** The cost model runs at compile time on a function's structure; decisions are pure functions of those three inputs. Within that key, the same source must always yield the same parallelization graph. Across compiler versions or target profiles, decisions can differ (per G1). Note this is orthogonal to *runtime* scheduling determinism — the scheduler retains its existing freedom over task placement and ordering across threads (Implementation Freedom under § Specification Layers). The cost model decides *whether* a group forks at compile time; the scheduler decides *how* a forked group is placed at runtime.

**G3 — Mend or Parallax-lite as first benchmark.** **Parallax-lite.** Same answer as Item 7's F3, and for the same reason: Mend (Demo 2) is single-function transcripts about AI writing Kāra — it has no parallelization shape worth measuring. Parallax-lite has the actual fan-out + join workload (HTTP server, multiple providers, multiple resources) that exercises every cost-model heuristic — the loop-body parallelization rule, the fork threshold, the distinctness policy under dynamic keys. Same workload as Item 7's Cumulative Cost Surface measurement; one binary, two analyses. Tracked in `roadmap.md` Phase 8 under "Parallax-lite — first ground-truth measurement workload."

**Follow-up implementation work.**
- ✅ design.md Feature 5 → "Cost Model — v1 Status" subsection (between `seq { }` discussion and Runtime).
- ✅ design.md Seed Classification → cost-model decisions listed under Reported Behavior with cross-ref to Feature 5.

## Item 9 — Effect polymorphism viral propagation + monomorphization scaling

**Status:** RESOLVED 2026-04-30 — Option A: accept and document, with mitigations and escape hatches named explicitly. Both 9a and 9b are consequences of correct decisions (soundness for 9a, precision for 9b); the right v1 answer is to (a) name the trade-offs in the spec so users can plan around them, (b) document the first-class mitigations that already exist (`with E` over `with _` when threading; `dyn Trait` as the monomorphization escape hatch), and (c) ship the *tracking mechanism* for monomorphization in v1 while deferring the *budget threshold* to v1.x once Phase 7.1 has empirical data — the same pattern as Item 8's cost model. Option B (effect-joining call-site modifier) and Option C (implicit `with _` propagation) both stay deferred: B is worth revisiting if Phase 7.1 data shows the monomorphization explosion is severe; C re-opens the "declared vs. inferred" question Feature 2 carefully closed.

**Observation.**

**9a. `with _` is viral through the stdlib.** The convention that unbound trait methods declare `with _` (e.g. `Iterator.next`) is correct for soundness, but it means every public function that touches an iterator must declare `with _` itself. With the heavy stdlib use the language encourages, most public signatures will become `with _ with _ with _` — defeating the documentation value that explicit declarations were supposed to provide.

**9b. Effect-polymorphic monomorphization explodes.** Compound generics `[T, with E]` × N effect variables × M distinct closure-effect sets at call sites = `M^N` instances. With effect variables resolved by unification rather than join (Feature 2), every distinct call shape produces a new monomorphization. Codegen size and compile time aren't budgeted anywhere.

**Why this matters.** These aren't bugs — they're consequences of correct decisions (soundness for 9a, generic-by-default precision for 9b). But the cumulative effect on a real program with iterator-heavy code and several effect-polymorphic combinators is a signature surface that loses its documentation value (9a) and a binary size that grows quadratically or worse with combinator depth (9b). The doc treats both as solved-by-construction. They're solved at the formal level; the ergonomic and engineering consequences are not addressed.

**Resolution.** Option A. The cost-properties subsection lands now; the monomorphization tracking mechanism lands in Phase 7.1 alongside `karac query monomorphization`; the budget threshold lands in v1.x after Phase 7.1 measurements exist. Both 9a's `with E` mitigation and 9b's `dyn Trait` escape are existing parts of the design — the spec edit names them explicitly so users hit the cost properties section before reaching for an unsafe shortcut.

**H1 — Iterator-heavy worked example from the stdlib.** A generic public function operating on `I: Iterator` already pays the wildcard cost: `pub fn enrich[I: Iterator](items: I, f: Fn(I.Item) -> Output with _) -> Vec[Output] with _` carries two `with _` slots — one on the closure, one on the function. Both are required for soundness (`Iterator.next`'s `with _` ceiling makes the source effects unknown until the impl is concrete) but they encode "any effects" rather than "this function does X." The mitigation is named in the new subsection: prefer `with E` over `with _` whenever the effect identity threads through the signature. The same `enrich` rewritten as `pub fn enrich[I: Iterator, with E](items: I, f: Fn(I.Item) -> Output with E) -> Vec[Output] with E` is precise — the result's effects are exactly the closure's effects. Iterator adaptors already use this pattern (`map[U, with E]`, `filter[with E]`, `fold[A, with E]`).

**H2 — Codegen monomorphization budget.** Two-step. **v1 ships the tracking mechanism** (compiler counts per-generic instantiations during type checking / lowering, exposes via `karac query monomorphization`); **v1.x ships the budget threshold** (`--monomorphization-budget=warn:N,error:M` with default thresholds tuned against Phase 7.1 codegen output on real programs — Parallax + stdlib-heavy combinator code). Picking 100 vs. 500 vs. 1000 without empirical data risks locking in numbers that turn out to be wrong, exactly the trap Item 8's cost-model resolution avoids. Mechanism is non-breaking; tightening the default later does not break anyone (and users who want stricter ceilings can opt in via the flag before the default lands). Tracked in `implementation_checklist.md` Phase 7.1.

**H3 — `dyn Trait` as monomorphization escape.** **Yes — confirmed by construction.** Dyn dispatch erases the type, so there is exactly one instantiation per dyn-using site; the method effect at that site is `with _` (the impl is unknown) or the trait's per-method declaration (if the trait declared a ceiling), which the caller declares once at the call boundary. Trade-off: indirect dispatch in exchange for single instantiation. The lever is already first-class in the design (see Effect Polymorphism > `dyn Trait` subsection); this resolution names it explicitly as the answer to monomorphization pressure so library authors who hit the budget have a ready escape hatch instead of redesigning their generic API.

**Follow-up implementation work.**
- ✅ design.md Effect Polymorphism → "Cost Properties — Viral `with _` and Monomorphization Surface" subsection covering 9a (named mitigations: prefer `with E`; declare per-method ceilings on library traits), 9b (mechanism in v1, budget number in v1.x, `dyn Trait` escape hatch).
- ✅ design.md Compiler Query API → `karac query monomorphization` listed alongside the other queries; output schema documents per-generic instantiation tables; future `--monomorphization-budget` flag described.
- ✅ design.md Seed Classification → monomorphization counts classified as Reported Behavior (counts may shrink across releases via improved de-duplication; underlying instantiation-identity rule is Guaranteed).
- ✅ implementation_checklist.md Phase 7.1 → "Monomorphization tracking + `karac query monomorphization`" entry plus "`--monomorphization-budget=warn:N,error:M` flag (v1.x)" entry tracking the deferred budget threshold.

## Item 10 — REPL ↔ provider-rooted resources collide

**Status:** RESOLVED 2026-04-30 — Option A: `:provide R = expr` / `:end-provide R` REPL meta-commands (and `%provide` / `%end-provide` Jupyter magics) that compile in the saved session as `with_provider[R](expr, || { /* cells */ })` blocks. The trade-off is named explicitly in the spec — bindings declared inside the provide scope are scoped to it (visible only within, not after `:end-provide`) — and the REPL surfaces "name not in scope" eagerly when a later cell references such a binding. This consistency-with-the-file-form rule turns the closure-capture concern (I2) into a special case of the binding-scope rule, eliminating special-case logic. Jupyter widget UI (active-provider panel) is committed as a Tier 3 stretch goal, presentation-only on top of the same magics.

**Observation.** `with_provider` (Feature 2) is strictly scoped — the provider's effects are consumed at the boundary; you can't "stash" a provider in a global. The Interactive Evaluation Model (REPL session) wants the opposite: set up a fake DB in cell 1, run analysis cells 2–10 against it, swap in a real DB in cell 11, continue. The REPL and provider-scoping rules were designed in different parts of the doc and aren't reconciled.

**Why this matters.** Provider injection is the load-bearing mechanism for testing — and Tier 2 of the interactive surface (Jupyter kernel) is positioned as a data-science differentiator. If providers can't span cells, the canonical data-science workflow ("connect once, analyze for an hour") is incompatible with the safety model.

The REPL's implicit-`fn main` framing makes every cell an inner statement of one growing function body. A provider scope opened in cell 1 *should* be syntactically capable of spanning cells N..M because cells are statements within `main`. But the REPL also reuses cell scope for new bindings — the scoping rule needed is "block-scoped from the user's perspective, function-scoped from the implicit-main perspective." Those don't trivially compose.

**Resolution.** Option A. Both Option B (forbid cross-cell providers) and Option C (special REPL-mode semantics) are wrong: B breaks the canonical data-science workflow; C creates a divergent dialect that contradicts the "interactive surface is a faster path to the same language, not a divergent dialect" closure of § Interactive Evaluation Model. Option A keeps the file-form `with_provider` semantics intact and adds two meta-commands as syntactic sugar; the binding-scope cost is real and documented.

**I1 — Implicit-`fn main` framing permits cell-spanning `with_provider`, with an explicit binding-scope restriction.** Cells are statements inside the implicit `fn main()` body. A `with_provider[R](expr, || { stmt1; stmt2; ... })` is itself a statement; wrapping cells N..M inside one is just nesting statements in a block. **The trade-off, named:** bindings declared inside the provide scope are visible only within that scope — they're block-local in the saved form, and the REPL projects the same scoping. After `:end-provide R`, references to those bindings fail with "cannot find value `X`" + a notebook-aware hint naming the provider scope that closed and the cell where `X` was declared. This is the price of consistency with the file form; the REPL does not implicitly hoist bindings out of provider scopes (that would be a divergent dialect). Users who need bindings to outlive `:end-provide` either (a) delay closing the scope, (b) write the value to an external sink before closing, or (c) inline `with_provider` blocks directly instead of using the meta-command.

**I2 — Closure capture is not a special case; it's the binding-scope rule applied to closure values.** A closure that captures a provider-resource binding is itself a binding inside the provide scope. If the closure escapes (assigned to a name referenced after `:end-provide`), the saved form is a closure-escape error and the REPL surfaces it eagerly via the same diagnostic shape as I1's binding case. No separate closure-capture rule is needed — the binding-scope restriction subsumes it.

**I3 — Both the magic and the widget, staged: magic in v1.1, widget in Tier 3 (post-MVP).** Jupyter MVP magic surface ships `%provide R = expr` / `%end-provide R` in parity with the REPL meta-commands using the same compilation path. A richer widget UI (active-provider panel showing resource bindings, opened-in-cell, one-click `%end-provide`, per-cell-effect timeline) is a Tier 3 stretch goal — presentation-only on top of the same `%provide`/`%end-provide` machinery. Saved-session export ignores widget actions; the underlying meta-command log is what survives.

## Item 11 — `shared struct` → `par struct` migration is mandatory and manual

**Status:** RESOLVED 2026-04-30 — B + D shipped (Tier 2 perf note at definition time + `karac migrate shared-to-par` tool). Added 2026-04-29.

**Observation.** Add a single `par {}` access to a previously-sequential program using `shared struct Elevator`, and the compiler rejects it and demands a per-field `Mutex`/`Atomic` rewrite (Feature 4 Part 5). The structured fix-diff is a real tool, but the migration is a forced refactor of every consumer of `Elevator`, often spanning the codebase.

**Why this matters.** This is exactly the friction Rust gets criticized for — `Send`/`Sync` retrofitting, Cell→Mutex migrations, and the cascade of API changes that follow. Kāra's design avoids the *annotation* burden but inherits the *migration* burden. The spec presents the structured fix-diff as a feature; in real adoption it's "the compiler told me to rewrite half my code."

The deeper issue: `shared struct` is the intuitive default for "shared mutable state." `par struct` is the correct one for parallel access. Choosing wrongly costs a refactor — and you can't tell at definition time which one a future consumer will need.

**Options.**
- **A. Lazy promotion.** The compiler infers `par struct` requirements from cross-task reachability and promotes `shared struct` automatically — at the cost of mandatory `Mutex`-ish per-field overhead applied silently. Goes against the no-silent-cost principle.
- **B. Per-field migration suggested at definition time.** When the compiler sees a `shared struct` with mut fields, emit a perf note ("this struct cannot cross task boundaries; if you might want that, define as `par struct` from the start"). Cheap; nudges users toward the right default.
- **C. Make `par struct` the default; `shared struct` is the opt-in narrow form.** Reverse the polarity. Default is the parallelism-safe form; the narrow form is named explicitly. Aligns with the language's parallelism-first posture.
- **D. Structured-edit tooling.** Beyond the fix-diff, ship `karac migrate shared-to-par` that performs the rewrite across the workspace. Real engineering; users-meet-the-tool moment.

**Default lean.** B + D. B is cheap and prevents the surprise; D is the production-grade answer when the migration does happen. C is tempting but is a v1 redesign.

## Item 12 — Closure capture mode footgun

**Status:** RESOLVED 2026-04-30 — Option A shipped (explicit `move` / `ref` / `mut ref` prefix on closure expressions, per-closure semantics, body-vs-declared conflict diagnostic). Added 2026-04-29.

**Observation.** Feature 4 Part 1 says closure capture mode is inferred from the *first use* in the body. Mixed use resolves to consume. This rule is footgun-prone: refactoring a closure body to add an early `.clone()` (or moving an existing line above another) may flip the capture mode and silently change RC decisions in the enclosing function.

**Why this matters.** The "deterministic across compiler versions" promise (Feature 4) holds only if the use-predicate table is frozen forever. But the user-side determinism is fragile: a benign refactor changes capture mode, which changes parameter-mode inference upstream, which can flip an `own` to a `ref` (or trigger RC fallback) at the call site. The change is local in source but global in performance characteristics. `karac explain` will catch it; programmers won't run `karac explain` on every refactor.

**Options.**
- **A. Explicit capture mode on closures.** `|x| body` infers as today; `move |x| body` consumes; `ref |x| body` borrows; `mut ref |x| body` mut-borrows. Removes the implicit-flip footgun by making the choice explicit at the syntactic site. Mirrors Rust's `move` keyword.
- **B. Diagnostic on capture-mode change.** When a refactor changes inferred capture mode, the compiler emits a perf note pointing at the changed first-use site. Requires per-version baseline; impractical without a compile-time diff tool.
- **C. Leave as-is; rely on `karac explain`.** Document the rule, trust the user.

**Default lean.** A. Explicit `move` / `ref` keywords on closures are well-precedented and align with the rest of the language's "ownership is declared, checked from body" posture. The implicit inference can stay as the default; explicit keywords win when the user wants stability.

**Resolution.** Option A. Implicit first-use inference (current Rule 2) stays as the default — annotation-free, body-checked, matches the rest of the language's posture. The fragility footgun is real but local: when the user wants stability against refactoring, the explicit prefix pins the contract at the closure expression site. B (per-version diagnostic on inferred-mode change) is impractical without compile-time diff tooling; C (do nothing) leaves the footgun visible only to programmers who run `karac explain` on every refactor, which is the wrong default.

**K1 — This is a design addition.** Current Rule 2 specifies *inference only*: capture modes are determined by the body's first-use scan, with no syntactic surface for an explicit override. Rule 2½ adds the override; Rule 2 stays as the default behavior when no prefix is written.

**K2 — Conflict diagnostic: declared mode is the floor, body usage may be weaker but never stronger.** Read modes as ordered by strength: `ref < mut ref < move`. Body usage demands a *minimum* mode (read → `ref`, mutate → `mut ref`, consume → `move`). The declared mode must be ≥ body usage. Equal is consistent; declared strictly greater than body is fine and emits a perf note only in the specific `mut ref`-declared / read-only-used case (`perf[unused-mut-capture]`); declared less than body is a hard compile error at the closure expression, naming the capture and the offending use line. Symmetric to declared-vs-body parameter-mode mismatch on `fn`. The diagnostic shape mirrors the once-callable closure diagnostic (Rule 3): name the specific capture, the line of the offending use, offer the concrete fix.

**K3 — Explicit override is warranted iff (i) the inferred property changes a contract visible to a different scope AND (ii) the inference is locally fragile.** Closures meet both criteria — capture mode visibly changes the enclosing function's ownership analysis, and the use-predicate scan flips on body line ordering. Other inferred properties in v1 don't:

- Function parameter modes are *signature-declared* (`bare T` / `ref T` / `mut ref T` is at the signature) — no fragility.
- Private-function effect inference is *quarantined behind the public boundary* — public functions explicitly declare effects.
- RC vs owned is body-inferred but *observable via Tier 1 inline notes* (`perf[rc-fallback]`) and `karac explain`.

Closure capture is the only v1 case that hits both criteria *and* lacks a default visibility surface. The principle isn't "all inferred properties get explicit overrides"; it's "explicit overrides where the local-fragility cost lands without the user having visibility to catch it."

**Per-closure, not per-capture.** `move |x, y| body` forces every captured name to move into the closure regardless of body usage. This matches Rust's `move ||` semantics and is the simple shape that handles the common case (user wants stability against refactoring). Per-name override syntax (one capture by move, another by ref) is deferred — if real programs surface the need, the per-closure form composes forward without breakage.

**Follow-up implementation work.**
- ✅ design.md § Closure Behavior → new "Rule 2½: Explicit capture-mode override on closure expressions" subsection: three optional prefix keywords (`move` / `ref` / `mut ref`), per-closure semantics, K2 conflict table (7 rows), `perf[unused-mut-capture]` Tier 2 note for unused `mut ref`, K3 principle for when explicit override surface is warranted.
- ✅ syntax.md § 5.11 → closure grammar extended with optional `CAPTURE_MODE` prefix (`move | ref | "mut" "ref"`); examples added; cross-link to design.md Rule 2½.
- ✅ implementation_checklist.md Phase 2 → new entry covering parser/AST changes (closure prefix tokens, `Closure.capture_mode: Option<CaptureMode>` field, walker-site updates across 16 sites in `src/`); existing OwnershipChecker closure entry extended with step 9 (honor declared mode when present, K2 table validation, 7 conflict tests + 1 bare-form regression).

## Item 13 — Effect-system asymmetries

**Status:** RESOLVED 2026-04-30 — A + D shipped as new § "Effect-System Design Principles" (three named principles + forward-compatibility note). Documentation-only; no compiler change. Added 2026-04-29.

**Observation.**

**13a. Cancellation isn't an effect, but `panics` is.** The spec's rationale for not making cancellation an effect ("would force every `-> Result` function to widen") applies equally to `panics`, which the spec does require. The two aren't reconciled.

**13b. `allocates(Heap)` is default-permitted; the opt-out is the real-time guarantee.** Real-time guarantees depend on omitting `allocates(Heap)`. This is at odds with the elsewhere-stated principle that "absence of declaration is denial of capability." Most other effects work the opposite way — you declare the capability you need.

**13c. `transparent` effects are excluded from conflict analysis but counted for inference.** The split is correct (you want logging not to serialize requests; you want logging to show in `karac explain`), but it creates a category of effects with asymmetric standing across the analysis pipeline. New effect designers will repeatedly ask "is this verb `transparent` or not?" with no clear procedure.

**Why this matters.** Asymmetries are a tax on language users and on future spec authors. Each one needs a rationale that holds up to "why isn't this rule the same as the others?" The current spec has rationales for each individually but no overarching principle.

**Options.**
- **A. Document each asymmetry with explicit rationale in the spec.** Cheap. Lets future spec edits know the choice was deliberate.
- **B. Resolve 13a by making cancellation an effect.** Probably wrong (spec rationale for not doing this is solid), but worth naming as the alternative.
- **C. Resolve 13b by inverting `allocates(Heap)` default.** Heap allocation is opt-in via `allocates(Heap)` declaration. Embedded becomes the default-permitted profile; app/lib programs declare the capability. Significant migration cost; probably correct long-term.
- **D. Resolve 13c with a procedure for classifying new effects.** "An effect is `transparent` iff it has no observable program-state side-effect that another concurrent operation could depend on." Frees future authors from the case-by-case decision.

**Default lean.** A + D. B is wrong; C is too disruptive for v1. The asymmetries are real but each is justified — making the justifications visible and giving future authors a procedure is the cheap, correct response.

**Resolution.** A + D, articulated as new § "Effect-System Design Principles" — three named principles that reconcile all three asymmetries under one frame. The single insight is that *effects are capabilities the function exercises*, and the three asymmetries fall out as deliberate consequences. B (cancellation as effect) and C (invert `allocates(Heap)` default) stay rejected for v1 — B contradicts the capability framing; C is principled for v2 but a disruptive edition-gated migration. The principles document the present state and gate future additions without precluding C as a v2/v3 edition change.

**The three principles, named.**

1. **An effect is a capability the function exercises, not a control-flow event imposed externally.** Resolves 13a: cancellation is external interruption (parent cancels child), surfaced through the existing `Result` channel; `panics` is a capability the function chose to exercise. Same shape as the existing resource-vs-execution-effect distinction, made explicit.

2. **Default-permitted-then-deny is appropriate for *substrate* effects; declare-then-permit is appropriate for *scoped capabilities*.** Resolves 13b: heap allocation is exercised ambiently by ~90%+ of functions in non-RT programs; declaring it would be noise rather than information. Substrate set in v1 is exactly `{allocates(Heap)}`; future substrate proposals must show ambient ubiquity. Profile-level inversion (`embedded`/`isr` set `default_no_alloc`) is the right granularity because the substrate/scoped distinction is program-class-dependent.

3. **An effect is `transparent` iff it has no observable program-state side-effect that another concurrent operation in the same program could depend on.** Resolves 13c: gives future effect authors a one-line decision procedure. Tested against `traces` (append-only sink, no readback → transparent ✓), hypothetical `metrics` (commutative writes → transparent ✓), `reads(DB)` (downstream dependency → not transparent ✓), `panics` (currently not transparent — clean partition between resource verbs and `transparent`).

**L1 — Yes. § "Effect-System Design Principles" is the new closing subsection of Feature 2** (between the existing Effect Semver Rules subsection and Feature 3). Carries the three principles plus a forward-compatibility note covering: (i) what a future edition could change (e.g., inverting heap default — Item 11 / option C path), (ii) the bar for new substrate proposals, (iii) how the transparent set may grow under Principle 3.

**L2 — D classifies `traces` correctly and extends to the natural class of logging/tracing/append-only-sink effects.** Procedure passes `traces`, `metrics` (commutative), and `audit_log` (append-only). Procedure fails `reads(X)` / `writes(X)` (downstream dependency present) — also the right answer. Edge cases that pass D but might surprise: an effect that writes to an external sink the program *does* read back from (e.g., a hypothetical `cache_write` followed by a `cache_read`) would correctly be non-transparent — the procedure phrases the test as "another concurrent operation in the same program could depend on," which catches the readback case. Case-by-case judgment is preserved for genuine novelty that the procedure cannot decide unambiguously.

**L3 — Edition-gated migration is the path for option C, deferred to v2.** v1 doesn't paint into a corner: documenting the substrate/scoped distinction as deliberate doesn't preclude future inversion. A future edition (Kāra 2.0) could flip the `allocates(Heap)` default with `karac migrate` automating the rewrite (add `allocates(Heap)` to every fn that transitively allocates, carried behind the existing edition mechanism for cross-edition non-breaking). v1 work is just naming the choice and the principle behind it; v2 inherits a clean foundation if the inversion proves correct empirically.

**Follow-up implementation work.**
- ✅ design.md → new closing § "Effect-System Design Principles" subsection in Feature 2 (between Effect Semver Rules and Feature 3): three named principles (capability-not-control-flow; substrate vs scoped; transparent classification procedure), each tied to one of 13a/13b/13c, plus a forward-compatibility note covering future substrate proposals, the transparent set, and the v2/v3 edition path for Item 11 / option C.
- *(no implementation_checklist.md entry — Item 13 is documentation-only; the principles are a design policy that future effect designers consult, not work to be tracked.)*
