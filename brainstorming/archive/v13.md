# Brainstorming v13 — Syntax Naturalness Review

A review of Kāra's surface syntax for naturalness and programmer readability. Each item identifies a specific syntax pattern that could be clearer or more consistent, proposes concrete alternatives, and notes trade-offs. The review reads example code from `examples/` and the full grammar from `docs/syntax.md` through the lens of "could a programmer from Rust/Python/TypeScript/Go read this without a manual?"

Organized into three tiers by impact. No unilateral changes — each item is a design discussion.

## Tier 1: High Impact

### 1. ~~Effect annotations lack a delimiter from the return type~~ ✅ RESOLVED

**Decision: Always require `with` before any effect clause. Effects are space-separated (no `+`, no `,`). `+` may be added as optional sugar later (non-breaking).**

This also resolves item #5 (effect separator inconsistency) — space separation replaces both `,` in function signatures and `+` in effect groups. Effect group definitions keep `+` because they are declarative equations (`GroupName = A + B`), not annotation lists.

`with` becomes the universal "effects start here" marker. Every function signature reads as: `fn name(params) [-> Type] [with effects] [contracts] { body }`. Space separation was chosen over `+` because: (1) effect verb keywords (`reads`, `writes`, `sends`) already provide visual separation; (2) starting with spaces is non-breaking if `+` is later added as optional sugar; (3) `+` adds noise without semantic meaning (effects are a set, not a sum).

**Examples:**

```kara
fn greet() { ... }                                          // pure — unchanged
fn add(a: i64, b: i64) -> i64 { ... }                      // pure — unchanged
fn log(msg: String) with writes(FileSystem) { ... }         // was: writes(FileSystem)
fn save(user: User) -> Result[(), Error]
    with writes(UserDB) { ... }                             // was: writes(UserDB)
fn process(id: i64) -> Result[Report, Error]
    with reads(UserDB) writes(OrderDB) sends(Network) { ... }
fn f() with OrderProcessing { ... }                         // unchanged
fn f() with OrderProcessing reads(FileSystem) { ... }       // was: with OrderProcessing, reads(FileSystem)
fn run[T: Processor](p: T) -> Result[Output, Error]
    with _ { ... }                                          // unchanged
extern "C" fn write(fd: i32, buf: *const u8, count: usize) -> isize
    with writes(FileSystem);                                // was: writes(FileSystem);

// Trait methods
fn process(self, data: Data) -> Result[Output, Error] with _;

// Effect groups — definition syntax unchanged (+ is a declarative equation)
effect group OrderProcessing = Validation + Fulfillment;
```

**Breaking change:** Yes — every effectful function without `with` needs `with` added, and `,` between effects in function signatures becomes a space. Non-breaking for pure functions and functions already using `with` with a single effect.

---

**Original problem and options (kept for context):**

**Location:** syntax.md §3.1 (Functions), §7.2 (Placement).

**Problem.** There is no syntactic marker between the return type and the effect clause. A reader scanning left-to-right hits `-> Result[(), Error]` and then `writes(UserDB)` with nothing separating them:

```kara
fn save(user: User) -> Result[(), Error] writes(UserDB) { ... }
fn process(id: i64) -> Result[Report, Error]
    reads(UserDB), writes(OrderDB), sends(Network) { ... }
```

The `with` keyword already exists for groups and polymorphism, but bare verbs don't use it, creating an asymmetry:

```kara
fn f() -> T reads(X) { ... }           // bare verb — no introducer
fn f() -> T with Group { ... }         // group — uses `with`
fn f() -> T with _ { ... }             // polymorphic — uses `with`
fn f() -> T with Group, reads(X) { }   // mixed — `with` applies to group only
```

**Why it matters.** The function signature is the most frequently read construct in any language. A missing delimiter between return type and effects forces the reader to use context (recognizing `reads`/`writes`/`sends` as keywords) to find the boundary. For unfamiliar readers or LLM consumers, this is a real parsing obstacle.

**Options:**
- (a) Always require `with` before any effect clause.
- (b) Require `with` only when effects follow a return type.
- (c) Leave as-is.

**Separator sub-question (discussed during resolution):**
- `+` between effects — mirrors trait bounds (`where T: A + B`) but adds noise; semantically meaningless since effects are a set.
- `,` between effects — looks like `with` applies only to the first item.
- Spaces — minimal; verb keywords provide natural visual separation; non-breaking path to add `+` later as optional sugar.

**Recommendation:** (a) with space separation.

**Fix type:** `design-discussion`

### 2. ~~`ensures |result| expr` uses closure syntax for a non-closure binding~~ ✅ RESOLVED

**Decision: Use parentheses — `ensures(result) expr`. Matches `errdefer(e) { ... }` binding syntax.**

Both `ensures` and `errdefer` bind a value produced by the runtime to a name for use in a checking/cleanup expression. Using the same paren-binding syntax for both reduces cognitive load. The pipe syntax (`|result|`) misleads readers into thinking `ensures` takes a closure, raising false questions about capture semantics and effect inference.

**Examples:**

```kara
fn binary_search[T: Ord](haystack: ref Vec[T], needle: ref T) -> Option[i64]
    with reads(SearchIndex)
    requires haystack.is_sorted()
    ensures(result) match result { Some(i) => haystack[i] == *needle, None => true }
{ ... }

fn deposit(self, amount: i64) -> (Account, Receipt)
    requires amount > 0
    ensures(result) result.0.balance == old(self.balance) + amount
{ ... }
```

**Breaking change:** Yes — all `ensures` clauses need updating. Mechanical find-replace: `ensures |` → `ensures(`, then closing `|` → `)`.

---

**Original problem and options (kept for context):**

**Location:** syntax.md §3.1 (Functions — CONTRACT), design.md § Contracts.

**Problem.** `ensures` uses pipe-delimited binding (`|result|`) which is closure syntax, but `ensures` is not a closure. `errdefer` already uses parens for the same pattern: `errdefer(e) { ... }`.

**Options:**
- (a) Use parentheses: `ensures(result) expr`. ← **chosen**
- (b) Use colon: `ensures result: expr`.
- (c) Leave as-is.

**Fix type:** `design-discussion`

### 3. ~~`providers { } { }` double-block has no separator keyword~~ ✅ RESOLVED

**Decision: Add `in` between the bindings block and the body block — `providers { ... } in { ... }`.**

`in` is already a keyword (`for x in collection`), so no new keyword is needed. It reads as "given these providers, *in* their scope, run this code." The `} {` double-block was the only construct in Kāra with two adjacent braced blocks and no separator — it looked like a formatting accident.

**Examples:**

```kara
fn main() -> Result[(), AppError] {
    let config = load_config("app.toml")?;

    providers {
        OrderDB => PostgresOrderDB.connect(config.db_url)?,
        UserDB  => PostgresUserDB.connect(config.db_url)?,
        Cache   => RedisCache.connect(config.cache_url)?,
    } in {
        run_server()
    }
}
```

**Breaking change:** Yes — all `providers` blocks need `in` inserted between `} {`. Mechanical.

---

**Original problem and options (kept for context):**

**Location:** syntax.md §3.15 (Providers Block).

**Problem.** Two consecutive braced blocks with no separator keyword — the only such construct in Kāra.

**Options:**
- (a) Add `in` separator. ← **chosen**
- (b) Use `with providers` as introducer.
- (c) Flatten into a single block.
- (d) Leave as-is.

**Fix type:** `design-discussion`

### 4. ~~`self` in refinement type predicates is misleading~~ ✅ RESOLVED — no change

**Decision: Keep `self` in refinement type predicates. No change to syntax.**

On closer examination, `self` works naturally in this context — English "itself" reads correctly: `type NonZero = i32 where self != 0` = "an i32 where *itself* is not zero." Keeping `self` also maintains consistency with struct invariants (one rule: `self` refers to "the thing being described" in all declarative predicates) and avoids introducing a new special identifier.

**Why not `it`:**
- `it` is heavily used in test frameworks (Jest, Mocha, RSpec) and as a common variable name (`for it in items`). Reserving it for a narrow use case would block legitimate identifiers.
- Kāra may want `it` for BDD-style testing or Kotlin-style implicit lambda parameters in the future.
- The benefit over `self` is small — both are readable, but `self` requires no new reserved word.

**Why not `_`:** `_` means wildcard/discard everywhere in Kāra (`let _ = ...`, `match _ => ...`, pipe placeholder). Using it to mean "the value under test" would invert its meaning.

---

**Original problem and options (kept for context):**

**Location:** syntax.md §3.12 (Type Aliases and Refinement Types), §3.13 (Distinct Types).

**Problem.** `self` in a type predicate is outside a method context, which could surprise programmers who associate `self` with instance methods.

**Options:**
- (a) Use `it`.
- (b) Use `value`.
- (c) Use the type name itself.
- (d) Leave as-is. ← **chosen**

**Fix type:** `no-change`

## Tier 2: Medium Impact

### 5. ~~Effect groups use `+` but function effect lists use `,`~~ ✅ RESOLVED (via item #1)

**Decision: Space-separated effects in function signatures. Effect group definitions keep `+` (declarative equation syntax). See item #1 for full rationale.**

Function signatures use spaces: `with reads(UserDB) writes(OrderDB) sends(Network)`. Effect group definitions keep `+` because they are equations combining named groups: `effect group OrderProcessing = Validation + Fulfillment`. The two contexts are syntactically distinct (annotation vs. declaration), so using different separators is not confusing.

---

**Original problem and options (kept for context):**

**Location:** syntax.md §3.6 (Effect Groups), §7.1 (Effect List).

**Problem.** Two different operators mean "combine these effects": groups use `+`, function effect lists use `,`.

**Options:**
- (a) Use `+` everywhere.
- (b) Use `,` everywhere.
- (c) Leave as-is.
- (d) Use spaces in function signatures, keep `+` in group definitions. ← **chosen**

**Fix type:** `design-discussion`

### 6. ~~`collect_all` requires closure wrappers for every branch~~ ✅ RESOLVED

**Decision: Auto-thunk `collect_all` arguments. The compiler implicitly wraps each argument expression in a closure, so `||` is optional.**

`collect_all` is a compiler builtin with special parallel-execution semantics — the laziness is an implementation detail the compiler handles regardless. Removing the mandatory `||` aligns with Kāra's philosophy of "write sequential-looking code, get parallelism for free." Existing `||`-wrapped calls continue to work (non-breaking).

**Examples:**

```kara
// New (clean):
let (profile, orders, notifs) = collect_all(
    fetch_profile(user_id),
    fetch_orders(user_id),
    fetch_notifs(user_id),
);

// Old (still valid):
let (profile, orders, notifs) = collect_all(
    || fetch_profile(user_id),
    || fetch_orders(user_id),
    || fetch_notifs(user_id),
);
```

**Breaking change:** No — existing `||`-wrapped calls still work. The change makes `||` optional, not forbidden.

---

**Original problem and options (kept for context):**

**Location:** syntax.md §5.23 (Built-in Functions).

**Problem.** Every branch in `collect_all` must be wrapped in a zero-argument closure:

```kara
let (profile, orders, notifs) = collect_all(
    || fetch_profile(user_id),
    || fetch_orders(user_id),
    || fetch_notifs(user_id),
);
```

The `||` adds visual noise and conveys no information — every branch is always a zero-argument closure. Since `collect_all` is already a compiler builtin (not a library function), it has the freedom to desugar expressions directly.

**Options:**
- (a) Auto-thunk: the compiler wraps each argument expression in a closure implicitly. `collect_all(expr1, expr2, expr3)` desugars to `collect_all(|| expr1, || expr2, || expr3)`.
- (b) Use a block syntax instead: `collect_all { fetch_profile(user_id), fetch_orders(user_id), fetch_notifs(user_id) }`.
- (c) Leave as-is. The closure syntax makes it explicit that each branch is a separately scheduled unit of work, not an eagerly-evaluated argument. This is a meaningful semantic distinction.

**Recommendation:** (a), with a caveat. The closure syntax genuinely communicates "these are lazy — they run in parallel, not at the call site." However, since `collect_all` is a compiler builtin, the laziness is an implementation detail the compiler handles regardless. The programmer cares about *what* runs in parallel, not the thunking mechanism. If explicitness is valued, (c) is defensible; if readability is prioritized, (a) removes noise from every `collect_all` call.

**Examples after (a):**

```kara
let (profile, orders, notifs) = collect_all(
    fetch_profile(user_id),
    fetch_orders(user_id),
    fetch_notifs(user_id),
);
```

**Fix type:** `design-discussion`

**Breaking change:** No — existing `||`-wrapped calls would continue to work (a closure returning a value is still a value). The change makes `||` optional, not forbidden.

### 7. ~~Lock blocks require repeating the variable name~~ ✅ RESOLVED

**Decision: Add optional positional alias — `lock variable alias { alias.field }`. Grammar: `LOCK_EXPR = "lock" IDENT [ IDENT ] BLOCK`.**

When variable names are long, repeating them inside the lock block creates noise. The alias is optional — omitting it preserves today's behavior (non-breaking). The alias works naturally with nesting since each lock introduces its own named alias. This avoids the problems with `self` (ambiguous inside methods, broken in nested locks) and implicit receivers (ambiguous in nested locks).

**Examples:**

```kara
// Short name — alias optional, original syntax still works:
lock counter { counter.count += 1; }

// Long name — alias reduces noise:
lock user_notification_preferences prefs {
    prefs.email_enabled = true;
    prefs.push_enabled = false;
    print(prefs.summary());
}

// Nesting — each lock has its own alias:
lock account_ledger ledger {
    lock transaction_log log {
        ledger.balance += amount;
        log.append(entry);
    }
}

// Expression form:
let val = lock counter c { c.count }
```

**Breaking change:** No — alias is optional. Existing lock blocks without an alias are unchanged.

---

**Original problem and options (kept for context):**

**Location:** syntax.md §5.9 (Lock Blocks).

**Problem.** The locked variable must be named explicitly inside the block:

```kara
let val = lock counter { counter.count }

lock counter {
    counter.count += 1;
    print(counter.count);
}
```

The variable name `counter` appears 3 times in 2 lines. In longer lock blocks this gets worse. Since `lock` already introduces a scope tied to a specific variable, the variable is unambiguous inside the block.

**Options:**
- (a) Allow leading-dot syntax (implicit receiver) inside lock blocks: `lock counter { .count }`. The `.` refers to the locked variable. Used by Swift (`self.` elision), D, and Nim.
- (b) Allow `it` as an implicit binding: `lock counter { it.count }`. Matches item #4's use of `it`.
- (c) Leave as-is. The explicit variable name is clearer when lock blocks are long or nested, and avoids introducing a new scoping rule.
- (d) Positional alias: `lock variable alias { alias.field }`. ← **chosen**

**Recommendation:** (d). Optional alias that the programmer chooses explicitly — no magic binding, no ambiguity in nesting or methods, non-breaking.

**Fix type:** `design-discussion`

**Breaking change:** No — either option is additive (existing explicit syntax continues to work).

## Tier 3: Low Impact

### 8. ~~No raw string literals~~ ✅ RESOLVED — deferred

**Decision: Defer. Add `r"..."` when stdlib regex support lands. Purely additive, non-breaking. Track in future_decisions.md.**

---

**Original problem and options (kept for context):**

**Location:** syntax.md §1.3 (Literals).

**Problem.** The spec defines `"..."`, `"""..."""`, and `f"..."` but no raw strings. Regex patterns, Windows paths, and escape-heavy content need `\\` everywhere:

```kara
let pattern = "\\d{3}-\\d{4}";    // regex for ###-####
```

**Options:**
- (a) Add `r"..."` (Rust/Python convention): `let pattern = r"\d{3}-\d{4}";`.
- (b) Defer until stdlib has regex support and the need is concrete. ← **chosen**

**Recommendation:** (b). Non-breaking to add later. Worth tracking as a future decision.

**Fix type:** `deferred`

**Breaking change:** No — purely additive.

### 9. ~~`effect resource` two-keyword verbosity~~ ✅ RESOLVED — no change

**Decision: Keep `effect resource`. No change.**

The verbosity cost is low — resource declarations are written once per module, not per function. The two-keyword form is self-documenting and avoids `resource` colliding with user identifiers. Not worth a breaking change.

---

**Original problem and options (kept for context):**

**Location:** syntax.md §3.6 (Effect Declarations).

**Problem.** Every resource declaration starts with 16 characters before the name:

```kara
effect resource UserDB;
effect resource UserDB: DatabaseProvider;
```

`resource` alone would be cleaner but could conflict with user identifiers.

**Options:**
- (a) Shorten to `resource UserDB;`.
- (b) Leave as-is. ← **chosen**

**Recommendation:** (b). The verbosity cost is low (resource declarations are written once per module, not per function). The two-keyword form is self-documenting. Not worth a breaking change.

**Fix type:** `no-change`

### 10. ~~Interaction between items 1 and 5~~ ✅ RESOLVED

Items #1 and #5 were resolved together. The function signature grammar is now:

```
fn NAME [GENERICS] (PARAMS) [-> TYPE] [with EFFECTS] [requires EXPR] [ensures(NAME) EXPR] [where BOUNDS] BLOCK
```

Where `EFFECTS = EFFECT { EFFECT }` (space-separated) and `EFFECT = VERB "(" RESOURCES ")" | GROUP_NAME | "_"`.

**Full example combining items 1, 2, and 5:**

```kara
fn audit_transfer(
    from: ref Account,
    to: ref Account,
    amount: i64,
) -> Result[Receipt, TransferError]
    with reads(AccountDB) writes(AuditLog) sends(NotificationService)
    requires amount > 0 && from.balance >= amount
    ensures(receipt) receipt.amount == amount
    where Receipt: Serialize
{
    // ...
}
```

Compare with current syntax:

```kara
fn audit_transfer(
    from: ref Account,
    to: ref Account,
    amount: i64,
) -> Result[Receipt, TransferError]
    reads(AccountDB), writes(AuditLog), sends(NotificationService)
    requires amount > 0 && from.balance >= amount
    ensures |receipt| receipt.amount == amount
    where Receipt: Serialize
{
    // ...
}
```

Each clause starts with a keyword (`with`, `requires`, `ensures`, `where`), making the signature scannable.
