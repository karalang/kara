# Brainstorming v47 — Explicit `dyn` keyword vs compiler-inferred dynamic dispatch

**Status:** ✓ RESOLVED — 2026-04-23. Adopted **Approach A (explicit `dyn` keyword)** — `dyn Trait` is required in every trait-object type; a bare trait name in type position is a compile error.

## Disposition

| Axis | Landed |
|---|---|
| Syntax decision | A — explicit `dyn` required in type positions |
| Approach B (inferred) | Rejected |
| Approach C (hybrid: explicit for owned, inferred behind `ref`) | Rejected |
| `impl Trait` return types (open question 1) | Preserved as a future option — explicit `dyn` avoids the collision |
| `karac explain` dispatch-mode surfacing (open question 3) | Not needed — locality is preserved at the type itself |
| Object safety, cycle handling, effect semantics | Out of scope for v47 — already settled in prior docs |

## Why A won

1. **"Costs are legible" posture.** `Vec[dyn Animal]` makes the heap allocation visible at the type itself; `Vec[Animal]` hides it behind "reader must know Animal is a trait." Kāra's effect system, ownership modes, `shared`, and `allocates(Heap)` all push the same direction — hiding dynamic dispatch behind inference is the one place in the language that would reverse the posture.

2. **`impl Trait` future-proofing.** `impl Trait` return types are architecturally on the table (v47 open question 1). If they ever land, bare `Trait` in return position must mean `impl Trait`, not a trait object — Approach B would need to reverse itself. Committing to A now avoids that migration.

3. **Mechanical refactorability.** Renaming a trait, swapping trait-for-struct in a return type, or moving a trait between modules changes dispatch semantics silently under B and noisily (with a diagnostic) under A. Robustness outweighs three saved characters under `memory/feedback_design_weighting_llm_era.md`.

4. **LLM generation clarity.** Explicit keywords are unambiguous tokens — easier for a generator to emit correctly from prose, easier to diagnose when wrong. Approach B creates non-local coupling between a type annotation and the declaration of the trait it names; generators misread that.

5. **Spec already assumes A.** ~20 occurrences of `dyn Trait` across the effect system, cycle detection, GPU restrictions, ownership, and function-type variance sections. The decision consolidates what was already written; switching cost was a meaningful weight even before the design arguments did the work.

The one genuine argument for B — section C, "fewer keywords for first-time readers" — is the weakest axis under `feedback_design_weighting_llm_era.md`. `dyn` is teachable in ch08 alongside traits, which hadn't written the subsection anyway.

## Decision lens — what we're optimizing for

### 1. Locality of reasoning *(primary)*
Given a random function signature or type annotation in the middle of a file, can a reader (human or LLM) determine whether dispatch is static or dynamic *without cross-referencing another file*? This is the core question. Explicit `dyn` answers locally; inference answers "maybe — depends on how the trait is used elsewhere."

### 2. LLM-friendliness *(primary)*
- **Generation:** Can an LLM emit correct code from a prose description? Explicit syntax usually wins — keywords are unambiguous tokens.
- **Comprehension:** Can an LLM analyze existing code without needing whole-program context? Inference creates non-local coupling.
- **Error recovery:** When an LLM generates slightly-wrong code, is the diagnostic it gets back actionable? Explicit mismatches are easier to diagnose than "dispatch mode changed because of a distant call site."

### 3. Compiler-verifiable robustness *(primary)*
Does the syntax let the compiler catch more classes of error — e.g., "intended dynamic dispatch but got static," "accidentally monomorphized when vtable was expected"?

### 4. Human ergonomics *(secondary)*
Typing `dyn` is three characters plus a space. That's the entire cost side on the ergonomics axis. Readability is ergonomics too — often explicit wins even here.

### 5. Migration / precedent signal *(informational)*
Rust's 2015→2018 migration *added* the keyword after living with implicit. The rationale in RFC 2113 (clarity, static-vs-dynamic locality, `impl Trait` disambiguation) is public and applicable. Use as prior, not proof.

## Context — the two approaches

### Approach A: Explicit `dyn` keyword

```kara
let pets: Vec[dyn Animal] = [cat, dog]
fn render(shape: ref dyn Shape) -> String { ... }
let storage: Box[dyn Storage] = Box.new(FileStorage.new())
```

**Mechanical signal:** `dyn` in the type is the *only* way to get dynamic dispatch. Bare `Animal` in a type position is a type error (or a type alias / associated type, depending on context) — never silently a trait object.

### Approach B: Compiler-inferred

```kara
let pets: Vec[Animal] = [cat, dog]        // Animal is a trait — compiler infers dyn
fn render(shape: ref Shape) -> String { ... }   // if Shape is a trait, dyn
let storage: Box[Storage] = Box.new(FileStorage.new())
```

**Mechanical signal:** When a trait name appears in a type position where a concrete type would be expected, compiler synthesizes `dyn`. `karac explain` and diagnostics surface when this happens.

### Approach C (hybrid, for completeness)

Require `dyn` in owned positions (`Vec[dyn Animal]`, `Box[dyn Animal]` — heap-allocated, visible cost) but infer in `ref` positions (`ref Animal` becomes `ref dyn Animal` — the borrow makes it cheap, less worth typing).

Not recommended on face — adds a third rule — but worth naming so the option space is explicit.

## Discussion — TODO

### A. Against Approach B: the `impl Trait` collision
When Kāra gains `impl Trait` return types (deferred; not in v1 but architecturally on the table), bare `Trait` in return position already has to mean something else. Rust RFC 2113 called this out as the strongest argument. *(expand)*

### B. Against Approach B: hidden heap allocation
Owned `dyn Trait` costs a heap allocation (design.md:4244). Approach B makes that cost invisible at the type level — a reader has to know "Animal is a trait, not a struct" to know this is a heap allocation. Conflicts with Kāra's broader "costs are legible" posture (effects, ownership modes, `shared` keyword). *(expand)*

### C. For Approach B: fewer keywords for first-time readers
The user's own v44/v46 framing — struct/trait/impl is already a lot to load. Adding `dyn` to the introductory surface is another concept. *(expand — counter: can be taught in ch08 alongside traits, not ch02)*

### D. For Approach A: mechanical refactorability
Renaming a trait, moving it between modules, swapping trait-for-struct in a return type — all of these change semantics silently under Approach B and noisily (with a diagnostic) under Approach A. *(expand)*

### E. Interaction with the decision lens
- Locality: A wins decisively.
- LLM-friendliness: A wins on generation clarity; B wins on token count. Net: A, per `feedback_design_weighting_llm_era.md`.
- Compiler robustness: A wins — ambiguity is a rejectable error, not a silent default.
- Ergonomics: B wins marginally on three characters.
- Precedent: A, per Rust RFC 2113.

### F. Transition path
If we land on A (which the spec already assumes), the action is: canonicalize the decision in `design.md:7764`, remove the deferred entry, close out `implementation_checklist.md:7512` row, add a `ch08-traits-and-generics.md` subsection on `dyn Trait` citing the settled syntax. No code changes — parser has not yet landed `dyn` parsing (per `design.md:4315` Phase 8 note on effect checking; parser acceptance presumably tracks).

If we land on B, the action is: rewrite all ~20 `dyn Trait` occurrences in `design.md`, update `implementation_checklist.md:512` decision (1), specify the inference rule precisely (what counts as a "trait in type position"?), and spec the diagnostic that tells readers dispatch is dynamic.
