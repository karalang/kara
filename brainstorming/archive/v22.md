## Codegen memory layout

### 1. `Vec[T]` memory representation

**Question:** What are the concrete fields of a compiled `Vec[T]`? Options:

- **(a) `{ ptr data, i64 len, i64 capacity }`** — classic Rust/C++ layout. 24 bytes on 64-bit. Capacity stored in the stack/struct slot. Good locality for length check.
- **(b) `{ ptr data, i64 len }` with capacity in a header word at `data[-1]`** — 16 bytes on the stack. Capacity costs an extra indirection on growth but is never touched on read. Penalizes `push`, neutral on everything else.
- **(c) Small-vector optimization (SVO):** inline storage for N elements, heap-allocate only above that. Tempting but tangles with ownership (moving a Vec with inline storage is not a pointer copy).

Tradeoffs:
- Option (a) is the obvious default; matches everything programmers expect; simplest codegen.
- Option (b) saves 8 bytes in every Vec-valued local. Relevant if Vec is common as a return type or struct field. `push` gets a load to find capacity.
- Option (c) is a real perf win on small collections but breaks the "move = pointer copy" invariant the rest of the codegen relies on.

**✓ RESOLVED — (a).** `{ ptr, len, cap }` (24 bytes). No SVO in v1 — move stays pointer copy. Captured in implementation_checklist.md Phase 7.2.

---

### 2. `Vec[T]` growth strategy

**Question:** When `push` exceeds capacity, how does capacity grow?

- **(a) Doubling (`cap = max(1, cap * 2)`):** C++ `std.vector`, Rust `Vec` (historically 2x, now 2x with a minimum of 4 elements). Amortized O(1) push.
- **(b) Golden ratio (`cap = cap * 1.5`):** Facebook folly recommendation; allows more capacity reuse when allocator returns the old block's neighborhood.
- **(c) 2x with a minimum floor:** e.g., first grow goes from 0 to 4, then doubles. Avoids a flurry of tiny allocations for new Vecs.

**✓ RESOLVED — (c).** 2x doubling with a floor of 4 on first growth. Captured in implementation_checklist.md Phase 7.2.

---

### 3. `Array[T, N]` codegen representation

**Question:** How does `Array[T, N]` appear in LLVM IR?

- **(a) `[N x T]` directly.** No wrapper. Element-count is part of the LLVM type. Indexing is `getelementptr`.
- **(b) Struct wrapper `{ [N x T] storage }`** — adds one named field for future extensibility (bounds-check flags, invariants). Small overhead, flexibility in exchange.
- **(c) Heap-allocated `{ ptr data }`** for uniformity with `Vec`. Overkill for a fixed-size type — pays indirection for no benefit.

**✓ RESOLVED — (a).** LLVM `[N x T]` directly, no wrapper. Captured in implementation_checklist.md Phase 7.2.

---

### 4. `String` memory representation

**Question:** What's the layout for compiled `String`?

- **(a) `{ ptr data, i64 len, i64 capacity }`** — a `Vec[u8]` with a UTF-8 invariant on construction/mutation. Consistent with Vec.
- **(b) `{ ptr data, i64 len }` — immutable-buffer String.** Mutation always reallocates. Saves 8 bytes but makes `push_str` expensive.
- **(c) `{ ptr data, i64 len, i64 capacity, u8 is_heap }` — SSO-capable.** Small-string optimization; strings under (say) 23 bytes live inline. Worth considering for interned-heavy workloads; adds complexity.

Tradeoffs match the Vec question. `String` and `Vec` should probably use the *same* layout shape so code that handles one handles the other.

**✓ RESOLVED — (a).** `{ ptr, len, cap }` mirroring Vec; UTF-8 invariant enforced at construction and mutation. SSO deferred to Phase 8. Captured in implementation_checklist.md Phase 7.2.

## Layout codegen semantics

### 5. Which collection types does layout codegen cover in 7.2?

**Question:** Design.md § Feature 1 specifies `Vec[Entity]` in the example. Does 7.2 codegen ship with just Vec? Also Array[T, N]? Both?

- **(a) Vec only** — matches the design example. Requires Vec codegen to land first.
- **(b) Array[T, N] only** — simpler (fixed size, no growth). Enough to demonstrate SoA but forces users with dynamic collections to wait for Phase 8.
- **(c) Both** — Vec and Array[T, N]. More implementation work but gives users a real choice.

**✓ RESOLVED — (c).** Ship Array[T, N], Vec[T], and String in Phase 7.2. Implementation order: Array first (simplest), Vec second, String alongside Vec. Captured in implementation_checklist.md Phase 7.2.

---

### 6. SoA physical representation

**Question:** For `layout entities: Vec[Entity] { group physics { pos, vel } group combat { hp } }`, what's the physical shape?

- **(a) Split allocations:** one heap block per group. Outer Vec is `{ ptr physics, ptr combat, ptr metadata, i64 len, i64 capacity }`. Grows all groups together on `push`.
- **(b) Single allocation, interleaved group segments:** outer Vec is `{ ptr data, i64 len, i64 capacity }` and `data` points at `[group0_element_0 .. group0_element_cap | group1_element_0 .. group1_element_cap | ...]`. One allocation, one free — but grow requires moving every segment.
- **(c) Struct-of-pointers with separate capacities:** `{ ptr physics, i64 physics_cap, ptr combat, i64 combat_cap, ..., i64 len }`. All groups have the same `len` but potentially different allocated capacities (e.g., after a partial realloc failure — which we'd never want, so this buys nothing).

Tradeoff: (a) is simplest to reason about and matches GPU-buffer intuition (one buffer per group). (b) allocates less but complicates growth. (c) over-engineers.

**✓ RESOLVED — (a).** One heap allocation per group; outer Vec carries N group pointers + len + capacity. Captured in implementation_checklist.md Phase 7.2.

### 7. SoA field access translation

**Question:** How does `entities[i].position` compile under SoA layout?

- **(a) Compiler rewrites `entities[i].position` directly to `entities.physics[i].position`.** No intermediate "element view" materialized. Works for all plain field access. Breaks down when a program writes `let e = entities[i]` then uses `e.position` and `e.health` across group boundaries.
- **(b) `entities[i]` materializes a logical-view struct** gathering one element's worth of fields from every group. Used for `let e = entities[i]`. Field access on the view is simple AoS. Costs: one copy of every field on element load.
- **(c) Hybrid:** field access through `entities[i].field` short-circuits to the direct group load (option a). Binding `let e = entities[i]` materializes the full logical struct (option b) only when required.

**✓ RESOLVED — (c).** Hybrid: `entities[i].field` lowers to a direct group-indexed load; `let e = entities[i]` materializes an AoS copy. Purely a codegen optimization — no language-level signal. Captured in implementation_checklist.md Phase 7.2.

### 8. Reference semantics for SoA fields

**Question:** What references are allowed under SoA layout?

- `ref entities[i].position` — returns `ref Vec3`, pointing into the `physics` group's array at index `i`. **Straightforward.**
- `mut ref entities[i].position` — returns `mut ref Vec3` into the `physics` group. **Straightforward; single group write, standard aliasing rules.**
- `ref entities[i]` — returns what? A single logical `Entity` doesn't exist as a contiguous memory region under SoA. Options:
  - **(a) Compile error.** "Cannot take a reference to a single element of a SoA-laid-out collection; use field-level references or iterate." Clear but restrictive.
  - **(b) Fat reference** — a struct `{ ref physics_elem, ref combat_elem, ref metadata_elem }` synthesized on the fly. Callers see a `ref Entity` by static type but the fat ref is passed as three pointers. Works until the fat ref escapes into a context expecting a single pointer (FFI, generic code treating refs uniformly).
  - **(c) Materialize and borrow temporary** — synthesize a stack-allocated AoS `Entity`, copy the three group slots in, borrow that. Subsequent mutations through the ref don't write back. Semantic footgun.

**✓ RESOLVED — (a).** Compile error on `ref entities[i]` (whole element). Field-level refs allowed. Same rule for `ref enum_value` under `split_by_variant`. User-visible rule captured in design.md § Layout Rules under "References into SoA-laid-out collections".

---

### 9. Mutation and aliasing under SoA

**Question:** Standard borrow-check forbids `mut ref a` and `ref b` overlap when they alias. Under SoA, `mut ref entities[i].position` and `ref entities[j].health` never alias (different groups, different allocations) even when `i == j`. Does the borrow checker know this?

Implications:
- **If yes:** the hot path `for e in entities { e.position += e.velocity; read_health(e.health); }` runs without any aliasing objection even though both refs come from the same index. This is a key SoA performance win.
- **If no:** users write awkward code to route around a conservative checker, or the checker uses whole-collection pessimism.

**✓ RESOLVED.** Borrow checker treats borrows into distinct groups as disjoint — idiomatic `for e in entities { e.position += e.velocity; read_health(e.health) }` compiles clean. User-visible rule captured in design.md § Layout Rules under "Cross-group disjointness". Implementation extends the existing ownership pass (`src/ownership.rs`).

### 10. `layout` interaction with owned fields (drop on remove)

**Question:** If `Entity.name: String` (owned heap value) and the Vec removes an element, what owns the `name` buffer?

- Under AoS this is automatic — moving/dropping an `Entity` moves/drops `name` with it.
- Under SoA, `name` lives in its own group array. Removing element `i` requires explicitly dropping the `String` at `metadata[i].name`.

This is a codegen concern (emit the right destructor sequence on `pop` / `remove` / `drop_in_place`), not a user-visible semantic change. Document that the compiler handles this correctly.

**✓ RESOLVED.** Captured in implementation_checklist.md Phase 7.2 under "SoA drop semantics": on `pop`/`remove`/drop, emit destructor calls for every owned field in every group at the removed index. No user-facing surprise.

### 11. `layout` and generics

**Question:** Design.md § Feature 1 says "Monomorphized per layout variant. Layout is part of the concrete type at codegen level." How does the monomorphization key change?

Scenarios:
- `fn sum_health(entities: ref Vec[Entity]) -> f32` — if two call sites pass Vecs with different layouts (one AoS, one SoA `physics/combat/metadata`), are they two monomorphs?
- A library that declares `struct Entity` — the library has no control over whether downstream users apply a layout. The library's `fn on_entity(e: ref Entity)` must work on any layout.

Options:
- **(a) Layout is part of the concrete type.** `Vec[Entity]@AoS` ≠ `Vec[Entity]@SoA<physics/combat/metadata>`. Functions are monomorphized per layout. Libraries that accept generic collections get per-instantiation code.
- **(b) Layout affects codegen of field access only.** The type `Vec[Entity]` is one type regardless of layout; but inside a function body that accesses `e.position`, the compiler uses the layout information of the *actual* collection passed in. Requires passing a layout witness at runtime, or monomorphizing anyway.
- **(c) Layouts are project-internal and only affect code in the same project.** A library gets AoS; the project using the library can re-declare with its own layout. No cross-crate generic issue.

**✓ RESOLVED — (c) at the library boundary + (a) within a project.** Layout binds to the binding site, not to the type globally. Library code always sees the AoS layout of types it declared; projects that apply a layout create project-local monomorphs. User-visible rule captured in design.md § Layout Rules under "Layout binds to the binding site".

### 12. Unassigned-field group: physical placement

Design.md § Layout Rules: "Fields not listed in any group are collected into an implicit trailing AoS group, stored after all explicitly named groups. The compiler emits a warning once per layout block listing the unassigned fields — suppress with `#[allow(layout_unassigned_fields)]`."

**Open question:** Is the implicit trailing group another group (option 6a — its own heap allocation) or does it get special treatment?

**Suggest:** treat it as just another group. Same heap allocation, same indexing. No special case in the codegen path. The warning belongs to validation (design already specifies it); the runtime shape is identical to explicit groups.

**✓ RESOLVED.** No special case — implicit trailing group is an ordinary group. Captured in design.md § Layout Rules (existing "Unassigned fields" bullet extended).

## Validation layer

### 13. Validation pass location

**Question:** Where does "layout field is real and unique across groups" validation run?

- **(a) Resolver (`src/resolver.rs`).** Layout blocks live among top-level items; resolver already walks items; adding a pass that links `LayoutDef.collection_type` → `StructDef` and checks field lists is a natural fit.
- **(b) TypeChecker (`src/typechecker.rs`).** If the validation needs to know the concrete monomorphized type of the collection (e.g., "is `T` in `Vec[T]` really the struct we expect?"), type checker has the right info.

Since field names resolve before generic instantiation and layouts target concrete structs (not generic params), (a) is the right home.

**Suggest:** resolver pass. Fits the existing phase boundary.

**✓ RESOLVED.** Layout validation lives in `src/resolver.rs`. Captured in implementation_checklist.md Phase 7.2 under "Layout validation pass (resolver)".

## Smaller implementation concerns

### 14. Format specifiers in `f"..."` on compiled Strings

Currently f-strings go through `Display` via the interpreter's runtime formatter. Compiled String codegen will need a runtime formatter too (itoa, dtoa, padding, width). Scope question for 7.2: how much of this lands alongside String codegen?

**Suggest:** minimum formatter in 7.2 — integer and float to-string, raw concatenation for `f"…{s}…"` with `s: String`. Format specifiers (`{x:.2}`, width, alignment) deferred to Phase 8.

**✓ RESOLVED.** Int/float to-string + raw concatenation in 7.2; format specifiers (`{x:.2}`, width, alignment) deferred to Phase 8. Captured in implementation_checklist.md Phase 7.2 under "Minimum f-string formatter".

---

### 15. Drop-on-scope-exit for compiled collections

Existing codegen has `scope_rc_vars` for `shared struct` cleanup. Owned `Vec`/`String` needs a similar mechanism — free the heap buffer at scope end. Do we extend `scope_rc_vars` (rename it or generalize), or add a separate `scope_owned_heap_vars`?

**Suggest:** generalize to `scope_cleanup_actions` with a per-entry kind (RC dec, free heap buffer, free SoA groups). One mechanism, uniform treatment.

**✓ RESOLVED.** Generalize `scope_rc_vars` → `scope_cleanup_actions` with per-entry kinds (RC dec / free heap buffer / free SoA groups). Captured in implementation_checklist.md Phase 7.2.

---

### 16. Interpreter parity for compiled-type semantics

When Vec/String/Array become compiled types, should the tree-walk interpreter also switch to the same runtime representation, or stay with its `Value.Vec(Vec<Value>)` model?

- **(a) Interpreter stays on `Value`-based representation.** Codegen runs against compiled types; interpreter stays simple. Semantics are tested twice (interpreter + compiled E2E).
- **(b) Interpreter moves to the compiled representation.** Single source of truth. Significant interpreter rewrite.

**Suggest:** (a). Interpreter is for validating language semantics, not runtime representation. Compiled binaries test the representation; interpreter tests the behavior. Both should produce equivalent observable output.

**✓ RESOLVED.** Interpreter keeps its `Value`-based representation; compiled binaries test layout, interpreter tests behavior. Both must produce equivalent observable output. Captured in implementation_checklist.md Phase 7.2 under "Interpreter parity scope".
