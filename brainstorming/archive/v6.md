# Brainstorming v8 — Low-Level and Hardware-Level Development

A focused gap analysis of Kāra's spec against the requirements of low-level
systems programming: OS kernels, embedded microcontrollers, device drivers,
bare-metal firmware, and hardware-adjacent compute (SIMD, DMA, interrupt handling).

Kāra already has meaningful low-level foundations — the `kernel` and `embedded`
profiles, `unsafe` blocks, FFI with C ABI, `#[gpu]` GPU subsets, and layout
blocks for memory organization. This document looks at what is still missing
or underspecified when those foundations hit real hardware.

Reference languages surveyed: **Rust (embedded), Zig, C, C++, Ada/SPARK.**

---

## Priority 1 — Critical Gaps (required for any real low-level use)

### 1.1 Volatile Memory Access / Memory-Mapped I/O ✓ Resolved

**Source:** All embedded languages, C `volatile`, Zig `@volatileLoad`/`@volatileStore`.

The most fundamental requirement of embedded and kernel programming:
memory-mapped hardware registers must be read and written with **volatile
semantics**. Without it, the compiler may reorder, coalesce, or eliminate
reads/writes to I/O registers because they look like dead stores to dead loads
on normal memory.

The current spec has:
- Layout blocks — for data-oriented performance layout (SoA). Not related.
- `unsafe` blocks — allows raw pointer arithmetic, but no volatile guarantee.
- FFI `extern "C"` — can call C `volatile` functions, but no Kāra primitive.

No volatile primitive exists. There is no way to write a memory-mapped UART
driver, GPIO toggle, or hardware timer read in safe or unsafe Kāra today.

```kara
// Desired: explicit volatile access for hardware registers
unsafe {
    let uart_dr = 0x4000_4400 as *mut u32;        // UART data register address
    volatile_write(uart_dr, b'A' as u32);          // must not be optimized away
    let status = volatile_read(uart_dr as *const u32);  // must not be cached
}
```

**Design recommendation:** Two intrinsics in the `unsafe` surface:

```kara
unsafe fn volatile_read[T: GpuSafe](ptr: *const T) -> T
unsafe fn volatile_write[T: GpuSafe](ptr: *mut T, val: T)
```

Additionally: a `VolatileCell[T]` wrapper type for safe-ish MMIO register access,
similar to Rust's `vcell.VolatileCell`. This lets hardware register maps be
expressed as structs with fields that always read/write with volatile semantics —
no raw pointer arithmetic at every use site.

```kara
struct Uart {
    data:    VolatileCell[u32],   // 0x000 — transmit/receive data
    status:  VolatileCell[u32],   // 0x004 — status flags
    control: VolatileCell[u32],   // 0x008 — configuration
}

// Then:
uart.data.write(b'A' as u32);    // volatile, no unsafe at call site
let s = uart.status.read();      // volatile, no unsafe at call site
```

**Effect integration:** `volatile_read`/`volatile_write` imply `reads(Hardware)`
and `writes(Hardware)` respectively. `Hardware` is a new built-in primitive
resource, parallel to `FileSystem`, `Stdio`, and `Network`. This makes MMIO
access visible in the effect system — kernel functions that touch hardware
registers advertise it in their signature.

**Questions to resolve:**
- Is `VolatileCell[T]` a language primitive or a stdlib type built on `unsafe`?
- Should `Hardware` be a single undifferentiated resource, or parameterized
  (`Hardware[uart0]`) for finer effect tracking in drivers?
- `T` in `VolatileCell[T]` — should it be restricted to integer/primitive types?

### 1.2 `#[repr]` — ABI and Physical Layout Control ✓ Resolved

**Source:** Rust (`#[repr(C)]`, `#[repr(packed)]`, `#[repr(align(N))]`), C structs, Ada record representation clauses.

The spec has **layout blocks** for data-oriented SoA/AoS transformations for
performance. That is a different problem from ABI layout control. Hardware
programming requires exact, predictable memory layout for:

- **Protocol wire formats** (USB descriptors, Ethernet frames, register maps)
- **C ABI compatibility** (sharing struct types with C libraries or the OS kernel)
- **DMA buffers** (hardware DMA engines require specific alignment)
- **Interrupt vector tables** (exact memory layout mandated by CPU architecture)

Without `#[repr]`, the compiler may add padding, reorder fields, or use an
internal layout that breaks any of the above.

```kara
#[repr(C)]               // same layout as equivalent C struct — no padding surprises
struct UsbDescriptor {
    length:       u8,
    descriptor_type: u8,
    bcd_usb:      u16,
    device_class: u8,
    // ...
}

#[repr(C, packed)]       // no padding at all — every byte counts (protocol framing)
struct EthernetFrame {
    dst_mac: Array[u8, 6],
    src_mac: Array[u8, 6],
    ether_type: u16,
}

#[repr(align(4096))]     // page-aligned — required for DMA and MMU page tables
struct PageTableEntry {
    entries: Array[u64, 512],
}
```

**Why layout blocks don't cover this:**
Layout blocks rearrange *which* fields are stored together (SoA transformation)
and hint at cache-line groupings. They don't commit to C ABI layout, guarantee
no padding, or set alignment for external requirements. The two features solve
orthogonal problems and should coexist.

**Interaction with `unsafe`:** `#[repr(packed)]` makes taking a reference to a
misaligned field undefined behavior — accessing packed fields by reference should
require an `unsafe` block. `#[repr(C)]` is safe.

**Questions to resolve:**
- Does `#[repr(C)]` disable layout block annotations on the same struct, or are
  they orthogonal?
- Is `#[repr(u8)]` (enum discriminant representation) in scope? Needed for
  `enum` types used in C FFI.
- Should alignment be declarable per-field (`#[align(8)] field: u32`) in addition
  to per-struct? C11 `_Alignas` supports this.

### 1.3 Inline Assembly — Full Specification ✓ Resolved

**Source:** Rust (`core.arch.asm!`), Zig (`asm` builtin), GCC/Clang extended `asm`.

The spec mentions inline assembly in the `unsafe` section with one line:
_"unsafe blocks for raw pointer arithmetic, unchecked casts, bypassing ownership,
inline assembly"_ — no syntax, no operand model, no clobber list, nothing.

Inline assembly is essential for:
- Privileged instructions (CPU mode switches, MMU control, cache flush)
- Architecture-specific instructions not expressible in IR (e.g., ARM `wfi`,
  x86 `hlt`, RISC-V `fence`)
- Hand-tuned inner loops where LLVM auto-vectorization falls short
- Reading CPU special registers (CPSR, control registers, cycle counters)

```kara
// Proposed syntax (Rust-inspired, since LLVM is the backend):
unsafe {
    // Disable interrupts on ARM Cortex-M (CPSID i)
    asm!("cpsid i", options(nomem, nostack, preserves_flags));

    // Read the cycle counter on ARM
    let cycles: u32;
    asm!(
        "mrs {}, pmccntr_el0",
        out(reg) cycles,
        options(nomem, nostack),
    );

    // x86: CPUID
    let eax: u32;
    let ebx: u32;
    asm!(
        "cpuid",
        inout("eax") 0u32 => eax,
        out("ebx") ebx,
        options(nostack, preserves_flags),
    );
}
```

The LLVM inline asm model (which Rust also uses) is the natural fit given Kāra's
LLVM backend: operands are bound with `in(constraint)`, `out(constraint)`,
`inout(constraint)`, and options declare side effects to the compiler
(`nomem`, `nostack`, `volatile`, `preserves_flags`, `att_syntax`).

**Effect integration:** Any `asm!` block with `options(volatile)` or without
`nomem`/`nostack` implies `reads(Hardware)` or `writes(Hardware)`. The compiler
should conservatively assume hardware effects on all `asm!` unless the options
explicitly prove otherwise.

**Questions to resolve:**
- Syntax: Rust's `asm!` macro-like syntax, or Zig's `asm` expression form?
- How do inline asm operands interact with ownership? Can you pass an owned value
  as an `out` operand?
- Should there be a `global_asm!` equivalent for file-scope assembly (needed for
  interrupt vector tables and bootstrap code)?

### 1.4 Atomic Operations and Memory Ordering ✓ Resolved

**Source:** Rust (`std.sync.atomic`, `std.sync.atomic.Ordering`), C11 `_Atomic`, Zig `@atomicLoad`/`@atomicStore`.

The spec lists "Channels / atomics" as a single deferred item. For high-level
concurrency, atomics and channels go together. For low-level work, atomics are
needed even in single-threaded interrupt-driven programs where a channel
abstraction would be overkill or unavailable.

The `embedded` profile forbids concurrency but still needs atomics:
- Sharing data between interrupt handlers and main code (ISR + main thread)
- Spinlocks, RCU, lock-free queues in kernel code
- Memory barriers for DMA completion signaling
- Enforcing ordering around MMIO (ensure memory writes reach the bus before the DMA start register is written)

Missing:
- `Atomic[T]` type (load, store, compare-exchange, fetch-add, etc.)
- Memory ordering enum: `Relaxed`, `Acquire`, `Release`, `AcqRel`, `SeqCst`
- Standalone fence: `fence(Ordering)` / `compiler_fence(Ordering)`

```kara
// Inter-ISR flag:
let flag: Atomic[bool] = Atomic.new(false);

// In interrupt handler:
flag.store(true, Release);

// In main loop:
while !flag.load(Acquire) { /* spin */ }
// Memory ordering guarantees the flag write is visible before we proceed

// Memory fence for DMA:
unsafe {
    dma_buffer[0] = payload;
    fence(Release);           // all writes complete before DMA sees the buffer
    dma_start_reg.write(1);   // trigger DMA
}
```

**Effect model:** `Atomic[T]` operations on a shared atomic don't fit neatly into
`reads`/`writes` with ownership. Options:
1. Atomic operations on owned `Atomic[T]` — callers with a shared reference can
   still call `load`/`store` (as with Rust's `&AtomicBool`).
2. Atomic types are implicitly `shared struct` — RC semantics, accessible through
   shared references.
3. A new effect verb `synchronizes(X)` for atomic operations (very heavy).

Recommendation: `Atomic[T]` is a `shared struct` (reference semantics). Effect
tracking is coarse: `reads(X)` / `writes(X)` on the resource that the atomic
guards, exactly as it would be for a mutex. The atomicity is a runtime guarantee,
not an effect-system guarantee.

**Questions to resolve:**
- Should `fence` / `compiler_fence` be in `unsafe` or safe?
- How do atomics interact with the `embedded` profile that bans concurrency?
  (ISR sharing is single-core but multi-context — arguably not "concurrency"
  in the task-scheduler sense.)
- Is `compare_exchange` expressible given Kāra's ownership model?

## Priority 2 — Important for Embedded/Kernel Work

### 2.1 `no_std` Boundary Definition ✓ Resolved

**Source:** Rust's `core` / `alloc` / `std` split, Zig's `std` vs freestanding.

The spec mentions `no_std = true` in the `kernel` and `embedded` profiles but
**never defines what `std` is or what remains available without it**.

For embedded, this boundary is the most operationally critical design decision
after the memory model. Programmers need to know precisely:

| Category | `no_std` available? | Notes |
|---|---|---|
| Primitive types (`i64`, `f64`, `bool`, `char`) | Yes | Must be — they're language-level |
| `Array[T, N]` | Yes | Stack-allocated, no heap required |
| `Vec[T]`, `HashMap[K, V]`, `String` | No | Require heap allocation |
| `Option[T]`, `Result[T, E]` | Yes | Pure algebraic types |
| Formatting / `f"..."` interpolation | Partial — no heap forms | `f"..."` strings require allocation in general |
| Math (`sin`, `cos`, `sqrt`) | Yes — libm | Available as `core.math` or equivalent |
| `panic` handler | Required — user-supplied | `no_std` projects must define a panic handler |
| Effect system | Yes — pure compile-time | Effects are a type-system feature |
| `Pool[T]`, `Handle[T]` | Depends — arena-backed | Fixed-size pool can work without heap |

**Recommendation:** Follow Rust's three-layer model:

- **`core`** — language fundamentals: primitives, `Option`, `Result`, `Array`,
  refinement types, traits, effects. Zero dependencies. Always available.
- **`alloc`** — heap-requiring types: `Vec`, `HashMap`, `String`, `shared struct`.
  Available in `no_std` projects that provide an allocator.
- **`std`** — OS-dependent: file I/O, networking, threads, environment. Not
  available in `no_std`.

The `kernel` profile implies `no_std = true` and `no_alloc = true`. The
`embedded` profile implies `no_std = true`; `no_alloc` is opt-in (some embedded
systems have heap allocators).

**Questions to resolve:**
- What is the name of the Kāra equivalent of `core`? Just `core`?
- Can `shared struct` exist in `alloc` (heap-backed RC)? Or only in `std`?
- Does `f"..."` string interpolation require `alloc`? If so, what's the `no_std`
  alternative for formatting to fixed-size buffers?

### 2.2 Interrupt Handler ABI and `#[interrupt]` ✓ Resolved

**Source:** Rust's `cortex-m-rt` crate, AVR-hal, RTIC framework; Zig's interrupt
vectors; Ada's interrupt aspect.

Interrupt service routines are not normal functions:
- They are called by hardware, not software
- They have a different calling convention (all registers saved, `iret`/`eret` return)
- They cannot return values
- They must be registered in a specific location (the interrupt vector table)
- They interact with volatile shared state (see §1.1)

Without language-level support, embedding ISRs requires either unsafe linker
hacks or FFI shims — the norm in Rust's embedded ecosystem today.

```kara
// Proposed: #[interrupt] attribute
#[interrupt(TIMER1)]    // name matches the MCU's interrupt vector
fn timer1_isr() {
    // Runs with hardware interrupt context:
    // - all registers saved/restored by compiler
    // - returns via interrupt return instruction
    // - implicitly reads(Hardware), writes(Hardware)
    TIMER1_FLAG.store(true, Release);  // see §1.4
}

// Critical section — temporarily disable interrupts
fn update_shared_state() writes(Hardware) {
    let _guard = critical_section.acquire();  // disables interrupts
    // ...
    // _guard drop re-enables interrupts via RAII
}
```

**Interaction with profiles:** The `isr` custom profile example in the spec is
exactly this use case. The `#[interrupt]` attribute should:
- Force the function to use the platform's interrupt calling convention
- Forbid it from being called directly from normal code (compile error)
- Automatically emit a placement directive for the vector table entry
- Apply `isr` profile restrictions (no heap, no panics) automatically

**Questions to resolve:**
- Is the interrupt name a string, identifier, or platform-defined enum value?
- How does the MCU's interrupt controller model map to the vector table? Is this
  target-specific and expressed in a platform crate, or built into the language?
- Critical sections: a `CriticalSection` token type (capability pattern) vs
  an explicit `disable_interrupts()` / `enable_interrupts()` pair?

### 2.3 Linker Control Attributes ✓ Resolved

**Source:** Rust's `#[link_section]`, `#[no_mangle]`, `#[used]`; GCC `__attribute__((section(...)))`.

Embedded and kernel code needs to place specific data and functions at exact
memory addresses, which requires controlling the linker's view of symbols:

```kara
// Place the interrupt vector table at a fixed address (required by ARM Cortex-M)
#[link_section(".vectors")]
#[used]                              // don't dead-strip even if not called by Kāra
let INTERRUPT_VECTORS: Array[*const fn(), 256] = [...];

// Export a symbol with a stable name for the C linker / OS loader
#[no_mangle]
pub fn kernel_main() { ... }

// DMA buffer in a specific memory region (fast SRAM on STM32)
#[link_section(".dtcmram")]
#[repr(align(32))]                   // cache-line aligned for DMA
let mut dma_rx_buffer: Array[u8, 512] = [0; 512];
```

Three attributes needed:

| Attribute | Purpose |
|---|---|
| `#[link_section("name")]` | Place symbol in named linker section |
| `#[no_mangle]` | Use the Kāra identifier as-is for the symbol name (no mangling) |
| `#[used]` | Prevent dead-code elimination even if the symbol has no Kāra callers |

These are already in the `unsafe` ecosystem by convention (they don't bypass
the type system but they have linker-level consequences). They are low-risk to
add — pure annotations with no semantic weight in the effect system.

**Questions to resolve:**
- Does `#[no_mangle]` also imply `extern "C"` ABI, or just name stability?
- Does Kāra's module system mangle names by default? If not, `#[no_mangle]` is
  a no-op and the real issue is ABI, not name mangling.
- `#[used]` and the auto-concurrency dead-code pass — does the compiler dead-strip
  unreferenced globals? If so, `#[used]` is needed from day one.

### 2.4 C Calling Convention Variants ✓ Resolved

**Source:** Rust's `extern "C"`, `extern "C-unwind"`, `extern "stdcall"`, `extern "fastcall"`, `extern "interrupt"`.

The spec's FFI section only shows `extern "C"`. Real systems work requires more:

| Convention | Used for |
|---|---|
| `extern "C"` | Standard C ABI (already present) |
| `extern "C-unwind"` | C functions that may propagate exceptions (e.g., C++ interop) |
| `extern "stdcall"` | Win32 API (Windows kernel and driver development) |
| `extern "fastcall"` | Some x86 calling conventions |
| `extern "win64"` | Windows x64 ABI (differs from System V AMD64 ABI on Linux) |
| `extern "sysv64"` | System V AMD64 ABI explicitly named |
| `extern "interrupt"` | ISR entry points on some architectures |

For the first release, only `extern "C"` and `extern "C-unwind"` are urgent.
The others can be deferred, but the syntax must allow for them — `extern "<abi>"`
as a general form is the right design, not hardcoding only `"C"`.

**Questions to resolve:**
- Kāra's effect system ensures no panics propagate across `extern "C"` — what
  happens with `extern "C-unwind"`? Can C++ exceptions propagate through Kāra?
- Should `extern "interrupt"` be the implementation of `#[interrupt]` (§2.2)?

## Priority 3 — Track With Existing Deferred Items

### 3.1 SIMD / Explicit Vectorization ✓ Resolved

Already captured in `v5.md §3.3` as "track alongside GPU story."
From the hardware perspective: SIMD is needed on CPU too, not just GPU.

Current spec: LLVM auto-vectorization for CPU. This is often sufficient, but:
- Auto-vectorization is not guaranteed
- It cannot use architecture-specific instructions (`AVX-512`, ARM SVE)
- Performance-critical DSP, signal processing, and codecs need explicit SIMD
- The `#[gpu]` `Vector[T, N]` primitive (v5 §3.3 candidate) would satisfy CPU
  SIMD as well if designed with both targets in mind

**Interaction with `#[repr]`:** SIMD vector types require specific alignment
(`__m256` needs 32-byte alignment). `#[repr(align(32))]` from §1.2 is a prerequisite.

Track: defer SIMD alongside GPU backend, but note that `#[repr(align(N))]`
(§1.2) is a prerequisite that should land first.

### 3.2 Bitfield Types for Hardware Registers ✓ Resolved

**Source:** C bitfields, Zig packed structs, Ada record representation clauses.

Hardware registers are almost universally described as bitfields:

```kara
// ARM UART status register: specific bits have specific meanings
#[repr(C)]
struct UartStatus {
    // bits [31:8] — reserved
    // bit [7]    — tx_fifo_empty
    // bit [6]    — rx_fifo_full
    // bit [5:0]  — error flags
    raw: u32,   // current workaround: access via masks and shifts
}

// Desired — but Kāra has no bitfield syntax:
#[repr(C, bitfield)]
struct UartStatus {
    error_flags:    u6,     // bits [5:0]
    rx_fifo_full:   u1,     // bit [6]
    tx_fifo_empty:  u1,     // bit [7]
    _reserved:      u24,    // bits [31:8]
}
```

The workaround (raw integer + manual bit masks) is common but error-prone.
Zig's packed structs handle this more cleanly. Ada's representation clauses are
the gold standard.

**Connection to `#[repr]`:** Bitfield support requires `#[repr(packed)]` and
`#[repr(C)]` (§1.2) to be in place first.

Recommendation: defer until `#[repr]` lands. Track as a follow-on.

### 3.3 Static Stack Depth Analysis ✓ Resolved

**Source:** Ada/SPARK's stack usage analysis, MISRA C, embedded-hal guidelines.

Microcontrollers have 2KB–256KB of stack. Stack overflows are silent (no
guard pages, no MMU) and often cause difficult-to-debug corruption. Static
stack usage analysis — determining the maximum stack depth of a call tree at
compile time — is a standard practice in safety-critical embedded work.

```kara
// Proposed: optional annotation + compiler analysis
#[max_stack(512)]    // compile error if this function's call tree uses > 512 bytes
fn handle_interrupt() { ... }

// Or: query mode
karac query stack src/isr.kara.handle_interrupt
// → { "max_stack_bytes": 312, "chain": [...] }
```

Recursion makes this undecidable, but:
- The `embedded` profile already forbids (and the GPU subset already forbids) recursion
- Without recursion, the call graph is a DAG — stack depth is statically computable
- The `#[tailrec]` annotation from v5 §3.1 is orthogonal (TCO for CPU) but the
  no-recursion requirement for stack analysis aligns with `embedded`

**Questions to resolve:**
- Is stack analysis opt-in (`#[max_stack(N)]`) or always computed for `embedded`
  profile builds?
- Should it appear in `karac build --output=json` as a standard field, or only
  on explicit query?

## What's Already Solid

These low-level features are **already in the spec** and well-designed:

- **`kernel` / `embedded` profiles** — project-wide effect restrictions enforced
  at compile time. The right model (vs Rust's per-item `#[no_std]`).
- **`unsafe` blocks** — raw pointer arithmetic and unchecked casts are the
  correct escape hatch. The decision to keep effects active inside `unsafe` is
  correct — effects on the enclosing function remain enforced.
- **`wrapping_add` / explicit wrapping arithmetic** — overflow traps by default
  with explicit opt-out. Correct for both safety and hardware (CRC, checksum,
  ring buffer index arithmetic all need wrapping).
- **`usize` for FFI only** — using `i64` for all indices and `usize` only for
  `size_t` interop is clean. No constant casting noise in application code.
- **`#[gpu]` with `GpuSafe` trait and explicit `gpu.dispatch`** — the model
  (programmer controls dispatch site, compiler validates constraint propagation)
  is exactly right and generalizes to future hardware targets (FPGA, DSP).
- **Layout blocks** — SoA/AoS transformation and field grouping for cache
  locality. Unique to Kāra; directly benefits game engines and simulation.
- **`Pool[T]` / `Handle[T]`** — generational arenas as a first-class stdlib
  type. Correct for OS-style descriptors, ECS, and compiler IR nodes.
- **`extern "C"` with mandatory effect annotations** — correct. FFI calls making
  their effects visible to the type system is the right call.
