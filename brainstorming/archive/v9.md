# Brainstorming v9 ŌĆö Design Review Findings

Issues identified during a full review of design.md. Split into two categories: spec gaps that could force design changes if left unresolved, and lower-risk items that are cheap to address or can be deferred safely.

## Category A: Worth Discussing (Potential Spec Gaps)

### 1. `shared struct` synchronization at the `Arc` boundary ŌĆö RESOLVED

**Context:** Brainstorming v8 resolved that `mut` field mutation uses RefCell semantics (runtime borrow checking). That decision covers single-threaded RC. But when the compiler promotes `Rc` to `Arc` (value's live range overlaps a parallel region), RefCell is no longer sufficient ŌĆö `RefCell` is not thread-safe.

**Problem:** The spec says the compiler chooses `Arc` when a `shared struct` value crosses task boundaries. But `RefCell` panics are single-threaded runtime checks. Under `Arc`, two tasks mutating the same `mut` field concurrently would race on the RefCell borrow flag itself ŌĆö undefined behavior, not a clean panic.

**What needs deciding:**

- Does `Arc` promotion also promote `RefCell` ŌåÆ `RwLock` (or `Mutex`) per `mut` field? If so, the performance characteristics change significantly ŌĆö lock contention, potential deadlocks, and the `panics` effect now includes lock poisoning.
- Or does the effect system's `writes` conflict detection guarantee that two tasks never concurrently mutate the same `shared struct` instance? If so, this guarantee needs to be stated explicitly and the mechanism described ŌĆö effect conflicts are per-resource, not per-instance.
- Or is `shared struct` simply not safe to share across tasks without explicit synchronization (user wraps in `Mutex`)? If so, this contradicts the "compiler handles Arc wrapping" promise.

**Options:**

- **(a) `RefCell` ŌåÆ `RwLock` auto-promotion.** When the compiler chooses `Arc`, each `mut` field's borrow flag becomes an `RwLock`. Reads take a shared lock; writes take an exclusive lock. Thread-safe, but adds lock overhead and deadlock risk. Poisoned locks propagate `panics`.
  - Pro: Fully automatic. No user burden.
  - Con: Hidden locks. Performance cliff when crossing the `Rc`/`Arc` boundary. Deadlock risk if a method locks field A then field B while another locks B then A.

- **(b) Per-instance effect tracking prevents concurrent mutation.** The effect system guarantees that two tasks never hold mutable access to the same `shared struct` instance simultaneously. `RefCell` is sufficient because concurrency is serialized at a higher level.
  - Pro: No lock overhead. RefCell is always enough.
  - Con: Effect conflicts are per-resource-type, not per-instance (brainstorming v8 item 2 resolved this as "untracked"). This option contradicts that resolution unless a new mechanism is introduced.

- **(c) `shared struct` is `Send` but not `Sync` by default.** A `shared struct` can be sent to another task (ownership transfer via `Arc`), but concurrent access from multiple tasks requires explicit `Mutex` wrapping by the user. The compiler rejects code that would allow concurrent `mut` field access across tasks.
  - Pro: Explicit. No hidden synchronization. Users opt into the complexity they need.
  - Con: Undermines the "compiler decides how" philosophy. Graph structures shared across tasks become verbose.

- **(d) Field-granularity locks, compiler-inserted.** Each `mut` field gets its own lock (lightweight spinlock or similar). The compiler inserts lock/unlock around every field access. No deadlock risk because locks are never held across user code ŌĆö they protect only the single field read/write.
  - Pro: Fine-grained. No deadlock. Automatic.
  - Con: Spinlocks under contention burn CPU. Every field access pays lock cost even in single-threaded code that happens to use `Arc`.

**Decision:** RESOLVED ŌåÆ **Option (c) with `lock` block syntax.**

`shared struct` is `Send` but not `Sync` by default. A `shared struct` can be sent to another task (ownership transfer via `Arc`), but concurrent access from multiple tasks requires explicit `Mutex[T]` wrapping. The compiler rejects code that would allow concurrent `mut` field access across tasks without `Mutex`.

Access to `Mutex[T]`-wrapped values uses `lock` block syntax (not `.lock()` method):

```kara
let bank = Mutex(Bank { balances: ... })

// Single access
let bal = lock bank { bank.balances.get("Alice") }

// Multi-step atomic operation
lock bank {
    let bal = bank.balances.get("Alice")
    if bal >= amount {
        bank.balances.set("Alice", bal - amount)
        bank.balances.set("Bob", bank.balances.get("Bob") + amount)
    }
}
```

**Why `lock { }` blocks instead of `.lock()` guards:**
- Scope is visual ŌĆö braces show exactly where the lock is held; no risk of holding a guard too long.
- No new concepts ŌĆö block syntax is control flow, like `if` or `for`. No guard values, drop semantics, or lock poisoning questions.
- Consistent with K─üra's philosophy ŌĆö the compiler handles the mechanism (lock/unlock), the programmer declares intent (exclusive access scope).
- Early return, break, or panic inside a `lock` block automatically releases the lock.

**Why not the other options:**
- **(a) RefCell ŌåÆ RwLock auto-promotion** ŌĆö hidden performance cliff at the Rc/Arc boundary; deadlock risk with multiple `mut` fields locked in different orders.
- **(b) Per-instance effect tracking** ŌĆö contradicts v8 ┬¦2 resolution (effects are per-resource-type, not per-instance). Would require a new tracking mechanism.
- **(d) Per-field spinlocks** ŌĆö lock cost on every field access even in single-threaded code; burns CPU under contention.

**Rationale for (c):** Most `shared struct` values crossing task boundaries are either transferred (one task hands off to another) or read-only. Only concurrent mutation ŌĆö the case where hidden locking causes the most surprising performance ŌĆö requires `Mutex`. This keeps the explicit/implicit boundary reasonable: the compiler decides Rc vs Arc; the user opts into synchronization when they genuinely need concurrent mutation.

### 2. Lifetime inference for borrowed structs ŌĆö RESOLVED

**Context:** The spec says borrowed structs (structs with `ref` fields) have their scope bounded by borrowed sources, and "the compiler infers and enforces the constraint" without named lifetime parameters.

**Problem:** The inference algorithm is not specified. Simple cases work:

```kara
struct Parser {
    source: ref String,
    position: i64,
}

fn make_parser(s: ref String) -> ref Parser {
    Parser { source: s, position: 0 }
}
```

But what about borrowed structs with multiple `ref` fields from different sources?

```kara
struct Joiner {
    left: ref String,
    right: ref String,
}

fn make_joiner(a: ref String, b: ref String) -> ref Joiner {
    Joiner { left: a, right: b }
}
```

Is the return implicitly `ref(a, b) Joiner`? What about:

```kara
struct Nested {
    parser: ref Parser,   // Parser itself borrows a String
    config: ref Config,
}
```

Now `Nested` transitively borrows whatever `Parser` borrows, plus `Config`. The compiler must track transitive borrow chains without named lifetimes.

**What needs deciding:**

- **Inference rules for multi-source borrowed structs.** When a struct has `ref` fields borrowing from different parameters, does the compiler require `ref(a, b)` on the return type, or does it infer the sources automatically?
- **Transitive borrows.** If `Nested` contains a `ref Parser` which itself borrows a `String`, does the compiler track the transitive chain? How deep?
- **Failure mode.** When inference fails (the compiler can't determine the borrow source), what happens? Hard error? Suggest RC fallback? This is the escape hatch question ŌĆö Rust's answer is "write lifetime annotations." Kara doesn't have those.

**Options:**

- **(a) Require explicit `ref(...)` on all multi-source returns.** The compiler infers single-source returns automatically. For multiple sources, the programmer writes `ref(a, b)`. Transitive borrows flatten ŌĆö if `Parser` borrows from `s`, then a function returning `ref Nested` that contains `ref Parser` must list `s` in its `ref(...)` set.
  - Pro: Simple. Predictable. The programmer names the sources.
  - Con: For deeply nested borrowed structs, the `ref(...)` list grows long. Still simpler than `'a, 'b` but similar in spirit.

- **(b) Full inference ŌĆö compiler traces all borrow sources automatically.** The compiler walks the struct's `ref` fields, traces them to parameters, and infers the borrow set. No annotation beyond `ref` on the return type.
  - Pro: Zero annotation. Maximum ergonomics.
  - Con: Complex inference. Error messages may be opaque ("cannot infer borrow source for field `parser.source`"). May hit undecidable cases with generic borrowed structs.

- **(c) Borrowed structs are rare ŌĆö push users toward RC.** If the compiler can't infer the borrow set in one pass, it falls back to RC with a performance note. Borrowed structs remain a power-user feature for simple cases.
  - Pro: Pragmatic. Most users never hit the edge cases.
  - Con: Penalizes a legitimate performance pattern. "Your struct is too complex, here's an RC" is a poor error experience.

**Decision:** RESOLVED ŌåÆ **Full inference (option b). No `ref(a, b)` on borrowed struct returns.**

The compiler fully infers the borrow set for borrowed struct returns by tracing each `ref` field back to its source parameter. No annotation beyond `ref` on the return type is ever needed for borrowed structs.

**Inference algorithm ŌĆö the compiler traces `ref` fields through:**
1. **Direct assignment** ŌĆö `Parser { source: s }` ŌåÆ source borrows from `s`
2. **Local variable chains** ŌĆö `let src = s; Parser { source: src }` ŌåÆ trace through `src` to `s`
3. **Conditionals** ŌĆö union of all branches. `if flag { a } else { b }` ŌåÆ borrows from `{a, b}`
4. **Function call returns** ŌĆö follow the callee's `ref` return annotation. `pick(x, y)` where `pick` returns `ref(x, y)` ŌåÆ borrows from `{x, y}`
5. **Field access on `ref` parameters** ŌĆö `u.name` ŌåÆ borrows from `u`
6. **Transitive borrows** ŌĆö tracked to arbitrary depth. If `Parser` borrows from `s`, and `Nested` contains `ref Parser`, the compiler traces through to `{s}`

**Why inference always succeeds for borrowed structs:** Every `ref` value in a well-typed program already has a traceable source ŌĆö that's a requirement of source pinning (brainstorming v7 ┬¦10). A `ref` field in a borrowed struct is just a `ref` value assigned to a field. If the compiler can't trace it to a parameter, that's already a source pinning error ("cannot return borrowed struct ŌĆö `ref` field borrows from local variable"), not an inference failure.

**No `ref(a, b)` on borrowed struct returns.** `ref(a, b)` remains available for plain `ref` returns (e.g., `fn longer(a: ref String, b: ref String) -> ref(a, b) String`), where the compiler needs disambiguation between parameters. For borrowed structs, the struct's field assignments already provide the disambiguation ŌĆö the compiler knows which field borrows from which parameter by tracing the construction.

**Escape hatch when borrow complexity is too high:** If a programmer finds themselves building deeply nested borrowed structs with complex transitive chains, the answer is `shared struct` (RC), not annotations. This aligns with K─üra's ownership tiers: owned ŌåÆ ref ŌåÆ RC. Borrowed structs are the zero-cost path; `shared struct` is the ergonomic path. The programmer never writes lifetime annotations.

**Why not the other options:**
- **(a) Explicit `ref(...)` on multi-source returns** ŌĆö if the compiler can't infer the borrow set, asking the user to write `ref(a, b)` is asking them to do the same dataflow analysis manually. That's strictly harder. And for deeply nested cases, `ref(...)` lists become K─üra's version of lifetime annotations ŌĆö exactly what the language rejected.
- **(c) RC fallback on inference failure** ŌĆö inference doesn't fail for well-typed programs (see above). Silent RC promotion would be surprising and contradict the explicit ownership tier model.

### 3. RC cycles through dynamic dispatch ŌĆö RESOLVED

**Context:** The spec says "the compiler analyzes the type graph at compile time and rejects types that form ownership cycles." This works for concrete types.

**Problem:** When `dyn Trait` is involved (deferred to Phase 8 but the design must account for it), cycles can form through trait objects that are invisible at the type level:

```kara
trait Node {
    fn children(ref self) -> Vec[ref dyn Node];
}

shared struct TreeNode {
    mut children: Vec[dyn Node],  // could contain TreeNode instances ŌåÆ cycle
}
```

The static type graph sees `TreeNode ŌåÆ Vec[dyn Node]`. It doesn't see `dyn Node ŌåÆ TreeNode` because that relationship exists only at runtime. The compile-time cycle detector cannot catch this.

**What needs deciding:**

- Does `dyn Trait` inside a `shared struct` require `weak` by default?
- Or does the compiler reject `shared struct` fields of type `dyn Trait` unless explicitly marked `weak`?
- Or is this accepted as a known limitation, with runtime leak detection as the mitigation?

**Options:**

- **(a) `dyn Trait` in `shared struct` requires `weak` annotation.** Any `shared struct` field that holds a `dyn Trait` value (directly or inside a container) must be `weak`. This is conservative ŌĆö it prevents cycles but forces `Option[T]` access on fields that may not actually cycle.
  - Pro: No leaks possible. Simple rule.
  - Con: Overly restrictive. A `shared struct Widget { children: Vec[weak dyn Renderable] }` where children never reference the parent is forced into `weak` for no reason.

- **(b) Reject `dyn Trait` inside `shared struct` entirely.** Use generic type parameters instead, which the compiler can analyze statically.
  - Pro: Preserves the static cycle guarantee completely.
  - Con: Severely limits `shared struct` usability for heterogeneous collections (the primary use case for `dyn Trait`).

- **(c) Allow it, accept potential leaks, provide runtime leak detection.** A debug-mode leak detector runs at program exit and reports unreachable `shared struct` instances with non-zero RC. The programmer adds `weak` where cycles are identified.
  - Pro: Pragmatic. Most programs don't have cycles through trait objects.
  - Con: Leaks in production. Contradicts "no runtime cycle collector."

- **(d) Require explicit `#[acyclic]` annotation on traits used in `shared struct` fields.** The compiler trusts the annotation and does not insert `weak`. If a cycle forms at runtime, the RefCell/RC machinery detects it (or it leaks). The annotation is a contract: "implementations of this trait will not form cycles back to the containing type."
  - Pro: Explicit intent. No false positives. Documents the design constraint.
  - Con: Trust-based ŌĆö the compiler cannot verify the annotation. A violation is a silent leak, not a compile error.

**Decision:** RESOLVED ŌåÆ **Option (e) `#[cyclic]` annotation + debug-mode leak detector.**

Default: `dyn Trait` inside `shared struct` is allowed without `weak`. The compiler assumes no cycles. When the programmer knows a trait can form cycles through dynamic dispatch (e.g., bidirectional tree structures, observer patterns), they annotate the trait with `#[cyclic]`:

```kara
#[cyclic]
trait Node {
    fn parent(ref self) -> Option[dyn Node]
    fn children(ref self) -> Vec[dyn Node]
}

shared struct Element {
    mut parent: weak dyn Node,    // compiler enforces weak ŌĆö Node is #[cyclic]
    mut children: Vec[dyn Node],  // strong refs downward ŌĆö fine
}
```

When a trait is marked `#[cyclic]`, the compiler requires `weak` on any `shared struct` field of type `dyn Trait` (or a container of `dyn Trait`) that could form a back-edge. Without `#[cyclic]`, no `weak` is required.

**Debug-mode leak detector (safety net).** In debug builds, the runtime tracks all `shared struct` allocations. At program exit, any instance with RC > 0 that is unreachable is reported:

```
leak[rc-cycle]: 2 shared struct instances not freed at program exit
  Element at src/ui.kara:15 ŌĆö RC count: 1, allocated at line 42
  Element at src/ui.kara:15 ŌĆö RC count: 1, allocated at line 43
  hint: these may form a reference cycle through `dyn Node`
  hint: consider marking trait `Node` as #[cyclic] and using `weak` on back-edges
```

In release builds, the leak detector is compiled out entirely ŌĆö zero overhead.

**Why this combination:**
- **`#[cyclic]` covers the known cases.** The programmer (or AI) identifies traits where cycles can form and annotates them. The compiler then enforces `weak` at the field level ŌĆö a compile-time guarantee for annotated traits.
- **Debug leak detector covers the forgotten cases.** If `#[cyclic]` is missed, the leak detector catches it during development. This is reactive (requires test coverage that triggers the cycle), but cycles through `dyn Trait` are rare in practice ŌĆö they occur almost exclusively in bidirectional tree/graph structures with parent pointers and observer patterns.
- **Low annotation burden.** Most traits are not cyclic. Only traits used in bidirectional shared data structures need the annotation. AI-generated code can learn this pattern reliably.
- **No runtime cycle collector.** The debug leak detector is a post-mortem check at program exit, not a runtime collector. Production builds have zero overhead.

**Why not the other options:**
- **(a) Force `weak` on all `dyn Trait` in `shared struct`** ŌĆö overly restrictive. Most `dyn Trait` usage doesn't form cycles. Forcing `Option[T]` access everywhere penalizes the common case.
- **(b) Ban `dyn Trait` in `shared struct`** ŌĆö kills heterogeneous collections in shared types, the primary use case for `dyn Trait`.
- **(c) Runtime leak detection only** ŌĆö no compile-time guarantee at all. Entirely dependent on test coverage.
- **(d) `#[acyclic]` annotation** ŌĆö inverts the burden. Most traits would need the annotation since most are acyclic. `#[cyclic]` targets the minority that actually has cycles.

### 4. Module system details ŌĆö RESOLVED

**Context:** The spec defines `pub` vs private visibility and fully qualified paths, but omits structural details that implementers need.

**What needs deciding:**

- **File-to-module mapping.** Is it one file per module (Go-style)? Directory with `mod.kara` (Rust-style)? Implicit from directory structure? This affects how the compiler discovers source files and how developers organize code.
- **Circular module dependencies.** Are they allowed? Rust forbids circular `mod` declarations but allows circular `use` imports. Go allows circular references within a package but not across packages. The effect system's per-module inference assumes modules can be processed in dependency order ŌĆö circular dependencies break topological ordering.
- **Re-exports.** Is `pub use other_module.SomeType;` supported? This is essential for library API design (exposing types from internal modules at the crate root).
- **Conditional compilation.** Is there a `#[cfg(...)]` equivalent? The project profiles (`kernel`, `embedded`) imply platform-specific code paths. Without conditional compilation, platform-specific code requires separate files and a build-system-level selection mechanism.

**Options:**

These sub-questions are somewhat independent. A few high-level stances:

- **(a) Rust-like.** Explicit `mod` declarations, `mod.kara` for directories, `pub use` for re-exports, `#[cfg]` for conditional compilation. Proven model, but verbose.
- **(b) Go-like.** Directory = package, all files in a directory share a namespace, no `mod` declarations. Simpler but less control over visibility and namespacing.
- **(c) Hybrid.** Directory structure defines modules implicitly (no `mod` declarations), but `pub use` and `#[cfg]` are available. Reduces boilerplate while keeping power features.

**Decision:** PARTIALLY RESOLVED ŌĆö file-to-module mapping and visibility decided. Re-exports and conditional compilation still open.

**Resolved: File-to-module mapping ŌĆö directory structure defines modules implicitly (hybrid approach).**

The directory structure defines the module tree. No `mod` declarations needed. Each `.kara` file is its own module, namespaced by its path relative to the project root. `src/db/connection.kara` defines the `db.connection` module. The compiler discovers all `.kara` files automatically. Entry points: `main.kara` for executables, `lib.kara` for libraries.

**Why not Rust-style:** `mod` declarations are pure boilerplate ŌĆö they repeat what the file system already says. This adds ceremony that doesn't align with K─üra's "compiler handles the mechanism" philosophy.

**Why not Go-style:** All files in a directory sharing private access undercuts K─üra's encapsulation model. A "private" function visible to 10 other files in the same directory isn't meaningfully private.

**Resolved: Three-level visibility model.**

| Keyword | Visible to |
|---|---|
| `pub` | End users + all project files + same directory |
| *(default, no keyword)* | All files in the project + same directory (not end users) |
| `private` | Same directory only |

- **`pub`** ŌĆö public API, visible to end users of the package. Public functions must declare effects explicitly.
- **Default (no keyword)** ŌĆö project-internal. Visible to all files in the project, not to end users. The most common case. Effect inference applies.
- **`private`** ŌĆö same directory only. Does not leak into parent or child directories. Useful for internal helpers shared between related files in the same directory (e.g., `db/connection.kara` and `db/schema.kara` sharing a helper in `db/helpers.kara`).

**Rationale for three levels:** Two levels (`pub` + private) forces a choice between "end users can see it" and "only this file can see it." The most common need ŌĆö sharing code across files within a project without exposing it to end users ŌĆö has no keyword, making it the zero-annotation default. `private` is opt-in for the rare case where directory-level encapsulation matters.

**Resolved: No circular module dependencies.** Circular imports are a compile error. Modules are processed in dependency order for effect inference.

**Resolved: Re-exports (`pub use`) ŌĆö supported.**

`pub use path.Item` re-exports an item at the current module level. Essential for library API design ŌĆö end users import from the package root without knowing the internal file structure. Internal file reorganization is not a breaking change.

```kara
// lib.kara
pub use db.connection.Connection
pub use db.pool.Pool
pub use auth.token.Token

// End users write:
use mylib.Connection   // not mylib.db.connection.Connection
```

**Resolved: Conditional compilation ŌĆö platform directories only (v1). `#[cfg]` deferred.**

Platform-specific code uses file suffixes:
```
net/
  poller_linux.kara
  poller_macos.kara
```

The compiler picks the right file based on the target platform. For cases where a single function differs, extract the platform-specific part into a function in the platform file and call it from shared code.

`#[cfg]` annotations for fine-grained per-item conditional compilation are deferred to post-v1. See future_decisions.md ┬¦12. Platform directories cover the common case; every `#[cfg]` use case can be handled by extracting into a platform file.

## Category B: Could Go Either Way (Low-Risk, Additive)

### 5. Float NaN semantics at FFI boundaries ŌĆö RESOLVED

**Context:** Kara defines `NaN == NaN` as `true` with a total order. C/LLVM follow IEEE 754 where `NaN != NaN`. The spec acknowledges this but provides no compiler-level mitigation.

**Problem:** A float returned from `extern "C"` may be NaN. Code that checks `result == result` (a common C idiom for "is not NaN") behaves differently in Kara ŌĆö it's always `true`. This is a subtle semantic mismatch at a boundary where bugs are already hard to find.

**Decision:** RESOLVED ŌĆö add the warning.

The compiler emits a warning when a `f32`/`f64` value crosses an FFI boundary (passed to or returned from an `extern` function). The warning suggests using `is_nan()` for explicit NaN checks rather than relying on equality semantics. Suppressible with `#[allow(ffi_float_semantics)]`.

**Why a warning, not a hard error:** Most FFI float values aren't NaN. Forcing every float through `is_nan()` would be overkill. The warning educates the programmer about the semantic mismatch; they decide whether NaN is a concern for their specific use case.

**Cost:** Low ŌĆö a simple lint on `extern` function signatures involving float types.

### 6. Production contract checking (`#[checked]`) ŌĆö RESOLVED (deferred to post-v1)

**Context:** `requires`/`ensures`/`invariant` are stripped in release builds. For a language targeting AI-generated code, production-time verification has value.

**Problem:** Contracts that vanish in production are documentation, not safety. An AI-generated function with `requires x > 0` that receives `x = -1` in production silently proceeds with undefined results. The contract was the safety net, and it was removed.

**Resolution:** Deferred to post-v1. For v1, contracts are stripped in release builds ŌĆö no exceptions. If production-time validation is needed, write an explicit `if` check and return `Result`. This is already the idiomatic pattern for input validation.

**Rationale for deferral:**
- `Result`-based failure mode is too restrictive (only works on functions returning `Result`).
- `panics` failure mode is simpler and more honest (a violated precondition is unrecoverable), but adds complexity to the effect system.
- `#[checked]` creates a third category between "debug assertion" and "real validation logic" that may confuse users.
- The feature is purely additive ŌĆö can be added post-v1 without breaking existing code.

**When added:** `#[checked]` should use `panics` semantics (violated contract = panic, adds `panics` to the function's effect set). A violated precondition in production is an unrecoverable situation, consistent with how `as` (asserting cast) and out-of-bounds indexing already work.
