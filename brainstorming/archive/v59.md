## Findings

### F1. Auto-concurrency diagnostics for resource modeling

Auto-concurrency correctness depends heavily on resource modeling. Cross-resource dependencies are pushed to the programmer through data flow, resource aliasing, parameterized resources, or `seq {}`. That is acceptable, but it means the language needs excellent diagnostics around "you probably modeled this too coarsely or too finely." Otherwise users will either overuse `seq {}` or create fake resources to force ordering.

**Resolution: YES (light).** The reactive surface (`karac query concurrency` + diagnostic explanations) already lets users diagnose their own resource modeling — the existing AI-First Compiler Interface § 2 commitment ("every major inferred decision must be queryable") covers the "why didn't this parallelize?" case. The reviewer's concern is about *proactive* lints, which are bonus features, not a v1 gap. Speccing those lints before real Kāra programs exist would be guessing — what looks like over-coarse modeling in one domain may be legitimate sequential business logic in another. Action taken: (1) added a single paragraph in design.md § Feature 5 (after the existing `seq { }` language-health-metric paragraph) extending the same rationale to other resource-modeling friction signals — dense `independent` declarations, phantom resources for ordering — and committing them to `karac build --perf-report` rather than normal compilation; (2) added a P2 entry in `deferred.md` ("Resource-Modeling Friction Lints") with explicit non-breaking guarantees so the lints can land in a minor version once real-world patterns are known.

### F2. Transparent effects carve-out — keep surface tiny

Transparent effects are useful but dangerous conceptually. The carve-out for tracing, logging, and nondeterminism is ergonomic, but it creates a second class of effects that do not really count. Keep this surface tiny. If too many things become transparent, the effect system loses trust.

**Resolution: NO.** The reviewer's concern is generic "be careful with carve-outs" rather than engagement with the actual design. Multiple tells: (1) the reviewer lumps logging into the transparent carve-out, but the design *explicitly forbids* structured logging from being transparent (lines 4502, 4817 reserve transparent verbs for diagnostic bailout only); (2) "second class of effects that do not really count" misreads Principle 3 — transparent effects *do* participate in inference and `karac explain`, they are simply excluded from conflict analysis (visibility ✓, propagation ✓, conflict ✗); (3) the review never engages with Principle 3 itself, which is the existing gate against surface growth; (4) the v1 built-in transparent surface is exactly one verb (`debugs`) — already as tight as the use case admits. Existing governance (Principle 3 + explicit denials of `logs`/`panics`/`allocates(Heap)` + the "only concession to ergonomics" framing on line 4815) addresses the concern without textual change. One residual: line 5714's "may grow" is mildly permissive; if real-world transparent-surface growth pressure ever materializes, that line is where to tighten — not worth pre-empting now.

### F3. `shared struct` / `par struct` split — early-demo simplification

The `shared struct` / `par struct` split is principled, but complex. `shared struct` with hidden per-field borrow flags, RC-only behavior, migration to `par struct`, `Mutex` / `Atomic` rules, and effect resources for mutation is a lot. Consider making `par struct` the only shared reference-semantics story for early public demos, and leave `shared struct` as advanced or preview.

**Resolution: NO.** Surface-level review point that doesn't engage with the actual design. Tells: (1) the listed complexity (per-field borrow flags, `Mutex`/`Atomic` rules, RC-only behavior) cannot be eliminated by removing `shared struct` — it must exist somewhere, and pushing it all to `par struct` forces single-task tree/graph code onto Arc with global atomic-refcount overhead; (2) the review treats the two forms as redundant siblings, but they encode structurally different aliasing models (single-task RC vs. cross-task Arc with `Atomic`/`Mutex` field constraints enforced at the definition site); (3) the design already explicitly considered making `par struct` the default — line 6712 documents this as a deferred decision empirically gated on Phase 8 single-task overhead measurement (~5 ns/field access threshold), with a `karac` migration tool already specified for the inversion path if it materializes; (4) line 6632 explicitly notes the cost the reviewer ignored — "a `par struct` used exclusively in single-task contexts pays Arc's atomic reference-count overhead everywhere." The fragment of the concern that's real — *which shared form leads in public demos* — is a v60 messaging question (route to D3, the 5-step demo path), not a v59 design change. No action.

### F4. Layout-block expressivity gap (GPU SoA footgun)

Layout blocks are promising, but "same type, different physical layout, no API distinction" is a double-edged sword. It preserves ergonomics, but makes layout requirements hard to express. The spec admits GPU kernels that require SoA must document it separately. That is a future footgun. At some point the language may need a lightweight way to express layout capability without making every layout part of the public type.

**Resolution: YES (light).** The gap is real and the design already admits it explicitly at line 4171 ("the type system does not enforce it"). The reviewer's hedged phrasing ("the language *may* need a lightweight way...") flags the gap rather than proposing a mechanism — the right design depends on Phase 10 GPU code that doesn't exist yet. Action taken: (1) added a P2 deferred entry in `deferred.md` ("Layout-Capability Bound") that captures the gap, sketches plausible mechanism shapes (where-bound, attribute, structural trait), gates revisit on Phase 10 GPU code, and commits non-breaking-by-construction (opt-in bound, existing signatures unaffected); (2) added a one-sentence cross-reference at design.md line 4171 pointing readers to the deferred entry — the existing line-4171 admission already does most of the work; the cross-reference closes the loop. Same template as F1: acknowledge real gap, defer mechanism to post-MVP empirical data, lock the non-breaking commitment.

### F5. Add `karac tdd` watch driver with unified JSONL stream

The design already has most of the raw material for LLM-driven test-first development:

1. Colocated `_test.kara` files with private sibling access.
2. Fresh per-test provider injection via `#[with_provider]`.
3. Structured JSONL test-runner output.
4. Streaming compiler phase events for LLM clients.
5. Root-cause diagnostics, derivation chains, and machine-oriented compiler output.

The missing piece is not another language feature. It is a small orchestration surface that turns these pieces into a tight red-green loop between an LLM code author and the compiler:

```bash
karac tdd --watch --output=jsonl
karac tdd src/foo.kara::function_name --output=jsonl
```

This mode should emit one unified event stream for build, diagnostics, test execution, and cycle summary:

```json
{"type":"cycle_start","changed":["src/foo.kara","src/foo_test.kara"]}
{"type":"phase_start","phase":"parse"}
{"type":"diagnostic","phase":"parse","id":"d1","primary":true}
{"type":"test_fail","test":"foo::test_empty_input","left":"...","right":"..."}
{"type":"cycle_complete","status":"red","next_best_action":"fix_primary_diagnostic"}
```

Recommendation: do not build a heavyweight reactive compiler server yet. Add a thin `karac tdd` / watch driver that reuses existing compiler phases, emits one unified JSONL stream, supports related-test selection, and treats red-green-refactor as the native development workflow.

**Resolution: YES (capstone framing).** The watch driver is a real productized loop on top of pieces that are 80% specced (`karac test` JSONL, `karac build --output=jsonl`, structured diagnostics) and 20% missing (envelope unification, red-state cycle summary, test selection, scaffolding, JSON explain — F6-F11 in this doc, plus F18 from the test-shaped extensions). F5 is the integrating capstone; F6-F11 / F18 are the sub-pieces that compose into it. Action taken: added a P2 deferred entry "`karac tdd` Watch Driver" that captures the cycle envelope sketch, lists the prerequisite items by name (each decided independently in this v59 walk-through), commits non-breaking-by-construction (new subcommand wraps existing tools), and explicitly leaves the cycle event schema TBD until prerequisites land — specifying it now would over-spec without implementation experience. No design.md change; this is pure tooling. The capstone framing exists so prerequisite items have a destination to aim at and so the watch driver doesn't get lost as a coherent product after individual prerequisites resolve.

### F6. Unify build and test event envelopes

`karac build --output=jsonl` uses `"type"`, while `karac test` currently uses `"event"`. A TDD client should consume one shape. Future TDD mode can wrap test events as `type: "test_pass"` / `"test_fail"` without breaking the existing test runner.

**Resolution: YES.** Confirmed against `src/cli.rs:2919` — implementation diverged from the build pipeline (`"type"`) by using `"event"` for test events. design.md was silent on the discriminator key for test events, so this was an under-specified place where the implementation made an inconsistent choice. Pre-1.0, cheap to fix. Migrated test runner `"event"` → `"type"` to match the build pipeline. Action taken: (1) `src/cli.rs:emit_test_event` now writes `"type"` instead of `"event"`, with the doc comment updated; (2) `tests/cli.rs:jsonl_lines` tightened to filter on the five known test-runner event types (`run_start` / `test_pass` / `test_fail` / `test_skip` / `summary`) since the unified discriminator alone can no longer distinguish test events from build-pipeline diagnostics that share the same envelope key; (3) design.md test runner output spec updated to explicitly require `"type"` and cross-reference the streaming JSONL envelope, so future implementers can't drift again. Build + clippy + full test suite (74 cli tests + 49 other suites) all pass.

### F7. Make intentional red states first-class in cycle summary

TDD starts with failing tests. The cycle summary should distinguish compile failure, no tests discovered, tests compiled and failed, tests passed, and tests skipped because resources were unavailable.

**Resolution: YES (a) — taxonomy locked in `karac tdd` deferred entry.** The five distinct end-states the reviewer named are all observable from already-specced events; the watch driver synthesizes them into a single `status` field. Action taken: extended the existing `karac tdd` Watch Driver entry in `deferred.md` with a "Cycle-summary status taxonomy" subsection that pins the five status values (`compile_error`, `no_tests_discovered`, `tests_failed`, `tests_passed`, `tests_skipped_unavailable`), the conditions that trigger each, the LLM-action guidance per state, the precedence order when multiple conditions apply within one cycle (`compile_error` > `tests_failed` > `tests_skipped_unavailable` > `tests_passed` > `no_tests_discovered`), the `--all` interaction (any skip becomes a fail), and a forward-compat rule (new `test_skip` reasons route through `tests_skipped_unavailable`; expanding the top-level vocabulary is a major-version decision). No design.md or code change — the taxonomy is part of the deferred capstone, locked so the watch driver doesn't have to invent it later.

### F8. Targeted test selection: `--failed` / `--related` / `--since`

Substring filtering is useful but crude. LLM loops want `--failed`, `--related src/foo.kara`, `--since HEAD`, and precise function/module filters.

**Resolution: YES (a) — flag set locked in `karac tdd` deferred entry.** Five flags pinned with semantics, dependencies, and combination rules: `--failed` (no compiler dependency, pure runner book-keeping via `.kara/test-state.json` cache), `--related <FILE>` (depends on `karac query affected-by`), `--since <REV>` (depends on the same query plus git diff), `--module <path>` (no dependency, prefix filter on FQ ID), `--exact <full::test::id>` (no dependency, equality filter on FQ ID). The existing substring filter remains the casual default; new flags compose orthogonally as set intersections. `--related` and `--since` block on the affected-by query landing — without it, both would need ad-hoc heuristics that miss real reach edges (closure captures, trait-object dispatch, generic monomorphizations). Cache schema for `--failed` is internal, not part of the JSONL contract, so it can evolve freely. No design.md or code change — taxonomy is part of the deferred capstone, locked so the watch driver doesn't have to invent it later.

### F13. Signature-from-call-site stubs as structured diagnostic

The classic TDD opener is a test that calls a function that does not yet exist. Today the resolver emits `unresolved identifier`. This reuses the existing `hints[].diff` shape — no new protocol — and lets the LLM begin every cycle from one `karac test --output=jsonl` parse instead of two roundtrips. The stub is a suggestion, not an action: the compiler proposes, the LLM accepts or rewrites.

**Resolution: YES (b) — light P1 spec; inference policy deferred to implementation.** Action taken: (1) added a P1 deferred entry "Signature-from-Call-Site Stub Diagnostic" in `deferred.md` (placed adjacent to `karac explain --format=json` since both are structured-diagnostic enhancements) — fully specs the diagnostic shape, the activation gate (`_test.kara` only — production-file unresolved calls keep the plain diagnostic), the body convention (cross-references F9's compiling-skeleton policy for `todo()` defaults), and notes that the inference layer (resolver-time best-effort vs. post-typecheck refinement) is left to implementation time since the diagnostic shape is identical at either layer; (2) added an `[→ P1]` checklist entry in `implementation_checklist.md` Phase 5.3 (Advanced Diagnostics) pointing to the deferred entry. ### F14. Promote passing test assertions into contracts

When `test_sort_preserves_length` asserts `sort(v).len() == v.len()` and passes, that is runtime evidence; an `ensures(result) result.len() == v.len()` would make it static, and per the existing static-discharge rule (design.md § Static discharge under Contracts) the compiler may already prove and strip the runtime check. Add `karac test --suggest-contracts` that, for each passing test, emits structured suggestions mapping assertion expressions to candidate `requires` / `ensures` clauses on the function under test, with the same `derivation` shape used elsewhere. The LLM authors a test, the compiler proposes the contract, the next build discharges it. This is the natural inverse of derivation chains and the only place in the design where test artifacts feed back into the declarative surface of the language.

**Resolution: YES (c) — P2 with explicit promotion gates and additivity note.** Three open questions block a confident P1 commitment: (1) specific-vs-universal classification of assertions is fundamentally heuristic and we have no Kāra test corpus to calibrate against, (2) pre-condition inference from passing tests is unsound (a test that doesn't exercise the empty case doesn't logically *require* non-empty input — `ensures` is the tractable direction, `requires` isn't), (3) static-discharge integration is the unique compiler value-add (LLMs can analyze passing tests via prompting on the same source) and that depends on Phase 9 contract prover maturity which doesn't exist yet. Action taken: (a) added a P2 deferred entry "Promote Passing Test Assertions into Contracts" with the `suggest_contract` event shape, the three deferral reasons, and **explicit promotion gates** for P2 → P1 (Phase 9 prover handles len/equality/arithmetic refinements; ≥10 real Kāra projects with substantive test suites exist for calibration; prototype implementation shows honest acceptance rate); (b) explicitly noted that F14 is *additive* to the F5 `karac tdd` Watch Driver capstone, not a prerequisite — the LLM TDD loop (F5/F6/F7/F8/F9/F13) ships fine without F14; (c) refined the broader P0/P1/P2/P3 framing in `deferred.md` and the project memory to clarify that P2 ≠ "may never ship" — P2 = "important post-v1 language features, will be built by author or community"; P3 = libraries/frameworks where the may-or-may-not is open. No checklist entry — P2 entries don't need v1 implementation_checklist coverage; the deferred entry is the tracking surface.

### F15. Auto-derived `Arbitrary`/`Shrink` honoring refinements and invariants

The current `#[derive(Arbitrary)]` generates fields independently; types with refinements or `invariant` blocks must implement `Arbitrary` by hand (design.md § Property tests). For LLM-driven property testing this is the largest grunt-work tax in the test surface. Since refinement predicates and invariants are already typed expressions, the derive machinery can plumb them into a rejection-sampling generator and an invariant-respecting shrinker. Predicates the refinement system can prove become direct constructors; predicates it cannot become rejection filters with a configurable bailout. The deeper consequence is that one piece of source — the predicate — does triple duty: type rule, contract, and test-input generator. Worth a separate design pass before committing, because the rejection-vs-construction split has real implications for shrinking quality and test runtime.

**Resolution: YES (c) — P2 with explicit promotion gates and additivity note.** The reviewer's own caveat ("worth a separate design pass before committing") plus genuine implementation hardness (especially invariant-aware shrinking, which is research territory across PBT frameworks — neither QuickCheck, Hypothesis, proptest, nor jqwik solves it generically) plus dependence on Phase 9 refinement-type / `invariant`-block maturity all point to P2. Three substantive open questions block a confident P1 commitment: (1) predicate-pattern catalogue (which patterns get a direct-constructor vs. rejection-filter — empirically calibrated, not theoretically derivable), (2) invariant-aware shrinking quality (the naive filter-through-predicate approach can produce poor shrinking; constraint-solving alternatives are more general but expensive), (3) rejection bailout default (too low → false-negative test failures, too high → tests hang). Action taken: added a P2 deferred entry "Auto-Derived `Arbitrary` and `Shrink` Honoring Refinements and Invariants" with the two-strategy generator description (direct constructor for structural predicates, rejection filter for non-structural), the invariant-respecting shrinker discussion, the three open questions, **explicit promotion gates** for P2 → P1 (Phase 9 substrate maturity, prototype implementation with measured catalogue handling common cases, shrinker prototype showing acceptable quality on a benchmark suite, ≥10-project corpus for bailout-default calibration), and the additivity note (F15 doesn't block F5 capstone; hand-written `Arbitrary` / `Shrink` remains the supported path until F15 ships). Stronger deferral case than F14 — F14's translation is fuzzy but its output (a contract suggestion) is well-understood; F15's output (generator + shrinker) is well-understood but its *implementation* requires real research into invariant-aware shrinking.

### F16. `karac query affected-by` and `karac query coverage`

The existing follow-up calls for `--related` and `--since HEAD` as test-selection flags. Add the underlying query as a first-class entry in the compiler query API, alongside `effects` / `ownership` / `concurrency`. Inputs: a file:line or function path. Output: the transitive callers and callees the call graph already computes for effect inference, plus the test functions that reach the input through that graph. This satisfies the design's commitment that every major inferred decision must be queryable (design.md AI-First Compiler Interface § 2), and it gives the test-selection flags a well-defined backing surface that other tools (IDE plugins, CI affected builds) can reuse.

**Resolution: YES (a) — full P1 spec; structural prerequisite for prior commitments.** Three reasons converge for the strongest commitment: (1) F8 / F11 / F5 (all P1) depend on this query — without it, `--related` / `--since` test selectors and the `coverage_delta` event ship with ad-hoc heuristics that the design entries themselves call out as inadequate ("miss real reach edges — closure captures, trait-object dispatch, generic monomorphizations"); (2) the call graph is already computed for effect inference, so F16 is fundamentally exposing existing data via the query API rather than computing new analysis — low marginal implementation cost, no research questions; (3) design.md § AI-First Compiler Interface § 2 already commits to queryable-everything for inferred decisions — F16 makes good on that for the call graph specifically. The reviewer's "and `karac query coverage`" framing turns out to refer to the test-coverage relationship (which tests reach which code) which is just the test-filtered view of affected-by — one query with views, not two endpoints. Action taken: (a) added P1 deferred entry "`karac query affected-by` — Call-Graph Reach Query" with full spec covering invocation forms (file path, file:line range, function path), JSONL output schema (`callers` / `callees` / `tests` arrays with `{fn, file, line}`), call-graph subtleties (trait-object dispatch / generic monomorphization / closure captures + escape / FFI boundaries / SCCs — well-understood engineering with known approaches), structural-prerequisite rationale, and non-breaking guarantee; (b) added `[→ P1]` checklist entry in Phase 5.3 alongside the other `karac query` subcommands; (c) cleaned up cross-references in F8 (the `karac tdd` test-selection-flags table) and F11 (the `coverage_delta` paragraph) to point at the F16 entry directly instead of "separate deferred item" hedging.

### F17. Doctests as `#[example]` blocks on `pub` items

Not present in the current testing section. Doctests are the smallest TDD unit: an example in the docstring that the compiler runs as a test, with the docstring as the assertion site. They fit the existing test-prelude injection mechanic (`assert`, `assert_eq` already injected into `_test.kara`) by extending it to `#[example]` attribute blocks on `pub` items in regular source files under `karac test`. The constraint is that examples must declare the effects they touch — which falls out of the existing public-effect declaration rule with no special casing.
