# Kāra v2: Brainstorming — A Language for the AI Era

This document captures a from-first-principles rethinking of what Kāra could be. It sets aside all previous design decisions and asks: if we were building a language today with full freedom, what would it look like?

## Starting Assumptions

1. **AI writes most code** — optimize for machine readability, formal verifiability, and unambiguous semantics rather than human ergonomics.
2. **Compiler should extract maximum performance** — give the compiler every advantage to optimize aggressively.
3. **Silicon matters** — modern CPUs are memory-bound, not compute-bound. Cache misses, branch mispredictions, and memory layout dominate performance.
4. **Take the best from everything** — no loyalty to any paradigm.

---

## The Core Problem

The fundamental problem with every mainstream language is that **the programmer describes computation, and the compiler has to reverse-engineer intent.**

The programmer writes `for i in 0..n { result += data[i]; }` and the compiler has to figure out: can I vectorize this? Can I parallelize it? Is there aliasing? Are there dependencies between iterations? The compiler does heroic work (LLVM has millions of lines of optimization passes) to recover information that the programmer had in their head but didn't express.

**What if the language expressed intent directly, and the compiler didn't have to guess?**

## Features

### Feature 1: Data Layout as a First-Class Concept — COMMITTED (opt-in)

**Status:** Committed. Optional — programmer uses it only when needed.

The language separates **logical structure** from **physical layout**:

```
// Logical: how the programmer thinks about data
struct Entity {
    id: u64,
    name: String,
    position: Vec3,
    velocity: Vec3,
    health: f32,
    armor: f32,
    is_alive: bool,
}

// Physical: how data is actually stored in memory
// The compiler chooses, or the programmer overrides
layout entities: Vec[Entity] {
    // Store position and velocity together (hot path: physics)
    group physics { position, velocity }

    // Store health and armor together (hot path: combat)
    group combat { health, armor, is_alive }

    // Everything else is cold
    group metadata { id, name }
}
```

This is struct-of-arrays (SoA) vs array-of-structs (AoS) — the most impactful performance optimization in data-heavy systems — expressed as a language feature rather than a manual refactoring nightmare. The programmer writes code against `Entity`. The compiler generates code that accesses the correct physical layout. If you iterate over `entities` and only touch `position` and `velocity`, only the `physics` group is loaded into cache.

**Who does this today?** ECS frameworks (Unity DOTS, Bevy) do this manually. Zig lets you control memory layout. But no language makes it a first-class concept with compiler-managed mapping between logical and physical.

#### Discussion Notes

- **How it works:** Group names (`physics`, `combat`, `metadata`) are arbitrary programmer-chosen labels. The compiler doesn't care about names — it cares about which fields are in each group. Fields in a group are stored contiguously in memory so a single cache line fetch loads related data together.
- **Applicability:** Most impactful for game engines, simulations, scientific computing, database engines, and ML/data pipelines. For typical webservers and LeetCode-style problems, data usually fits in cache regardless of layout, so this feature would sit unused. This is a niche optimization for ~5% of programs, but powerful for that 5%.
- **The programmer still decides:** The programmer must decide which fields are accessed together (hot path grouping). The compiler handles the mechanical transformation but can't determine optimal groupings automatically — that requires runtime profiling knowledge.
- **Scope:** Only applies to homogeneous collections (arrays of the same type). Doesn't apply to HashMap, trees, or other pointer-based data structures where layout is dictated by the data structure's algorithm.
- **Not a Python dict:** This is a compile-time directive, not a runtime data structure. No hash map, no keys, no runtime overhead. The closest real-world equivalent is database column stores (Parquet, DuckDB).
- **Rust equivalent:** Multiple `Vec`s — one per field group. The `layout` block automates what a programmer would do manually by splitting `Vec[Entity]` into `Vec[Vec3]` (positions), `Vec[Vec3]` (velocities), `Vec[f32]` (healths), etc.

### Feature 2: Effect Types — COMMITTED (Option B: fixed verbs + user-defined resources)

**Status:** Committed. Fixed verbs with user-defined resources.

Every function declares what it does to the world:

```
fn calculate(x: f64, y: f64) -> f64 { ... }
// Inferred effect: none — the compiler knows this is pure

fn read_config(path: String) -> Result[Config, Error]
    reads(FileSystem) { ... }
// Declared effect: reads from filesystem

fn save_user(user: User) -> Result[(), Error]
    writes(Database) { ... }
// Declared effect: writes to database

fn process(id: u64) -> Result[Report, Error]
    reads(Database), writes(FileSystem), sends(Network) { ... }
// Multiple effects — composed from callees
```

The compiler uses effects for:

- **Auto-parallelism:** Functions with non-conflicting effects run in parallel without programmer intervention.
- **Testing:** Mock any effect boundary automatically.
- **Sandboxing:** Restrict what a module can do by limiting its allowed effects.
- **Dead code elimination:** If nothing reads from a write, the write can be eliminated.

Effects are inferred when possible. If your function only calls pure functions, its effect is `none` without you saying anything. You only annotate when using FFI or built-in I/O primitives.

**Who does this today?** Koka, Eff (research languages). Nobody in mainstream.

#### Design Decision: Option B — Fixed Verbs + User-Defined Resources

The **verbs** are a small fixed set built into the language:

```
reads, writes, sends, receives, allocates, panics
```

The **resources** are user-defined and typed by their provider trait (see Q30 for provider-rooted resource aliasing):

```
effect resource UserDB: DatabaseProvider;
effect resource OrderDB: DatabaseProvider;
effect resource FileSystem;

fn save_user(user: User) writes(UserDB) { ... }
fn load_orders(id: u64) reads(OrderDB) { ... }
```

**Note:** The final design evolves this into provider-rooted resources backed by traits (`effect resource UserDB: DatabaseProvider`) with `with_provider` injection — see design.md Feature 2 for the full syntax. The general mechanism described here is correct; the provider trait layer adds swappable implementations and instance-based aliasing detection on top.

The compiler understands verb semantics and uses them to reason about conflicts:

| Combination | Same resource | Different resources |
|---|---|---|
| `reads` + `reads` | Safe | Safe |
| `reads` + `writes` | **Conflict** — serialize | Safe |
| `writes` + `writes` | **Conflict** — serialize | Safe |
| `sends` + `sends` | Safe | Safe |
| `sends` + `receives` | Safe | Safe |
| `receives` + `receives` | Safe | Safe |
| `allocates` + `allocates` | Safe | Safe |
| `panics` | N/A — no resource | Never conflicts |

**`panics` has no resource** — it is a resource-less verb signaling potential program termination. It never participates in conflict analysis (cannot serialize with anything). It must still be declared on public functions so callers know the function can abort. See Q6 for the full primitive resource table. Although `panics` has no resource and never conflicts with other verbs, it still propagates through the call graph and must be declared on public functions — see G26 for full semantics (tracking through `unwrap()`, call chains, and the compile-time crash-freedom guarantee).

**`allocates(Heap)` is transparent by default** — it does not propagate for ordinary CPU code, so everyday functions creating `Vec` or `String` do not carry `allocates(Heap)` through every call site. `allocates` is a *documentation and gating verb*, not a concurrency verb: it never conflicts (even with itself), so it plays no role in parallelism decisions. At GPU/embedded boundaries, heap allocation is enforced via `GpuSafe` type checks and project profiles in `kara.toml` — not via conflict analysis. See Q6 for the full design rationale.

**Effect groups** let public functions name a shared effect profile rather than listing every individual effect. See Q3 for the full design. Groups also compose with `+` for modular declarations (e.g., `effect group Full = A + B`), allowing sub-groups to be composed into larger profiles without listing every individual effect again:

```
effect group OrderProcessing =
    reads(UserDB, InventoryDB),
    writes(OrderDB),
    sends(PaymentGateway, NotificationService);

pub fn process_order(order: Order) with OrderProcessing { ... }
```

**Transparent verbs** are the user-defined extension point for adding verbs beyond the six built-ins. Observability effects (logging, tracing, metrics) should never affect parallelism or propagate to callers, and transparent verbs provide this guarantee. See Q1 for the full design:

```
transparent effect verb traces;
fn debug_log(msg: String) traces(Console) { ... }
// Adding traces(...) inside a private helper never changes the public effect signature
```

This gives the compiler enough information for auto-concurrency without requiring the programmer to learn a complex effect algebra.

#### Why This Is Better Than fn/flow

The earlier `fn` (pure) vs `flow` (impure) design was binary — a function is either pure or it isn't. Effect types are a spectrum. The compiler can reason about *which specific effects* conflict and parallelize everything that doesn't conflict. Two functions that both `reads(Database)` can run in parallel. A function that `reads(UserDB)` and a function that `writes(OrderDB)` can run in parallel because they touch different resources.

### Feature 3: Algebraic Data Types with Layout-Aware Matching — COMMITTED

**Status:** Committed. Rust-style enums and pattern matching.

```
enum Shape {
    Circle { radius: f64 },
    Rectangle { width: f64, height: f64 },
    Triangle { a: Vec2, b: Vec2, c: Vec2 },
}

// Exhaustive matching — compiler error if a variant is missed
match shape {
    Circle { radius } => pi * radius * radius,
    Rectangle { width, height } => width * height,
    Triangle { a, b, c } => triangle_area(a, b, c),
}

// Layout optimization for collections of enums
layout shapes: Vec[Shape] {
    split_by_variant  // circles stored separately from rectangles
    // Now iterating over all circles is cache-perfect
}
```

Rust's `match` semantics are already the best design. Keep them exactly as-is. The addition is `split_by_variant` in the layout system, which stores each variant contiguously in memory for cache-friendly iteration.

### Feature 4: Tiered Ownership with Restricted Return Borrows — COMMITTED

**Status:** Committed. Rust ownership semantics with parameter mode inference, owned returns by default, explicit `ref` for borrow returns, and controlled RC fallback.

The key insight: **ownership inference and lifetime inference are two different problems.** Parameter mode (own vs. borrow) is easy to infer from function body analysis. Return-position borrows (which input does the return borrow from?) are the hard part — that's what Rust's `'a` annotations solve. Kāra restricts return-position borrows to eliminate ~80% of the complexity while keeping the zero-cost ownership model.

#### Part 1: Parameter Mode Inference

The compiler infers whether each parameter is owned, borrowed, or mutably borrowed by analyzing the function body:

```
fn process(data: Data, config: Config) -> Summary {
    let result = transform(data, config.threshold);
    //                     ^^^^ consumed — compiler infers: own
    //                           ^^^^^^ read-only — compiler infers: ref
    result.summarize()
}
// Generated signature: fn process(data: own Data, config: ref Config) -> Summary
```

This is dataflow analysis. If the body moves/consumes the parameter, it's `own`. If it only reads, it's `ref`. If it mutates, it's `mut ref`. No annotation needed in the common case.

**Ambiguity rule:** If *any* code path consumes the parameter, it's `own`. Conservative but always correct:

```
fn maybe_take(data: Data, condition: bool) -> Option[Data] {
    if condition {
        Some(data)    // consumes data
    } else {
        None          // data dropped here — but must be owned to handle the other branch
    }
}
// Unambiguous: data must be owned
```

#### Part 2: Owned Returns by Default

Functions return owned values by default. This single rule eliminates most lifetime complexity:

```
fn longest(a: String, b: String) -> String {
    if a.len() > b.len() { a } else { b }
}
// Compiler infers: takes ownership of both, returns one.
// No lifetime annotation needed — returns are always owned by default.

fn first_word(s: String) -> String {
    s.split(' ').first().to_owned()
    // Returns an owned String — no lifetime relationship to s.
}
```

Sometimes this means a clone where Rust would return a borrow. The compiler reports this via performance notes:

```
perf[owned-return]: src/parser.kara:42:5
  `first_word` clones return value — input `s` is borrowed but return is owned.
  hint: add `ref` return annotation to avoid the clone (see Part 3).
```

#### Part 3: Explicit `ref` for Borrow Returns

When returning a borrow is performance-critical, the programmer opts in with `ref`:

```
// Single ref parameter — borrow source is unambiguous
fn first_word(s: ref String) -> ref String {
    s.split(' ').first()
}

// Multiple ref parameters — annotate the source
fn longer(a: ref String, b: ref String) -> ref(a, b) String {
    if a.len() > b.len() { a } else { b }
}

// Struct holding a borrow — ref on the field
struct Parser {
    source: ref String,    // this struct borrows from something
    position: i64,
}
```

`ref` replaces Rust's lifetime annotations. It's less powerful (can't express `'a: 'b` lifetime outlives relationships) but covers 95%+ of real-world cases. The remaining cases use RC.

**Key: `ref` is opt-in and rare.** Most functions return owned values. You only add `ref` when the perf note tells you a clone is expensive and you want to eliminate it.

#### Part 4: RC Fallback with Budget Controls

When the compiler can't determine ownership statically and `ref` isn't used, it falls back to RC. Budget controls prevent silent performance erosion:

```
// Module-level RC budget — compile error if exceeded
#[rc_budget(max: 5)]
mod order_processing;

// Per-function strict mode — no RC allowed
#[no_rc]
fn hot_loop(data: ref Vec[Entity]) -> f64 {
    // Compiler error if any value here needs RC
    data.iter().map(|e| e.position.magnitude()).sum()
}

// Project-level setting in build config
[ownership]
rc_budget = "warn_above_10_per_module"  // or "deny" for strict mode
```

Start permissive (RC as needed), tighten as the code matures.

#### Error Message Design

When inference fails or RC kicks in, the error teaches the model:

```
note[ownership-decision]: src/order.kara:42:9  `user`

  Decision: reference counted (Rc)

  Why: `user` is returned from two branches with different lifetimes.
    Line 45: returned from `if` branch (borrows from `db_result`)
    Line 48: returned from `else` branch (newly constructed)

  The compiler cannot prove a single owner at compile time.

  Options:
    1. Accept RC (~2ns/access overhead) — no code change needed
    2. Restructure to return owned value from both branches:
         let user = if condition {
             db_result.user.clone()  // explicit clone
         } else {
             User.new(...)
         };
    3. Add #[allow(rc_fallback)] to silence this note
```

#### What This Gives Up vs. Rust

- Can't express `'a: 'b` (lifetime outlives relationships) — use RC for these cases
- Can't return borrows from multiple unrelated inputs with different lifetimes — use RC or restructure
- Some unnecessary clones in naive code — but the perf system flags them

#### What This Gains Over Rust

- No lifetime annotations in 95%+ of code
- No borrow checker fights for beginners or AI
- Clear escalation path: owned → `ref` → RC, each explicitly chosen
- RC budget prevents the "everything is secretly RC" failure mode

**Who does this today?** Swift does ARC everywhere (no zero-cost path). Rust does full borrow checking (complex annotations). Lobster does compile-time RC optimization. Mojo has ownership with `borrowed` keyword. Nobody combines parameter mode inference + owned-return-by-default + opt-in `ref` + budgeted RC fallback.

### Feature 5: Auto-Concurrency via Effect Analysis — COMMITTED

**Status:** Committed. Derivative of Feature 2.

No `async/await`. No colored functions. No choosing between threads and async.

```
fn load_dashboard(user_id: u64) -> Dashboard {
    // The compiler sees these have non-conflicting effects
    // and runs them concurrently automatically
    let profile = fetch_profile(user_id);       // reads(UserDB)
    let orders = fetch_orders(user_id);         // reads(OrderDB)
    let notifications = fetch_notifs(user_id);  // reads(NotifDB)

    // This depends on all three — compiler inserts sync point
    build_dashboard(profile, orders, notifications)
}
```

No `async` keyword. No `.await`. No `join!`. No `Promise.all`. The compiler handles concurrency because it understands effects and data dependencies. The programmer writes sequential-looking code. The compiler generates concurrent execution.

This only works because of the effect system (Feature 2). Without effects, the compiler can't know that `fetch_profile` and `fetch_orders` don't conflict.

**Who does this today?** Nobody, really. This is the combination of algebraic effects + automatic parallelism that hasn't been realized in a practical language.

### Feature 6: Gradual Verification — DEFERRED (may or may not implement)

**Status:** Deferred to late-stage. Level 2 is achievable; Level 3-4 may never be built.

Types are just the first level. The language supports increasingly strong guarantees:

```
// Level 1: Basic types (always on — this is just the type system)
fn add(a: i64, b: i64) -> i64 { a + b }

// Level 2: Constrained types (opt-in, achievable without SMT solver)
type PositiveInt = i64 where value > 0;
type Percentage = f64 where 0.0 <= value <= 100.0;

// Level 3: Pre/post conditions (opt-in, requires SMT solver)
fn binary_search(list: Vec[i64], target: i64) -> Option[usize]
    requires list.is_sorted()
    ensures result.is_some() implies list[result.unwrap()] == target
{ ... }

// Level 4: Full formal verification (opt-in, requires SMT solver)
fn transfer(from: Account, to: Account, amount: Money)
    requires from.balance >= amount
    ensures from.balance == old(from.balance) - amount
    ensures to.balance == old(to.balance) + amount
    ensures from.balance + to.balance == old(from.balance) + old(to.balance)
{ ... }
```

#### Why Deferred

- **Level 2** (constrained types) is achievable without Z3 — implement as range checks at boundaries. `type Percentage = f64 where 0.0 <= value <= 100.0` is already differentiating and valuable. No mainstream compiled language offers this.
- **Level 3-4** requires integrating an SMT solver (Z3, ~1M lines of C++). This is the hardest feature to build — harder than effects, ownership inference, or LLVM codegen. SMT queries can time out on complex constraints, error messages are notoriously bad ("can't satisfy constraint" with no actionable guidance), and the programmer ends up debugging the theorem prover instead of their code.
- **Recommendation:** Build Level 2 first. It's useful, implementable, and doesn't require Z3. Evaluate Level 3-4 later based on whether Level 2 proves valuable in practice.

**Who does this today?** Dafny, F*, Liquid Haskell, Ada/SPARK. But they're all verification-first languages. Nobody offers this as a gradient.

### Feature 7: Compilation Target Flexibility — COMMITTED

**Status:** Committed.

The same language compiles to:

- **Native code** via LLVM (systems, CLI tools, game engines)
- **WebAssembly** (browser, edge computing)
- **GPU compute shaders** (data processing, ML inference)
- **FPGA bitstreams** (embedded, ultra-low-latency) — future goal

The data layout system (Feature 1) makes this natural. When targeting a GPU, `Vec[Entity]` with `group physics { position, velocity }` maps directly to a GPU buffer with the right memory layout for coalesced access.

## Type System and Object Model

**No classes. No inheritance. No OOP hierarchy.**

Kāra follows the Rust/Go approach — composition over inheritance:

- **Structs** — plain data containers, no methods attached
- **Enums** — algebraic data types with pattern matching
- **Traits** — interfaces that types can implement
- **Functions** — standalone, not owned by a type

This fits naturally with:
- **Effects** — effects annotate functions, not objects
- **Data layout** — the compiler needs full struct definitions to decide layout; inheritance hierarchies make this impossible because subclasses can add unknown fields

## Data Structures

Any general-purpose language needs these as standard library items (not language-level features, built on top of generics/enums/traits):

| Data structure | Kāra type | Purpose |
|---|---|---|
| Fixed-size array | `[T; N]` | Stack-allocated, known size at compile time |
| Dynamic array | `Vec[T]` | Heap-allocated, growable |
| Hash map | `HashMap[K, V]` | Key-value lookup |
| Hash set | `HashSet[T]` | Unique values, membership testing |
| String | `String` | Text |
| Tuple | `(A, B, C)` | Lightweight grouping |
| Option | `Option[T]` | Nullable values |
| Result | `Result[T, E]` | Error handling |

The `layout` feature (Feature 1) is orthogonal — it controls how a `Vec[T]` is physically arranged in memory, but only applies to homogeneous collections. HashMap, trees, and other pointer-based structures have layouts dictated by their algorithms.

### The Pitch

**Kāra is what you build AI on.**

The database AI queries. The runtime AI executes on. The network stack AI communicates through. The game engine AI controls characters in. The compiler that checks AI-generated code. All of it needs to be fast, safe, and trustworthy. Kāra is the language for that layer.

**The value of Kāra isn't "easier to write." It's "easier to trust code you didn't write."**

As AI generates more code, the problem shifts. Writing code gets cheaper. Understanding code gets harder. Every function in a million-line AI-generated codebase is code you didn't write. Kāra makes that code trustworthy without requiring you to read it:

- **Effect signatures are reviewable contracts.** `reads(UserDB), writes(OrderDB)` tells you what a function does to the world — without reading the implementation. Review the signature, trust the compiler to enforce it.
- **Ownership guarantees memory safety on code no human reviewed.** No use-after-free, no data races, no matter who (or what) wrote the code.
- **Exhaustive matching rejects incomplete logic at compile time.** The compiler catches missing cases before deployment, not at 3 AM in production.
- **Auto-concurrency from effect analysis** means the compiler extracts parallelism correctly — no hand-written async, no race conditions from manual thread management.

The compiler's job isn't just to optimize good code. **It's to reject bad code.** The more code AI generates, the more valuable a strict compiler becomes. Effects, ownership, exhaustive matching — these are rejection mechanisms that scale with codebase size, regardless of who authored it.

**In one sentence:** A language where the programmer declares *what* and *why*, and the compiler decides *how* and *where* — with the freedom to lay out memory, schedule concurrency, and target hardware without the programmer specifying any of it.

## What to Build First

**Effect types + auto-concurrency** (Features 2 + 5).

Rationale:
- No mainstream language has it.
- It solves the async/await colored-function problem that plagues every language.
- It makes the auto-parallelism story actually work (unlike the earlier fn/flow design which was too coarse-grained).
- It composes naturally with the rest of the features.
- The effect system can be built incrementally — start with a few built-in effects (reads, writes, sends) and expand over time.

---

## Previously Explored and Dropped Ideas

These ideas were explored in earlier design sessions and dropped after stress testing:

1. **Pure functions (fn/flow split):** Binary pure/impure distinction was too coarse-grained. The friction (no debugging prints, infectious refactoring, Plan/Execute verbosity) outweighed the benefits (speculative auto-parallelism, niche memoization). Effect types (Feature 2) subsume this with finer granularity.

2. **Context at boundaries (semantic types as core differentiator):** Through stress testing with 7+ examples, this was found to be essentially Rust newtypes / TypeScript branded types — a pattern that already exists in mainstream languages. Useful but not a language differentiator.

3. **Dataflow `->` operator:** Required purity guarantees (which were dropped) for auto-parallelism. Without effects, `->` is just a pipe operator with no semantic benefit over normal function calls. Awkward when mixed with loops and conditionals.

4. **Sandhi (boundary transformations):** Sanskrit-inspired automatic field mapping between record types at boundaries. Useful but marginal — it saves a few lines of field mapping code. The complexity of implicit transitive chains made debugging and readability concerns outweigh the benefit.

## Open Questions

### Effect System

1. **Effect inference scope:** ~~Should effects be inferred within a module and explicit at module boundaries? Or inferred everywhere? Fully inferred is more ergonomic but makes APIs unstable (adding a log statement changes the function's effect signature, breaking callers).~~

   **RESOLVED: Infer within modules, explicit at public boundaries.**

   - Effects are fully inferred for private/internal functions within a module. The compiler checks that inferred effects are compatible with the enclosing public function's declared effects.
   - Public function signatures must explicitly declare their effects. This provides API stability — adding a *transparent* log call inside a private helper doesn't silently change the module's public contract (transparent effects don't propagate). Adding a real effect like `reads(DB)` inside a private helper *will* propagate to the public boundary and require an explicit annotation update — surfaced as a compile error, never silently.
   - This matches the universal pattern of successful type systems (Rust infers types locally but requires explicit function signatures; Haskell convention annotates top-level definitions).
   - **Observability effects (logging, tracing, metrics) are transparent** — they don't propagate through the effect system. `transparent effect verb` is **the extension point for adding user-defined verbs beyond the six built-ins**, declaring new verbs that are *exempt from conflict analysis* — they never conflict with anything and never propagate to callers. The six built-in verbs (`reads, writes, sends, receives, allocates, panics`) are fixed because they drive parallelism and safety decisions; transparent verbs are for everything else (observability, instrumentation, auditing). Transparent verbs are purely documentary — the compiler tracks them but ignores them for scheduling:
     ```
     transparent effect verb traces;
     fn debug_log(msg: String) traces(Console) { ... }
     ```
     This prevents the "adding a log breaks callers" problem without weakening the effect system for effects that actually matter (reads, writes, sends).
   - Since AI writes most code (Starting Assumption #1), maintaining explicit effect annotations at boundaries is trivial for the primary code author.

2. **Effect polymorphism:** ~~Can a higher-order function be generic over effects? E.g., `fn map[F, E](list: Vec[T], f: F) -> Vec[U] where F: Fn(T) -> U + E` — the `map` function inherits whatever effects `f` has. Without this, higher-order functions become painful.~~

   **RESOLVED: Yes — use `with _` wildcard syntax for effect-polymorphic closures.**

   - Without effect polymorphism, higher-order functions like `map` would only accept pure closures, reintroducing the colored function problem the effect system is designed to eliminate.
   - Syntax: `Fn(T) -> U with _` means "accepts any effects and propagates them." The `_` wildcard signals effect polymorphism in the type signature:
     ```
     fn map[T, U](list: Vec[T], f: Fn(T) -> U with _) -> Vec[U]
     ```
   - Omitting `with _` means "requires pure." This distinction is intentional — `sort` genuinely needs a pure comparator, while `map` doesn't care:
     ```
     fn sort[T](list: Vec[T], cmp: Fn(T, T) -> Ordering) -> Vec[T]  // pure required
     fn map[T, U](list: Vec[T], f: Fn(T) -> U with _) -> Vec[U]     // any effects ok
     ```
   - Multiple closure arguments each carry their own effects; the compiler unions them:
     ```
     fn filter_map[T, U](list: Vec[T], pred: Fn(T) -> bool with _, transform: Fn(T) -> U with _) -> Vec[U]
     ```
   - The compiler infers concrete effects at each call site. `with _` is a type-level marker, not a runtime mechanism.
   - **Deferred: constrained effect polymorphism** (e.g., `with _ except writes` for a `parallel_map` that rejects write effects). Start with the simple binary — pure vs. `with _` — and evaluate whether constrained polymorphism is needed based on real usage.
   - **Note:** These closure forms (pure, `with _`, and concrete effect expression for stored closures) are later formalized and rationale-explained in G6, which resolves the ambiguity around stored vs. immediately-invoked closures across concurrency boundaries.

3. **Effect composition in practice:** ~~When function A calls function B which calls function C, effects bubble up. A large function deep in the call stack could accumulate many effects. Does the programmer have to declare all of them? Or does inference handle the propagation?~~

   **RESOLVED: Inferred internally, declared at public boundaries (follows from Q1). Named effect groups for ergonomics.**

   - Within a module, effects propagate automatically through the call chain. If private `validate` calls private `check_inventory` which `reads(InventoryDB)`, the compiler infers `validate` also `reads(InventoryDB)`. No annotation needed.
   - At public boundaries, the function declares its full effect set. A public function that touches six resources lists all six — this is honest documentation, not boilerplate:
     ```
     pub fn process_order(order: Order) -> Result[Receipt, Error]
         reads(UserDB, InventoryDB),
         writes(OrderDB),
         sends(PaymentGateway, NotificationService),
         reads(FileSystem)
     { ... }
     ```
   - If a public function's effect list looks ugly, that's a **design smell** — the function does too much. The effect system surfaces complexity that was always there.
   - **Named effect groups** provide ergonomic relief for functions that legitimately touch many resources:
     ```
     effect group OrderProcessing =
         reads(UserDB, InventoryDB),
         writes(OrderDB),
         sends(PaymentGateway, NotificationService);

     pub fn process_order(order: Order) -> Result[Receipt, Error]
         with OrderProcessing, reads(FileSystem)
     { ... }
     ```
     Effect groups are purely syntactic sugar — the compiler expands them. They reduce repetition when multiple public functions in a module share the same effect profile.
   - **Effect groups compose with `+`** for modular, refactorable effect declarations (see G5):
     ```
     effect group Validation = reads(UserDB, InventoryDB), sends(FraudService);
     effect group Fulfillment = writes(OrderDB), sends(PaymentGateway);
     effect group OrderProcessing = Validation + Fulfillment;
     ```
     Changes to sub-groups propagate automatically to composed groups. Libraries publish effect groups as API stability boundaries — callers using the group absorb individual effect changes without code updates.

4. **Conflicting effects at fine granularity:** ~~Two functions both `writes(UserDB)` — are they always unsafe to parallelize? What if they write to different rows? The resource granularity determines how much parallelism the compiler can extract vs. how much annotation burden the programmer bears.~~

   **RESOLVED: Resource-level granularity by default. Parameterized resources opt-in for finer partitioning.**

   - **Default:** `writes(UserDB)` + `writes(UserDB)` = conflict, serialized. Conservative and correct — the compiler never introduces a data race. For most code this is the right tradeoff; serializing two database writes costs little compared to I/O latency.
   - **Opt-in parameterized resources** unlock more parallelism when the programmer knows operations on different keys are independent:
     ```
     effect resource UserDB[user_id: u64];

     fn update_profile(id: u64) writes(UserDB[id]) { ... }
     fn update_settings(id: u64) writes(UserDB[id]) { ... }

     // Same user_id → conflict, serialized
     update_profile(42);
     update_settings(42);

     // Different user_ids → safe, parallelized
     update_profile(42);
     update_profile(99);
     ```
   - The parameter is a **compile-time partition key**. The compiler checks whether parameters are statically distinguishable at the call site. If values aren't statically known, the compiler conservatively treats them as the same resource.
   - **No finer than parameterized resources.** Field-level granularity (`writes(UserDB.email)`) leaks schema into effect signatures, is brittle, and most databases don't guarantee field-level isolation. Row-level with runtime checks adds overhead incompatible with a systems language. The 80/20 rule applies — resource-level covers most cases, parameterized resources cover the important rest. Note: field-level granularity (`writes(UserDB.email)`) is explicitly out of scope — the finest granularity is the parameterized resource.

5. **Effect handlers:** ~~Do we need them? Koka's power comes from effect handlers — the ability to intercept and reinterpret effects (e.g., swap a real database for an in-memory mock). Without handlers, effects are just annotations. With handlers, they become a control flow mechanism. This significantly increases implementation complexity.~~

   **RESOLVED: No full algebraic effect handlers. Use trait-based effect injection instead.**

   - Full Koka-style handlers require **delimited continuations** — the runtime must capture and resume continuations, which requires stack manipulation, complicates LLVM codegen, and is fundamentally at odds with zero-cost abstractions. This is the single hardest runtime feature to implement efficiently.
   - The three main use cases for handlers (testing, sandboxing, context switching) are all solved by a simpler mechanism — **effect resources are backed by swappable trait implementations**:
     ```
     effect resource UserDB;

     // The trait that backs the resource
     trait UserDBProvider {
         fn read_user(id: u64) -> User;
         fn write_user(user: User);
     }

     // Production implementation
     impl UserDBProvider for PostgresUserDB { ... }

     // Test implementation
     impl UserDBProvider for InMemoryUserDB { ... }

     // Swap the backing implementation at the call site
     with_provider[UserDB](InMemoryUserDB.new(), || {
         process_order(test_order);
     });
     ```
   - This is essentially dependency injection integrated with the effect system. The compiler enforces that all required resource providers are supplied. The pattern is familiar (Rust's `dyn Trait`, Go interfaces) and requires no runtime continuation magic.
   - **What you lose:** Koka's ability to suspend and resume computations mid-function — effect-based coroutines, automatic backtracking, nondeterminism handlers. These are research language features with very niche use cases. For a practical systems language, the tradeoff is overwhelmingly worth it.
   - **Effects remain a static analysis tool.** The compiler uses them for parallelism decisions. The programmer uses trait injection for runtime flexibility. No stack manipulation, no hidden control flow.

6. **What are the built-in I/O primitives?** ~~Effects need to bottom out somewhere. What are the leaf-level operations that have intrinsic effects? `read_file`, `write_file`, `send_tcp`, `receive_tcp`, `allocate_memory`, `get_time`? This defines the effect system's foundation.~~

   **RESOLVED: Six verbs, five primitive resources, plus `Heap` (transparent) and `panics` (resourceless).**

   Primitives map to syscalls, not abstractions. Databases, HTTP, and gRPC are user-defined resources built on top of these.

   | Verb | Primitive resource | Maps to |
   |---|---|---|
   | `reads` | `FileSystem` | `read`, `pread`, `stat`, `readdir` |
   | `writes` | `FileSystem` | `write`, `pwrite`, `fsync`, `mkdir`, `unlink` |
   | `reads` | `Clock` | `clock_gettime`, `gettimeofday` |
   | `reads` | `Env` | `getenv`, `getcwd`, command-line args |
   | `writes` | `Env` | `setenv`, `chdir` |
   | `sends` | `Network` | `send`, `sendto`, `connect` |
   | `receives` | `Network` | `recv`, `recvfrom`, `accept`, `listen` |
   | `reads` | `RandomSource` | `getrandom`, `/dev/urandom` |
   | `allocates` | `Heap` | `mmap`, `brk` (transparent by default) |
   | `panics` | *(no resource)* | `abort`, unwinding, `exit` |

   Design notes:
   - **`FileSystem` is one resource at the primitive level.** User code creates finer-grained resources on top (`effect resource ConfigFiles; effect resource LogFiles;`). The trait-based injection from Q5 connects these to filesystem primitives.
   - **`Clock` and `RandomSource` are read effects** because they're non-deterministic — calling them twice gives different results. Making them explicit effects lets tests inject a fake clock or seeded RNG for reproducibility.
   - **`Heap` allocation is transparent by default.** Otherwise every function that creates a `Vec` would carry `allocates(Heap)`. `allocates` is a *documentation and gating* verb, not a concurrency verb — it never conflicts (even with itself), so it plays no role in parallelism decisions. Its purposes are: (1) making heap allocation visible in function signatures where opted into, (2) enabling `#[no_effect(allocates(Heap))]` for GPU/embedded targets where allocation is genuinely forbidden (Q23), and (3) `Heap` stays transparent for ordinary code so it doesn't pollute every function that touches a `Vec`. At GPU/embedded boundaries, enforcement uses two committed v1 mechanisms: (a) *structural* — the `GpuSafe` type check rejects heap-allocated types so any function using them fails compilation; (b) *project profiles* — `no_effects = ["allocates(Heap)"]` in `kara.toml` applies a project-wide gate for every public function in that profile. Transparency is a CPU-code concern; GPU/embedded contexts opt in to hard enforcement via profiles.
   - **`panics` has no resource** — it's a control flow effect. A function that can panic carries `panics` so callers know. This composes with error handling (Q19).
   - **FFI is where primitives originate.** Every primitive ultimately calls into C/OS code via FFI. The FFI boundary is where effect annotations are mandatory — the programmer declares what effects a foreign function has, and the compiler trusts that declaration. This is the "unsafe" boundary of the effect system.
   - **Not primitives:** `Database`, `HTTP`, `gRPC` (user-defined on `Network`), `Console`/`Stdout` (either `writes(FileSystem)` on fd 1/2 or transparent `traces`), `Thread`/`Process` (concurrency managed by compiler per Feature 5).

### Ownership and Memory

7. **RC fallback granularity:** ~~When the compiler falls back to reference counting, is it per-value or per-type? If a single value in a function needs RC, does everything in that scope get RC? What's the blast radius of a single inference failure?~~

   **RESOLVED: Per-value, always. No infection.**

   - Only the specific value with ambiguous ownership gets RC. Adjacent values, same-type values, and same-scope values are unaffected. This is the only option consistent with "compiler extracts maximum performance" (Starting Assumption #2).
   - **RC is invisible in the type system.** The programmer always sees `T`, never `Rc[T]`. The compiler decides per-value whether storage is inline (owned), borrowed (reference), or reference-counted — this is a codegen decision, not a type-level one:
     ```
     fn example(condition: bool) -> String {
         let a = "hello".to_string();  // owned, no RC
         let b = "world".to_string();  // owned, no RC

         // Compiler can't prove which branch executes, so the
         // return value needs RC. But a and b themselves stay owned.
         if condition { a } else { b }
     }
     ```
   - **Transitive through reference graphs, not lexical scope.** If value A holds a reference to value B and A needs RC, B's lifetime must also be extended (RC or lifetime-pinned). But value C in the same function with no relationship to A stays zero-cost.
   - This matches how Swift's ARC works — proven at scale, per-value, no infection.

8. **Performance notes UX:** ~~The compiler emits "performance notes" when it falls back to RC or inserts unexpected clones. What does this look like in practice? Warnings? A separate report? How does the AI (or human) act on them?~~

   **RESOLVED: Three-tier structured diagnostics, designed for machine consumption first.**

   Since AI writes most code (Starting Assumption #1), the primary consumer is a machine, not a human reading terminal output.

   - **Tier 1: Inline notes (always emitted, never block compilation).** Structured diagnostics attached to source locations with category, location, explanation, and actionable hint:
     ```
     perf[rc-fallback]: src/order.kara:42:9
       `user` uses reference counting — returned from two branches with different lifetimes.
       hint: restructure to return owned value from a single path, or accept the ~2ns/access overhead.

     perf[implicit-clone]: src/order.kara:58:22
       `order_items` cloned when passed to `validate` — original used again on line 63.
       hint: pass by reference if `validate` doesn't need ownership.
     ```
     Machine-parseable fields (category, location, hint) let an AI code reviewer act on these programmatically. Never errors — the code compiles and runs correctly.

   - **Tier 2: Summary report (opt-in, `karac build --perf-report`).** Per-module summary for performance review passes:
     ```
     Performance report for src/order.kara:
       RC fallbacks:     2 values (user, session)
       Implicit clones:  3 (order_items, config, user_name)
       Estimated overhead: ~14ns per call to process_order()
     ```

   - **Tier 3: Suppression annotations** for accepted overhead:
     ```
     #[allow(rc_fallback)]
     fn example(condition: bool) -> String { ... }
     ```
     Silences the note for that specific site. Prevents warning fatigue on code where RC is intentional.

   - **Why not warnings?** Warnings imply "you should fix this." RC fallback isn't broken — it's a valid compiler strategy. Warning fatigue is real. Notes are informational; the programmer (or AI) decides whether to optimize.

9. **Cycles in RC:** ~~Reference counting can't handle cycles (A references B, B references A). Rust avoids this by preventing cycles through the borrow checker. If Kāra uses RC as a fallback, how are cycles detected or prevented? Weak references? Cycle collector (adds runtime overhead)?~~

   **RESOLVED: Compile-time cycle detection + explicit `weak` annotation. `shared struct` for ergonomic graph/tree types. Arena as stdlib primitive for performance-critical paths. No runtime cycle collector.**

   - The compiler analyzes the type graph at compile time and **rejects types that form ownership cycles**:
     ```
     struct Parent { children: Vec[Child] }
     struct Child { parent: Parent }  // ERROR: Parent → Child → Parent cycle
     ```
   - To break a cycle, the programmer explicitly marks the back-edge with `weak`:
     ```
     struct Child { parent: weak Parent }  // OK: weak breaks the cycle
     ```
     A `weak` reference may become `None` if the referent is deallocated. Accessing it returns `Option[T]`, forcing the caller to handle the possibility. Matches Swift's `weak` semantics exactly. **Assigning a strong reference where a `weak` type is expected automatically creates a weak reference — no explicit conversion needed.** An explicit `.upgrade()` method is also available to convert a `weak` field to `Option[T]` on demand.
   - **`shared struct` and `shared enum` for trees, graphs, and linked lists** — types that are inherently shared:
     ```
     shared struct TreeNode {
         val: i64,
         left: Option[TreeNode],
         right: Option[TreeNode],
     }

     shared struct GraphNode {
         id: u64,
         neighbors: Vec[weak GraphNode],
     }

     // shared enum for recursive types like JSON or ASTs
     shared enum Json {
         Null,
         Bool(bool),
         Number(f64),
         Str(String),
         Array(Vec[Json]),
         Object(HashMap[String, Json]),
     }
     ```
     `shared struct` and `shared enum` opt types into reference semantics with automatic RC. Assignment shares (RC increment) instead of moving. **`shared struct` fields are immutable by default; declare `mut` on each field that needs to be reassigned after construction.** This matches the language's general immutability-by-default principle (local `let` vs `let mut`) and is safer for shared-reference semantics: anyone reading the struct definition sees immediately which fields can change. The primary use case — trees and graphs — only needs `mut` on the structural link fields (`left`, `right`, `neighbors`); payload fields (`val`, `id`) stay immutable. Attempting to assign to a non-`mut` field is a compile error. Methods may mutate `mut` fields without additional annotation; effect annotations are not required for field mutation. `shared enum` carries the same RC semantics and is useful for recursive types where the value is inherently shared (JSON trees, AST nodes, expression DAGs). This makes trees/graphs feel as natural as any other struct, without requiring Arena boilerplate. Inspired by Swift's `class` vs `struct` distinction — Kāra uses `shared struct`/`shared enum` to avoid OOP connotations.
   - **Arena as stdlib primitive, not language feature.** `Arena[T]` + `ArenaRef[T]` remain available for performance-critical bulk allocation (cache-friendly, zero RC overhead per node), but as a standard library type, not a language keyword. No mainstream language has arena as a language feature — Go tried it (1.20), then removed it (1.23). Arena produces identical performance whether it's a language feature or stdlib.
   - **No runtime cycle collector.** Zero overhead, deterministic deallocation. Cycles are rare in practice — mostly parent-child back-references and observer patterns. Making the programmer annotate them with `weak` is consistent with the ownership philosophy: two things can't own each other.

### Auto-Concurrency

10. **Concurrency runtime:** ~~Green threads (Go-style), OS threads, or a hybrid? Green threads need a runtime scheduler (adds binary size, complicates FFI). OS threads are heavyweight but simple. A work-stealing scheduler (like Tokio) is efficient but complex to build.~~

    **RESOLVED: Work-stealing task scheduler, compiler-managed, zero-cost when unused.**

    - **Work-stealing over green threads** because: green threads require a mandatory runtime (~10MB for Go), break C FFI assumptions (stack switching), and can't target GPU/WASM. A work-stealing scheduler adds runtime only when the program actually uses concurrency.
    - **Compiler generates tasks from effect analysis**, not manually by the programmer. The compiler decides granularity — don't spawn a task for a 10ns pure function, do spawn one for a 10ms database query. The effect system provides enough information to make this decision.
    - **Implementation:** Fixed-size OS thread pool (default = CPU cores, configurable via runtime config). Per-thread work queues with work stealing. Tasks are lightweight: function pointer + captured state (~64 bytes), no stack allocation. I/O operations use OS-level async (epoll/kqueue/io_uring) under the hood. Language-level concurrency annotation uses `#[concurrency(max_tasks: N)]` — thread pool sizing is a runtime concern, not a language annotation (see G2).
    - **Zero-cost when unused.** If a program has no concurrent effects (pure computation), the scheduler is never initialized. No binary size or startup cost for programs that don't need concurrency.
    - **FFI compatible.** Tasks run on real OS thread stacks, so C FFI works normally. No stack switching, no segmented stacks.
    - **Graceful degradation.** Falls back to sequential execution on targets without thread support (WASM without SharedArrayBuffer, single-core embedded). The programmer writes the same code; only the execution strategy changes.

11. **Determinism:** ~~If the compiler auto-parallelizes, does the program produce the same output on every run? Non-deterministic scheduling can cause heisenbugs. Should there be a "deterministic mode" for debugging?~~

    **RESOLVED: Deterministic by construction. No special mode needed.**

    - The effect system already guarantees determinism for correct programs. The compiler only parallelizes operations with **non-conflicting effects** — if two operations don't read/write the same resource, their execution order doesn't affect the result. That's the whole point.
    - **Timing is the only source of non-determinism.** If `fetch_profile` returns in 5ms on one run and 50ms on another, the *values* are identical but the *wall-clock timing* differs. This only matters if the program observes time (`reads(Clock)`), which is an explicit effect that the compiler accounts for.
    - **Error ordering is non-deterministic** — if two concurrent operations both fail, which error surfaces first depends on scheduling. This is an error semantics question (Q12), not a determinism-of-correct-execution question.
    - **Debug support via compiler flags, not language constructs:**
      - `karac build --sequential` — disable all auto-parallelism, run everything in source order. For stepping through with a debugger.
      - `karac build --concurrency-report` — show what the compiler parallelized, for transparency into the execution model.

12. **Error handling in concurrent effects:** ~~If `fetch_profile` and `fetch_orders` run concurrently and `fetch_orders` fails, what happens to `fetch_profile`? Is it cancelled? Does it run to completion and the result is discarded? How does error propagation work across concurrent branches?~~

    **RESOLVED: Cancel sibling tasks on first error (structured concurrency). Return the first error. Opt-in `collect_all` for gather-all-errors.**

    - **Default: cancel on first error.** When one branch of a compiler-generated concurrent fork fails, all sibling branches are cancelled. If `fetch_orders` fails while `fetch_profile` is in-flight, `fetch_profile` is cancelled. If it already completed, its result is dropped. The function returns the first error.
    - **Why cancel, not "let them finish"?**
      - Resource efficiency: if all results are needed to proceed, completing the others after one fails wastes I/O and CPU.
      - Predictable behavior: the programmer wrote sequential-looking code and expects sequential-looking error semantics. Concurrency is invisible, so error behavior should match the sequential mental model.
    - **No rollback.** Cancellation stops future work, it doesn't undo past side effects. If a cancelled task already wrote to a database, that write persists. Rollback is a transaction concern:
      ```
      fn transfer(from: Account, to: Account, amount: Money)
          writes(AccountDB) {
          transaction(AccountDB, || {
              debit(from, amount)?;
              credit(to, amount)?;
              // If credit fails, debit is rolled back by the transaction
          })
      }
      ```
    - **Opt-in `collect_all` for gather-all-errors** (e.g., validating multiple fields):
      ```
      let results = collect_all(
          || fetch_profile(user_id),
          || fetch_orders(user_id),
          || fetch_notifs(user_id),
      );
      // results: (Result[Profile], Result[Orders], Result[Notifs])
      // All three run to completion regardless of failures
      ```
    - **Error ordering:** When multiple concurrent branches fail, the error from the source-order-first operation is returned. This makes error reporting deterministic even though execution order isn't.

13. **Backpressure and resource limits:** ~~If the compiler aggressively parallelizes, it could open 1000 database connections simultaneously. Is there a way to limit concurrency per resource? E.g., `effect resource UserDB { max_concurrent: 10 }`?~~

    **RESOLVED: Concurrency limits belong on the provider (runtime), not the effect declaration (compile-time). No new language syntax needed.**

    - **Limits are a deployment concern, not a semantic one.** The effect declaration says *what* a function does (`reads(UserDB)`). The provider (from Q5) says *how*. Different environments have different limits (dev: 2 connections, staging: 10, production: 100). Baking limits into the effect declaration makes them compile-time constants when they should be runtime configuration.
    - **Providers implement backpressure internally** via connection pools / semaphores:
      ```
      let db = PostgresUserDB.new(ConnectionPool.new(max_connections: 10));

      with_provider[UserDB](db, || {
          // Provider's internal semaphore limits concurrent access to 10,
          // regardless of how many tasks the compiler spawns
          process_all_orders(orders);
      });
      ```
    - **The compiler uses heuristics for batching.** If it sees 1000 loop iterations each calling `reads(UserDB)`, it doesn't naively spawn 1000 tasks. It batches into chunks sized to the thread pool (e.g., `num_cores * 2`). Effect types (I/O vs. compute) inform batching strategy.
    - **Blocked tasks yield naturally.** The work-stealing scheduler (Q10) handles provider-level blocking — a task waiting for a connection yields its thread, which picks up other work. No deadlock risk from exhausting the thread pool.

### Data Layout

14. **Default layout:** ~~When no `layout` block is specified, what's the default? AoS (like every other language) is the safe default. But should the compiler auto-detect opportunities for SoA based on access pattern analysis?~~

    **RESOLVED: AoS default. No compiler auto-detection. Performance notes suggest SoA opportunities.**

    - **AoS is the only sane default.** Every other language uses it. It's cache-friendly when you access most fields of each element (the common case). Changing the default would make every tutorial and AI training dataset produce code with unexpected memory layouts.
    - **No compiler auto-detection of SoA.** Sounds appealing but has fatal problems:
      - Compile-time analysis can't see *runtime* access patterns — a function that reads `position` and `name` looks like it needs both, even if 99% of calls only touch `position`.
      - Layout changes are observable — FFI breaks, serialization changes, memory-mapped I/O produces different results. Layout can't be a silent optimization.
      - Non-deterministic builds — adding a new function that reads a field could change layout of collections used elsewhere.
    - **Performance notes fill the gap** (same tier-1 diagnostic system as Q8):
      ```
      perf[layout-opportunity]: src/physics.kara:12
        `entities` iterated 3 times — each iteration accesses only {position, velocity}.
        hint: consider adding a layout block with `group physics { position, velocity }`.
      ```
      The AI or programmer evaluates and adds a `layout` block if it makes sense. The decision is explicit and intentional.

15. **Layout and generics:** ~~If `Vec[T]` can have custom layouts, what happens with generic functions that operate on `Vec[T]` without knowing T's layout? Does the compiler monomorphize for each layout variant?~~

    **RESOLVED: Monomorphize per layout variant. Layout is part of the concrete type.**

    - The compiler monomorphizes generic functions for each concrete layout — just like Rust monomorphizes generics for each concrete type. A generic `sum_positions[T: HasPosition](entities: Vec[T])` generates different code for AoS vs. SoA layouts. Zero runtime dispatch.
    - **Layout doesn't appear in generic constraints.** You don't write `where T: SoALayout`. Layout is an attribute of the collection, not the element type. Generic functions are layout-agnostic in source; the compiler handles it during monomorphization.
    - Two `Vec[Entity]` with different layouts are different types at the monomorphization level — same as how `Vec[u32]` and `Vec[String]` are different types in Rust.

16. **Layout migration:** ~~If you change a `layout` block, does all code that accesses the collection need to be recompiled? Is layout part of the API boundary or an implementation detail?~~

    **RESOLVED: Layout is an implementation detail, not part of the public API. Changing layout triggers recompilation of the defining module and direct consumers only.**

    - **Layout is module-private.** Code outside the module accesses collections through the logical struct type (`Entity`), never through the physical layout. Public functions return logical types, not layout-specific representations.
    - **Recompilation scope is standard.** Changing a `layout` block requires recompiling the defining module and any modules that directly iterate the collection (because their generated access code changes). This is normal incremental compilation behavior — no different from changing a function body.
    - **No ABI impact on public APIs.** A function that returns `Entity` returns a single struct regardless of whether the collection internally uses AoS or SoA. Layout only matters during iteration over the collection.

### Type System and Language Design

17. **Methods vs. standalone functions:** ~~Should Kāra have `impl` blocks (Rust-style `user.save()`) or pure standalone functions (`save(user)`)? Methods enable chaining (`user.validate().save().notify()`). Standalone functions are simpler and more consistent with the effect system.~~

    **RESOLVED: `impl` blocks with Rust-style method syntax. UFCS (Uniform Function Call Syntax) as the bridge.**

    - **Methods are too ergonomic to give up.** Chaining is essential for readable data transformation: `users.filter(|u| u.is_active).map(|u| u.name).sort()` vs. the unreadable nested `sort(map(filter(...)))`.
    - **`impl` blocks work with effects.** Effects annotate methods the same way as standalone functions: `fn save(self) -> Result[(), Error] writes(UserDB) { ... }`. No conflict.
    - **UFCS for flexibility.** Both `user.validate()` and `User.validate(user)` call the same function. Standalone functions with a type as their first argument can also be called with method syntax. No need to choose — you get both.
    - **No inheritance, so methods are simple.** Methods in `impl` blocks are just syntactic sugar for functions with `self` as the first argument. No vtables, no dynamic dispatch (unless through trait objects).

18. **Generics implementation:** ~~Monomorphization (Rust-style, zero cost, slow compilation, large binaries) vs. boxing (Go-style, faster compilation, runtime overhead)? Monomorphization fits the "maximum performance" assumption but may hurt compilation speed.~~

    **RESOLVED: Monomorphization.**

    - Already implicitly committed — Q15 resolved layout generics via monomorphization, and Starting Assumption #2 says "maximum performance." Monomorphization is the only option consistent with zero-cost generics.
    - Compilation speed is a tooling problem (incremental compilation, caching), not a language design problem. Rust's slow compilation is partly due to LLVM's optimizer, not monomorphization itself.

19. **Error handling:** ~~`Result[T, E]` with `?` operator (Rust-style)? Exceptions? Something else? This needs to compose well with effects — an error is arguably an effect (`panics` or `throws`).~~

    **RESOLVED: `Result[T, E]` with `?` operator. No exceptions. `panics` for unrecoverable errors only.**

    - Rust got this right. Errors are values, not control flow. The `?` operator provides ergonomic propagation. The return type tells you if a function can fail — no hidden control flow.
    - **`panics` is for truly unrecoverable situations** (out of memory, assertion failures, index out of bounds). It's an effect (Q6) so callers know. `Result` is for expected failure modes (file not found, invalid input, network timeout).
    - **No `throws` effect.** The return type `Result[T, E]` already communicates fallibility. Adding a `throws` effect would be redundant — you'd have the error channel in both the return type and the effect signature.

20. **Module system:** ~~How are effect resources scoped? If two libraries both define `effect resource Database`, are they the same resource or different? This is a module system / namespace question with concurrency implications.~~

    **RESOLVED: Fully qualified paths. Same name in different modules = different resources.**

    - Effect resources are namespaced by their module path, just like types. `myapp.db.Database` and `otherlib.db.Database` are distinct resources. The compiler treats them as non-conflicting (different resources = safe to parallelize).
    - If they're actually the same underlying database, the programmer connects them through the provider (Q5) — both map to the same `PostgresDB` instance at runtime.

21. **FFI story:** ~~How does Kāra interoperate with C/Rust/Python? FFI functions are where effects originate — they're the leaf nodes of the effect system. The FFI boundary design is critical.~~

    **RESOLVED: C-compatible ABI at the boundary. Effect annotations mandatory on `extern` functions.**

    - FFI functions are where effects originate. The programmer declares what a foreign function does, and the compiler trusts that declaration — this is the "unsafe" boundary of the effect system:
      ```
      extern "C" fn write(fd: i32, buf: *const u8, count: usize) -> isize
          writes(FileSystem);
      ```
    - Start with C FFI (most universal). Rust interop via C ABI. Python interop via C extension API or WASM.
    - The standard library wraps these raw `extern` declarations in safe Kāra functions with proper types and error handling.

22. **Closures and effects:** ~~If a closure captures a variable and is passed to another function, what effects does the closure carry? Are captured variables' effects part of the closure's type?~~

    **RESOLVED: Closures carry the effects of operations they perform when called, not captured variables.**

    - Capturing a variable is an ownership operation (handled by Q4/Q7), not an effect. The closure's effects come from what it *does* when called.
    - Already addressed in Q2: `Fn(T) -> U with _` expresses that a closure can carry any effects. `Fn(T) -> U` (no `with _`) requires the closure body to be pure. The captured variables don't change this — only the operations inside the closure body matter.

### Compilation Targets

23. **GPU compilation scope:** ~~Which subset of Kāra is valid for GPU targets? No heap allocation, no recursion, limited control flow. Is this enforced by effects (e.g., GPU code can't have `allocates` effect) or by a separate compilation mode?~~

    **RESOLVED: Enforced by effects. No separate compilation mode.**

    - GPU code can't have `allocates(Heap)` — that's why `allocates` exists as a non-transparent verb (Q6). The compiler checks at the target boundary: if a function compiled for GPU has `allocates`, `panics`, or recursive calls, it's a compile error.
    - No separate compilation mode needed. The effect system already encodes the restrictions. A function that's pure + no allocation + no recursion is GPU-eligible. The compiler tells you if it isn't.
    - The `layout` feature (Feature 1) maps naturally to GPU buffers — `group physics { position, velocity }` becomes a coalesced-access GPU buffer.

24. **WASM limitations:** ~~WASM doesn't have threads (yet, outside of SharedArrayBuffer). How does auto-concurrency degrade on WASM targets? Does the compiler fall back to sequential execution?~~

    **RESOLVED: Sequential fallback. Already answered in Q10.**

    - Q10 resolved this: "Falls back to sequential execution on targets without thread support (WASM without SharedArrayBuffer, single-core embedded)."
    - The programmer writes the same code; the compiler generates sequential execution for WASM. When WASM threads mature, the compiler generates concurrent code for that target without source changes.

### Practical Concerns

25. **Debugging:** ~~How do you debug auto-parallelized code? If the compiler reorders operations, breakpoints and step-through debugging become confusing. Is there a "sequential mode" for debugging?~~

    **RESOLVED: Already answered in Q11.**

    - `karac build --sequential` disables all auto-parallelism. Every operation runs in source order. Breakpoints and step-through debugging work normally.
    - `karac build --concurrency-report` shows what the compiler parallelized, for understanding the execution model without disabling it.

26. **Ecosystem bootstrapping:** ~~A language is only as good as its libraries. What's the plan for building an ecosystem? Start with a strong standard library? FFI bridge to Rust crates?~~

    **RESOLVED: FFI bridge to C/Rust libraries first. Strong standard library built iteratively.**

    - The C FFI (Q21) is the immediate path to a usable ecosystem — wrap existing C libraries (libc, OpenSSL, SQLite) with Kāra types and effect annotations. A Rust-crate bridge via C ABI follows.
    - Build the standard library iteratively: start with what's needed for self-hosting (Q27), expand based on real usage. Don't try to compete with mature ecosystems on day one.
    - The effect system makes wrapping external libraries safe — every FFI function declares its effects, so the compiler can reason about parallelism even across FFI boundaries.

27. **Self-hosting feasibility:** ~~The north star is rewriting the Kāra compiler in Kāra. This requires Kāra to support: file I/O, string processing, data structures (trees, hash maps), error handling, command-line argument parsing, and recursive data types. Are all of these covered by the current feature set?~~

    **RESOLVED: Yes. The current feature set covers all self-hosting requirements.**

    - File I/O: effects + `FileSystem` resource (Q6)
    - String processing: `String` type + `impl` methods (Q17)
    - Data structures: generics (Q18) + standard library `HashMap`, `Vec`, trees
    - Error handling: `Result[T, E]` + `?` operator (Q19)
    - CLI argument parsing: `reads(Env)` (Q6)
    - Recursive data types: enums (Feature 3) + `shared struct` for trees/graphs (Q9)
    - The only gap is build system integration (dependency management, compilation orchestration), which is a tooling concern, not a language feature.

### Additional Design Gaps (identified via external review)

28. **Unsafe escape hatch:** Every systems language needs raw access to memory, unchecked operations, and inline assembly. The doc mentions FFI as the "unsafe boundary" but doesn't formalize an escape hatch for low-level operations within Kāra code itself.

    **RESOLVED: Add `unsafe` blocks, following Rust's model.**

    - An `unsafe` block allows: raw pointer arithmetic, unchecked casts, bypassing the ownership system, and inline assembly:
      ```
      unsafe {
          let ptr = &value as *const T;
          let raw = ptr.offset(4);
          let val = *raw;  // unchecked dereference
      }
      ```
    - **Effects declared inside `unsafe` are trusted but unverified.** The compiler takes the programmer's word for what effects an unsafe block has. This is the same contract as FFI (Q21) — the programmer asserts correctness, the compiler can't check it.
    - **`unsafe` does not disable effects.** You still declare effects on the enclosing function. What `unsafe` disables is ownership checking, bounds checking, and type safety within the block. The effect system remains sound — it just trusts the programmer's annotations inside unsafe.
    - **Use cases:** Custom allocators, OS interfaces, MMIO for embedded targets, performance-critical inner loops, and FFI wrapper implementations. Without this, Kāra cannot be a systems language.
    - **Minimal surface area.** Like Rust, the goal is to make `unsafe` rare and auditable. The vast majority of code never needs it. But when it's needed, there's no workaround.

29. **Async I/O mechanism:** Q10 committed to a work-stealing scheduler but didn't specify whether I/O is blocking-on-threads or compiler-transformed state machines.

    **RESOLVED: Blocking I/O on OS threads for v1. State machine transform is a future optimization. Language surface is execution-model-agnostic (see G2).**

    - **Blocking I/O on real OS threads.** When a task calls `read_file()` or `recv()`, it blocks the OS thread. The work-stealing scheduler compensates — other tasks run on other threads. This is Go's model (goroutines on OS threads via `GOMAXPROCS`), proven to scale to ~10K concurrent connections.
    - **Why not state machines (Rust async)?** The state machine transform (CPS/generator lowering) is the single hardest compiler feature in Rust. It requires: splitting functions at every await point into enum variants, transforming local variables into struct fields, handling drop semantics across yield points, and generating correct `Pin` semantics. This is an order of magnitude more implementation effort than blocking I/O, with marginal benefit for most workloads.
    - **Why blocking works for Kāra:** The compiler controls task spawning (Feature 5). It knows from effect analysis which operations are I/O-bound. It can size the thread pool accordingly — if 8 tasks are all `reads(Network)`, use 8+ threads so blocking doesn't stall the program. This adaptive sizing is cheap.
    - **Scaling limits are acceptable for v1.** Blocking I/O limits concurrency to ~10K connections (one thread per blocked I/O). For the 99% of programs that don't need 100K concurrent connections, this is fine. For the 1% that do (web servers at scale), state machine transform can be added later — the language surface stays the same.
    - **FFI advantage.** Blocking I/O means tasks run on normal OS thread stacks. C FFI works without special handling. Future runtime models preserve this: FFI calls always run on real OS threads regardless of execution strategy (see G2).
    - **Cancellation.** Cooperative cancellation via compiler-inserted checks at effect boundaries (see G2). No `pthread_cancel` (unsafe — can leave mutexes locked). No fd closing for cancellation (violates structured concurrency). Cancelled tasks return a `Cancelled` error variant, propagated via `?`.
    - **Execution-model agnostic surface.** The language API exposes tasks, not threads. Concurrency annotations use `max_tasks`, not `max_threads`. Thread pool sizing is runtime config. This ensures code written for v1 runs unchanged on future runtimes (state machines, io_uring) without language-level changes.

30. **Resource aliasing for cross-library safety:** Q20 resolved that same-name resources in different modules are different resources. But if two libraries independently define resources that map to the same underlying system (e.g., both wrap the same Postgres instance), the compiler will parallelize them — introducing a data race it can't see.

    **RESOLVED: Provider-rooted resources with three layers of defense (see G3).**

    - Resources are typed by their provider trait: `effect resource UserDB: DatabaseProvider;`. Two resources backed by the same provider trait type are the same *kind* of resource, though still logically distinct.
    - **Layer 1 — Same-instance detection (automatic).** When the same provider instance is passed to two `with_provider` calls, the compiler detects it and treats those resources as aliased. No annotation needed. This catches the most common case.
    - **Layer 2 — Explicit `alias` (manual fallback).** When two libraries internally create separate provider instances that happen to hit the same system, the compiler can't detect it. The programmer declares the relationship:
      ```
      alias mylib.UserDB = theirlib.TheirDB;
      ```
      After aliasing, the compiler treats `writes(mylib.UserDB)` and `writes(theirlib.TheirDB)` as conflicting — they serialize correctly.
    - **Layer 3 — `--strict-effects` mode (opt-in).** Refuses to auto-parallelize across module boundaries unless resource relationships are explicitly declared with `alias` (same system) or `independent` (different systems). Falls back to sequential execution for undeclared pairs. For high-assurance code.
    - **Aliasing is explicit and auditable.** The programmer declares the relationship, not the compiler. This is consistent with the effect system philosophy — the programmer provides semantic knowledge, the compiler enforces it.
    - **The soundness gap is minimized.** Same-instance detection handles the common case automatically. `--strict-effects` closes the remaining gap entirely for code that needs it. The gap only exists for cross-library-internal-connection scenarios in default mode.

31. **Rc vs Arc decision rule:** Q7 resolved that RC fallback is per-value. But `Rc` (non-atomic reference counting) is not thread-safe — if a value crosses a concurrency boundary, it needs `Arc` (atomic reference counting, ~10x slower). Since concurrency is compiler-managed and invisible to the programmer, the compiler must choose correctly.

    **RESOLVED: The compiler uses Arc when a value crosses a concurrency boundary, Rc otherwise.**

    - The compiler already knows concurrency boundaries — it generates them from effect analysis (Feature 5). When a value is captured by a concurrent task, the compiler uses `Arc`. When a value stays within a single task, it uses `Rc`. The programmer never sees either type.
    - **The effect system provides the information.** A value captured by `fetch_profile` (which the compiler runs concurrently) crosses a concurrency boundary. A value used only within a single function doesn't. The compiler has this information from its concurrency analysis.
    - **Performance notes (Q8) report the choice.** If a value gets `Arc` and the programmer thinks it shouldn't, the performance note explains why:
      ```
      perf[arc-upgrade]: src/order.kara:42:9
        `config` is shared across a concurrency boundary — compiler uses atomic reference
        counting (~10x overhead vs single-threaded RC).
        hint: clone `config` before the concurrent section to give each task its own
        owned copy, eliminating the atomic overhead entirely.
      ```
    - **Conservative default: Arc when uncertain.** If the compiler can't prove a value stays within one thread (e.g., it's stored in a data structure that might be shared), it uses `Arc`. This is safe but potentially slower. The performance note system makes this visible.

## Deferred Design Items

Items explicitly evaluated during the design phase and postponed. These are **not rejected ideas** — they're good features that don't need to be designed before implementation begins. They either depend on implementation experience, are additive (don't change core semantics), or belong to the standard library / tooling layer rather than the language itself.

### D1: Compile-Time Computation (Comptime / Const Generics)

**What:** Zig-style compile-time function evaluation. Functions marked `comptime` execute during compilation, producing constants, lookup tables, specialized code, and layout decisions. Const generics (`fn matrix_mul[const N: usize]`) enable zero-cost linear algebra.

**Why it matters:**
- Eliminates the need for a separate macro/metaprogramming system (Zig's key insight)
- Layout decisions, buffer sizes, serialization code all benefit from comptime
- Const generics are essential for zero-cost linear algebra (GPU/ML targets)
- Comptime functions are trivially pure — composes naturally with the effect system

**Why deferred:**
- Strictly additive. Comptime doesn't change the semantics of any existing feature — it adds a new evaluation phase.
- The core compiler must exist before you can evaluate functions at compile time. Designing comptime now would be speculative about compiler internals that don't exist yet.
- Const generics are a simpler subset that can come with the generics implementation (Q18) without full comptime.

**Revisit when:** The core compiler can parse, type-check, and generate code for basic programs. At that point, the compiler's IR is stable enough to add a comptime evaluation pass.

**Risk of deferring:** May need a temporary macro system for code generation during bootstrap. If so, design it as minimal and explicitly temporary.

### D2: Interior Mutability

**What:** Equivalents to Rust's `Cell<T>`, `RefCell<T>`, and `Mutex<T>` — mechanisms for mutating data behind a shared reference.

**Why it matters:**
- Rust needs these because the borrow checker is too restrictive for some patterns (caching, lazy initialization, observer pattern)
- If Kāra's compiler infers borrowing, it will hit the same patterns where shared mutation is needed
- Without explicit interior mutability primitives, the compiler might silently insert locks — a performance and correctness landmine

**Why deferred:**
- The right design depends on how ownership inference (Feature 4) actually plays out in practice
- If the inference engine is smart enough, some patterns that need `RefCell` in Rust may "just work" in Kāra (e.g., the compiler infers unique access through a shared reference when it can prove no aliasing)
- Designing interior mutability now would be guessing at which patterns the inference engine can't handle — better to let implementation reveal the gaps

**Revisit when:** The ownership inference engine is implemented and tested against real-world patterns. The specific patterns that fail inference will dictate what interior mutability primitives are needed.

**Risk of deferring:** Low. The compiler can always fall back to RC (Q7) for values it can't infer. Interior mutability is an optimization over RC, not a correctness requirement.

### D3: Self-Referential Structs

**What:** A struct containing a field that borrows from another field (e.g., a parsed AST node that references the source string stored in the same struct). Rust requires `Pin<T>` for this, which is notoriously complex.

**Why it matters:**
- Common in parsers, compilers (Kāra's own AST), and string processing
- Without explicit support, programmers use indices or unsafe code as workarounds

**Why deferred:**
- Rust took ~4 years from 1.0 to stabilize `Pin` (2018), and the design is still considered one of Rust's worst ergonomic failures
- The right solution depends on the ownership inference engine's capabilities — if the compiler can track intra-struct borrows, `Pin` may be unnecessary
- Arena allocation (Q9) and `shared struct` cover many of the same use cases with simpler semantics

**Revisit when:** The ownership system is stable and real programs hit self-referential patterns that Arena doesn't solve. Gather concrete failing examples before designing a solution.

**Risk of deferring:** Medium. The compiler itself (self-hosting, Q27) will need AST nodes that reference source text. `shared struct` or Arena allocation is the likely workaround.

### D4: Streaming / Lazy Iterators

**What:** Rust's `Iterator` trait — lazy, composable, zero-cost iteration. Also async iterators/streams for processing data that arrives over time (streaming HTTP, database cursors, event streams).

**Why it matters:**
- Processing a 10GB file line by line (can't load into memory)
- Streaming HTTP responses, reactive event streams, database cursors
- These are inherently sequential-with-occasional-parallelism patterns that the fork-join model (Feature 5) doesn't cover

**Why deferred:**
- Iterators are a trait + standard library feature, not a language primitive. The language already has traits (committed), generics (Q18), and `impl` blocks (Q17). Iterators are built on top.
- Async iterators depend on the async I/O model (Q29) being stable first
- The composability of iterators with effects needs real-world testing: `fn lines(file: File) -> Iterator[String] reads(FileSystem)` — does the effect annotation go on the iterator constructor or on each `next()` call?

**Revisit when:** Building the standard library. The trait system and generics must be implemented first.

**Risk of deferring:** Low for language design. High for practical usability — most real programs need iterators. Standard library work should start early.

### D5: Concurrency Primitives (Channels, Atomics, Synchronized Collections)

**What:** Communication and synchronization primitives for concurrent tasks — channels (message passing), atomics (lock-free operations), concurrent hash maps, queues.

**Why it matters:**
- The effect system handles fork-join parallelism (spawn N independent tasks, join results)
- Real-world concurrent programs also need: producer-consumer pipelines, event loops, long-lived background workers, lock-free counters
- Without channels, there's no way for concurrent tasks to communicate intermediate results

**Why deferred:**
- Channels are a standard library type, not a language feature. They compose with effects naturally: `sends(Channel[T])`, `receives(Channel[T])`
- Atomics are low-level primitives exposed via `unsafe` (Q28)
- Synchronized collections are built on top of atomics + locks
- The work-stealing runtime (Q10) must be implemented first to know what primitives it needs

**Revisit when:** The auto-concurrency runtime is implemented and real programs hit patterns that fork-join can't express. Channels are likely the first addition.

**Risk of deferring:** Medium. Fork-join covers the demo cases but not production concurrent systems. Channels should be among the first standard library additions.

### D6: Numeric Tower / SIMD

**What:** Explicit SIMD types (`f32x4`, `f32x8`), fixed-point arithmetic, and units of measure (`meters * seconds → meter_seconds`).

**Why it matters:**
- SIMD is essential for game engines, ML inference, and scientific computing — all stated target domains
- Auto-vectorization (LLVM) is unreliable and opaque — explicit SIMD types are more consistent with the "express intent" philosophy
- Units of measure (F# influence) would be a genuine differentiator for scientific computing
- Fixed-point arithmetic is needed for financial and embedded domains

**Why deferred:**
- Auto-vectorization is an LLVM backend concern handled by optimization passes. Kāra gets it "for free" from LLVM.
- SIMD intrinsics can be exposed via FFI + `unsafe` (Q28) as a stopgap
- Units of measure are a type system extension that can be added without changing existing code
- Fixed-point is a library concern (newtype over integer with operator overloading)

**Revisit when:** Targeting GPU/ML workloads where auto-vectorization isn't sufficient. Units of measure can be revisited when the type system is mature and there's a scientific computing user base.

**Risk of deferring:** Low. LLVM auto-vectorization covers most cases. Explicit SIMD can be added incrementally via traits/generics.

### D7: Build System / Package Manager

**What:** Cargo-equivalent: dependency management with lockfiles, reproducible builds, cross-compilation support, workspace management.

**Why it matters:**
- Cargo is widely cited as why Rust succeeded. npm is why Node succeeded. The build system is the language for adoption purposes.
- Cross-compilation is critical for Feature 7 (multi-target)
- For an AI-first language, the build system should expose a machine-readable project model that AI tools can query and modify

**Why deferred:**
- The build system can't be designed without knowing: file extension, module system semantics, compilation unit boundaries, FFI linking model — all of which come from implementation
- Build systems are best designed by building real projects and feeling the pain points
- A simple `Makefile` or `build.sh` suffices during early development

**Revisit when:** Multiple files need to be compiled together, or the first external dependency is needed. Likely during standard library development.

**Risk of deferring:** Low for language design, high for adoption. Plan to invest heavily in tooling before any public release.

### D8: AI-First Features (partially promoted to v1 — see G4)

**Promoted to v1:**
- **Structured compiler output (JSON diagnostic protocol)** — all compiler output has machine-readable format with applicable fix diffs
- **Compiler query API** — programmatic access to effect inference, ownership decisions, and concurrency analysis
- **Canonical formatting** — deterministic formatter output so AI edits produce clean semantic diffs

**Still deferred:**
- **Bidirectional compiler communication:** AI asserts "this loop is embarrassingly parallel" as a checkable annotation, not a hope. The compiler verifies or rejects.
- **Machine-verifiable intent annotations:** `@invariant(sorted)`, `@complexity(O(n log n))`, `@security(no_user_input_in_query)` — constraints for the AI to satisfy and the compiler to check.
- **Formal specification as primary artifact:** If AI writes code, the specification (pre/post conditions) becomes the artifact humans care about. This makes Level 3-4 verification (Feature 6) more central, not less.

**Why the remaining items are still deferred:**
- Bidirectional hints are enhancements to the effect system — the effect system must work first
- Intent annotations overlap with Feature 6 (Gradual Verification), already deferred
- We don't yet know which of these matter vs. which are theoretically interesting but unused

**Revisit when:** The effect system is implemented and AI is actually writing Kāra code. Real-world AI authoring will reveal which features are missing.

**Features developed after BRAINSTORMING_V1:**

- **`shared struct` / `shared enum` (reference semantics with RC):** A major ownership feature not anticipated here. `shared struct` and `shared enum` opt types into reference semantics with automatic RC, making recursive types (trees, graphs, JSON) ergonomic without requiring Arena boilerplate. This was developed as part of Feature 4 (Tiered Ownership). See design.md Feature 4 for the full design.

- **Project profiles (`kernel`, `embedded`, `server`):** Project-wide effect restriction sets declared in `kara.toml`. Profiles like `kernel` (no heap, no panics) and `embedded` (no heap, no panics, no concurrency) enforce constraints compiler-wide without per-function annotations. Designed after BRAINSTORMING_V1. See design.md Project Profiles section for the full spec.

## Open Gaps and Concerns

**Note:** This section is a later audit layer, separate from the original design above. These are inconsistencies, ambiguities, and soundness holes found by reviewing the design after it was written — not original design choices. Resolved items have been fixed in design.md; unresolved items remain open.

### G1: ~~Ownership Inference Failure UX~~ — RESOLVED

**Resolved by: Redesign of Feature 4 — Tiered Ownership with Restricted Return Borrows.**

The original concern was that "Rust semantics, fully inferred" conflated two problems of different difficulty: parameter mode inference (easy) and return-position lifetime inference (hard). Hiding the hard problem behind inference would produce worse errors than Rust's explicit annotations.

**Resolution:** Feature 4 was redesigned to separate these concerns:
1. **Parameter mode inference** — compiler infers own/ref/mut ref from body analysis. Easy, no annotations needed.
2. **Owned returns by default** — eliminates return-position lifetime inference entirely for 95%+ of functions.
3. **Explicit `ref` for borrow returns** — lightweight opt-in for the rare case where returning a borrow matters for performance.
4. **RC fallback with budget controls** — `#[no_rc]`, `#[rc_budget(max: N)]`, and project-level settings prevent silent performance erosion.
5. **Structured error messages** — show the compiler's ownership decision, why RC was chosen, and concrete alternatives.

The key insight: restricting return-position borrows (requiring explicit `ref`) removes ~80% of Rust's lifetime complexity. The programmer escalates intentionally: owned → `ref` → RC. No hidden inference failures.

### G2: ~~Blocking I/O Migration Path~~ — RESOLVED

**Resolved by: Abstract Task Model with Cooperative Cancellation. Updates to Q10, Q12, Q29.**

The original concern was that v1's blocking I/O would leak thread-specific semantics (pthread_cancel, thread counts, fd-closing for cancellation) into the user-facing API, making future runtime evolution a breaking change.

**Resolution:** The execution strategy (blocking threads vs. state machines) is a runtime/compiler concern, not a language concern. Three changes seal off the thread-specific leaks:

**1. Cooperative cancellation at effect boundaries (replaces pthread_cancel/fd closing in Q29).**

The compiler inserts cancellation checks at every effect boundary — the natural point, since the effect system already marks these locations:

```
fn fetch_profile(id: u64) -> Profile reads(UserDB) {
    // [compiler inserts: check_cancellation()?]
    let row = db_query("SELECT * FROM users WHERE id = ?", id);
    // [compiler inserts: check_cancellation()?]
    Profile.from_row(row)
}
```

- No `pthread_cancel` — unsafe (can leave mutexes locked, resources leaked).
- No fd closing for cancellation — side effect that violates structured concurrency.
- A cancelled task returns a `Cancelled` error variant, propagated via `?` like any other error.
- Works identically for blocking threads, state machines, and io_uring.

**2. The abstraction is "task", not "thread" (updates Q10).**

The runtime exposes tasks, never threads. Thread count is a runtime config, not a language annotation:

```
// Language level — task-oriented
#[concurrency(max_tasks: 100)]

// Build config — runtime concern, may change between runtime versions
[runtime]
max_concurrent_tasks = 100
io_thread_pool_size = 8       // v1 implementation detail
```

Q10's `#[concurrency(max_threads: 4)]` is replaced. Users never see thread IDs, thread-local storage, or thread counts. The compiler/runtime translates task limits into thread pool sizing (v1) or state machine scheduling (future).

**3. FFI always gets real OS threads, regardless of runtime model.**

```
extern "C" fn blocking_c_call(fd: i32) -> i32
    reads(FileSystem);
// Always runs on a real OS thread stack.
// v1: runs inline (everything is threads anyway).
// Future state-machine runtime: scheduled on a blocking thread pool
//   (like Rust's spawn_blocking, Go's LockOSThread).
```

FFI is never a migration concern — it works the same regardless of execution model.

**4. v1 runtime scope is explicitly documented.**

```
v1 Runtime: Blocking I/O on OS Threads

Scaling: ~10K concurrent I/O-bound tasks (one OS thread per blocked task).

Suitable for:
- CLI tools, compilers, desktop apps
- Backend services with moderate concurrency
- Game engines (compute-bound, not I/O-bound)

Not suitable for (requires future runtime evolution):
- High-connection web servers (>10K simultaneous connections)
- Real-time event streaming at scale

The language surface is identical regardless of runtime. Code written for v1
runs unchanged on future runtimes with better scaling characteristics.
```

**Summary of changes to existing resolutions:**
- **Q10:** `#[concurrency(max_threads: 4)]` → runtime config. Language annotation is `#[concurrency(max_tasks: N)]`.
- **Q12:** Cancellation mechanism is cooperative flags at effect boundaries. Structured concurrency model unchanged.
- **Q29:** Add: "The language surface (tasks, cancellation, concurrency annotations) is execution-model-agnostic. v1 implements tasks as blocking OS threads. Future versions may use state machines, io_uring, or other strategies without language-level changes." Remove `pthread_cancel` and fd-closing references.

### G3: ~~Resource Aliasing Soundness Hole~~ — RESOLVED

**Resolved by: Provider-Rooted Resources with three layers of defense. Updates to Q30.**

The original concern was that `alias` declarations in Q30 were the sole mechanism for connecting abstract resources to real infrastructure, making the effect system's safety guarantee conditional on programmer diligence — exactly the problem effects are supposed to solve.

**Resolution:** Make providers the primary source of truth for resource identity. Three layers of defense close the soundness gap progressively:

**Layer 1: Same-instance detection (compile-time, automatic).**

Resources are typed by their provider trait. When the application passes the same provider instance to two `with_provider` calls, the compiler detects it and treats those resources as aliased — no annotation needed:

```
// Define provider trait
trait DatabaseProvider {
    fn query(sql: String) -> Result[Rows, Error];
    fn execute(sql: String) -> Result[(), Error];
}

// Resources backed by the same provider trait type
effect resource UserDB: DatabaseProvider;
effect resource OrderDB: DatabaseProvider;

let shared_db = PostgresDB.connect("postgres://...");

// Same instance → compiler detects conflict automatically
with_provider[UserDB](shared_db, || {
    with_provider[OrderDB](shared_db, || {
        process();  // writes(UserDB) + writes(OrderDB) → CONFLICT
        // Compiler: UserDB and OrderDB share the same provider instance.
        // Auto-parallelization of conflicting effects is disabled.
    });
});

// Different instances → safe to parallelize, no annotation needed
with_provider[UserDB](PostgresDB.connect("postgres://users-db"), || {
    with_provider[OrderDB](PostgresDB.connect("postgres://orders-db"), || {
        process();  // writes(UserDB) + writes(OrderDB) → safe
    });
});
```

This catches the most common case — the application is the integration point where resources connect to real infrastructure, and the compiler already sees `with_provider` call sites.

**Layer 2: Explicit `alias` for non-obvious cases (compile-time, manual).**

When two libraries internally create their own provider instances that happen to hit the same system (e.g., both read a config and connect to the same DB), the compiler can't detect it. The `alias` declaration from Q30 remains as a fallback:

```
alias mylib.UserDB = theirlib.TheirDB;
```

Same mechanism as Q30, but now it's the fallback, not the primary defense.

**Layer 3: `--strict-effects` mode (compile-time, opt-in).**

For high-assurance code, a compiler flag that refuses to auto-parallelize across module boundaries unless resource relationships are explicitly declared:

```
// With --strict-effects, every cross-module resource pair requires:
independent mylib.UserDB, theirlib.TheirDB;  // asserts: different systems
// OR
alias mylib.UserDB = theirlib.TheirDB;        // asserts: same system
```

Without a declaration, `--strict-effects` falls back to sequential execution for cross-module resource pairs. Safe by default — you opt *in* to parallelism rather than hoping aliasing is declared correctly.

**Behavior by mode:**

| Scenario | Default mode | `--strict-effects` mode |
|---|---|---|
| Same provider instance | **Auto-detected** — serialize | **Auto-detected** — serialize |
| Cross-library, no declaration | Assumes independent (parallelize) | **Error** — requires `alias` or `independent` |
| Explicit `alias` | Serialize | Serialize |
| Explicit `independent` | Parallelize | Parallelize |

**Summary of changes to Q30:**
- Resources gain a provider trait type: `effect resource UserDB: DatabaseProvider;`
- Same-instance detection is automatic at `with_provider` call sites.
- `alias` remains for cases the compiler can't detect.
- `independent` declaration added for `--strict-effects` mode.
- The soundness gap shrinks from "all cross-resource parallelism is trust-based" to "only cross-library-internal-connection parallelism is trust-based." `--strict-effects` closes even that gap.

### G4: ~~AI-First Identity Gap~~ — RESOLVED

**Resolved by: AI-First Through the Compiler Interface. Three concrete v1 commitments. D8 partially promoted.**

The original concern was that Starting Assumption #1 ("AI writes most code") had no substance in the v1 feature set. The resolution: **AI-first is a compiler UX problem, not a language syntax problem.** The language itself doesn't need AI-specific syntax. What changes is that every compiler output has two consumers (human and AI agent), and the compiler exposes a programmatic interface designed for agents.

**1. Structured Compiler Output (JSON diagnostic protocol) — v1 commitment.**

Every compiler output — errors, warnings, performance notes, concurrency reports — has a machine-readable format alongside human-readable. Q8 already commits to "designed for machine consumption first" for performance notes; this extends to **all** compiler output:

```bash
# Human-readable (default)
$ karac build
error[E0412]: src/order.kara:42:9
  `user` moved on line 38, used again here.
  hint: clone `user` before the move, or use `ref` parameter mode.

# Machine-readable (for AI agents)
$ karac build --output=json
{
  "diagnostics": [{
    "severity": "error",
    "code": "E0412",
    "category": "ownership",
    "file": "src/order.kara",
    "line": 42,
    "column": 9,
    "message": "`user` moved on line 38, used again here.",
    "related": [
      {"file": "src/order.kara", "line": 38, "label": "moved here"}
    ],
    "hints": [{
      "description": "clone `user` before the move",
      "diff": {
        "file": "src/order.kara",
        "line": 37,
        "old": "process(user);",
        "new": "process(user.clone());"
      }
    }]
  }]
}
```

Every diagnostic includes machine-applicable fix suggestions as diffs, not just prose. The AI agent gets structured data it can act on programmatically.

**2. Compiler Query API (programmatic access to compiler analysis) — v1 commitment.**

An AI agent writing Kāra code needs to ask the compiler questions beyond "does it compile?" A programmatic interface exposes the compiler's internal analysis:

```bash
# What effects does this function have? (inferred, not just declared)
$ karac query effects src/order.kara.process_order
{
  "declared": ["writes(OrderDB)", "reads(UserDB)"],
  "inferred": ["writes(OrderDB)", "reads(UserDB)", "reads(ConfigDB)"],
  "source": {
    "reads(ConfigDB)": "called from load_config() at line 45"
  }
}

# What ownership decisions did the compiler make?
$ karac query ownership src/order.kara.process_order
{
  "parameters": [
    {"name": "order", "mode": "own", "reason": "consumed at line 48"},
    {"name": "config", "mode": "ref", "reason": "read-only access"}
  ],
  "rc_values": [
    {"name": "user", "reason": "returned from two branches with different lifetimes"}
  ]
}

# What did the compiler parallelize?
$ karac query concurrency src/order.kara.load_dashboard
{
  "concurrent_groups": [
    {
      "tasks": ["fetch_profile", "fetch_orders", "fetch_notifs"],
      "reason": "non-conflicting effects: reads(UserDB), reads(OrderDB), reads(NotifDB)",
      "sync_point": "line 52: build_dashboard(profile, orders, notifications)"
    }
  ]
}
```

The agent queries the compiler directly instead of compiling, parsing output, and guessing. No mainstream language has a compiler query API designed for AI agents — this is a genuine differentiator.

**3. Canonical Formatting (deterministic output from the parser) — v1 commitment.**

The formatter (`karac fmt`) produces a **canonical form** — semantically equivalent programs produce identical formatted output:

- Declaration order within a scope is canonicalized (e.g., `impl` methods sorted by name, `use` statements sorted alphabetically)
- Semantically equivalent expressions normalize to one form (e.g., `if !x { a } else { b }` → `if x { b } else { a }`)
- Whitespace and formatting are deterministic given the AST

When an AI regenerates a function, the diff shows only *semantic* changes, not reordering noise. Without canonical formatting, every AI edit produces noisy diffs that humans can't review efficiently.

**Implementation constraint:** The AST needs a canonicalization pass. AST nodes must carry enough semantic information for the formatter to determine canonical order. This must be designed into the AST from the start — adding it later requires changing every AST node.

**What stays deferred (D8 remainder) — RESOLVED, tracked in `docs/design.md` Deferred Items:**
- Bidirectional compiler hints (`@parallel`, `@complexity(O(n))`) — wait for real AI usage.
- Formal specification as primary artifact — depends on Feature 6 (Gradual Verification / Level 3).
- Machine-verifiable intent annotations — overlaps with verification.

**Positioning shift:** From "a language *for* AI" (requires AI-specific language features) to "a language *designed to be written by AI*" (requires a machine-friendly compiler interface). The three commitments give v1 real substance behind the AI-first claim without speculative language features.

| v1 Commitment | What it does | Why it's AI-first |
|---|---|---|
| Structured output | JSON diagnostics with machine-applicable fixes | AI reads structured data, not prose |
| Query API | Programmatic access to compiler analysis | AI asks questions, not trial-and-error |
| Canonical formatting | Deterministic code output | AI edits produce clean diffs |

### G5: ~~Effect System Composition at Scale~~ — RESOLVED

**Resolved by: Effect group composition + effect semver rules. Clarifies existing Q1/Q3 resolutions.**

The original concern was that the effect system might become infectious and tedious at scale — annotation explosion, refactoring cost, and unclear API stability. Analysis shows two of three sub-problems were already solved by existing resolutions; the third (semver) needed new design.

**Sub-problem 1: Annotation explosion — already solved by Q1 + Q3.**

Private functions: effects inferred, no annotations. Public functions: declare their effects (honest documentation). Named effect groups (Q3) reduce repetition. Long effect lists are a design smell — the function does too much. No additional design needed.

**Sub-problem 2: Refactoring cost — solved by adding effect group composition.**

Q1's `transparent` effects handle the common case (adding a log doesn't break callers). Q1's inference handles private function changes. The remaining case — a public function adds a real new effect — is solved by composable effect groups:

```
// Sub-groups for logical concerns
effect group Validation = reads(UserDB, InventoryDB), sends(FraudService);
effect group Fulfillment = writes(OrderDB), sends(PaymentGateway);
effect group Notification = sends(NotificationService);

// Composed from sub-groups — changes to sub-groups propagate automatically
effect group OrderProcessing = Validation + Fulfillment + Notification;

pub fn process_order(order: Order) -> Result[Receipt, Error]
    with OrderProcessing
{ ... }
```

Adding `sends(FraudService)` to `Validation` automatically propagates to `OrderProcessing` and all its callers. The refactoring is one line. Effect groups compose with `+`, following the same pattern as trait bounds.

**Sub-problem 3: Cross-module effect visibility and semver — new design.**

Adding a new effect to a public function is a breaking API change — callers that declared their effects based on the library's function now have incomplete declarations (compile error), and the compiler's parallelism decisions change.

**Effect semver rules:**

| Change | Semver impact | Why |
|---|---|---|
| Adding an effect to a public function | **Breaking** (major) | Callers' declarations are now incomplete |
| Removing an effect from a public function | **Non-breaking** (minor) | Callers' declarations are over-specified but correct |
| Changing a private function's effects | **Non-breaking** | Private effects are inferred, invisible to callers |
| Adding a `transparent` effect | **Non-breaking** | Transparent effects don't propagate |
| Splitting a resource into parameterized resources | **Breaking** (major) | Changes effect signatures |
| Adding an effect to a published effect group | **Non-breaking** (minor) | Callers using the group absorb the change |

**Library effect groups as versioning boundary:**

Libraries publish effect groups as part of their API. Callers depend on groups, not raw effect lists:

```
// Library defines and exports its effect group
pub effect group OrderServiceEffects =
    reads(UserDB), writes(OrderDB), sends(PaymentGateway);

// Caller uses the library's group — insulated from individual effect changes
pub fn handle_request(...) with orderlib.OrderServiceEffects, reads(FileSystem) { ... }
```

When the library adds `sends(FraudService)` to `OrderServiceEffects`, the caller's code compiles unchanged — the group absorbed the new effect. Libraries should document: "depend on our effect groups, not on individual effect lists." Same principle as "depend on our public API types, not on our internal struct layout."

This makes effect groups serve double duty: ergonomic shorthand (Q3) and API stability boundary (new). Depending on a group means accepting that its effect surface can grow in minor versions, which may affect auto-parallelism with the caller's own effects — a deliberate trade-off for API stability insulation.

**Sub-problem 4: `main()` effect visibility — resolved by build output.**

`main()` is not `pub`, so its effects are inferred and never written in source. This creates a gap: every public function declares its effects for callers to see, but the entry point — whose effects represent the full observable behavior of the program — declares nothing. Requiring annotations on `main()` would impose a burden with no API-stability benefit (no external callers exist). The solution is the compiler: after a successful build, it emits a program-level effect summary as standard output and includes `"program_effects": [...]` as a top-level key in `--output=json`. The full effect surface is always visible without source annotations.

### G6: Stored vs Immediate Callback Effects — RESOLVED

**The problem:** The current effect system propagates closure effects via `with _`, but doesn't distinguish between "I call your callback now" and "I store your callback for later invocation." This distinction has concurrency implications:

```
// Case 1: Immediate invocation — effects happen in caller's context
fn map[T, U](list: Vec[T], f: Fn(T) -> U with _) -> Vec[U] {
    // f is called synchronously, effects execute here
}

// Case 2: Stored for later — effects happen in a different context
fn on_click(button: Button, handler: Fn() with _) {
    // handler is stored, invoked later — possibly in a concurrent task
}

// Case 3: Stored and invoked across concurrency boundaries
fn spawn_worker(work: Fn() writes(OrderDB)) {
    // work is sent to another thread — the writes(OrderDB) effect
    // now happens concurrently with the caller's continuation.
    // Does the compiler know to serialize with other writes(OrderDB)?
}
```

**Why it matters:** When a closure is invoked immediately, its effects compose naturally with the caller (union of effects, standard conflict detection). When a closure is *stored* and invoked later — especially across a concurrency boundary — the compiler needs to know that the deferred effects may conflict with effects that happen after the storing call returns.

**Resolution: Three forms for closure parameters.**

The key insight: **storing a closure doesn't perform its effects.** Effects track to the point of invocation, not creation or storage. The mechanism already exists in the effect system — `with _` means "invoke and propagate." The gap was that there was no way to accept an effectful closure *without* propagating its effects.

The three forms:

| Form | Meaning | Effect propagation |
|---|---|---|
| `Fn(T) -> U` | Pure closure required | None — closure has no effects |
| `Fn(T) -> U with _` | Invoke and propagate | Closure's effects become caller's effects |
| `Fn(T) -> U with writes(R)` | Accept and store — don't invoke yet | Effects tracked on closure type, NOT propagated to accepting function |

In the third form, `writes(R)` is a concrete effect expression — any effect expression can appear after `with`. This is not a type variable: it names the exact effects the stored closure will perform when eventually invoked. The difference from `with _` is intentional: `with _` says "I call this now and inherit its effects"; a concrete expression says "I store this and someone else will call it later."

The third form is the new clarification. It lets you pass effectful closures to functions that store them without forcing those functions to declare effects they don't perform.

```
struct EventBus {
    handlers: Vec[Fn(Event) with writes(OrderDB)],
}

// Stores a handler — no writes(OrderDB) effect on register
fn register(bus: mut ref EventBus, handler: Fn(Event) with writes(OrderDB)) {
    bus.handlers.push(handler);
}

// Invokes handlers — writes(OrderDB) effect declared
fn emit(bus: ref EventBus, event: Event) writes(OrderDB) {
    for handler in bus.handlers {
        handler(event);
    }
}

// Auto-concurrency works correctly:
fn setup_and_load(bus: mut ref EventBus, user_id: u64) {
    register(bus, |e| { update_order(e) });  // no OrderDB effect
    let profile = fetch_profile(user_id);    // reads(UserDB)
    // ^ compiler can parallelize these — register doesn't touch OrderDB
}
```

**Soundness argument:** If `register` stores a `writes(OrderDB)` closure without invoking it, no write to OrderDB happens during `register`. The compiler is correct to not count `writes(OrderDB)` as an effect of `register`. When `emit()` eventually calls the stored closure, *that* function declares `writes(OrderDB)`, and the compiler serializes it with conflicting operations at that call site. Kāra manages all concurrency — there's no manual thread spawning that could invoke the closure outside the compiler's knowledge.

**Inference for private functions:** The compiler already does body analysis for effect inference. If it sees the closure parameter is invoked → infer the closure's effects. If it sees the closure is only stored → don't infer those effects. This is the same kind of dataflow analysis as parameter mode inference (own vs ref).

**For public functions:** The programmer declares only the effects the function actually performs. Storing a closure without invoking it means you don't declare the closure's effects.

### G7: ~~v1 Blocking I/O — 10K Concurrency Ceiling~~ — RESOLVED

**Resolved by: Phased runtime evolution — v1 blocking, v1.1 network event loop, v2 full hybrid. Option A (hybrid runtime) confirmed as architecture.**

G2 resolved the *migration path* concern — ensuring that v1 code works unchanged on future runtimes. But the ~10K concurrent task ceiling itself is a structural problem that G2 doesn't address.

**Why this matters more than G2 acknowledged:**

1. **Backend services are the dominant adoption vector for new languages.** Go, Rust, and Zig all gained traction through backend/infrastructure use cases first. A language that can't serve >10K concurrent connections is excluded from the most common proving ground.

2. **The ceiling isn't 10K tasks — it's 10K *blocked* tasks.** One OS thread per blocked I/O operation means 10K simultaneous in-flight I/O operations. For a web server with keep-alive connections (most of which are idle), this limit is hit at far fewer than 10K *active* requests. A server with 50K idle WebSocket connections and 100 active requests would need 50K threads under the blocking model.

3. **"State machines later" is underspecified.** G2 says the language surface is execution-model-agnostic, but doesn't commit to *when* or *how* the async runtime arrives. Without a concrete plan, "later" becomes "never" — and early adopters build on the blocking model, creating ecosystem pressure against the transition.

4. **Thread overhead compounds.** Each OS thread costs ~8MB stack (default on Linux). 10K threads = ~80GB of virtual address space for stacks alone. Even with reduced stack sizes, the scheduler overhead of 10K OS threads (context switches, TLB flushes) is significant compared to Go's ~4KB goroutine stacks or Rust's state-machine futures.

**Possible approaches (need evaluation):**

**Option A: Hybrid runtime for v1 — blocking compute + epoll/kqueue for I/O.**

The compiler already knows from effect analysis which operations are I/O-bound (`reads(Network)`, `sends(Network)`) vs compute-bound. Use this information:
- Compute-bound tasks: run on OS threads (current model, FFI-compatible).
- I/O-bound tasks: register with epoll/kqueue/io_uring, park the task (not the thread), resume on completion.

This is essentially what Go does internally (netpoller) and what Rust's Tokio does. The difference: Kāra's effect system *tells* the compiler which operations are I/O, so there's no need for the programmer to choose between sync and async APIs. The compiler routes to the right execution strategy.

**Complexity cost:** Requires an event loop integrated with the work-stealing scheduler. Significantly more runtime complexity than pure blocking. But avoids the full state machine transform (functions still have real stacks — only the I/O wait is non-blocking).

**Option B: io_uring-first on Linux, kqueue on macOS.**

Skip the epoll generation entirely. io_uring provides kernel-level async I/O with completion queues — the OS handles the non-blocking mechanics. The runtime submits I/O requests to the ring and polls for completions. Tasks suspend at I/O boundaries without blocking a thread.

**Advantage:** Less userspace complexity than Option A. The kernel manages the event loop.
**Risk:** io_uring is Linux 5.1+ only (2019). macOS and Windows need different backends. Platform-specific runtime complexity.

**Option C: Accept the ceiling, target differently.**

Keep v1 as-is, but don't position Kāra for high-concurrency backend services until v2. Target:
- CLI tools and compilers (no concurrency ceiling concern)
- Game engines (compute-bound, not I/O-bound)
- Embedded/systems (few concurrent connections)
- Data pipelines (batch processing, not connection-heavy)

**Risk:** Concedes the most important adoption vector. Go and Rust dominate backend because they got concurrency right in v1.

**Option D: Lightweight thread stacks (segmented or growable).**

Reduce per-thread cost from ~8MB to ~8KB using segmented stacks (Go's original approach) or growable stacks. This pushes the ceiling from ~10K to ~1M concurrent tasks without changing the blocking model.

**Risk:** Segmented stacks break C FFI (Go abandoned them for this reason). Growable stacks add complexity and can cause stack relocations. Both approaches conflict with the "FFI always gets real OS threads" guarantee from G2.

**Recommendation:** Option A (hybrid runtime) is the most promising. The effect system provides exactly the information needed to route I/O to an event loop without programmer intervention. This preserves the "no async/await" promise while scaling to 100K+ connections. The complexity is in the runtime, not the language surface.

#### Option A Downside Analysis

Option A is the right *direction*, but carries significant implementation and design risks:

**1. The yield point problem — the reason Rust has `.await`.**

When a task parks on an event loop, it must save its local state and resume later. That's a state machine transform — the same thing Rust does with `async/await`. The difference: Rust makes yield points *explicit* (`.await`), Kāra would make them *implicit* (every effect boundary).

Implicit yield points mean:
- **Drop order becomes non-obvious.** A `MutexGuard` held across an implicit yield point keeps the mutex locked while the task is parked. Rust catches this at `.await` because the programmer can see the yield. In Kāra, the programmer doesn't know a yield happened.
- **Stack variables must be captured.** Every local variable alive across an effect call must be saved into the state machine. The compiler can do this, but it increases the size of each task's captured state — potentially defeating the memory advantage over blocking threads if functions have many live locals across I/O calls.

**2. Two execution modes = two sets of bugs.**

Compute tasks run on real OS thread stacks. I/O tasks run as state machines on an event loop. Two fundamentally different execution models coexisting means:
- Different stack behavior (real stack vs captured state)
- Different cancellation mechanics (thread interruption vs future dropping)
- Different debugging experiences (real stack traces vs state machine frames)
- Different performance characteristics (context switch vs poll overhead)

Every interaction between the two modes is a potential edge case. What happens when a compute-bound function calls an I/O function? The compute thread must... block? Spawn a future? Park itself on the event loop? Each answer has tradeoffs.

**3. FFI at the boundary gets complicated.**

G2 guarantees "FFI always runs on real OS threads." If a task is running as a state machine on the event loop and calls into C via FFI, the runtime needs to:
- Move the task to an OS thread (`spawn_blocking` in Tokio's terms)
- Block that thread for the FFI call
- Resume the state machine after completion

This is solvable (Tokio does it) but adds latency and complexity at FFI boundaries. The effect system must distinguish between "I/O effect handled by event loop" and "I/O effect behind FFI that must block a real thread."

**4. Effects become load-bearing for correctness, not just optimization.**

With blocking I/O, effects are advisory — getting them wrong just means missed parallelism. With Option A, effects *route execution strategy*:
- If a function is marked pure but actually does I/O, it blocks the event loop thread (stalls all other tasks on that thread).
- If an FFI function is marked as compute but actually blocks on I/O, same problem.
- The consequence of a wrong effect annotation goes from "suboptimal parallelism" to "entire event loop stalls."

This is a meaningful shift in the severity of effect annotation errors.

**5. Runtime complexity and binary size.**

A blocking-only runtime is simple: thread pool + work queue. Option A adds:
- Event loop (epoll/kqueue/io_uring abstraction layer)
- State machine transform in the compiler
- Task parking/resumption machinery
- Cross-mode scheduling (compute ↔ I/O task handoff)

This is essentially building Tokio into the language runtime. Tokio is ~50K lines of carefully optimized Rust. Doable, but a significant implementation investment and a larger runtime binary.

**6. Debugging and profiling.**

With blocking threads, every task has a real stack. `gdb`, `perf`, `strace` all work naturally. With state machines, tasks don't have stacks — they're enums being polled. Custom tooling is needed to reconstruct logical call stacks from state machine variants. Rust's async debugging story is still poor after 7+ years.

#### Option A Tradeoff Summary

| | Blocking (current v1) | Option A (hybrid) |
|---|---|---|
| Concurrency ceiling | ~10K | ~1M+ |
| Implementation complexity | Low | High (Tokio-level) |
| Effect annotation errors | Suboptimal perf | Correctness bugs |
| Debugging | Native tools work | Needs custom tooling |
| FFI | Trivial | Needs spawn_blocking |
| Binary size | Small runtime | Larger runtime |

#### Prior art: How Rust handles this

Rust's answer is `async/await` with state machine compilation. `async fn` compiles to an enum where each variant holds locals alive across `.await` points. Futures are ~dozens of bytes (not ~8MB thread stacks), enabling millions of concurrent connections via Tokio.

What Rust proves: the *mechanism* (state machines + event loop) scales. The *problem* is the user-facing API — colored functions, `Pin`, `async fn` in traits (stabilized 8 years after async was proposed). Kāra's leverage is that the effect system can replace `async` as the compiler's routing signal, but the underlying state machine complexity doesn't disappear — it just moves from the programmer's syntax to the compiler's implementation.

#### Resolution: Phased Runtime Evolution

Option A (hybrid runtime) is confirmed as the architecture. Phased delivery manages implementation risk while committing to a concrete path:

**Phase 1 (v1): Blocking I/O on OS threads.**

Scope v1 to domains where ~10K concurrent tasks is sufficient:
- CLI tools, compilers, desktop apps
- Game engines (compute-bound, not I/O-bound)
- Backend services with moderate concurrency

The ceiling is documented explicitly. The language surface is execution-model-agnostic — code written for v1 runs unchanged on all future phases.

**Phase 2 (v1.1): Event-loop I/O for network effects only.**

Add epoll/kqueue-based task parking for `sends(Network)` and `receives(Network)` only:
- The compiler knows from effect analysis which calls are network I/O
- Network I/O tasks park on the event loop (no thread blocked), resume on completion
- File I/O stays blocking (fast on local disks; truly async file I/O requires `io_uring` which is Linux-only)
- FFI stays simple — only network effects are routed to the event loop
- State machine transform is limited to network-boundary functions, reducing compiler complexity

This covers the 90% case — high-connection web servers, WebSocket servers, and API gateways. Concurrency ceiling rises from ~10K to ~1M+ for network-bound workloads.

**Phase 3 (v2): Full hybrid runtime.**

Extend event-loop I/O to all effect boundaries:
- All I/O effects (`reads`/`writes` on `FileSystem`, `Network`, etc.) can park on the event loop
- Add io_uring backend for file I/O on Linux
- Build async-aware debugging tooling (source-mapped state machine frames)
- Full state machine transform at all effect boundaries

**Why the downsides are acceptable:**

1. **Implicit yield points:** The compiler knows *exactly* where yields happen (effect boundaries) and can warn when RAII resources (mutex guards, file handles) span them. This is *better* than Rust's `.await`, where the programmer must visually spot the problem.

2. **Two execution modes:** The programmer never sees them. The compiler routes internally based on effects. Go does this transparently with its netpoller — proven to work.

3. **FFI complexity:** FFI functions already declare effects. The compiler sees `extern "C" fn ... writes(FileSystem)` and runs it on a blocking thread. No ambiguity.

4. **Effects load-bearing for correctness:** Effects are *already* load-bearing — wrong effects cause incorrect parallelism decisions (data races). The shift from "wrong parallelism" to "event loop stall" is a severity change, not a category change. And event loop stalls are *observable and debuggable* (server stops responding), while data races from wrong parallelism are *silent and intermittent*. The current design already has the harder problem.

5. **Runtime complexity:** Phasing limits v1.1 to network-only event loop — significantly less complex than a full Tokio-equivalent runtime. v2 builds on proven v1.1 infrastructure.

6. **Debugging:** The compiler generates state machines and can generate debug info mapping back to source locations. Investment in async debugging tooling starts in v1.1, is mature by v2.

**Kāra's advantage over prior art:**

| Problem | Go | Rust | Kāra |
|---|---|---|---|
| Which calls yield? | All (runtime guesses) | Programmer marks `.await` | Compiler knows (effect boundaries) |
| RAII across yield | No RAII (GC) | Programmer must check | Compiler warns automatically |
| FFI blocking | `cgo` overhead | `spawn_blocking` (manual) | Compiler routes (effect on `extern`) |
| Colored functions | No (goroutines) | Yes (async/sync split) | No (effect-routed) |

The effect system gives Kāra more information than either Go or Rust has. Go's runtime guesses which calls might block. Rust requires the programmer to say. Kāra's compiler *knows* from effect declarations — and those declarations are compiler-verified (G9), so they can't be wrong for non-FFI code.

**Summary of changes to existing resolutions:**
- **Q29:** v1 blocking I/O unchanged. Add: v1.1 introduces network event loop; v2 extends to all I/O effects.
- **G2:** Unchanged — the abstract task model and cooperative cancellation work across all phases.
- **Q10:** Add: work-stealing scheduler integrates with event loop in v1.1+. Compute tasks stay on OS threads; I/O tasks park on event loop.

**CLOSED.** All risks addressed above. Architecture confirmed. Implementation tracked in `roadmap.md` Phase 6.2 (v1 blocking) and Phase 6.3 (v1.1 network event loop).

### G8: ~~Effects Through Generic Trait Bounds~~ — RESOLVED

**Resolved by: Effect polymorphism on traits via `with _`, consistent with closure effect polymorphism (Q2).**

Q2 resolves effect polymorphism for closures via `with _`. Trait method dispatch introduces the same problem: different implementations of the same trait method can have different effects.

```
trait Processor {
    fn process(self, data: Data) -> Result[Output, Error];
}

impl Processor for LocalProcessor {
    fn process(self, data: Data) -> Result[Output, Error] {
        // Pure computation — no effects
        Ok(compute(data))
    }
}

impl Processor for RemoteProcessor {
    fn process(self, data: Data) -> Result[Output, Error]
        sends(Network) {
        // Network call — has effects
        remote_call(data)
    }
}

fn run[T: Processor](processor: T, data: Data) -> Result[Output, Error] {
    processor.process(data)  // What effects does this have?
}
```

**Three options were evaluated:**

**Option 1: Effect bounds on trait methods (upper bound).** The trait declares the *maximum* effects any impl can have. Every impl stays within that bound. Problem: over-approximation — callers that only use pure impls still carry worst-case effect annotations. A `sort` function using a `Comparator` trait would carry `writes(Database)` if any comparator impl touches a database, even though the sort-relevant impls are pure.

**Option 2: Effect polymorphism on traits via `with _`.** The trait method uses `with _` to indicate effects vary by impl. Each impl declares its actual effects. The compiler monomorphizes per concrete type and resolves effects at each call site. Consistent with the existing `with _` syntax for closures (Q2).

**Option 3: Implicit inference through monomorphization.** No effect annotation on trait methods. The compiler infers effects per monomorphized instantiation. Inconsistent with the explicit-at-public-boundaries rule (Q1/G9) — trait methods are public API.

**Decision: Option 2. Reasons:**

**1. Consistent with existing `with _` for closures.**

The `with _` syntax already means "effects vary by context — propagate from the concrete type." Extending this to trait methods is a natural generalization, not a new concept:

```
// Closures: with _ means "effects depend on what the closure does"
fn map[T, U](list: Vec[T], f: Fn(T) -> U with _) -> Vec[U]

// Traits: with _ means "effects depend on which impl is used"
trait Processor {
    fn process(self, data: Data) -> Result[Output, Error] with _;
}
```

Same syntax, same meaning: the effects are polymorphic and resolved at the call site.

**2. Monomorphization resolves effects per concrete type.**

Since generics are monomorphized (Q18), the compiler generates separate code for each concrete type. Effect resolution follows the same path:

```
// Monomorphized: run[LocalProcessor] → no effects (pure)
// Monomorphized: run[RemoteProcessor] → sends(Network)

pub fn run[T: Processor](p: T, data: Data) -> Result[Output, Error] with _ {
    p.process(data)
}
```

The `with _` on `run` means "my effects depend on what `T.process` does." At each call site, the concrete type is known, and the effects are resolved:

```
run(LocalProcessor.new(), data);   // compiler resolves: pure
run(RemoteProcessor.new(), data);  // compiler resolves: sends(Network)
```

**3. Public generic functions use `with _` — consistent with G9.**

G9 establishes that public functions must declare effects explicitly. For generic functions whose effects depend on type parameters, `with _` is the explicit declaration — it says "my effects are polymorphic." The compiler verifies that the function body's effects come only from `with _` trait method calls and doesn't introduce undeclared effects of its own.

**4. Trait methods without `with _` require pure impls.**

If a trait method omits `with _`, all impls must be pure — same rule as closures:

```
trait Comparator[T] {
    fn compare(self, a: T, b: T) -> Ordering;  // no with _ → all impls must be pure
}

impl Comparator[i64] for RemoteComparator {
    fn compare(self, a: i64, b: i64) -> Ordering
        reads(Network) {  // ERROR: trait method requires pure, impl has effects
        ...
    }
}
```

This is the right default for most traits — comparators, iterators, formatters should be pure. `with _` is opt-in for traits where impls legitimately vary in effects.

**5. Dynamic dispatch — effect semantics settled, implementation tracked separately.**

Dynamic dispatch (trait objects) is deferred to Phase 8. The effect semantics are resolved here; syntax and codegen are tracked in [implementation_checklist.md](../implementation_checklist.md) (Phase 8 — Dynamic dispatch / trait objects).

When the concrete type is unknown at compile time (no monomorphization), the compiler computes the *union* of all known impl effects as the worst-case bound:

```
// All impls of Processor have effects: {} (LocalProcessor) ∪ {sends(Network)} (RemoteProcessor)
// → trait object carries sends(Network)

fn run_dynamic(p: dyn Processor, data: Data) -> Result[Output, Error]
    sends(Network) {  // must declare worst-case
    p.process(data)
}
```

The programmer can also constrain which impls are accepted via explicit effect bounds on the trait object:

```
// Only accept impls with no effects
fn run_pure(p: dyn Processor with ()) -> Result[Output, Error] {
    p.process(data)  // guaranteed pure — only LocalProcessor-like impls allowed
}
```

**Summary** (syntax for dynamic dispatch rows uses `dyn` as a placeholder — final syntax decided at implementation time):

| Context | Syntax | Effect resolution |
|---|---|---|
| Trait method (pure required) | `fn method(self) -> T` | All impls must be pure |
| Trait method (effect-polymorphic) | `fn method(self) -> T with _` | Resolved per impl at monomorphization |
| Generic function calling trait | `fn f[T: Trait](x: T) with _` | Resolved per concrete `T` at call site |
| Dynamic dispatch | `fn f(x: dyn Trait) sends(Network)` *(provisional)* | Worst-case union of all impl effects |
| Dynamic dispatch (constrained) | `fn f(x: dyn Trait with ())` *(provisional)* | Only pure impls accepted |

### G9: ~~Explicit vs Inferred Effect Annotations on Public Functions~~ — RESOLVED

**Resolved by: Explicit annotations with compiler-suggested signatures. Confirms Q1's original resolution.**

**The question:** The compiler already infers effects for private functions by tracing the call graph. It also verifies that public function declarations match their bodies. If the compiler is smart enough to *flag* stale annotations, why can't it *infer* them — eliminating annotations entirely?

**Two options were evaluated:**

**Option A: Fully inferred with locked metadata.**

Public functions have no effect annotations in source. The compiler infers effects on first compile and records them in metadata (`.karameta` or equivalent). On subsequent compiles, the compiler checks whether inferred effects changed and errors if they did, requiring the programmer to acknowledge the change.

**Option B: Explicit annotations with compiler-suggested signatures.**

Public functions must declare effects explicitly (current Q1 design). When a public function is written without an annotation or when the body changes to include new effects, the compiler errors with a machine-applicable fix diff showing the exact annotation to add.

**Decision: Option B (explicit annotations). Reasons:**

**1. Source code should be self-contained.**

The effect signature is part of a function's public contract — as important as the return type. Storing it in metadata means the source code alone doesn't tell you what a function does to the world. This directly undermines the effect system's core value proposition: *"Effect signatures are reviewable contracts."*

With explicit annotations:
```
pub fn process(id: u64) -> Report
    reads(UserDB), writes(Cache) {
    ...
}
```

Without — you need a separate query to learn the same information:
```bash
$ karac query effects src/order.kara.process
```

The source code should be readable without tooling.

**2. Code review diffs are better with explicit annotations.**

When effects change, the diff shows it:
```diff
 pub fn process(id: u64) -> Report
-    reads(UserDB) {
+    reads(UserDB), writes(Cache) {
```

A reviewer immediately sees: "this function now writes to Cache — is that intentional?" Without explicit annotations, this change is invisible in the source diff. The reviewer must diff a metadata file or manually query effects before and after.

**3. Metadata is a second source of truth.**

Inferred metadata introduces:
- A file that must stay in sync with source code
- Build system complexity to track metadata
- Merge conflicts in metadata when two branches modify the same function differently
- A metadata file that can be corrupted, lost, or out of sync

Explicit annotations have one source of truth: the source code.

**4. The "accept change" workflow is worse than writing annotations.**

With inferred metadata, when effects change, the programmer must acknowledge a change to an annotation they never wrote. They must understand an inferred signature to decide whether a change to it is acceptable. This is harder than writing the annotation in the first place.

**5. Annotation burden is near-zero with AI authoring.**

Since AI writes most code (Starting Assumption #1), the "typing cost" of annotations is irrelevant. The workflow:

1. AI writes or modifies a public function body
2. Compiler emits structured error with machine-applicable fix:
   ```json
   {
     "severity": "error",
     "code": "E0502",
     "message": "public function `process` missing effect declaration",
     "inferred_effects": ["reads(UserDB)", "writes(Cache)"],
     "hints": [{
       "description": "add effect declaration",
       "diff": {
         "old": "pub fn process(id: u64) -> Report {",
         "new": "pub fn process(id: u64) -> Report reads(UserDB), writes(Cache) {"
       }
     }]
   }
   ```
3. AI applies the fix automatically — mechanical, no judgment needed
4. Compiler verifies body matches declaration — compiles clean

The AI never "struggles" with annotations. The compiler tells it exactly what to write. The human reviews the diff later and sees effect changes highlighted — the annotation change is a signal that draws attention to meaningful semantic changes.

**6. The AI does not need to prompt the user for effect changes.**

The AI can accept compiler-suggested effect annotations autonomously because:
- The compiler *verifies* correctness — the annotation either matches the body or doesn't compile
- The annotation is a mechanical reflection of what the code does, not a design choice
- The human review happens asynchronously in code review, where the diff shows the change clearly

The AI workflow is: write code → compiler suggests effects → apply suggestion → move on. No human-in-the-loop needed for annotation maintenance.

**7. Effect groups keep annotations concise.**

For functions with many effects, named groups (G5) prevent annotation explosion:
```
pub fn process_order(order: Order) -> Result[Receipt, Error]
    with OrderProcessing { ... }
```

One annotation covering multiple effects. Changes to the group's constituent effects propagate automatically — callers using the group don't need updates.

**What stays the same:**
- Private functions: fully inferred, no annotations (Q1 unchanged)
- Public functions: explicit declaration, compiler-verified against body (Q1 confirmed)
- FFI functions: explicit declaration, compiler-trusted (Q21 unchanged)

**What's clarified:**
- The compiler's role is to *suggest and verify*, not to *manage silently*
- AI applies effect annotations mechanically via structured compiler output (G4)
- Annotations exist in source code, not metadata — single source of truth
- Human review sees effect changes in diffs, catching unintended semantic changes

### G10: RC Creep in AI-Generated Code

**Status: DESIGN RESOLVED — `warn_above_0` default committed in design.md. Implementation tracked in [implementation_checklist.md](implementation_checklist.md) (Phase 7).**

**The problem:** The tiered ownership model (F4) uses RC as a fallback when the compiler can't determine ownership statically. Budget controls (`#[no_rc]`, `#[rc_budget(max: N)]`) exist to prevent silent performance erosion. But the default is permissive — RC "just works," and AI code generators have no incentive to restructure code for zero-cost ownership when RC resolves ambiguity immediately.

**The failure mode:** AI writes code → ownership is ambiguous → compiler inserts RC → code compiles and runs correctly → AI moves on. Over time, a codebase accumulates RC in places where a few minutes of restructuring would eliminate it. By the time someone notices performance degradation, the RC is deeply embedded and hard to remove.

**Why existing mitigations are insufficient:**

Current controls are all **opt-in**:
- `#[no_rc]` on hot-path functions — programmer must know which functions are hot
- `#[rc_budget(max: N)]` per module — programmer must set a budget
- Performance notes flag each RC insertion — AI must act on them
- Project-level `rc_budget = "warn_above_10_per_module"` — project must configure it

A new project starts with no budget. The AI generates code freely. Nobody tightens constraints until there's a performance problem. This is the "boiling frog" pattern — each individual RC is harmless, the cumulative effect is death by a thousand cuts.

**Proposed mitigations:**

**1. Default `rc_budget` is `warn_above_0` for new projects.**

Every RC insertion produces a compiler note (not error — code still compiles). The AI sees the note and tries to restructure:

```
note[rc-fallback]: src/order.kara:42:9
  `user` uses reference counting — returned from two branches with different lifetimes.
  hint: restructure to return owned value, or add #[allow(rc_fallback)] to accept.
```

If the AI can't eliminate the RC, it adds `#[allow(rc_fallback)]` explicitly. This annotation:
- Shows up in code review diffs — the reviewer sees "this value uses RC, the author accepted it"
- Is auditable and searchable — `grep allow(rc_fallback)` shows every accepted RC in the codebase
- Prevents invisible accumulation — every RC is a conscious choice, not a silent compiler decision

**2. Compiler query API reports RC density.**

`karac query ownership --summary` returns structured data about ownership decisions:

```json
{
  "module": "src/order.kara",
  "rc_values": 12,
  "arc_values": 3,
  "estimated_overhead_per_call_ns": 24,
  "hottest_rc": {"name": "user", "line": 42, "access_count_estimate": 1000}
}
```

AI agents use this as a code quality metric. A CI check can enforce: "RC density must not increase without justification."

**3. Structured RC reduction suggestions.**

The compiler doesn't just flag RC — it suggests the specific restructuring to eliminate it:

```
note[rc-fallback]: src/order.kara:42:9
  `user` uses reference counting — returned from two branches with different lifetimes.

  Option 1: Clone before the branch (eliminates RC, adds one clone):
    diff: line 40: let user = if cond { fetch_a() } else { fetch_b() };
      →  let user = if cond { fetch_a().clone() } else { fetch_b() };

  Option 2: Restructure to single return path:
    diff: [multi-line restructuring suggestion]

  Option 3: Accept RC (~2ns/access overhead):
    diff: line 39: add #[allow(rc_fallback)]
```

The AI applies the best option automatically. The human reviewer sees the choice in the diff.

**Resolution:** Mitigation #1 (default `warn_above_0`) is committed — see design.md. Every RC insertion produces a compiler note in new projects. `#[allow(rc_fallback)]` is the acknowledgement mechanism. Mitigations #2 and #3 (compiler query API extensions) are implementation-phase work tracked in the checklist.

### G11: FFI Effect Annotation Completeness

**Status: DEFERRED TO IMPLEMENTATION — Phase 8. Tracked in [implementation_checklist.md](implementation_checklist.md).**

**The concern:** Effect annotations are mandatory on `extern` functions. Wrapping OS-level syscalls for the standard library requires meticulous effect mapping. Getting an annotation wrong at the FFI boundary (e.g., marking a function `reads(FileSystem)` when it actually writes) is a soundness hole — the compiler trusts FFI annotations.

**Why this is low severity:** The mapping is straightforward — syscalls have well-defined semantics that map cleanly to the six verbs:

| Syscall category | Effect mapping |
|---|---|
| `read`, `pread`, `stat`, `readdir` | `reads(FileSystem)` |
| `write`, `pwrite`, `fsync`, `mkdir`, `unlink` | `writes(FileSystem)` |
| `send`, `sendto`, `connect` | `sends(Network)` |
| `recv`, `recvfrom`, `accept`, `listen` | `receives(Network)` |
| `clock_gettime`, `gettimeofday` | `reads(Clock)` |
| `getenv`, `getcwd` | `reads(Env)` |
| `setenv`, `chdir` | `writes(Env)` |
| `getrandom` | `reads(RandomSource)` |
| `mmap`, `brk` | `allocates(Heap)` |

This is front-loaded work, not ongoing work. ~50-100 syscall wrappers need correct annotations once. After that, all user code builds on top of safe wrappers and never touches FFI directly.

**The real risk:** Composite syscalls that have multiple effects. For example, `sendfile()` both reads from a file and sends to a socket — it needs `reads(FileSystem), sends(Network)`. Missing one effect could allow incorrect parallelization.

**Mitigation:** Standard library FFI wrappers should have integration tests that validate effect annotations against actual syscall behavior. A test that calls `sendfile` and verifies the compiler serializes it with both `writes(FileSystem)` and `reads(Network)` operations catches annotation errors before they ship.

**Next action:** During Phase 8 (standard library), create a comprehensive syscall-to-effect mapping table and test suite. Review POSIX/Linux syscall man pages for composite effects.

### G12: Integer Overflow Behavior

**Status: RESOLVED — Option A. Always trap on overflow. Explicit wrapping operations for intentional wrapping.**

Every arithmetic operation traps on overflow in all build modes (debug and release). When wrapping is intentional, use explicit methods:

```
let x: i64 = i64.MAX;
let y = x + 1;                  // RUNTIME ERROR: integer overflow
let z = x.wrapping_add(1);      // wraps to i64.MIN — explicit intent
```

**Rationale:** Consistent behavior between debug and release (unlike Rust). Silent wrapping is a bug in 99% of cases. Fits Kāra's philosophy of explicit intent — the programmer declares what they mean.

**The problem:** What happens when `let x: i64 = 9223372036854775807; x + 1` executes?

| Language | Behavior | Consequence |
|---|---|---|
| C/C++ | Undefined behavior | Security vulnerabilities for decades |
| Rust | Debug: panic. Release: silent wrap | Tests pass, production wraps — inconsistent |
| Go | Silent wrap | Same class of silent bugs |
| Python | Arbitrary precision | No overflow, but unbounded memory |
| Swift | Trap (crash) in all modes | Consistent, predictable, safe |
| C# | Silent wrap by default, `checked { }` for trapping | Opt-in safety — most code is unsafe |

**Why it matters:** Every arithmetic expression is affected. Changing this after code exists is a breaking semantic change — programs that relied on wrapping would break, programs that relied on trapping would silently change behavior.

**Options:**

**Option A — Always trap (Swift model):**
```
let x: i64 = i64.MAX;
let y = x + 1;   // RUNTIME ERROR: integer overflow
```
Provide explicit wrapping when intended:
```
let y = x.wrapping_add(1);   // wraps to i64.MIN — explicit intent
```

**Option B — Always wrap (Go model):**
```
let x: i64 = i64.MAX;
let y = x + 1;   // silently wraps to i64.MIN
```

**Option C — Rust model (mode-dependent):**
Debug builds trap, release builds wrap. Inconsistent but pragmatic.

**Recommendation:** Option A. Kāra values safety and explicit intent. Silent wrapping is a bug in 99% of cases. When you mean wrapping, say so with `wrapping_add`. This is consistent with Kāra's philosophy of "the programmer declares what, the compiler decides how."

### G13: Equality and Hashing Contract

**Status: RESOLVED — Option B. Separate traits with compiler-enforced dependency. `Hash` requires `Eq`.**

```
trait Eq {
    fn eq(self, other: ref Self) -> bool;
}
trait Hash: Eq {     // Hash requires Eq — compiler enforces
    fn hash(self) -> u64;
}
```

Some types are equatable but not hashable (valid). No type can be hashable without being equatable (enforced). `#[derive(Eq, Hash)]` generates both consistently.

**Rationale:** More flexible than bundling into a single trait, but prevents the most common bug class (inconsistent Eq/Hash). Java's biggest HashMap pitfall, eliminated at compile time.

**The problem:** If `a == b` then `hash(a)` must equal `hash(b)`. Every hash-based collection depends on this. But if `Eq` and `Hash` are separate traits, nothing prevents inconsistent implementations.

| Language | Approach | Consequence |
|---|---|---|
| Java | `equals()` and `hashCode()` are separate methods | One of the most common bug classes in Java |
| Python | `__eq__` and `__hash__` are separate | Same bugs, mutable objects in sets break |
| Rust | `Eq` and `Hash` are separate traits | `clippy` warns but doesn't prevent |
| Haskell | `Eq` is separate from `Hashable` | Same issue, less common in practice |

**Options:**

**Option A — Bundled trait:**
```
trait Hashable {
    fn eq(self, other: ref Self) -> bool;
    fn hash(self) -> u64;
}
// Implementing Hashable requires both. Can't have one without the other.
```

**Option B — Compiler-enforced dependency:**
```
trait Eq {
    fn eq(self, other: ref Self) -> bool;
}
trait Hash: Eq {     // Hash requires Eq
    fn hash(self) -> u64;
}
// Compiler: if you impl Hash, you must impl Eq.
// Compiler: if Eq changes, Hash is re-validated.
```

**Option C — Separate traits with lint (Rust model):**
Allow separate implementation, warn on inconsistency.

**Recommendation:** Option B. Separate traits are more flexible (some types are equatable but not hashable), but the compiler enforces the dependency direction. `Hash` requires `Eq`. The compiler could also verify consistency for `#[derive]`-style automatic implementations.

### G14: Float Ordering and NaN

**Status: RESOLVED — Option A. Pragmatic total order. `NaN == NaN` is `true`. NaN sorts after `+Infinity`. `f64` implements both `Eq` and `Ord`.**

`f64` has a total ordering: `-Infinity < ... < 0.0 < ... < +Infinity < NaN`. NaN equals itself for `Eq` and `Hash` purposes. HashMap with NaN keys works. `sort()` works on `Vec[f64]` with no ceremony.

**Rationale:** IEEE 754 pedantry (`NaN != NaN`) has caused more bugs than it's prevented. Languages exist to solve problems, not enforce floating-point standards. Users who need strict IEEE 754 semantics can use explicit bit-level comparison functions.

**The problem:** IEEE 754 says `NaN != NaN` and `NaN` is unordered relative to all values. This breaks total ordering, which breaks sorting.

| Language | Approach | Consequence |
|---|---|---|
| Rust | `f64: PartialOrd` but NOT `Ord` | Can't sort `Vec[f64]` without a wrapper — extremely common complaint |
| Java | `Double.compareTo`: NaN > everything | Pragmatic but mathematically wrong |
| Python | `sorted()` with NaN gives unpredictable results | Silent bugs |
| Swift | `<` returns false for NaN comparisons, sort uses `<` | NaN placement is unstable |
| Go | NaN comparisons return false, no Ord trait | Less of an issue — no generic constraints |

**Options:**

**Option A — Total order with NaN-at-end (Java-ish):**
`f64` implements `Ord`. `NaN` is defined to be greater than all non-NaN values. `sort()` just works.

**Option B — PartialOrd only (Rust model):**
`f64` doesn't implement `Ord`. Sorting requires a wrapper or explicit comparator. Mathematically correct but ergonomically painful.

**Option C — No NaN in the type system:**
`f64` traps on NaN-producing operations. NaN never exists at runtime. Radical but eliminates the entire problem class.

**Recommendation:** Option A. Pragmatic total order. `NaN` sorts to the end, `NaN == NaN` is `true` for the purpose of `Eq`. Mathematicians will object, but 99.9% of programmers want `numbers.sort()` to work. IEEE 754 pedantry has caused more bugs than it's prevented.

**Counter-argument considered:** Floats arriving via FFI (C, Rust, Python, SQL, BLAS, GPU shaders) all follow IEEE 754, where `NaN != NaN`. At FFI boundaries, a value that compares as `NaN == NaN` in Kāra would compare differently in the calling language — a real semantic divergence. However, this is an edge case: most FFI interop never compares two NaN values directly. The ergonomic benefit (`sort()`, `HashMap`) is pervasive and affects every Kāra programmer; the FFI NaN comparison edge case affects a narrow class of numerical code. Users in that class can use bit-level comparison functions. Decision stands as Option A.

### G14b: String Indexing Semantics — RESOLVED

**The problem:** `s[i]` on a UTF-8 string is ambiguous — does it return a byte, a character, or something else? Rust chose byte-offset access with a panic on non-character-boundary, which is one of Rust's most cited ergonomic pain points. For a language where AI writes most code and strings commonly contain non-ASCII text, the default must be safe and unsurprising.

**Decision (revised — see v4.md §7):** `s[i]` on `String` is a **compile error**. Use `s.char_at(i) -> Option[char]` for the i-th Unicode scalar value (O(n), no panic). Byte access requires explicit `s.bytes()[i]` (returns `u8`, panics on out-of-bounds). For iteration, `for c in s` gives characters; `for b in s.bytes()` gives bytes. When repeated indexed access is needed, convert first: `let chars: Vec[char] = s.chars().collect()`.

**Rationale:** The original decision to allow `s[i]` with a compiler warning was an admission the syntax was wrong — a warning can be ignored, a compile error cannot. The explicit method name `char_at` makes the O(n) cost impossible to miss. UTF-8 is retained; UCS-4 fixed-width would buy only O(1) indexing at the cost of 4× memory for ASCII and unpredictable per-string storage width. Most string processing uses iteration anyway; `s.char_at(i)` covers the remaining cases clearly.

### G15: Default Integer Literal Type

**Status: RESOLVED — Option A. `i64` default, `i64` indexing.**

```
let x = 42;          // i64 (default)
let y = items[x];    // no cast needed — index is i64
let z: i32 = 42;     // explicit annotation overrides default
```

Type inference for `let` bindings is already implemented (Phase 3.2). This decision sets what unadorned integer literals (`42`) infer to. Explicit type annotation (`let x: i32 = 42`) overrides the default.

**Rationale:** On 64-bit systems (everything Kāra targets), `i64` is the natural width. Using `i64` for both literals and indexing eliminates the constant `as usize` casting that plagues Rust. Users who need smaller types annotate explicitly.

**The problem:** What type is `42`? What type do array indices use?

| Language | Default int type | Array index type | Pain point |
|---|---|---|---|
| Rust | `i32` | `usize` | Constant `as usize` / `as i32` casting between them |
| Go | `int` (platform-dependent) | `int` | Different behavior on 32-bit vs 64-bit |
| Java | `int` (32-bit) | `int` | `long` for large values requires `L` suffix |
| Swift | `Int` (platform-width, usually 64) | `Int` | Clean — no casting |
| Python | Arbitrary precision | Same | No issue — but no fixed-width |

**Options:**

**Option A — `i64` default, `i64` indexing (Swift-like):**
```
let x = 42;          // i64
let y = items[x];    // no cast needed — index is i64
```

**Option B — `i32` default, `usize` indexing (Rust-like):**
```
let x = 42;          // i32

```

**Recommendation:** Option A. On 64-bit systems (which is everything Kāra targets), `i64` is the natural width. Using `i64` for both literals and indexing eliminates the most common casting friction in Rust. Users who need smaller types can annotate: `let x: i32 = 42;`.

### G16: Testing as a Language Feature

**Status: RESOLVED — Option B with co-location. `_test.kara` files in the same directory as code. `#[test]` attribute on test functions. `karac test` CLI command.**

Tests live in separate files, co-located with the code they test:

```
src/
  parser.kara
  parser_test.kara       // same directory = same module = private access
  lexer.kara
  lexer_test.kara
```

```
// parser_test.kara
#[test]
fn test_parse_expression() {
    let tokens = tokenize("1 + 2");
    let expr = parse_expression(tokens);   // private fn — accessible because same module
    assert(expr == Expr.Binary { ... });
}
```

**Key decisions:**
- `_test.kara` files are in the same module — full access to private functions, no `pub` escape hatches
- `#[test]` attribute marks test functions (already in attribute syntax)
- `karac test` discovers all `_test.kara` files recursively
- `karac build` ignores `_test.kara` files — no test code in production
- Built-in `assert()` and `assert_eq()` functions
- One convention, no exceptions — no same-file tests

**Rationale:** Co-location scales from small projects to large monorepos (separate `tests/` directory breaks down at scale). Go, Rust, Swift, Kotlin all co-locate. Single convention means every Kāra project looks the same — AI agents and humans always know where tests live.

**The problem:** Languages that didn't include testing from the start fragmented their ecosystems (C, C++, Java all have 5+ competing test frameworks). Languages that included it from day 1 (Go, Rust) have unified ecosystems.

| Language | Testing | Consequence |
|---|---|---|
| Go | `go test` + `testing` package built-in | Universal adoption, consistent project structure |
| Rust | `#[test]` + `cargo test` built-in | Same — every Rust project uses the same test runner |
| Java | JUnit (external) | Framework wars, inconsistent test discovery |
| C++ | None built-in | Google Test, Catch2, Boost.Test, CTest — fragmented |

**Options:**

**Option A — Attribute-based (Rust model):**
```
#[test]
fn test_addition() {
    assert(1 + 1 == 2);
}

#[test]
fn test_overflow() {
    assert_err(|| { let x: i64 = i64.MAX; x + 1; });
}
```
Run with `karac test`. Tests live alongside code.

**Option B — Separate test files (Go model):**
```
// In word_count_test.kara
fn test_count_words() {
    let result = count_words("hello world");
    assert(result == 2);
}
```
Files ending in `_test.kara` are test files. `karac test` discovers them.

**Option C — Both:**
Support `#[test]` attributes AND `_test.kara` file convention. `karac test` runs both.

**Recommendation:** Option A with `_test.kara` convention too. `#[test]` is minimal syntax — just an attribute, which Kāra already supports. The effect system makes testing especially powerful — `with_provider` for dependency injection in tests is already designed. A built-in `assert()` and `assert_eq()` are just functions, not language features.

**What to decide now:** Add `#[test]` to the known attributes table and plan `karac test` as a CLI command. The rest can be filled in during implementation.

### G17: Variable Shadowing Rules

**Status: RESOLVED — Option A. Allow shadowing in all scopes, including cross-scope.**

```
let x = "42";
let x = parse_int(x);    // same-scope shadowing — transformation chain

let y = 10;
if condition {
    let y = 20;           // cross-scope shadowing — allowed
    process(y);
}
// y is still 10 here
```

**Rationale:** Shadowing bugs are logic errors, not language safety concerns. The compiler prevents memory bugs, type bugs, and concurrency bugs — variable naming is the programmer's responsibility. Cross-scope shadowing is common in practice (using the same name inside conditional blocks). Logic errors from shadowing are caught by unit tests, not by compiler restrictions.

**The problem:** Can you rebind a variable name in the same scope?

```
let x = "42";
let x = parse_int(x);    // shadow with different type — is this allowed?
```

| Language | Shadowing | Consequence |
|---|---|---|
| Rust | Allowed, even with type change | Useful for transformations, can hide bugs |
| Go | Allowed with `:=`, source of subtle bugs | Go 1.22 added a vet check for some cases |
| Java | Not allowed in same scope | Safer but more verbose (`parsedX`) |
| Swift | Allowed | Same tradeoffs as Rust |

**Options:**

**Option A — Allow shadowing (Rust model):**
```
let input = read_line();        // String
let input = input.trim();       // ref String — shadowed, old binding inaccessible
let input = parse_int(input);   // i64 — shadowed again
```
Useful for transformation pipelines. Old binding is inaccessible after shadowing.

**Option B — Disallow shadowing:**
```
let input_raw = read_line();
let input_trimmed = input_raw.trim();
let input = parse_int(input_trimmed);
```
Forces unique names. Verbose but no hidden rebinding.

**Recommendation:** Option A. Shadowing is especially valuable in a language with ownership — after a value is moved, shadowing the name with the result avoids needing a new variable name for what is logically the same data at a different stage. Rust's experience shows this is a net positive.

### G18: Zero/Default Value Policy

**Status: RESOLVED — Option A. Require explicit initialization. No zero values.**

```
let x: i64;          // declared but not usable
print(x);            // ERROR: x is not initialized
x = 5;
print(x);            // OK
```

**Rationale:** Consistent with ownership model — a value that was never assigned has no meaningful owner. Prevents Go-style bugs where uninitialized variables silently have zero/empty values. The compiler enforces that every variable is assigned before use.

**The problem:** Can a variable exist without an explicit initial value?

| Language | Policy | Consequence |
|---|---|---|
| Go | Everything has a zero value | Use of uninitialized variables with surprising defaults. `nil` map access panics |
| Rust | Must be explicitly initialized | Safer. `let x: i64;` is an error unless assigned before use |
| Java | Fields default to zero/null; locals must be initialized | Inconsistent rules |
| C/C++ | Uninitialized = undefined behavior | Decades of security bugs |

**Options:**

**Option A — Require explicit initialization (Rust model):**
```
let x: i64;            // OK — declared but not yet usable
println(x);            // ERROR: x is not initialized
x = 5;
println(x);            // OK
```

**Option B — Zero values (Go model):**
```
let x: i64;            // x is 0
let s: String;         // s is ""
let opt: Option[i64];  // opt is None
```

**Recommendation:** Option A. Explicit initialization prevents an entire class of bugs and is consistent with Kāra's ownership model — a value that was never assigned has no meaningful owner. This is already how the ownership checker would need to work (you can't move something that doesn't exist).

### G19: Implicit Numeric Widening

**Status: RESOLVED — Option A refined. Implicit guaranteed-lossless widening only.**

Allowed (always lossless):
```
i8  → i16, i32, i64           // integer widening
i16 → i32, i64
i32 → i64
i32 → f64                     // 32-bit int fits in f64's 53-bit mantissa
f32 → f64                     // float widening
```

Blocked (requires explicit `as` cast):
```
i64 → f64                     // can lose precision above 2^53
any narrowing                  // i64 → i32, f64 → f32, etc.
float → int                   // f64 → i64 (truncation)
```

```
let a: i32 = 42;
let b: i64 = a;       // OK — lossless widening
let c: f64 = a;       // OK — i32 fits in f64
let d: f64 = b;       // ERROR — i64 → f64 can lose precision
let d: f64 = b as f64; // OK — explicit, programmer accepts precision loss
```

**Rationale:** Balances safety with ergonomics. No silent data loss ever. The compiler just checks a static widening table — ~10 lines in the type checker, zero runtime cost. The only common case that requires an explicit cast is `i64 → f64`, which is one keyword (`as f64`) when needed.

**The problem:** Should `let x: i64 = some_i32;` compile without an explicit cast?

| Language | Policy | Consequence |
|---|---|---|
| C/C++ | Implicit widening AND narrowing | Truncation bugs everywhere |
| Java | Implicit widening only | Subtle `int → float` precision loss |
| Rust | No implicit conversions | Safe but verbose: `x as i64` everywhere |
| Go | No implicit conversions | Same verbosity |
| Swift | No implicit conversions | Same |

**Options:**

**Option A — Implicit widening only, never narrowing:**
```
let a: i32 = 42;
let b: i64 = a;       // OK — i32 → i64 is lossless
let c: i32 = b;       // ERROR — i64 → i32 could lose data, need explicit cast

let d: f32 = 1.5;
let e: f64 = d;       // OK — f32 → f64 is lossless
let f: f32 = e;       // ERROR — f64 → f32 could lose precision
```

**Option B — No implicit conversions (Rust model):**
```
let a: i32 = 42;
let b: i64 = a;       // ERROR — must write: a as i64
```

**Recommendation:** Option A for integer widening (i8 → i16 → i32 → i64 is always safe). Be cautious about int → float widening (i64 → f64 can lose precision for large values). Narrowing always requires explicit cast. This balances safety with ergonomics — you never lose data silently, but you don't need casts for obviously safe conversions.

### G21: Standard Error Trait Design

**Status: DEFERRED TO IMPLEMENTATION — Phase 8. Tracked in [implementation_checklist.md](implementation_checklist.md).**

**The problem:** Kāra has `Result[T, E]` and `?`. But without a standard `Error` trait, every library invents its own error type and they don't compose.

| Language | Approach | Consequence |
|---|---|---|
| Rust | `std.error.Error` trait + ecosystem crates (`anyhow`, `thiserror`) | Trait exists but ecosystem fragmented on ergonomics |

| Java | `Exception` class hierarchy | Checked exceptions are universally hated |

trait Error {

### G22: Loop Variable Capture in Closures

**Status: RESOLVED — Option A. Each loop iteration creates a fresh binding. Closures capture the value at that iteration.**

```
let callbacks = Vec.new();
for i in 0..5 {
    callbacks.push(|| { print(i); });
}
// Prints: 0 1 2 3 4 (each closure captured its own copy of i)
```

**Rationale:** This is what users expect. Go shipped the alternative (shared variable), had a decade of bugs, and switched to fresh binding in Go 1.22. JavaScript had the same problem with `var` (fixed by `let`). No practical downsides — negligible memory overhead, and if someone needs shared state across closures they use an explicit shared variable.

**The problem:** When a closure is created inside a loop, does it capture the loop variable's current value or a reference to it?

| Language | Behavior | Consequence |
|---|---|---|
| Go (pre-1.22) | Capture by reference | Most common Go bug for 10 years. Fixed in Go 1.22 |
| JavaScript (`var`) | Capture by reference | Classic bug, fixed by using `let` instead |
| Python | Late binding | `[lambda: x for x in range(5)]` — all return 4 |
| Rust | Capture by value or reference (explicit via `move`) | Explicit, no surprise |
| Swift | Capture by reference, but value types copy | Less surprising for value types |

```
// The bug (Go pre-1.22 / JS var / Python):
let callbacks = Vec.new();
for i in 0..5 {
    callbacks.push(|| { print(i); });
}
// User expects: 0 1 2 3 4
// Actually prints: 4 4 4 4 4 (all captured the same variable)
```

**Options:**

**Option A — Each iteration gets its own binding (Go 1.22 fix):**
The loop variable is a fresh binding per iteration. Closures capture the value at that iteration.

**Option B — Explicit `move` for closures (Rust model):**
Default captures by reference. `move || { print(i); }` captures by value.

**Option C — Always capture by value for loop variables:**
Loop variables are always copied into closures, even without `move`.

**Recommendation:** Option A. Each loop iteration creates a fresh binding. This is what Go 1.22 adopted after a decade of bugs. It matches user intent 99% of the time. Combined with Kāra's ownership model (values are moved or copied, not silently shared by reference), this should be the natural behavior — loop variables are logically a new value each iteration.

### G23: Derive / Compiler-Synthesized Traits

**Status: RESOLVED — Option A. `#[derive]` as compiler built-in. Explicit opt-in per struct.**

```
#[derive(Eq, Hash, Display)]
struct Point { x: i64, y: i64 }
// Compiler generates Eq (field-by-field ==), Hash (field-by-field hash), Display (formatted string)
```

**Derivable traits (initial set):**
- `Eq` — field-by-field equality
- `Hash` — hash-combine all fields (requires `Eq` per G13)
- `Display` — `TypeName(field1=val1, field2=val2)` format
- `Clone` — field-by-field clone

**Later (Phase 8+):**
- `Ord` — lexicographic field ordering
- `Serialize` / `Deserialize` — field-by-field to/from JSON/binary (G28)

**Rationale:** Explicit `#[derive]` over auto-generation because: not every struct should be equatable (e.g., `DatabaseConnection`), not every struct should be hashable (float-containing structs), and `Clone` can be expensive for large types. One line of opt-in is the right trade-off — conscious intent without boilerplate. No macro system needed — the compiler knows struct fields and generates implementations mechanically.

**The problem:** Without `#[derive]`, users must manually implement `Eq`, `Hash`, `Display`, and eventually `Serialize` for every struct and enum. This is massive boilerplate that makes the language impractical for real-world use.

```
// Without derive — user writes this for EVERY struct:
struct User { name: String, age: i64 }

impl Eq for User {
    fn eq(self, other: ref User) -> bool {
        self.name == other.name && self.age == other.age
    }
}

impl Hash for User {
    fn hash(self) -> u64 {
        hash_combine(self.name.hash(), self.age.hash())
    }
}

impl Display for User {
    fn to_string(self) -> String {
        f"User(name={self.name}, age={self.age})"
    }
}
```

```
// With derive — one line:
#[derive(Eq, Hash, Display)]
struct User { name: String, age: i64 }
```

**How others handle it:**

| Language | Approach | Mechanism |
|---|---|---|
| Rust | `#[derive(Debug, Clone, Eq, Hash)]` | Proc macros (complex build system) |
| Swift | Auto-synthesized `Equatable`, `Hashable`, `Codable` | Compiler knows struct fields, generates conformance |
| Kotlin | `data class` keyword | Compiler generates `equals()`, `hashCode()`, `toString()`, `copy()` |
| Go | None | Manual implementation — top complaint |
| Java | Lombok `@Data` or IDE generation | External tooling, not language |
| Python | `@dataclass` decorator | Runtime metaprogramming |

**Options:**

**Option A — Compiler-built-in derive (Swift model):**
`#[derive(Eq, Hash, Display)]` is a compiler directive. The compiler knows the struct's fields and generates the trait implementations. No macro system needed. Works for a predefined set of traits.

**Option B — `data struct` keyword (Kotlin model):**
```
data struct User { name: String, age: i64 }
// Auto-generates: Eq, Hash, Display, Clone
```
Single keyword, opinionated set of generated traits.

**Option C — Defer until comptime/macros exist:**
Wait for a general-purpose metaprogramming system. Risk: every struct is boilerplate until then.

**Recommendation:** Option A. The compiler already has all the information it needs (field names, field types, field order). For a defined set of "derivable" traits (`Eq`, `Hash`, `Display`, `Clone`, and later `Serialize`), the compiler can mechanically generate implementations by iterating over fields. This is exactly what Swift does — no macro system, just compiler knowledge.

**Derivable traits (initial set):**
- `Eq` — field-by-field equality
- `Hash` — hash-combine all fields
- `Display` — `TypeName(field1=val1, field2=val2)` format
- `Clone` — field-by-field clone

**Later (Phase 8+):**
- `Serialize` / `Deserialize` — field-by-field to/from JSON/binary
- `Ord` — lexicographic field ordering

### G24: Optional Chaining (`?.` and `??`)

**Status: RESOLVED — Option A. Add `?.` and `??` operators. No `!` operator.**

```
let name = user.address?.city?.name ?? "unknown";
// ?.  → short-circuits to None if any level is None
// ??  → provides default if the chain results in None
```

**Syntax (same as TypeScript/Swift/Kotlin):**
- `expr?.field` — if `None`, result is `None`; if `Some(val)`, result is `val.field`
- `expr?.method()` — if `None`, result is `None`; if `Some(val)`, result is `val.method()`
- `expr ?? default` — if `None`, result is `default`; if `Some(val)`, result is `val`
- `??` has very low precedence (lower than `||`)

**No `!` operator.** TypeScript's `!` (non-null assertion) is a workaround for a weak type system. Kāra's `unwrap()` + `panics` effect (G26) serves the same purpose with compile-time tracking — strictly better.

**The problem:** Deeply nested `Option` access requires verbose pattern matching:

```
// Current Kāra — verbose:
match user.address {
    Some(addr) => match addr.city {
        Some(city) => match city.name {
            Some(name) => print(name),
            None => print("unknown"),
        },
        None => print("unknown"),
    },
    None => print("unknown"),
}
```

Every modern language has solved this:

| Language | Optional chaining | Default value |
|---|---|---|
| Swift | `user.address?.city?.name` | `?? "unknown"` |
| Kotlin | `user.address?.city?.name` | `?: "unknown"` |
| TypeScript | `user.address?.city?.name` | `?? "unknown"` |
| C# | `user?.Address?.City?.Name` | `?? "unknown"` |
| Rust | `user.address.as_ref().and_then(\|a\| a.city.as_ref())...` | `.unwrap_or("unknown")` |

Rust is the outlier — no optional chaining syntax. It's one of Rust's most requested features.

**Options:**

**Option A — `?.` operator + `??` for default:**
```
let name = user.address?.city?.name ?? "unknown";
// Type: String (resolved by ??)
```
`?.` short-circuits to `None` if any level is `None`. `??` provides a default value if the chain results in `None`.

**Option B — Method chaining only (Rust model):**
```
let name = user.address
    .and_then(|a| a.city)
    .and_then(|c| c.name)
    .unwrap_or("unknown");
```

**Recommendation:** Option A. `?.` and `??` are simple, readable, and universally understood. They affect the lexer (two new token types) and parser (postfix operator + infix operator). Minimal implementation cost, massive ergonomic gain.

**Syntax details:**
- `expr?.field` — if `expr` is `None`, result is `None`; if `Some(val)`, result is `val.field`
- `expr?.method()` — if `expr` is `None`, result is `None`; if `Some(val)`, result is `val.method()`
- `expr ?? default` — if `expr` is `None`, result is `default`; if `Some(val)`, result is `val`
- `??` has very low precedence (lower than `||`)

### G25: `?` Operator for Option (not just Result)

**Status: RESOLVED — Yes. `?` works for both `Option` and `Result`.**

```
// For Option — in a function returning Option[T]:
fn get_city(user: User) -> Option[String] {
    let addr = user.address?;     // returns None if None
    let city = addr.city?;        // returns None if None
    Some(city.name)
}

// For Result — in a function returning Result[T, E]:
fn load_user(path: String) -> Result[User, Error] {
    let data = read_file(path)?;  // returns Err if Err
    parse_json(data)?
}
```

Desugaring:
- Option: `expr?` → `match expr { Some(val) => val, None => return None }`
- Result: `expr?` → `match expr { Ok(val) => val, Err(e) => return Err(e) }`

**Rationale:** Type checker determines which desugaring to use based on the function's return type. Same syntax, same mental model. Rust does this and it's universally loved.

**The problem:** The `?` operator is currently defined only for `Result`:

```
// Current: works for Result
fn load_user(id: u64) -> Result[User, Error] {
    let data = read_file(f"users/{id}.json")?;  // propagates Err
    parse_user(data)?
}
```

Should it also work for `Option` in functions returning `Option`?

```
// Proposed: also works for Option
fn get_city_name(user: User) -> Option[String] {
    let addr = user.address?;     // returns None if None
    let city = addr.city?;        // returns None if None
    Some(city.name)
}
```

**How others handle it:**
- **Rust:** `?` works for both `Option` and `Result`. Universally loved.
- **Swift:** `guard let` + early return serves the same purpose.
- **Kotlin:** `?:` + `return` — `val addr = user.address ?: return null`

**Recommendation:** Yes. `?` should work for both `Option` (in functions returning `Option[T]`) and `Result` (in functions returning `Result[T, E]`). The desugaring is straightforward:

```
// For Option:
expr?  →  match expr { Some(val) => val, None => return None }

// For Result:
expr?  →  match expr { Ok(val) => val, Err(e) => return Err(e) }
```

This is a type checker concern (check that the return type matches) not a parser concern (`?` syntax already exists).

### G26: `unwrap()` and the Panics Effect

**Status: RESOLVED — Option A. `unwrap()` produces `panics` effect. Tracked through call chain.**

```
fn get_first(items: Vec[i64]) -> i64 panics {
    items.first().unwrap()   // panics effect inferred
}

// Public function without panics — compiler prevents unwrap()
pub fn safe_first(items: Vec[i64]) -> i64 {
    items.first() ?? 0       // must use safe alternative
}
```

**Panic-producing methods:**
- `unwrap()` — panics if None/Err
- `expect("message")` — panics with custom message
- `unwrap_or(default)` — no panics (safe, returns default)
- `unwrap_or_else(f)` — no panics (safe, calls f)

**Rationale:** Minimal compiler complexity — `panics` is just one more entry in the existing effect table. The primary value is completeness of the effect system, code review visibility (diffs that add `panics` are red flags), and library-level guarantees (a library with zero `panics` in its public API is crash-free). The real safety comes from `Option`, `Result`, `?`, and `??` making `unwrap()` avoidable — `panics` tracking is the last line of defense.

**The problem:** How do you get the value out of an `Option` or `Result` when you're "sure" it's `Some`/`Ok`?

| Language | Approach | Consequence |
|---|---|---|
| Rust | `.unwrap()` panics at runtime | Convenient but crashes in production |
| Swift | `!` force unwrap | Same — runtime crash |
| Kotlin | `!!` operator | Same — NullPointerException |
| Go | No optionals — everything is nullable | NPE-equivalent panics |

**Options:**

**Option A — `unwrap()` requires `panics` effect:**
```
fn get_first(items: Vec[i64]) -> i64 panics {
    items.first().unwrap()   // panics if None — effect propagates
}

// Callers must handle it:
pub fn process(items: Vec[i64]) -> i64 panics {  // must declare panics
    get_first(items)
}
```
The effect system tracks where panics can happen. Public functions must declare it. Code review shows `panics` spreading.

**Option B — `unwrap()` exists, no special tracking:**
Just a method that panics. No effect integration.

**Option C — No `unwrap()` at all:**
Force pattern matching or `?` or `??` always. Safest but most verbose.

**Recommendation:** Option A. This is where Kāra's effect system shines uniquely. No other language can do this — `panics` is already a built-in effect verb. Making `unwrap()` produce a `panics` effect means the compiler tracks exactly which functions can crash and forces public APIs to declare it. This is genuinely novel: compile-time panic tracking through the effect system.

**Additional methods to provide:**
- `unwrap()` — panics if None/Err (requires `panics` effect)
- `unwrap_or(default)` — returns default if None/Err (no panics, pure)
- `unwrap_or_else(f)` — calls f to produce default (effects of f propagate)
- `expect("message")` — panics with custom message (requires `panics` effect)

### G27: NaN Equality Semantics

**Status: RESOLVED — `NaN == NaN` is `true`. Consistent with G14 (total order).**

Follows directly from G14. `f64` implements both `Eq` and `Ord` with pragmatic total order: `-Infinity < ... < +Infinity < NaN`. NaN equals itself, can be used as a HashMap key, and sorts predictably. No separate decision needed.

**The problem:** Beyond ordering (G14), NaN needs a complete semantics story.

**IEEE 754 says:**
- `NaN != NaN` (reflexive equality violated)
- `NaN` is unordered relative to all values
- Any comparison with `NaN` returns `false`

**Real-world consequences:**
```
let x = 0.0 / 0.0;     // NaN

// IEEE 754 behavior:
x == x       // false (!!!)
x != x       // true  (the only value where this is true)
x < 0.0      // false
x > 0.0      // false
x == 0.0     // false

// HashMap nightmare:
map.insert(x, "value");
map.get(x)    // None! Key can never be found because NaN != NaN
```

**Options:**

**Option A — Pragmatic equality (NaN == NaN is true):**
`f64` has total ordering and reflexive equality. `NaN` equals itself, sorts to the end. IEEE 754 purists object, but HashMap works, sorting works, everything is predictable.

**Option B — IEEE 754 strict (NaN != NaN):**
Mathematically correct. HashMap with NaN keys is a bug (compiler could warn). Sorting requires explicit NaN handling.

**Option C — NaN-free floats:**
Operations that would produce NaN instead return `Result[f64, MathError]`. `0.0 / 0.0` returns `Err(DivisionUndefined)`. NaN never exists at runtime. Radical but eliminates the entire problem.

**Recommendation:** Option A. Pragmatic total order. Languages exist to solve problems, not to enforce IEEE 754 pedantry. `NaN == NaN` is `true`, `NaN` sorts after `Infinity`, HashMap with NaN keys works. This is what most programmers expect. The 0.01% who need strict IEEE 754 behavior can use explicit bit-level comparison functions.

Combined with G14 (float ordering), this gives Kāra a complete, predictable story: `f64` implements `Eq` and `Ord` with total ordering. No special cases, no surprises.

### G28: Serialization / Deserialization

**Status: OPEN — Critical. Depends on G23 (derive). Target: Phase 8.**

**The problem:** Every web server, every API client, every config file reader needs to convert between structs and data formats (JSON, TOML, YAML, binary). Without a serialization story, Kāra is impractical for web development.

**How others handle it:**

| Language | Approach | Ergonomics |
|---|---|---|
| Rust | `serde` + `#[derive(Serialize, Deserialize)]` | Excellent — one line per struct |
| Go | `encoding/json` + struct tags + reflection | Good — struct tags control field names |
| Swift | `Codable` protocol + compiler synthesis | Excellent — just conform to `Codable` |
| Python | `json.dumps()/loads()` + dataclasses | Trivial — dynamic typing |
| Java | Jackson + annotations + reflection | Verbose but capable |

**Kāra's path (depends on G23):**

If derive is compiler-built-in (G23 Option A), then serialization becomes:

```
#[derive(Serialize, Deserialize)]
struct User {
    name: String,
    age: i64,
    #[serialize(rename: "email_address")]
    email: String,
}

// Serialize
let json = User.to_json(user);    // writes(FileSystem) if writing to file

// Deserialize
let user = User.from_json(json)?; // Result[User, ParseError]
```

The compiler generates `to_json()` / `from_json()` by iterating over struct fields — same mechanism as derive for `Eq` and `Hash`.

**What to decide now:**
1. Serialization depends on derive (G23) — solve that first
2. Attribute syntax for field customization (`#[serialize(rename: "...")]`) — decide whether attributes can have string values
3. Which formats to support in stdlib (JSON at minimum, probably TOML for config)

**Next action:** Resolve G23 (derive) first. Then design the `Serialize`/`Deserialize` traits.

### G29: Channels / Inter-Task Communication

**Status: OPEN — Medium severity. Should decide before Phase 5.**

**The problem:** Auto-concurrency handles fork-join parallelism. But many real systems need tasks that communicate while running:

```
// Producer-consumer — can't express with fork-join alone
let ch = Channel.new();

// Producer task
spawn(|| {
    for item in data {
        ch.send(process(item));    // sends to channel
    }
    ch.close();
});

// Consumer
for result in ch {
    aggregate(result);
}
```

**Use cases that need channels:**
- Web servers: request queues, connection pools
- Game engines: job systems, event buses
- Data pipelines: streaming transformations
- Actor model: message passing between independent agents

**How others handle it:**

| Language | Approach |
|---|---|
| Go | `chan T` — first-class channels with `select` |
| Rust | `std.sync.mpsc` — multi-producer single-consumer channels |
| Erlang | Mailboxes — built into the actor model |
| Java | `BlockingQueue`, `CompletableFuture` |

**Questions to answer:**
- Are channels a language feature (`chan T`) or stdlib type (`Channel[T]`)?
- What effects does `send`/`receive` on a channel carry?
- How do channels interact with auto-concurrency? Can the compiler auto-parallelize into channel-connected stages?
- `select` over multiple channels — language feature or library?

**Recommendation:** Stdlib type, not language feature. `Channel[T]` with `send()` and `receive()` methods. Channel operations carry effects that the compiler can reason about. This can wait until Phase 5 when the concurrency runtime exists.

### G30: `#[no_std]` / Bare Metal Support

**Status: OPEN — Low severity. Can defer.**

**The problem:** OS kernels, bootloaders, and embedded systems can't use heap allocation or standard library I/O. They need a way to opt out.

| Language | Approach |
|---|---|
| Rust | `#![no_std]` — disables stdlib, allows `core` only |
| Zig | No implicit allocations — allocator is always explicit |
| C | No stdlib by default — you choose what to link |

**Questions:**
- Can Kāra compile without a heap? (No RC, no Vec, no String)
- Can the effect system enforce no-heap? (`#[no_effect(allocates(Heap))]`)
- Is this even a goal? Only Rust, C, C++, and Zig play in this space.

**Recommendation:** Defer. Document as a future possibility. The effect system actually provides a natural path — `allocates(Heap)` is already a built-in effect. A `#[no_effect(allocates(Heap))]` module-level attribute could enforce no-heap allocation in the future without language changes. But building this now adds complexity for a niche use case.

## Null / None / NaN — Complete Story

Kāra's approach to "absence of value" across the entire language, consolidated from individual gap items.

### The Three Kinds of "Nothing"

| Concept | What it means | Kāra's answer | Status |
|---|---|---|---|
| **Null / None** | Value doesn't exist | `Option[T]` — `Some(v)` or `None` | Designed |
| **Error** | Operation failed | `Result[T, E]` — `Ok(v)` or `Err(e)` | Designed |
| **NaN** | Math undefined | G27 — total order, `NaN == NaN` is `true` | **OPEN** |

### What Kāra Gets Right (Already Designed)

1. **No null keyword.** Values are `T` (exists) or `Option[T]` (might not). Compiler enforces handling.
2. **No implicit conversions.** `Option[i64]` is not `i64`. Can't accidentally use `None` as `0`.
3. **Pattern matching forces handling.** Incomplete match is a compile error.
4. **`?` operator.** Propagates `None`/`Err` without boilerplate (G25: extend to Option too).
5. **Effect tracking for panics.** `unwrap()` produces `panics` effect (G26).

### What Needs Decisions

1. **Optional chaining** (G24): `user.address?.city?.name ?? "unknown"` — avoids nested matching.
2. **`?` for Option** (G25): `let addr = user.address?;` in functions returning `Option[T]`.
3. **`unwrap()` semantics** (G26): Requires `panics` effect — compile-time panic tracking.
4. **NaN equality** (G27): `NaN == NaN` is `true` — pragmatic total order.
5. **`??` default operator** (G24): `value ?? default` — unwrap with fallback.

### JavaScript's Disaster (What Kāra Avoids)

JavaScript has seven falsy values (`false`, `0`, `""`, `null`, `undefined`, `NaN`, `-0`), implicit type coercion between them, and two equality operators (`==` vs `===`). This produces:

```javascript
null == undefined     // true
null === undefined    // false
NaN == NaN            // false
"" == false           // true
0 == false            // true
null == false         // false (inconsistent!)
```

Kāra avoids ALL of this by having:
- No null (Option instead)
- No implicit conversions (strong typing)
- One equality operator (`==` always structural via Eq trait)
- No falsy/truthy coercion (`if` requires `bool`, period)
- NaN == NaN is true (G27, if adopted)
