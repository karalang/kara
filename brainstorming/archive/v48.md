## Findings

**Phantom gaps (C5, C6).** Both flagged as "possible gap" in the original scoping turned out fully specified in canonical docs:

- C5: `extern "C"` defaults to `{blocks}`, `extern "C-unwind"` adds `panics`; `@noblock` opts out for pure-CPU helpers; resource effects are programmer-declared under a trust-not-verify rule at the FFI boundary. Complete contract.
- C6: Integer overflow always traps in every build mode (stricter than Rust's debug-trap/release-wrap split). Full `checked_*` / `wrapping_*` / `saturating_*` / `overflowing_*` method families ship per integer primitive in v1.

**C3 — one micro-fix.** Float reassoc is IEEE-strict globally (no `@fast` attribute planned; the proposed attribute is out of step with Kāra's "costs are legible" posture and can reopen only if a concrete program needs FMA/associative rewrites). Hashmap hasher is DoS-resistant and seeded per process, which implies non-deterministic iteration across runs; `TreeMap[K, V]` is the sorted-iteration opt-in. The `Hash` / `Hasher` stability policy stated the seeding rule, but the `Map[K, V]` method-table block did not — a reader encountering `for (k, v) in map` would not know to reach for `TreeMap` for stable iteration. Added one sentence to the `Map[K, V]` block making the non-determinism explicit at the type surface and pointing to `TreeMap`.

**Deferred items (C1, C2, C4, C7, C8).** Each migrated to `docs/deferred.md` with enough shape to pick up cold when the respective work begins:

- C1 — post-v1 tooling waiting for the diagnostic catalogue to stabilize (need ~20+ diagnostics shipped before enumerating the class enum).
- C2 — pure tooling waiting for real consumers (LLM agents, IDE plugins); builds in concert with C1 given the shared JSONL schema vocabulary.
- C4 — bounded audit (~30–60 min) that produces a useful decision only once Phase 4+ real programs exist; current examples are spec-illustration sized.
- C7 — comptime architectural choice (AST→AST vs value-level `const fn`), waiting for comptime itself to come into scope.
- C8 — runtime tracing, blocked on mature codegen and scheduler (Phase 8+).

**C9 folds into C1** per the candidate's own note. Target-incompatibility errors are one class in the shared error-class enumeration, not a separate diagnostic category.

## Non-candidates (decisions visible, for record)

LLM-friendliness-maximizing ideas that Kāra declined, documented here so the choice is on the record:

- **AST-as-source-of-truth** — premise fork; discards the lever that humans can read and mentally execute code.
- **Content-addressable identifiers (`fn#a3b2f17c`)** — requires an ecosystem (migration contracts, view renderer, hash registry) out of scope for a solo timeline.
- **Mnemonic-free internal names** — contradicts `memory/project_design_principle.md` (mainstream-legible syntax).
- **"No hand-written comments — intent lives in spec blocks"** — comments-for-non-obvious-WHY is already Kāra's stance; spec-block replacement depends on full SMT verification, deferred to Level 3–4.
- **Everything-first-party stdlib** — scope explosion; contradicts "user code can grow the ecosystem."
- **Panics reserved for compiler-proved unreachable** — Kāra's broader `panics` surface (runtime contract violations, refinement-type `as` conversions) is intentional under Levels 1–2; a stricter stance requires SMT verification the current level does not earn.
