## Framing

Client-side browser app development is today restricted to JS or WASM-on-top-of-JS-host. Both are mature, but both inherit a platform (HTML/CSS/JS + the DOM + same-origin security + URL-based distribution) that was designed for documents and retrofitted for apps. Every modern "better app" attempt is a layer on top: SPA frameworks, virtual DOMs, service workers, COOP/COEP, Electron, Tauri, React Native, Flutter, Capacitor. Each closes one gap and opens another.

The thought experiment: go back to the drawing board, no scope or time constraints (new runtime, new browser, new OS, all on the table). How would you actually solve the fragmentation?

## Problem statement

The browser's original sin is trying to be both a **document viewer** and an **application runtime**. HTML/CSS/JS is fine for documents — articles, blogs, reference pages, wikis. Retrofitting it for apps has produced layer-upon-layer of workaround:

- **Two+ language problem** (JS for logic, HTML for structure, CSS for styling — different mental models stitched by frameworks).
- **WASM is a second-class citizen**: can't touch DOM directly, every host call is an FFI boundary, GC integration is recent, threads require COOP/COEP header negotiation.
- **The DOM**: retained-mode tree designed for documents, retrofitted with accessibility, animation, layout-thrashing mitigations.
- **Rendering model**: everything through the browser compositor; GPU access mediated through WebGPU/WebGL host calls.
- **Security**: origin-based (correct for documents, wrong for apps) — hence CORS, COOP/COEP, cookies-as-ambient-auth, permission prompts per API.
- **Distribution**: "visit URL, trust whatever JS the server serves today."
- **Capabilities**: filesystem, USB, Bluetooth, camera, clipboard — each behind a different permission model, inconsistent across browsers.
- **Native fragmentation**: iOS / Android / desktop each want different wrappers (Cordova, Capacitor, Tauri, React Native, Flutter, MAUI).

## Proposed shape

**An application runtime where the WASM Component Model is the system ABI — not a browser feature, a full platform.**

Concretely:

- **App binaries are WASM components.** All capabilities (GPU, filesystem, network, audio, camera, clipboard, input, sensors) are typed WIT interfaces. No DOM. No HTML. No CSS.
- **UI is a library choice**, not a platform mandate. Skia/immediate-mode, Flutter-style retained tree, imgui-style, or a DOM-like retained tree — the runtime renders a portable pixel buffer via GPU; UI framework ships as a library the app binary depends on.
- **Capability-scoped security replaces same-origin.** Each capability is a typed token the app requests on install and can delegate. This kills CORS / COOP / COEP / cookies-as-ambient-auth as categories.
- **Real threads by default.** OS processes or real OS threads, not "Web Workers pretending via SharedArrayBuffer behind a header negotiation."
- **One runtime ships everywhere:**
  - Desktop: native executable running the runtime + the app bundle.
  - Mobile: app-store signed bundle (satisfies iOS/Android distribution requirements).
  - Browser: compatibility shim runs the same WASM component inside a browser tab via WebGPU + Workers + SharedArrayBuffer. **Zero install** — no plugin, no extension, no browser modification (see "Installation model" below). You pay the web tax only here, and only during transition.
  - Embedded: runtime runs on-device directly.
  The *same* signed bundle runs on all of them.
- **Distribution is signed bundles**, not "visit URL and re-download JS on every page load." Bundles are small (<1MB target after dead-code elimination), cached after first fetch. You get near-instant install without the "new untrusted code on every visit" insecurity.

## Installation model (what the user actually experiences)

A common early confusion: "does this runtime need to be installed in every browser before it can be used?" **No.** Three delivery paths, same signed bundle, only one requires anything installed anywhere:

- **Browser (zero install).** User clicks a URL. Web server returns a tiny HTML+JS loader (~5-10KB) alongside the signed WASM bundle. Loader runs the component in a `<canvas>` using the browser's **already-shipped** WASM engine + WebGPU + Web Workers + SharedArrayBuffer. **No plugin, no extension, no browser modification** — works in any modern browser as-is, indistinguishable from a normal webapp load to the user.
- **Desktop / mobile (one-time runtime install).** The application runtime ships as a small system-level runtime — equivalent of installing the JVM or .NET runtime once — or is bundled with each app binary. Apps fetched from web or app store run against the installed (or bundled) runtime.
- **Embedded.** Runtime ships on device firmware directly; no user-facing install step.

**The browser path is load-bearing for adoption.** Without it, the application runtime would face the cold-start problem every new platform faces ("why would I build for a runtime no one has?"). With it, apps can ship against the runtime *today* with zero-install delivery to every browser user, while the native runtime installs accumulate over time as the ecosystem grows. The browser is effectively a free, universal, pre-installed host for the runtime — because everything the runtime needs (WASM engine, WebGPU, Workers, SAB) is already in every modern browser.

## Developer mental model (JVM-shaped, not CDK-shaped)

From a developer's perspective, AR follows the familiar **SDK + runtime + portable bundle** pattern that JVM, .NET, and Flutter all use:

- **SDK** — build toolchain that compiles source (in any language that targets WASM Component Model — Rust, Kāra, Go, Swift, Zig, C#) against the AR's WIT capability interfaces. Produces a signed WASM-component bundle.
- **Runtime** — installed once per platform (desktop / mobile / embedded), shared by every AR app on that platform. Equivalent to installing the JVM once, **not** per-app-bundled engines like Flutter ships today. For browsers specifically, a ~5–10KB loader ships alongside the bundle and runs it against the browser's already-shipped WASM engine + WebGPU + Workers primitives — no install anywhere on the browser side.
- **Bundle** — the deliverable. Signed, under 1MB target after dead-code elimination, same bytes on every target. Developer builds once, ships everywhere.

This is **not** AWS CDK / Pulumi / Terraform-shaped — those are infrastructure-as-code frameworks for provisioning cloud resources. AR is a runtime-portable application ABI. The closest analogies are the runtime ancestors: **JVM** (cross-platform bytecode + once-per-platform runtime install), **.NET** (same shape), **Flutter** (cross-platform UI runtime, with important caveats — see "Why not just Flutter?" below).

## Threading and GPU as day-1 capabilities

The two categories most broken on today's web platform — and the clearest differentiators of this proposal — deserve explicit call-out, because the uniform-surface story is the whole point.

**Multi-threading:**
- **Native targets** (desktop / mobile / embedded): real OS threads via the runtime's threading capability. No COOP/COEP, no SharedArrayBuffer header negotiation. The WASM threads proposal is shipped; outside the browser sandbox, implementing real threads is straightforward.
- **Browser-tab compatibility shim**: still pays the web tax — Web Workers + SharedArrayBuffer + COOP/COEP headers at the hosting server. Unavoidable while the browser security model is what it is.

**GPU:**
- **Native targets**: direct Vulkan / Metal / DX12 access via a `wgpu`-style abstraction. One shader language (WGSL lean, since it compiles cleanly to all GPU backends through existing tooling), lowered per backend. No host-call mediation.
- **Browser-tab shim**: WebGPU, which is itself the browser's interface to Vulkan/Metal/DX12 under the hood. More overhead than native but functional.

**The point: developer-facing API is identical across targets.** App code says "spawn a worker, send a message" or "create a compute pipeline, dispatch these buffers." Runtime lowers to OS thread / real GPU on native; to Web Worker / WebGPU on shim. Target-specific cost, uniform surface.

**Why this is the day-1 differentiator vs. today's stack.** In the current world, web multi-threading requires deployment-level COOP/COEP coordination (most sites never bother), and GPU APIs fragment across native (Metal/Vulkan/DX12) vs. web (WebGPU) — you either write twice or adopt a heavyweight framework like Unity/Unreal that bundles its own abstraction. In the proposed runtime: threads and GPU are first-class capability interfaces everywhere, from day 1, single API.

**Honest caveats:**
- Browser-shim threads pay COOP/COEP — can't escape it without leaving the sandbox. App code is identical across targets, but the performance degradation on the shim is real.
- GPU feature parity across backends isn't perfect (Vulkan features ≠ Metal features ≠ DX12 features). `wgpu` already deals with this via "core features + opt-in extensions"; same pattern inherited.
- Shader language needs a single pick. WGSL is the likely choice — already spec'd, already compiles to SPIR-V / MSL / HLSL / DX12-bytecode via existing tooling.

## Performance expectations

Performance of an AR app depends strongly on which delivery path runs it. There are three regimes, each with a different envelope.

**Three-phase model:**

| Phase | Where AR runs | Perf vs. fully-native platform-specific app |
|---|---|---|
| 1. Browser-shim (today; adoption vector) | WASM engine + WebGPU + Workers + SAB inside a browser tab | 70–90% of native on typical workloads; 50–70% on threading-heavy or syscall-heavy; cold start slow. Acceptable for ~80% of apps. |
| 2. Natively-installed runtime on a traditional OS (mid-term) | AR runtime as a JVM-class system-level install on Windows / macOS / Linux / iOS / Android | 90–99% on typical workloads; 80–90% on syscall-heavy; hand-tuned SIMD 85–95%; cold start mitigated by AOT at install. Invisible to ~95% of apps. |
| 3. AR-native OS (long-term strategic play) | Kernel ABI *is* WIT; WASM Component *is* the executable format; no POSIX layer | **Can exceed** traditional-native perf on traditional OSes — AR apps outperform because traditional native pays abstraction taxes that don't exist on an AR-native OS. See below. |

**Sources of the delta in phase 2 (native install on traditional OS):**

- **WIT → runtime → POSIX syscall → kernel:** three indirection layers. Typically adds 1–10ns per syscall; visible only on microbenchmarks.
- **WASM bounds-checks on loads/stores:** 1–5% CPU overhead on memory-heavy loops; near-zero on hardware with safe-read extensions (MTE, CHERI).
- **Component Model marshaling for host calls:** 10–100ns per crossing (JNI-class); matters only for APIs called thousands of times per frame.
- **GPU via `wgpu`-style abstraction:** 5–10% over raw Vulkan/Metal/DX12 — but this is the same cost a native `wgpu`-using Rust app pays today. Only hand-rolled Vulkan/Metal/DX12 apps beat it, and most apps already don't do that.
- **Threading:** essentially zero — AR runtime wraps `pthread_create` / `CreateThread` with a thin typed surface.
- **Cold start:** WASM compile on first load. AOT compilation at install time eliminates this; warm start is native-equivalent.

**Why phase 3 can invert the comparison.** The baseline isn't "AR vs. a theoretical zero-overhead native." Traditional "native" apps on traditional OSes already pay layered abstraction taxes that compound:

- **POSIX syscall surface** — 50 years old, designed around PDP-11-era assumptions. `io_uring` and similar mechanisms exist precisely to bypass it and are materially faster than classical syscalls.
- **Dynamic linking** via `dlopen` / `.so` / `dyld` / `.dll` — load-time resolution cost and runtime PLT/GOT indirection on every external call.
- **libc / glibc** — decades of ABI-compat constraints; specialized allocators (mimalloc, jemalloc, tcmalloc) routinely beat glibc `malloc` by 2–10× because they aren't locked to the legacy ABI.
- **ELF / Mach-O / PE** — legacy executable formats with historical metadata. WASM Components are cleaner and smaller.
- **Kernel schedulers** (Linux CFS, Windows NT, XNU) — designed for POSIX threads + `fork`/`exec` + signals. Go's goroutine scheduler beats them for goroutine-heavy workloads *because* it can be designed for its own execution model. A component-native scheduler has the same freedom.
- **GPU drivers** — Vulkan/Metal/DX12 are themselves translations of older graphics APIs, implemented in C against legacy driver ABIs. Every abstraction layer costs.

An AR-native OS drops all of this. WIT calls *are* the kernel ABI (no translation). WASM Components *are* the executable format (no ELF/Mach-O loader). The scheduler, allocator, GPU surface, and linker are designed around the runtime's execution model directly. Nothing legacy to preserve.

**Hypothetical phase-3 comparison (AR-native OS vs. traditional OSes):**

| Workload | Native app on traditional OS | AR app on AR-native OS |
|---|---|---|
| CPU compute | 100% baseline | 100–105% (no POSIX tax on syscalls and memory ops) |
| Memory allocation | 100% (glibc-bound) | 105–120% (capability-scoped allocator, no ABI compat constraint) |
| I/O-heavy | 100% | 110–140% (io_uring-class default, no classical syscall layer) |
| GPU rendering | 100% (via wgpu / Vulkan) | 100–110% (no driver translation layer) |
| Thread spawn | 100% (pthread) | 105–150% (component-native scheduler, no POSIX overhead) |
| Cold start | 100% | 95–100% (AOT at install, native-equivalent) |
| Security context switches | 100% | 120–200% (capability-scoped, no MMU context switch for in-process isolation) |

Numbers are directional, not benchmarked — they reflect the shape of what removing each abstraction layer allows, not measurements from a system that does not yet exist. The point is **direction of the inequality, not magnitude**: the comparison runs in AR-native OS's favor on multiple axes, not against it.

**The strategic punchline.** Phase 3 is what makes the whole AR story more than "another cross-platform runtime." The traditional-native baseline is not a ground-truth optimum — it's a stack of historical compromises a fresh OS design can drop. An AR-native OS inherits every AR app ever built *and* runs them potentially faster than any traditional OS runs its own native apps. This is the moment the 25+ years of OS-ecosystem lock-in breaks — new OSes get day-1 app compatibility plus a perf story that traditional OSes structurally cannot match without throwing away their own legacy.

**Caveats inherited from § What this unlocks:**

- **Drivers still have to exist.** A new OS inherits the hardware-tail problem. Tractable engineering, years of work — the genuine blocker, not apps.
- **Hardware vendor engagement is load-bearing for the top of the envelope.** Hardware shipping WASM-aware primitives (MTE/CHERI bounds-check enforcement, GPU exposing WIT natively, first-class NPU/TPU/accelerator components) is what delivers the *best-case* phase-3 numbers. Without it, phase 3 is still meaningfully faster than phase 2 but doesn't reach the full hardware-synergy upside.
- **Runtime governance.** If one vendor controls the AR spec, new OSes are hostage. Open-spec governance (W3C / IETF-class) is a non-negotiable prerequisite for the OS-unlock story to hold.

## Why not new-browser-from-scratch or new-OS-from-scratch

Both are correct in instinct but wrong in leverage.

- **New browser**: has to out-compete Chrome/Safari/Firefox on the *legacy* web *and* bring the new model. You die on the first axis before the second matters. Even Servo/Ladybird, which are real engineering projects, are years from legacy-parity and carry zero new-platform story.
- **New OS**: hardware driver tail is 40 years of inertia. Needs decades. Google tried with Fuchsia — right instinct (capability-scoped OS, Zircon kernel) — and shelved it.
- **Runtime-as-ABI** (the proposal): *additive*. Runs **inside** today's browsers and **alongside** today's OSes. Adoption does not need anyone to switch platforms. This is the path that can actually ship.

## What this unlocks: OS-level innovation (the strategic reframe)

The above asks "what's the right *app platform*?" But the bigger consequence is downstream: **a capability-scoped application runtime is the decoupling mechanism that lets OS-level innovation restart after 25+ years of ecosystem lock-in.**

**The hidden barrier to new OSes is not technical — it's app compatibility.** Evidence:

- **Linux desktop** has been technically capable for 25+ years and is free. Never broken 5% desktop market share. Reason: apps (Office, Adobe, pro tools, major games).
- **Windows Phone** was by many measures the cleanest mobile OS in 2013. Died on the app gap.
- **Fuchsia** — years of Google engineering, capability-scoped, modern microkernel (Zircon). Shelved. No path to an app ecosystem.
- **HarmonyOS** outside China — technically credible, cannot break out. App gap.
- **Serenity / Redox / Haiku** — real engineering, zero mainstream trajectory. App gap.

Meanwhile every incumbent OS carries decades of baggage it cannot drop:

- **Windows**: Win32, 40 years of COM/OLE/DCOM, registry semantics, NT internals exposed via undocumented APIs apps rely on.
- **macOS**: Carbon→Cocoa lineage, iOS-app compatibility layers, aging BSD userland.
- **Linux**: POSIX assumptions, X11/Wayland dual-reality, SysV init DNA, 40-year-old syscall shapes.

**What an application runtime unlocks.** A new OS only needs to implement the runtime port once — ~10-20k LoC of adapter code wiring the WIT capability interfaces to the new kernel's syscalls, windowing, GPU, and networking. Day 1 after the port lands: **every app ever built for the runtime runs on the new OS, unmodified.**

With the app-compatibility chain cut, an OS designer is free to:

- **Target modern hardware directly.** Many-core CPUs, hundreds of GB of RAM, NVMe as primary storage (not "a faster disk"), GPUs / NPUs / TPUs as first-class compute citizens, heterogeneous cores. Existing OSes schedule and address-map like it's still 1995.
- **Drop POSIX / Win32 / Cocoa entirely.** No `fork()`, no signal handlers as a universal abstraction, no file descriptors as the one-true-handle, no `mmap` as a leaky contract. Design the syscall surface for today's workloads (async-first, GPU-aware, NPU-aware, heterogeneous-scheduler-aware).
- **Make security capability-based from the kernel up.** No ambient authority, no setuid, no UID/GID inheritance as the security model. Match what the app layer already assumes.
- **Rethink process / thread / memory models.** Hardware-tagged address spaces (ARM MTE, CHERI), io_uring-only IO, coroutine-native scheduler, shared-nothing by default with explicit delegation.
- **Drop ELF / PE / Mach-O.** WASM components *are* the binary format. No separate executable loader surface to maintain.

**The strategic reframe.** Read literally, the proposal is "a better app platform." Read at full stakes, it is **the unlock for OS-level and hardware-level innovation** — the thing that would let a small, focused team ship an OS designed for 2030 hardware, without having to first replicate 40 years of app ecosystem. The app platform stops being the chain dragging hardware innovation, and becomes portable cargo that moves with it.

**Caveats honestly:**
- **Drivers still have to exist.** A new OS still inherits the hardware-tail problem (GPU drivers, Wi-Fi firmware, device trees, USB stacks). But this is a tractable engineering problem — and historically it's the *app* compatibility problem that kills new OSes long before drivers become the bottleneck. Decoupling apps turns driver work into the remaining major cost, not the second-of-many.
- **System-level software** (databases, web servers, JVMs, container runtimes) is written against POSIX today. A new OS either ports these once per project (already the baseline for any platform) or provides a POSIX compatibility shim for server workloads. The app-facing side of the OS doesn't need POSIX at all.
- **Runtime governance is load-bearing.** Whoever controls the canonical runtime gains the kind of leverage V8/Chrome has today over the web. Open specification, multiple independent implementations, W3C/IETF-style process — this has to be designed in from the start, not retrofitted. The whole unlocking story collapses if one vendor ends up as gatekeeper.
- **OS and runtime are mutually dependent for credibility.** The runtime needs credible diverse OS targets to prove portability; new OSes need the runtime to have real app density. Chicken-and-egg. The browser-tab compatibility shim breaks the cycle — apps ship against the runtime using the browser as universal initial deployment target, building app density *before* any new OS exists.

## Real-world partial attempts

Each got part of the answer right and is instructive about what's hard:

| Project | Got right | Stopped short at |
|---|---|---|
| **Flutter** | Skip the DOM, render via Skia to a portable pixel buffer. Ship same UI on mobile/desktop/web. | Dart isn't portable substrate (not an open ABI); web target leans on HTML/canvas fallback; no capability model. |
| **Fuchsia** | Capability-scoped OS with typed IPC (FIDL). | Google shelved it; also still required hardware adoption. |
| **Tauri** | Native shell instead of Chromium per-app; small footprint. | UI layer is still WebView (HTML/CSS/JS). |
| **Electron** | Unified runtime ships everywhere; "one app, all desktops." | Ships a full Chromium per app (~100MB); desktop-only; UI layer is still HTML. |
| **WASI + Component Model** | Right substrate: typed interfaces, portable binaries, capability-scoped. | No UI story. No bundle-distribution story. No opinionated runtime ships on every device. Stopped short of being a platform. |
| **React Native / Capacitor** | Write-once, deploy-to-mobile. | Per-platform shims, JS-to-native bridge cost, divergent capabilities per target. |
| **Fermyon / wasmCloud** | WASM components as server-side ABI, capability-scoped. | Server-only; no client/UI story. |

The through-line: **nobody has combined Flutter's rendering approach + WASI's capability-scoped typed-interface substrate + mobile-app-store-style signed bundle distribution** into one runtime that ships on every device including inside browsers.

## Why not just Flutter?

Flutter is the closest real-world pretender and any reader coming to this proposal will naturally ask this question. Flutter got the rendering approach genuinely right (Skia-to-portable-pixel-buffer, skip the DOM, same UI on mobile/desktop/web) and that lesson is inherited here. But Flutter stops short on four dimensions the AR proposal is specifically trying to cross:

| Dimension | Flutter | AR proposal |
|---|---|---|
| **Language** | Dart only. Locked substrate; not an open ABI that other languages can target. | Language-neutral. WASM Component Model is the portable ABI — any language that compiles to components produces apps (Rust, Kāra, Go, Swift, C#, Zig). |
| **Runtime install shape** | Engine is bundled **per-app** (~5–10MB copy of the Flutter engine in each binary). No shared runtime — every app ships its own. | One runtime install per platform, shared by every AR app on that platform. JVM-style: install once, all subsequent apps are small bundles. |
| **Web target** | Compromised. Skia rendered over HTML `<canvas>`, with HTML/canvas fallback paths for some widgets; inherits DOM / same-origin / COOP/COEP baggage. Dart also transpiles to JS for some delivery shapes. | Ships the *same* signed WASM-component bundle to browsers via the zero-install loader. Pure canvas + WebGPU, no HTML rendering. Browser is one delivery target among many, not a special case. |
| **Security / capabilities** | Platform channels — per-target shims, no unified typed capability model. File system access on iOS ≠ Android ≠ desktop; each is implemented and typed differently. | Capability-scoped WIT interfaces. Same typed API for filesystem / network / GPU / audio / input on every target. Replaces origin-based web security with capability tokens. |
| **Primary role** | UI framework first (widgets, layout, animation), platform second. UI shape is part of the framework; you opt in to Flutter's UI or you don't use Flutter. | Platform runtime first. UI is a **library choice** on top — Dioxus-style, Flutter-style, imgui-style, immediate-mode, retained-tree, whatever the app picks. |

**Summary:** Flutter is a cross-platform UI framework with a bundled engine, locked to Dart, compromised on web, without capability-scoped security. AR is a platform runtime: take Flutter's rendering lesson, pair it with WASI's capability substrate, and add signed-bundle distribution on top. The result is a true runtime rather than another cross-platform UI framework.

## Honest tradeoffs

- **You lose "just visit a URL."** The web's installation-free ubiquity — click a link, app runs — is genuinely valuable and is the single biggest reason HTML won. Signed bundles + browser-tab compatibility shim preserves ~90% of this (bundles under 1MB install in under a second on first visit; cached after), but it's not 100%.
- **You lose view-source.** Also genuinely valuable — the web's openness came from the fact that you could read any site's code. WASM components are opaquer. Counter-argument: minified JS with sourcemaps stripped is already opaque; the openness battle was lost in practice a decade ago.
- **Accessibility is load-bearing and currently handled by the DOM.** If UI becomes a library choice, accessibility has to be a runtime-level capability interface, not a UI-library responsibility — or we get a worse accessibility story than today. This is the single hardest subproblem and the one most likely to sink the whole thing.
- **Text input / IME / composition across languages is similarly load-bearing.** Platforms have decades of investment here. Runtime has to expose it as a capability; UI libraries must use that capability rather than reimplementing.
- **OS integration** (notifications, sharing, file pickers, deep links, protocol handlers) has to be capability-scoped from day one or apps feel second-class on each platform.
- **Performance on the browser-tab compatibility shim** is necessarily worse than native. That's fine for adoption (the shim is the transition path, not the destination), but it means the pitch to developers is "native on desktop/mobile, merely acceptable in browser" — not "uniformly great everywhere."

## One-line version

Finish what the WASM Component Model started — build an application runtime that treats WASM as the portable OS ABI, makes UI a library not a platform, makes security capability-scoped not origin-scoped, and lets the browser become one deployment target among many instead of the center of the app universe.

## Revisit triggers

Reasons to come back to this seed and decide whether it's worth turning into an actual project:

- **Component Model reaches broad adoption outside browsers** (wasmCloud / Fermyon / server-side WASI) — indicates the substrate is real, not speculative.
- **Browser vendors ship direct-Component-Model execution** (no JS-shim required) — removes the browser-tab compatibility tax and makes the shim path viable.
- **A real accessibility-capability spec emerges** in the WASI working groups — removes the largest blocker.
- **Flutter or a similar cross-platform runtime stalls or fragments** — opens a market gap.
- **App stores relax on signed-bundle-format requirements** or Apple/Google accept WASM components natively — removes the distribution blocker.
- **A credible reference UI library emerges in the Rust/WASM ecosystem** (Dioxus/Leptos/Xilem-style but ABI-portable) — removes the "ship without UI" problem.
- **A credible new-OS project with real funding emerges** (post-Fuchsia, post-HarmonyOS-restart, or a well-funded startup). This is the single strongest trigger — it turns the strategic-reframe section from theoretical to concrete, because the new OS becomes the forcing function for a portable runtime to exist. If this happens, the revisit should be immediate, not delayed.
- **Major hardware vendor (Apple / Nvidia / emerging China silicon) starts pushing a vertically-integrated stack** that signals app-portability is becoming a strategic weapon — same forcing function from a different direction.

Absent any of these, this stays parked. Revisit in 2–3 years unless a trigger fires earlier.

## Out of scope for this seed

- Specific UI library design (immediate vs retained, reactive vs imperative, etc.).
- Specific bundle format bytes / manifest schema.
- Specific capability interface WIT definitions.
- Specific compatibility-shim implementation for existing browsers.
- Language choice for writing apps against this runtime (Kāra would be one option among many; so would Rust, Go, Swift, Zig, anything that targets WASM Component Model).

These are entire design areas of their own and only worth working on if/when the seed is promoted to an actual project.
