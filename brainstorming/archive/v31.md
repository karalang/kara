# Brainstorming v31 — Compiler hint for explicit `ref T` on Copy primitives

**Part 1 status: RESOLVED 2026-04-18 — parked as (a) silent.** No lint, no auto-rewrite. Rely on `karac explain` to surface inferred-vs-written modes for the teaching channel, and on standard backend optimizer passes (argument promotion, inlining + SROA) to narrow the observable perf gap at the machine-code level. Guiding principle: parameter modes are semantic signature, not optimizer hints — the compiler must not silently override explicit annotations, and frontend lint/rewrite conflates concerns that belong in different layers. Captured as a P3 entry in `docs/deferred.md § Lint on Explicit ref T for Copy Primitives`, with three re-evaluation triggers (corpus evidence, `karac explain` insufficiency, independently-justified lint framework). **Parts 2 and 3 remain open.** Original analysis preserved below for future reference.

---

Triggered while explaining `ref` semantics in `docs/book/src/ch12-ownership.md` to a user walking through `ref String` vs `StringSlice`. The conversation drifted to `ref i64` — a borrow of a Copy primitive — and established:

- `i64` is 8 bytes; `ref i64` is also an 8-byte pointer on a 64-bit target. **Same argument size.**
- Reading through `ref i64` adds one indirection the optimizer can't always eliminate.
- For small Copy primitives, `ref T` is *strictly worse* than `own T` unless the callee needs `mut ref T`.

Kāra's parameter-mode inference (`docs/design.md` Part 1) already picks `own` for Copy types whose only uses are reads, so the pessimization only appears when the programmer **explicitly overrides** inference by writing `ref` in the parameter list.

Question: should the compiler flag explicit `ref T` on Copy primitives where inference would have chosen `own`, as a structured diagnostic?

---

## Background

- Parameter modes are inferred (`docs/design.md` Part 1, line 4313). Explicit annotation overrides inference.
- Inferred mode is classified as *reported behavior* (`docs/design.md:123`) — `karac explain` is expected to surface it.
- No lint infrastructure in the compiler today. Diagnostics are phase-structured errors per `CLAUDE.md`.
- `#[allow(...)]` / attribute syntax for lint suppression is not yet speced.

**When explicit `ref` on a Copy type is genuinely suboptimal:**

- `fn f(x: ref i64)` — 8-byte pointer vs 8-byte value; extra indirection.
- `fn f(c: ref char)` — 4-byte value, still passed in a register under every mainstream ABI.
- `fn f(b: ref bool)` — 1-byte value, same story.

**When explicit `ref` on a Copy type is *not* suboptimal:**

- `fn f(m: ref Matrix4x4)` — 64-byte Copy struct. `own` copies 64 bytes per call; `ref` passes 8-byte pointer. `ref` wins.
- Generic contexts: `fn f[T: Copy](x: ref T)` — `sizeof(T)` unknown at definition site.
- Library authors keeping `ref T` uniform across Copy and non-Copy instantiations of the same pattern.

---

## Framings

- **(a) Silent — status quo.** Compiler accepts explicit `ref` without comment.
- **(b) Surface via `karac explain` only.** Tool reports "this `ref` would have been inferred as `own`" on demand; compilation stays silent.
- **(c) Lint / warning at definition site.** Non-fatal structured diagnostic pointing at the `ref` keyword: "explicit `ref T` on Copy type `T`; inference would pick `own`, which avoids one indirection."
- **(d) Hard error.** Reject the function definition; user must drop `ref` or add `mut`.

---

## Arguments for (c) — the lint

- **Genuine pessimization, easy to explain.** For small Copy types, `own` is the same argument size as `ref` and avoids a load. Users coming from Rust habits write `ref` reflexively to mean "don't move my variable" — not realizing that Copy + owned already preserves the caller's binding.
- **Reinforces inference-as-default.** Kāra's design principle is "don't hand-write parameter modes." A lint on redundant/worse overrides is a teachable nudge toward idiom.
- **Symmetric precedent.** Rust's Clippy has `needless_pass_by_value` targeting the inverse (owned where `&` is cheaper). Kāra's case is the mirror image.
- **Cost is local.** Only triggers on explicit `ref` — never on inferred modes. Default behavior unchanged.

## Arguments against (c) — the lint

- **Threshold rule is non-trivial.** "Copy + ≤ pointer-size" covers the obvious cases but not all. Tuples of primitives (`(i64, i64)` is 16B but still register-passable on some ABIs), `distinct type` wrappers (`docs/design.md:4833-4835`, Copy is user choice), and word-size-variable architectures all muddy the rule. Pinning it commits the compiler to an ABI-ish heuristic.
- **Generic parameters.** `fn f[T: Copy](x: ref T)` — at definition site, size of `T` is unknown. The lint would have to either (i) fire pessimistically on every `ref T` with a Copy bound, (ii) fire at monomorphization (noisy, per-instantiation), or (iii) skip generics entirely (inconsistent).
- **Signature-uniformity overrides.** Trait methods, overload sets, or pattern-consistent APIs may want `ref T` uniformly. A lint here is noise until a suppression mechanism exists.
- **Documentation channel.** A programmer may write `ref` explicitly to signal "read, not consumed" in long signatures where the body isn't obviously visible. The lint removes a valid annotation use case.
- **Requires attribute syntax.** Without `#[allow(...)]` or equivalent, the lint has no opt-out. Spec'ing attribute syntax is larger than the lint itself.

## Arguments for (b) — `karac explain` only

- Information available on demand, no build-time noise.
- Consistent with the existing promise that inferred modes are reported behavior (`docs/design.md:123`).
- Downside: invisible to programmers who don't run `karac explain`. Fails the teaching use case for newcomers.

## Arguments against (d) — hard error

- Over-reaches. The user wrote a legal program with suboptimal performance; that's a warning, not an error. Hard errors are reserved for soundness issues.
- Breaks the "explicit override always wins" property of the inference system.

---

## Threshold rules (if (c) is chosen)

| Rule | Fires on | Misses / false-positives |
|---|---|---|
| **R1.** `T: Copy ∧ sizeof(T) ≤ word` | `ref i64`, `ref bool`, `ref char`, `ref f64` | Misses `ref (i64, i64)`; word-size varies across targets |
| **R2.** `T: Copy ∧ sizeof(T) ≤ 2 × word` | All register-passable Copy types | Platform-dependent ABI leak |
| **R3.** `T: Copy` (any Copy type) | Every Copy type | False-positives on large Copy structs where `ref` actually saves bytes |
| **R4.** `T` is a built-in primitive scalar (`i*`, `u*`, `f*`, `bool`, `char`) | Language-defined set — no ABI dependence | Misses `distinct type` newtypes over primitives, tuples of primitives |

R4 is the easiest to explain and the most conservative. R1 is narrowest on semantics but ABI-dependent. R2/R3 trade precision for platform coupling.

---

## Interaction with existing rules

- **Override precedence.** Explicit annotation still wins (`docs/design.md` Part 1). The lint flags, never rewrites.
- **Reported behavior clause.** Lint wording is *reported behavior*, not semantics (`docs/design.md:99`, `:124`). Free to evolve across releases.
- **Trait method signatures.** Traits have no body to infer from; `ref` is mandatory there. Lint needs a carve-out — fire inside `impl` blocks and free functions only, not on trait declarations.
- **Closures.** Closure parameter modes are inferred by the same scan (`docs/design.md:2539`). An explicit `|x: ref i64| ...` would lint under the same rule.
- **`karac explain`.** Under (c), `explain` would also surface the same fact; the lint is the build-time variant of (b), not an alternative.

---

## Current lean

**Weakly toward (c) with rule R4 (primitives only), but not urgent.** The lint is narrow, easy to explain ("don't borrow a primitive unless you need to mutate it"), and fires only on explicit overrides. R4 avoids the ABI-heuristic trap of R1/R2 and the false-positive rate of R3.

Three prerequisites before the lint can actually ship:

1. A lint/warning emission framework exists in the compiler (today there is none — only phase errors).
2. An attribute or suppression syntax exists for the library-uniformity use case (otherwise every false-positive is a cliff).
3. Empirical evidence that explicit `ref T` on primitives shows up in real Kāra code. A scan of `examples/` and `tests/` today would likely find zero occurrences — if the corpus has <5 instances, this is a lint chasing a non-problem.

None of these is a blocker for the decision; they are preconditions for implementation.

---

## Resolution path

- **Short-term:** park in `docs/deferred.md § Compiler Lints` as P2. Re-evaluate once the lint framework lands.
- **Pre-decision spike:** when considering implementation, grep the corpus for explicit `ref <primitive>` parameters. Decide based on incidence, not hypotheticals.
- **Backstop:** reconsider once `karac explain` ships — if it already surfaces inferred-vs-written modes well, (b) may make (c) redundant.

---

## Out of scope

- **Auto-rewriting `ref` away.** Explicit annotations are honored. This discussion is about diagnostics, not transformations.
- **`ref` on non-Copy types.** Those parameters genuinely need `ref` to avoid a move; no lint applies.
- **The inverse lint** — `own T` where `ref T` would be cheaper (e.g., a large non-Copy struct). Different tradeoffs; separate question.
- **Attribute syntax design** (`#[allow(...)]` form). Larger feature; referenced here only as a precondition.

# Part 2 — Call-site `mut` marker for `mut ref T` / `mut Slice[T]` parameters: required or optional?

**Part 2 status: RESOLVED 2026-04-18 — doc served its purpose as the framing pass; decision carried forward to [v32.md](v32.md) with a fresh problem statement.** The analysis below (variant (a2'), edge-case enumeration E1–E15, open loops O1–O5) informed the framing but is not itself the final decision. Do not treat as authoritative — v32 is where the decision lands.

---

## Resolution (a2') — details for review

### The rules

**Call-site markers:**

- **`mut x` required** when passing a *fresh* binding to a parameter whose inferred mode is `mut ref T` or whose declared type is `mut Slice[T]`.
- **No marker** for owned, borrowed-immutable, or forwarded-mut-ref arguments. Plain `f(x)`.
- **`ref x` and `mut ref x` at call sites are not legal syntax** under (a2'). They are rejected, not just discouraged.

**Signatures (non-self parameters):**

- **`x: T`** — binding immutable, ownership mode inferred from body.
- **`mut x: T`** — binding mutable, ownership mode inferred from body.
- **Explicit `ref T` / `mut ref T`** remain legal as an author-written *override* of inference (rare; used only for library-uniformity or forward-compat reasons). Never required. Not idiomatic.

**Signatures (self receivers):**

- **`self`** — binding immutable, ownership mode inferred.
- **`mut self`** — binding mutable, ownership mode inferred.
- **Idiomatic code has no `ref self` / `mut ref self`.** Inference picks the right mode from the body (reads → ref, mutates in place → mut ref, moves or returns self → own).
- **Explicit `ref self` / `mut ref self`** remain legal as overrides (same as non-self params). Rare.

**Type-level markers — stay in signatures (no change from today):**

- **`mut Slice[T]`** is a *type*, not a mode annotation. Distinct from `Slice[T]` with different aliasing rules (unique vs. shared). Author must pick which slice type the parameter accepts. Inference cannot derive "this function accepts mutable views" from a body that happens not to mutate.
- `String` / `StringSlice` follow the same pattern — distinct types, author writes them.

**The "fresh vs. forwarded" rule for call-site `mut`:**

- **Fresh:** argument expression resolves to a variable binding (or field access of one) that currently holds the *value*, not a borrow to it. `let mut v = [...]; f(v)` → `f(mut v)`. Fires the marker requirement.
- **Forwarded:** argument is already a `mut ref T` / `mut Slice[T]` in the current scope. No marker. Example: inside `fn helper(m: mut ref Map)`, calling `other(m)` is `other(m)` — no marker, because `m` is already a mut-ref, there's no fresh mutation intent.
- Recursive calls through a forwarded mut-ref: unmarked (they're forwarding, not initiating).
- Chain of forwarded refs: each downstream call unmarked until the original root caller.

### The design principle this falls out of

> The user declares *what* they want. The compiler decides *how* to do it.

- **What the user writes (at call sites):** intent to mutate. `mut x` = "yes, this variable of mine may change because of this call."
- **What the user writes (in signatures):** binding mutability (`mut x` / `mut self` if they reassign inside the body), and type-level slice/receiver shape (`mut Slice[T]`, `String` vs `StringSlice`). Not ownership mode.
- **What the compiler decides (at call sites):** whether to lower as move or borrow based on the callee's inferred parameter mode.
- **What the compiler decides (in signatures):** ownership mode per parameter and receiver, derived from how the body uses it.

### What the compiler is inferring — explicit list

This is the "lot of magic" the user wants to audit. Every inference decision the compiler makes under (a2'):

| Decision | Input | Output | Where it fires |
|---|---|---|---|
| Parameter mode | Function body (uses of parameter) | `own` / `ref` / `mut ref` | At function definition |
| Receiver mode | Method body (uses of self) | `own` / `ref` / `mut ref` | At method definition |
| Borrow vs. move at call site | Callee's inferred/declared mode + caller's use-after pattern | ABI lowering only (semantic mode already fixed by callee) | At each call |
| Whether argument is "fresh" or "forwarded" | Type of the argument expression in the caller's scope | Whether `mut` marker is required / rejected | At each call site involving a mutating param |
| Stability of inferred mode across body edits | Cross-boundary call graph | Diagnostic if an inferred public-fn mode shifts | At compile time, after body edits |

The first two already exist in the current compiler. The rest are new or extended under (a2').

### Edge cases — explicit enumeration

For review before spec/compiler changes:

**E1. Non-Copy own param, caller uses x after the call.**
```kara
fn consume(s: String) -> u64 { s.len() }  // inferred: own (body could be ref, but whatever)
let s = "hello";
consume(s);
println(s);  // ERROR: s moved by consume
```
Current Rust-like ownership error, unchanged. No caller-side marker needed — the error message surfaces the move.

**E2. Forwarded mut ref — recursive call.**
```kara
fn dp(memo: mut ref Map[i64, i64], k: i64) -> i64 {
    if let Some(v) = memo.get(k) { return v; }
    let result = dp(memo, k - 1) + 1;  // UNMARKED — memo is already mut ref in scope
    memo.insert(k, result);
    result
}

let mut cache = Map.new();
dp(mut cache, 100);  // MARKED — cache is fresh
```

**E3. Forwarded mut ref — chain through helpers.**
```kara
fn outer(m: mut ref Map[K, V]) {
    helper(m);  // UNMARKED — forwarding
}
fn helper(m: mut ref Map[K, V]) {
    m.insert(k, v);
}

let mut map = Map.new();
outer(mut map);  // MARKED — fresh
```

**E4. Method call on mut-ref receiver — no marker (method convention).**
```kara
let mut vec = Vec.new();
vec.push(42);  // UNMARKED — method syntax doesn't carry the marker, period
vec.clear();   // UNMARKED
```
This is the documented asymmetry between free-function and method call sites. `mut` marker applies only at free-function call sites, not method receivers. Method mutation is signaled by convention (name: `push`, `clear`, `set_*`, `*_in_place`) + tooling, not by a caller-side keyword.

**E5. Nested call returning mut ref.**
```kara
fn get_inner(outer: mut ref Outer) -> mut ref Inner { ... }

let mut o = Outer.new();
mutate_inner(get_inner(mut o));  // MARKED on outer (fresh); get_inner's return is already a mut ref, so mutate_inner gets it unmarked via forwarding
```
The marker fires on `o` (fresh), not on `get_inner(...)` (which returns a mut ref — already-a-borrow).

**E6. Chained mutators that return self (builder / fluent style).**
```kara
fn with_x(mut self, x: i64) -> Self { self.x = x; self }  // inferred: own self (body returns self)
fn with_y(mut self, y: i64) -> Self { self.y = y; self }

let cfg = Config.new().with_x(1).with_y(2);  // NO markers anywhere — each call moves self
```
Builder pattern compiles naturally. No caller-side markers because each method infers owned-self.

**E7. In-place mutator with a return value.**
```kara
fn push_and_len(mut self, x: T) -> u64 {
    self.items.push(x);
    self.items.len()  // returns a value, but self is not returned — inferred mut ref
}

let mut v = Vec.new();
let n = v.push_and_len(42);  // method call — UNMARKED (method convention)

// If called as a free function:
fn push_and_len(v: mut ref Vec[T], x: T) -> u64 { ... }  // hypothetical
push_and_len(mut v, 42);  // MARKED (free function, fresh)
```
Inference picks `mut ref self` because self isn't returned. Method call stays unmarked. Free-function form would require `mut`.

**E8. Generic higher-order code.**
```kara
fn apply[F: Fn(mut ref T)](f: F, x: mut ref T) { f(x) }
// Inside apply: x is already mut ref (forwarded) — f(x) is unmarked.

let mut v = Vec.new();
apply(|y| y.push(42), mut v);  // MARKED on v (fresh)
```
The F bound carries the `mut ref` requirement; closures at the call site participate in the same inference. Closure arguments to `f` inside `apply` are forwarded, so unmarked.

**E9. Trait method dispatch.**
```kara
trait Container {
    fn add(mut self, item: Item);  // trait declares mut self
}

impl Container for Vec[Item] {
    fn add(mut self, item: Item) { self.items.push(item); }
}
```
The trait's `mut self` fixes the receiver mode at the interface; implementors must match. The `ref` is not written because it's inferred from the body. If the body moves self out (returning it), inference picks own. If it mutates in place (like above), inference picks mut ref. **Open question:** does the trait declaration lock the mode, or is it still inferred per impl? Needs a spec decision — see O1 below.

**E10. Public API stability.**
```kara
pub fn process(data: Vec[i64]) -> Summary {
    // v1: body only reads data — inferred ref.
    data.iter().sum()
}

// Later edit: body mutates data
pub fn process(data: Vec[i64]) -> Summary {
    data.sort();  // now mutates — inferred mut ref
    data.iter().sum()
}
```
Under (a2') with inference-only signatures, this edit silently shifts the mode from `ref` to `mut ref`. Every external caller now passes via `mut` instead of unmarked — **breaking change** that the source didn't advertise. Needs replacement enforcement — see O2 below.

**E11. Field assignment.**
```kara
let mut s = Struct.new();
s.field = 5;  // no marker; field assignment mutates in place with no caller-side marker
```
Existing behavior, unchanged. Field assignment is not a function call — no marker applies.

**E12. Indexing and subscript.**
```kara
let mut v = Vec.from([1, 2, 3]);
v[0] = 99;  // desugars to IndexMut — no marker
```
Existing behavior, unchanged. Subscript assignment doesn't carry a call-site marker either.

**E13. Owned-mutable builder (theoretical but not observed in corpus).**
Previously `mut self` ambiguously meant "owned mutable local binding" (builder) OR "mutable borrow" (mut-ref-self). Under (a2') with inference, both cases are written `mut self` and the compiler picks based on body:
- Body returns self → owned.
- Body mutates but doesn't return → mut ref.
- Neither common nor used in the Kāra corpus as of 2026-04-18.

**E14. Closures capturing mutable state.**
```kara
let mut count = 0;
let incr = || count += 1;  // closure captures count as mut ref
incr();  // method-like call — unmarked
incr();  // unmarked
```
Closure call sites are like method call sites — no caller-side marker. The closure-param-mode inference handles the capture.

**E15. Explicit override kept as legal escape hatch.**
```kara
pub fn required_shape(x: ref T) { ... }  // explicit override — pins ref even if body could accept own
```
Remains legal. Use cases: locking mode for public APIs, library-wide uniformity, traits that need a fixed receiver shape.

### Open loops (parked for follow-up — do NOT block (a2') commit)

**O1. Trait signature and mode inference.**
Does `trait T { fn f(mut self); }` declare the receiver mode for all impls, or is each impl's mode inferred from its body? If the trait locks it, we need trait-declaration syntax that pins `mut ref self` vs. `mut self` (owned) — which reintroduces the `ref` we just removed, only at trait declarations. If the trait leaves it inferred, different impls could have different receiver modes for the "same" trait method, which is incoherent for dispatch.
- **Likely resolution:** traits lock the mode by convention (the first impl or a written annotation); others must match. Needs spec.
- **Impact on (a2'):** traits are an edge where explicit mode annotation may be mandatory in signatures. Retain `ref self` / `mut ref self` for trait declarations specifically? Or allow a non-inference form like `fn f(self: mut ref Self)` that avoids the shorthand keyword conflict?

**O2. Stability enforcement for public inferred modes.**
Under (a2'), a public function's mode can shift silently across body edits (see E10). Need a diagnostic: "inferred mode of `pub fn f` shifted from `ref` to `mut ref`; this is a breaking change for external callers at X:12, Y:34. Opt in with explicit annotation or revert the body change."
- **Prerequisite:** the compiler has to read (or compute) a manifest of previously-inferred public modes. Simplest: a `karac.lock` or similar; alternative: the rustc-like crate metadata in compiled artifacts.
- **Timing:** can be deferred to a follow-up phase; (a2') can ship without it if public API stability is not yet a shipped promise.

**O3. ABI and separate compilation.**
The compiler must emit the inferred mode into the compiled artifact's interface metadata so that callers in another crate compile correctly. Straightforward mechanically, but worth pinning in the spec.

**O4. `karac explain` surfacing.**
For readers who cannot infer modes from source alone (because inference hides them), `karac explain fn_name` should print the inferred modes. Not a blocker; essential for the teaching story.

**O5. Error messages when marker is missing/wrong.**
- **Missing marker on mutating call:** "function `f` mutates its parameter `x`; this call passes the fresh binding `v` without `mut`. Write `f(mut v)` to permit the mutation."
- **Stray marker on non-mutating call:** "function `f` does not mutate its parameter; `mut` marker is not legal here. Remove `mut`."
- **Stray `ref` marker:** "Kāra does not require `ref` at call sites. Remove the keyword; the borrow is implicit."

## Original analysis (pre-resolution, preserved for context)

Triggered in the same ch12 walk-through while investigating why `examples/slice_basics.kara:47` works. The example calls `double_each(nums)` without an explicit `mut` marker, but `docs/design.md:2303` and `docs/book/src/ch12-ownership.md:116` both show the marker as required: `sort_in_place(mut v)`. The compiler currently accepts both forms — there is a typechecker bug that fails to enforce the marker.

Before "fixing" the bug to match spec, the underlying design question needs a second look: **why require the marker at all when the function signature already declares `mut ref T` / `mut Slice[T]`?**

---

## Background

- `docs/design.md:2303`: `sort_in_place(mut v);            // ref mut Vec[i64] coerces to mut Slice[i64]` — marker is part of the canonical call syntax.
- `docs/book/src/ch12-ownership.md:116`: same form.
- `examples/slice_basics.kara:47`: `double_each(nums);` — no marker. Violates spec.
- `tests/codegen.rs:1509` (`test_e2e_slice_basics_example`): pins the mutated output, so the typechecker is accepting the unmarked form and the mutation is happening.
- Parameter-mode inference (`docs/design.md` Part 1): Kāra infers `own` / `ref` / `mut ref` from the body. The signature `mut Slice[T]` is one of the places mutability is *explicit in the type* rather than inferred, but most mutating `ref` parameters on regular types are inferred silently.

Relevant stability guarantee (`docs/design.md:114`): "The inferred parameter mode of a function (`own` / `ref` / `mut ref`) once a call from external code has been type-checked against the function's signature" is guaranteed within a release. This is the property that makes a call-site marker load-bearing *if* we want inference decisions to be locked in visibly at each call.

---

## Framings

- **(a) Status quo per spec — marker required at call site.** `f(mut x)` for every `mut ref T` / `mut Slice[T]` parameter. Fix the compiler to enforce, fix the example.
- **(b) Drop the marker.** Signature declares the mode; the compiler resolves coercion without programmer repetition. Current compiler leniency becomes correct; update design.md, book, and the test to match.
- **(c) Context-sensitive — marker required only when the *variable* isn't `let mut`.** If `v` is declared `let mut v = ...`, mutation is already telegraphed; omit the call-site marker. If `v` is not `let mut`, require `mut` at the call site as a local override signal.
- **(d) Require marker only on `mut ref` / `mut Slice` — not on plain `ref`.** Plain borrows get no marker (cheap, silent). Mutable borrows get the marker (because mutation is the side effect worth flagging). This is the *intent* of (a) already; listing it as a distinct framing is just to pin that `ref x` at call sites is off the table.

---

## Arguments for (a) — require the marker (status quo)

- **Reader scannability.** The single-most-asked question in a codebase is "where does this variable get mutated?" `f(mut x)` answers it at the call site; `f(x)` forces the reader to jump to `f`'s signature (or body, if the mode is inferred). Scales with codebase size.
- **Mutation-at-a-boundary convention.** Kāra already marks mutation at the declaration point (`let mut x`). Call sites are the other boundary where a value enters territory that can mutate it. Marking both boundaries is consistent.
- **Locks inference visibly.** When a private function infers `mut ref` for a parameter, every call site *currently reading as unmarked* would quietly gain mutation semantics. The call-site marker forces callers to acknowledge the mode, and — importantly — forces a compile error at every call site if a future body edit changes the inferred mode from `ref` to `mut ref`. This is how `docs/design.md:114`'s "parameter mode must not silently shift" promise is mechanically enforced.
- **Rust precedent.** `&mut x` is near-universally accepted by Rust users. The mental model transfers for newcomers.
- **Marker is cheap.** Three characters. Not the same redundancy cost as the struct-literal-prefix case in v30, where prefixes could be 20+ chars in deeply-nested literals.

## Arguments for (b) — drop the marker

- **Compiler knows.** The signature is unambiguous. The marker carries zero information the typechecker needs. Pure annotation tax.
- **Kāra-style inference direction.** The language's stated direction is "don't hand-write what inference can determine." Parameter modes are inferred; type annotations are often elided in check mode; generic arguments are inferred from call context. Requiring the call-site marker is the odd one out.
- **"Semantics-Rust, syntax-mainstream" tiebreaker** (per `MEMORY.md`). Python, Java, C#, Go, JavaScript, Kotlin, Swift — none mark mutation at call sites. Rust is the outlier; the tiebreaker says to align with the mainstream when semantics aren't at stake, and here the semantics are the same either way.
- **Consistent with v30's direction.** The leaning resolution for struct-literal prefix is "drop the redundant annotation when the target is known." Call-site `mut` is the same shape of redundancy.
- **`let mut x` already marks intent.** The programmer who wrote `let mut v = ...` has already said "this variable can be mutated." Requiring a second marker at every mutating call is belt-and-suspenders.

## Arguments for (c) — context-sensitive

- Splits the difference: marker appears where it adds information (non-`mut` locals being borrowed mutably via coercion, if that's even legal) and is silent where it doesn't.
- **But:** "when does the marker appear?" becomes a rule most users won't internalize. Worse teaching story than either (a) or (b).
- **Also:** mutably borrowing a non-`mut` local is probably a compile error anyway (you can't take `mut ref x` from an immutable binding), so the "context-sensitive" cases collapse to always-required or always-optional. (c) is likely not coherent once the details are nailed down.

## Arguments against (b) — drop the marker

- Loses the scan-for-mutation affordance. In a 500-line file, `f(v)` vs `f(mut v)` is the difference between "I can see the mutation sites at a glance" and "I have to jump to definitions."
- Weakens the `docs/design.md:114` stability promise. If a body edit changes inferred mode from `ref` to `mut ref`, callers compile silently; the only way to notice is through `karac explain` or downstream behavior. With the marker, the compiler tells callers to acknowledge the change.
- Newcomers from Rust expect the marker. Not a decisive argument, but a non-zero transition cost.

---

## Interaction with existing rules

- **Parameter mode inference** (`docs/design.md` Part 1). Under (a), the inferred mode of a private function determines what call sites must write — a change to the body may ripple to callers as a required edit. Under (b), the inferred mode is invisible at call sites; body changes that shift `ref` → `mut ref` are silent to the reader (still caught by the borrow checker if they introduce aliasing violations).
- **`docs/design.md:114` stability clause.** Under (a), the marker is the enforcement mechanism. Under (b), the clause needs a different enforcement story (diagnostic on mode change? `karac explain` pinning? Or loosen the clause to "inferred mode is advisory, callers rely on aliasing semantics only"?).
- **`mut ref self` method calls.** Currently `v.push(x)` does not require the caller to write `v.mut.push(x)` or similar — method call syntax doesn't carry a mutation marker, even when `self` is `mut ref`. This is an existing inconsistency with (a): if call-site markers are required on free functions, why not on methods? Under (b), the inconsistency disappears.
- **Field assignment.** `s.field = 5` already mutates without an explicit marker (because it's desugared to an `IndexMut`/setter). Another data point suggesting Kāra doesn't uniformly require mutation markers at mutation sites — (a) is already an exception, not the rule.
- **v30 resolution pattern.** v30's lean (remove redundant struct prefix in check mode) is grounded in "every other check-mode position in Kāra elides redundant annotations." The call-site marker is the `ref`/`mut` equivalent of that observation.

---

## Current lean

**Weakly toward (b) — drop the marker — mostly on consistency.** The method-call inconsistency is the strongest argument: if `v.push(x)` silently mutates `v` (with no caller-side marker), then `process(mut v)` on a free function looks arbitrary. The stability-enforcement concern (`docs/design.md:114`) is real but can be addressed by a diagnostic on mode change rather than a syntactic marker at every call.

However: this is exactly the kind of decision that should not be made in a vacuum. The method-call vs free-function asymmetry is load-bearing and deserves its own walk-through before committing. And (b) would make Kāra a distinctly less Rust-looking language in a visible way, which is a signaling choice worth thinking about even if the tiebreaker formally allows it.

---

## Resolution path

**Don't fix the bug yet.** Enforcing the spec (making `examples/slice_basics.kara:47` fail) locks in (a). Accepting the current leniency (updating spec + book + example to match the compiler) locks in (b). The bug is *informative*: the compiler has been shipping with the more-permissive behavior and no downstream code broke.

Suggested sequence:

1. **Audit method calls.** Walk through how `mut ref self` methods are called in existing examples. If no call-site markers appear (and they don't, by spec), then (a) is already inconsistent and the delta to ship (b) is just freeing free-function calls.
2. **Spike on scannability.** Pick a non-trivial Kāra file (real or synthetic) and ask: with and without call-site markers, how long does it take to answer "what does this function mutate?" The scannability argument either survives the test or doesn't.
3. **Decide, then act.**
   - **If (a):** fix `examples/slice_basics.kara:47`, add a typechecker test that rejects unmarked mutable borrows, fix the typechecker. Add a test for the method-call asymmetry and decide whether methods should also get markers (probably not, but worth pinning).
   - **If (b):** update `docs/design.md:2303` and surrounding, update `docs/book/src/ch12-ownership.md:116`, add a typechecker test that pins "unmarked call is accepted for `mut Slice[T]` / `mut ref T`." Consider a replacement enforcement for the `docs/design.md:114` stability clause.

---

## Out of scope (Part 2)

- **Whether `ref` on plain borrows should get a marker.** Not currently required, not proposed. (d) above just pins that this stays out.
- **Mutation markers on method call receivers.** Mentioned as an inconsistency argument; fully resolving method-call syntax is a separate question with its own tradeoffs (method resolution, trait coherence).
- **`karac explain` UX for inferred modes.** Related to the stability-enforcement story under (b) but independent of the marker decision.

# Part 3 — Standard-library traits for common resource categories (SQL, HTTP, KV cache, …)?

**Part 3 status: RESOLVED 2026-04-18 as (a) — thin stdlib, ship only primitive resources.** Non-primitive categories (SQL, HTTP, cache, queue) remain ecosystem-defined at this phase. (b) opinionated stdlib is premature without an ecosystem; (c) marker traits solve no concrete problem; (d) dropping the trait bound breaks the provider injection model. Non-breaking to add stdlib traits later if the ecosystem converges on shapes. Captured as a P3 entry in `docs/deferred.md § Stdlib Scope for Non-Primitive Resources` with two re-evaluation triggers (package manager + ≥2 independent libraries in a category; shape convergence ≥70% method-signature overlap). **Pre-resolution task to do when the book is drafted:** add a one-line note wherever `DatabaseProvider` appears (currently `examples/phase0/dashboard.kara:79`, plus any upcoming book chapters) clarifying it's an illustrative user-written trait, not a stdlib import. Do not churn unwritten prose now. Original analysis preserved below.

---

Triggered while walking through `examples/phase0/dashboard.kara` with a user. The example declares `effect resource UserDB: DatabaseProvider;`, where `DatabaseProvider` is shown in `docs/design.md:3914` as a **user-written** trait — there is no `std.db.DatabaseProvider` in the stdlib today. The user's questions pulled out the next layer:

> "Do users need to write traits for each resource type, or is there a predefined lib?"
> "If I want DynamoDB with the AWS SDK, or without an SDK, what does the trait bind look like?"
> "Who builds the `aws-sdk` / DB providers?"

Current spec answer: built-in primitive resources (`FileSystem`, `Clock`, `Env`, `Network`, `Stdin/Stdout/Stderr`, `RandomSource`, `Heap`, `Hardware`, `GpuBuffer`) are hardwired by the compiler/stdlib — no user trait. Everything else (databases, caches, HTTP clients, queues, vendor APIs) is user- or library-written. Rust made the same choice (no `std::db`, no `std::http`).

The question: **should Kāra stdlib ship opinionated traits for a small set of common resource categories, or leave that to the ecosystem entirely?**

---

## Background

- `docs/design.md:3336-3353` lists the built-in *primitive* resources. The list is language-level, not library-level — these are resources the compiler knows about, with hardcoded verbs.
- `docs/design.md:3914-3935` shows the binding pattern: `effect resource UserDB: DatabaseProvider;` where `DatabaseProvider` is any trait with appropriate methods. The effect system is indifferent to *who wrote* the trait.
- `docs/design.md:2347` ("stdlib minimum viable I/O surface") scopes the stdlib narrowly — names, module paths, signatures, effect contracts for I/O primitives only. No mention of higher-level categories.
- No prior brainstorming item covers this. The closest is v1/v2-era discussion of stdlib scope, which settled on "thin core, ecosystem above."
- Ecosystem doesn't exist yet — Phase 0 compiler, no package manager, no external libraries. So the question is aspirational: what should Kāra's posture be when the ecosystem starts forming?

---

## Why this matters for effect-system UX

The effect system's value proposition (auto-parallelism, injection, test isolation) depends on resources being *named consistently across libraries*. Concretely:

- If `pg-kara` declares `trait PgConnection` and `mysql-kara` declares `trait MySqlConnection` and neither share a common parent, then a user with both drivers writes:
  ```kara
  effect resource Primary: pg_kara.PgConnection;
  effect resource Replica: mysql_kara.MySqlConnection;
  ```
  Callers passing a `ref Primary` to a function that wanted "any SQL connection" need generics. Fine in the small, painful for library interop.

- If the stdlib ships `std.sql.Connection`, both drivers implement it, and the user's resource binds to it:
  ```kara
  effect resource Primary: std.sql.Connection;
  ```
  Swapping providers is a `with_provider` change; consumer code is driver-agnostic.

The tradeoff is the one every stdlib faces: **standardization vs. innovation.** A stdlib trait shipped too early ossifies design choices; shipped too late, the ecosystem fragments.

---

## Framings

- **(a) Thin stdlib — status quo.** Ship only primitive resources. Every non-primitive category (SQL, HTTP, cache, queue) is ecosystem-defined. Libraries may coordinate via independent "consensus" crates. Rust's model.
- **(b) Opinionated stdlib — ship standard traits.** Stdlib provides `std.sql.Connection`, `std.http.Client`, `std.cache.KV`, maybe one or two more, each minimal (just the methods needed to anchor interop). Providers in ecosystem libraries implement them. Go's model (`database/sql`, `net/http`).
- **(c) Marker traits only.** Stdlib ships empty or near-empty **marker traits** (`std.resource.Sql`, `std.resource.Http`) with no methods. Libraries extend with their own richer traits. The markers exist purely so `effect resource X: std.resource.Sql + pg.PgSpecific;` can express "this resource is a SQL thing, also with Postgres specifics." Enables category-level parallelism analysis without committing to method shapes.
- **(d) Trait-less capability model.** Drop the trait bound requirement on `effect resource` entirely. Resources are just named capabilities; the methods callable on `UserDB` come from `impl` blocks in scope. Trait-based dispatch becomes optional.

---

## Arguments for (a) — thin stdlib (status quo)

- **Ecosystem-led design is resilient.** Every category (SQL, HTTP, caching) has unresolved design questions that the language shouldn't freeze. Rust's `hyper`/`reqwest`/`actix-web` evolution would have been impossible if `std::http` shipped in 1.0.
- **Kāra's Phase 0 posture.** No package manager, no ecosystem, no empirical data on what patterns win. Speculative stdlib traits ship baggage that survives forever.
- **Narrow stdlib is a feature.** `docs/design.md:2347` explicitly scopes stdlib to "minimum viable I/O surface." Expanding into domain traits violates that posture.
- **Marker-less interop is workable.** Libraries that care can publish a lightweight "interop" crate (`kara-sql-interop`) with the shared trait. Users opt in; the stdlib stays clean.

## Arguments for (b) — opinionated stdlib

- **Avoids N-squared fragmentation.** Without a stdlib anchor, each new SQL driver plus each new ORM plus each new migration tool negotiates its own trait shape. Go's `database/sql` made the ecosystem interoperable in a way Rust's didn't.
- **Teaching story.** The book example (`dashboard.kara`) currently references `DatabaseProvider` as if it were a thing. Either the book handwaves "you or your library writes this" or it points at `std.sql.Connection`. The latter is pedagogically cleaner.
- **Effect-system interop payoff.** Generic code over `effect resource R: std.sql.Connection` can be written once and reused across providers. Without a stdlib trait, this only works within a single ecosystem silo.
- **Small, well-chosen set.** Not every category; only the 3–5 with clear shape consensus (SQL connection, HTTP request/response, KV cache get/set, maybe message queue send/receive, maybe time/random beyond the primitives).

## Arguments against (b)

- **Premature standardization is worse than fragmentation.** A bad `std.sql.Connection` is harder to fix than no `std.sql.Connection`. Versioning and deprecation are harder at the stdlib level.
- **"Minimal trait" is a negotiation.** Is `query(sql) -> Rows` minimal, or does it need parameter binding, prepared statements, transactions, connection pooling? Each omission is a reason for a library to ignore the trait and ship its own.
- **No empirical basis.** Choosing which 3–5 categories without real-world Kāra usage is guesswork.

## Arguments for (c) — marker traits only

- **Solves the effect-system interop question without solving the method-shape question.** A resource declared `: std.resource.Sql` is known to be SQL-shaped for parallelism/conflict analysis; the actual callable surface is supplied by library-specific traits added via `+`.
- **Escape hatch for categories without obvious shape.** `std.resource.Http` can exist even if `std.http.Client` is contentious.
- **Cheap to ship, cheap to change.** No methods = no ABI/API commitment.

## Arguments against (c)

- **Unclear what problem markers actually solve.** The effect system already treats every resource as independent; the parallelism analysis doesn't need categories. If the only use is "group resources by kind for tooling," that's a thin value proposition.
- **Risk of cargo-culting.** Users add markers because they look official, without any semantic payoff.

## Arguments for (d) — drop the trait bound

- **Simpler mental model.** A resource is just a name. Methods come from `impl Name { ... }` blocks. No trait negotiation.
- **Escapes the whole stdlib-vs-ecosystem question.** The language doesn't take a side.

## Arguments against (d)

- **Breaks the provider injection model.** `with_provider[UserDB](InMemoryUserDB.new(), ...)` needs a trait to dispatch against; without it, the provider *is* the resource, and swapping providers means swapping types. Test-time in-memory providers become awkward.
- **Breaks the test story.** One of Kāra's design goals is easy test-mode substitution. The trait is what makes `PostgresUserDB` and `InMemoryUserDB` interchangeable.
- **Large spec surface rewrite.** `docs/design.md:3914-4054` is built on the trait-bound model.

---

## Interaction with ecosystem questions

Parked user question: **"Who builds AWS SDK / DB providers?"** — Part 3 is the design framing for that answer.

- Under (a), answer is "whoever writes the crate, using whatever trait shape they choose." Ecosystem is maximally free, minimally interoperable.
- Under (b), answer is "library authors, implementing the stdlib trait where applicable." Stdlib sets the shape; libraries fill it in.
- Under (c), answer is "library authors, optionally tagging with marker traits." Similar to (a) for method-level concerns, better for category-level tooling.
- Under (d), answer is "library authors, no trait involved." Provider substitution becomes a library concern (likely via generics or dyn).

---

## Interaction with existing rules

- **`docs/design.md:3914-3935` (provider pattern).** (a)/(b)/(c) all preserve it. (d) requires a rewrite.
- **`docs/design.md:2347` (stdlib scope).** (b) expands this; needs an explicit policy update on what stdlib covers and what it doesn't. Deserves its own discussion before (b) lands.
- **Same-instance detection (`docs/design.md:4054`).** Works the same under (a)/(b)/(c); changes meaning under (d).
- **`with_provider` (`docs/design.md:3909-3982`).** Trait-based injection. Only (d) is disruptive.

---

## Current lean

**Weakly toward (a) — thin stdlib — for now, with the door open to (c) later.** The argument is time-based:

- Kāra has no ecosystem. Shipping stdlib traits today is pure speculation on shapes. Rust's evolution (web frameworks, async runtimes, DB libraries) repeatedly demonstrates how much the "right" shape only emerges after real usage.
- The book's `DatabaseProvider` can say "you or your library writes this trait" — a one-line caveat, not a teaching catastrophe. The example's pedagogical value (effect-driven concurrency) is independent of where the trait lives.
- (c) is a low-cost addition later: marker traits can be introduced without breaking existing code. Waiting costs little.
- (b) is a one-way door. Once `std.sql.Connection` ships, removing it breaks every library that implemented it.

**But:** reconsider (b) or (c) once two conditions hold:

1. A package manager exists and several independent libraries in one category have shipped.
2. The shapes have converged enough that a stdlib trait would codify consensus, not impose opinion.

---

## Resolution path

- **Short-term:** park as P3 in `docs/deferred.md § Stdlib Scope`. No code or spec change today.
- **Book fix (small, now):** `docs/book/src/ch11-effects.md` (or wherever `DatabaseProvider` is first introduced, if at all) should say explicitly that `DatabaseProvider` is an illustrative user-written trait, not a stdlib import. Removes the ambiguity that triggered this brainstorming.
- **Trigger for re-evaluation:** when the ecosystem has >1 independent library in any category (SQL, HTTP, cache, queue), revisit. The specific question at that point is "does a stdlib trait now reduce fragmentation, or would it impose a losing design?"

---

## Out of scope (Part 3)

- **Package manager / registry design.** Prerequisite for ecosystem formation; separate question.
- **`std.fs`, `std.time`, `std.env`** and other stdlib modules that wrap primitive resources. These already exist per `docs/design.md:3363-3365` and are not in question.
- **Specific trait shapes** for hypothetical `std.sql.Connection` etc. If (b) or (c) is ever chosen, each trait deserves its own brainstorming.
- **Versioning policy for stdlib traits** (how to evolve `std.sql.Connection` v1 → v2). Only relevant if (b) is adopted.
- **The "who builds AWS SDK" ecosystem question.** Part 3 is the framing; the actual answer depends on which framing wins and is further out.
