## Tier 1: High Impact — Inconsistencies

### 1. `.await` exists despite "No async/await" claim

**Location:** "What Kara Is Not" section vs. "Async I/O in Practice" section and "Execution Effects" section.

**Problem.** The "What Kara Is Not" section states:

> **No async/await.** No `async fn`, no `.await`, no colored functions. The compiler determines concurrency from effect analysis.

But the "Async I/O in Practice" section and multiple examples throughout the spec use `.await` extensively:

```kara
let resp = http_get(format("/users/{}", id)).await?
```

The language *does* have `.await` — it just doesn't have `async fn` or function coloring. The current claim is factually incorrect.

**Options:**

- **(a) Narrow the claim.** Change "No async/await" to "No `async fn`. No function coloring." Keep `.await` as a yield-point marker. The key innovation is that functions are not colored — `.await` is a call-site annotation, not a function-level annotation.
- **(b) Remove `.await` entirely.** If the claim is aspirational, design an alternative yield-point syntax (e.g., implicit yield at effect boundaries). This would be a significant design change.
- **(c) Rename `.await` to something else** (e.g., `.yield`, `.park`, `.suspend`) to avoid confusion with traditional async/await semantics.

**Recommendation:** (a) — the innovation is the absence of `async fn`, not the absence of `.await`. The claim should be precise.

### 2. `usize` in iterator adaptors contradicts numeric semantics

**Location:** Iterator Adaptors section (lines ~1018-1032) vs. Numeric Semantics section (line ~777).

**Problem.** The Numeric Semantics section explicitly states:

> `usize` exists only for FFI interop (maps to C's `size_t`); idiomatic Kara code uses `i64` for all sizes and indices.

But several iterator adaptor signatures use `usize`:

```kara
fn enumerate(self) -> impl Iterator[Item = (usize, Self.Item)]
fn take(self, n: usize) -> impl Iterator[Item = Self.Item]
fn skip(self, n: usize) -> impl Iterator[Item = Self.Item]
fn count(self) -> usize
```

These should use `i64` to be consistent with the language's own numeric rules. `enumerate` should yield `(i64, Self.Item)`, `take`/`skip` should accept `i64`, and `count` should return `i64`.

**Options:**

- **(a) Change all iterator adaptors to use `i64`.** Consistent with the stated language rule. Negative values for `take`/`skip` would be runtime errors (same as negative array indices).
- **(b) Keep `usize` for iterator adaptors.** Acknowledge an exception to the "i64 everywhere" rule for iterator operations that are inherently non-negative.
- **(c) Use a refinement type.** `type Count = i64 where self >= 0` for return positions; plain `i64` for parameters.

**Recommendation:** (a) — the language already chose `i64` for indices and committed to "negative indices are runtime errors." Iterator adaptors should follow the same rule.

### 3. Auto-concurrency and `.await` interaction underspecified

**Location:** Feature 5 (Auto-Concurrency) vs. Async I/O in Practice section.

**Problem.** The auto-concurrency section says the compiler parallelizes independent calls with no conflicting effects. The async section shows:

```kara
let profile = http_get(format("/users/{}", user_id)).await?
let orders  = http_get(format("/orders?user={}", user_id)).await?
let notifs  = http_get(format("/notifs?user={}", user_id)).await?
```

And claims the compiler can parallelize these (line ~4316). But `.await` in every other language means "suspend here and wait for the result before continuing to the next line." If Kara's `.await` allows the compiler to reorder and parallelize, this is a fundamentally different semantic that needs explicit specification:

- Does `.await` mean "this is a yield point" (where the task may suspend) rather than "wait here"?
- When the compiler parallelizes three `.await` calls, does it transform them into a `par { }` region internally?
- What happens to error propagation order? If all three run concurrently and the second one errors, does `?` on the first still fire?
- Is there any way for the programmer to force sequential execution of `.await` calls? (Presumably `sequential { }` block.)

**Options:**

- **(a) Specify `.await` as a yield-point marker, not a sequencing operator.** The compiler is free to reorder `.await` calls subject to the same data-dependency and effect-conflict rules as any other calls. `.await` means "this call may suspend the task" not "wait here before proceeding."
- **(b) Specify `.await` as sequential by default, parallelized only in explicit `par { }` blocks.** Matches programmer expectations from other languages. Auto-concurrency applies to non-`.await` calls only.
- **(c) Remove `.await` from user-facing syntax entirely.** The compiler inserts yield points at effect boundaries automatically. No visible syntax change between sync and async code.

**Recommendation:** Needs discussion. (a) is the most aligned with Kara's philosophy but is surprising. (b) is safe but undermines the "no function coloring" pitch. (c) is the purest but may be hard to implement.

## Tier 2: Medium Impact — Underspecified Areas

### 4. Closure capture semantics scattered across multiple sections

**Location:** Multiple — First-Class Functions, Closures and Effect Capture, RC Dataflow Specification, Parameter Mode Inference.

**Problem.** The spec references "the closure-capture inference subsection" and "Closures: parameter modes, capture, and escape" in several places, but closure semantics are spread across at least four sections with cross-references between them. Key questions that require reading multiple sections to answer:

- How does the compiler decide whether a closure captures by move or by ref?
- When does a closure capture trigger RC fallback?
- How do closure effects interact with the enclosing function's effect set?
- What are the rules for closures that escape their defining scope?

**Options:**

- **(a) Consolidate into a single "Closures" section.** Move all closure-related rules (capture, effects, escape, RC interaction) into one place. Other sections reference it.
- **(b) Add a cross-reference index.** Keep the current structure but add a "Closure semantics roadmap" note listing which section covers which aspect.

**Recommendation:** (a) for the next spec revision.

### 5. `shared struct` interior mutability (`RefCell`) never formally defined

**Location:** Feature 4 Part 5 (Shared Types) and Deferred Items.

**Problem.** The spec says `shared struct` `mut` fields use "RefCell semantics" (line ~5586) and mentions "runtime borrow checking" for `shared struct` (line ~3941), but `RefCell` is never formally defined as a language or stdlib type. Key unanswered questions:

- Is `RefCell` an explicit type the programmer uses, or is it the compiler's internal implementation of `mut` field access on `shared struct`?
- What happens at runtime when two code paths try to mutably borrow the same `mut` field simultaneously (in single-threaded code)? Panic? Compile error?
- Is the borrow state (borrowed/not-borrowed) tracked per-field or per-struct?
- How does `RefCell` interact with the `lock` block syntax for `Mutex`-wrapped `shared struct`?

**Options:**

- **(a) Define `RefCell` formally in the spec.** As a stdlib type with explicit semantics, or as the compiler-generated mechanism for `shared struct` `mut` field access.
- **(b) Replace "RefCell semantics" with a precise rule.** E.g., "`mut` field writes on `shared struct` are exclusive at runtime — two simultaneous mutable borrows from the same thread panic. The compiler enforces this via a hidden borrow flag per `mut` field."
- **(c) Defer to implementation.** Mark the interior mutability model as implementation-defined (reported behavior, not guaranteed semantics).

**Recommendation:** (b) — define the actual runtime behavior without naming a specific implementation type.

### 6. Generic vs index disambiguation rule not stated

**Location:** Type System (generics syntax) and Standard Data Structures (subscript/index).

**Problem.** Kara uses `[T]` for generics (`Vec[i32]`, `fn sort[T: Ord](...)`) and `[i]` for indexing (`arr[0]`, `map["key"]`). The spec claims `[T]` is unambiguous because `[` is not a comparison operator, but does not state the disambiguation rule between:

```kara
foo[bar]     // Is this: generic instantiation foo[bar], or index access foo[bar]?
```

If `foo` is a type name, it's a generic. If `foo` is a variable, it's an index. But what about a function name that could be either?

```kara
let x = get[i32](data)    // generic call
let x = get[0]            // index into `get` (if it's a collection)
```

The parser needs a rule. Rust avoids this with `<>` for generics and `[]` for indexing. Go 1.18 faced this exact problem and resolved it with type-context rules.

**Options:**

- **(a) State the disambiguation rule explicitly.** E.g., "in expression context, `ident[...]` is always an index. Generic arguments in expression context use `ident[T](args)` — the presence of `(` after `]` triggers generic parsing."
- **(b) State that disambiguation relies on name resolution.** "The parser produces an ambiguous node; the resolver determines whether `foo` is a type/generic function (→ generic instantiation) or a value (→ index access)."
- **(c) Require a syntactic marker.** E.g., `foo.[T]` for generics in expression context (turbofish-like). But the spec explicitly rejects turbofish.

**Recommendation:** (a) or (b) — the rule exists implicitly in the parser implementation but should be stated in the spec.

### 7. Variadic `format()` function uses undeclared language feature

**Location:** String section, Collection Core Methods (String table).

**Problem.** The `format` function signature is:

```kara
fn format(template: String, args: ...) -> String
```

The `...` variadic syntax is never defined anywhere in the spec. Kara has no variadic function mechanism — no `...` parameter syntax, no variadic generics, no `va_list`. The `format` function either:

- Needs a defined variadic mechanism, or
- Should use a different design (e.g., `f"..."` string interpolation handles all formatting needs, and `format` takes a fixed number of args via overloading or a builder pattern)

**Options:**

- **(a) Define a variadic mechanism for `format` only.** A compiler built-in that is type-checked at each call site (like Rust's `format!` macro). Not available to user code.
- **(b) Remove `format()` as a free function.** `f"..."` interpolation is the primary mechanism; `format()` becomes `String.format(template, args_vec)` taking a `Vec[dyn Display]` or similar.
- **(c) Defer.** Mark `format()` signature as placeholder pending a compile-time string formatting design.

**Recommendation:** (a) — `format` as a compiler built-in is the simplest path and matches Rust's approach.

### 8. `Vec.filled` and `Vec.from_fn` missing from Collection Core Methods table

**Location:** Standard Data Structures section.

**Problem.** `Vec.filled` and `Vec.from_fn` are defined in prose (lines ~620-638) with full signatures and examples, but are not listed in the `Vec[T]` methods table (lines ~653-673). The table is the authoritative reference for collection methods; omitting these two constructors makes them hard to find.

**Options:**

- **(a) Add both to the `Vec[T]` table.** Between `with_capacity` and `push`.
- **(b) Create a separate "Constructors" sub-table** for static methods (`new`, `with_capacity`, `filled`, `from_fn`).

**Recommendation:** (a) — keep the single table.

**Location:** Derive > Serialization via Derive section.

**Problem.** The `Serializer` trait definition uses `usize` for length parameters:

fn serialize_seq_begin(mut ref self, len: usize)

### 10. No `process.exit()` may be too restrictive for signal handlers

**Location:** Entry Point (`main()`) section.

**Problem.** The spec says:

> There is intentionally no `process.exit()` primitive: a function that aborts the process mid-frame would bypass `defer`/`errdefer`.

The reasoning is sound for normal application code. However, some scenarios genuinely need immediate termination:

- Signal handlers (SIGTERM, SIGINT) where the process must exit promptly
- Fatal resource exhaustion where cleanup may itself fail or hang
- CLI tools where `main` returning `ExitCode` is overly verbose for a 10-line script

The `ExitCode` return pattern handles the CLI case adequately. Signal handling is a separate concern that may need `unsafe` escape hatches regardless.

**Options:**

- **(a) Keep the current design.** Signal handling is an OS-level concern handled by the runtime, not by user code calling `exit()`. CLI verbosity is acceptable.
- **(b) Add `process.exit()` as an `unsafe` function.** Bypassing cleanup is a safety concern, so `unsafe` is appropriate. The programmer explicitly opts into the risk.
- **(c) Add `process.abort()` only.** For truly catastrophic failures. No cleanup, immediate termination. `exit()` with cleanup remains the `main() -> ExitCode` pattern.

**Recommendation:** (a) for v1 — revisit if real-world usage reveals a gap.
