## Original framing (for record)

**Status:** OPEN — 2026-04-21.

**Framing.** CR-24 follow-up slice 3 says the `karac test` runner wraps each test body in `with_provider(constructor_fn(), || { ... })` so tests get a fresh provider per run. Slices 1 and 2 shipped. Slice 3 stalled when implementation started because the stated dependency — "the existing provider-injection machinery" — does not exist. `with_provider` is not a runtime primitive, `effect resource X: ProviderTrait` parses but the resolver only records the name, and no per-call provider dispatch is wired at any layer (interpreter, codegen, or otherwise). Design.md § Provider-Rooted Resources and § Testing assume the machinery; the implementation never built it.

This brainstorm decides the sequencing and semantics of (1) the `with_provider` runtime itself and (2) how the test runner interacts with it. It does not re-litigate the surface language design already in `design.md § Provider-Rooted Resources` — only the implementation questions that have to be answered before slice 3 can ship, plus the handful of slice-3-specific questions that design.md doesn't pin down.

**Ground rule (from `feedback_no_design_compromise.md`):** decide each item on Kāra's own merits. "Rust / effect-system frameworks do X" is a surfacing signal, not a deciding signal.

## Context — what exists today

- `EffectResourceDecl { name, generic_params, provider_trait }` parses: `effect resource UserDB: DatabaseProvider;` round-trips through the parser and formatter (`src/parser.rs:759`, `src/formatter.rs:560`).
- Resolver collects the resource name and marks it accessible (`src/resolver.rs:949`). It does *not* register the `provider_trait` bound anywhere, does not resolve method calls against the trait, and has no concept of "a call in a resource-using function should look up its provider."
- `Token::Providers` tokenizes (`src/lexer.rs:751`) but the parser has no `providers { ... in body }` block rule. No AST node exists for it.
- No `with_provider` builtin in the interpreter. No `with_provider` LLVM intrinsic / runtime call. No per-thread / per-task "active providers" stack at any layer.
- The ownership/escape check for provider-rooted closure capture (implementation_checklist.md:77) is unticked.
- Slice 2 of `karac test` landed `requires`-gating with env-var probes; the runner never touches the resource at runtime beyond the probe.
- Design.md § Testing (lines 2480–2491) commits the surface syntax: `#[with_provider(resource_path, constructor_fn)]`, multiple attrs allowed, fresh per test.
- Design.md § Provider-Rooted Resources (line 4522+) commits the surface syntax for `with_provider[R](provider, || body)`, `providers { ... in body }`, `Arc`-wrapping for task-spawn, `Send + Sync` at the call site, and the escape rule.

## Context — what design already commits to

These are **decided** and not open for this brainstorm:

- `#[with_provider(resource_path, constructor_fn)]` surface syntax and multi-attr allowance (design.md:2480–2491).
- `with_provider[R: effect resource, T, P: R.Provider + Send + Sync](provider: P, f: Fn() -> T with R) -> T` signature (design.md:4553).
- Arc-wrapping internally, two atomic ops total in single-threaded tests, Send + Sync at call site (design.md:4558–4565).
- `providers { p0, p1, ... in body }` desugars to nested `with_provider` calls, evaluate-all-then-scope (design.md:4616).
- Provider-rooted closures cannot escape their `with_provider` block; ambient program-rooted resources are exempt (design.md:4567–4600).
- `#[with_provider]` is distinct from `#[test(requires = [...])]` — requires is for tests that need a real external service, with_provider is for tests that supply an in-memory fake (design.md:2491).

## Scope of this brainstorm

- **Runtime mechanics** for `with_provider`: how a resource-using function finds its provider, where the scope lives, how `Arc` is threaded, how the interpreter and codegen paths split.
- **Test runner integration**: how slice 3 wraps tests, what happens when constructors fail, what JSONL events carry about providers, how `#[with_provider]` interacts with `#[test(requires = [...])]` on the same function.
- **Escape check dependency**: whether slice 3 ships before or after the escape check.
- **Sequencing**: what CR ordering gets us to green slice 3 fastest without painting us into a semantic corner.

Out of scope:
- Package-manager / workspace questions.
- The broader effect-system integration (resource verb propagation through provider calls) — that's § Effect System's job, only referenced here where it pins a runtime decision.
- Codegen-specific IR shape — runtime semantics first, IR details later in implementation.

## Group M — Runtime machinery

### M1. Where does the "active provider" live at runtime?  *(open)*

**The question.** When a function declared with `reads(UserDB)` runs inside a `with_provider[UserDB](p, || body)` block, how does the code inside `body` locate `p`?

**Options:**

- **M1a. Thread-local stack of provider maps.** Each task has a thread-local `Vec<HashMap<ResourceId, ProviderPtr>>`. `with_provider` pushes a frame, `body` reads from the top-of-stack, exit pops. Spawned tasks inherit the current map by cloning (Arc-sharing) the top frame.
  - Pro: zero compile-time cost, works uniformly in interpreter and codegen (both platforms have thread-locals), matches the design.md "Arc-wrapping internally" note naturally.
  - Con: every call into a resource-using function is a dictionary lookup; the design claims "reading through an Arc pointer is a plain pointer dereference — identical cost to Rc or Box" which this doesn't achieve without caching.

- **M1b. Implicit parameter threading (compile-time).** The typechecker rewrites each resource-using function to take an extra parameter per resource it uses; `with_provider` becomes a thunk call that supplies the parameter; call sites get the implicit argument inserted by the compiler.
  - Pro: the pointer-deref cost claim literally holds (provider is just another argument). No thread-local cost.
  - Con: significant typechecker surgery, hard to interpret (the interpreter has no such rewriting pass), crosses module boundaries in a way that forces monomorphization on provider types.

- **M1c. Hybrid — parameter threading for monomorphized paths, thread-local fallback.** Compile-time rewrite when the provider type is known at the call site; thread-local otherwise (dyn trait objects, indirect calls).
  - Pro: keeps the hot path pointer-cheap.
  - Con: most complexity of both approaches.

**Tiebreaker.** Interpreter first (that's what `karac test` uses today), which strongly favors M1a — the interpreter has no rewriting infrastructure. Codegen can optimize M1a later if profiling shows dictionary-lookup cost matters. Design.md's "two atomic ops total" cost claim is already about Arc clone/drop, not about lookup cost, so M1a doesn't violate any committed cost model as long as the lookup itself is a thread-local read + hash.

### M2. How does `with_provider` bind "this provider satisfies this resource"?  *(open)*

**The question.** A provider value `p: P` with `impl DatabaseProvider for P`. A resource `effect resource UserDB: DatabaseProvider`. A call `with_provider[UserDB](p, || body)` — how does the runtime know `p` is the provider for `UserDB` specifically, not for some other `DatabaseProvider`-constrained resource?

**Options:**

- **M2a. Resource name is the key.** The `with_provider[UserDB]` generic arg is erased at runtime to a stable resource id (u32 or interned path string); the runtime stores `(resource_id → provider_ptr)`. Resolving a `UserDB.method(...)` call looks up by resource id.
  - Pro: same-instance detection (design.md:4667) fits naturally — two `with_provider[UserDB]` and `with_provider[OrderDB]` with the same `p` map two distinct ids to the same pointer, compiler sees aliasing.
  - Con: requires resource-id assignment to be stable across modules (project-wide interning at the tree level).

- **M2b. Trait vtable indexing.** Store providers keyed by `(trait_id, slot)` where each `effect resource R: Trait` picks a slot within the trait's vtable.
  - Pro: compact.
  - Con: doesn't model the "resources are named" intuition; same-instance aliasing needs its own bookkeeping.

**Tiebreaker.** M2a lines up with how paths are already interned in the resolver (every resource has a fully-qualified name by module position — slice 5 of CR-24 landed the machinery). Lightweight extension to the existing resolver table. Picks the semantics that matches design.md's "same-instance detection is instance-based".

### M3. Interpreter vs codegen split.  *(open)*

**The question.** The interpreter is the MVP runtime for `karac test`. Codegen already compiles real programs under `--features llvm`. Do they share an implementation of `with_provider` or diverge?

**Options:**

- **M3a. Interpreter-only for v1.** Ship `with_provider` in the interpreter immediately; codegen emits an unimplemented-error if it encounters a `with_provider` call. Slice 3 ships. Codegen follow-up CR lands later.
  - Pro: unblocks slice 3 in weeks not months.
  - Con: splits the "does X feature work?" table by target. Tests that use `#[with_provider]` only work under `karac test` (interpreter); `karac build && ./dist/...` is broken.

- **M3b. Both simultaneously.** Land interpreter + codegen in one CR, slice 3 follows.
  - Pro: no target matrix bifurcation.
  - Con: much bigger CR, much longer timeline.

- **M3c. Interpreter-first with a codegen-known stub.** Interpreter full impl; codegen emits a call to a runtime-library function that panics with "with_provider not yet codegen'd" so the codegen path is at least compilable and the LLVM IR hook is reserved.
  - Pro: forward-compatible IR shape.
  - Con: trap-panic binaries in production are worse than compile-time rejection.

**Tiebreaker.** M3a is the direct application of our existing sequencing discipline (interpreter is authoritative for language semantics; codegen follows). Precedent: trait dispatch, closures, and pattern exhaustiveness all landed interpreter-first with codegen following. `karac test` uses the interpreter so slice 3 is green the day interpreter support lands. Codegen tracked as a separate follow-up CR with its own slot in `implementation_checklist.md`.

### M4. `Send + Sync` enforcement site.  *(open)*

**The question.** Design.md:4556 is clear: "`Send + Sync` is enforced at the call site on the concrete provider type — not on the provider trait itself." How do we actually enforce it, given auto-traits don't exist yet in Kāra's type system?

**Options:**

- **M4a. Defer until concurrency lands.** Until Kāra has `spawn`, tasks, and a concrete concurrency model, `with_provider` is single-threaded in practice. Enforce `Send + Sync` as a name-check (hard-coded list of types known to be thread-safe) for now, full auto-trait work ships with concurrency.
  - Pro: unblocks slice 3 and single-threaded use.
  - Con: doesn't match design.md's stated constraint.

- **M4b. Block all `with_provider` work until auto-traits.** Wait for the full Send/Sync infrastructure.
  - Pro: no interim design debt.
  - Con: indefinitely blocks slice 3 and all provider work, which touches many downstream features.

**Tiebreaker.** M4a. Single-threaded interpreter has no Send/Sync concern today; the constraint is forward-looking. We add the auto-trait check when concurrency work adds `spawn`. Meanwhile, the interpreter's `with_provider` documents the forward constraint in-code and the typechecker can plant a TODO that becomes a real error later. Prefer shipping single-threaded correctly over blocking on concurrency that isn't on the near-term roadmap.

## Group T — Test runner integration

### T1. Where does slice 3's "wrap in `with_provider`" live?  *(open — depends on M1)*

**The question.** When the runner executes `test_foo` with `#[with_provider(db.UserDB, InMemoryUserDB.new)]`, is the wrapping:

**Options:**

- **T1a. Source-level AST rewrite** before compile: take the test function body, wrap in a synthesized `with_provider[UserDB](InMemoryUserDB.new(), || { <original body> })` expression, re-typecheck the rewritten body.
  - Pro: test body runs under the exact same code path as hand-written `with_provider`. No duplicated logic in the runner.
  - Con: the rewrite has to produce valid AST with fresh spans; error messages need to map back to the original test; resolver/typechecker must already support `with_provider` as a parsed construct (currently it's only a runtime primitive, nothing resolves it).

- **T1b. Interpreter-level scope management** from the runner: before `call_function(name, &[])`, the runner pushes a provider frame with the constructed instance; after the call, it pops.
  - Pro: no AST rewriting, works even if the parser never learns a `with_provider` keyword (the runtime just has the builtin).
  - Con: tests get a codepath nothing else exercises — hand-written `with_provider` in test body (if ever used) goes through a different code path than the runner's wrap.

**Tiebreaker.** Depends on M1. If M1a (thread-local stack), T1b is natural: the runner pushes a frame, the interpreter's resource-using calls read from top-of-stack regardless of who pushed. The "two codepaths" worry dissolves because hand-written `with_provider` pushes the same frame.

### T2. Multiple `#[with_provider]` attributes — order.  *(open)*

**The question.** 
```
#[with_provider(Clock, FakeClock.at(T))]
#[with_provider(RandomSource, FakeRandom.from_seed(42))]
fn test_foo() { ... }
```
Which provider is the outermost wrapper? Does it matter? Do multiple attributes on the same test function evaluate all-before-scope-entered (like the `providers { }` block per design.md:4616) or sequentially?

**Options:**

- **T2a. Source order = outer-to-inner (first attr is outermost).** Evaluate all constructors before entering any scope (`providers {}` semantics). Attribute order matches how the user reads top-to-bottom.
- **T2b. Source order = inner-to-outer.** Attributes are listed in the order they'll be pushed from the innermost — unusual, violates reading order.
- **T2c. Order is undefined; providers are a set.** Valid since the design says providers for distinct resources don't aliase semantically; emit them in source order as a convention.

**Tiebreaker.** T2a. Matches the natural reading direction and matches `providers {}`'s evaluate-all-then-scope semantics. Document this in design.md § Testing as a one-line note.

### T3. Constructor failure.  *(open)*

**The question.** What happens when `constructor_fn()` panics, returns `Err`, or (in a hypothetical future) triggers an effect not declared on the test? The constructor runs before the test body — is the failure mode `test_fail`, a new `test_error` event, or something else?

**Options:**

- **T3a. `test_fail` with reason `"provider_construction_failed"`.** Constructors that panic or return Err surface as a normal test failure with a distinct `reason` field. JSONL consumers filter on it like they do `unsatisfied_requires`. `duration_ms: 0` (the body never ran).
- **T3b. New `test_error` event.** Separate event kind to distinguish "the test's preconditions were broken, not the assertion" from "the test's assertion failed".

**Tiebreaker.** T3a. The JSONL schema from slice 2 already uses `reason` as the discriminator; piling a new event kind on when a `reason` tag suffices grows schema for no reader benefit. Tests that succeed get `test_pass`; tests that fail for any reason get `test_fail` with a `reason` tag that disambiguates.

### T4. Interaction with `#[test(requires = [...])]` on the same function.  *(open)*

**The question.** Is this legal?
```
#[test(requires = [db.UserDB])]
#[with_provider(db.UserDB, InMemoryUserDB.new)]
fn test_foo() { ... }
```
Design.md:2491 says the two are distinct: `requires` is for tests that need a real external service, `with_provider` is for tests that supply a fake. A test that uses both on the same resource is contradictory — either you're using the real thing (requires) or the fake (with_provider).

**Options:**

- **T4a. Reject at discovery time.** If a test has both `#[test(requires = [X])]` and `#[with_provider(X, ...)]` for the same resource X, emit a `test_error` / `test_fail` with `reason = "requires_and_with_provider_conflict"` and don't run the test.
- **T4b. Silently let `#[with_provider]` take priority.** Probe for `requires` as usual; if the env var / health check is absent, still run the test as long as the resource is provided by `#[with_provider]`.
- **T4c. Silently run both — `requires` gate passes only if the env var is set, independent of `#[with_provider]`.** This is the simplest "attributes are independent" reading but produces confusing skip messages (test was skipped because resource X was unavailable — but the test supplied its own fake).

**Tiebreaker.** T4a. The design's framing that these attributes serve disjoint purposes ("real service" vs "in-memory fake") means the combination is almost always a user mistake — a copy-paste from a different test. Loud rejection beats silent strange behavior. Different resources in the two lists are fine (e.g., requires real database but supplies a fake clock).

### T5. `#[with_provider]` on built-in primitives (Clock, RandomSource, Env).  *(open)*

**The question.** Design.md:3994 shows `#[with_provider(Clock, FakeClock.at(...))]` and `#[with_provider(RandomSource, FakeRandom.from_seed(...))]`. These are program-rooted primitives, not user-declared `effect resource`s. Can the test runner inject them the same way as a user-defined resource?

**Options:**

- **T5a. Same mechanism, same code path.** Built-in primitives are registered in the interpreter's resource registry with a default provider at program start; `#[with_provider]` pushes a replacement on the thread-local stack for the duration of the test. Identical to user resources after registration.
- **T5b. Separate knob — test-only builtin-override.** Special-case built-in primitives with their own override plumbing.

**Tiebreaker.** T5a is both simpler and matches design.md's framing ("This is the same mechanism already described for user-defined resources; the nondeterminism resources are blessed so that every Kāra program can use the same pattern"). Ensures the feature set is uniform — users aren't surprised that `#[with_provider(Clock, ...)]` works in some contexts but not others.

### T6. JSONL surface — does `test_pass` / `test_fail` mention providers?  *(open)*

**The question.** Current events carry `test`, `duration_ms`, optional `reason`/`resources`/`message`/`left`/`right`/`location`. Should they grow a `providers: [...]` field listing which providers were active for this test?

**Options:**

- **T6a. No new field.** The attribute is visible in source; reproducing it in events duplicates what consumers already have via `gh api ... pulls/diff` or static analysis.
- **T6b. Add `providers`.** LLM tooling that analyzes a JSONL stream in isolation (no source access) benefits from seeing which fixtures wrapped a failing test.

**Tiebreaker.** T6a for the common case, T6b for failure events only. `test_fail` gets `providers: [...]` so post-hoc triage is self-contained; `test_pass` stays lean. Consistent with how `resources` only shows up on skip/fail events, never on pass.

## Group E — Escape check sequencing

### E1. Does slice 3 block on the provider-escape check?  *(open)*

**The question.** Implementation_checklist.md:77 commits to rejecting code paths that let closures capturing provider-rooted resources escape their `with_provider` block. Without that check, a test can leak provider state across tests via a shared scheduler / worker pool / static — the exact class of bug design.md:4585 flags as "a bug no typechecker could catch after the fact."

**Options:**

- **E1a. Escape check must land before `with_provider` runtime.** Otherwise the design's "test isolation is mechanical" promise is empty.
- **E1b. Escape check lands after runtime but before slice 3 ships to users.** Sequence: runtime CR → escape check CR → slice 3.
- **E1c. Escape check is independent of slice 3.** The runtime itself is correct for single-threaded, non-leaky tests; escape leaks are a safety upgrade that can land any time.

**Tiebreaker.** E1c in practice: most tests don't spawn tasks or store closures in statics. The common failure mode from a missing escape check (test leaks provider into a global, next test sees stale state) is observably a flaky-test bug the user will debug and fix — not a silent corruption. E1a would block slice 3 on an ownership-pass item that's a separate CR with its own complexity. Document the gap in slice 3's landing note: "Provider escape check (implementation_checklist.md:77) not yet enforced — tests that leak providers into statics or spawned tasks will produce cross-test state contamination; track and land the check as a follow-up." User accepts, or upgrades to E1a.

## Group S — Sequencing

### S1. CR order.  *(open — depends on M3, E1)*

**The question.** What's the minimal CR chain that ships slice 3?

**Options (assuming M3a interpreter-first and E1c escape-check-deferred):**

- **S1a. One CR: `with_provider` interpreter + slice 3 test wiring + `providers {}` block parse.** Big CR but all one theme.
- **S1b. Two CRs: (1) `with_provider` interpreter only, (2) slice 3 + `#[with_provider]` attribute wiring.** Smaller CRs, each reviewable alone.
- **S1c. Three CRs: (1) `effect resource X: Trait` trait dispatch (the resolver work), (2) `with_provider` interpreter, (3) slice 3.** Most granular.

**Tiebreaker.** S1b. The trait-dispatch work (implementing resource method calls to dispatch through the active provider) is what makes `with_provider` do anything useful — separating it from the runtime just spreads one logically-inseparable change over two CRs. Slice 3 is cleanly decoupled (it's just the attribute + test-runner integration) and benefits from being its own commit in the CR-24 follow-up series.

### S2. Does slice 4 (coverage) jump ahead?  *(open)*

**The question.** Slice 3 now has a multi-CR dependency chain (at minimum: `with_provider` runtime). Should slice 4 (`--coverage`) ship before slice 3, since it has no upstream gap?

**Options:**

- **S2a. Ship slice 4 next, slice 3 trails.** Breaks the 1-2-3-4 sequence but unblocks coverage for users today. Slice 3 lands whenever the runtime CR resolves.
- **S2b. Hold slice 4 until slice 3 lands.** Keeps sequencing tidy.

**Tiebreaker.** S2a. Sequencing discipline isn't worth holding shippable work hostage to an upstream gap. Slice 4's coverage plumbing (LLVM instrumentation, lcov output, `--min=N` gating) is independent of provider machinery. Rename slice 3 to "slice 3 (deferred pending `with_provider` runtime)" in implementation_checklist.md and proceed to slice 4.
