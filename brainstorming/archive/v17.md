## Tier 1: High Impact — Blocks Implementation or Creates Ambiguity

### 1. Atomics: fully specified yet listed as deferred

**Locations:** `Atomic[T]` spec at lines ~5247–5310; "channels / atomics" in Deferred Items section.

**Problem.** The Unsafe Escape Hatch section contains a complete specification of `Atomic[T]`: the type, all memory orderings (`Relaxed`, `Acquire`, `Release`, `AcqRel`, `SeqCst`), load/store/CAS methods, fences, and the `Copy` bound. This reads as a committed design. Meanwhile, the Deferred Items section lists "channels / atomics" as Phase 6 (P0).

A reader cannot tell whether:
- (a) `Atomic[T]` ships in v1 and "atomics" in the deferred list refers only to async channel primitives, or
- (b) the entire atomic specification is aspirational and deferred to Phase 6, or
- (c) ISR-safe atomics (the embedded use case) ship in v1 but general-purpose atomics are deferred.

This matters for the implementation checklist: if `Atomic[T]` is v1, the interpreter and future codegen need to support it. If it's deferred, the spec text should be moved to `deferred.md` or clearly labeled.

**Options:**

- **(a) Split the deferred entry.** Rename the deferred item to "channels (async message passing)" and confirm `Atomic[T]` as v1. The atomic spec stays in design.md as committed. This is the least disruptive reading of the existing text.
- **(b) Defer everything.** Move the full `Atomic[T]` spec to `deferred.md` and note it as Phase 6. Remove atomic examples from the unsafe section. Embedded users would lose atomics until Phase 6.
- **(c) Ship a minimal subset.** `Atomic[T]` with only `Relaxed` and `SeqCst` orderings ships in v1. `Acquire`/`Release`/`AcqRel` and fences are deferred. Simplifies the implementation while covering the most common patterns.

**Recommendation:** (a) — the embedded use case (ISR + main loop synchronization) is a stated project goal, and the spec is already complete.

### 2. `global_asm` effects not integrated into the effect system

**Locations:** Inline Assembly section (lines ~5180–5210); effect system core rule "every effect has an attributable source."

**Problem.** The spec states that `global_asm` has "unrestricted hardware effects by definition." But it provides no mechanism for those effects to propagate to callers. A function containing `global_asm` has no syntactic site where the compiler can infer or verify effects — the assembly is a black box.

This contradicts the core design principle stated in Module-Level Bindings (line ~517): "Kāra's entire effect-tracking discipline rests on the rule that every effect has an attributable source." If `global_asm` silently escapes effect tracking, it creates an invisible hole.

The inline `asm!` expression is handled correctly — it lives inside a function body, and the programmer annotates with `reads(Hardware)` / `writes(Hardware)`. The gap is specifically `global_asm`, which sits at module scope.

**Options:**

- **(a) Require effect annotation on `global_asm`.** `global_asm` blocks must declare their effects explicitly: `global_asm("...", effects: writes(Hardware) reads(Hardware))`. Any function that calls a symbol defined in `global_asm` must declare compatible effects. The compiler trusts the annotation (same trust model as FFI).
- **(b) Treat `global_asm` symbols as `extern "C"` with default effects.** Symbols defined in `global_asm` automatically get the FFI default effect set (`{blocks}`, empty resource effects). The programmer over-declares on the Kāra-side `extern` declaration that references the symbol.
- **(c) Exempt `global_asm` from effect tracking explicitly.** State that `global_asm` is outside the effect system by design — it's for interrupt vector tables, linker-script glue, and startup code that runs before `main`. Effects apply only to code reachable from `main`. Document the boundary clearly.

**Recommendation:** (b) — `global_asm` symbols are always accessed through an `extern` declaration on the Kāra side, so the existing FFI effect annotation mechanism covers them. The spec just needs to state this explicitly.

### 3. Effect-free atomics guarding effectful resources — no formal model

**Locations:** Atomic Operations section (~5247); Auto-Concurrency section (~4220); effect conflict rules (~2500).

**Problem.** Atomics are declared "effect-free synchronization primitives" — they don't contribute to a function's effect set. But in practice, atomics guard access to resources that *do* carry effects. Consider:

```kara
let lock = Atomic[bool].new(false)

fn writer_a() with writes(SharedBuffer) {
    while !lock.compare_exchange(false, true, Acquire, Relaxed) {}
    write_to_buffer()              // writes(SharedBuffer)
    lock.store(false, Release)
}
```

The auto-concurrency system sees `writes(SharedBuffer)` + `writes(SharedBuffer)` and serializes the two calls. The atomic spin-lock is redundant from the effect system's perspective. But what if the programmer uses a parameterized resource (`writes(Buffer[0])` vs `writes(Buffer[1])`) with an atomic to coordinate — does the effect system still serialize?

The fundamental question: does the effect system's conflict detection make atomics unnecessary for intra-program synchronization? Or are there cases where atomics are the only coordination mechanism and the effect system needs to be told "this is safe"?

**Options:**

- **(a) Atomics are for FFI/hardware only.** Within Kāra, the effect system handles all synchronization. Atomics exist for ISR communication, hardware registers, and FFI interop where the effect system can't see the other side. Document that intra-Kāra-code synchronization should use effects + `par {}` + `sequential {}`, never raw atomics.
- **(b) Atomics + `#[atomic_guard]` annotation.** A new annotation tells the auto-concurrency system that a pair of atomic operations constitutes a critical section. The effect system defers to the atomic guard within that region. Complex, potentially fragile.
- **(c) Atomics override effect serialization via `unsafe`.** Wrapping concurrent access in `unsafe` tells the compiler "I'm handling synchronization manually." The effect system's conflict detection is suppressed within the `unsafe` block. This is the Rust model.
- **(d) State the relationship informally.** Add a paragraph saying: "Atomics are orthogonal to the effect system. The effect system serializes conflicting operations at the granularity of function calls. Atomics provide finer-grained synchronization within a single function or across FFI. Using atomics to bypass effect-system serialization is unsound unless the programmer also uses `unsafe` to suppress effect checking." No new mechanism needed.

**Recommendation:** (a) with (d) as documentation — the effect system is designed to make manual synchronization unnecessary within Kāra. Atomics exist for the hardware/FFI boundary. State this clearly.

### 4. Type inference bidirectional spec — v1 scope unclear

**Locations:** Type Inference section (lines ~155–247); implementation gap notes (lines 237–247).

**Problem.** The Type Inference section specifies a full bidirectional system: check mode, synthesis mode, subsumption, generic metavariable instantiation, effect variable solving, refinement proof obligations, and variance rules. The "Implementation gap" note at lines 237–247 then lists five missing pieces in the current `typechecker.rs`.

The gap note says these are tracked as a "Phase 3 upgrade" — but Phase 3 in the implementation checklist covers the effect checker and ownership checker, not the type system. And the spec doesn't distinguish which parts of the bidirectional system are v1 targets vs aspirational.

Specifically:
- **Check mode and subsumption** — needed for closures to type-check without explicit annotations. Likely v1.
- **Generic call-site metavariables** — needed for any generic function call. Definitely v1.
- **Effect variable machinery** — depends on the effect checker. Likely tied to the effect checker phase.
- **Refinement proof obligations** — could be v1 (for basic constant-folding discharge) or post-v1 (for SMT integration).
- **Bidirectional subsumption through function types** — needed for effect-set covariance. Likely tied to effect polymorphism.

Without labeling, an implementer doesn't know what to build first or what can wait.

**Options:**

- **(a) Label each gap item with its target phase.** Add a column to the implementation gap list: "Phase 2" for check mode and metavariables, "Phase 3" for effect variables, "Phase 5+" for refinement proof obligations and full subsumption.
- **(b) Split the spec into "v1 target" and "full target" subsections.** The v1 target includes check mode, generic metavariables, and basic subsumption. The full target adds refinement proofs and effect-variable solving. Both live in design.md but are clearly labeled.
- **(c) Move the implementation gap to `implementation_checklist.md`.** The design spec describes the target state. The checklist tracks what's built. The gap list belongs in the checklist, not the spec.

**Recommendation:** (a) + (c) — label the phases in the checklist, remove the gap note from design.md (it's implementation status, not design), and add a one-line forward reference: "See `implementation_checklist.md` for current implementation status."

## Tier 2: Medium Impact — User-Facing Ambiguity

### 5. `ref(a, b)` vs "no named lifetime parameters"

**Locations:** "What Kāra Is Not" section (~5640); Part 3: Explicit `ref` for Borrow Returns (~3850).

**Problem.** The "What Kāra Is Not" section states:

> **No named lifetime parameters.** No `'a`, no lifetime annotations on every function.

But Part 3 introduces `ref(a, b) T` syntax for disambiguating borrow returns when multiple parameters could be the source:

```kara
fn pick(ref x: Vec[i64], ref y: Vec[i64], cond: bool) -> ref(x, y) i64 {
    if cond { x[0] } else { y[0] }
}
```

`ref(x, y)` is a parameter-name-based annotation that tells the compiler which parameters the return value borrows from. A reader familiar with Rust will reasonably ask: "How is `ref(x, y)` different from `'a` and `'b`?" The spec doesn't explain the distinction.

The distinction *does* exist — `ref(x, y)` is a set of source parameters, not a reusable scope variable. You can't write `ref(x, y) T` on a struct field or thread it through a chain of calls the way Rust lifetimes compose. It's a disambiguation annotation, not a lifetime system. But this isn't stated.

**Options:**

- **(a) Clarify the distinction in "What Kāra Is Not."** Change the text to: "No named lifetime parameters. No `'a`, no lifetime annotations on struct fields, no lifetime threading through generic bounds. Borrow returns use `ref(param_name)` disambiguation — a fixed annotation listing which parameters the return borrows from, not a reusable lifetime variable. `ref(x)` does not compose, does not appear on struct fields, and does not require callers to annotate." This explains the boundary.
- **(b) Add a "Comparison with Rust lifetimes" note in Part 3.** Keep the "What Kāra Is Not" text as-is but add a callout box in the borrow-returns section explaining why `ref(x, y)` is not a lifetime system.
- **(c) Rename `ref(x, y)` to avoid confusion.** E.g., `ref from(x, y)` or `borrows(x, y)`. This makes the "not a lifetime" nature syntactically obvious.

**Recommendation:** (a) — the clarification belongs where the claim is made.

### 6. `Hardware` resource scope and custom resource interaction

**Locations:** Volatile Memory Access section (~5200); Effect Types core definitions (~2496); Parameterized Resources (~3301).

**Problem.** The Unsafe Escape Hatch section introduces `Hardware` as a "built-in primitive resource" used by `volatile_read`, `volatile_write`, `VolatileCell`, and inline `asm!`. But the spec doesn't define:

1. **What `Hardware` covers.** Is it a single coarse resource for all hardware interaction (MMIO, ISR, DMA, GPIO)? Or is it intended as a fallback when the programmer hasn't defined something more specific?

2. **Interaction with custom resources.** An embedded programmer will likely define `resource Uart0`, `resource Gpio`, etc. Can `volatile_write` to a UART register carry `writes(Uart0)` instead of `writes(Hardware)`? Or does it always carry `writes(Hardware)`, and `Uart0` is a separate, higher-level resource?

3. **Aliasing.** If both `Hardware` and `Uart0` refer to the same physical register, does the effect system detect the alias? The resource aliasing section (~3440) describes alias detection for provider-rooted resources, but `Hardware` is a built-in primitive — does it participate in the same aliasing rules?

**Options:**

- **(a) `Hardware` is coarse; custom resources are layered on top.** `volatile_read`/`volatile_write` always carry `reads(Hardware)`/`writes(Hardware)`. Programmers define domain-specific resources (`Uart0`, `Gpio`) for their own functions. The two layers are independent — `Hardware` captures the physical operation, the custom resource captures the domain intent. No aliasing between them (they're at different abstraction levels).
- **(b) `Hardware` is parameterizable.** `volatile_write` carries `writes(Hardware[addr])` where `addr` is the memory-mapped address. Custom resources like `Uart0` alias `Hardware[0x4000_1000..0x4000_1FFF]`. The effect system detects conflicts between overlapping address ranges.
- **(c) `Hardware` is a default that custom resources replace.** When the programmer defines `resource Uart0` and annotates their UART wrapper functions with `writes(Uart0)`, the `writes(Hardware)` from the underlying `volatile_write` is subsumed. Only the most-specific resource propagates to callers.
- **(d) Leave `Hardware` coarse and add guidance.** Define `Hardware` as a single, non-parameterized built-in resource. Add a guidance note: "For fine-grained effect tracking in embedded code, wrap `volatile_read`/`volatile_write` in helper functions annotated with domain-specific resources. The `Hardware` effect on the inner call is an implementation detail; callers see only the wrapper's declared effects." This works because public function effects are declared, not inferred from the body.

**Recommendation:** (d) — it requires no new mechanism. The existing effect declaration rules on public functions already support this pattern. Add a worked example showing a UART wrapper.

### 7. String interpolation with nested brackets/quotes

**Locations:** String Interpolation section (~2026–2050); Generic vs Index Disambiguation (~146).

**Problem.** The String Interpolation section defines `f"..."` with `{expr}` interpolation but doesn't specify how the parser handles expressions containing characters that could terminate the interpolation. Specifically:

```kara
f"value: {map["key"]}"      // nested quotes
f"item: {arr[idx]}"         // nested brackets
f"pair: {(a, b)}"           // nested parens
f"nested: {f"{inner}"}"     // nested interpolation
```

The parser needs a rule for:
1. **Nested quotes** — does `"key"` inside `{ }` terminate the outer string?
2. **Nested brackets** — `[idx]` is fine (balanced), but what about `{` inside the expression (e.g., a struct literal `{Point { x: 1, y: 2 }}`)?
3. **Nested interpolation** — is `f"{f"{x}"}"` legal?

Most languages with string interpolation handle this via brace-counting (Kotlin, C#) or require escaping (Python f-strings handle nested quotes via different quote types). The spec is silent.

**Options:**

- **(a) Brace-counting with full expression parsing.** The parser counts matched `{`/`}` pairs inside an interpolation. `f"value: {map["key"]}"` works because `"key"` is inside a balanced `{...}` and the parser knows it's in expression mode. Nested `f"..."` inside an interpolation is allowed because the inner `f"..."` is a complete expression. This is the Kotlin/C# model.
- **(b) No nested quotes or interpolation.** `f"{ expr }"` allows only simple expressions — identifiers, field access, method calls, index operations. Struct literals, nested strings, and nested interpolations require a `let` binding. This is simpler to parse and to read.
- **(c) Allow nested brackets and parens but not nested quotes or interpolation.** `f"item: {arr[idx]}"` and `f"pair: {(a, b)}"` work via delimiter balancing. `f"value: {map["key"]}"` is a compile error — use `let v = map["key"]; f"value: {v}"`. This is a pragmatic middle ground.
- **(d) Full expression parsing (option a) but prohibit nested `f"..."`.** Nested interpolation (`f"{f"{x}"}"`) is a compile error. All other expressions are fine inside `{ }` because the parser enters expression mode and handles quotes, brackets, and braces normally. This avoids the hardest edge case while keeping the common cases ergonomic.

**Recommendation:** (d) — full expression parsing inside `{...}` is the user expectation (every major language with interpolation does this), but nested interpolation is genuinely confusing to read and rarely needed. Prohibiting it loses almost nothing.

## Tier 3: Lower Impact — Future Feature Boundary

### 8. Generators (`yield`) vs `suspends` — desugar boundary

**Locations:** Deferred Items section (generators listed as P0); Execution Verbs section (`suspends` defined, ~2548); "What Kāra Is Not" section (~5640).

**Problem.** The Deferred Items section lists "generators (`yield`)" as a P0 item (committed design direction, implementation deferred). The `yield` keyword is reserved. Meanwhile, the `suspends` execution verb is fully specified — the compiler inserts yield points automatically at `suspends` call boundaries.

When generators land, the design needs to answer:
1. Does `yield` inside a generator function produce a `suspends` effect?
2. Or is `yield` a separate mechanism (lazy iteration, not concurrency)?
3. Can a generator be used as a concurrent task (yielding values into a channel)?
4. Does a generator's caller see `suspends` in the effect signature, or does iteration hide it?

The tension: `suspends` means "cooperative yield to the runtime scheduler." Generator `yield` means "produce the next value in a lazy sequence." These are semantically different — one is about concurrency, the other is about iteration. But they share the mechanism of suspending and resuming a stack frame.

If generators desugar to `suspends`, then every `for` loop over a generator becomes a suspending operation, which would infect effect signatures throughout the codebase. If they don't, the language has two independent suspension mechanisms that look similar but behave differently.

**Options:**

- **(a) Generators are pure iterators, no `suspends`.** `yield` produces values lazily but does not interact with the concurrency runtime. A generator function's effect set includes whatever effects its body has (e.g., `reads(Db)` if it reads from a database) but not `suspends`. The `Iterator` trait is the interface. `suspends` remains concurrency-only.
- **(b) Generators carry `suspends` and the effect is hidden by `for`.** `yield` desugars to a `suspends` checkpoint. But `for x in generator` is sugar that absorbs the `suspends` — callers of the `for` loop don't see it in their effect set. The runtime can preempt inside the generator's yield points. This unifies the mechanism but requires an effect-absorption rule.
- **(c) Generators are a separate IR transform, unrelated to `suspends`.** The compiler transforms a generator function into a state machine (like Rust's async transform or Python's generator protocol). No effect system involvement. `suspends` is about the scheduler; generators are about the iterator protocol. Two implementations, clearly separated.
- **(d) Defer the decision.** State that generators are deferred and that their interaction with `suspends` will be resolved when the design is committed. Reserve `yield` and move on. Add a note: "Generators will not carry `suspends` unless the design explicitly opts in — lazy iteration is the default assumption."

**Recommendation:** (d) with a lean toward (a) — lazy iteration should not infect effect signatures. State the default assumption now so that downstream design (effect polymorphism, closure capture) doesn't accidentally close off option (a).

## Appendix: Editorial Issues (No Design Decision Needed)

The following issues were identified in the same review but require only editorial changes, not design decisions. They are listed here for completeness and can be resolved without brainstorming:

1. **Document size and navigability.** At 6042 lines, design.md is a book, not a spec page. Sections like Feature 2 (Effect Types) span ~1000+ lines. Consider splitting into separate files per feature with a top-level index document, or at minimum adding sub-ToC anchors within long sections. The current single-file structure makes it difficult to review individual features in isolation, increases merge conflict risk, and exceeds context limits for AI-assisted workflows.
2. **"Implementation gap" block in Type Inference section** (lines 237–247) — belongs in `implementation_checklist.md`, not design.md. Move it and add a forward reference.
2. **Phase numbering inconsistency** — some items reference "Phase 3," others "Phase 6," others "P0/P1/P2." Add a notation section to design.md defining these terms, or cross-reference the roadmap.
4. **Scattered deferred-item references** — inline deferrals should all point to `deferred.md` with a consistent format.
5. **Missing sub-ToC for Feature 2** — the effect system section spans ~1000 lines. Add sub-anchors to the main ToC.
6. **`program_effects: null` protocol contract** — state explicitly in the streaming phase events section that `program_effects` is `null` when the effect phase is skipped.
7. **Error return traces in release builds** — state whether the `error_return_trace` JSON field is omitted or `null` in release mode.
8. **`#[repr(packed)]` + layout block interaction** — add a one-line rule: "packed disables SoA transforms" (or doesn't — but state which).
