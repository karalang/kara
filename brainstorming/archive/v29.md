# Brainstorming v29 — Path separator: `::` vs `.`

**Status: RESOLVED 2026-04-18 — (b) wins.** `.` is now the single path separator across every path-valued position; `::` has been removed from the lexer, parser, grammar, spec, book, examples, tests, and brainstorming citations. The parser disambiguates `.`-as-path from `.`-as-field-access by the case class of the leading identifier (Type/Const → path; Value → postfix), per `docs/design.md § Identifiers and Naming`. The companion example files (`v29_example_colon.kara`, `v29_example_dot.kara`) have been deleted. Original analysis preserved below for future reference.

---

## Original analysis

Triggered while reading `docs/book/src/ch13-modules.md`: "could we use `.` instead of `::` for imports?" On inspection this is a language-wide design call, not an import-site tweak.

---

## Background

Kāra uses `::` as the single path separator across every path-valued position:

- **Imports / re-exports:** `use db::connection::Connection;`
- **Associated-type projections:** `Option[Self::Item]`
- **Associated-function calls:** `Vec::new()`, `ExitCode::from(...)`
- **Enum-variant paths:** `IoError::InvalidUtf8`
- **Effect-group references:** `use logging::traces;`
- **`alias` and `independent` declarations**

Grammar (`docs/syntax.md:570`): `PATH_PREFIX = IDENT { "::" IDENT }`. Lexer emits `ColonColon` (`src/token.rs:145`). `docs/design.md` has ~231 occurrences of `::`.

`.` is the value-level operator: field access, method calls, `?.`, tuple indexing, pipe RHS. No grammar ambiguity with paths today because `::` and `.` never collide.

---

## Prerequisite — Identifier case classes as grammar ✓ RESOLVED & FORMALIZED

Settled first because it changes what is technically possible for Q1. **Spec has landed:** see `docs/design.md § Identifiers and Naming` (rules CN-1 through CN-7) and `docs/syntax.md §1.5 Identifier Classes` for the grammar terminals (`TYPE_IDENT`, `CONST_IDENT`, `VALUE_IDENT`).

**Summary of the rule:**
- Types, enum variants, traits, generic params, effect resources → **Type class** (PascalCase, e.g. `UserAccount`, `IoError`).
- Module-level `const` bindings → **Const class** (all-uppercase, e.g. `MAX_RETRIES`).
- Functions, `let` bindings, params, fields, modules, effect groups, effect verbs → **Value class** (snake_case or leading `_`).
- Acronyms treated as words: `HttpClient` not `HTTPClient` (CN-4).
- FFI escape hatch via `#[kara_name = "..."]` on extern items (CN-7).
- Non-ASCII identifiers deferred.

**Why this had to land first:**
- Makes `.`-as-universal-separator *lexer-deterministic*. The main grammar objection to option (b) below only exists in the absence of this rule.
- Independent payoff regardless of Q1's outcome — ecosystem uniformity, no PR bikeshedding.

**Enforcement implementation** is tracked as a separate work item in `docs/implementation_checklist.md § 5.2`.

## Q1. Path separator — `::` or `.`?

### Framings

- **(a) Keep `::` everywhere.** Status quo.
- **(b) `.` everywhere; `::` removed.** With enforced caps (above), disambiguation is lexer-deterministic.

### Arguments for (b)

- **Audience size.** Dot-path family (Java, Kotlin, Swift, C#, Python, JS/TS, Go) is far larger than `::` family (Rust, C++). Adoption friction matters.
- **Writer-vs-reader asymmetry.** With LLMs writing most code, humans mostly read. Most readers default-parse dot-paths.
- **Visual weight.** `use std.collections.Map;` genuinely reads lighter than `use std::collections::Map;`.
- **One operator vs. two.** No decision at every use-site. IDE autocomplete mitigates for writers, but not for learners, whiteboard, plain-text editors.
- **Enforced caps collapses the only technical objection** — ambiguity becomes a solved lexer problem.

### Arguments for (a)

- **Zero migration cost.** ~231 occurrences in `design.md` alone, plus lexer/parser, book chapters, examples, tests, brainstorm citations.
- **Consistent with the `[T]` precedent** of unambiguous-by-grammar over convention-based. *Soft* — enforced caps keeps (b) unambiguous-by-grammar too, just with a capitalization rule instead of a separator rule.
- **Semantic-family signaling.** `::` primes readers for Rust-family semantics (borrows, moves, effects); `.` might prime them for Java semantics. *Weak* — Swift uses `.` with ARC/value semantics and nobody confuses it with Java. Readers learn semantics from behavior, not separators.

### Not an argument

**Safety.** `vec.new()` vs `Vec::new()` fails either way. The separator isn't preventing bugs, just signaling visually.

### Current lean

**~60/40 toward (b).** The writer-overhead and audience-size arguments compound over the life of the language; migration tax is paid once. Not strong enough to commit blind.

## Q2. Timing — when, not whether

The real deadline is **before external readers first encounter the language.** Once `::` appears in a tutorial, README, blog post, or talk, switching creates reader churn and outdated material that can't be walked back.

**Cost-of-migration curve:**

| Milestone | Migration cost |
|---|---|
| Today (Phase 7.x) | ~4–6 hours focused work |
| Post-stdlib (Phase 8) | ~2× today |
| Post-first-public-moment | Internal cost still bounded; **external cost unrecoverable** |

**First public moment envelope:** post-Phase-8, one polished book chain end-to-end (getting-started → guessing game → small CLI tool), compiler stable enough not to surprise first-time users. Rough estimate 3–6 months from now.

### Proposal: resolve via spike, not calendar wait

Rather than set a date and wait, run a **one-session spike**:

1. Migrate lexer + parser to `.` on a branch.
2. Convert `design.md` and one book chapter.
3. Read the result end-to-end.
4. Commit the branch or revert.

Converts the 60/40 into 80/20 or 35/65 with real evidence, which is what the decision is actually missing. Bounded cost, resolves faster than waiting for more Kāra code to exist.
