# Brainstorming v18 — Deferral Candidates and Phase Renumbering

A review of `docs/design.md` and `docs/roadmap.md` identifying features currently scheduled for Phases 3-8 that can be safely deferred. Items 5-7 (Gradual Verification enforcement) move to a **new Phase 9** (post-MVP correctness layer). The remaining items move to Phase 10 (P1). Current Phase 9 (Additional Targets) and Phase 10 (Self-Hosting) shift to Phases 10 and 11 respectively.

**Phase renumbering:**

| Old | New | Content |
|---|---|---|
| Phase 1-8 | Phase 1-8 | Unchanged — MVP compiler |
| *(new)* | **Phase 9 (P0)** | **Gradual Verification Enforcement** — refinement types, distinct types, contracts (parsing already done in Phase 2; enforcement deferred to here) |
| Phase 9 | **Phase 10** | Additional Compilation Targets (WASM, GPU, FPGA) |
| Phase 10 | **Phase 11** | Self-Hosting |

**Rationale:** Phases 1-8 build the MVP. Phase 9 adds the verification/correctness layer before new backends — language semantics should be fully locked before compiling to WASM/GPU. All three Phase 9 items are purely additive and non-blocking; parsing is complete, only type-checker enforcement and interpreter runtime checks are deferred.

## 1. Generators (`yield`)

**Current location:** Deferred Items table in design.md, listed as P0 targeting Phase 5-6. Depends on traits + generics.

**What it is:** `yield` keyword reserved. Generators desugar to `Iterator` implementations, providing lazy iteration without manual state-machine boilerplate.

**Why safe to defer to P1:** The deferred items table itself says "Add generators when manual `Iterator` boilerplate becomes friction." Manual `Iterator` impls work correctly without generators. No v1 feature depends on generators. `yield` is already a reserved keyword, so adding it later is non-breaking. The design is settled (v17 item 8 confirmed: `yield` is pure iteration, orthogonal to `suspends`).

**Dependencies:** None in either direction. No feature in Phases 1-8 is blocked on generators. The `Iterator` trait (Phase 8) works without generator sugar.

**Action:** Move from P0 (Phase 5-6) to P1 (Phase 10) in the deferred items table. Add entry to `deferred.md`.

## 2. Effect variable bounds (`with E: no writes(R)`)

**Current location:** Deferred Items table in design.md, P0 targeting Phase 7. Full design spec at design.md § Effect Variable Bounds.

**What it is:** Upper-bound constraints on effect variables — `with E: no writes(R)` (exclusion) and `with E: only reads(AuditLog)` (inclusion). Checked at monomorphization.

**Why safe to defer to P1:** The design spec itself states: "Unbounded effect variables (`with E`) cover the current systems coding use cases. Bounds require a more complex checker (effect set subsumption, not just propagation) and add surface area to error messages." The feature is explicitly opt-in per function. No existing or planned v1 code requires bounded effect variables — unbounded `with E` is sufficient.

**Dependencies:** Requires effect inference to be stable (already true by Phase 7). No downstream feature depends on effect variable bounds.

**Action:** Move from P0 (Phase 7) to P1 (Phase 10) in the deferred items table. Move detailed spec to `deferred.md`.

## 3. Complexity budgets (G43)

**Current location:** Deferred Items table in design.md, P0 targeting Phase 6-7. Depends on profiles (G46).

**What it is:** Static enforcement of stack depth and heap allocation budgets. The heap case is already covered by the effect system (`allocates(Heap)` detection). The remaining work is stack budget analysis requiring transitive call graph computation.

**Why safe to defer to P1:** The effect system already handles the heap allocation case (the more common concern). Stack depth analysis requires a fully resolved call graph and is primarily useful for embedded builds — a niche use case that doesn't block the core language. The deferred items table itself says "Revisit when profiles land."

**Dependencies:** Depends on profiles, which are themselves a Phase 6-7 concern. No v1 feature depends on complexity budgets.

**Action:** Move from P0 (Phase 6-7) to P1 (Phase 10) in the deferred items table. The related "Static stack depth analysis" entry (Phase 6+) already exists with a committed design shape — merge the two into a single P1 entry in `deferred.md`.

## 4. REPL (`karac repl`)

**Current location:** Phase 4 roadmap (unchecked item), Phase 5 roadmap (under CLI).

**What it is:** Interactive expression evaluation with persistent state — `karac repl` command.

**Why safe to defer to P1:** The REPL is a developer convenience tool, not a language feature. `karac run` for executing `.kara` files and `karac check` for type-checking are the critical CLI tools. A REPL requires significant additional infrastructure (incremental compilation, state persistence, expression-vs-statement disambiguation) that is orthogonal to the compiler pipeline. No test or example depends on REPL availability.

**Dependencies:** Requires the interpreter (Phase 4) and CLI infrastructure (Phase 5), but nothing depends on the REPL.

**Action:** Move from Phase 4/5 to P1 (Phase 10) in the roadmap. Add entry to `deferred.md`.

## 5. Refinement types — type-checker and interpreter

**Current location:** Phase 3.2 (unchecked: "Refinement types: constraint validation at construction boundaries; compile-time elision when provable; `TryFrom` generation") and Phase 4 (implicit — runtime enforcement of refinement constraints).

**What it is:** `type Percentage = f64 where self >= 0.0 && self <= 100.0` — numeric constraint validation at construction boundaries, compile-time elision when provable, and `TryFrom` generation for fallible construction.

**Why safe to defer:** Refinement types are a Gradual Verification Level 2 feature. Parsing is already complete (Phase 2, done). The type-checker and interpreter work is substantial — it requires proof obligation discharge, compile-time elision logic, and `TryFrom` generation. No other v1 feature depends on refinement type enforcement. Programs can use plain types with explicit validation for the same effect. The feature is explicitly labeled as part of the gradual verification roadmap, which is designed to be incrementally adopted.

**Dependencies:** Parsing is done. `TryFrom` generation depends on the `TryFrom` trait (Phase 8), so full refinement types can't ship until Phase 8 anyway. No feature depends on refinement type enforcement.

**Action:** Move type-checker and interpreter work from Phase 3-4 to new Phase 9 (Gradual Verification Enforcement). Parsing remains complete (Phase 2). Update roadmap.

## 6. Distinct types — type-checker and interpreter

**Current location:** Phase 3.2 (unchecked: "Distinct types: enforce opacity (no implicit operations); verify `#[derive]` compatibility").

**What it is:** `distinct type UserId = i64` — newtypes where the underlying type's operations are not implicitly available. `#[derive]` selectively re-enables specific traits.

**Why safe to defer:** Parsing is already complete (Phase 2, done). The type-checker enforcement (opacity checking, derive compatibility verification) is a self-contained addition that no other v1 feature depends on. Programmers can use wrapper structs with a single field for the same semantic effect. Distinct types are a convenience/safety feature, not a prerequisite for any pipeline stage.

**Dependencies:** Parsing is done. No downstream feature depends on distinct type enforcement.

**Action:** Move type-checker enforcement from Phase 3 to new Phase 9 (Gradual Verification Enforcement). Parsing remains complete. Update roadmap.

## 7. Contracts (`requires` / `ensures` / `invariant`)

**Current location:** Phase 3.2 (unchecked: "Contracts: verify `requires`/`ensures`/`invariant` expressions are pure; insert runtime checks in debug builds") and Phase 4 (implicit — runtime contract evaluation).

**What it is:** Gradual Verification Level 2.5. Function preconditions and postconditions (`requires`/`ensures`), struct invariants (`invariant`). Runtime-checked in debug builds, stripped in release.

**Why safe to defer:** Parsing is already complete (Phase 2, done). Like refinement types, contracts are part of the gradual verification gradient and are designed to be incrementally adopted. No v1 feature depends on contract enforcement. The purity verification for contract expressions and the runtime check insertion are both self-contained additions. Programs work correctly without contracts — they're a correctness aid, not a semantic requirement.

**Dependencies:** Parsing is done. The `#[checked]` production-mode variant is already deferred to P1 in `deferred.md`. No feature depends on contract enforcement.

**Action:** Move type-checker verification and interpreter runtime checks from Phase 3-4 to new Phase 9 (Gradual Verification Enforcement). Parsing remains complete. Update roadmap.

## 8. `#[tailrec]` verification

**Current location:** Phase 3.2 (unchecked: "`#[tailrec]` verification: verify every recursive call is in tail position; compile error if not"). Also listed in `deferred.md` as P1 for the codegen side (`musttail`).

**What it is:** Compile-time verification that every recursive call in a `#[tailrec]`-annotated function is in tail position. The codegen side (LLVM `musttail`) is already deferred to P1 in `deferred.md`.

**Why safe to defer to P1:** The verification pass is only meaningful paired with the codegen guarantee — verifying tail position without emitting `musttail` gives the programmer a false promise. Since the codegen side is already P1, the verification should move with it. No v1 feature depends on `#[tailrec]`. Users who need guaranteed non-overflow can use loops.

**Dependencies:** The codegen side is already P1 in `deferred.md`. Moving verification to P1 keeps them together.

**Action:** Move verification from Phase 3 to P1 (Phase 10). Update the existing `deferred.md` entry to note that both verification and codegen are P1.

## 9. `#[must_use]` enforcement

**Current location:** Phase 3.2 (unchecked: "`#[must_use]` enforcement: warn on unused values of annotated types or unused function return values").

**What it is:** Compiler warning when a value of a `#[must_use]`-annotated type or a `#[must_use]`-annotated function's return value is discarded without being used.

**Why safe to defer to P1:** This is a lint/diagnostic feature, not a semantic one — it produces warnings, not errors. No program behavior changes with or without `#[must_use]`. No v1 feature depends on it. The annotation is purely additive and can be enforced whenever the infrastructure is ready.

**Dependencies:** None. Purely additive.

**Action:** Move from Phase 3 to P1 (Phase 10). Add entry to `deferred.md`.

## 10. `stable` modifier on effect groups

**Current location:** Phase 2.6 (unchecked: "`stable` modifier on effect groups: `stable effect group Name = ...;`"). Not in the deferred items table.

**What it is:** An annotation on effect groups that marks them as part of the semver contract — adding or removing effects from a `stable` group is a breaking change.

**Why safe to defer to P1:** The `stable` modifier is only meaningful for library authors publishing packages with semver guarantees. Until the package manager and registry exist (Phase 7-8), there is no semver boundary to enforce. Effect groups themselves work without the `stable` modifier. The modifier is purely additive — existing effect groups are unaffected.

**Dependencies:** Only meaningful with a package ecosystem. No v1 feature depends on it.

**Action:** Move from Phase 2-3 to P1 (Phase 10). Add entry to `deferred.md`.
