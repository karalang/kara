## 1. `with` keyword inconsistently used in design.md effect examples

**Source:** syntax.md §7.2 vs design.md throughout

**Issue:** syntax.md grammar is clear — `EFFECT_LIST = "with" EFFECT_SPEC` — the `with` keyword is mandatory before any effect annotation. But many design.md examples omit `with`:

```
// design.md — MISSING `with`:
fn access(ref self, key: Key) -> Value reads(DB);        // line ~1346
fn handler(e: ParseError) -> AppError writes(Log)        // line ~1512
fn debug_log(msg: String) traces(Console) { ... }        // line ~2733
fn log_structured(msg: String) logs(Stdout) reads(Clock)  // line ~3013

// design.md — HAS `with`:
fn save(u: User) -> () with writes(UserDB) { ... }       // line ~2092
fn save_user(user: User) with writes(UserDB) { ... }     // line ~2610
```

**Decision needed:** This is a design.md bug — `with` is required per the grammar. All design.md examples missing `with` should be fixed.

> **RESOLVED:** Added `with` keyword to all 15 affected function signatures in design.md (lines ~1346, 1350, 1356, 1360, 1506, 1512, 2733, 2738, 2997, 3013, 3019, 3075, 3076, 3231, 5630).

## 2. Missing grammar: non-transparent user-defined verbs (`effect verb`)

**Source:** syntax.md §3.6, design.md §2723

**Issue:** design.md describes non-transparent user-defined verbs:

> Declaring `effect verb logs;` (without `transparent`) creates a user-defined verb that propagates like the eight built-in verbs.

But syntax.md §3.6 only has grammar for `TRANSPARENT_EFFECT`:

```
TRANSPARENT_EFFECT = "transparent" "effect" "verb" IDENT ";"
```

There is no grammar rule for plain `effect verb logs;`. This declaration form is missing from both the grammar and the ITEM production list.

**Resolution:** Add a grammar rule for non-transparent user-defined verb declarations:

```
EFFECT_VERB_DECL = [ "pub" ] [ "transparent" ] "effect" "verb" IDENT ";"
```

And add `EFFECT_VERB_DECL` to the ITEM production (replacing `TRANSPARENT_EFFECT`).

> **RESOLVED:** Replaced `TRANSPARENT_EFFECT` with `EFFECT_VERB_DECL = [ "pub" ] [ "transparent" ] "effect" "verb" IDENT ";"` in syntax.md §3.6 grammar and ITEM production. Added `effect verb logs;` example.

## 3. Missing grammar: module-level `let` / `let mut` bindings

**Source:** syntax.md §2 PROGRAM, design.md §486-511

**Issue:** design.md describes module-level bindings with `let` and `let mut`:

```
pub let MAX_RETRIES: i32 = 5
pub let TIMEOUT_MS: i64 = 60 * 1000
let mut SCRATCH: Array[u8, 256] = [0; 256]
```

But syntax.md's PROGRAM production and items list has no `LET_BINDING` at the top level. The only top-level binding form is `CONST_DECL` (`const NAME: TYPE = EXPR;`).

**Decision needed:** Are `const` and `let` both valid at module scope, or did design.md replace `const` with `let`? If both exist, what distinguishes them? If `let` replaces `const`, syntax.md §3.9 needs updating. The `const` keyword would also need to be reconsidered.

> **RESOLVED:** Kāra has no `const` binding form — `let` is immutable by default, making `const` redundant. Replaced `CONST_DECL` with `LET_DECL = [ "pub" ] "let" [ "mut" ] IDENT ":" TYPE "=" EXPR ";"` in syntax.md §3.9 and ITEM production. Added compile-time initializer rule description and examples matching design.md. Moved `const` keyword from "Declarations" to "Type system" group (still needed for `*const T` and const generics). Updated `pub const MAX` example to `pub let MAX`.

## 4. Missing keyword: `private` visibility modifier

**Source:** syntax.md §1.1, design.md §358

**Issue:** design.md describes a three-tier visibility model with an explicit `private` keyword:

> `private` restricts visibility to files in the same directory.

Example: `private let SCRATCH: i64 = 0`

But `private` is not listed in syntax.md §1.1 keywords, and syntax.md §9.4 says:

> No `pub(crate)` or `pub(super)` variants. Only `pub` (fully public) and private (default).

These contradict: design.md has an explicit `private` keyword, syntax.md says private is the default with no keyword.

**Decision needed:** Does `private` exist as a keyword? If yes, add it to syntax.md keywords and update §9.4. If no, update design.md.

> **RESOLVED:** `private` is an intentional keyword per design.md. Added `private` to syntax.md §1.1 keywords (Visibility group). Rewrote §9.4 with the three-tier visibility table from design.md (`pub` / default / `private`). Added `VISIBILITY = "pub" | "private"` production in §2. Updated all grammar rules from `[ "pub" ]` to `[ VISIBILITY ]` (except struct fields which only support `pub`).

## 5. Visibility section contradicts design.md three-tier model

**Source:** syntax.md §9.4, design.md §354-360

**Issue:** Beyond the `private` keyword question, the models themselves differ:

- **design.md**: Three tiers — `pub` (public to end users), default (project-wide), `private` (same directory only)
- **syntax.md**: Two tiers — `pub` (fully public) and private (default, module-internal)

The design.md model has a middle tier (project-internal but not public to end users) that syntax.md doesn't mention at all.

**Resolution:** Align syntax.md §9.4 with design.md's three-tier model, or explicitly note the simplification.

> **RESOLVED:** Fixed as part of item #4 — syntax.md §9.4 now matches design.md's three-tier model exactly.

## 6. `todo()` / `unreachable()` argument type: `String` vs `str`

> **RESOLVED:** Changed design.md from `str` to `String`. Kāra has no `str` type — string literals have type `String`, consistent with syntax.md.

## 13. Import syntax incomplete: no glob, rename, or multi-imports

**Source:** syntax.md §3.8, design.md (silent)

**Issue:** syntax.md defines minimal import syntax:

```
USE_DECL = "use" PATH ";"
PATH     = IDENT { "." IDENT }
```

This doesn't support:
- Glob imports: `use std.collections.*`
- Rename: `use std.collections.HashMap as Map`
- Multi-import: `use std.collections.{HashMap, HashSet}`
- Re-export: `pub use internal.Widget`

design.md mentions `pub use path.Item` for re-exports but doesn't define the full import grammar.

**Decision needed:** Which import forms does Kāra support? At minimum, `pub use` for re-exports needs grammar support since design.md relies on it.

> **RESOLVED:** All four forms supported. Rewrote syntax.md §3.8 USE_DECL grammar with: `[ VISIBILITY ]` prefix for re-exports, `"as" IDENT` for renames, `"{" USE_ITEM { "," USE_ITEM } "}"` for multi-imports, `"*"` for glob imports. Added examples for each form.

## 14. Pattern syntax gaps: no range patterns, no `@` binding

**Source:** syntax.md §5.6

**Issue:** syntax.md's PATTERN grammar is missing two forms common in pattern-matching languages:

- **Range patterns** in match: `0..=9 => "digit"` — neither doc addresses this
- **Binding with `@`**: `x @ Some(_) => use(x)` — neither doc addresses this

These are standard in Rust and likely expected. If intentionally omitted, this should be documented as a design decision.

**Decision needed:** Are range patterns and `@` bindings supported? If deferred, add to deferred.md.

> **RESOLVED:** Both supported. Design finalized now, implementation in Phase 9. Added `LITERAL "..=" LITERAL` (range) and `IDENT "@" PATTERN` (`@` binding) to syntax.md §5.6 PATTERN grammar with rules and examples. Added design spec to design.md (range patterns: integer/char only, no floats; `@` bindings: capture + test, composes with or-patterns and ranges). Added implementation checklist to roadmap.md Phase 9 "Extended Patterns" section. Updated design.md exhaustiveness text to remove "future extension" framing for range patterns.

## 17. Literal suffixes not in lexical grammar

**Issue:** design.md describes literal suffixes for forcing numeric types:

```
INTEGER = (DECIMAL | HEX | BINARY | OCTAL) [ TYPE_SUFFIX ]
TYPE_SUFFIX = "i8" | "i16" | "i32" | "i64" | "u8" | "u16" | "u32" | "u64"
```

**Resolution:** Add numeric literal suffix syntax to §1.3 grammar.

> **RESOLVED:** Added `INT_SUFFIX` and `FLOAT_SUFFIX` rules to syntax.md §1.3 lexical grammar. INTEGER now includes optional `INT_SUFFIX`, FLOAT includes optional `FLOAT_SUFFIX`.

## 20. `spawn()`, `TaskGroup`, channels not in syntax.md

**Source:** syntax.md (absent), design.md §4551-4584

**Issue:** design.md describes `spawn()`, `TaskGroup`, and channels as concurrency primitives. These are stdlib functions/types, not syntax, so their absence from syntax.md is arguably correct. However, syntax.md does document `par {}` and `seq {}` which are also concurrency constructs.

The inconsistency is that `collect_all` (a stdlib-like builtin) is documented in syntax.md §5.24 but `spawn()` and `TaskGroup` are not — even though they have comparable compiler-known semantics (scope-bound lifetime, structured concurrency invariant).

**Decision needed:** Should `spawn()` be listed as a compiler builtin in §5.24? Its scope-binding behavior is enforced by the compiler, similar to how `collect_all` has compiler-specific semantics.

> **RESOLVED:** `spawn()`, `TaskGroup`, and channels are **stdlib**, not builtins — they don't need special evaluation rules like `collect_all`'s auto-thunking. The compiler-enforced scope-binding of `TaskHandle` is a type system property, not a calling convention. Added a "Stdlib concurrency primitives" table and examples to syntax.md §5.9 (after `par`/`seq` blocks) documenting `spawn`, `TaskGroup`, and `Sender[T]`/`Receiver[T]` channels.
