# Brainstorming deferred_v2 — Layout blocks: scope, extensibility, and the gap between the pitch and the v1 mechanism

**Status:** RESOLVED 2026-04-24 — all five questions decided. Opened 2026-04-23 during a strategic pressure-test of Kāra's pitch against `docs/design.md`. The question was whether `layout` blocks, as currently spec'd, deliver the "cache-aware layouts" part of Kāra's pitch, or whether the mechanism was narrower than the framing suggested.

## Context

### What `layout` blocks currently are

Per `docs/design.md:3369–3402`, `docs/design.md:3451–3456`, and `docs/design.md:132`:

- **Opt-in per binding.** `layout entities: Vec[Entity] { group physics { position, velocity } ... }` attaches a layout plan to the binding `entities`, not to the type `Entity`.
- **SoA / field grouping.** `group` names a set of fields that are stored contiguously; cross-group borrows are proven non-aliasing (`docs/design.md:3400`).
- **Enums: `split_by_variant` only.** Each variant stored contiguously. Per-variant field grouping is out of scope (`docs/design.md:3395`).
- **No auto-detection.** Programmer must write the layout block (`docs/design.md:7490`). The compiler never infers SoA.
- **Project-internal.** A library exposing `Entity` always sees `Entity` as AoS in the library code; the consuming project's `layout` block affects only the project's binding (`docs/design.md:3401`).
- **Monomorphization per layout variant.** Two layouts of the same collection type produce distinct codegen monomorphs within a project (`docs/design.md:3393, 3401`).
- **`#[repr(C)]` / `#[repr(packed)]` take precedence.** Layout block becomes documentation-only; for `pub` types this is a compile error (`docs/design.md:3451–3456`).

### Where the pitch promises more

The "systems language with cache-aware layouts when the hot path needs them" pitch implies:

1. **Broad coverage** — any data type, including ones defined in libraries, can be laid out for the consuming workload.
2. **More strategies than SoA+groups** — hot/cold split, padding-for-alignment, interleaving, cache-line packing, array-of-bitfields, GPU mappings.
3. **Low-friction adoption** — ideally the compiler suggests a layout, or at least makes switching between layouts trivial.
4. **Composition with generics** — a `Vec[T]` laid out SoA works regardless of whether `T` came from this crate or another.

The v1 mechanism delivers #4 cleanly (monomorphization handles generics) and delivers a narrow slice of #1 (project-internal binding). It does **not** deliver the rest.

### Why the gap matters

- **Library types are the majority of types.** Most user code works with types from the stdlib or third-party packages. If `layout` only applies to project-internal bindings of project-internal types in homogeneous collections, the feature's reach in real programs is small. The pitch is "cache-aware when you need it" but the feature is "cache-aware when *all five* of these constraints align."
- **Auto-detection was deliberately rejected** (`docs/design.md:7490`), on the reasonable grounds that auto-SoA is semantic-changing and should be explicit. But "no auto" plus "project-internal only" together leaves no story for the 95% case. A user who reads "cache-aware layouts" and then finds they can't apply it to `Vec[T]` returned from a stdlib `parse_csv` call will feel the mechanism doesn't match the promise.
- **The ECS / game / HPC workloads** where SoA matters most also use third-party libs for physics, rendering, and data loading. A layout story that doesn't cross package boundaries loses those exact workloads.
- **LLM-era implication.** An LLM asked to optimize a hot loop will want to reach for `layout` and find it inapplicable to the data it received from a library function. The LLM-friendly "you can tell from the code what will happen" property breaks at the library boundary.

### What already exists in the design

- Intentional project-internal scoping (`docs/design.md:3401`): *"library code always sees the AoS layout of the types it declared."* This is a **committed** design choice, not an oversight — libraries must be stable against consumer layout decisions.
- `split_by_variant` for enums (`docs/design.md:3395, 4936`).
- Explicit rejection of field-level effect granularity as an analogous "schema leakage" concern (`docs/design.md:7488`).

## Prior-art reference

- **Jai** (Blow's language): `#as SOA` on a declaration; no cross-package layout story either.
- **Odin**: `#soa [N]T` type-level operator; SoA is part of the type, not the binding. Library types can be used SoA.
- **Mojo**: `@parameter` and layout-generic types with compile-time layout selection.
- **Rust**: no first-class SoA; community crates (`soa-rs`, `soa_derive`) macro-generate SoA wrappers at the type level. No compiler support.
- **C++ (experimental `std::mdspan`)**: layout policy as a template parameter. Very general; very verbose.

## Open questions

### Q1 — Is the v1 scope the right v1 scope, or is it too narrow to support the pitch?

**Decision (2026-04-24): C — cross-package binding-site layout.** `docs/design.md:3401` updated.

The "library always sees its own AoS" invariant is preserved: only the consuming project generates a new monomorph. The consumer may reference any `pub` field of the element type; private or nonexistent fields are compile errors. A library field rename or removal produces a compile error at the consumer's layout block — no silent fallback to AoS. This is the same cross-crate monomorphization mechanism used for generics; no new compiler concept is introduced.

D was ruled out: allowing library authors to set layout defaults creates subtle cross-crate coupling that Kāra's package-versioning model wants to avoid.

See **P1** below for implementation tasks.

### Q2 — Which additional layout strategies are worth committing to in the spec?

**Decision (2026-04-24): B — add `cold` section and `align(N)` group modifier in v1.** `docs/design.md` updated.

Rules committed:
- One `cold` section per layout block (compile error if more than one). Fields in `cold` go to a separate heap allocation. Unassigned fields remain hot by default (implicit trailing hot group — same as before). A field cannot appear in both a `group` and `cold`.
- `align(N)` modifier on any `group`. `N` must be a power of two. Primary use: false-sharing avoidance between threads. Independent of `#[repr(align(N))]`.

Deferred: custom interleaving, bitfield packing, `aos` override, GPU mappings.

Note: the `layout { }` block is already implicitly a directive DSL — `group`, `split_by_variant`, `cold`, and `align` are all directives. This is worth a sentence in the spec eventually but no formal grammar extension is needed now.

See **P1** below for implementation tasks.

### Q3 — Auto-detection: ever, never, or behind a flag?

**Decision (2026-04-24): B — `karac explain` layout suggestions, diagnostic-only.** `docs/design.md:7492` updated.

The compiler never auto-transforms AoS to SoA. `karac explain` statically analyzes field access patterns and emits a concrete ready-to-paste `layout` block. No code changes without the programmer writing the block. C ruled out: flag-gated auto-layout makes behavior diverge between builds in hard-to-debug ways. D is a natural future extension of B once tooling exists.

See **P1** below for implementation tasks.

### Q4 — Composition with generics: is monomorphization per layout enough?

**Decision (2026-04-24): A — monomorphization is sufficient for v1.** No spec change needed; monomorphization per layout variant is already committed at `docs/design.md:3393, 3401`.

B ruled out: layout-generic functions only pay off for library authors writing functions that explicitly handle multiple layout variants — a narrow use case. Most functions access fields normally and get the correct monomorph automatically. C is the right long-term direction but is a separate design problem; deferred to v2.

### Q5 — What does the pitch actually promise, given the answer to Q1–Q4?

**Decision (2026-04-24): stronger pitch, earned by Q1=C and Q2=B.** `docs/design.md` feature intro updated.

Committed pitch: *"Three layout directives — `group` (SoA field grouping), `cold` (separate allocation for infrequently-accessed fields), `align(N)` (cache-line alignment for false-sharing avoidance) — applicable to any collection binding, including types from external packages. The library always sees its own layout unchanged; only the consumer generates a specialized monomorph. `karac explain` analyzes your loops and suggests the layout block. You write it; the compiler generates the code."*

Differentiation: Odin and Jai have SoA but no cross-package story and no tooling-assisted suggestion. Rust has no first-class layout control.

## Decisions

1. **Q1** — C: cross-package binding-site layout. `docs/design.md:3401` updated.
2. **Q2** — B: `cold` section and `align(N)` group modifier added to v1. `docs/design.md:3396–3397` updated.
3. **Q3** — B: `karac explain` suggests concrete layout blocks, diagnostic-only. `docs/design.md:7492` updated.
4. **Q4** — A: monomorphization is sufficient. No spec change needed.
5. **Q5** — stronger pitch committed, earned by Q1=C and Q2=B. `docs/design.md` feature intro updated.

## P1: Cross-Package Binding-Site Layout (Q1=C)

Implementation tasks, in dependency order:

1. **Field resolution across package boundaries.** The layout block's field names must resolve against the element type's `pub` fields, even when the type is declared in an external package. The resolver needs to look up the type's public field list from the package's exported symbol table (same mechanism as struct literal construction for imported types). Private or missing fields → compile error with a span pointing to the offending name in the layout block.

2. **Error on library field rename/removal.** When a library upgrades and a previously-referenced field no longer exists, the compile error must land at the consumer's layout block with a message like `field 'position' no longer exists on 'lib::Widget' — update or remove the layout block`. No silent fallback.

3. **Cross-package monomorph generation.** The codegen backend must emit a layout-specific monomorph for `Vec[lib.Widget]` inside the consumer's compilation unit. The monomorph is identical in structure to project-internal layout monomorphs; no new codegen path needed — only the triggering condition (element type from external package) is new.

4. **Incremental-build invalidation.** A consumer's layout monomorph must be invalidated when the referenced library type's public field set changes. The build system needs to track which external field names each layout block depends on, separately from the general "library API changed" invalidation.

5. **Diagnostic quality.** The `karac explain` output for a cross-package layout block should name the source package of the element type and confirm which fields resolved successfully. This makes debugging library-upgrade breakage straightforward.

**Q2 — `cold` and `align(N)`:**

6. **`cold` section parsing and validation.** The parser must accept a `cold { ... }` section in a layout block. Validation: at most one `cold` per block (compile error otherwise); a field may not appear in both a `group` and `cold` (compile error); unassigned fields go to the implicit trailing hot group, not `cold`.

7. **`cold` allocation codegen.** At collection construction, allocate a separate heap region for cold fields. Index into it by element index, same as hot groups. Drop semantics must cover both allocations. Hot-path iteration must never touch the cold allocation.

8. **`align(N)` parsing and validation.** The parser must accept `align(N)` as an optional modifier after a group's field list. Validation: `N` must be a power of two (compile error otherwise); `align(N)` on a `cold` section is not allowed in v1 (no defined semantics yet).

9. **`align(N)` codegen.** The group's backing array must be allocated with alignment `N`. Verify with static assertions in generated code that the alignment holds.

**Q3 — `karac explain` layout suggestions:**

10. **Field access pattern analysis.** A compiler pass that, given a collection binding, walks loop bodies and records which fields are accessed together. Output: a frequency map of (field set → loop count) per binding.

11. **Hot/cold classification.** From the frequency map, classify fields as hot (appear in at least one loop body) or cold (never appear in any loop body over this binding). This is purely static for v1 — no profiling required.

12. **Suggestion generation.** Format the analysis result as a concrete `layout` block. Group hot fields that co-occur in the same loops; place never-accessed fields in `cold`. Fall back to a single `group` if co-occurrence data is ambiguous.

13. **`karac explain` integration.** Wire the analysis into the `karac explain` command output. Each suggestion should include: the binding name, which loops triggered it, and the ready-to-paste `layout` block.

## Risks / follow-up

- **Incremental-build cost under Q1=C.** Cross-package monomorphs mean consumers recompile library generics for their layout variants. May interact badly with Kāra's package-manager caching story. Worth a dedicated follow-up if C is chosen.
- **`#[repr(C)]` restrictions tighten** as more strategies land. Hot/cold split cannot be applied to `#[repr(C)]` types at all (ABI-breaking). Alignment directives mostly can. Need a clear table in `docs/design.md:3451–3456`.
- **Interaction with shared types.** A `shared struct` with `layout group { ... }` — does SoA work over an Rc-managed collection? The RC header layout affects this. Probably yes (RC header is per-container, groups are internal), but worth a sentence.
- **Interaction with the effect system.** If hot/cold split moves a field to a separate allocation, does accessing the cold field contribute `allocates(Heap)`? Almost certainly no (the allocation is at container creation, not at field access) but worth being explicit.
- **Layout for `Map`, `Set`, tree-shaped containers.** Currently out of scope (homogeneous only). Probably stays out of scope for v1 — these containers' performance is dominated by the shape of the data structure, not field layout.
