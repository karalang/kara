## Tier 1: High Impact — Missing Definitions

### 1. `Send` and `Sync` traits used but never defined

**Locations:** Lines 2169, 3396, 3401, 3404, 3410, 3411, 4096, 4099, 5282, 5611.

**Problem.** `Send` and `Sync` are used as trait bounds throughout the spec — `with_provider` requires `P: Send + Sync` (line 3401), `shared struct` is described as "`Send` but not `Sync`" (line 4096), and closures escaping across parallel regions trigger `Send` channel analysis (line 2169). But neither trait is ever formally defined. Key questions are unanswered:

- Are they marker traits (no methods, compiler-enforced)?
- Are they auto-derived based on field types, or explicitly opted into?
- What are the rules for when a type is `Send` vs `Sync`?
- Where do they live in the stdlib layers (`core`? `alloc`? `std`)?
- Does `shared struct` have a blanket `impl Send` and explicit `impl !Sync`?

These are load-bearing concepts — `with_provider`'s signature requires them, `shared struct` concurrency safety depends on the `Send`-not-`Sync` distinction, and the Rc→Arc promotion algorithm uses `Send` as a trigger.

**Options:**

- **(a) Add a subsection under Type System and Object Model.** Define `Send` and `Sync` as auto-derived marker traits in `core`, following Rust's model: `Send` = safe to transfer ownership across threads, `Sync` = safe to share references across threads. `T: Sync` iff `ref T: Send`. State the auto-derive rules (all fields `Send` → type is `Send`; all fields `Sync` → type is `Sync`). State the exceptions: `shared struct` is `Send` but not `Sync` because interior mutability without synchronization is unsound under concurrent access.
- **(b) Add to the Shared Types section.** Define them inline where they're first load-bearing (Part 5). Less discoverable but keeps the definition near its primary use.
- **(c) Defer to a future concurrency section.** Note them as "defined when the concurrency runtime is specified" and add forward-reference stubs.

**Recommendation:** (a) — these are fundamental enough to deserve a top-level definition.

### 2. `spawn()` used in examples but never specified

**Locations:** Lines 2160, 2169, 4352–4353, 4493, 5290.

**Problem.** The async I/O example at line 4352 uses `spawn(|| handle_client(conn))` as if it's a known primitive. The embedded profile section (line 5290) bans `spawn` in embedded contexts. The closure escape section (line 2160) references `spawn` as a trigger for escape analysis. But `spawn` has no section, no signature, no effect declaration, and no relationship to structured concurrency.

Key questions:

- Is `spawn` a language primitive or a stdlib function?
- What is its effect signature? Does it carry `allocates(Heap)`? `suspends`?
- Does a spawned task participate in the structured concurrency join barrier described in `defer`/`errdefer` (line 339)?
- How does `spawn` interact with `collect_all` and `par {}`?
- Does a spawned task inherit the parent's provider-rooted resources?

**Options:**

- **(a) Add a Structured Concurrency subsection under Feature 5.** Define `spawn` as a stdlib function that creates a child task within the current scope's structured concurrency boundary. Signature: `fn spawn[T, with E](f: Fn() -> T with E) -> TaskHandle[T] with E`. The task joins at scope exit (structured). Escaping a scope without joining is a compile error.
- **(b) Define `spawn` as unstructured and separate from `par {}`.** `spawn` creates an independent task not bound to a scope. This matches the example at line 4353 (accept loop spawning per-connection tasks). Requires defining task lifetime, cancellation, and panic propagation independently.
- **(c) Defer `spawn` entirely.** Remove the examples that use it. The auto-concurrency system handles implicit parallelism; explicit `spawn` is a Phase 2+ feature.

**Recommendation:** (a) or (b) — the examples already use it, so it needs at least a stub definition. If deferred, the examples must be rewritten.

### 3. `par {}` referenced repeatedly but never formally defined

**Locations:** Lines 2654, 2656, 2661, 2675, 2705, 3961, 4493, 4624.

**Problem.** The document references "inside a `par {}` region" in the `dbg()` section (line 2654), logging span propagation (line 2705), and scheduler backpressure (line 4493). But no section defines `par {}`. Auto-concurrency (Feature 5) is implicit — the compiler parallelizes based on data deps and effect conflicts. So what is `par {}`?

Possibilities implied by context:
- An explicit parallel block (the counterpart to `sequential {}`)?
- Shorthand for "a region the compiler chose to parallelize"?
- The structured concurrency scope for `spawn`?

Without a definition, readers cannot distinguish compiler-implicit parallelism from programmer-explicit parallelism.

**Options:**

- **(a) Define `par {}` as an explicit parallel block.** The counterpart to `sequential {}`. Statements inside are parallelized regardless of effect conflicts (the programmer asserts safety). This is dangerous — contradicts the "no implicit parallelism of conflicting effects" non-goal.
- **(b) Define `par {}` as an explicit fork-join scope.** A structured concurrency block where each statement is a task. Effect conflicts within the block are still checked; `par {}` provides the join barrier. This is the structured concurrency primitive that `spawn` would operate within.
- **(c) Clarify that `par {}` is informal shorthand.** Every reference means "a region the compiler chose to parallelize." Replace `par {}` with "a compiler-parallelized region" in prose. No new syntax.
- **(d) Reserve `par` as a keyword (per v14 item 9) and define it properly.** v14 already resolved to reserve `par` and `seq` as keywords. This item follows through on that decision.

**Recommendation:** (b) or (d) — the language needs an explicit fork-join scope to complement the implicit auto-parallelism. `sequential {}` already exists as the opt-out; `par {}` is the natural opt-in.

### 4. Error return traces referenced but no section exists

**Location:** Line 257 references `(see [Error return traces](#error-return-traces))`.

**Problem.** The Error Handling section references an "Error return traces" section that does not exist as a heading anywhere in the document. The concept is described inline — line 257 explains `?` propagation traces, lines 286–298 show a JSON trace format, and line 4444 mentions `Cancelled` exclusion from traces, line 4781 shows another trace JSON — but there is no dedicated section defining:

- When traces are collected (debug builds only? always?)
- The exact data model (the JSON at line 289 is an example, not a schema)
- How traces interact with `.context()` messages
- Whether traces are part of the guaranteed, reported, or implementation-freedom layer
- How traces are accessed programmatically (are they in the `Err` value? a side channel?)

**Options:**

- **(a) Add a subsection `### Error Return Traces` under Error Handling.** Define the trace data model, collection rules, specification layer (reported behavior — traces may change format across compiler versions), and programmatic access.
- **(b) Fold into the existing `.context()` subsection.** Expand lines 257–298 into a complete specification rather than a cross-reference to a nonexistent section.

**Recommendation:** (a) — the concept is referenced from multiple locations and deserves its own anchor.

## Tier 2: Medium Impact — Design Tensions

### 5. `allocates(Heap)` default-permitted vs. viral annotation rule

**Locations:** Lines 2677 (default-permitted), 3039 (viral annotation rule).

**Problem.** Two rules interact in a potentially confusing way:

1. `allocates(Heap)` is default-permitted — public functions don't need to declare it (line 2677).
2. The viral annotation rule says `with _` must be declared at every public boundary when calling a polymorphic function, "regardless of whether the closure's effects are default-permitted" (line 3039).

Consider: `pub fn map(list: Vec[T], f: Fn(T) -> U with _) -> Vec[U] with _`. If a caller passes a closure whose only effect is `allocates(Heap)`, does the caller need `with _` on its own public signature?

The answer per line 3039 is yes — "the annotation is about the mechanism." But this creates a situation where `with _` is required even when the resolved effect set is entirely default-permitted and would require no annotation if the effects were direct rather than polymorphic.

**Options:**

- **(a) Add a worked example to the viral annotation section.** Show the `allocates(Heap)`-only case explicitly. Confirm that `with _` is still required. Explain that the mechanism annotation serves a forward-compatibility purpose — the closure may gain non-permitted effects later.
- **(b) Exempt default-permitted effects from the viral rule.** If the resolved effect set at a call site contains only default-permitted effects, `with _` is not required on the caller. This reduces annotation burden but creates a subtle rule interaction.
- **(c) Keep as-is but add a note.** The current rules are self-consistent; the confusion is about expectations, not correctness. A brief note in the viral annotation subsection ("even when the resolved effects are default-permitted, `with _` is required because...") would suffice.

**Recommendation:** (a) or (c) — the rules are sound, but the interaction needs a worked example to prevent implementer confusion.

### 6. No flow-sensitive narrowing for refinement types — ergonomic cost

**Location:** Lines 1601–1605.

**Problem.** The spec explicitly defers flow-sensitive narrowing:

> `if x > 0 { f(x) }` where `f` expects `Positive` is a compile error inside that branch.

The workaround is `Positive.try_from(x)?` inside the branch. This is correct but creates friction, especially given the "AI writes most code" assumption (line 80). An AI generating `if x > 0 { f(x) }` will hit a compile error every time and need to restructure to `if x > 0 { f(Positive.try_from(x)?) }` — which is redundant (the `try_from` can never fail given the guard).

The spec acknowledges this: "Occurrence typing is deferred indefinitely — it interacts with pattern matching, closures, and effect analysis in ways that are disproportionate to the ergonomic win for numeric refinements."

**This is not a request to implement flow-sensitive narrowing.** The concern is that the ergonomic cost should be documented more prominently — perhaps as a known limitation in a "Refinement Types — Known Limitations" subsection — so that AI tooling can be trained to emit the `try_from` pattern proactively rather than discovering it through compile errors.

**Options:**

- **(a) Add a "Known Limitations" note to the Refinement Types section.** List the guard-then-use pattern as a known ergonomic gap with the canonical workaround.
- **(b) Add a compiler diagnostic for the specific case.** When `f(x)` fails because `x: i64` where `f` expects `Positive`, and the call is inside an `if x > 0` branch, emit a targeted suggestion: "use `Positive.try_from(x)?` — the check is guaranteed to succeed given the guard."
- **(c) No change needed.** The spec already acknowledges the deferral and the existing diagnostic infrastructure handles it.

**Recommendation:** (b) — a targeted diagnostic is low-cost and directly helps the AI-first workflow.

### 7. `Contextual[E]` and blanket `From[E]` potential coherence conflict

**Location:** Line 284.

**Problem.** The spec says `From[E]` is implemented for `Contextual[E]`, allowing `?` to propagate a bare `E` into a function returning `Contextual[E]`. But there is a standard blanket impl `impl[T] From[T] for T` (identity conversion). If `E` is itself `Contextual[SomeError]`, then `From[Contextual[SomeError]] for Contextual[Contextual[SomeError]]` competes with the identity impl.

More concretely: the spec says `impl[E] From[E] for Contextual[E]`. This means for any concrete `E`, there's a `From` impl that wraps it. But what about `From[Contextual[E]] for Contextual[Contextual[E]]`? This overlaps with the identity `From[T] for T` when `T = Contextual[E]`.

**Options:**

- **(a) Clarify that the identity `From[T] for T` blanket wins.** Kāra rejects overlapping impls (line 1178), so one must be excluded. The identity impl is the more general; the `Contextual`-wrapping impl should be written as `impl[E] From[E] for Contextual[E] where E: !Contextual` — but negative bounds don't exist in the spec.
- **(b) No blanket `From[T] for T` in Kāra.** Unlike Rust, Kāra may not have the identity impl. Clarify.
- **(c) `Contextual` wrapping is a compiler-inserted desugaring, not a trait impl.** `.context()` returns `Contextual[E]` directly; the `From` impl exists only for `?` propagation across error types, and the coherence issue doesn't arise because the compiler disambiguates at the `?` desugaring site.
- **(d) Acknowledge and defer.** Note the potential overlap and state that the coherence rules will resolve it when `Contextual` is implemented.

**Recommendation:** (c) or (d) — this is a subtlety that matters at implementation time. At minimum, note the interaction.

### 8. `collect_all` fixed-arity overloads mechanism unspecified

**Location:** Lines 4396–4398.

**Problem.** The spec says `collect_all` is "a compiler builtin resolved via fixed-arity overloads — maximum 8 branches." But there is no mechanism in the language for fixed-arity overloads:

- Kāra has no function overloading (no two functions with the same name and different arities).
- Kāra has no variadic generics.
- Kāra has no macro system.

So how does `collect_all` work? Is it compiler magic (a special-cased builtin)? A family of functions (`collect_all_2`, `collect_all_3`, ...)? A trait-based mechanism?

**Options:**

- **(a) Define `collect_all` as a compiler builtin.** Like `dbg()` or `todo()`, it's recognized by the compiler and desugared to the appropriate arity. This is the simplest explanation and matches the "compiler builtin" phrasing.
- **(b) Define it as a family of functions.** `collect_all` is overloaded by arity in the same way that tuple types are — the compiler generates `collect_all` for arities 2–8. This is closer to how Rust handles tuple impls.
- **(c) Note it as a spec gap to be resolved when the stdlib is designed.** The semantics are clear; the mechanism is an implementation detail.

**Recommendation:** (a) — state explicitly that `collect_all` is a compiler builtin with fixed-arity desugaring, same category as `dbg()` and `todo()`.
