## Category A: High-Risk Semantic Core

### 1. ~~Effect lattice structure and SCC termination not formally defined~~ ✅ RESOLVED

**Decision: Option (a) — a formal lattice spec written directly into design.md, with one shared carrier for resource and execution atoms (no separate product lattice).**

The effect lattice is now specified as `L = ⟨ P(Atoms), ⊆, ∪, ∩, ∅, Atoms ⟩`, where `Atoms = ResourceAtoms ∪ ExecutionAtoms` are both members of a single finite powerset carrier. An SCC mixing `writes(DB)` with `blocks` is handled by a single join, not a product construction. The surface-level distinction between resource and execution verbs remains (effect groups, profile rules, diagnostics) but is invisible to the fixed-point computation — rejecting the agent's premise that a product lattice was required.

The transfer function `T_f(φ) = direct(B_f) ∪ ⋃{φ(g) | g private callee} ∪ ⋃{declared(g) | g public callee}` is monotone in `φ` because public-callee contributions are constant and private-callee contributions grow monotonically. Knaster–Tarski + finiteness gives a unique least fixed point `φ*`. The per-SCC iteration bound `n · m` (functions × atoms in scope) is stated as a hard bound, enforced as an internal-compiler-error safety net in `effectchecker.rs` against future monotonicity regressions.

**Effect variables are unification variables, not lattice elements.** A polymorphic SCC is analyzed once in symbolic form with `T_f` parameterized by `E`; the variable resolves by unification at call sites outside the SCC, not by join. Monomorphized instances inherit the symbolic fixed point with `E` substituted — they are not re-iterated. This closes the "effect variables have no stated role in the lattice" gap by clarifying that they deliberately sit outside it.

**Rationale for (a) over (b):** citing Lucassen–Gifford by reference would have required the reader to cross a bibliographic edge to understand Kāra's actual lattice, and the LG formulation uses a different carrier (effect rows with read/write/alloc verbs tied to region variables). Writing the few paragraphs in-place was cheaper than annotating how Kāra's lattice diverges from the reference. (c) was rejected because `src/effectchecker.rs` already exists — leaving the spec silent while the implementation locks in would invert the source-of-truth relationship.

**Changes applied to design.md:**
- New subsection **Effect Lattice** inserted in Feature 2 before **Mutual Recursion and SCCs**. Defines atoms, the lattice `L`, transfer function `T_f`, monotonicity, the `n · m` iteration bound (and ICE safety net), effect variables as unification variables, effect groups as sugar, and the `karac explain` chain format.
- **Mutual Recursion and SCCs** convergence sentence tightened to cite `T_f`, `L`, and the `n · m` bound.
- **Performance Principles › Known-risky passes › Effect inference SCC fixed-point** updated to cite Feature 2 › *Effect Lattice* and state the `n · m` bound as the reason the pass stays linear.

### 2. ~~Type inference algorithm not specified~~ ✅ RESOLVED

**Decision: Option (a) — bidirectional type inference, declared as the target algorithm in design.md, with an explicit implementation-gap note pointing at Phase 4 work in the checklist.**

The Type System and Object Model section now has a `### Type Inference` subsection specifying two judgments (`synthesize ⇒` and `check ⇐`), a subsumption rule connecting them through function-type variance, a mode table listing which contexts drive each judgment, per-expression-form rules for the core language (literals, let, if, match, call, method, lambda, struct literal, cast), a generics section on per-call-site fresh metavariables with local solving, an effect-variables-as-separate-namespace rule, and refinement-type proof obligations as check-mode side effects.

**Why bidirectional over HM or ad-hoc synthesis:**
- **Effect sets** are ordered by subset inclusion, not unified. Check mode flows the caller's expected effect set into the closure in the one direction that matches the subtyping relation; an HM row-variable encoding of effects can be made to work but duplicates machinery the effect lattice already provides.
- **Refinement predicates** (Feature 6 Level 2.5) are proof obligations, not equalities. They are discharged in check mode (`prove e ⇒ P`) because the expected predicate is known; synthesis mode cannot produce obligations without an expected type to prove against.
- **Closure parameter inference from context** is a known gap in the current typechecker. Check mode is exactly the mechanism that resolves it — `|x| body` in a slot expecting `Fn(A) -> B` gives `x : A` for free, whereas synthesis mode must fall back to explicit annotations.

**Why not (b) HM with constraint generation:**
- Effects don't fit the unification framing cleanly (subset, not equality).
- Refinement obligations would need a second pass or a separate solver, losing the compositional advantage HM is chosen for in the first place.
- The HM "infer everything at once, then solve" model makes diagnostics worse — an error at the solver stage often points far from the offending expression, whereas bidirectional typing catches errors at the site where check mode fails.

**Why not (c) ratify current implementation:**
- `src/typechecker.rs` today is bottom-up synthesis with post-hoc `check_assignable`. Declaring that as the spec would lock in the unresolved-closure-type-variable bug as designed behavior, which is wrong.
- The current implementation also has no generic-call metavariable machinery, no effect variable solving, and no refinement obligation handling. Ratifying it would require deferring items 6, 7, 8, 10, and 14 of this brainstorm too — the gap is load-bearing, not cosmetic.

**Implementation gap acknowledged, not hidden.** The `### Type Inference` subsection ends with an "Implementation gap" paragraph listing the five concrete differences between the spec and `src/typechecker.rs`, and a matching Phase 4 entry in `docs/implementation_checklist.md` records the upgrade as the first TypeChecker work item. This avoids the drift failure mode where the spec says one thing and the implementation quietly does another.

**Changes applied to design.md:**
- New subsection **Type Inference** inserted in `## Type System and Object Model` between the type-system bullets and `### Error Handling`. Covers: choice of algorithm and rationale for rejecting HM and bottom-up synthesis, the two judgments, subsumption rule, mode-selection table, per-expression-form rules, generics and metavariable solving, effect-variables as a separate namespace, refinement proof obligations in check mode, structural variance table, and the implementation-gap paragraph.

**Changes applied to implementation_checklist.md:**
- New Phase 4 entry **TypeChecker: upgrade to bidirectional inference** listing six ordered upgrade steps: (1) introduce a `check` entry point and thread it through the contexts listed in the mode table; (2) per-call-site fresh metavariables for generic functions with local substitution solving; (3) closure parameter types from context in check mode; (4) subsumption through function-type variance replacing `check_assignable`; (5) effect variable solution map (blocked on Phase 3 `with E` work); (6) unsolved-metavariable diagnostics.

### 3. ~~RC fallback dataflow algorithm — exact dominance definition~~ ✅ RESOLVED

**Decision: Option (a) — a formal dataflow specification written into design.md, with pseudocode, a unified use-predicate table, explicit answers to all five sub-questions, and a pointer to the Phase 4 ownership-checker upgrade in the implementation checklist.**

**Answers to the five sub-questions from the original item:**

1. **Forward or backward dataflow?** Forward, over a structured CFG lowered from Kāra's `if`/`match`/`loop`/`while`/`break`/`continue`/`return`/`?` and `defer`/`errdefer` exit edges. The CFG is reducible because Kāra has no `goto`; Cooper–Harvey–Kennedy computes the dominator tree in near-linear time. No SSA — Kāra's move semantics already forbid re-definition, so the phi-insertion motivation for SSA does not arise.

2. **What counts as a "use"?** A unified predicate with a 15-row classification table covering every AST form that names a value — field reads, index reads, method calls by receiver mode, function calls by inferred parameter mode, let re-binding, container inserts, closure captures by mode, `return`, enum/struct constructors, and `match` scrutinees — each classified as `Read` or `Consume`. Closure captures are located at the closure *creation site*, not where the body runs, because local dataflow must see the consume at the point the programmer wrote it.

3. **How is live range computed?** Not SSA, not linear scan — a standard forward liveness interval on the CFG, formally defined as the basic blocks between a value's def and its last use on each reaching path. The move-semantics invariant (no re-definition) collapses the general forward-liveness equation to this interval form. Implementation computes it as a side effect of the use-site collector, which walks the CFG in reverse post-order.

4. **Nested parallel region order?** Pre-order on the region nesting tree — outer regions promoted before inner ones. This is an implementation-chosen traversal, not a semantic requirement: `Arc` promotion is monotonic and live ranges are contiguous, so any order reaches the same fixed point. Pre-order is fixed so that `karac explain` references regions in source order.

5. **Can RC decisions differ between branches?** No. A value is `Rc`, `Arc`, or owned for its *entire* live range, as a single bit per function. Per-branch RC would require runtime dispatch on the ownership tier, making the tier observable — violating *Equality and Mutation Semantics for Implicitly RC'd Values* — and breaking the `warn_above_0` audit model where each RC shows up exactly once in code review diffs.

**The formal RC condition.** `v` is marked RC iff there exist `C, U ∈ U(v)` with `C` a consume, `U ≠ C`, and neither `C` nor `U` dominates the other in the function's dominator tree. Two non-consuming reads on divergent branches never trigger RC — the consume requirement is essential. The three named triggers in the existing design section (re-use after consume, closure capture + outer use, container store + outer use) are user-facing shape-classifications of instances of this formal rule; the `classify(c, u)` routine maps pairs to triggers for diagnostic messages but is not itself a separate dataflow round.

**Rationale for (a) over (b) and (c):**
- **(b) Reference LLVM's liveness:** rejected because LLVM liveness operates on a lowered IR with different primitives (no Kāra closures, no structured `par` regions, no `defer` exit edges). The "document deltas" version would end up longer than a self-contained spec, and every delta would be a place where Kāra's semantics could silently diverge from the reference.
- **(c) Leave to implementation:** rejected because RC stability is part of the language's performance contract, as the original recommendation noted. A programmer who restructures code to eliminate an RC note needs to trust that a compiler upgrade will not reintroduce one without a source change. A dataflow form committed in the spec is the only way to make that promise.

**Implementation gap acknowledged.** `src/ownership.rs` today is a linear HashMap-based state machine (`Live`/`Moved` per binding, cloned at branch points) — no CFG, no dominator tree, no use-site collector, no RC promotion machinery at all. The gap is tracked as two Phase 4 entries in `docs/implementation_checklist.md`: one for the Phase 1 dataflow upgrade (CFG → dominators → use-site collection → RC condition), and one for the Phase 2 Rc→Arc promotion (blocked on the dataflow upgrade and on auto-concurrency work). The current use-after-move and double-move checks remain sound on the richer dataflow and will run on the same CFG walk — nothing is lost in the migration.

**Changes applied to design.md:**
- New subsection **RC Dataflow Specification** inserted in Feature 4 › Part 4 between **RC Insertion Algorithm** and **Rc vs Arc — Two-Phase Algorithm**. Covers: analysis form (forward over structured reducible CFG, Cooper–Harvey–Kennedy dominators, per-function), unified `use(v)` predicate as a 15-row classification table, closure-capture-site location rule, the formal RC condition, the "two reads alone never trigger RC" clarification, pseudocode for Phase 1, the per-value-not-per-branch rule, and the cross-version stability guarantee.

### 4. ~~Drop order × parallel regions × RC × defer interaction~~ ✅ RESOLVED

**Decision: unified drop+defer LIFO stack, with an `errdefer` phase running first on error paths. Destructors (including `Rc`/`Arc` decrements) are not a separate group — they interleave with `defer` blocks in a single program-order-LIFO stack. A new `#### Drop ordering within a branch` subsection in "Parallel Failure and Cleanup" is the authoritative statement; the error-handling bullet list forwards to it.**

**Answers to the three sub-questions from the original item:**

1. **Arc decrement vs. sibling's `defer` blocks.** Interleaved in a single LIFO stack, ordered by program-order of introduction — not "before all defers" and not "after all defers." A `let x = Arc.new(...)` followed by `defer foo()` in a cancelled sibling drains on cleanup as `foo()` first, then `drop(x)` (the atomic `Arc` decrement); swap the source lines and the order swaps. This is the same rule as sequential code. Two sibling branches may atomically decrement the same `Arc` concurrently without synchronization; the final decrement (whichever runs last) frees the payload.
2. **`defer` referencing a value owned by a sibling.** Forbidden by scope rules — the resolver rejects it at name-resolution time. Branches are distinct lexical scopes within the enclosing `par` region, and a sibling branch's locals are not in scope. The only cross-boundary values a branch can see are outer-scope bindings either borrowed by reference (outer scope is live past the join barrier per step 5 of the join-then-cleanup sequence) or moved into exactly one branch (so only that branch owns it). Cross-branch `defer` liveness is a non-question by construction.
3. **`errdefer(e) = Cancelled` in a cancelled sibling's destructor phase.** Yes. Every `errdefer(e)` block in a cancelled sibling binds `e` to the implicit `Cancelled` value of the function's `Err` type, regardless of the originating branch's actual error. Parameterless `errdefer` fires normally. This was already stated in the existing `errdefer(e)` binding table; the new subsection repeats it in the drop-ordering context because it is the most common subtle question.

**Worked sequence (cancelled sibling `S`):**
1. `S` observes cancellation at its next effect-boundary check and takes the error path with `e = Cancelled`.
2. `S` drains its `errdefer` stack LIFO — all bindings still live — with `errdefer(e)` blocks seeing `e = Cancelled`.
3. `S` drains its unified drop+defer stack LIFO in program-order — `Arc` decrements interleaved with `defer` blocks at the position each was introduced.
4. `S` reports completion to the join barrier.
5. Only after every sibling has finished steps 1–4 does the enclosing scope run its own drop+defer phase against the scope's return value.

**Rationale for a unified LIFO over the alternative ("`defer` group then drops group" or vice versa):**
- **Contradicts the sequential-code mental model.** Rust (drops only) and Zig (drops + defers + errdefers all LIFO) both use a single interleaved LIFO stack. Grouping destructors separately would force programmers to keep two different ordering rules in their head depending on whether a resource is released by a destructor or a `defer`.
- **Breaks the program-order invariant.** The `errdefer`/`defer` design already says LIFO-by-declaration-order. Putting destructors in a fixed position relative to that stack would mean "closing `x` via `defer x.close()`" and "closing `x` via destructor" produce different relative ordering with a sibling's cleanup — a silent hazard.
- **Live-range-end drops would still not fit any grouped model.** Per the RC Dataflow Specification (item 3), destructors fire at live-range end via the NLL rule, which can be mid-scope. A "destructor group at start/end of cleanup" is incompatible with mid-scope drops; a unified LIFO stack handles both uniformly (mid-scope drops fire inline, end-of-scope drops drain in the stack).

The unified-stack answer matches what every existing language with both RAII and `defer` does, and is consistent with the NLL drop-placement rule already committed under RC Dataflow Specification.

**Implementation gap acknowledged.** The Phase 5 `defer`/`errdefer` AST+parser work is done, but the interpreter does not yet interleave destructor calls with `defer` blocks in a unified LIFO stack, nor does it honor live-range-end (NLL) destructor placement — the latter is blocked on the OwnershipChecker RC dataflow upgrade (item 3), which is where live-range end is computed. Tracked as a new Phase 4 entry in `docs/implementation_checklist.md`.

### 5. ~~Atomics × effect system × `shared struct` interaction~~ ✅ RESOLVED

**Decision: atomic operations are effect-free synchronization primitives. `load`/`store`/`swap`/`compare_exchange`/`fetch_*` contribute zero effects to a function's inferred or declared set. The function's effect set comes from ordinary reads/writes on the resources the atomic is *synchronizing access to*, not from the atomic operations themselves. Memory orderings are codegen attributes, not effect atoms. `Atomic[T]` fields in a `shared struct` are covered uniformly by the struct's Phase 2 `Rc → Arc` promotion. `lock` blocks are effect-transparent: body effects flow to the enclosing function unchanged.**

**Answers to the four sub-questions from the original item:**

1. **What effect does `load(Ordering.Relaxed)` carry?** None. Atomic method calls contribute zero to the effect transfer function. The function's effect set is determined by ordinary reads/writes on *other* resources — the ones the atomic is synchronizing access to. In the canonical signal-flag pattern, `sensor_buf.write(reading)` produces `writes(SensorData)` and `flag.store(true, Release)` produces nothing; the function's declared `writes(SensorData)` is satisfied by the literal buffer write, not by the flag store.
2. **Is an `Atomic[T]` field in a `shared struct` subject to the same RC→Arc promotion rule?** Yes. The Phase 2 `Rc → Arc` promotion in Feature 4 Part 4 operates on the whole struct. When the enclosing struct is promoted, every field — atomic or otherwise — moves with it. Atomic ops work identically under both tiers because the atomicity guarantee is a property of hardware instructions, not the RC wrapper. A `shared struct` that uses *only* `Atomic[T]` fields for cross-task mutation does not need `Mutex[T]` — the `Sync`-forbidden rule that mandates `Mutex[T]` for shared mutable access applies to non-atomic `mut` fields only. Atomic fields are themselves the synchronization.
3. **How does `lock` block syntax interact with effect inference?** Effect-transparent. `lock node { ... }` does not scope, mask, or rename effects; the body's effects contribute to the enclosing function's inferred set exactly as if they were written outside the block. `lock` is a runtime-synchronization code-generation construct; the effect system and the mutex layer are complementary (coarse compile-time serialization per-resource from effects, plus precise runtime serialization for intra-resource granularity from `Mutex[T]`).
4. **Do memory orderings appear in `karac explain` output?** No. `Relaxed`/`Acquire`/`Release`/`AcqRel`/`SeqCst` are codegen attributes that constrain hardware memory barriers. They do not appear in effect-inference output, do not participate in effect subsumption, and are not rejected by an effect-boundary check. A future `karac explain --memory-model` view may surface them as a separate reasoning layer; that is a deferred tool feature, not part of the effect verb vocabulary.

**"Effect-free is not conflict-free."** This is the subtle clarification that makes the whole story consistent. Two private functions that both call `counter.fetch_add(1, Relaxed)` on the same `Counter` both infer empty effect sets, and auto-concurrency is therefore free to run them in parallel. This is *correct* — `fetch_add` is linearizable by construction, so any interleaving produces a well-defined result (one of the valid orderings). The effect system does not need to forbid parallelism here because the hardware already guarantees safety. Programmers who need a specific ordering between two atomic updates use memory orderings or a higher-level synchronization primitive, neither of which is expressed as an effect atom.

**Rationale for effect-free over the original recommendation's `reads(R)`/`writes(R)` tagging:**
- **No clean source of `R`.** The original recommendation said "tag with `reads(R)`/`writes(R)` where `R` is the resource the atomic is a field of." But atomics are most often *not* fields of a resource-tagged container — they are standalone `let counter: Atomic[i64]` bindings or fields of ordinary structs. Picking `R` would require either a new type-level annotation (`Atomic[i64] @ Counter`) or implicit resource synthesis from binding-site heuristics. Both were rejected as adding surface area for a feature (atomics as resources) whose semantics are already cleaner when factored out.
- **Tracking atomics as `writes(R)` would serialize `fetch_add`.** Auto-concurrency would see two `fetch_add` calls as conflicting on `R` and serialize them, which defeats the *entire purpose* of having a lock-free counter. The effect system is the wrong layer to reason about atomic linearizability.
- **Tracking atomics as `reads(R)` would misrepresent stores.** `store` and `fetch_add` mutate state but would register as `reads(R)`, inverting the verb vocabulary and misleading both the programmer and the auto-concurrency analyzer.
- **Effect-free separates layers cleanly.** Effects answer "which resources does this function touch"; the memory-ordering layer answers "in what order do these operations observe each other." The two layers never overlap, and the language commits to not collapsing them into a single notion. This is the same architectural split as in Rust (atomics live in `core.sync.atomic`, outside the borrow checker's primary abstractions) and C11 (memory model is orthogonal to the type system).

**The atomic-field / shared-struct / `Sync` question resolved.** Before this resolution, the `shared struct` section said "`shared struct` is `Send` but not `Sync`; concurrent `mut` field access from multiple tasks requires explicit `Mutex[T]` wrapping." This was ambiguous for atomic fields — would they require `Mutex[Atomic[i64]]`? No: atomic fields are themselves the synchronization, so the `Mutex[T]` requirement applies only to non-atomic `mut` fields. The new Effect Model subsection states this explicitly.

**Implementation gap acknowledged.** Atomics are a Phase 8 feature and do not yet exist in the compiler. When atomics land, the effect checker must treat all atomic method calls as zero-effect (the transfer function ignores them), and Phase 6 auto-concurrency must treat atomic-only function bodies as conflict-free. The existing Phase 2 `Rc → Arc` promotion (item 3) handles atomic fields inside `shared struct` uniformly with non-atomic fields — no special case. Tracked as an extension to the existing Phase 8 `Atomic[T] and memory ordering` entry in `docs/implementation_checklist.md`.

**Changes applied to design.md:**
- Atomic Operations › **Effect Model** subsection rewritten comprehensively. Replaced the vague "tagged with `reads(R)` / `writes(R)` accordingly" phrasing with the explicit "effect-free synchronization primitives" rule. Updated the `signal_ready` / `wait_and_read` example so the function literally writes to `sensor_buf` (producing `writes(SensorData)`) instead of declaring the effect without any code that justifies it. Added six new paragraphs: **Why effect-free** (linearizability rationale), **Effect-free is not conflict-free** (the counter case), **Memory orderings are not effects** (codegen attribute rule), **RC integration** (atomic fields inside `shared struct` covered by Phase 2 promotion; `Sync`-forbidden rule clarified to apply to non-atomic fields only), **`lock` blocks and effect inference** (effect-transparent code-generation construct), and a **Summary** paragraph stating the two-layer architecture (effects = what is touched, memory model = when it is safe to touch).

### 6. ~~Pattern exhaustiveness algorithm~~ ✅ RESOLVED

**Decision: commit to Maranget's usefulness/exhaustiveness algorithm ("Warnings for pattern matching", JFP 2007) — the same decidable algorithm used by Rust, OCaml, and Scala. Specified explicitly in a new `### Pattern Exhaustiveness` subsection under Feature 3 so implementers do not have to chase a paper.**

**Answers to the four sub-questions from the original item:**

1. **How do refinement types narrow the scrutinee?** They don't — not for the purpose of exhaustiveness. The algorithm operates over the **base type's constructor space**. A match on `type Positive = i64 where self > 0` still requires a wildcard arm because exhaustiveness is checked against `i64`'s full 2^64-value space. The refinement constraint `self > 0` is a runtime narrowing enforced at construction and cross-refinement casts; reasoning about which base values are excluded by an arbitrary predicate is SMT territory, deferred to Level 3-4 gradual verification. Programmers who need an exhaustive match over a narrow value set should use an enum, not a refinement.
2. **Do distinct types participate in literal overlap checks?** Yes — as opaque nominal wrappers with the base type's constructor space. Two `UserId(5)` patterns are the same literal; `UserId(0)` and `UserId(1)` are distinct literals; a wildcard is still required to cover the full `UserId` space. Distinct types prevent cross-type interchange at the type-checker level but are transparent to the pattern algorithm. Matches Rust's newtype handling.
3. **How do or-patterns desugar formally for exhaustiveness (without the "shared guards exponentiate" problem)?** An arm `P1 | P2 | … | Pn => body` is, for the algorithm, `n` matrix rows pointing at the same body index: `[P1; i], [P2; i], …, [Pn; i]`. The specialization step of `U(P, q)` recurses *through* or-patterns (case 4 of the four recursion cases: `U(P, q1 | q2) = U(P, q1) ∨ U(P_after_q1, q2)`) rather than materializing the cross-product, so a nested `Foo(A | B | C)` pattern does not blow up to `3^d` rows. For guards specifically: a guarded or-pattern arm `P1 | P2 if G => body` evaluates `G` **once** per successful match, not once per alternation — the or-pattern decides which side matched, then the guard runs once against the resulting (single) binding environment. The "shared guards exponentiate" concern disappears because the algorithm and the runtime both treat the or-pattern arm as one evaluation of the guard.
4. **Does a provably-true guard satisfy exhaustiveness?** No. Not even `if true`, `if 1 == 1`, or any other trivially-true expression counts toward exhaustiveness. The compiler does not const-evaluate guard expressions for the exhaustiveness check. Rationale: (a) the compiler has no general const-evaluator, and building one purely for exhaustiveness is a large feature for a small benefit; (b) "trivially provable" creates an inconsistency cliff between guards the prover handles and guards that are one character away from handled, surprising the programmer; (c) the workaround is trivial — write an unguarded `_` arm. Matches Rust.

**Additional cases specified (beyond the four original sub-questions, needed for a complete algorithm):**

- **Integer, float, string, and `char` literal patterns.** The base type's constructor space is not enumerable; a match is exhaustive only with a wildcard arm (`_`, an unadorned binding, or a future range pattern that covers the full range).
- **`Array[T, N]` (fixed-size arrays).** Behaves like a tuple — `N` field positions, exhaustiveness propagates to each element. `[a, b, c]` is exhaustive on `Array[T, 3]` iff `a`, `b`, `c` are each exhaustive on `T`. Rest patterns `[first, ..rest]` cover the range the `..` spans.
- **`Vec[T]`, `HashMap`, `String`.** Variable-length; wildcard required. Slice patterns with a rest cover non-empty cases but still need `[]` or `_` for the empty case.
- **`Never`.** `match x { }` is exhaustive iff `x: Never`. The only type for which an empty match is well-formed.
- **`let PAT = expr`.** Pattern must be irrefutable (single-row matrix `[PAT]` covers every value of the type). Refutable patterns are rejected with a diagnostic suggesting `if let` or `let ... else`.
- **`if let` / `while let`.** Single-pattern refutable tests — not exhaustiveness-checked. Irrefutable pattern is a warning (unreachable else / infinite loop), not an error.
- **`let ... else`.** Refutable `PAT` allowed; the `else` block must diverge (typechecked via the `Never` type on the `else` block's result). No exhaustiveness needed.
- **Unreachable arms.** Produce a warning (not an error) via the reachability judgment `U(P[0..i], P[i])`. Unreachable arms are retained in generated code so rebuilding with a changed earlier arm does not resurrect unrelated code.
- **Empty match on non-`Never`.** Rejected with full-type-uncovered diagnostic.

**Rationale for Maranget over the alternatives:**
- **Why not "handwave: Rust-style"?** Rust-style is not a spec — it is a pointer to a moving target. The current resolution writes the four recursion cases directly into design.md so an implementer can build the checker without reading foreign source code, and so future divergences from Rust are caught at the design-review layer rather than discovered at "why does our match compile but Rust's reject it" debugging time.
- **Why not a simpler per-type checker?** The current code implements a flat `HashSet<variant_name>` approach that counts `Some(0) | Some(1)` as covering *all* `Some` variants — a false-negative that would let a real non-exhaustive match compile. Fixing this for nested patterns, integer literals, or-patterns, and refinement/distinct types without a general framework would accumulate ad-hoc cases and still miss corners. The usefulness framework subsumes every case in one algorithm.
- **Why not defer to SMT / Level 3-4 verification?** The usefulness algorithm is decidable on finite constructor spaces; no SMT solver is required. Reserving SMT for refinement-aware exhaustiveness (the "check whether `u16 where self <= 10` is exhausted by the 11 relevant literal arms" case) is the right split — the base-type algorithm runs at every match, SMT would run only for refinement-narrowed matches and only after Level 3-4 verification lands (deferred).

**Implementation gap acknowledged.** `src/typechecker.rs` around line 2255 has a very partial exhaustiveness check: flat `HashSet<String>` of enum variant names, no recursion into nested patterns, no handling of integer/float/string literals, no or-pattern specialization (duplicated alternations are implicitly fine), no witness production, and no reachability pass. A match like `match opt { Some(0) => a, None => b }` is currently accepted as exhaustive because `Some` is counted as a covered variant regardless of its payload — a real soundness hole. Tracked as a new Phase 4 entry in `docs/implementation_checklist.md` with the seven ordered upgrade steps.

**Changes applied to design.md:**
- New subsection **Pattern Exhaustiveness** inserted in Feature 3 between the or-patterns rules block and the Layout integration note. Contents: (1) commitment to Maranget's algorithm with the usefulness judgment `U(P, q)` defined formally; (2) the four recursion cases (empty-columns base, constructor specialization, wildcard with default matrix, or-pattern alternation); (3) derivation of exhaustiveness and reachability from usefulness; (4) type-specific rules for enums, structs/tuples, integer/float/string/char, `Array[T, N]`, `Vec[T]`/`HashMap`/`String`, `Never`, refinement types, distinct types, and `shared struct`/`shared enum`; (5) or-patterns and guards subsection answering the "shared guards" concern explicitly; (6) `let`/`if let`/`while let`/`let ... else` subsection; (7) empty match and unreachable arms subsection; (8) a worked example showing or-pattern specialization on an `Event` enum with step-by-step U-recursion trace.

**Changes applied to implementation_checklist.md:**
- New Phase 4 entry **TypeChecker: upgrade exhaustiveness to Maranget's algorithm** listing the seven ordered upgrade steps (matrix representation, core `U` function, nested-pattern recursion, type-specific rules, witness emission, reachability/unreachable-arm pass, and unified `let`/`if let` refutability check) plus a note about refinement/distinct type handling via base-type delegation.

### 7. ~~Trait method resolution order and coherence with effects~~ ✅ RESOLVED

**Decision: commit to a Rust-adapted four-step method resolution algorithm — receiver candidate chain, per-candidate probe with inherent-beats-trait priority, first-match-wins across candidates with ambiguity-rejection within a candidate, UFCS escape. Two Kāra-specific simplifications from language-level decisions already made elsewhere: (1) no user-defined `Deref`, so the receiver candidate chain is statically bounded by built-in rules for `shared struct` / `Rc` / `Arc` / refinement types only; (2) no separate "bring a trait into scope via `use`" layer, so a trait's methods participate in lookup iff the trait itself is name-resolvable from the call site under existing visibility rules. Effect conflicts across two traits are always legal coexistence — they only surface as ambiguity errors at the method-lookup layer when both traits happen to declare a method with the same name, and the programmer disambiguates via UFCS, which selects one trait and therefore one effect ceiling. There is no "union of effects" semantics.**

**Answers to the two sub-questions from the original item:**

1. **Which order are inherent impls, direct trait impls, blanket impls, and conditional impls consulted?** Per-candidate, in the receiver chain order `[T, ref T, mut ref T, (shared-struct / Rc / Arc inner), (refinement base)]`. Within each candidate level: inherent methods *first* (and they win outright — trait candidates at the same level are dropped when an inherent method exists), then trait methods from every visible trait. Conditional impls are filtered by attempting to solve their bounds against the concrete receiver type; impls whose bounds fail to resolve are silently dropped from the candidate set (not an error). Blanket impls are not a separate category — they are conditional impls whose bound is general enough to match many types; they go through the same "solve the bound, include if it resolves" filter. First receiver level that yields any method wins — autoref does not override inherent on the value level, auto-deref does not override methods on the outer type. Same receiver-chain-and-priority ordering Rust uses.
2. **Can one type implement two traits with conflicting effect declarations?** Yes, always. Each trait's per-method effect declaration is independent; the existing coherence rule (`Trait Coherence and Effects`) checks each impl's method effects against *that impl's* trait ceiling, not against other traits the same type also implements. A type can `impl Reader with reads(DB)` and `impl Writer with writes(DB)` simultaneously without any cross-trait effect check. The effect conflict becomes observable only when both traits also declare a method with the same name, at which point the call site is ambiguous and the programmer disambiguates via UFCS. The selected trait is the sole source of the method's effect set for that specific call; there is no "union over candidates" semantics. This is important because unioning would either force all candidates to have compatible effects (overly restrictive, disallowing exactly the legitimate two-trait case) or silently widen the caller's inferred effects based on which traits happen to be in scope (fragile under library upgrades — adding a new trait with an `access` method in a dependency could retroactively widen a caller's effect inference).

**Rationale for the two simplifications vs. Rust:**

- **No user-defined `Deref`.** Rust walks an unbounded deref chain because any type can `impl Deref<Target = U>` and reroute lookup through `U`. Kāra has no such trait, so the receiver candidate chain is a small finite list determined by the type's structural form. Programmers cannot silently extend the lookup path, and the compiler cannot get stuck chasing a cyclic user-defined deref. This is a strict improvement: same ergonomic coverage (`shared struct`, `Rc`, `Arc` all auto-deref via built-in rules), zero extensibility risk, simpler implementation.
- **No Rust-style "bring trait into scope via `use`".** Rust requires `use Trait` to make a trait's *methods* discoverable even when the trait name itself is already resolvable. Kāra's module visibility is already project-wide for default items, so adding a second layer ("trait is visible but its methods are invisible until `use`'d") would be a second rule doing roughly the same work as the first, with the added failure mode of the "I can name the trait but its methods don't show up" confusion. The simpler rule — *trait is visible iff its methods are candidates* — matches programmer intuition and is equally sound because ambiguities at the lookup step are still caught and rejected with a clear diagnostic.

**Worked example (effect-conflict resolution):**

```kara
trait Reader with reads(DB)  { fn access(ref self, key: Key) -> Value reads(DB); }
trait Writer with writes(DB) { fn access(ref self, key: Key) -> Value writes(DB); }

struct Connection { ... }
impl Reader for Connection { fn access(ref self, k: Key) -> Value reads(DB)  { ... } }
impl Writer for Connection { fn access(ref self, k: Key) -> Value writes(DB) { ... } }

fn use_conn(conn: ref Connection, key: Key) {
    let v = conn.access(key);            // ERROR: ambiguous — Reader.access or Writer.access?
    let r = Reader.access(conn, key);   // OK — Reader impl; effect at this call = reads(DB)
    let w = Writer.access(conn, key);   // OK — Writer impl; effect at this call = writes(DB)
}
```

**Generics and trait bounds.** At a generic call site `fn f[T: Reader](x: T) { x.access(key); }`, resolution is performed against the bound `T: Reader`, not against any concrete type. The single-trait bound guarantees exactly one candidate (`Reader.access`) regardless of which concrete type eventually instantiates `T` — even if that concrete type also impls `Writer`, the call inside `f` resolves through `Reader` because that is the only trait `f` named. Multi-bound generics (`fn f[T: Reader + Writer]`) re-introduce the ambiguity, and the compiler rejects with the same diagnostic as the concrete case. UFCS is again the resolution. Intentional: multi-bound generics should not silently pick one trait over another.

**Implementation gap acknowledged.** `src/typechecker.rs` around line 1876 (`infer_method_call`) currently does a naive first-match scan over `env.impls` filtered by target type name — no receiver candidate chain, no inherent-vs-trait priority, no ambiguity detection, no UFCS, no autoref, no visible-trait filtering for conditional impls, and it silently returns `Type.Error` when no method is found rather than emitting a diagnostic. This is a real soundness hole for the same reason the exhaustiveness checker is (item 6): programs can compile with one impl "winning" not because of any principled rule but because of HashMap iteration order. Tracked as a new Phase 4 entry in `docs/implementation_checklist.md` with eight ordered upgrade steps.

**Changes applied to design.md:**
- New subsection **Method Resolution** inserted between `### Orphan Rules for impl Blocks` and `### Associated Types in Traits`. Covers: (1) the two Kāra-specific simplifications (no user-defined `Deref`; no separate `use Trait` layer) with rationale; (2) the four-step algorithm (receiver candidate list, per-candidate probe with inherent priority, first-match-wins/ambiguity-within-level, UFCS escape); (3) the effect-conflict case with the full worked example above; (4) interaction with generics and trait bounds (single-bound generics resolve uniquely, multi-bound generics produce the same ambiguity as concrete sites); (5) conditional impl handling (bound-satisfaction filter, silent drop not error); (6) "Why not" subsection rejecting alternative designs (Rust-style `use`, effect unioning, specialization).

**Changes applied to implementation_checklist.md:**
- New Phase 4 entry **TypeChecker: implement full method resolution algorithm** listing the eight ordered upgrade steps (receiver chain construction, inherent/trait probe, inherent-priority filter, ambiguity detection + diagnostic, UFCS parsing/resolution, conditional-impl bound-satisfaction filter, no-method-found diagnostic with typo suggestions, generic-bound-driven resolution at polymorphic call sites).

### ~~8. Refinement type proof procedure and call-site behavior~~ ✅ RESOLVED

**Decision:** The v1 elision procedure is exactly **two rules**, and nothing else. **Rule 1 (const-evaluable narrowing):** if the initializer of a binding or the argument at a call site has a refined target type and is a const-evaluable expression, the compiler reduces it via the existing const-evaluator, evaluates the refinement predicate against the resulting value, and on success emits no runtime check and no error path. On failure it emits a **compile error** (not a runtime error — const-eval has the value in hand). **Rule 2 (type-identity narrowing):** if the static type of the initializer is exactly the target refined type (same name, same predicate, same base), no check is emitted. Every other narrowing form — runtime-value binding, runtime-value call-site argument, cross-refinement coercion — requires an explicit `Refined.try_from(x)?` or `x as Refined` at the narrowing site. There is no implicit runtime-value narrowing and no flow-sensitive narrowing anywhere in v1.

**Sub-questions answered:**

1. **What is the provability procedure — interval arithmetic, SAT/SMT, pattern-based, or structural?** None of interval arithmetic, SMT, or syntactic predicate matching. The procedure is: (a) run the const-evaluator on the expression; (b) if it produces a concrete value, evaluate the predicate against that value at compile time. That's the entire procedure. No symbolic reasoning, no implication, no range propagation. This is deliberately below even "structural" — we don't even check whether the source type's predicate is structurally identical to the target's, because distinct refined types are always opaque nominal wrappers and cross-refinement is always explicit in v1.

2. **At `f(x)` where `f` expects `Positive` and `x: i32`, is the check runtime always, or can it be elided via flow-sensitive typing?** **Compile error.** Not a runtime check, not an elision — a hard compile-time rejection with the message "cannot narrow `i32` to `Positive` implicitly; use `Positive.try_from(x)?` or `x as Positive`". The call-site rule is identical to the binding-initialization rule: if the argument is const-evaluable the value is const-checked (rule 1); if the argument's static type matches the parameter's type exactly it passes through (rule 2); otherwise it is rejected. This eliminates the "silent runtime check at the argument boundary" surface entirely — the only place a runtime check can silently appear is inside an explicit `try_from` call, which is syntactically visible.

3. **Is there any flow-sensitive narrowing (`if x > 0 { f(x) }` lets `f` accept `x`)?** **No.** Occurrence typing is not in v1 and is explicitly deferred indefinitely. Inside the `if` branch, the static type of `x` remains `i32`, and `f(x)` is still a compile error. Programmers who want the flow-sensitive effect write `let x: Positive = 5` at the binding site (which triggers rule 1) or use explicit `Positive.try_from(x)?` inside the branch. Occurrence typing interacts with pattern matching, closures, effect variables, and inference in ways that are disproportionate to the ergonomic win for numeric refinements, and we'd rather ship a smaller, consistent surface.

4. **What do generic functions see?** A polymorphic function `fn f[T](v: T)` operates on an abstract `T` inside its body — the body cannot inspect the predicate even when `T` is eventually instantiated with a refined type, so **no elision happens inside generic bodies**. The two elision rules apply only at the *caller's* narrowing boundaries against the concrete instantiated type. This keeps monomorphization decoupled from refinement-elision decisions — type substitution comes first, and the same v1 rules then govern narrowing/widening at the caller site as they would for any concrete type.

**Internal contradiction fixed as part of this resolution.** The original design.md had `let port: ValidPort = 80;` as an implicit-narrowing example (line 1185 of the pre-edit doc) while the `(2) Narrowing` subtyping rule (line 1203 of the pre-edit doc) simultaneously stated narrowing required explicit `try_from`. - The original recommendation in the problem statement was "structural elision only (if the type of `x` is already `Positive`, elide; otherwise runtime check)", which is broader than what landed: it would silently emit a runtime check at call sites for non-matching types. Landing a *rejection* instead of a *runtime check* for the "neither rule applies" case is a strict improvement — it keeps the set of places where a refinement check can silently appear down to zero at the surface language, and it forces the programmer to name the narrowing explicitly at every point where one occurs. This makes the effect-system story cleaner too: an `as` coercion propagates `panics` at a syntactically visible site rather than being buried in an implicit call-site widening.
- Rule 1 (const-evaluable narrowing) is the only implicit form because it is guaranteed to be decidable (const-eval either reduces or it doesn't), and a const-evaluable failure produces a **compile error** rather than a runtime panic — it's as safe as a literal mismatch anywhere else. ### ~~9. Contract evaluation semantics~~ ✅ RESOLVED

**Decision:** Contract evaluation semantics are pinned down by **four rules**, added as a new "Contract evaluation semantics" subsection inside the Contracts section of design.md. **(1) Purity is "effect set ⊆ `{panics}`"** — the effect checker imposes this constraint on every contract expression (`requires`/`ensures`/`invariant`); `panics` is admitted because indexing/division/unwrap are idiomatic in predicates, no other effect is admitted. **(2) Panic-in-contract is a distinct fault category** (`contract predicate panicked at <loc>`) separate from predicate-returned-false (`contract violated`), reported with distinct messages so programmers can tell "bug in checked code" from "bug in contract". **(3) Invariants check at the exit of every `pub` method and every constructor, structurally** — applies to `struct`, `shared struct`, `shared enum`, every receiver mode, regardless of whether the method syntactically mutates. Private methods do not check, allowing multi-step updates with temporarily-broken invariants. **(4) Postconditions reference pre-state via `old(expr)`** — a special form valid only inside `ensures`, evaluated at function entry and captured via `Clone`; direct reference to a consumed receiver in `ensures` is a compile error suggesting `old(self)` or `old(self.field)`. All four rules are stripped in release.

**Sub-questions answered:**

1. **Can contract expressions call effectful functions? If not, how is "effect-free" checked?** Contract expressions must have an effect set that is a subset of `{panics}`. The effect checker infers the set of the contract expression using the same machinery as ordinary expressions, then imposes the `⊆ {panics}` constraint at the contract boundary. Any of the seven non-panic effects — `reads`, `writes`, `sends`, `receives`, `allocates`, `blocks`, `suspends` — appearing in a contract expression is a compile error with the diagnostic pointing at the offending call and naming the forbidden effect. Method calls inside contracts are admitted iff the called method's effect set is `⊆ {panics}`, which is the same rule applied inductively.

   `panics` is the one allowed effect (not zero) because: (a) the existing `binary_search` example in design.md already uses `haystack[i] == *needle` in an `ensures` clause, which has `panics`; (b) indexing, integer division, `unwrap()`, and explicit `panic!()` are idiomatic in contract predicates; (c) forbidding `panics` would force contracts to be written through `.get()` / `checked_div()` / `match`-on-`Option` machinery, making contracts harder to write than the code they check. Non-panic effects remain forbidden because they introduce *observable* side effects from runtime predicate evaluation — mutation, I/O, message sending — which is the actual hazard "pure" is ruling out.

2. **What happens if a contract predicate panics during evaluation?** A panic during contract evaluation is reported as the fault category `contract predicate panicked at <loc>: <panic message>`, **distinct** from `contract violated: <predicate>` for the predicate-returned-false case. The two faults are reported separately and not conflated.

   The distinction matters because the two cases have different root causes: "contract violated" means the checked code misbehaved (bug in the *checked code*); "contract predicate panicked" means the predicate itself failed to evaluate (bug in the *contract*, e.g., `ensures |result| haystack[i] == *needle` panicking on out-of-range `i` even though the checked code is correct). Collapsing both into one fault would misdirect programmers debugging contract failures. Both aborts the program in debug; both are stripped in release. `karac test` catches both and reports them as test failures with their distinct messages.

3. **When is `invariant` checked on a `shared struct` — every method, only `pub mut ref`, only syntactically mutating methods?** **At the exit of every `pub` method**, including constructors, regardless of receiver mode, regardless of whether the method syntactically mutates, regardless of whether the struct is `struct`, `shared struct`, or `shared enum`. The rule is **structural**, not dataflow-driven — no "does this method actually mutate" analysis. Private methods do **not** check.

   The structural rule is chosen because: (a) a private helper leaking through a shared-interior-mutable field is a real hazard that a dataflow-driven "does this mutate" check would miss — the helper would be marked non-mutating but can still break the invariant through atomics or an unexpected aliased write; (b) multi-step updates that temporarily break the invariant can factor through private helpers, which freely leave the invariant violated between calls as long as the outermost pub method restores it before return; (c) it's trivially implementable (insert the invariant check at every pub-method return point; done). Constructors check too so every externally observable `Self` value satisfies the invariant from creation onward.

4. **Can a postcondition reference `self` if `self` is consumed by the return?** Not directly — a direct reference to a consumed parameter (`own self` or any `own`-parameter whose value was moved) in an `ensures` clause is a **compile error**. The error suggests `old(self)` or `old(self.field)`.

   `old(expr)` is a special form valid only inside `ensures` (not `requires`, not `invariant`). It captures a snapshot of `expr` at function entry via `Clone` and makes the snapshot available in the postcondition. In debug builds, the compiler desugars `old(expr)` to a function-entry local that clones `expr` before the body runs, and rewrites the ensures clause to reference the local. Non-cloneable types inside `old()` are a compile error with message `"\`old(...)\` requires the captured expression to be Clone"`. In practice this pushes programmers toward `old(self.balance)` (a `Copy` field) rather than `old(self)` (the whole aggregate, which may not be Clone) — a natural pressure toward narrow captures matching the postcondition's actual data dependency.

   Scope restrictions: `old(...)` is rejected in `requires` (preconditions run at entry, so pre-state is just "state"), in `invariant` (invariants are checked after every pub method exit and do not reference a particular method's pre-state), and with `old(result)` (`result` does not exist at function entry). In release builds contracts are stripped, so the `old()` clone has zero runtime cost.

**Rationale for the four choices vs. alternatives:**

- **Why `⊆ {panics}` and not empty?** Empty would outlaw indexing, division, unwrap, and panic!() — a strict ban on half the idiomatic contract surface. It would also invalidate the existing `binary_search` example that uses `haystack[i]` in ensures. Allowing `panics` (and only `panics`) keeps the "no observable side effects" guarantee while letting contract authors write predicates naturally. The effect system's existing `panics` atom already captures exactly the operations we want to admit.
- **Why a distinct panic-in-contract fault rather than treating panic as contract violation?** Treating "predicate panicked" as "contract violated" would misattribute the bug to the checked code when the actual problem is usually in the predicate. The two messages point the programmer at different files — the distinction pays for the small implementation cost (catching panics at the contract-expression boundary) by reducing debug time for contract bugs.
- **Why structural invariant-check-every-pub-method instead of "only methods that mutate"?** A dataflow-driven "does this method mutate" check would miss interior mutability through atomics on shared structs, miss accidental aliased writes, and add a nontrivial per-method analysis step. Structural insertion is simple (every pub method, every constructor, every return point), catches every externally observable state change by construction, and requires zero new analysis. The cost is a runtime check in debug that the dataflow check *might* have elided — but contracts are a debug-only tool, so the cost is negligible.
- **Why `old(expr)` with Clone requirement instead of an implicit "ghost self" available everywhere?** Implicit ghost pre-state would have to capture the entire function-entry environment for every function with an ensures clause, including types that may not be Clone — which either requires silently failing on non-Clone types or adds an implicit `self: Clone` bound to every function with an ensures clause that references self. Explicit `old(expr)` narrows the capture to exactly what the postcondition reads, fails loudly on non-Clone (with a suggestion to capture a field instead), and requires zero implicit bounds. It also makes the debug-build cost visible at the source level: you can see where the clones happen.

**Implementation gap acknowledged.** Contracts are not yet implemented — they are a Phase 5 item. The existing Phase 5 Contracts entry is extended in `docs/implementation_checklist.md` with eight ordered steps covering the four rules above: parse `old(expr)` as an ensures-only special form, effect-set check for contract expressions (`⊆ {panics}`), `old(expr)` desugaring with Clone requirement, consumed-parameter reference check, invariant insertion at pub-method exit (including constructors, including shared struct, every receiver mode), distinct "contract predicate panicked" fault category with panic-catch wrapper in debug, test-runner integration for both fault categories, and release-mode stripping.

**Changes applied to design.md:**
- New **Contract evaluation semantics** subsection inserted between the "Contract expressions must be pure" paragraph and the "Verification strategy" paragraph in the Contracts section. Covers the four rules explicitly: effect-set `⊆ {panics}` purity, distinct "contract predicate panicked" fault category, structural invariant-check-every-pub-method rule, `old(expr)` form for pre-state with Clone requirement and worked example.
- Existing "Contract expressions must be pure" paragraph rewritten to point at the new subsection instead of duplicating the rule.

**Changes applied to implementation_checklist.md:**
- Existing Phase 5 **Contracts** entry extended with eight ordered implementation steps listed above.

### ~~10. Effect polymorphism × generics × higher-order boundaries~~ ✅ RESOLVED

**Decision:** The compound case (generic function with both type parameters and effect variables accepting closures whose types depend on those parameters) is resolved at every call site via a **types-first, effects-second** four-step algorithm: (1) constraint generation collects both kinds of constraints without unifying; (2) type substitution runs to completion, resolving all type parameters, projected types, and trait bounds; (3) effect unification resolves all effect variables against the now-concrete closure argument types; (4) call-site effect contribution is the union of fixed effects, resolved effect variables, and trait-method effects, added to the caller's inferred set. Ordering is not a preference — it is **forced** by the dependency structure: effect constraints may depend on type constraints (closure parameter types must be concrete before bodies can be effect-checked) but type constraints never depend on effect constraints. Nested calls resolve bottom-up: each call site runs the four-step algorithm independently, innermost first.

**Sub-questions answered:**

1. **At monomorphization, does the type parameter resolve first or the effect variable?** Types first. Every call site runs the four-step algorithm in a fixed order: constraint generation, full type substitution (including bound checking and projected-type resolution), full effect unification, call-site effect contribution. Step 2 finishes entirely before step 3 begins. The uniqueness of this order is forced by the one-way dependency: a closure body cannot be effect-checked until its parameter types are concrete, but type inference needs no effect information. "All types first, then all effects" is the unique consistent topological order; "effects first" would require deferring closure body typechecking; "interleaved per-parameter" would create cascading constraint dependencies and muddle error reporting.

2. **How does the trait bound on T interact with effect analysis of cb?** Indirectly, through two independent channels, neither of which creates a circular dependency. **Channel 1 (method effects):** the bound `T: Iterator` makes per-method effect declarations from the trait available for any `it.method()` calls *inside* `f`'s body; these contribute to `f`'s total inferred effect set via the existing "static dispatch narrows to per-method declarations" rule. **Channel 2 (projection):** the bound makes `T.Item` resolvable after step 2, so the closure slot `Fn(T.Item) -> U with E` has a concrete argument type ready for effect unification at step 3. The closure's *own* effect analysis is entirely separate from the trait bound — `cb`'s effects come from unifying `E` against the closure's inferred body effects, not from anything the bound specifies. The two sources compose freely and commutatively at step 4 as a union.

3. **Multi-variable compound case.** When a function declares multiple type parameters (`[T, U, V]`) and multiple effect variables (`[with E1, E2]`), all type parameters resolve together in step 2 (standard multi-variable type inference) and all effect variables resolve together in step 3. The ordering is strictly "all types before any effects," not per-parameter interleaved. This gives the error-ordering guarantee: a programmer with a type mismatch sees the type error first, never an effect diagnostic that is secretly caused by an unresolved type variable upstream.

4. **Nested polymorphism — closure calling another generic effect-polymorphic function.** Resolved via **inside-out / bottom-up expression-typing order**. For `outer(x, |n| inner(n, |m| write_db(m)))`, the innermost `inner(n, |m| write_db(m))` call runs the four-step algorithm first, producing a concrete effect set for its call-site context (`E2 = {writes(DB)}`). That concrete set becomes part of the enclosing closure's inferred body effects, which then feed the `outer(...)` call's four-step algorithm (`E1 = {writes(DB)}`). No global constraint system, no deferred resolution across call sites, no fixed-point iteration between call sites — each call site is resolved independently in one pass.

**Rationale for types-first vs. alternatives:**

- **Why not "effects-first, types-second"?** Effect constraints depend on types (closures cannot be effect-checked until parameter types are concrete), so effects-first would require deferring closure body typechecking until after effect unification — a two-pass dance where both passes repeatedly revisit the same closure bodies, with no upside.
- **Why not "interleaved per-parameter"?** Interleaving would mean "resolve `T1`, then `E1`, then `T2`, then `E2`" with possible back-propagation when a later type constraint affects an earlier effect. This creates cascading constraint dependencies with no clean error-reporting story: a type error on `T2` could appear as an effect error on `E1`, which is the opposite of the programmer-intuitive order.
- **Why per-call-site and not a global constraint system?** A global system would be more powerful in principle (it could defer resolution across call sites) but Kāra's effect inference is already designed around per-SCC fixed-point processing, and per-call-site resolution composes cleanly with that: each call site in an SCC runs the four-step algorithm independently, and the SCC-level fixed-point deals only with the cross-function propagation of *already-resolved* effect sets. Mixing a global constraint system with SCC fixed-point would either double the work or require a new hybrid algorithm with no clear correctness story.
- **Why is nested polymorphism bottom-up and not "resolve outer first"?** Bottom-up matches ordinary expression type-checking (inner expressions are typed before outer ones) and ensures that by the time `outer`'s call site runs, the outer closure's body already has a concrete effect set from the inner resolution. Top-down would require *outer* to somehow pre-commit to an effect set before the inner closure's types are even known, which is impossible without speculative execution.

**Worked example — pipeline.** A concrete walk-through of the four-step algorithm:

```kara
fn pipeline[T: Iterator, U, with E](
    it: T,
    transform: Fn(T.Item) -> U with E,
) -> Vec[U] with E
where T.Item: Clone
{
    let mut out = Vec.new();
    for item in it { out.push(transform(item)); }
    out
}

fn log_and_transform(x: i32) -> String reads(Log) { ... }

let numbers: Vec[i32] = vec![1, 2, 3];
let result = pipeline(numbers.iter(), |&x| log_and_transform(x));
```

Step 1 (constraint generation): `T = Iter[i32]`, closure argument shape `Fn(&i32) -> String with {reads(Log)}`, target shape `Fn(T.Item) -> U with E`. Step 2 (type substitution): unify `T = Iter[i32]`, so `T.Item = &i32` and `U = String`. Check `Iter[i32]: Iterator` ✓ and `&i32: Clone` ✓. Signature now concrete except for `E`. Step 3 (effect unification): closure's inferred body effects are `{reads(Log)}` (from `log_and_transform`), so `E := {reads(Log)}`. Step 4 (call-site effect contribution): `pipeline`'s body calls `it.next()` (pure for `Iter[i32]`, method-effects channel) and `transform(item)` (`{reads(Log)}` via the resolved effect variable). Union: `{reads(Log)}`. This is added to the caller's inferred effect set, which must declare `reads(Log)` or `with _` on its public signature (viral rule).

**Error-reporting guarantee.** Because step 2 runs to completion before step 3 is attempted, a programmer debugging `pipeline(wrong_iterator_type, closure)` always sees a type-inference diagnostic about `wrong_iterator_type`, never an effect diagnostic about the closure cascading from the type failure. When step 3 fails (e.g., a closure whose effect set exceeds a narrower slot), the diagnostic names the slot, the closure, and the offending effect with the signature fully monomorphized — no type variables in the error text.

**Implementation gap acknowledged.** The current effect checker handles the simple case (`with E` + single closure arg) but does not implement the compound case (type parameters with projected types plus effect variables plus closure arguments depending on the projections). Tracked as a new Phase 3 entry in `docs/implementation_checklist.md` — `TypeChecker + EffectChecker: compound effect polymorphism at call sites` — with eight ordered upgrade steps covering the four-step algorithm, nested-call bottom-up ordering, error-ordering guarantee, and the interaction with trait-method effects.

**Changes applied to design.md:**
- New subsection **Monomorphization order for compound polymorphism** in Effect Polymorphism, placed between "Private functions: polymorphism is inferred" and "Effect bounds on type parameters (planned, Phase 6)". Covers: the four-step algorithm, the "types-first is forced by dependency structure" rationale, the multi-variable case, nested polymorphism with bottom-up ordering, error-reporting guarantee, worked `pipeline` example with step-by-step resolution, and the interaction with trait-method effects from `T`-bound method calls.

### ~~11. Ownership syntactic rule — edge cases for parameter mode inference~~ ✅ RESOLVED

**Decision:** The consume predicate is the single source of truth for both parameter mode inference (Feature 4 Part 1) and RC dataflow (Feature 4 Part 4). It is a **syntactic AST-level classifier** that runs over function bodies as a pre-order traversal and labels each occurrence of a value as *read* or *consume*. Classification for call-form occurrences is driven by **method/function resolution**, not name matching: the compiler looks up the resolved method's receiver mode (or the resolved callee's parameter mode) and reads the classification off that signature. The predicate is **insensitive to nesting depth, control-flow reachability, loop-execution count, and `unsafe` blocks**: a consume is a consume regardless of what it is wrapped inside. The existing use-predicate table in design.md Part 4 is extended with missing rows (partial moves through field projection, tail expression in return position), and a new "The consume predicate" subsection is added to Part 1 spelling out the five edge cases from the brainstorming plus the meta-rules.

**Sub-questions answered:**

1. **Is `.clone()` a consume?** **No — it's a read.** `.clone()` resolves to `Clone.clone(ref self) -> Self`; the use-predicate row "method call with `ref self` receiver" classifies it as a read. There is no special case for the method name — the classification is purely a function of the resolved method's receiver mode. Two methods both named `clone` with different receiver modes would classify differently.

2. **Is `.into_iter()` a consume?** **Yes.** `.into_iter()` resolves to `IntoIterator.into_iter(own self) -> Self.IntoIter`; the row "method call with `own self` receiver" classifies it as a consume. Same mechanism — receiver-mode lookup on the resolved method.

3. **Does capture-by-move in a closure count as a consume of the captured binding?** **Yes, at the closure creation site.** The existing table row "`|args| body` captures by move" already covers this: the consume is located at the closure-creation expression, not where the closure body later runs. This is necessary for per-function dataflow to be sound — the closure may execute much later (stored, returned, called lazily), so the outer function's parameter mode must be determined at the point the closure is constructed. How the capture mode is *inferred* for each captured variable is closure-capture inference territory (separately tracked under item 12 of this brainstorming doc).

4. **Conditional consume inside a loop nested inside a conditional?** **Nesting depth is irrelevant.** The predicate is a pre-order AST traversal with no branch pruning — a consume at depth 17 inside five nested `if`s inside two nested `while` loops is the same as a consume at the top of the function body. There is no reachability reasoning, no branch-condition evaluation, no loop-execution-count analysis, and no path-sensitivity. The "dead branches count as consuming" rule in Part 1 is exactly this principle applied to branch conditions.

5. **Consume inside an `unsafe` block?** **Counts exactly like a consume outside an `unsafe` block.** `unsafe` disables ownership/bounds/type *checking* (the pass that rejects use-after-move, out-of-bounds indexing, and type-punning violations), but it does not mask consumes from the syntactic-scan that feeds the use-predicate table. A function whose only consume of its parameter happens inside an `unsafe` block still has inferred parameter mode `own`. This is the natural counterpart to the existing "unsafe disables ownership/bounds/type checking, not the effect system" rule: unsafe also does not disable the syntactic inference of signatures. The programmer cannot use `unsafe` to hide a consume from the function's public contract.

**Additional edge cases enumerated as part of this resolution:**

- **Partial move through field projection.** `consume(v.field)`, `consume(v.a.b)`, and any deeper projection consume the *root* binding `v` for parameter-mode purposes. After a partial move, the parameter is no longer fully owned. A function with only `consume(x.inner_vec)` in its body infers `x` as `own`, even though the rest of `x`'s fields remain unused.
- **Match scrutinee with move patterns.** `match v { ... }` is a consume of `v` when *any* arm binds a field or the whole value by-move (e.g., `Some(x)` extracting owned `x`). It is a read when all arm patterns are ref-only. The classification is computed per scrutinee by walking each arm's pattern and detecting move-binding positions.
- **Tail expression in return position.** Both `return v` and a tail-position `v` in a block that serves as a function body (or a return expression) are consumes. The scan treats the tail expression of a return-position block as equivalent to `return <expr>` for classification purposes.
- **Method/function resolution drives classification, not name matching.** This is the meta-rule. Every call-form row that references receiver mode or parameter mode assumes the compiler has already resolved the call to a specific signature before classifying. Two same-name methods with different receiver modes classify differently. Because ownership analysis processes the call graph bottom-up with SCC fixed-point for cycles, every callee signature is known at the point its call site is classified.

**Rationale for making the predicate purely syntactic:**

- **Predictability beats precision at the signature boundary.** Parameter mode is part of a function's public contract. A rule that depended on control-flow reachability or effect-reachability would force programmers to run dataflow analysis in their heads to predict their own function's mode. A pure AST-scan predicate can be understood by grepping the body for consume positions, full stop.
- **Stability across compiler versions.** Because the predicate is exactly a syntactic scan, a function's inferred mode cannot silently flip between compiler releases as the analyzer gets smarter. What the source says now is what every future compiler will continue to say. This is the stability guarantee stated at design.md Part 1 paragraph 4; the edge-case formalization in this resolution extends that guarantee to cover `.clone()`, `.into_iter()`, partial moves, unsafe blocks, and nesting — all are tied to the same "pure syntactic scan" principle.
- **Sharing the predicate between Part 1 and Part 4 avoids drift.** Parameter mode inference (Part 1) and RC dataflow (Part 4) both need to know whether each AST position is a consume or a read. Sharing one predicate means the two passes see exactly the same classification for every position, forever. Two separate predicates would be a source of subtle bugs where a consume classified one way for one pass disagrees with the classification for the other pass.
- **Method-resolution-driven classification (not name matching) is the only sound option in a language with traits.** Name-based heuristics would break the moment two crates defined methods with overlapping names and different semantics. Receiver-mode lookup on the resolved method is sound and compositional — it extends uniformly to user-defined traits, blanket impls, and generic code.

**Implementation gap acknowledged.** The current `src/ownership.rs` is a linear forward state machine that does not implement the full predicate — it misses partial moves through field projection, the `unsafe`-transparency rule, the match-scrutinee-by-pattern-mode rule, and (most critically) the method-resolution-driven classification for call-form occurrences. The existing Phase 4 entry **OwnershipChecker: implement RC fallback dataflow** handles the dataflow skeleton; the new Phase 4 entry **OwnershipChecker: consume predicate — formalized AST-level enumeration** covers the eight ordered steps needed to implement the full predicate: method-resolution-driven receiver mode lookup, function-resolution-driven parameter mode lookup, partial move through field projection, match scrutinee classification by pattern binding mode, return/tail-expression consume, `unsafe` block transparency, nesting-depth invariance, and shared-predicate reuse between Part 1 and Part 4.

### ~~12. Closure capture mode inference and escape rules~~ ✅ RESOLVED

**Decision.** Closures need no closure-specific ownership or effect analysis. Parameter mode, capture mode, escape behavior, and effect set are all derived from three existing mechanisms: the shared consume predicate (Part 1 / Part 4), the RC dataflow pass with its two-phase Rc→Arc promotion, and the creation-site effect inference rule. The resolution is a consolidation-and-cross-reference pass that collects the derivation in one place so two implementers reach the same programmer-visible behavior.

**Sub-question answers:**

1. **Mode of closure parameters — inferred, same rule as fn parameters.** A closure `|x| body` has its parameter modes inferred by running the same use-predicate scan over `body` that `fn f(x: ...)` inference uses, classifying each use of `x` as read or consume. In check mode against an expected `Fn(A) -> B`, each parameter's mode is taken *from the expected signature* (expected `Fn(ref T)` → `x` is `ref`); a body that then consumes `x` is a compile error at the closure expression. In synthesis mode, the mode is inferred from the body's uses. Nesting depth is irrelevant. Explicit `|x: ref T| body` annotations are allowed but rarely needed.

2. **Closure escape — composes existing rules, no new mechanism.** Four sub-cases cover every escape form (return, field store, channel send, spawn):
   - **Clean escape, no outer use of captures.** Captured owned values move into the closure cleanly; no Rc, no Arc. Returning the closure moves it.
   - **Escape with outer use.** RC trigger 2 fires (closure creation site + outer-scope use, neither dominating). Part 4 Phase 1 tentatively marks the captured value as Rc.
   - **Escape across a parallel region.** Phase 2 of the two-phase Rc→Arc algorithm promotes Rc to Arc when the closure's live range (and therefore the captured values' live range) overlaps a parallel region. Unchanged from §Rc vs Arc — Two-Phase Algorithm.
   - **`ref`-captured value outliving its source.** Standard borrow-check error at the closure expression's escape site; the diagnostic offers three fixes (clone in body, restructure, drop upstream `&` so capture becomes owned and RC fallback handles sharing).
   
   Escape is detected by the RC dataflow pass as the closure value participating in uses whose live range extends beyond the current function. There is no global "promote all captures on escape" rule — each capture is analyzed independently.

3. **Effect variable × capture kind — orthogonal.** Capturing is an ownership operation, never an effect. Capturing a `ref Database` does not give the closure `reads(Database)`; only invoking `db.query()` in the body does. The closure's effect set is determined at elaboration time (creation site) from the body's inferred effects, and it flows into `with E` unification at the call site per the compound-polymorphism four-step algorithm (item 10). Capturing and *invoking* an effectful closure-typed value contributes that value's effects to the outer closure — the regular closure-invocation rule applied recursively. Capturing without invoking carries no effect. `with _` composes the same way: capture-and-invoke a `Fn() with _` → outer closure is `with _`; capture without invoking → no wildcard propagation.

**Rationale.** The three questions in item 12 are all *derivable* from existing rules but never collected or cross-referenced in one place. Silent workarounds in implementation would produce closure-specific paths in `ownership.rs` that duplicate the consume predicate and drift from it over time. The resolution writes the derivation down and locks in the shared-predicate reuse. The explicit "no global promote-on-escape rule" is the load-bearing sentence: it forces each capture to go through the same RC dataflow every other value uses, which is the only way the "Rc for single-thread, Arc for parallel-region overlap" rule stays consistent.

**Implementation gap.** Current `src/ownership.rs` does not yet have the shared consume predicate (item 11 blocks this too) and does not have closure-aware capture collection. The new Phase 4 checklist entry *OwnershipChecker: closure parameter mode, capture mode, and escape analysis* is intentionally separated from the existing RC fallback entry so that when the ownership checker is rebuilt, closure support is wired through the same passes by construction rather than added as a special case after the fact.

### ~~13. FFI effect defaults beyond `blocks`~~ ✅ RESOLVED

**Decision.** Formalize the scattered FFI effect rules into a single `Effect defaults for extern functions` subsection in the FFI section, keyed on ABI, with an explicit trust-not-verify rule. Four concrete rules:

1. **Resource effects default to empty for all `extern` ABIs.** Resource identity is a Kāra-side concept that does not exist in C, so there is no reasonable default. Programmers must annotate any resource effect the foreign function performs. A stale annotation is the programmer's responsibility — the compiler cannot check it because there is no body.

2. **Execution effect defaults are ABI-keyed:**
   - `extern "C"` → `{blocks}`. `@noblock` opts out for pure-CPU helpers. `panics` is *not* defaulted (no ABI-visible unwinding intent); programmers must annotate for `abort` / `exit` / `longjmp` / signal-terminating cases.
   - `extern "C-unwind"` → `{blocks, panics}`. Selecting the `"C-unwind"` ABI is itself the declaration that the function may unwind, so `panics` is implied — no explicit annotation needed. `@noblock` still valid. Calling still requires the caller to carry `panics` on its own signature.

3. **FFI annotations are trusted declarations, not verified.** This is the explicit exception to the general "declared and inferred effects must match" rule at §Effect Inference and Boundaries. For `extern` there is no inferred side, because there is no body. A `extern "C" fn foo() reads(Network);` contributes `reads(Network)` to every caller's inferred set, but the compiler cannot verify that `foo` actually touches the network — trust-the-signature, same boundary Rust draws at `unsafe extern`.

4. **"Effect annotations mandatory" was a stylistic instruction, not a compile-error rule.** The compile-error cases at FFI are narrower: profile constraints violated by an `extern`'s declared set (e.g., `embedded` profile + `allocates(Heap)`) and malformed effect syntax. The linter's pattern-match hints for known-blocking/allocating libc symbols are advisory, not enforcement.

**Sub-question answers:**

1. **What resource effects does `extern "C"` default to?** Empty. The programmer must annotate any resource effect. See Rule 1 above.

2. **How does `extern "C-unwind"` differ from `extern "C"` for effect inference?** The `"C-unwind"` ABI adds `panics` to the default set automatically — a change from the previous design, which required explicit `panics` annotation on `extern "C-unwind"` declarations. Resource effects still default to empty for both ABIs. `blocks` default and `@noblock` override are unchanged for both.

3. **How is an annotation like `extern "C" fn foo() reads(Network)` validated?** It is not validated — it is trusted. The effect checker treats FFI effect annotations as declarations of record and contributes them to callers' inferred sets without ever attempting to verify them against a body (there is no body). A stale annotation is a programmer-correctness boundary, not a compiler-checkable property.

**Rationale.** The four rules collect and formalize what is already scattered across §2222–2235 (blocks leaves), §2290 (parenthetical FFI exemption), §4950–4972 (`C-unwind`), and §4983–5000 (FFI section). The one substantive change is making `panics` a default on `"C-unwind"` instead of a required explicit annotation: since the ABI selection itself is the unwinding declaration, requiring a redundant `panics` suffix was ceremony without information. Everything else is consolidation.

**Implementation gap.** `src/effectchecker.rs` does not yet handle `extern` items at all. The new Phase 3 checklist entry wires the ABI-keyed default set, `@noblock` override, verification exemption, profile-compatibility check, and linter hints into a single pass. Test coverage for the four-way combination (`"C"` / `"C-unwind"` × with/without `@noblock`) is part of the checklist entry.

### ~~14. Blanket impl + effect variable propagation~~ ✅ RESOLVED

**Decision.** Add `with _` to the four conversion trait method declarations (`From.from`, `Into.into`, `TryFrom.try_from`, `TryInto.try_into`). Effects on the blanket `Into`/`TryInto` impls are then inherited per monomorphization from the bound `From`/`TryFrom` impl, using the existing effect-polymorphism rules — no new mechanism. Impl-level effect-variable quantification (`impl[..., with E]`) is not supported in v1, because the method-level `with _` already expresses what it would.

**Four concrete rules:**

1. **Conversion trait methods are declared `with _`.** Before this item the stdlib definitions at §1082–1122 had no effect annotations, which per §2617 would forbid any effectful conversion impl. After this item:
   ```kara
   trait From[T]    { fn from(value: T) -> Self with _ }
   trait Into[T]    { fn into(self) -> T with _ }
   trait TryFrom[T] { type Error; fn try_from(value: T) -> Result[Self, Self.Error] with _ }
   trait TryInto[T] { type Error; fn try_into(self) -> Result[T, Self.Error] with _ }
   ```
   Pure impls (`impl From[i32] for i64`) satisfy the wildcard ceiling trivially — backwards-compatible.

2. **Blanket impl effect propagation is per-monomorphization.** The body of the blanket `Into` impl is a single call to `U.from(self)`. At each `(T, U)` monomorphization, the blanket's `into` method's inferred effect set is exactly the concrete `impl From[T] for U`'s declared effect set. This is not a blanket-impl-specific rule — it's the standard `with _` public-function-calling-less-effectful-helpers rule (§2509 effect set subtyping) applied to a one-line body. Same rule for `TryInto`.

3. **`x.into()` at a call site contributes exactly `U.from(x)`'s effects.** Method resolution picks the blanket, type inference resolves `U` from context, and the call-site effect contribution is the concrete `From[T] for U`'s effect set — no more, no less, no pooling across all possible `U`s.

4. **Impl-level `with E` quantification is rejected.** `impl[T, U: From[T], with E] Into[U] for T` is not valid v1 syntax. The compiler emits a structured diagnostic suggesting `with _` on the trait method instead. Rationale: method-level `with _` already expresses per-monomorphization effect inheritance; a second spelling would create two ways to say the same thing and duplicate work in the effect checker.

**Sub-question answers:**

1. **If `From.from` carries an effect, does the blanket `Into` impl inherit it?** Yes, per monomorphization. A concrete `impl From[ParseError] for AppError writes(Log)` produces a monomorphized `Into.into` for `(ParseError, AppError)` that also carries `writes(Log)`, inherited from the body's single call to `U.from(self)`.

2. **What's the effect of `x.into()` at a call site?** Exactly the effect set of `impl From[T] for U`'s `from` method for the `U` determined by type inference at that site. Worked example: given `impl From[ParseError] for AppError writes(Log)`, the call `e.into()` in a context where the expected type is `AppError` contributes `writes(Log)` to the caller's inferred set.

3. **Is `impl[T, U: From[T], with E]` valid syntax?** No. Impl-level effect-variable quantification is not supported in v1. Method-level `with _` on the trait handles all cases the feature would be needed for, and a second spelling would muddy the mental model.

**Rationale.** Item 14 is resolved by making a small spec change (add `with _` to four trait methods) and then letting the existing effect-polymorphism rules do the work. The alternative — writing a bespoke "blanket impl effect propagation" rule — would duplicate logic that already exists for every generic `with _` function body. Adding explicit effect variables on impl blocks would be two spellings for the same thing, so v1 forbids it. The current `From`/`Into` trait definitions were *too restrictive* — they would have rejected any effectful conversion impl, including refinement `try_from` paths that panic and I/O-backed conversions that allocate. This item catches the mismatch before any code depends on the broken version.

**Implementation gap.** Current `src/effectchecker.rs` does not yet process trait method `with _` declarations (blocked on the named-effect-variable work). When that work lands, the conversion traits should be the first test case. The new Phase 3 checklist entry lists the five steps needed to close the loop.

### ~~15. Const-eval placeholder decisions (deferred items)~~ ✅ RESOLVED

**Decision.** Commit to five placeholder rules for comptime's effect-system interactions and park them in `docs/future_decisions.md § 14`. The comptime feature itself stays deferred; only the effect-system questions are answered now, so the rest of the language does not freeze around conflicting assumptions. The only v1-visible change is reserving two resource names (`CompileTimeEnv`, `CompileTimeHeap`) so user programs cannot declare them.

**Five rules:**

1. **Permitted effect verbs at comptime.** A `const fn` body may perform `panics` (compile-error-with-rendered-message at the invocation site), `reads(CompileTimeEnv)`, and `allocates(CompileTimeHeap)`. All runtime resource verbs on runtime resources are forbidden — the violation is detected at the `const fn` declaration, not at call sites. `blocks` and `suspends` are also forbidden (no scheduler at compile time).

2. **`CompileTimeHeap` is distinct from `Heap`.** Compile-time allocation uses its own resource so `embedded` and `kernel` profiles can still use comptime — those profiles forbid `allocates(Heap)` but permit `allocates(CompileTimeHeap)`. Without this split, comptime would be unavailable in no-runtime-heap environments.

3. **Parameter modes and ownership unchanged for `const fn`.** The consume predicate, use-predicate table, Part 1 parameter mode inference, and Part 4 RC fallback all run unchanged on `const fn` bodies. No comptime-specific ownership system. Rationale: two ownership systems would be a major complexity burden for a feature whose main motivations (custom derives, compile-time validation, codegen) don't need them.

4. **Comptime values have stripped effects at the runtime boundary.** A `const fn`'s effects (`reads(CompileTimeEnv)`, `allocates(CompileTimeHeap)`, `panics`) are attributed entirely to compile-time evaluation. A runtime caller that references a comptime-computed constant sees it as a pure value with zero effects. Code-generating comptime: the generated expression is separately checked at its embedding site; the generating `const fn` itself stays effect-free at runtime.

5. **`CompileTimeEnv` and `CompileTimeHeap` resource names reserved from v1.** A v1 program that declares either name as a user-defined effect resource is a compile error. This is the only v1-visible change item 15 produces; zero existing code declares these names.

**Sub-question answers:**

1. **Which effect verbs are permitted at comptime?** `panics` (as compile error), `reads(CompileTimeEnv)`, `allocates(CompileTimeHeap)`. Everything else is forbidden — specifically, runtime resource verbs on runtime resources, `blocks`, and `suspends`.

2. **Heap vs. `CompileTimeHeap`?** Distinct resource, to keep `embedded`/`kernel` profile composition sound.

3. **Parameter modes?** Unchanged — `const fn` bodies use the same ownership rules as runtime functions.

4. **Runtime-boundary effect carrying?** Stripped at the boundary. Comptime effects are attributed to compile-time evaluation; they do not propagate to runtime call sites that consume the produced value.

**Rationale.** Comptime is the feature most likely to break something else in the language if its effect-system interactions are left unspecified. The five rules are the minimal commitment needed so that (a) the effect-system design is self-consistent, (b) profile composition rules stay sound for `embedded`/`kernel` when comptime lands, and (c) no v1 code accidentally squats on the reserved resource names. None of the rules constrains the comptime *syntax* or *evaluation semantics* — those are intentionally left open for the comptime design pass — they only pin down the effect-system boundary. The "comptime values strip at the runtime boundary" rule is the load-bearing one: it guarantees that runtime effect inference never has to reason about compile-time evaluation, which would otherwise couple two passes that should stay independent.

**Implementation gap.** One small v1 change: the resolver must reject `effect resource CompileTimeEnv;` and `effect resource CompileTimeHeap;` declarations with a pointer to `future_decisions.md § 14`. Tracked in implementation_checklist.md as a Phase 4 Resolver entry. No other v1 compiler work — the full rule set activates when comptime ships.

**Recommendation:** State the detection mechanism: the effect checker rejects any declared `panics`, `allocates(Heap)`, or other forbidden effect inside a function transitively reachable from a `#[gpu]` root. `unreachable()` and `todo()` desugar to `panic!()` and are caught by the same check.

**Fix type:** `design-discussion`

**Resolution:** ✅ RESOLVED — Specified the two-mechanism detection approach in the "Call graph validation" section. Feature violations (heap types, recursion, dynamic dispatch, host-capturing closures) are caught by call-graph validation during type checking. Both phases produce errors with full call chains. `todo()`, `unreachable()`, and `panic!()` all carry the `panics` effect. No dedicated GPU validation pass needed — existing phases handle it.
