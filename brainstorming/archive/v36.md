# Brainstorming v36 — WASM / front-end basics: what must land before Phase 10

**Status: ✓ RESOLVED.** Opened 2026-04-19, resolved 2026-04-19. All six questions (Q1–Q6) landed in canonical docs — `docs/design.md § Feature 7`, `docs/deferred.md` (P2 entries for `await`, transparent WASM threading, `.karapack` manifest), `docs/syntax.md § 3.16` and `§ 8`, `docs/roadmap.md § Phase 10`, `docs/implementation_checklist.md § Phase 10`.

Tracks the design-layer commitments that Phase 10 (WASM / GPU / FPGA codegen targets) needs to rest on so that downstream front-end work — both **server-side rendering (SSR)** and **client-side rendering (CSR)** — has room to grow without retrofit.

---

## Framing

Phase 10 currently specifies WASM as *"another codegen backend, with sequential fallback for concurrency."* That is necessary but not sufficient. A codegen backend produces `.wasm` bytes; it does not decide:

- What **effects** browser/host APIs live under.
- What **FFI shape** foreign functions use to cross the module boundary.
- How a **single source file** compiles to both native (server) and WASM (client) without conditional-compilation chaos.
- What **build artifacts** `karac build --target wasm` produces for downstream bundlers / loaders / runtimes.
- Whether the language's **concurrency story** allows opt-in multi-threaded WASM later, or closes that door.

The pattern from brainstorming v35 (data science) applies here: **commit to the minimal substrate at spec layer now** — types, effects, FFI shape, build-artifact contract — so community libraries can build UI frameworks, RPC layers, SSR renderers, and hydration runtimes on top without each re-inventing the boundary.

**What this doc is NOT.** It is not designing a UI component framework, a reactive-signal library, a JSX-like template syntax, or a hydration protocol. Those are library-level (see `brainstorming/v35.md:244` — frontend UI toolkit stays user-space). This doc is about the *substrate* those libraries will share.

This doc covers six questions. The first four are **forcing functions** — decisions that must resolve before Phase 10 codegen work lands or the stdlib/community will calcify around unexamined assumptions and retrofitting will be breaking. The remaining two are sequencing questions that can resolve during Phase 10 implementation.

Pairs with `docs/roadmap.md § Phase 10` (the codegen plan) and `brainstorming/v35.md:243-270` (the gap catalog that named "frontend UI toolkit" and "cross-compilation ergonomics" as open).

**Tiebreaker principle (added 2026-04-19).** When a question below has multiple reasonable answers of roughly equal cost today, bias toward **capability-neutral framings** (abstract surfaces like `Display`, `Storage`, `Timer`, `Input` that can lower to DOM today and to other host interfaces tomorrow) over **browser-specific framings** (hardcoded `DOM` / `Browser` / `WebAPI` names). Rationale: capability-neutral framings are also better language design on their own merits (same Kāra source compiles to server + browser + embedded without per-target names leaking), and they preserve optionality for future WASM host surfaces (WASI, component-model-based application runtimes, anything else that might emerge). The motivating scenario is sketched in `brainstorming/v37.md` (a parked seed about a post-browser application runtime) — the tiebreaker holds whether or not v37 ever materializes, because the principle is about language quality, not about a specific future platform.

## Forcing-function decisions (resolve before Phase 10 codegen lands)

### Q1. Effect vocabulary for browser / host surfaces ✓ RESOLVED (2026-04-19)

**Resolution.** Option **1D** (two-tier reuse-where-honest) with **capability-neutral naming**. `std.web` declares `Display`, `Storage`, `Console`, `Timer`, `Input` (capability names, not `DOM`/`WebAPI`). `fetch` reuses the existing `Network` primitive. `std.wasi` exists as a parallel module for headless WASM but stays thin in v1 because WASI's current surface maps onto the native primitives. All web/host resources are module-gated (not prelude) so native-only code never sees them.

**Resolution landed in:** `docs/design.md § Effects > Web / Host Effect Vocabulary`. Implementation tasks added to `docs/implementation_checklist.md § Phase 10`.

**Sub-question resolutions:**
- Prelude vs. module: module-gated (`std.web`, `std.wasi`), not prelude.
- Capability vs. host-specific names: **capability-neutral** per the v36:25 tiebreaker.
- `std.wasi` as sibling to `std.web`: **yes**, declared but thin for v1.

### Q1. Effect vocabulary for browser / host surfaces — original framing (resolved above)

**The question.** Kāra already names effect resources for native I/O (`FileSystem`, `Network`, `Env`, `Process`, `Time`). A WASM target — especially in a browser — introduces new surfaces that don't map cleanly onto those names. What effect vocabulary does Kāra commit to for browser/host APIs?

- **1A. Reuse existing resources where semantically close.** `fetch(url)` runs under `sends(Network)`; `localStorage.set(k, v)` runs under `writes(FileSystem)`; `console.log(x)` runs under `writes(Process)` (≈ stdout). Minimizes vocabulary; forces semantic stretching (localStorage is not a file system).
- **1B. Add browser-specific resources alongside.** Name `DOM`, `Storage` (localStorage/sessionStorage/IndexedDB), `Fetch` (or keep under `Network`), `Console`, `Timer` (setTimeout/requestAnimationFrame). Stdlib declares them; user code imports them from `std.web` or similar. Clean but more surface area.
- **1C. Generic `Host` effect — one effect, dynamic dispatch.** Every browser API is `reads(Host)` / `writes(Host)`. Minimal vocabulary; gives up almost all of the effect system's value.
- **1D. Two-tier — reuse where it fits, add where it doesn't.** `Network` covers `fetch` (semantically correct). `DOM`, `Storage`, `Console`, `Timer` are new. Hybrid of 1A and 1B based on whether semantics genuinely line up.

**Lean:** 1D. Reusing `Network` for `fetch` is honest — both are "this function can make a request over the network." Conflating `DOM` with `FileSystem` is not honest, and a UI library saying `writes(FileSystem)` to mean "mutates the DOM" will confuse every reviewer for the rest of Kāra's life. The small number of new resource names is a one-time cost; the per-library mental-translation cost of 1A compounds forever.

**Open sub-questions.**
- Are browser resources in `prelude` or behind `std.web`? Favor `std.web` so effect names don't exist for native-only code that will never touch them (no per-resource cost for SSR-server-only code).
- How does 1D compose with SSR? A component with effect `reads(DOM)` can only run client-side; how is that enforced at compile time when the same component is compiled for both targets? Links to Q3.
- What about WASI (server-side WASM runtimes like wasmtime)? Should `std.web` be browser-specific and a separate `std.wasi` exist for headless WASM? Probably yes — different resource sets, same target.

**Action item if resolved.** Add a "Web/Host Effect Vocabulary" subsection to `docs/design.md § Effects` listing the resource set and which stdlib modules contain them. Update `docs/syntax.md` if any new effect-declaration forms are needed (likely none — existing `reads`/`writes`/`sends` verbs suffice).

### Q2. Host-binding / FFI contract for WASM (and native) ✓ RESOLVED (2026-04-19)

**Resolution.** Option **2C** — language-level `host fn` declaration; compiler lowers per target. Keyword is `host fn` at module scope. Effect declarations are **required** (no default, unlike `extern "C"` which defaults `{blocks}`). Parameter and return types restricted to primitives, `Copy` types, and opaque-handle newtypes. Owned non-`Copy`, `ref`/`mut ref`, and generic `host fn` are **not supported in v1**.

WIT is the **lowering target** for server-WASM (not the user surface); until Component Model runtime support matures, `host fn` for server-WASM lowers to a generated C-ABI + shim pair — migration to WIT when stable is non-breaking at the source level. Browser-WASM lowers to WASM `import` + wasm-bindgen-compatible JS glue. Native lowers to `extern "C"` (identical code path).

`extern "C"` is **not deprecated** — it remains the raw C-ABI primitive; `host fn` is the recommended portable surface.

**Resolution landed in:** `docs/design.md § Host Functions`; grammar in `docs/syntax.md § 3.16`; `host` added to keyword list (`docs/syntax.md` §1). Implementation tasks added to `docs/implementation_checklist.md § Phase 10`.

**Sub-question resolutions:**
- Effects declared vs. inferred: **declared and required**.
- Ownership across boundary: opaque handles are scalars, do not participate in RC; safe lifecycle wrappers are library-layer.
- Generics on `host fn`: **deferred past v1**.

### Q2. Host-binding / FFI contract for WASM (and native) — original framing (resolved above)

**The question.** UI libraries, HTTP clients, DB drivers, and OS wrappers all need to call into host-provided functions. On native, that's `extern "C"` (libc, dlopen). On WASM in a browser, that's `importObject` / JS glue. On WASM on a server, that's WASI or WIT (Component Model). What is Kāra's single FFI surface, such that a library author can declare "this function is provided by the host" once and have it work across targets?

- **2A. Raw `extern` blocks, C-ABI-only.** Rust-style `extern "C" { fn dom_append(parent: i32, child: i32); }`. Simplest; requires hand-written JS shim per browser build; no type safety across the boundary beyond scalar ABI.
- **2B. WASM Component Model / WIT as the canonical contract.** Author libraries against WIT interface definitions; `karac` targets Component Model on WASM, lowers to C-ABI on native. Target-neutral, industry-aligned (wit-bindgen trajectory), supports rich types across the boundary. Cost: WIT is still evolving; runtime support for Components in browsers is a tooling burden today.
- **2C. Language-level `host fn` declaration.** `host fn dom_append(parent: Element, child: Element)` — compiler handles lowering to C-ABI / Component Model / JS binding per target. Highest ergonomics; biggest design commitment; requires a host-ABI mapping layer in the compiler.
- **2D. Layered — 2A as substrate, 2B/2C as opt-in.** Ship `extern` in Phase 10; add WIT / `host fn` later as separate layer(s) that desugar into the substrate. Deferrable, but commits every early library to regenerating itself once the richer layer lands.

**Lean:** 2C with WIT as the *lowering target* for WASM. `host fn` as the surface syntax is the "Rust without being Rust" move — it's the thing JS/Python programmers expect (a function declaration says "this is provided externally" without module-level ceremony), and the compiler does the target-specific lowering once. Cost is concentrated in the compiler, not spread across every library. If WIT is not stable enough at Phase 10 time, 2C can lower to a generated C-ABI + JS shim bundle in the interim and swap to Component Model when mature — the language surface stays stable across that migration.

**Open sub-questions.**
- Does `host fn` participate in effect inference, or must every host function have declared effects (like a public function)? Likely declared — the compiler can't infer effects of code it can't see.
- How does ownership cross the host boundary? For browser APIs, `Element` handles are opaque identifiers (i32 in today's JS interop); they don't participate in Kāra's RC. Probably `host fn` takes only *primitive* / *Copy* / *opaque-handle* types, and richer types are library-layer wrappers.
- Does `host fn` support generics? Probably no in v1 — generic host functions require monomorphization coordinating with the host, which is a research problem.

**Action item if resolved.** Add a "Host Functions" chapter to `docs/design.md` with the `host fn` syntax, effect-declaration rules, and the compilation-target lowering table (native → C-ABI, WASM browser → JS imports, WASM WASI → WIT). Update `docs/syntax.md` with the `host fn` grammar.

### Q3. Cross-target conditional compilation ✓ RESOLVED (2026-04-19)

**Resolution.** Option **3E** (effect-driven gating as primary + `#[target(...)]` escape hatch), with the explicit framing that **provider injection is the primary pattern for SSR-shared components**; `#[target(...)]` is reserved for items that genuinely cannot exist on all targets (entry points, platform-exclusive setup, irreducible forks).

**Key commitments:**
- **Attribute spelling is `#[target(...)]`** (not `@[target(...)]` — the v36 framing got the grammar prefix wrong; Kāra's attribute prefix is `#`).
- **Closed target set for v1:** `native`, `wasm_browser`, `wasm_wasi`, `gpu`. Opening to custom targets is additive post-v1.
- **Attribute argument grammar:** comma-separated target names, each optionally `not(...)`. No general boolean logic.
- **Effect-checker is parameterized over a target→provided-resource-set table.** Target correctness is enforced by rejecting effect sets that reference unprovided resources.
- **SSR-shared components stay target-agnostic**; different I/O providers are bound per target via the existing `with_provider` mechanism. Callers of `#[target(X)]` items from non-X call sites get a diagnostic at the call site.
- **No runtime `if target == ...` form.** Compile-time attribute + providers are the only mechanisms.
- **`karac check` under multi-target builds** type-checks once per target in the declared build matrix; diagnostics are tagged with the target that produced them.

**Deployment-topology independence (from discussion).** Q3's design is deployment-topology-agnostic. Whether the user deploys single-origin (server serves SSR + static WASM/JS), split (static CDN + separate API server), or edge-co-located, the effect-gating + attribute + provider-injection mechanism is identical. Kāra produces artifacts per target; deployment topology is the user's choice.

**Resolution landed in:** `docs/design.md § Feature 7 > Cross-target Compilation` (new subsection with target set, provided-resource tables, effect-gating rules, SSR provider-injection pattern, and `#[target(...)]` semantics); `docs/syntax.md § 8 Attributes` examples. Implementation tasks added to `docs/implementation_checklist.md § Phase 10`.

**Sub-question resolutions:**
- Attribute syntax: **`#[target(...)]`**.
- Closed vs. open target set: **closed** in v1; opening is additive.
- `karac check` behavior: check under every target in the declared build matrix; finite cost because target set is closed.

### Q3. Cross-target conditional compilation — original framing (resolved above)

**The question.** A single SSR application compiles the same component code to two targets: native (server) and WASM (browser). The components share 80–90% of their code, but diverge at the I/O edges — server reads from a DB, client reads from the DOM. How does Kāra express "this branch runs on server, that one on client," and can the type/effect systems prove no target leak happens?

- **3A. Rust-style `#[cfg(target = "wasm32")]` attributes on items.** Familiar; duplicates items; no effect-system integration (server item could accidentally reference client-only resource). Pragmatic.
- **3B. `if target == "wasm32" { ... } else { ... }` as a compile-time conditional.** Block-level, not item-level; bodies are type-checked against the target being compiled. Feels less Rusty, more like Zig's `comptime`.
- **3C. Target-parameterized traits.** Components are generic over a `Target` trait; native and WASM provide different implementations of shared trait methods. Purest ML-style approach; ceremony-heavy for one-off conditions.
- **3D. Effect-driven target gating.** Code that uses `reads(DOM)` *cannot* be compiled to native (no provider); code that uses `reads(FileSystem)` *cannot* be compiled to browser-WASM. No explicit `cfg` — the effect system enforces target correctness by what resources the target environment provides. Shared code uses only target-agnostic effects (`reads(Network)`, pure compute).
- **3E. Hybrid — 3D as the default, `cfg`-attributes only for cases where effect-level gating isn't enough.** Most SSR components would be target-agnostic by effect; the rare "this function is deliberately client-only" case gets an explicit attribute.

**Lean:** 3E. The effect system is already doing the work for free — if `DOM` and `FileSystem` are distinct resources (per Q1 lean), then shared code that uses only `Network` and pure compute is target-agnostic by construction, without any `cfg` annotations. The `cfg`-attribute escape hatch handles the cases where the programmer wants to hand-partition (e.g., "this logging function uses `std.fs` on server, `console.log` on browser"). This is a genuine Kāra differentiator — Rust needs `cfg` everywhere because its type system doesn't know about targets; Kāra's effect system already encodes the target's provided resources.

**Open sub-questions.**
- What's the `cfg`-attribute syntax? `@[target(wasm)]` fits Kāra's attribute grammar better than Rust's `#[cfg(target_arch = "wasm32")]`. Decide at resolution.
- Do target names form a closed set (`native`, `wasm_browser`, `wasm_wasi`, `gpu`) or an open one? Closed is simpler and matches Kāra's ethos of constrained choice; open supports custom targets (embedded, FPGA) without language changes.
- How does `karac check` behave for a file that uses `@[target(wasm)]`? Probably: type-check under every target's resource set, report errors from each. Expensive but correct.

**Action item if resolved.** Spec the target-gating rules in `docs/design.md § Compilation Targets` (new section). Add `@[target(...)]` (or chosen syntax) to `docs/syntax.md`. Update the effect-checker to be parameterized over target-provided resource set.

### Q4. Build artifact contract for `karac build --target wasm` ✓ RESOLVED (2026-04-19)

**Resolution.** Option **4D** — tiered `.wasm` substrate + optional bindings per host mode via `--bindings browser|component|none`. Ship 4B-style browser bindings, 4C-style Component Model bindings, and 4A raw output; `.karapack` manifest (4E) deferred to post-v1 (P2 in `docs/deferred.md`).

**Key commitments:**
- **Flag:** `karac build --target <T> --bindings <browser|component|none>`.
- **Default `--bindings` inferred from target:** `wasm_browser` → `browser`; `wasm_wasi` → `component`. No universal default; the `--target` choice declares the host family.
- **Browser artifacts** (`--bindings browser`): flat `dist/wasm/<pkg>.wasm` + `.js` (wasm-bindgen-compatible ES module) + `.d.ts` (TypeScript declarations). Compatible with vite / webpack / esbuild / rollup default loaders.
- **Component Model artifacts** (`--bindings component`): single `dist/wasm/<pkg>.wasm` with embedded WIT. Until Component Model runtime support is stable, `--bindings component` may emit paired `<pkg>.wasm` core module + `<pkg>.component.wit` descriptor; migration to embedded-WIT is non-breaking at source level with one release of deprecation notice for tooling that consumes the paired shape.
- **Component Model emission** via **external `wit-bindgen`** for v1 (pinned in `kara.toml` toolchain). In-compiler emission is post-v1, gated on spec stability.
- **Entry-point discovery** via `pub` + `#[target(wasm_browser)]` / `#[target(wasm_wasi)]`. No new `#[export]` attribute. Target-agnostic public functions transitively reachable from a target-tagged entry point are also exported (normal dead-code elimination).
- **Native and GPU artifacts** documented alongside for completeness; unchanged from existing behavior.

**Resolution landed in:** `docs/design.md § Feature 7 > Target Build Artifacts` (new subsection covering native, WASM per-bindings shape, Component Model emission, entry-point discovery, GPU); `.karapack` manifest deferral in `docs/deferred.md § P2 > Package Manifest Artifact Format`. Implementation tasks added to `docs/implementation_checklist.md § Phase 10`.

**Sub-question resolutions:**
- JS/TS glue shape: **wasm-bindgen-compatible** calling convention; flat layout (`<pkg>.wasm` + `<pkg>.js` + `<pkg>.d.ts` in `dist/wasm/`).
- Component Model emission: **external `wit-bindgen`**, toolchain-pinned in `kara.toml`.
- Directory vs. sibling files: **flat directory, sibling files**; package name comes from `kara.toml`.
- Default `--bindings`: **inferred from `--target`**, not from a universal WASM default.

### Q4. Build artifact contract for `karac build --target wasm` — original framing (resolved above)

**The question.** What does the WASM build output? This is the equivalent of Arrow-layout commitment for data science — downstream tooling (bundlers, loaders, runtimes) need a stable artifact contract.

- **4A. Raw `.wasm` module only.** Single file, host provides all glue. Minimal; unusable in a browser without hand-written JS.
- **4B. `.wasm` + generated JS glue + TypeScript type declarations.** wasm-bindgen / wasm-pack shape. Immediately usable from JS/TS; ties Kāra's output to browser-JS conventions.
- **4C. WASM Component Model `.wasm` (with WIT embedded).** Target-neutral; works in Component Model runtimes (wasmtime, jco for browsers); host-agnostic; tool ecosystem is still maturing.
- **4D. Tiered — `.wasm` substrate + optional generated bindings per host mode.** `karac build --target wasm --bindings browser` produces 4B-style output; `karac build --target wasm --bindings component` produces 4C; `karac build --target wasm --bindings none` produces 4A. Flexibility without lock-in.
- **4E. Manifest + modules — Kāra emits a `.karapack` manifest (or similar) describing the module set, exported entry points, embedded WIT, and declared effect requirements per export.** Highest integration; gives downstream tools (bundlers) structured info to work with; one more format to own.

**Lean:** 4D in the short term, with 4E as a later consolidation. Shipping 4B (browser-targeted bindings) unblocks the playground and early UI libraries; 4C (Component Model) covers server-WASM and non-browser embedders; 4A is always available. The `--bindings` flag lets users choose per build. 4E (manifest) is a valuable future consolidation once we see what tools actually want from us — premature now.

**Open sub-questions.**
- For 4B, what JS/TS glue shape? Hand-written or match wasm-bindgen's calling conventions so existing JS ecosystem tools (vite, webpack) work without custom loaders?
- For 4C, does `karac` depend on an external tool (wit-bindgen) or emit Component Model bytes directly? External dependency keeps compiler simpler but creates a version-coupling problem.
- What's the naming convention for bundles — single `.wasm` + sibling `.js`, or a directory? Affects every downstream integration story.
- Does `--target wasm` default to `--bindings browser` or require an explicit flag? Defaulting to browser matches the most common case; requiring explicit flag prevents accidental browser-JS lock-in.

**Action item if resolved.** Add a "Build Artifacts" section to `docs/design.md § Compilation Targets` documenting the per-target artifact contract. Update `docs/implementation_checklist.md § Phase 10` to enumerate the artifact-emission tasks.

## Sequencing decisions (can resolve during Phase 10 implementation)

### Q5. WASM threading — sequential, opt-in, or compiler-managed transparency ✓ RESOLVED (2026-04-19)

**Resolution: two-decision split.**

**Decision 1 (committed now — source-level).** `go` / channel / `par` semantics are specified **target-agnostically at the source level**. Task semantics, channel ownership transfer, `par` auto-concurrency rules, and ownership-proven data-race-freedom are language-level properties that hold on every target. This closes the "Q5 is a forcing function" door: no source-level bifurcation across targets, so any future lowering change (including 5D compiler-managed transparent threading) is non-breaking.

**Decision 2 (Phase 10 delivery shape).** Ship **5A sequential-default + 5B `--features wasm-threads` opt-in**. Defer **5D/5E** (compiler-managed transparent threading, WASM stack switching for fiber-weight tasks) to post-v1 with concrete re-evaluation triggers. **5F** (WebGPU as `.par_iter` pool) is orthogonal to Q5 and tracked separately in the existing `#[gpu]` / `gpu.dispatch` design. **5G** (native shell) is a distribution mode, not a threading model — out of Q5 scope. **5H** (shared-everything-threads) is a watch item; if it ships it's additive to 5B and/or the deferred 5D.

**Why not land 5D as v1.** Phase 10 needs to ship. Committing to compiler-managed transparent threading as the v1 story invites either missing the phase or shipping a half-working version that poisons the differentiator claim. The source-level commitment already preserves the 5D path.

**Why the source-level commitment matters.** The original framing treated Q5 as "reversible sequencing" but **5A → 5D is not purely additive** without source-level guarantees (channels carrying ownership metadata that survives worker transfer, spawn semantics target-agnostic, etc.). Decision 1 locks those in so 5D remains a non-breaking future change.

**Resolution landed in:** `docs/design.md § Feature 7 > Concurrency Across Targets` (new subsection — source-level commitments + WASM lowering table); `docs/deferred.md § P2 > Compiler-Managed Transparent Threading on WASM` (5D/5E with re-evaluation triggers); `docs/roadmap.md § Phase 10` (retired "sequential fallback" phrasing, replaced with sequential-default + opt-in + deferred link); implementation tasks in `docs/implementation_checklist.md § Phase 10`.

**Sub-question resolutions:**
- How does deferred 5D interact with `host fn` (Q2)? Out of scope for v1 decision; spec 5D's `host fn` worker-safety rules when (if) 5D escalates from deferred to active.
- How does deferred 5D interact with `#[target(...)]` (Q3)? Workers are not a target; the compiler would auto-partition. Not relevant to v1.
- Pool sizing for `--features wasm-threads`: **runtime-dynamic by default** (`navigator.hardwareConcurrency`), configurable by entry-point flag. Landing as an implementation detail in `implementation_checklist.md`, not a design.md spec.
- Graceful degradation when SAB is unavailable (COOP/COEP unset) with `--features wasm-threads` active: runtime emits `console.warn` and falls back to single-worker + main-thread scheduling unless the program's build manifest flags an opt-out. Implementation detail, tracked in `implementation_checklist.md`.

**Status escalation note.** v36 flagged Q5 as "sequencing, may become forcing-function if 5D is aspirational." Resolution takes the middle path: Decision 1 treats the source-level surface as *forcing* (commit now, non-negotiable for future 5D), Decision 2 treats the lowering shape as *sequencing* (Phase 10 ships 5A+5B; 5D is deferred not abandoned).

### Q5. WASM threading — sequential, opt-in, or compiler-managed transparency — original framing (resolved above)

**The question.** `docs/roadmap.md § Phase 10` currently says "WebAssembly: LLVM WASM backend, **sequential fallback for concurrency**." That framing implies a false binary. Browsers *do* support multithreading today, with caveats — what model does Kāra target, and is there a more ambitious option hiding behind the sequential/opt-in framing?

**Factual ground — what browsers actually support in 2026.**

- **Web Workers** (shipped ~2010) — full OS threads, isolated memory, communicate via `postMessage` (serialization cost).
- **SharedArrayBuffer + Atomics** (shipped in all modern browsers) — true shared memory between main thread and workers. **Requires COOP/COEP headers** (Cross-Origin-Opener-Policy / Cross-Origin-Embedder-Policy, Spectre mitigation). This header requirement is the actual real-world blocker — many hosts, CDNs, and iframe-embedding contexts cannot set it.
- **WASM threads proposal** (shipped) — uses SharedArrayBuffer + atomics under the hood. Rust's `rayon` + `wasm-bindgen-rayon` uses this today.
- **WASM stack-switching proposal** (in-flight) — gives fiber-weight concurrency, thousands of tasks over a small worker pool without per-task worker-creation cost.
- **WebGPU compute** (shipped) — orthogonal parallelism axis; not "threads" but covers a huge class of data-parallel workloads.
- **Shared-everything-threads proposal** (W3C / WASM-CG WIP) — emerging work to relax SAB's COOP/COEP requirement via redesigned isolation.

So "sequential fallback" is a **choice**, not a constraint. The question is which shape to pick.

**What's actually hard about browser multithreading.**

1. **COOP/COEP deployment friction** — blocks many embedding contexts. Real-world blocker, not runtime.
2. **Worker startup cost** — 10–100ms per worker; pool required for anything interactive.
3. **DOM isolation** — workers can't touch DOM; threading is compute-only, not UI.
4. **No in-WASM spawn** — WASM can't create workers itself; main JS must orchestrate.
5. **Debugging story** — browser devtools for multithreaded WASM is primitive.
6. **Memory-model stack** — WASM memory model + JS atomics + Kāra ownership on top.

**Options — grouped from most conservative to most ambitious.**

- **5A. Stay sequential.** Channels and `go`-style concurrency lower to cooperative single-thread scheduling on WASM. No atomics; no COOP/COEP headaches. Smallest bundles, widest deployment. Caps the language's differentiator story — Kāra's concurrency model sells compile-time data-race freedom, and a sequential-only flavor doesn't showcase it.

- **5B. Opt-in `--features wasm-threads` enables atomics + shared memory.** Default sequential for small-bundle always-works shipping; opt-in for compute-heavy browser apps that want Web Workers + SharedArrayBuffer (games, ML inference, simulation). User pays the COOP/COEP cost when they opt in. More work than 5A; preserves the concurrency story for users who need it.

- **5C. Always multi-threaded when concurrency is used, sequential when it isn't.** Compiler detects whether the program uses `go`/channels and picks the artifact shape automatically. Simpler UX but artifact shape silently changes based on imports — opaque to the user when deployment breaks.

- **5D. Compiler-managed transparency (the "wild" baseline).** Kāra's `go` / channels lower to workers + SharedArrayBuffer automatically; the worker/message boundary is invisible at the language level. The compiler decides which closures cross workers based on effect analysis. **Ownership already proves data-race freedom** — Kāra is the rare language that can emit this correctly without user annotation. Distinguishing from 5B: the user never writes worker / postMessage code, and never even sees that concurrency became multi-worker. The language surface is the same on native and WASM.

- **5E. 5D + WASM stack switching for fiber-weight concurrency.** Ride the in-flight stack-switching proposal to get thousands of `go`-tasks over a small worker pool without per-task worker-creation cost. Combines with 5D's transparency — users write `go { ... }` freely, the runtime pools + multiplexes. Matches Go's goroutine model on WASM.

- **5F. WebGPU as the thread-pool for data-parallel work.** For `.par_iter()` / `arr.map(f)`-shaped workloads, dispatch compute shaders from the main thread — no Web Workers, no COOP/COEP, no SAB. Not general threading but covers physics, simulation, ML, image processing. Complementary to 5A–5E: threading covers task parallelism, WebGPU covers data parallelism.

- **5G. Ship a Kāra runtime that bypasses COOP/COEP.** Browser extension or Tauri / Electron-style native shell that embeds Kāra modules with first-class threading, no browser deployment constraints. Drops the "it's just a website" property for games / heavy apps where the user downloads an installer anyway. Not practical as the default but a valid target for high-end demos.

- **5H. Bet on shared-everything-threads landing.** Design Kāra's threading model against the emerging W3C/WASM-CG proposal that removes SAB's COOP/COEP requirement. Kāra could be among the first languages designed around it shipping. Timing risk: proposal may not land, or may land with different shape.

**How these compose.** These are not mutually exclusive. A realistic ambitious target is **5D + 5E + 5F + 5H watch**: compiler-managed transparent threading (5D), stack switching for fiber-weight tasks (5E), WebGPU for data-parallel (5F), and design the model so shared-everything-threads (5H) drops in without source-level change when it ships. 5G is an orthogonal distribution mode, not a threading model.

**Leans (not decisions — user will prioritize p0/p1/p10/throwaway later):**

- If Phase 10 target is "compute-bound programs run in browser" (current roadmap): **5A is enough**.
- If the target is "games and heavy apps become viable": **5B minimum, 5D aspirational**.
- If the target is "Kāra is *the* language for safe, ergonomic browser concurrency": **5D + 5E is the story**. This is the only option set where Kāra says something that Rust, Go, and JS cannot — because ownership already proves the safety property, and no other language has both that property and a first-class browser story.

**Why this is not simple sequencing.** The original framing marked Q5 as "reversible later — sequential-only lowering can be upgraded without breaking source." That's true for 5A → 5B transitions (opt-in flag is additive). **But 5A → 5D is not purely additive**: 5D requires that `go`/channel semantics were always *target-agnostic* at the language level, which influences how the surface is specified today (e.g., whether channels carry ownership metadata that survives worker transfer). If 5D is a serious aspiration, Q5 becomes a forcing function, not a sequencing question. If 5D is throwaway, Q5 stays sequencing.

**Open sub-questions.**
- How does 5D interact with Q2 (`host fn`)? A `host fn` call from a background worker can only reach host functions the worker has bindings to. The compiler needs to know which host functions are worker-safe.
- How does 5D interact with Q3 (target gating)? Worker code is effectively a separate compilation target — does `@[target(worker)]` become a thing? Probably not; the compiler should auto-partition.
- Pool sizing — static (compile-time `--workers N`), dynamic (runtime detect `navigator.hardwareConcurrency`), or configurable at the entry point? Default dynamic.
- Graceful degradation — what happens when 5D compiles but the browser doesn't have SAB available (COOP/COEP unset, or old browser)? Fall back to single-worker + main-thread scheduling? Error at startup? Honest answer probably: fall back with a `console.warn` unless the program opts out with a manifest flag.

**Action item if resolved.** Update `docs/roadmap.md § Phase 10` with the chosen threading model (and explicitly retire the "sequential fallback" phrasing — it misrepresents the option space). If 5D or 5E are in scope for v1, escalate Q5 from sequencing to forcing-function and spec the effect-system interaction with worker partitioning in `docs/design.md § Concurrency`. If deferred past Phase 10, capture 5D/5E/5F/5H in `docs/deferred.md` as a P1 or P2 entry depending on aspiration level.

### Q6. Async runtime / event-loop integration ✓ RESOLVED (2026-04-19)

**Resolution.** Option **6C** — no new language surface in v1. Promise-returning host APIs declared with `suspends` effect on their `host fn` signature; call sites look like any other function call with no `.await` / `.then` / `async` ceremony. Callback-driven host APIs (timers, events) modeled as channel producers in library wrappers over `host fn`. Scheduler contract: channel-receive on a host-async channel yields the `go`-task and resumes on completion. `await` keyword is **deferred to post-v1** with concrete re-evaluation triggers.

**Resolution landed in:** `docs/design.md § Feature 7 > Async Host APIs on WASM` (new subsection — Promise-returning host-fn pattern, callback-to-channel wrapper pattern, scheduler contract); `docs/deferred.md § P2 > \`await\` Keyword for Async APIs` (deferred with triggers); implementation task in `docs/implementation_checklist.md § Phase 10` (scheduler yield-to-event-loop contract).

**Key points:**
- Consistent with existing "no `async fn`, no function coloring" stance (`design.md § What Kāra Is Not`).
- No `Future[T]` as a first-class language type in v1 — if one emerges, it's a library-level wrapper on channels.
- `suspends` effect propagates through host-async calls per normal effect-propagation rules.
- Same source compiles to native and WASM; the scheduler absorbs the yield-to-event-loop difference.

### Q6. Async runtime / event-loop integration — original framing (resolved above)

**The question.** Browser APIs (`fetch`, `setTimeout`, `requestAnimationFrame`) are Promise-based and event-loop-driven. Server runtimes (tokio-equivalent) use their own async machinery. Kāra's concurrency story today is channels + effects + `go`-style tasks. How do these map?

- **6A. Effect-tracked `await` keyword that integrates with each target's native event loop.** Compiler lowers `await expr` to `.then(...)` on WASM-browser, to `.await` on native async runtime. Surface is uniform; lowering is target-specific.
- **6B. Channels-only — wrap every async API as a channel-receive.** `fetch_channel.recv()` blocks the current `go`-task, which the scheduler translates into a Promise continuation on WASM. No `await` keyword; Kāra's existing concurrency primitives cover everything. More consistent; requires every async library to model itself as channels.
- **6C. Library-level for v1, language-level later.** Ship Phase 10 with channels-only (6B); evaluate `await` (6A) once we see what idioms emerge in user-space.

**Lean:** 6C. This is exactly the kind of decision that benefits from seeing real library shapes before committing language surface. Channels-over-Promise is implementable (the scheduler can yield to the event loop on receive), and if user experience proves channels awkward for UI code, `await` is additive. Avoid locking in a keyword that might interact badly with the effect system or ownership.

**Why sequencing (not forcing function).** Adding `await` later is additive — existing channel code continues to work. The forcing question is only: does the lowered channel-receive correctly integrate with the browser event loop? That's a Phase 10 runtime implementation detail, not a spec commitment.

**Action item if resolved.** Add a "Concurrency on WASM" subsection to `docs/design.md § Concurrency` documenting the channel-to-event-loop lowering. Defer `await`-keyword design to post-Phase-10 evaluation.

## Out of scope (explicitly not in this doc)

These are real design questions for front-end work but they are library-level / post-v1 and do not block Phase 10:

- **JSX / template literal syntax for HTML.** Library-level via `sql!(...)`-style macros once macros exist. Not a language feature.
- **Reactive primitives (signals, observables).** Library-level. Language may grow a hook later if a community consensus forms, but not in v1.
- **Hydration protocol.** Depends on whatever SSR framework emerges; language doesn't prescribe.
- **Component lifecycle.** Library-level.
- **Routing.** Library-level.
- **CSS-in-Kara / styling story.** Library-level.
- **Browser GPU graphics / rendering (games, 3D apps).** Distinct from the GPU *compute* story already specced at `docs/design.md:1075-1081` + `brainstorming/v2.md § GPU Subset Constraints`, which covers SPIR-V/WGSL codegen for `#[gpu]`-marked kernels. **GPU graphics** — textures, render passes, vertex/fragment pipelines, depth/stencil, swapchains, plus the adjacent Web Audio / Gamepad / PointerLock surfaces a real game needs — is a library-level concern and an entire design area of its own (Rust's `wgpu` is the reference shape at ~15k LoC). The language does not ship a graphics pipeline; a community `kara-gfx` (or equivalent) library is expected to fill that role once Phase 10 lands. **It rests on Q1 and Q2 of this doc**: Q1 decides whether `GPU` is a distinct effect resource alongside `DOM`/`Timer` (and how it differs from the compute-side resource, if at all), and Q2 decides the `host fn` shape such that WebGPU / Vulkan / Metal / DX12 bindings can be expressed uniformly across targets. Compute-shader codegen is a Kāra-compiler feature; graphics-pipeline binding is a library feature.

These are tracked in `brainstorming/v35.md:244` ("frontend UI toolkit") as user-space once Phase 10 lands. This doc's goal is to ensure that when community libraries ship those, they rest on a stable substrate (the six questions above).
