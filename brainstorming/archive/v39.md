## Group A — Trait-contract cluster

These traits are referenced as **bounds** or **derivable** throughout `design.md`, but the trait definitions themselves (signatures, associated types, semantics) are absent. Generic library code cannot name these bounds unambiguously until the trait surface is committed.

### A1. `Drop` — user-defined destructors  *(✓ RESOLVED — 2026-04-20)*

**Why foundation-breaking.** `design.md` describes destructor runtime behavior (panic unwinding at `:3621`, RC-decrement-then-drop, `defer`/`errdefer` as RAII substitute at `:375`, TaskGroup drop at `:5406`, "Drop ordering within a branch" at `:5498`), but there is no way for a user type to *declare* a destructor. Types like `File`, `Connection`, `MutexGuard`, socket handles, and transaction guards need deterministic close-on-scope-exit.

**Resolution.** Landed in `design.md` as Feature 4 Part 7: `Drop` — User-Defined Destructors. Decisions:

- **Signature:** `fn drop(mut ref self)` — drop must mutate but not consume; consuming cleanup is a separate explicit `close(self)` method.
- **`Copy` exclusion:** Drop and Copy are mutually exclusive; enforced at the impl site with a diagnostic naming both traits.
- **Panics:** forbidden statically. Effect checker rejects any `fn drop` whose body is not `panics`-free — no runtime fallback. Fallible cleanup must be swallowed or logged in-place, never propagated.
- **Relation to `defer`/`errdefer`:** distinct mechanisms that compose in the same LIFO stack. Drop is type-attached (inherent to the type); `defer` is scope-local (inherent to this code). Rule of thumb documented in the section.
- **Derivability:** manual only — no `#[derive(Drop)]`. A derived body would be either empty or duplicate the built-in field-drop pass.
- **Field drop order:** reverse declaration order, running after the user's drop body returns. Inside the drop body, all fields are still live.
- **Move interaction:** partial moves out of a field are rejected for types with `Drop` (the destructor assumes all fields are present); types without `Drop` may have fields moved out individually.

### A2. `Hash` — trait definition + `Hasher` abstraction  *(✓ RESOLVED — 2026-04-20)*

**Why foundation-breaking.** `T: Hash` is used as a bound 20+ places (`Map[K, V]` at `:830`, `Set[T]` at `:846`, `#[derive(Hash)]` at `:1464`, parameterized-resource keys at `:4168`, `F32`/`F64` total-order wrappers at `:1144-1148`). Late commit changes every `Map[K, V]` signature if `Hasher` becomes a type parameter, and hash stability is an observable semantic contract that cannot be retrofitted.

**Resolution.** Landed in `design.md` as a new `### Hash and Hasher` subsection after Operator Traits. Decisions:

- **Signature:** `fn hash[H: Hasher](ref self, hasher: mut ref H)`. `Hasher` is a separate trait with `write`, `finish`, and primitive-width convenience methods.
- **`BuildHasher` companion:** holds per-table seed/config; produces fresh `Hasher` per operation. Matches Rust split.
- **User-extensible:** `Hasher` and `BuildHasher` are ordinary user traits. No fixed set.
- **Stability:** default hasher is DoS-resistant, per-process-seeded. Hash values are **not stable across runs, versions, or targets**. Stable-digest use cases (content addressing, snapshot tests, sharding) go through an explicit `hash::stable` module with versioned, byte-oriented functions (`siphash24`, `xxh3`); cryptographic uses go through `crypto`.
- **Eq consistency:** documented contract (`a == b ⇒ same hash-byte stream`), not statically checked.
- **v1 default:** `SipHash13BuildHasher`. Changeable across versions; users who pin a faster hasher (`Map[K, V, FxBuildHasher]`) opt out of DoS-resistance and the change-across-versions escape hatch.
- **Derive:** `#[derive(Hash)]` feeds each field in declaration order (enums feed discriminant first, then active variant's fields). All field types must be `Hash`. Manual impls allowed but should be rare.

### A3. `PartialEq` / `PartialOrd` — formal traits  *(✓ RESOLVED — 2026-04-20)*

**Why foundation-breaking.** `design.md:1142` says "`f32`/`f64` implement `PartialEq` and `PartialOrd` but not `Eq`/`Ord`" — but neither trait was defined elsewhere. Generic float-compatible code had no bound to name, and the operator-trait lowering committed to `Eq.eq` / `Ord.cmp` which would have had to be retro-renamed.

**Resolution.** Folded into `design.md` Operator Traits section. Decisions:

- **Operator lowering:** `==`/`!=` lower to `PartialEq.eq`; `<`/`<=`/`>`/`>=` lower to `PartialOrd.partial_cmp` with `.is_lt()`/etc. helpers. Every comparison in the language goes through the partial trait — generic code that needs totality adds the `Ord` or `Eq` bound separately.
- **Hierarchy:** `Eq: PartialEq` (marker, adds no methods), `PartialOrd: PartialEq`, `Ord: PartialOrd + Eq` (adds `fn cmp -> Ordering` without the `Option`). Matches Rust; the single-supertrait pedagogical example in `### Trait Constraints (Supertraits)` was updated to the real multi-bound form.
- **`partial_cmp` return:** `Option[Ordering]`. `None` means incomparable. Rejected a fourth `Ordering` variant — would contaminate `Ord.cmp` users who should never branch on incomparability. `Option[Ordering]` gets its own `.is_lt()`/`.is_le()`/`.is_gt()`/`.is_ge()`/`.is_eq()` helpers that return `false` on `None` (matches IEEE — `NaN < 5.0` is `false`).
- **Float treatment:** `f32`/`f64` implement `PartialEq + PartialOrd`, not `Eq` or `Ord`. `F32`/`F64` total-order wrappers implement all four.
- **Stdlib impl list:** reordered to enumerate `PartialEq`/`PartialOrd` first, then the `Eq`/`Ord` markers excluding floats.
- **`Eq` marker body:** `impl Eq for MyType {}` is the full body a user writes. `Ord` adds `fn cmp -> Ordering`; writing `impl Ord` is a claim that `partial_cmp` never returns `None`.

### A4. `Clone` — trait definition  *(✓ RESOLVED — 2026-04-20)*

**Why foundation-breaking.** `.clone()` is load-bearing in the ownership design (`:4616`, `:2746`, RC-fallback-vs-deep-clone reasoning), `T: Clone` is used as a bound (`:773`, `:1562`), `#[derive(Clone)]` is listed as derivable (`:1469`), but no formal trait definition.

**Resolution.** Landed in `design.md` as Feature 4 Part 7: `Clone` — Explicit Duplication (between Part 6 Copy and Part 8 Drop; Drop renumbered from Part 7 to Part 8). Decisions:

- **Signature:** `fn clone(ref self) -> Self`. Fixed, no variants.
- **Infallible:** returns `Self`, not `Result[Self, E]`. Fallibility would poison every generic signature and every derive. Consistent with `allocates(Heap)` panic-on-OOM stance. Types needing visible fallibility expose a separate `fn try_clone(ref self) -> Result[Self, Error]` and do not implement `Clone`.
- **`Copy: Clone`:** `Copy` is a supertrait of `Clone`. Formalizes the existing `#[derive(Copy)]` + `#[derive(Clone)]` requirement at `:5192`. `T: Copy` implies `T: Clone` via subsumption.
- **Shallow vs. deep:** each type's impl decides per its ownership model. Owned data (`String`, `Vec`) is deep; `Rc`/`Arc` is a refcount bump; `ref T` doesn't implement `Clone`; primitives are bit-copy. A struct's derived `Clone` calls `.clone()` on each field and trusts the type — composition inherits its leaves' choices.
- **Derive:** `#[derive(Clone)]` clones each field (declaration order for structs; per active variant for enums). Compiler infers minimal bound on generic params (`impl[T: Clone] Clone for Pair[T]`), consistent with `:1568`.

### A5. `Debug` — programmer-facing formatter  *(✓ RESOLVED — 2026-04-20)*

**Why foundation-breaking.** `Display` (`:2562`) covers user-facing formatting. `dbg()`, `assert_eq` failures, panic messages, and trace output all need a programmer-facing formatter where `Display`'s prose is wrong (`String` should print with quotes; `None` should print as `None`). Adding `Debug` later shifts every `println`/`dbg`/`assert_eq` signature, every derive list, and the format-specifier minilanguage.

**Resolution.** Trait shape landed in `design.md` String Interpolation section (extended bullets for Display/Debug split, format-specifier routing, no-fallback rule, derive policy, tooling linkage). The *minilanguage* work remains a deferred item; the *split* is committed now so the minilanguage landing is additive. Decisions:

- **Separate trait, not a mode flag.** `trait Debug { fn fmt_debug(ref self) -> String }` sits alongside `Display`. The two formats have different audiences; a mode flag would force every impl to juggle both.
- **No fallback when `Debug` missing.** Compile error naming the trait and suggesting `#[derive(Debug)]`. Silent fallback to `Display` would surface the exact class of bugs the split exists to prevent.
- **Opt-in derive.** No auto-derive. Matches Kāra's stance on all derives (explicit, because bound propagation / code size / semver is observable). Compiler *suggests* `#[derive(Debug)]` whenever a type first appears in `dbg`, `assert_eq`, or a `{:?}` slot. Added to derivables list at `:1469`.
- **Format-spec routing pinned:** `{}` → `Display`, `{:?}` → `Debug`, `{:#?}` → `Debug` pretty (multi-line, indented). Three symbols reserved now; minilanguage (width/precision/alignment) lands later without rerouting. `{:?}`/`{:#?}` in v1 emit a "deferred, coming with format specifiers" diagnostic rather than a syntactic error.
- **Linkage:** `dbg` formats via `Debug`; `assert_eq!` bound becomes `T: Debug + PartialEq`; panic messages mix `{}` (Display) and `{:?}` (Debug) freely; test-framework failure output uses `Debug`. All upgraded as one batch when the split lands.

## Group B — Semantics commitments

These are spec gaps where the language has a decision *implicit in examples* but not pinned in text. Late pinning is observable.

### B1. Enum discriminant runtime surface  *(✓ RESOLVED — 2026-04-20)*

**Why foundation-breaking.** `#[repr(uN)]` is defined (`:3135`, `:3178`) and explicit discriminants are shown (`UsbClass { Audio = 0x01 }` at `:6162`). Reading discriminants at runtime, int→enum conversion, semver of discriminant values, and variant iteration all need a pinned story before protocol/FFI libraries lock in idioms.

**Resolution.** Landed in `design.md` as `#### Enum Discriminant Runtime Surface` inside the `#[repr]` section. Decisions:

- **Read mechanism:** auto-generated `fn discriminant(ref self) -> D` method (D = repr type on `#[repr(intN)]`, conservative `u32` on non-repr C-like). Not `as` coercion — method makes call sites self-documenting and doesn't conflate safe/unsafe cases.
- **Payload enums excluded in v1.** Niche optimization and variant-layout stability aren't committed. Users needing a stable tag write an explicit `fn tag(ref self) -> u8` with a `match`. Lifting the restriction later is additive.
- **`int → enum`:** auto-generated `impl TryFrom[intN] for Foo` for every `#[repr(intN)]` C-like enum, returning `Err(DiscriminantError.OutOfRange { value })` for unmapped values. No `From` impl (most ints are out-of-range). No dedicated `unsafe_from_discriminant` — drop to `unsafe { transmute }` if needed.
- **Variant iteration:** auto-generated type-level `fn values() -> Slice[Self]` in declaration order. Only on C-like enums.
- **Semver:** `#[repr(intN)]` declaration locks discriminant values across minor versions. New variants at the end are minor-safe; reordering/removal/value-change is breaking. Non-repr'd enums have compiler-chosen discriminants that may shift — `karac explain` on `.discriminant()` for a non-repr enum notes this.
- **`.values()` bundled with this decision:** yes, all four parts (`.discriminant()`, `TryFrom`, `.values()`, semver lock) land as one surface because they share the C-like-enum restriction and the `#[repr]` semver promise.

### B2. Module-level `let mut` — concurrency + profile semantics  *(✓ RESOLVED — 2026-04-20)*

**Why foundation-breaking.** Design.md (`:594`) permitted `let mut` at module scope but left the concurrency model, effect attribution, and profile gating unpinned. Without a spec, libraries and programs would drift into patterns that couldn't be sandboxed later.

**Resolution.** Landed in `design.md` by expanding the `### Module-Level Bindings` section. Decisions:

- **Effect attribution:** synthetic per-binding resource (`effect resource BINDING_resource;`). Read contributes `reads(...)`, assignment contributes `writes(...)`. Fine-grained so unrelated bindings don't serialize against each other. Project-internal, non-namable in user code.
- **`par {}` interaction:** compile error to write a bare `let mut` inside `par { }` or `spawn()`. Requires wrapping in `Atomic[T]`, `Mutex[T]`, `RwLock[T]`, or `Arc[shared struct]`. Reads unrestricted. Static rule — upgrade the existing conflict-serialization into an explicit error because silent serialization is almost never what the programmer meant.
- **`#[thread_local]`:** ships in v1 as the per-task alternative. Effect becomes `writes(ThreadLocal[BINDING_resource])`, parameterized so tasks on different threads never conflict. Permitted everywhere module `let mut` is, except `gpu`.
- **`LazyLock[T]`:** stdlib wrapper for expensive-init, write-once, thread-safe globals. Declared with `let` (not `let mut`) at module scope because it's logically immutable after init. First-access init attributes to the *calling* function, not to the module — sidesteps the module-init hazard entirely. Constructor `LazyLock.new` is a `const fn` taking a closure.
- **Profile gating:** `app` — compile error (use context-struct, `Mutex`, `Atomic`, `#[thread_local]`, or `LazyLock`). `embedded` — permitted (MMIO justifies the feature). `gpu` — compile error. `lib` (default) — warning (`module_mut_binding`) suppressible per-binding with `#[allow(...)]`. Table added to design.md.
- **Constant-init rule (`:560`):** orthogonal — still applies. Constrains *what* can initialize; the concurrency rule constrains *where and how* it can mutate.

### B3. Integer division, modulo, and shift semantics  *(✓ RESOLVED — 2026-04-20)*

**Why foundation-breaking.** `:1097` pinned integer-overflow trap semantics; but division sign, modulo sign, `iN::MIN / -1`, `>>` on signed, and shift-by-width behavior were all unpinned. Every pick leaks into observable arithmetic.

**Resolution.** Landed in `design.md` Numeric Semantics section (new "Division, modulo, and shift" block + checked/wrapping/saturating/overflowing method-family table). Decisions:

- **`/` truncates toward zero.** `-7 / 2 == -3`. Rust/C/Java/Go/Swift alignment — it's the native CPU instruction with no correction step. Euclidean alternative is `x.div_euclid(n)`.
- **`%` takes dividend sign.** `-7 % 2 == -1`. Forced by the `(a/b)*b + (a%b) == a` identity once `/` is pinned. `x.rem_euclid(n)` for always-non-negative.
- **`iN::MIN / -1` traps as overflow.** Same error family as `iN::MAX + 1`. Division by zero is a separate trap with its own message.
- **`>>` on signed = arithmetic shift.** `(-8i32) >> 1 == -4`. Logical shift via `(x as uN) >> k`. `<<` has no sign-dependent form.
- **Shift by bit-width-or-more traps.** `(x: i32) << 32` panics. `wrapping_shl` masks modulo bit width — explicit method, no silent C/Rust-release masking.
- **`checked_*` / `wrapping_*` / `saturating_*` / `overflowing_*` method families ship in v1** on every integer primitive. Trivial wrappers over LLVM intrinsics; standard idiom (Rust/Swift/Zig); tutorial prose can reference them from day 1.
- **Profile sets default; attribute regions deferred.** `app`/`lib` = `checked`; `embedded` `unsafe` MMIO = `wrapping`. Per-function/per-block `#[checked]`/`#[wrapping]` attributes defer until layering questions (inlining boundaries, generics propagation) surface with real-user pressure. Named methods cover every non-default case explicitly.

## Group C — Lexical

### C1. Byte and byte-string literals  *(✓ RESOLVED — 2026-04-20)*

**Why foundation-breaking.** `design.md:6162` uses `b'A'` in the MMIO example, but `syntax.md § 1.3` defined no byte-literal production. Adding prefixes later risks collision with identifiers users rely on.

**Resolution.** Grammar landed in `syntax.md § 1.3`; semantics landed in `design.md` as a new `### Byte and Byte-String Literals` subsection within Strings. Decisions:

- **Prefixes reserved now:** `b` (v1 supported), `r` (raw), `br`/`rb` (raw byte — either order), `c` (C-string) — all grammatically reserved. Reserved-but-unimplemented prefixes emit a targeted "not yet implemented" diagnostic, not a generic syntax error. Prefix recognized only when immediately followed by `'` or `"` — bare identifiers named `b`, `r`, `c`, `br`, `rb` stay legal.
- **Types:** `b'A' : u8` (matches Rust); `b"hi" : [u8; N]` (owned fixed-size array with length in the type — preserves compile-time length guarantees for MMIO/protocol code). Coerces to `Slice[u8]` at call sites via normal array-slice coercion.
- **Escapes in byte literals:** `\n`, `\t`, `\r`, `\0`, `\\`, `\'`, `\"`, plus `\xHH`. `\u{...}` forbidden — lex error pointing at `\x` in diagnostic. `\x` now legal in text literals too (additive, non-breaking).
- **Grammar generalization:** single `STRING_PREFIX := "b" | "r" | "br" | "rb" | "c"` production. Future prefixes slot into the same rule. String-typed vs byte-typed escape rules split via `ESCAPE_SEQ_FOR_PREFIX`.
- **Conversion to `String`:** explicit method (`String.from_utf8(bytes)` returns `Result`), never implicit — type system keeps byte vs text distinction visible.

## Non-breaking FYI (additive-later)

These are gaps that surfaced in the audit but are safe to add post-v1 without breaking existing code. Listed here to avoid re-surfacing in a future audit.

- **`Send` / `Sync` as nameable traits.** Implicit today; referenced at `:4205`, `:4914`, `:5060`. Adding the surface later is additive.
- **`AsRef` / `AsMut` / `Borrow`.** `ref T` subsumes the common case; library ergonomics (`fn open(path: impl AsRef[Path])`) are additive.
- **Hex / octal char escapes (`\x7F`, `\0FF`).** Grammar (`syntax.md:160`) has `\u{...}` + short escapes. Adding more escapes is additive.
- **Doc-comment semantics.** Grammar defines `///` (`syntax.md:183`); spec of *what it does* (attached-to-next-item, markdown-rendered, rustdoc-style) is not pinned. Can be pinned later without breaking.
- **`From[T] for T` / `Into[T] for T` identity / reflexivity.** Purely additive.
- **Enum variant iteration (`Color.values()`).** Useful for serde / CLI enums. Tie-in with B1 above — may be worth deciding together, but *itself* is additive.
- **Attribute grammar unified spec.** `#[...]` is used extensively (`#[derive]`, `#[cfg]`, `#[repr]`, `#[with_provider]`, `#[test]`, `#[gpu]`, `#[interrupt]`, `#[link_section]`, `#[max_stack]`, `#[must_use]`, `#[allow]`, `#[checked]`) with no unified spec for parsing, values, nesting, or user-defined attributes. Additive but flag for early cleanup pass.
