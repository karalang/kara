## Q1. Float literal exponent notation

The `FLOAT` production in `docs/syntax.md ┬¦ 1.3` is:

```
FLOAT = DIGIT { DIGIT | "_" } "." DIGIT { DIGIT | "_" } [ FLOAT_SUFFIX ]
```

Scientific notation (`1e10`, `1.5e-3`, `6.022e23`) is not accepted. `design.md` does not mention exponent notation either way.

**Options:**
- **1A.** Support `e`/`E` exponents: `FLOAT = ... [ ("e" | "E") [ "+" | "-" ] DIGIT { DIGIT | "_" } ] [ FLOAT_SUFFIX ]`. Matches Rust / C / most mainstream languages. Necessary-in-practice for scientific computing, graphics, physics constants.
- **1B.** Reject exponent notation; require callers to write out the number or use `pow()`. Simpler lexer, but friction for any numeric-heavy domain.

**Lean:** 1A. Exponent notation has no syntactic ambiguity with other K─üra constructs, costs one grammar alternative, and every target audience (systems, numerical, graphics) expects it. K─üra's stated tiebreaker ŌĆö "Rust without being Rust" (semantics-Rust, syntax-mainstream) ŌĆö points the same direction.

**Action item if resolved:** Extend `FLOAT` production in `syntax.md ┬¦ 1.3` and note in `design.md` that the literal grammar follows standard IEEE-754 decimal notation with optional exponent.

## Q2. Half-open range expressions and patterns

`syntax.md ┬¦ 5.14` defines:

```
RANGE = EXPR ".." EXPR        // exclusive end
      | EXPR "..=" EXPR       // inclusive end
```

Both endpoints are required. `design.md:1115` likewise shows only `lo..hi` and `lo..=hi`. There is no syntax for:

- `0..` (from 0 to the end / unbounded above)
- `..5` (from the start up to, excluding, 5)
- `..=5` (from the start up to and including 5)
- `..` (the full range)

These appear naturally in slicing (`items[2..]`, `items[..n]`) and half-open iteration.

**Options:**
- **2A.** Support all four half-open forms for both expressions and (where total-ordered) patterns. Pattern-side already restricts to integer/char ŌĆö half-open patterns like `n @ 0..` (non-negative) and `..=-1` (negative) fit cleanly. Matches Rust.
- **2B.** Support only in expression position (for slicing). Disallow in patterns ŌĆö exhaustiveness checking already punts to wildcards anyway.
- **2C.** Support only `expr..` and `..expr` in slice indexing as a special case; no first-class open-ended `Range` types. Cheapest, but special-cased.
- **2D.** Keep status quo: require both endpoints. Callers write `0..items.len()` etc.

**Lean:** 2A. Once the lexer/parser handles `..` and `..=` as operators in binary position, extending to unary prefix/postfix is straightforward. The `Range` / `RangeFrom` / `RangeTo` / `RangeFull` / `RangeInclusive` / `RangeToInclusive` library split is the main cost, and Rust has already demonstrated it's absorbable.

**Action item if resolved:** Extend `RANGE` production in `syntax.md ┬¦ 5.14`, extend range-pattern rules in `┬¦ 5.6`, and document the library types in `design.md`.

## Q3. Trait-level associated functions (no `self` receiver)

`syntax.md ┬¦ 3.4` defines:

```
TRAIT_METHOD = "fn" IDENT [ GENERIC_PARAMS ] "(" METHOD_PARAMS ")" ... ";"
             | "fn" IDENT [ GENERIC_PARAMS ] "(" METHOD_PARAMS ")" ... BLOCK
METHOD_PARAMS = SELF_PARAM [ "," PARAM_LIST ]
```

Every trait method is required to take `self` / `ref self` / `mut ref self`. There is no way to declare trait-level associated functions like `Default.default() -> Self` or `FromStr.from_str(s: StringSlice) -> Result[Self, Error]`.

Inherent `impl` blocks *can* have associated functions without `self` ŌĆö e.g., `Vec.new()`, `Map.new()` appear throughout `design.md` ŌĆö but these are inherent, not trait-dispatched.

**Options:**
- **3A.** Allow trait methods to omit the receiver: `TRAIT_METHOD` takes `PARAM_LIST` (which no longer requires `SELF_PARAM`). Enables `trait Default { fn default() -> Self; }`. Matches Rust.
- **3B.** Keep the ban; require users to define a free function or an extension trait that *does* take `self`. Loses the ergonomics of `Default::default()`-style factories and blocks common constructor traits (`FromStr`, `Default`, `TryFrom`'s associated-function form).
- **3C.** Allow them but only with a default body (so there is always a dispatch target without a receiver). Half-measure, not clearly motivated.

**Lean:** 3A. The block is purely grammatical ŌĆö there is no semantic reason a trait cannot contain a static factory, and several stdlib traits need it. Dispatch on a no-receiver trait method is resolved via the caller specifying the type (e.g., `T.default()` where `T: Default`), which parses the same way as inherent-method `.`-access because K─üra's `.`-based UFCS already handles type paths.

**Action item if resolved:** Relax `TRAIT_METHOD` in `syntax.md ┬¦ 3.4` to either `METHOD_PARAMS` or `PARAM_LIST`, add a short design.md note on trait associated-fn dispatch (`T.method(args)` when `T: Trait`).

## Q4. Explicit dereference operator (`*expr`)

`syntax.md ┬¦ 1.2` lists unary operators as `!`, `-`, `~`. No `*` prefix. `syntax.md ┬¦ 5.1` matches. Yet `syntax.md` line 310 (the `binary_search` example) writes `*needle` where `needle: ref T`:

```kara
ensures(result) match result { Some(i) => haystack[i] == *needle, None => true }
```

K─üra has auto-deref for method calls and field access (`.`-through-`ref`), but no specified operator for "give me the value behind this reference" in an expression context.

**Options:**
- **4A.** Add `*expr` as a prefix operator that dereferences `ref T` / `mut ref T` to `T` (with the usual borrow/move rules on the result). Matches Rust. Necessary for value-level operations on refs ŌĆö comparison (`a == *b`), assignment through refs (`*r = value`), arithmetic on ref-wrapped numerics.
- **4B.** No explicit deref. Rely entirely on auto-deref via method calls (`ref_val.eq(&other)` or similar) and fix the `*needle` example. Cleaner surface, but forces users into method-call syntax for what is fundamentally a primitive operation. Assignment-through-ref (`*r = v`) becomes method-call soup.
- **4C.** Introduce a `.deref()` method / `.*` postfix instead of `*` prefix. Keeps the unary-operator slot open for future use. Postfix `.*` (Zig-style) reads left-to-right with chains; downside is unfamiliar to Rust/C/C++ audiences.

**Lean:** 4A. `ref T` is part of the core type system; it needs a primitive syntactic form for unwrapping that doesn't go through a trait. Using `*` is the zero-surprise choice given the audience. Conflict with the multiplication operator is resolved the same way Rust resolves it (unary vs binary position is unambiguous at parse time).

**Action item if resolved:** Add `*` to the unary operator row in `syntax.md ┬¦ 5.1`, add a `"*" EXPR` alternative to the unary-expression grammar, add a `PLACE_EXPR` rule including `"*" EXPR` so `*r = v` is a legal assignment target, document the semantics briefly in `design.md` alongside `ref T`.
