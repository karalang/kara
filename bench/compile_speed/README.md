# compile_speed — cold compile-elapsed CI gate

**Status:** Implemented, 2026-07-22 (scaffolded 2026-05-20). Graduated from [`brainstorming/archive/v69_go_parity_gaps.md § Gap 1`](../../brainstorming/archive/v69_go_parity_gaps.md); deliverables per [`docs/roadmap.md § Phase 8.5 Track 5`](../../docs/roadmap.md).

Tracks cold compile-elapsed time for `karac build` against `rustc -O` (peer baseline). The PR-trigger workflow ([`compile-speed.yml`](../../.github/workflows/compile-speed.yml)) gates on the **karac/rustc ratio** per workload vs `baseline.json` and fails on >30% ratio regression (the initial threshold; tightens toward ≤5% as data accumulates). `clang -O2` stays opportunistic and `go build` a launch-time reference — neither is wired into the gate.

## Corpus

| Member | Source | Shape | Status |
|---|---|---|---|
| `seed_kata.{kara,rs}` | Plain copy of `kara-katas/leetcode/1-100/1-two-sum/bench/hash_map.{kara,rs}` | ~40 LOC, algorithmic (Map + Slice + match) | ✅ in corpus |
| `synthetic.{kara,rs}` | Generated by [`gen_synthetic.py`](gen_synthetic.py) (deterministic, seed in file) | **~10.9K LOC**, 100 clusters: small fns in a linked call graph, generics with trait bounds, 34 traits × 100 impls, enums + match, declared effects on pub fns, Vec + closure bodies | ✅ in corpus (v1-required) |
| Backend-shape kata | `kara-katas` backend-service kata (PLAN.md priority #1) | ~500–1000 LOC, generics + IO + concurrency | ⏳ copies in when it lands |

Each member is a `.kara`/`.rs` **twin pair** with identical arithmetic (non-negative, mod-bounded — overflow-checked Kāra and release rustc agree), and `bench.sh` runs a **twin-equivalence oracle** on the synthetic pair before benching: both binaries must print the same checksum, so a miscompile can't hide under a good compile time. The seed kata is compile-only here (its `main` is a runtime benchmark).

Regenerating the synthetic: `python3 gen_synthetic.py` (commit both outputs; the generator is deterministic).

## Reproduction

```bash
# Prerequisites (one-time)
cargo build --release --features llvm --bin karac
cargo rustc -p karac-runtime --release --crate-type staticlib   # karac build links it
cargo install hyperfine    # or: apt-get install hyperfine / brew install hyperfine

# Run
./bench.sh                 # oracle + hyperfine tables + merged latest.json
python3 compare.py --latest latest.json --baseline baseline.json   # the gate, locally
```

`bench.sh` guards the llvm-feature footgun (a `karac` built without `--features llvm` silently type-checks instead of building — it probes for the `Built:` line before measuring), deletes artifacts before every run (`--prepare`, full cold compiles), and follows the short/long-workload hyperfine discipline from [`../README.md`](../README.md): 30 runs for the sub-100ms seed kata, 10 runs for the multi-second synthetic.

## CI gate

One workflow, two legs ([`.github/workflows/compile-speed.yml`](../../.github/workflows/compile-speed.yml)):

- **PR leg**: runs `bench.sh`, then `compare.py --baseline baseline.json`. Fails if any workload's karac/rustc **ratio** regressed >30% over baseline. Posts a sticky PR comment (same-repo PRs) + step summary with per-workload times, ratios, and deltas.
- **Main leg**: re-measures after each merge and rewrites `baseline.json` only when a ratio drifted >5% (no churn). `main` is protected (required status checks), so the refresh rides an automated PR from `bench/compile-speed-baseline` — an occasional one-click merge. The merge commit touches only `baseline.json`, which is `paths-ignore`d, so it can't retrigger the main leg. Every run also uploads `latest.json`/`gate.md` as a workflow artifact.

The gate is on the **ratio**, not absolute karac seconds: both compilers shift together across runner generations and machines, so the ratio is the stable quantity (this also makes a locally-run `compare.py` meaningful against the CI-measured baseline).

`baseline.json` bootstraps empty: the first main-leg run writes it. While empty, the PR leg passes vacuously with a note.

## First measurements (2026-07-22, container x86-64 — indicative; the gate baseline is CI-measured)

| workload | karac build | rustc -O | ratio |
|---|---|---|---|
| seed_kata (~40 LOC) | 229 ms | 203 ms | **1.13×** |
| synthetic (~10.9K LOC) | 5.90 s | 2.12 s | **2.79×** |

Split on the synthetic: `karac check` (parse → desugar → resolve → typecheck → effectcheck → ownership) is **0.56 s** of the 5.90 s — the front end handles 10K LOC in ~half a second, and ~90% of cold build time is LLVM codegen + linking. That's where any future compile-speed engineering should aim; the ratio gate keeps it from silently getting worse meanwhile.

## Threshold trajectory

30% initial (catches order-of-magnitude blowups — the 2026-05-12 Maranget O(N²) shape — without flaking on routine work) → tighten stepwise toward ≤5% as baseline variance data accumulates. The current value lives in `compare.py --threshold` / the workflow; this doc commits to the trajectory.
