# wasm_size — WASM module size: Kāra vs Rust vs TinyGo

On WASM the headline metric is **module size** — every byte is downloaded, the
way startup density is the headline on native. This track sizes the same two
programs compiled three ways and records the numbers, so the WASM positioning
rests on a measured receipt rather than a hypothesis.

**Result:** [`sizes.md`](sizes.md) (human) · [`sizes.json`](sizes.json) (machine).
Both are regenerated by [`bench.sh`](bench.sh) — never hand-edited.

## What it measures

Two workloads, two artifact flavors, three toolchains:

- **Workloads** — `hello` (a minimal stdout program) and `filter_core` (the real
  [Iris](../../examples/iris) image kernel: six filters over a 512×384 procedural
  image, integer math).
- **Flavors** — **core module** (`wasm32-wasip1`, the directly-comparable raw
  `.wasm` and the byte-shape Kāra's `--target=wasm_browser` ships) and
  **component** (WASI 0.2 / `wasm32-wasip2`, the modern embedded-WIT artifact and
  Kāra's `wasm_wasi` default).
- **Toolchains** — Kāra (`karac`), Rust (`cargo`, size-tuned release profile),
  TinyGo (`-opt=z -no-debug`).

For each cell it records three numbers: `raw` (build output), `stripped`
(`wasm-tools strip`, DWARF/custom-section removal), and `brotli` (`brotli -q 11`
of the stripped module — the real download cost). The headline is `brotli`.

## Honest-comparison discipline

The point of this track is a defensible number, so two guards keep it honest:

1. **Same kernel, three ways.** `filter_core` is a line-for-line port across the
   three languages (`src/kara`, `src/rust`, `src/tinygo`), all computing the
   identical integer kernel. `bench.sh` runs each core-module `filter_core` under
   `node:wasi` and asserts all three print the **same FNV-1a checksums** (the
   byte-exact values from the Iris native oracle). A mismatch fails the bench —
   you cannot accidentally compare a stripped-down stand-in against a full one.
2. **Like-for-like flavors.** Core modules are compared only against core
   modules, components against components — never a raw module against a
   component (the WASI preview1→preview2 adapter is ~tens of KB and would skew
   any cross-flavor row).

Each toolchain uses its idiomatic size-conscious release config (documented in
`sizes.md` and in the build flags in `bench.sh`), i.e. what a developer shipping
that toolchain to WASM would actually use.

## Reproduction

```bash
./bench.sh        # builds all 12 cells, cross-checks checksums, writes sizes.{md,json}
```

Requires `karac` (from this checkout, `--features llvm`), `cargo` + the
`wasm32-wasip1` / `wasm32-wasip2` targets (`rustup target add …`), `tinygo`
(`brew install tinygo-org/tools/tinygo`), `wasm-tools`, and `brotli`; `node` is
optional and used only for the checksum cross-check. Missing tools fail loudly
with the install command.

## Gate shape

**Not a CI gate** — a credibility receipt. WASM module size is not a launch
*claim* yet, so there is no `baseline.json` regression gate here (unlike
`compile_speed/`). If size becomes a published claim, add a `baseline.json` + a
PR-trigger workflow per the [parent README](../README.md) "Adding a new track"
protocol. Until then this is a checked-in measurement, refreshed by re-running
`bench.sh` after a toolchain bump.

## Known caveat

Kāra's **component** emission is not yet byte-deterministic (same length,
reordered content across rebuilds), so the Kāra-component `brotli` figure jitters
~0.2% run to run. Core-module figures are fully deterministic. Tracked as a
compiler follow-up; see the caveat note in `sizes.md`.
