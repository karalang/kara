# WASM module size — Kāra vs Rust vs TinyGo

> Regenerated by [`bench.sh`](bench.sh) — **do not hand-edit**. Sizes in bytes.
> `brotli` = `brotli -q 11` of the `wasm-tools strip`-ed module: the real
> download cost, since every byte of a wasm module is fetched. The headline
> metric on WASM is module size, the way startup density is the headline on native.

All three `filter_core` ports print **identical FNV-1a checksums** (asserted in
`bench.sh`), so this compares the *same* kernel three ways — an honest size
comparison, not a stripped-down stand-in. Workloads: `hello` (minimal stdout
program) and `filter_core` (the [Iris](../../examples/iris) image kernel —
6 filters over a 512×384 procedural image).

## Core module (`wasm32-wasip1`)

The directly-comparable axis: a raw core `.wasm` all three toolchains emit
natively. This is also the byte-for-byte shape Kāra's `--target=wasm_browser`
ships (the browser module is the same wasip1 command module behind a JS glue
polyfill). Kāra: `--bindings none`; Rust: `--target wasm32-wasip1` (release,
`opt-level="z"`, lto, `panic="abort"`, strip); TinyGo: `-target=wasi -opt=z -no-debug`.

| workload | toolchain | raw | stripped | brotli |
|---|---|---:|---:|---:|
| hello | Kāra | 3,540 | 3,540 | **1,579** |
| hello | Rust | 40,115 | 40,115 | **13,678** |
| hello | TinyGo | 162,034 | 161,722 | **58,334** |
| filter_core | Kāra | 39,180 | 39,180 | **13,600** |
| filter_core | Rust | 43,116 | 43,116 | **14,676** |
| filter_core | TinyGo | 188,845 | 188,533 | **65,672** |

## Component (WASI 0.2 / `wasm32-wasip2`)

The modern artifact — a single embedded-WIT component. This is Kāra's
`--target=wasm_wasi` **default** (`--bindings component`). Rust:
`--target wasm32-wasip2`; TinyGo: `-target=wasip2`.

| workload | toolchain | raw | stripped | brotli |
|---|---|---:|---:|---:|
| hello | Kāra | 37,679 | 37,679 | **12,736** |
| hello | Rust | 53,635 | 50,105 | **16,291** |
| hello | TinyGo | 321,843 | 315,750 | **100,099** |
| filter_core | Kāra | 60,866 | 60,866 | **20,863** |
| filter_core | Rust | 56,714 | 53,184 | **17,361** |
| filter_core | TinyGo | 349,270 | 343,177 | **107,277** |

> _Caveat: Kāra's component emission is not yet byte-deterministic — repeated
> builds of identical source produce the same byte **length** but reordered
> content, so the Kāra-component `brotli` cell jitters ~0.2% (~50 B) run to run
> (raw/stripped are stable). Tracked as a follow-up; the core-module figures
> above are fully deterministic._

## Reading the numbers

- **Core modules:** Kāra's AOT/no-GC/no-async posture (the wasm archive strips
  tokio/mio/socket2; `wasm-tools strip` removes DWARF by default; wasm-ld
  section-GC = native `-dead_strip`) lands module size in the same class as
  size-tuned Rust and far below TinyGo (which ships a GC + runtime). On
  `hello` Kāra is markedly smaller than Rust — Rust's `println!` pulls in std
  fmt + the panic/backtrace machinery that Kāra's lean runtime does not.
- **Components:** the WASI preview1→preview2 adapter inflates every toolchain's
  component over its core module. Kāra's adapter is heavier than Rust's, so on
  the larger `filter_core` workload Rust's component edges ahead of Kāra's
  (Kāra still wins `hello` and both core-module rows). Shrinking the embedded
  adapter is the open size follow-up; the core-module number is the one Kāra's
  `--target=wasm_browser` positioning actually ships.

_Toolchains: karac 0.1.0, rustc 1.95.0, tinygo 0.41.1. Regenerate with `./bench.sh`._
