  *Sub-steps.*

  *Hard-stop triggers.*
  - The effect-checker doesn't auto-seed `writes(Env)` for stdlib registrations the way it does for other resource verbs — would need design discussion.
  - `std::env::set_var` flagged unsafe in current Rust toolchain due to multi-threaded modification semantics; if this surfaces as a compile-time issue, halt.

  *Deviations from plan.* Two honest tags: (1) **Effect-checker seeding required two keys, not zero.** Plan said "rejection at non-`writes(Env)`-declaring callers should fire automatically through existing infra"; in practice neither `Env.var` nor `env.var` are seeded today (the existing `with reads(Env)` typecheck tests pass vacuously — there is no enforcement on the existing surface). Adding the seed for the new `env.set` / `Env.set` keys is one explicit `for fn_name in ["Env.set", "env.set"]` block mirroring the `Client.get` / `Client.post` Network seeding three blocks up; it is the minimum to make the negative test (a real `MissingEffectDeclaration` error fires when a `pub fn` calls `env.set` without declaring `writes(Env)`) work. (2) **MethodCall lowercase-alias call-key emission.** The capitalized form `Env.set(...)` parses as `Call(Path([Env, set]))` and routes through `extract_callee_name` → `Env.set` automatically. The lowercase form `env.set(...)` parses as `MethodCall { object: Identifier("env"), method: "set" }` because `starts_upper("env") == false` keeps it out of the parser's `Path`-rooted branch — so the call-collection loop in `collect_calls_in_expr`'s `MethodCall` arm needed an explicit lowercase-alias check (peer to the typechecker's `infer_method_call` lowercase-module branch and the interpreter's `eval_method_call` alias map) emitting `Env.<method>` as the call key. The added check is conservative — fires only when the receiver is the literal identifier `env` — so it cannot create false positives for user code. Both deviations are structural-not-semantic: same effect-resource verbs, same call-key shape, just an explicit registration site where the plan assumed implicit propagation.

  *Deviations from plan.* One inline-fix scope expansion: the interpreter's baked-stdlib registration walk only handled `Item::ImplBlock`, not `Item::EnumDef`. With baked enums un-walked, `let e = IoError.NotFound` (and any other path-style enum-variant *expression*) hit the `path '...' not found at ...:...; should be caught by resolver` `unreachable!` in `eval_path` because no `IoError.NotFound` value was bound in the interpreter env. Match-arm patterns worked already (the resolver tags variant patterns structurally), which is why nothing tripped earlier — every prior test using `IoError.NotFound` did so only in pattern position (e.g., `tests/interpreter.rs::test_filesystem_read_nonexistent_file_returns_err`). The pre-existing `Ordering` / `MemoryOrdering` enums dodged the issue via hand-written `register_items` blocks that pre-bind every unit variant. The fix in this slice generalizes that pattern: extend the baked-stdlib walk in `Interpreter::register_items` (the same `for item in &baked_items` loop that already calls `register_impl_methods` for `Item::ImplBlock`) to also enumerate `Item::EnumDef` and bind each unit variant under its qualified `Type.Variant` path. Tuple/struct variants are handled at call sites just like user-program enums, so they need no special-casing here. With the fix in place, every baked-stdlib enum (current: `Option`, `Result`, `Ordering`, `MemoryOrdering`, `IoError`, `VarError`, `Entry`, `RegexError`, `HttpError`, `DecodeError`) supports path-style construction at the use site. The hand-written `Ordering.{Less,Equal,Greater}` and `MemoryOrdering.{Relaxed,Acquire,Release,AcqRel,SeqCst}` registrations in `register_items` are now redundant on net, but kept in place this slice — removing them is a peer cleanup, not the slice's scope.

  *Out of scope (still open).* `impl From` for other I/O error pairs (network errors, fs errors) — only `VarError → IoError` is named in the deferred-portion sentence; reverse direction (`From[IoError] for VarError`) — not meaningful, one-way conversion only; the redundant hand-written `Ordering`/`MemoryOrdering` variant registrations in `Interpreter::register_items` — peer-to-this-slice cleanup, deferred so this slice stays single-commit.

- [x] **Lexer: reserve `expr_<NNNN>` fragment-specifier identifier namespace.** Closed 2026-05-04 in commit `1559772`. Raw-escape exemption is structural — `r#expr_2026` flows through `raw_identifier()`, which never enters the keyword match path, so no `was_raw_escaped` flag is needed at the token level. 7 tests in `tests/lexer.rs` cover boundary years (2020 / 2026 / 2099) at let / fn / struct-field positions, both fix-its in the diagnostic, the raw-escape exemption, and every accept-path negative case (no prefix, different prefix, non-year suffix, year out of range on both sides, wrong digit count). Original spec preserved below for the historical record. Spec at `design.md § Reserved Fragment-Specifier Identifier Namespace` (v60 item 62). Slices: (1) **Lexer regex check.** In `src/lexer.rs`, after recognising an identifier token, run a pattern check against `^expr_(\d{4})$`. If the regex matches AND the captured digit string parses as a year in the range `2020`..=`2099` inclusive, emit `error[E_RESERVED_FRAGMENT_SPECIFIER_NAMESPACE]: 'expr_<NNNN>' is a reserved identifier name; this naming convention is reserved for future edition-versionable syntax categories in macros / comptime fragment specifiers — use 'r#expr_<NNNN>' if you need this exact identifier today, or rename to a non-year-suffixed form` with the offending span underlined and the two fix-its (the raw-identifier escape OR a rename suggestion). (2) **Raw-identifier exemption.** When the identifier is preceded by `r#` (raw-identifier escape per item 39), the reservation does NOT fire — `r#expr_2026` parses as the bare identifier `expr_2026` and is accepted. The exemption falls out of the existing raw-identifier handling: the regex is checked AFTER the raw-prefix has been stripped, so `r#expr_2026` strips to `expr_2026` for downstream consumers but the reservation is checked against the *original lexed form* before stripping. Implementation detail: track a `was_raw_escaped` flag on the identifier token; if true, skip the namespace check. (3) **Narrow scope at v1.** The reservation matches ONLY the `expr_` prefix. Other plausible fragment-specifier prefixes (`pat`, `ty`, `stmt`, `block`, `tt`, `ident`, `lit`, `path`, `vis`, `meta`, `item`, `lifetime`) are NOT reserved at v1 — extending the reservation post-v1 to additional prefixes is non-breaking-additive (existing user identifiers using those prefixes continue to work). (4) **Year range.** `2020`..=`2099` inclusive. Years outside this range (e.g., `expr_1999`, `expr_3000`) parse as ordinary identifiers without rejection. - [ ] **Comptime — AST→AST `comptime fn` (post-v1).** Spec lives at [`deferred.md § Comptime — AST→AST comptime fn`](deferred.md#comptime--asttoast-comptime-fn) (v60 item 31, decided 2026-05-02). Four discrete implementation substrates, all post-v1: (1) **Comptime evaluator** — treewalk interpreter over the typed AST, retargeted from the existing Phase 4 interpreter to compile-time invocation; covers value-level computation in `comptime fn` bodies excluding AST/type-as-value work. (2) **`Type` as first-class value + reflection API** — expose the compiler's existing type registry as immutable `Type` values; field/variant/method iteration; size/align queries; the `Type` pseudotype is comptime-only (runtime references emit `E_TYPE_VALUE_AT_RUNTIME`). (3) **AST builder + emission** — stdlib `compiler.ast` module with typed builder API (`ast.expr`, `ast.method`, `ast.impl_block`, etc.) plus quasi-quote shim (`ast.parse_expr(string)`); module-level splicing rules for items returned from comptime fns. (4) **Derive desugaring** — `#[derive(Trait)]` resolves to a `derive_<Trait>` `comptime fn` lookup; built-in derives (`Eq`, `Hash`, `Display`, `Debug`, `Clone`, `Copy`, `PartialEq`, `PartialOrd`, `Ord`, `Arithmetic`, `Serialize`, `Deserialize`) are stdlib `comptime fn`s with no special compiler treatment beyond the lookup convention; user-defined derives use the same mechanism. Each substrate is internally complete on its own — partial deployment never leaks a half-built feature into the v1 language. Effect-system wiring (already pinned in v1 — design.md § Comptime Effect Defaults): `reads(CompileTimeEnv)` / `writes(CompileTimeEnv)` / `allocates(CompileTimeHeap)` permitted; `panics` becomes a build-time diagnostic; runtime resource verbs forbidden via `E_RUNTIME_EFFECT_AT_COMPTIME`. Resource limits (configurable via CLI flags): 2^24 instructions, 512 MiB CompileTimeHeap, 1024 frames recursion. Tooling integration: `karac doc` documents comptime fns with a "comptime" badge; `karac explain --expand <span>` prints post-expansion AST; `karac` debugger sees comptime frames as a dedicated stack; `karac query monomorphization` tracks comptime invocations alongside type/effect tuples. (Post-v1.)
