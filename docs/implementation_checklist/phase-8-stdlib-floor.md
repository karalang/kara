### Backend Platform (v64-lifted)

> Lifted from Phase 11 under the v64 backend-first decision (2026-05-09). Spec at [`design.md ¬ß v1 Positioning ‚Äî Backend-First`](../design.md#v1-positioning--backend-first); roadmap pointer at [`roadmap.md ¬ß Phase 8 > Backend Platform`](../roadmap.md#backend-platform-v64-lifted).

- [ ] **P0 ‚Äî `std.http` (HTTP/1.1 server + client) ‚Äî stable v1 surface + `#[unstable]`-gated extensions.** Stable surface: `Server::bind` / `Server::serve` / `Request` / `Response` / `Client::get` / `Client::post`, body streaming, header manipulation, basic routing, keep-alive, chunked transfer, Host routing. Advanced surface (gated `#[unstable]` per [syntax.md ¬ß 8](../syntax.md#8-attributes)): connection-level customization, custom transport, low-level frame access. Pre-lock audit of API shape against Go `net/http`, Rust `hyper` + `axum`, Node `http` for known footguns (Go middleware composition, Rust body-ownership in handlers, Node error propagation through promise chains). Effects: `sends(Network)` / `receives(Network)` on every call into the network surface; routes through the event loop via Phase 6.3.
- [ ] **P0 ‚Äî TLS ‚Äî vendored rustls + aws-lc-rs default crypto provider.** `std.tls` API: cross-platform server + client surface; modern-TLS-only (no SSLv3, no insecure ciphers). rustls-provider plug points private at v1 (no public crypto-provider extension API, revisited at v2 if FIPS / post-quantum forces it). Audit posture: rustls + aws-lc-rs already audited upstream, but the FFI binding layer + `std.tls` API + verification callbacks + certificate-chain handling + error-mode coverage are new code and need their own audit pass before v1 ship.
- [ ] **P0 ‚Äî WebSocket ‚Äî RFC 6455.** Server-side framing (continuation, ping/pong, close), handshake, message/binary distinction, fragmentation, masking. Built on `std.http` upgrade handshake. The canonical idle-keep-alive workload ‚Äî Demo 1 (1M+ idle connections) is shaped around this surface. Effects: same `sends(Network)` / `receives(Network)` as `std.http`.
- [ ] **P0 ‚Äî `karac new <name>` default project template ‚Äî backend HTTP server skeleton.** Default scaffolding: `std.http` + `std.tracing` + `/health` endpoint + `/ws` WebSocket route. `--lib` for libraries, `--cli` for CLI tools, `--data` for data-pipeline scaffolding (Kafka consumer + processor + sink shape). Default-being-backend reinforces the positioning at the friction-zero entry point.
- [ ] **P1 ‚Äî HTTP/2 ‚Äî multiplexed streams + flow control.** HPACK header compression, stream prioritization, server push (default-off), `Server-Sent Events` interop. Required for gRPC (which is post-v1; see [`deferred.md ¬ß gRPC`](../deferred.md#grpc-streaming-reflection-server--client)). Sequenced after HTTP/1.1 in implementation order; ships at v1 launch.
- [ ] **P1 ‚Äî `std.tracing` ‚Äî structured logging + span/trace context propagation, OTel-export-ready.** Minimum surface: span creation + nesting, structured fields on spans, trace-context propagation across async boundaries (state-machine transform preserves the active span), exporter trait (default no-op exporter at v1; OTel exporter ships post-v1 as a community library on the same trait). Comptime-driven trace-context plumbing ‚Äî KƒÅra-native opportunity, no proc-macro indirection like Rust's `tracing`.

  - [x] `Exporter` trait + `NoOpExporter` default impl; verified user `impl Exporter for X` dispatches correctly (`test_tracing_user_can_implement_exporter_trait`) ‚Äî commit `1d172e9`

- [ ] **`OnceLock[T]` and `OnceCell[T]` lazy-init primitives.** Spec at `design.md ¬ß Module-Level Bindings > LazyLock[T] / OnceLock[T] / OnceCell[T]` (v60 item 45). Document the laptop number as "expected upper-bound ceiling on thermally-constrained hardware."

  - **The `is_trivial` boolean granularity gate is the right v1 abstraction** but loses signal at the cost-model level. v1.x cost-model spec should likely refine to per-branch cost estimate (`branch_cost: Option<Duration>` based on inferred-effect classification) rather than the binary `is_trivial`.

  - **Stdout contention is *not* a meaningful auto-par overhead source** at typical busy-compute / println balances. Initial hypothesis ruled out by the 500M-iter test (efficiency *dropped* rather than improved). The v1.x cost-model spec should ignore stdout-write effects in granularity heuristics unless the busy-compute kernel is unusually small relative to expected per-write overhead.

  **Spec status ‚Äî locked.** `design.md ¬ß Provider-Rooted Resources` already pins the rule (clarified during slice 6 follow-up reframing 2026-05-08): `R.method(args)` desugars to `Arc<dyn R::Provider>` lookup in the per-task provider stack followed by a vtable call; `with_provider[R](p, ||...)` pushes/pops a frame; parameterized instances `R[k]` always route through the base-resource binding. The interpreter implements this at `src/interpreter.rs::eval_resource_method` (line 7146). The remaining work is **codegen-only** ‚Äî the implementation, not the spec. The earlier "needs design discussion" framing in this entry referred to the spec gap that is no longer open; what remains is **implementation-design choice** on the codegen ABI, scoped in the slice plan below.

  **Codegen ABI design choices** (all decidable from existing constraints; no further spec discussion needed before slice plan execution):
  - **(a) Per-task provider stack representation.** Two natural shapes ‚Äî (i) thread-local `Vec<HashMap<ResourceId, Arc<dyn Any>>>` mirroring the interpreter, accessed through a runtime helper; (ii) frame-allocated linked list of `(resource_id, provider_arc)` cells, with the head pointer carried in `KaracFrame` (slice 4) so the existing per-thread frame plumbing extends naturally. **Recommend (ii)**: avoids `thread_local!` bookkeeping on Apple/Linux platforms with cheap-but-not-free TLS, and reuses the slice 4 KaracFrame infrastructure already wired into `karac_par_run`. Lookup is O(stack-depth √ó resource-count); typical depths are 1-3, so linear scan is fine.
  - **(b) Vtable layout.** Trait-object support is *new* ‚Äî KƒÅra v1 has no other trait-object lowering site. Two options ‚Äî (i) per-trait single-dispatch vtable (data ptr + fn ptrs), classic Rust shape; (ii) per-method-call lookup table keyed by `(resource_id, method_name) ‚Üí fn_ptr`, sidestepping vtable construction entirely. **Recommend (i)**: the v1 cost model already names "thin Arc deref + one vtable indirection" at `design.md ¬ß Resource call desugaring`; option (ii) adds a hash lookup per call which is observably slower and not what the spec promises. Vtable construction at `with_provider` call site is a one-time cost; subsequent calls are pointer-chases.
  - **(c) Arc-sharing on spawn.** `karac_par_run` (slice 2 + 4) already passes branch closures captured environments into worker threads. The provider stack head pointer needs to flow through that capture so spawned tasks see the parent's bindings. Implementation: add a `provider_stack_head: *const ProviderFrame` field to the env-struct emitted by `compile_par_block` / `emit_par_run`, populated from the calling thread's KaracFrame at par-block entry; each worker thread initializes its KaracFrame with this head when entering the branch fn. Arc clones happen lazily at first use, not eagerly at spawn (matches the cost note at `design.md:7078`).
  - **(d) Parameterized-resource binding.** `design.md ¬ß with_provider and parameterized resources` (line 7101) locks the rule: `Db["users"]`, `Db["orders"]` all key on the base resource `Db`. Codegen already has access to the resource's base name via `effectchecker.rs` resource-ID interning. The lookup uses base name only; the parameter is passed through to the provider's method as a normal argument. No new design.
  - **(e) Vtable storage on `with_provider` call.** When `with_provider[R](provider, ||body)` runs, we need the provider's vtable for trait `R::Provider` to be addressable. Option (i): emit a per-`(provider_type, trait)` static vtable global at codegen time, referenced by `with_provider`'s frame push. Option (ii): build the vtable on the fly at the `with_provider` call site. **Recommend (i)** ‚Äî vtables are statically known per impl; static globals avoid per-call construction overhead and match the standard Rust trait-object model.
  - **(f) Resource ID assignment.** `effectchecker.rs` already interns resource names to stable IDs for conflict analysis. Codegen reuses these IDs as the provider-stack lookup key ‚Äî no new ID space.

(3) Typechecker ‚Äî three new checks, all at the enum-decl site after const-eval has folded each `discriminant` to an integer literal: (a) **All-or-nothing rule.** Walk the variant list; if some variants have `discriminant.is_some()` and some don't, emit `error[E_PARTIAL_EXPLICIT_DISCRIMINANTS]: enum 'NAME' mixes explicit and implicit discriminants; declare a value on every variant or remove the explicit values and rely on declaration order` with the first explicit + first implicit variant spans labeled. (b) **Range check per `#[repr(intN)]`.** When `#[repr(intN)]` is present, every explicit value must fit the repr type's range ‚Äî otherwise `error[E_DISCRIMINANT_OUT_OF_RANGE]: discriminant 'N' on variant 'V' does not fit in 'intN' (range '[MIN, MAX]')` with the offending value's span. Under `#[repr(C)]` without an `#[repr(intN)]` companion, the range is the platform's `c_int` (signed `i32` on every supported target at v1; the diagnostic should name `c_int` rather than `i32` to make the C-ABI-driven choice explicit to the reader). (c) **Duplicate-value rejection.** Build a `HashMap<i64, Span>` over the resolved values; on any insert that collides, emit `error[E_DUPLICATE_DISCRIMINANT]: variant 'V2' has the same discriminant value 'N' as variant 'V1'` with both spans labeled. The diagnostic uses the *resolved* integer value so const-expression aliasing (e.g., two variants pointing at the same module-level constant) is caught. (d) **`#[repr]` requirement on payload variants.** If the enum has at least one variant with both a payload (`VariantKind::Struct` or `VariantKind::Tuple`) and a `discriminant`, and the enum's `#[repr]` set does not contain `#[repr(intN)]` or `#[repr(C)]`, emit `error[E_PAYLOAD_DISCRIMINANT_REQUIRES_REPR]: explicit discriminants on payload variants require '#[repr(intN)]' or '#[repr(C)]'; without one, the discriminant location is unspecified and the value commitment is unreachable` at the first offending payload-variant's discriminant span. The check rejects `#[repr(transparent)]`, `#[repr(packed)]`, `#[repr(align(N))]` as repr forms that satisfy the requirement (none commits to a discriminant position). Field-less variants with `= N` under a `#[repr]`-less enum continue to be legal per the existing C-like-enum rule. (4) Runtime-surface preservation. The existing payload-enum exclusion from `.discriminant()` / `TryFrom[intN]` / `.values()` continues to apply unchanged ‚Äî explicit discriminants on payload variants do not lift the exclusion. Calls to `.discriminant()` on a payload enum (with or without explicit discriminants) continue to fail with the existing diagnostic that points at the manual-`tag` workaround; the design.md cross-reference now mentions the explicit-discriminant declaration form as the source of the literal values the manual `tag` method's match arms reference. (5) Effect-checker / ownership-checker integration ‚Äî none. The discriminant declarations are pure data attached to variant decls and contribute no effects, no consume tracking, no reference flow. (6) Codegen ‚Äî the explicit values on payload variants are *declarations*, not layout commitments. At v1 the LLVM lowering of payload enums is unchanged: the compiler may still elide tags via niche optimization, store the discriminant non-contiguously, and otherwise treat the variant set as it does today. The declarations exist for documentation, semver-lock, and as the literal values a hand-written `tag` method's arms reference. Field-less explicit-discriminant enums under `#[repr(intN)]` continue to lower with the declared integer at the runtime-surface position, exactly as today. Test coverage: positive ‚Äî `#[repr(u8)] enum Op { Reset = 0x01, Connect { addr: u32 } = 0x05, Send(Bytes) = 0x06, Disconnect = 0xFF }` parses, resolves, type-checks, formats; `#[repr(C)] enum E { A = 1, B { x: i32 } = 2 }` accepted; signed-repr negative values `#[repr(i8)] enum E { A = -1, B(u32) = 5 }` accepted; a manual `fn tag(ref self) -> u8` calling `match self { Op.Connect { .. } => 0x05, ... }` type-checks normally and is the documented consumption pattern. Negative ‚Äî partial-explicit (`enum E { A = 1, B(u32) }`) rejected with `E_PARTIAL_EXPLICIT_DISCRIMINANTS`; out-of-range `#[repr(u8)] enum E { A = 256 }` rejected with `E_DISCRIMINANT_OUT_OF_RANGE`; duplicate values `#[repr(u8)] enum E { A = 1, B(u32) = 1 }` rejected with `E_DUPLICATE_DISCRIMINANT` naming both variants; payload variant with explicit discriminant on a `#[repr]`-less enum rejected with `E_PAYLOAD_DISCRIMINANT_REQUIRES_REPR`; same on `#[repr(transparent)]` rejected; `.discriminant()` call on a payload enum (regardless of explicit discriminants) continues to be rejected with the existing payload-exclusion diagnostic. Four discrete implementation substrates, all post-v1: (1) **Comptime evaluator** ‚Äî treewalk interpreter over the typed AST, retargeted from the existing Phase 4 interpreter to compile-time invocation; covers value-level computation in `comptime fn` bodies excluding AST/type-as-value work. (2) **`Type` as first-class value + reflection API** ‚Äî expose the compiler's existing type registry as immutable `Type` values; field/variant/method iteration; size/align queries; the `Type` pseudotype is comptime-only (runtime references emit `E_TYPE_VALUE_AT_RUNTIME`). Each substrate is internally complete on its own ‚Äî partial deployment never leaks a half-built feature into the v1 language. Effect-system wiring (already pinned in v1 ‚Äî design.md ¬ß Comptime Effect Defaults): `reads(CompileTimeEnv)` / `writes(CompileTimeEnv)` / `allocates(CompileTimeHeap)` permitted; `panics` becomes a build-time diagnostic; runtime resource verbs forbidden via `E_RUNTIME_EFFECT_AT_COMPTIME`. Resource limits (configurable via CLI flags): 2^24 instructions, 512 MiB CompileTimeHeap, 1024 frames recursion. Tooling integration: `karac doc` documents comptime fns with a "comptime" badge; `karac explain --expand <span>` prints post-expansion AST; `karac` debugger sees comptime frames as a dedicated stack; `karac query monomorphization` tracks comptime invocations alongside type/effect tuples.
