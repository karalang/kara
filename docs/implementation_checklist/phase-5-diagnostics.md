### 5.2: Language Features

- [x] **Labeled loops.** Extend lexer/parser to support `label: for/while/loop` syntax. `break label` and `continue label` target the named loop. `break label expr` exits the named loop with a value. Labels are bare identifiers with colon (Go/Java style, no sigil). Resolver validates that break/continue labels refer to an enclosing loop.

- [x] **Labeled blocks (`label: { ... }`).** Extend the parser to accept `IDENT ":" BLOCK` as an expression form (`LABELED_BLOCK` per syntax.md ┬¦5.3) ŌĆö sibling to labeled loops. Resolver and typechecker treat a labeled block as a `break` target: `break label` and `break label expr` from inside the block exit it (with value), the block's type is the LUB of all reachable `break label expr` value types and the tail expression's type, and the existing labeled-loop machinery for label-scope and unknown-label diagnostics is shared. **`continue label` referring to a labeled block is a compile error** (`continue label refers to a labeled block; continue is only valid for loops`) with a fix-it pointing at the label binding. A `break` inside a closure body cannot target an enclosing label (parity with the existing labeled-loop closure-boundary rule). Test coverage: bare-`break label` exits with `()`, value-carrying `break label expr` joined with the tail expression, multi-`break` LUB inference, nested labeled blocks shadowing same-name label, unknown-label diagnostic, `continue label` referring to a labeled block diagnostic, closure-body `break` rejection, label scope ends at the closing `}` (later use of the same identifier as a `break` target outside is unknown-label). Phase target: 5.2 (Language Features). Cross-reference: `docs/design.md` ┬¦ Loops > **Labeled blocks**, syntax.md ┬¦5.3.

- [x] **Labeled blocks ŌĆö codegen + interpreter sibling slice.** Wire runtime semantics for `LabeledBlock` so `break label expr` actually exits the block with the value (codegen + interpreter), and route `break label` (without value) to the same exit. Frontend slice (commit `85e49c8`, 2026-05-09) shipped parser + resolver + typechecker; today both backends fall through their `_ =>` catch-alls (codegen returns the i64 zero sentinel; interpreter returns `Value::Unit`), so the typed program compiles but runs wrong. **Latent bonus fix:** today's `LoopFrame` carries no `label` field and `compile_break` always targets `loop_stack.last()`, so nested labeled loops where `break outer` should bypass `inner` would silently miscompile to break the innermost loop instead. No test fixture exercises this (grep `tests/` for nested labeled loops returns zero hits), but the codegen slice closes the gap as a side-effect because the same label-aware lookup serves both labeled blocks and labeled loops.

- [x] **Named/labeled function arguments.** Extend parser to accept `name: expr` at call sites. Labels must follow declaration order. Partial labeling allowed (any suffix). Receiver (`self`) cannot be labeled. Labels are documentation ŌĆö never required.

- [x] **`seq {}` block.** Suppress auto-parallelism within the block. All statements execute in source order. Block is an expression (value is last statement). Bindings declared inside are scoped to the block, consistent with all other blocks. Extract values via expression value or tuple.

### 5.5: Package Manager (v1.1)

All entries here are v1.1 work ŌĆö v1 ships with the current parsed-but-ignored `[dependencies]` posture; this section flips it to load-bearing. Decisions are recorded in `design.md ┬¦ Package System`.

- [ ] **PubGrub resolver.** Port `pubgrub-rs` or roll a K─üra-side equivalent. Inputs: `[dependencies]` + `[dev-dependencies]` + `[target.X.dependencies]` constraints across the dependency graph; output: a concrete version per package or a structured conflict diagnostic naming the constraint chain. Conflict diagnostics show the full chain ("`A` requires `C >=1.0`; `B` requires `C >=2.0`"), not a single "no solution" message. Honors the MSRV constraint via `[package].kara-version` (see toolchain entry).

- [ ] **Lockfile (`kara.lock`) reader/writer.** Schema: package name, exact version, source URL (registry-proxy mirror or git URL), BLAKE3 content hash, resolver-output dependency tree. Single lockfile across all build targets. Writer is invoked by the resolver after a successful resolve; reader is consulted on every build to pin versions. Library-vs-binary commitment is encoded by which projects check the file in (no compiler-side enforcement).

- [ ] **`karac update` and `karac update <pkg>`.** Bare form bumps every dep to the newest version compatible with declared semver constraints; surgical form bumps a single package (and any deps newly required by its update). Both rewrite `kara.lock`.

- [ ] **Registry proxy client.** HTTP client that fetches packages through `proxy.kara-lang.org` (or equivalent configured proxy) by default. Records both the original git URL (for human readability) and the proxy mirror reference (for fetch reproducibility) into `kara.lock`. `--no-proxy` flag for direct-from-source fetches in development workflows. The proxy itself is separate infrastructure (not part of `karac`); this entry is the client side.

- [ ] **Build artifact cache (`~/.kara/cache/`).** Global cache of compiled deps keyed on `(compiler-version, package-version, edition, profile, target-triple)`. Two projects with the same dep version under the same compiler / edition / profile / target share a cached object. Per-project `dist/` is already in place. Auto-invalidate on compiler upgrade (cache key includes version); never invalidate on time.

- [ ] **`karac clean` and `karac clean --global`.** Bare form removes the project `dist/`; `--global` removes the user-wide `~/.kara/cache/`.

- [ ] **`karac install <bin-spec>`.** Build a binary package and install it into `~/.kara/bin/`. Spec accepts `path = ...`, `git = ...`, or a registry-proxy reference. Useful pre-registry for path/git deps.

- [ ] **`karac vendor` and `karac build --offline`.** `karac vendor` copies all resolved dependencies into a `vendor/` directory at the project root. `karac build --offline` reads dependencies only from `vendor/` and refuses any network access. Air-gap workflow for regulated environments and offline CI.
