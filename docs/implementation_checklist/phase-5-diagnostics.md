### 5.2: Language Features

- [x] **Labeled loops.** Extend lexer/parser to support `label: for/while/loop` syntax. `break label` and `continue label` target the named loop. `break label expr` exits the named loop with a value. Labels are bare identifiers with colon (Go/Java style, no sigil). Resolver validates that break/continue labels refer to an enclosing loop.

- [x] **Labeled blocks (`label: { ... }`).** Extend the parser to accept `IDENT ":" BLOCK` as an expression form (`LABELED_BLOCK` per syntax.md ¬ß5.3) ‚Äî sibling to labeled loops. Resolver and typechecker treat a labeled block as a `break` target: `break label` and `break label expr` from inside the block exit it (with value), the block's type is the LUB of all reachable `break label expr` value types and the tail expression's type, and the existing labeled-loop machinery for label-scope and unknown-label diagnostics is shared. **`continue label` referring to a labeled block is a compile error** (`continue label refers to a labeled block; continue is only valid for loops`) with a fix-it pointing at the label binding. A `break` inside a closure body cannot target an enclosing label (parity with the existing labeled-loop closure-boundary rule). Test coverage: bare-`break label` exits with `()`, value-carrying `break label expr` joined with the tail expression, multi-`break` LUB inference, nested labeled blocks shadowing same-name label, unknown-label diagnostic, `continue label` referring to a labeled block diagnostic, closure-body `break` rejection, label scope ends at the closing `}` (later use of the same identifier as a `break` target outside is unknown-label). Phase target: 5.2 (Language Features). Cross-reference: `docs/design.md` ¬ß Loops > **Labeled blocks**, syntax.md ¬ß5.3.

  *Slice plan (drafted 2026-05-08) ‚Äî Labeled blocks.*

  Ship `label: { ... }` as a block-expression form: parser accepts the new shape, resolver registers the label with kind tag (Block, distinct from Loop), typechecker computes the block's type as the LUB of all reachable `break label expr` value sites and the tail expression. **Audit finding (2026-05-08):** the staging entry's claim that the existing labeled-loop machinery includes the closure-boundary rule is incorrect ‚Äî `src/resolver.rs:2300-2315` (`ExprKind::Closure` arm) does not save/restore `loop_labels`, and no test under `tests/{parser,resolver,typechecker}.rs` covers closure-body label rejection. This slice ships the closure-boundary rule for **both** loops and blocks together, which fixes the missed-rule for loops as a side-effect. **Locked design choices (2026-05-08):** (i) **LB1: new `ExprKind::LabeledBlock { label: String, body: Block }` AST variant** ‚Äî additive; does not modify the existing `Block` struct (which is widely used). Mirrors the per-loop-form label-on-AST-node convention used by `ExprKind::Loop` / `While` / `For` / `WhileLet`. (ii) **LB2: label-stack entries gain a kind tag** ‚Äî `loop_labels: Vec<Option<String>>` in resolver and `loop_labels: Vec<String>` in parser become `Vec<(Option<String>, LabelKind)>` and `Vec<(String, LabelKind)>` respectively, where `LabelKind ‚àà { Loop, Block }`. `continue label` site checks the resolved label's kind: `Block` ‚Üí `error[E_CONTINUE_LABEL_BLOCK]` with fix-it pointing at the label binding; `Loop` ‚Üí existing accept path. The kind tag is purely additive at use sites ‚Äî `break label [expr]` accepts both kinds. (iii) **LB3: LUB inference via per-label types collector** ‚Äî typechecker pushes a fresh `Vec<Type>` onto a `break_value_types: Vec<Vec<Type>>` stack at labeled-block entry, every `Break { label: Some(name), value: Some(e) }` arm appends `infer_expr(e)` to the matching frame, and at labeled-block exit the block's type is `lub(tail_expr_type, ...collected_break_types)`. **Loops keep their existing `Type::Never`-by-default behavior** (per `src/typechecker.rs:7423-7426` ‚Äî line 2564 of design.md's "loop type inference" rule is a separate gap, tracked at line 113's "Const generics" sibling ‚Äî explicit out-of-scope for this slice). (iv) **LB4: closure-boundary rule shipped for all label kinds** ‚Äî `ExprKind::Closure` arm in resolver and typechecker saves the current label stack, replaces it with an empty stack for body resolution, restores on exit. A `break label` / `continue label` inside a closure body whose label refers to an enclosing loop or block thus surfaces as `undefined loop label` (or the new block diagnostic). The closure body cannot target outer labels ‚Äî labels are lexical to the function-body control flow. Closure parameters and locals continue to resolve normally; only label resolution is scoped.

  *Slice plan (drafted 2026-05-09) ‚Äî Labeled blocks: codegen + interpreter sibling.*

  Ship label-keyed control-flow plumbing on both backends. Codegen: per-labeled-block result slot (analogous to Slice A's per-branch return slots, but in the value-block-exit flavor ‚Äî early-exit-with-value through a stack-allocated slot the labeled-block expression's parent reads); `LoopFrame` extended with `label: Option<String>`; `compile_break` walks the frame stack by label rather than always taking the innermost. Interpreter: `ControlFlow::Break { label, value }` already carries the label (parser/resolver wire it), so the new `LabeledBlock` arm just propagates the existing signal ‚Äî match on the frame's label at the labeled-block boundary, return the `value` as the block's result. **Locked design choices:** (i) **LBC1: shared frame stack across labeled loops + labeled blocks** ‚Äî codegen extends `LoopFrame` (`src/codegen.rs:452`) with `label: Option<String>` rather than introducing a parallel `block_stack`. Both labeled loops and labeled blocks push frames onto `loop_stack` with their label; `compile_break` does a label-aware lookup (walks stack from top, picks first frame matching the resolved label, falls back to innermost when label is `None`). Single mechanism is simpler to reason about and lights up nested-labeled-loop break correctness as a side-effect. The loop frame's `continue_bb` stays meaningful for loop frames; for block frames it is unreachable (resolver already rejects `continue label` on a block label per `E_CONTINUE_LABEL_BLOCK`), but keeping the field uniform avoids a `LoopOrBlockFrame` enum split. (ii) **LBC2: result-slot type via typechecker LUB** ‚Äî the labeled-block expression's type is already computed at typecheck time via `lub_block_type` (commit `85e49c8`, frontend slice LB3) and lives on the `infer_expr_type` side-table. Codegen at the new `ExprKind::LabeledBlock` arm reads the type, allocates a slot of the corresponding LLVM type at the `entry` BB (sibling to existing alloca patterns), pushes a frame with the slot, compiles the body, branches the post-body fallthrough to a freshly-created `lblock.exit` BB, position-after at exit, loads the slot. The slot stores both the tail-expression value (from a normal block exit) and any `break label expr` values (from early exits). (iii) **LBC3: `break label` (no value) stores a zero-equivalent** ‚Äî for typed-as-`()` blocks the slot is i64 (the existing unit-equivalent), the bare-break path stores zero. The slot's LLVM type matches the labeled-block expression type, so a bare break inside a `() : ()`-typed block just stores i64 zero; inside any block whose type is non-unit (always a value-carrying break per the typechecker's LUB constraint ‚Äî bare breaks alongside value breaks degrade the block to `()`), bare break can't typecheck, so this path never fires for non-unit slots. (iv) **LBC4: interpreter ‚Äî propagate existing `ControlFlow::Break`** ‚Äî `ControlFlow::Break { label: Option<String>, value: Value }` is already the runtime control-flow signal (`src/interpreter.rs:917-925`). The new `ExprKind::LabeledBlock { label, body, .. }` arm at `eval_expr_inner`'s match (the existing `_ => Value::Unit` catch-all per the frontend slice's "Out of scope, still open" line 157(b)): evaluate `body` via `eval_block`; on `Ok(v)` return `v`; on `Err(ControlFlow::Break { label: Some(l), value })` where `l == label`, return `Ok(value)`; on any other `Err(cf)` propagate (outer labels, returns, cancellation, etc.). Bare-break path's `value` is `Value::Unit`. **No new ControlFlow variants needed.** (v) **LBC5: closure-boundary parity in codegen** ‚Äî the existing `take(&mut self.loop_stack)` save/restore around closure compilation (`src/codegen.rs:14624 / 14966 / 15491`) already isolates closures from the parent's frame stack ‚Äî extending `LoopFrame` with a label field doesn't change that mechanism. Closure-side label-shadowing (resolver-rejected per LB4) is already prevented at the source level; codegen relies on the resolver's prior validation.

### 5.4: Interactive REPL and Jupyter Kernel

First-class interactive surface. **v0 ‚Üí v1 transition:** the MVP REPL shipped 2026-04-28 on top of the tree-walk interpreter; the v62 lock (2026-05-06) moved the runtime backend to **always-JIT via LLJIT** for v1, with tree-walk retained only as a dev/debug tool. ### 5.5: Package Manager (v1.1)

All entries here are v1.1 work ‚Äî v1 ships with the current parsed-but-ignored `[dependencies]` posture; this section flips it to load-bearing. Decisions are recorded in `design.md ¬ß Package System`.

- [ ] **PubGrub resolver.** Port `pubgrub-rs` or roll a KƒÅra-side equivalent. Inputs: `[dependencies]` + `[dev-dependencies]` + `[target.X.dependencies]` constraints across the dependency graph; output: a concrete version per package or a structured conflict diagnostic naming the constraint chain. Conflict diagnostics show the full chain ("`A` requires `C >=1.0`; `B` requires `C >=2.0`"), not a single "no solution" message. Honors the MSRV constraint via `[package].kara-version` (see toolchain entry).

- [ ] **Lockfile (`kara.lock`) reader/writer.** Schema: package name, exact version, source URL (registry-proxy mirror or git URL), BLAKE3 content hash, resolver-output dependency tree. Single lockfile across all build targets. Writer is invoked by the resolver after a successful resolve; reader is consulted on every build to pin versions. Library-vs-binary commitment is encoded by which projects check the file in (no compiler-side enforcement).

- [ ] **`karac update` and `karac update <pkg>`.** Bare form bumps every dep to the newest version compatible with declared semver constraints; surgical form bumps a single package (and any deps newly required by its update). Both rewrite `kara.lock`.

- [ ] **Registry proxy client.** HTTP client that fetches packages through `proxy.kara-lang.org` (or equivalent configured proxy) by default. Records both the original git URL (for human readability) and the proxy mirror reference (for fetch reproducibility) into `kara.lock`. `--no-proxy` flag for direct-from-source fetches in development workflows. The proxy itself is separate infrastructure (not part of `karac`); this entry is the client side.

- [ ] **Build artifact cache (`~/.kara/cache/`).** Global cache of compiled deps keyed on `(compiler-version, package-version, edition, profile, target-triple)`. Two projects with the same dep version under the same compiler / edition / profile / target share a cached object. Per-project `dist/` is already in place. Auto-invalidate on compiler upgrade (cache key includes version); never invalidate on time.

- [ ] **`karac clean` and `karac clean --global`.** Bare form removes the project `dist/`; `--global` removes the user-wide `~/.kara/cache/`.

- [ ] **`karac install <bin-spec>`.** Build a binary package and install it into `~/.kara/bin/`. Spec accepts `path = ...`, `git = ...`, or a registry-proxy reference. Useful pre-registry for path/git deps.

- [ ] **`karac vendor` and `karac build --offline`.** `karac vendor` copies all resolved dependencies into a `vendor/` directory at the project root. `karac build --offline` reads dependencies only from `vendor/` and refuses any network access. Air-gap workflow for regulated environments and offline CI.

- [ ] **`[target.X.dependencies]` and `[target.X.profile]` parsing.** Extend `manifest.rs` to recognize per-target dependency and profile blocks. Resolver consults the active target triple (from `karac build --target X` or `[build].target` default) to merge the relevant target block into the dep graph.

- [ ] **`[dev-dependencies]` excluded from non-test builds.** Resolver and build pipeline already separate test compilation from non-test compilation; ensure `[dev-dependencies]` participate only in test-mode resolution. No new infrastructure ‚Äî wiring in the existing test/non-test split.

- [ ] **`[package].kara-version` MSRV constraint.** Parser accepts the field; resolver enforces it as a constraint on the active toolchain version. A package whose `kara-version` exceeds the active toolchain version triggers a resolution failure with a diagnostic showing the constraint chain (which dep, what it required, what's installed).

- [ ] **`karac-toolchain.toml` reader.** `karac` parses the file from project (or workspace) root and either delegates to `karaup` (when it exists) to switch toolchain, or ‚Äî if running standalone ‚Äî emits a diagnostic indicating the version mismatch and the recommended action. v1 schema: `version` (required), `targets` (optional list). Channels / components / install profiles deferred.

- [~] **`:dep` REPL meta-command.** Parse `:dep name = "..."` (accepting the same right-hand side shape as a `[dependencies]` entry). Mutate the session's in-memory manifest, drive the resolver against the new constraint, fetch + compile the package, and bring its surface into scope for subsequent cells. Failures (resolution conflict, network error, missing package) surface in the cell output and leave session state unchanged. State is in-memory only ‚Äî no implicit persistence. Closed 2026-05-01 *for the syntax-parsing + registry portion only*: new `Session.pending_deps: BTreeMap<String, String>` accumulates `name ‚Üí normalized TOML value`; new `add_dep` parses the RHS by wrapping it as `[__kara_repl_dep__]\n{rest}\n` and leaning on the `toml` crate to validate the shape (bare semver string OR inline table ‚Äî `{ git = ... }` / `{ path = ... }` / `{ version = ... }`); failures surface `error: invalid \`:dep\` syntax ‚Äî ...` and leave state unchanged; success prints the recorded entry plus a v1.1 caveat note that the package's symbols are not yet in scope; `Session::pending_deps()` accessor exposed for the future package-manager wiring; `:help` updated. 9 tests in `tests/repl.rs`. The resolver / fetch / symbol-bring-into-scope plumbing is gated on the package-manager work (registry-proxy + resolver + lock-file, not yet shipped) and aligns with design.md's v1.1 tag for `:dep`.

- [ ] **`karac run <script>` script-dir manifest discovery.** Walk upward from the *script's own directory*, not cwd, looking for `kara.toml`.
