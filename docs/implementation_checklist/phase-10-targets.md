## Phase 10: Additional Targets

> **Priority: P0 (the WASM target leg specifically).** Greenfield-runtime is KДЃra's single largest *adoption* headwind вЂ” a language that ships its own runtime asks for a full rewrite, which is the switching cost that buried most technically-superior "level-2" languages (D, Nim, Crystal). The WASM target is the structural mitigation: it lets KДЃra ride an *existing, universal* host platform (browsers, edge, wasmtime, node) instead of demanding migration, so adoption can be incremental rather than all-or-nothing. That reclassifies WASM from "a target we support" to "the on-ramp that makes adoption physically possible" вЂ” hence P0, peer to the language core, not a post-v1 nicety. (Embedded/no_std and Cortex-M TLS remain post-v1 вЂ” see those entries; the P0 scope is the wasm32 browser + WASI command/component path.)

- [x] **WASM module-size benchmark vs Rust + TinyGo вЂ” measure before claiming parity.** On WASM the headline metric is *module size* (every byte is downloaded), the way startup density is the headline on native. KДЃra's structural story (AOT, no GC, no async runtime; the wasm archive strips tokio/mio/socket2; `+simd128` default; wasm-ld section-GC = native `-dead_strip` posture) *should* land it near Rust's tiny modules вЂ” but that's a hypothesis, currently **unmeasured**. Per "measure-first": size a real `hello.wasm` and one real workload (e.g. Iris's filter core) as **KДЃra vs Rust(`wasm-pack`/wasip2) vs TinyGo**, stripped + brotli'd, and record the numbers. Competitive в†’ it's a receipt that backs the WASM positioning; not competitive в†’ we need to know *before* someone benchmarks it in an HN thread. Definition of done: a checked-in size table (same source compiled three ways) refreshed through the canonical bench pipeline, not hand-edited. Gating only *if* WASM module size is made a launch claim вЂ” otherwise a credibility receipt, not a launch blocker. Pairs with the existing wall-time "keep the native/WASM comparison honest" note on the **Iris** dogfooding demo (size is the axis that demo does *not* yet measure).
  - **Prereq landed вЂ” strip-by-default (2026-06-10).** The "section-GC = `-dead_strip`" claim above was the trap: wasm-ld's section-GC drops unreachable *code*, but **keeps the `.debug_*` DWARF custom sections**, which are ~90%+ of an unstripped module (measured: browser hello 482 KiB = 93% DWARF). The native link path strips by default; the wasm path did not, so `karac build --target=wasm_*` shipped 482 KiB / 508 KiB modules whose *real* payload is ~25вЂ“30 KiB. Fixed: `karac build` now runs `wasm-tools strip` on every emitted `.wasm` (main module/component + any `.threads.wasm`) вЂ” `componentize::strip_debug`, called before the `Built:` line in `cmd_build`. Default `wasm-tools strip` keeps `name`/`component-type`/`dylink.0`, so components and shared-memory (threaded) modules stay valid (verified `wasm-tools validate`). `KARAC_WASM_KEEP_DEBUG=1` opts out (source-level wasm debugging); missing wasm-tools on a browser/raw build в†’ a note, not a failure. Measured after: browser hello **30 KiB**, wasi component **51 KiB**, raw module **17 KiB** вЂ” competitive with native (33 KiB hello). So the benchmark above must size the **stripped** output, and the structural-story parenthetical should read "section-GC + wasm-tools strip," not section-GC alone. E2E: `tests/cli.rs::wasm_build_strips_debug_info_by_default` (asserts >3x reduction vs the KEEP_DEBUG build).
  - **Landed вЂ” `bench/wasm_size/` track (2026-06-22).** The measurement is now a checked-in receipt: [`bench/wasm_size/`](../../bench/wasm_size/) builds `hello` + the Iris filter core three ways (KДЃra / Rust / TinyGo) in two flavors (core module = `wasm32-wasip1`; component = WASI 0.2 / `wasm32-wasip2`), strips with `wasm-tools strip`, brotli-compresses, and emits [`sizes.md`](../../bench/wasm_size/sizes.md) + `sizes.json` via `bench.sh` (not hand-edited, per DoD). **Honesty guard:** the three `filter_core` ports are a line-for-line port of the same integer kernel, and `bench.sh` runs each core module under `node:wasi` and asserts all three print **identical FNV-1a checksums** (the Iris native-oracle values) вЂ” a mismatch fails the bench, so it can't compare a stand-in against a full kernel. **Result (brotli'd stripped bytes, the real download cost):** *core module* вЂ” `hello` KДЃra **1,579** vs Rust 13,678 vs TinyGo 58,334; `filter_core` KДЃra **13,600** vs Rust 14,676 vs TinyGo 65,672 в†’ KДЃra edges size-tuned Rust and is ~4.8Г— under TinyGo, and is dramatically smaller on `hello` (Rust's `println!` drags in std fmt + panic/backtrace). *Component (wasip2)* вЂ” `hello` KДЃra **12,736** vs Rust 16,291 vs TinyGo 100,099; `filter_core` KДЃra 20,863 vs Rust **17,361** vs TinyGo 107,277 вЂ” the **one** cell where Rust wins: KДЃra's preview1в†’preview2 adapter is heavier; shrinking it is the open size follow-up. So the WASM-positioning receipt holds on the core-module axis (the shape `--target=wasm_browser` ships) and on component `hello`; the component-adapter weight on larger workloads is the named gap. Gate shape = credibility receipt, not a CI gate (no `baseline.json`) вЂ” matches the entry's "gating only *if* size is made a launch claim." **Surfaced (flagged, not fixed here):** KДЃra's **component** emission is byte-NONdeterministic (same length, reordered content across rebuilds вЂ” HashMap-ordering in `componentize`/`wit` emission; the core-module path is byte-stable) вЂ” separate compiler follow-up; the component brotli figure carries ~0.2% run-to-run jitter as a result, noted in `sizes.md`.

- [x] **wasm-threads Г— `host fn` вЂ” synchronous workerв†’main proxy.** *(2026-06-07)* Lifted the v1 build-time rejection of `host fn` under `--features wasm-threads`. The threaded program runs in a primary Web Worker (and pool workers), but `host fn` implementations are user-supplied main-thread JS closures, so each call round-trips workerв†’main synchronously across a shared control block вЂ” glue-side machinery only; codegen's `kara_host` import surface is unchanged. Landed:
  - **Shared control block (`HOST_CTL_BYTES` SharedArrayBuffer).** Layout (`wasm_glue.rs` static body): an i32 region вЂ” `HC_LOCK` (mutex), `HC_SEQ` (request doorbell), `HC_DONE` (result-ready), `HC_FN` (call index), `HC_ARGC` вЂ” plus an 8-byte-aligned payload region (`HC_PAYLOAD_OFF = 64`) of `HC_MAX_ARGS = 16` scalar slots + one return slot, viewed as overlaid `Float64Array`/`BigInt64Array` so each scalar reads/writes at its boundary class. One call in flight at a time (the mutex serializes workers; the main thread services serially anyway), so the slots are reused per call.
  - **Worker side (`makeHostProxy`).** Built per worker (primary AND pthread вЂ” both go through `karaThreadWorkerMain`, which now adds `imports.kara_host = makeHostProxy(hostCtl)` when `hostCtl` is present), all bound to the same shared block. Each `kara_host.<name>` stub: acquire `HC_LOCK` (`Atomics.compareExchange` + `Atomics.wait` on contention), marshal args per the `HOST_FN_SIGS` table, ring the doorbell (`Atomics.add(HC_SEQ)` + `notify` вЂ” release), `Atomics.wait` on `HC_DONE` until the result lands (acquire), read the return, release the lock. String `(ptr, len)` args pass through as scalars вЂ” the bytes live in the shared linear memory the main side reads directly.
  - **Main side (`startHostService`).** Runs on the main thread's event loop via `Atomics.waitAsync` (the page agent may be non-blockable вЂ” can't `Atomics.wait`; a `setTimeout`-poll fallback covers engines without `waitAsync`). On each doorbell it reads the call index/args, runs `hostImpls[name](...args, ctx)` with `ctx.memory`/`ctx.readString` over the shared memory, writes the return, sets `HC_DONE` + `notify`. `runThreaded(hostImpls)` validates impls up front (the same contract as the sequential `buildImports`), stands the service up before spawning the primary worker, and `stop()`s it in a `finally`. A throwing host impl logs + returns a zero result rather than deadlocking the worker.
  - **Signature-driven marshalling table (`HOST_FN_SIGS`).** Emitted by `render_glue` (always вЂ” `[]` for a host-fn-free program; the static body references it unconditionally) as `{ name, params: ["bigint"|"number"...], ret: "bigint"|"number"|"void" }` per fn, so the worker/main marshallers are data-driven, not per-fn-generated.
  - **Gates dropped.** The `cmd_build` (single-file) and `run_multi_file_codegen` (project) build-time rejections + their error text are gone; the existing build flow already collects host fns and renders the glue with the threaded config, so no new wiring was needed beyond removing the gates. - [ ] **Scheduler вЂ” yield-to-event-loop on WASM channel-receive.** Implement the WASM scheduler contract: `go`-task performing `channel.recv()` on a channel whose pending sender is a host-async operation yields to the browser event loop. Host-async completion (Promise resolution via registered `.then`, `setTimeout` firing, DOM event dispatch) posts to the scheduler's ready queue, which wakes the waiting `go`-task on the next tick. Source-level semantics match native (function appears to block on receive). Verify no user-observable difference from native except scheduling. Applies to both sequential-default lowering and `--features wasm-threads` lowering from Q5 вЂ” the yield-to-event-loop behavior is orthogonal to whether the underlying scheduling is single-threaded or multi-worker.

  *Re-scoped (2026-06-07).* This entry was investigated to be **two slices with wildly different cost**, split here. The source-level contract (design.md В§ Async Host APIs on WASM "appears to block, no ceremony") is **mechanism-agnostic** and already locked вЂ” asyncify / JSPI / a CPS state-machine transform all satisfy it identically вЂ” so adding any mechanism later is a **zero-source-break** change. The decision below keeps that door open without paying for it before a consumer exists.
  - **Threaded leg (`--features wasm-threads`) вЂ” already met; folds into the `std.web.time` producers entry below.** On a threaded target the contract is realized by the *existing* `recv` Condvar-block (`runtime/src/channel.rs`, the `any(not(wasm), wasm-threads)` arm): `recv` parks the calling worker, the host-async `send` fires on the **main thread's** event loop (a `setTimeout`/Promise/DOM callback re-entering wasm `send`) and notifies the Condvar, waking the worker. A blocked worker *is* the yield вЂ” the main-thread event loop is never occupied. No scheduler work remains here; what's left is the glue wiring so a host callback re-enters wasm `send`, which overlaps almost entirely with the producer wrappers. **Tracked under "`std.web.time.after` вЂ¦ and event-stream wrappers" below** (its threaded E2E is the natural place to prove the wake path), not as standalone work here. *(Wake path PROVEN 2026-06-07 by the `after` thin slice вЂ” `wasm_threads_timer_after_recv_e2e`; the glue's main-thread "service" instance is what re-enters wasm `send` from the `setTimeout` callback.)*
  - **Sequential leg (`wasm_browser` / `wasm_wasi` default) вЂ” DEFERRED post-v1.** On single-threaded wasm, block-then-resume is *physically impossible* without a task-suspension mechanism: control must unwind out of wasm to the JS event loop (so the host callback can fire) and re-enter вЂ” and you cannot spin-pump the event loop from inside a synchronous main-thread wasm call (`Atomics.wait` traps on the main thread; timers/microtasks don't run while the stack is busy). The tree has **no** such mechanism (no `wasm-opt`/asyncify, no JSPI, and the CPS full-hybrid transform is deferred per design.md В§ State-Machine Transform). So `seq_scheduler.rs::drive_until_terminal`'s "hook point" cannot resume a parked native stack вЂ” there is nothing to park *with*. **Rationale for deferring rather than building asyncify now:** (1) not a launch gate (backend-first; the flagship demos are idle-holder/Parallax/pipeline, none a browser event-driven UI); (2) the front-end **demo path is `--features wasm-threads`** вЂ” it honors the contract today and COOP/COEP headers are trivial on a host we control (Cloudflare Pages), so the demo does **not** force this leg; (3) asyncify is a *permanent* per-binary tax (instrumentation + a transitive reachable-from-spawn-can-reach-`recv` allowlist fed to `wasm-opt`) plus a new Binaryen toolchain dep, for a feature whose only producers (the entry below) don't exist yet. The narrow case this leg actually serves is **the default sequential target on hosts where the deployer cannot set COOP/COEP** (e.g. someone copy-pasting a demo onto a bare static host) вЂ” a v1.x/v2 *distribution* concern, not a demo/launch one. **When it surfaces:** reconfirm JSPI's runtime-support matrix first вЂ” if JSPI has shipped unflagged by then it is the cleaner default (no instrumentation tax); asyncify is the "works everywhere today" fallback. Deciding the mechanism now would lock in the heavier option prematurely. The companion correctness gate (so the deferral never *silently* mis-executes) is the entry immediately below.

- [x] **GPU routing decision вЂ” `#[gpu]` annotation design.** Decide how code is split between CPU and GPU. Recommendation: explicit `#[gpu]` attribute on functions, compiler verifies GPU-safety via effects (no `reads`, `writes`, `sends`, `receives`, `allocates(Heap)`, `panics`, no recursion). Layout groups (`group physics { position, velocity }`) map to GPU buffers. User declares intent, compiler enforces the rules. Design options: (A) fully automatic вЂ” too much magic, (B) explicit `#[gpu]` вЂ” recommended, (C) layout-driven вЂ” could complement B.

  *Decided 2026-06-28 вЂ” Option B, with C as a complement; A rejected. Authoritative spec already written: [`design.md В§ GPU Subset Constraints`](../design.md#gpu-subset-constraints).* This is a **design** slice (the decision + its home in the spec); the **codegen** that acts on the decision вЂ” WGSL/PTX lowering of `#[gpu]` bodies and the `gpu.dispatch` runtime wiring вЂ” is the separate v1-P1 implementation track at [`roadmap.md В§ Phase 10 > GPU compute shaders`](../roadmap.md), not this item. roadmap.md:847 confirms the split ("GPU is still at the `#[gpu]` routing-design stage вЂ¦ not yet built") and roadmap.md:852 confirms the home ("Full design вЂ¦ is already in `design.md В§ GPU Subset Constraints`").
  - **The decision (B):** GPU-safety is a **constraint the programmer declares at the definition site** with `#[gpu]`, and the compiler enforces вЂ” it never silently routes work to the GPU. `#[gpu]` means "this function uses only the GPU-compatible subset," nothing more; *dispatch* is always an explicit, separate `gpu.dispatch(fn, argsвЂ¦)` call. Small input в†’ call the same function on CPU; large batch в†’ dispatch it. The programmer controls *where* work runs; the compiler only guarantees the kernel *can* run there.
  - **Why not A (fully automatic):** auto-inferring which functions are GPU kernels from their bodies (or from monomorphization landing on only-`GpuSafe` types) is the "too much magic" failure mode вЂ” it makes a profound runtime/placement decision invisible at the call site, mirrors the rejected "infer effects at the call site" anti-pattern, and gives the author no declared intent to enforce against. `#[gpu]` is required even on a generic instantiated with only `GpuSafe` types: the question is "did the author declare this GPU-callable?", not "does this instantiation happen to be safe?" (the pre-monomorphization check, design.md В§ *Generics and `#[gpu]`*).
  - **C complements, does not replace:** layout groups (`group physics { position, velocity }`, Feature 1) map directly to GPU buffers, so the compiler emits coalesced bulk transfers and `gpu.dispatch` carries `reads/writes(GpuBuffer[buf])` parameterized by the buffer handle вЂ” two dispatches on disjoint buffers parallelize via the existing literal-distinctness conflict analysis. Layout-driven buffer mapping is the *data-movement* half; it rides on B's constraint declaration rather than being a third routing mechanism on its own.
  - **No dedicated GPU pass вЂ” enforcement reuses two existing phases** (design.md В§ *Call graph validation*): the **effect checker** rejects any forbidden effect (`panics` вЂ” incl. `todo()`/`unreachable()`/`panic()` вЂ”, `allocates(Heap)`, and `reads`/`writes`/`sends`/`receives` on I/O resources) in the transitive call graph from a `#[gpu]` root; the **type checker's** call-graph walk rejects structural incompatibilities (heap types like `String`/`Vec[T]`, recursion, `dyn Trait`, host-capturing closures) вЂ” `GpuSafe` is the structural trait it checks, auto-derived for primitives / `Array[T, N]` / all-`GpuSafe`-field structs & enums, and available as an explicit generic bound. Both surface the full call chain from the `#[gpu]` root to the violation (the `E0801` shape in the spec). Non-generic callees need no annotation (auto-compatible if clean); generics must carry `#[gpu]` to be GPU-callable.
  - **Decision is internally consistent with the rest of the spec** вЂ” `allocates(Heap)` is the "documentation+gating verb" whose only teeth are exactly this GPU/embedded boundary (design.md:5780), the `gpu` build profile already forbids module-level/`#[thread_local]` mutable state and defaults `panic = "abort"`, and the `GpuBuffer[buf_id]` parameterized resource is already in the effect-resource table (design.md:6351). Nothing in this decision changes with the v66 pull-forward of GPU codegen to v1-P1 вЂ” the constraint/trait/dispatch design is vendor-neutral by construction (wgpu-primary: Metal/Vulkan/DX12/WebGPU; CUDA opt-in via `--target cuda`).
