## Phase 10: Additional Targets

> **Priority: P0 (the WASM target leg specifically).** Greenfield-runtime is Kāra's single largest *adoption* headwind — a language that ships its own runtime asks for a full rewrite, which is the switching cost that buried most technically-superior "level-2" languages (D, Nim, Crystal). The WASM target is the structural mitigation: it lets Kāra ride an *existing, universal* host platform (browsers, edge, wasmtime, node) instead of demanding migration, so adoption can be incremental rather than all-or-nothing. That reclassifies WASM from "a target we support" to "the on-ramp that makes adoption physically possible" — hence P0, peer to the language core, not a post-v1 nicety. (Embedded/no_std and Cortex-M TLS remain post-v1 — see those entries; the P0 scope is the wasm32 browser + WASI command/component path.)

- [ ] **WASM module-size benchmark vs Rust + TinyGo — measure before claiming parity.** On WASM the headline metric is *module size* (every byte is downloaded), the way startup density is the headline on native. Kāra's structural story (AOT, no GC, no async runtime; the wasm archive strips tokio/mio/socket2; `+simd128` default; wasm-ld section-GC = native `-dead_strip` posture) *should* land it near Rust's tiny modules — but that's a hypothesis, currently **unmeasured**. Per "measure-first": size a real `hello.wasm` and one real workload (e.g. Iris's filter core) as **Kāra vs Rust(`wasm-pack`/wasip2) vs TinyGo**, stripped + brotli'd, and record the numbers. Competitive → it's a receipt that backs the WASM positioning; not competitive → we need to know *before* someone benchmarks it in an HN thread. Definition of done: a checked-in size table (same source compiled three ways) refreshed through the canonical bench pipeline, not hand-edited. Gating only *if* WASM module size is made a launch claim — otherwise a credibility receipt, not a launch blocker. Pairs with the existing wall-time "keep the native/WASM comparison honest" note on the **Iris** dogfooding demo (size is the axis that demo does *not* yet measure).
  - **Prereq landed — strip-by-default (2026-06-10).** The "section-GC = `-dead_strip`" claim above was the trap: wasm-ld's section-GC drops unreachable *code*, but **keeps the `.debug_*` DWARF custom sections**, which are ~90%+ of an unstripped module (measured: browser hello 482 KiB = 93% DWARF). The native link path strips by default; the wasm path did not, so `karac build --target=wasm_*` shipped 482 KiB / 508 KiB modules whose *real* payload is ~25–30 KiB. Fixed: `karac build` now runs `wasm-tools strip` on every emitted `.wasm` (main module/component + any `.threads.wasm`) — `componentize::strip_debug`, called before the `Built:` line in `cmd_build`. Default `wasm-tools strip` keeps `name`/`component-type`/`dylink.0`, so components and shared-memory (threaded) modules stay valid (verified `wasm-tools validate`). `KARAC_WASM_KEEP_DEBUG=1` opts out (source-level wasm debugging); missing wasm-tools on a browser/raw build → a note, not a failure. Measured after: browser hello **30 KiB**, wasi component **51 KiB**, raw module **17 KiB** — competitive with native (33 KiB hello). So the benchmark above must size the **stripped** output, and the structural-story parenthetical should read "section-GC + wasm-tools strip," not section-GC alone. E2E: `tests/cli.rs::wasm_build_strips_debug_info_by_default` (asserts >3x reduction vs the KEEP_DEBUG build).

- [x] **Effect-checker: target-provided-resource-set parameterization.** *(2026-06-05)* `src/effectchecker/target_gate.rs` (E0411), run after inference settles for the current target (`native` until `--target` lands; the per-target build matrix loops over this same pass). Key decisions, all from `design.md § Cross-target Compilation`:
  - **Reachability-rooted**: gate walks the transitive call graph from `main` ("a function in the transitive call graph of a native entry point") — unreachable declarations don't fire (they're dead code). Library builds / test entry points: follow-on noted below.
  - **Provider discharge**: the design's primary SSR pattern binds per-target providers over target-foreign resources (`Display => HtmlStringBuilder` on native), so a `providers { }` binding discharges the resource for the subtree. Effect inference is provider-transparent, so the gate recovers bindings via synthetic `__providers_bind::<R>` pseudo-calls recorded by the inference walk (the `__modbind_*` precedent; real consumers filter/miss harmlessly) and carries a discharged-set down the DFS. Granularity is the FUNCTION (a binding anywhere in `f` discharges for all of `f`) — slightly lenient for mixed bodies, documented in the module header.
  - **Direct-use attribution**: a violation fires at the function whose effect has `Direct` origin OR `Callee` origin naming a body-less callee (host fn / extern / builtin — not graph nodes, charged at the call site). Body-having callees are examined at their own node where discharge state may differ.

User resources are exempt (no host identity; they exist wherever a provider exists). A user-declared `effect resource Display;` claims the host identity by string — consistent with the `std.wasi` redeclaration philosophy.
  - **Table**: `target::{is_host_resource, target_provides}`; design.md's table gained a `ProcessTable` row (native-only — doc gap found while transcribing).
  - Diagnostic: ``target `native` does not provide resource 'Display' — required on the call chain main → helper → paint; …`` with actionable fixes (gate with `#[target]`, bind a provider, keep off entry path). Tests: effectchecker ×6 (chain, unreachable, SSR discharge, user exempt, native-provided pass, sibling-path fires), cli ×2 (alias-evasion check-path, SSR pass).

- [x] **wasm-threads × `host fn` — synchronous worker→main proxy.** *(2026-06-07)* Lifted the v1 build-time rejection of `host fn` under `--features wasm-threads`. The threaded program runs in a primary Web Worker (and pool workers), but `host fn` implementations are user-supplied main-thread JS closures, so each call round-trips worker→main synchronously across a shared control block — glue-side machinery only; codegen's `kara_host` import surface is unchanged. Landed:
  - **Shared control block (`HOST_CTL_BYTES` SharedArrayBuffer).** Layout (`wasm_glue.rs` static body): an i32 region — `HC_LOCK` (mutex), `HC_SEQ` (request doorbell), `HC_DONE` (result-ready), `HC_FN` (call index), `HC_ARGC` — plus an 8-byte-aligned payload region (`HC_PAYLOAD_OFF = 64`) of `HC_MAX_ARGS = 16` scalar slots + one return slot, viewed as overlaid `Float64Array`/`BigInt64Array` so each scalar reads/writes at its boundary class. One call in flight at a time (the mutex serializes workers; the main thread services serially anyway), so the slots are reused per call.
  - **Worker side (`makeHostProxy`).** Built per worker (primary AND pthread — both go through `karaThreadWorkerMain`, which now adds `imports.kara_host = makeHostProxy(hostCtl)` when `hostCtl` is present), all bound to the same shared block. Each `kara_host.<name>` stub: acquire `HC_LOCK` (`Atomics.compareExchange` + `Atomics.wait` on contention), marshal args per the `HOST_FN_SIGS` table, ring the doorbell (`Atomics.add(HC_SEQ)` + `notify` — release), `Atomics.wait` on `HC_DONE` until the result lands (acquire), read the return, release the lock. String `(ptr, len)` args pass through as scalars — the bytes live in the shared linear memory the main side reads directly.
  - **Main side (`startHostService`).** Runs on the main thread's event loop via `Atomics.waitAsync` (the page agent may be non-blockable — can't `Atomics.wait`; a `setTimeout`-poll fallback covers engines without `waitAsync`). On each doorbell it reads the call index/args, runs `hostImpls[name](...args, ctx)` with `ctx.memory`/`ctx.readString` over the shared memory, writes the return, sets `HC_DONE` + `notify`. `runThreaded(hostImpls)` validates impls up front (the same contract as the sequential `buildImports`), stands the service up before spawning the primary worker, and `stop()`s it in a `finally`. A throwing host impl logs + returns a zero result rather than deadlocking the worker.
  - **Signature-driven marshalling table (`HOST_FN_SIGS`).** Emitted by `render_glue` (always — `[]` for a host-fn-free program; the static body references it unconditionally) as `{ name, params: ["bigint"|"number"...], ret: "bigint"|"number"|"void" }` per fn, so the worker/main marshallers are data-driven, not per-fn-generated.

- [ ] **Scheduler — yield-to-event-loop on WASM channel-receive.** Implement the WASM scheduler contract: `go`-task performing `channel.recv()` on a channel whose pending sender is a host-async operation yields to the browser event loop. Host-async completion (Promise resolution via registered `.then`, `setTimeout` firing, DOM event dispatch) posts to the scheduler's ready queue, which wakes the waiting `go`-task on the next tick. Source-level semantics match native (function appears to block on receive). Verify no user-observable difference from native except scheduling. Applies to both sequential-default lowering and `--features wasm-threads` lowering from Q5 — the yield-to-event-loop behavior is orthogonal to whether the underlying scheduling is single-threaded or multi-worker.

  *Re-scoped (2026-06-07).* This entry was investigated to be **two slices with wildly different cost**, split here. The source-level contract (design.md § Async Host APIs on WASM "appears to block, no ceremony") is **mechanism-agnostic** and already locked — asyncify / JSPI / a CPS state-machine transform all satisfy it identically — so adding any mechanism later is a **zero-source-break** change. The decision below keeps that door open without paying for it before a consumer exists.
  - **Threaded leg (`--features wasm-threads`) — already met; folds into the `std.web.time` producers entry below.** On a threaded target the contract is realized by the *existing* `recv` Condvar-block (`runtime/src/channel.rs`, the `any(not(wasm), wasm-threads)` arm): `recv` parks the calling worker, the host-async `send` fires on the **main thread's** event loop (a `setTimeout`/Promise/DOM callback re-entering wasm `send`) and notifies the Condvar, waking the worker. A blocked worker *is* the yield — the main-thread event loop is never occupied. No scheduler work remains here; what's left is the glue wiring so a host callback re-enters wasm `send`, which overlaps almost entirely with the producer wrappers. **Tracked under "`std.web.time.after` … and event-stream wrappers" below** (its threaded E2E is the natural place to prove the wake path), not as standalone work here. *(Wake path PROVEN 2026-06-07 by the `after` thin slice — `wasm_threads_timer_after_recv_e2e`; the glue's main-thread "service" instance is what re-enters wasm `send` from the `setTimeout` callback.)*
  - **Sequential leg (`wasm_browser` / `wasm_wasi` default) — DEFERRED post-v1.** On single-threaded wasm, block-then-resume is *physically impossible* without a task-suspension mechanism: control must unwind out of wasm to the JS event loop (so the host callback can fire) and re-enter — and you cannot spin-pump the event loop from inside a synchronous main-thread wasm call (`Atomics.wait` traps on the main thread; timers/microtasks don't run while the stack is busy). The tree has **no** such mechanism (no `wasm-opt`/asyncify, no JSPI, and the CPS full-hybrid transform is deferred per design.md § State-Machine Transform). So `seq_scheduler.rs::drive_until_terminal`'s "hook point" cannot resume a parked native stack — there is nothing to park *with*. **Rationale for deferring rather than building asyncify now:** (1) not a launch gate (backend-first; the flagship demos are idle-holder/Parallax/pipeline, none a browser event-driven UI); (2) the front-end **demo path is `--features wasm-threads`** — it honors the contract today and COOP/COEP headers are trivial on a host we control (Cloudflare Pages), so the demo does **not** force this leg; (3) asyncify is a *permanent* per-binary tax (instrumentation + a transitive reachable-from-spawn-can-reach-`recv` allowlist fed to `wasm-opt`) plus a new Binaryen toolchain dep, for a feature whose only producers (the entry below) don't exist yet. The narrow case this leg actually serves is **the default sequential target on hosts where the deployer cannot set COOP/COEP** (e.g. someone copy-pasting a demo onto a bare static host) — a v1.x/v2 *distribution* concern, not a demo/launch one. **When it surfaces:** reconfirm JSPI's runtime-support matrix first — if JSPI has shipped unflagged by then it is the cleaner default (no instrumentation tax); asyncify is the "works everywhere today" fallback. Deciding the mechanism now would lock in the heavier option prematurely. The companion correctness gate (so the deferral never *silently* mis-executes) is the entry immediately below.

- [x] **Sequential-wasm host-async `recv` — hard compile error, not a silent lie.** *(SHIPPED 2026-06-07, with the `after` thin slice below.)* Companion to the deferred sequential leg above.
