What the entry actually demanded — *prove the provider cross-compiles to every v1 target, with a CI gate, before depending on it* — is done:
  - **Matrix + gate:** `runtime/tls-cross-probe/{std,embedded}` (workspace members, so they pin the runtime's exact rustls/ring versions via the shared lockfile) + CI `tls-cross` job; the pre-existing CI `cross` job additionally builds `-p karac-runtime` now — before this it only built the root package, whose rustls dep is dev-only, i.e. **the cross job compiled zero TLS code**.
  - **Per-target results** (local = verified on macOS M5 Pro 2026-06-05; CI = gated in `ci.yml`): macOS aarch64 ✅ local (full runtime `tls` tree) · macOS x86_64 ✅ local + CI `tls-cross` leg · Linux x86_64 ✅ CI `test` (native, default `tls`) · Linux aarch64 + armv7 ✅ CI `cross` (runtime build added) · Windows x86_64 ✅ CI `test` (native) · **wasm32-unknown-unknown ✅ local probe** — needs exactly three target-gated features: getrandom `js` (Web Crypto entropy), ring `wasm32_unknown_unknown_js` (SecureRandom impl gate), rustls-pki-types `web` (`UnixTime::now` via web-time) + a wasm-capable C compiler for ring's build.rs (ubuntu clang: stock; macOS: brew LLVM, Apple clang lacks the backend) · **wasm32-wasip1 ✅ local probe** — compiles as-is · **thumbv7em-none-eabihf (Cortex-M) ⏳ CI** — no_std rustls + getrandom `custom` resolve cleanly; ring's C build needs the arm-none-eabi newlib sysroot (locally absent — fails at `'assert.h' file not found`), CI leg installs `gcc-arm-none-eabi`.
  - **Scope boundary:** these probes gate the *provider*; the full `karac-runtime` can't compile on wasm32/bare-metal regardless (mio/socket2/tokio-net `compile_error!` on those targets) — that's the separate WASM-runtime work below, and the probes' feature recipes transfer to `runtime/Cargo.toml` when it lands.

- [x] **`std.web` module — browser/host effect resources.** *(2026-06-05)* Declared per `design.md § Web / Host Effect Vocabulary`, and landed the **first gated (non-prelude) stdlib mechanism** to make "native-only compilations must never see these resource names" real — no gating existed before this:
  - **Sources:** `runtime/stdlib/web.kara` (`Display`, `Storage`, `Console`, `Timer`, `Input` as ordinary `effect resource` decls) + `runtime/stdlib/web_net.kara` (`fetch(url) -> Result[Response, HttpError] with sends(Network) receives(Network)` — reuses the prelude `Network` resource, return shape mirrors `Client.get`; body is an honest stub until Q2 `host fn` wires host-call lowering).
  - **Gating mechanism:** `GATED_STDLIB_SOURCES` / `GATED_STDLIB_PROGRAMS` in `src/prelude.rs` — baked like `STDLIB_SOURCES` but with explicit module paths and ZERO scope-0 / `PRELUDE_*` / `register_baked_stdlib` registration. `build_program_tree` splices one `is_synthetic` module per entry (`std.web`, `std.web.net`), so `import std.web.{Display, ...};` is the only path into user scope. Three pipeline integrations: single-file mode expands gated imports into the real items pre-resolve (`prelude::expand_gated_stdlib_imports` — without it the blind import binding ICE'd in the interpreter at first use), project codegen appends imported gated items to the super-program (synthetic modules are skipped there), per-module typecheck already chases the tree.
  - **Resolver hardening (surfaced + fixed in-slice):** `resolve_effect_verb` was kind-blind — ANY in-scope symbol satisfied a verb clause, so unimported `writes(Display)` silently resolved against the prelude fmt **trait** `Display`, making the gate hollow for every colliding name. Now only `EffectResource` / `Import` / dotted-`Module` symbols pass; everything else gets `'X' is not an effect resource (it is a trait/struct/…)` with import guidance. Full suite green after the tightening (70 suites).
  - **Verified end-to-end** (debug + llvm builds): project + single-file × {with import → resolves/typechecks/runs/AOT-compiles, incl. the `fetch` stub body executing}, {without import → `undefined effect resource` / kind diagnostic}, alias form (`import std.web.Display as Screen` + `writes(Screen)`), repl accepts gated imports. Tests: `tests/module_graph.rs § Phase-10` (6) + `tests/cli.rs § Phase-10` (2, incl. the single-file ICE regression).
  - **Alias note for the target-gate entry below:** import-alias rename keeps effect-set *string* identity (`writes(Screen)` is the string "Screen" in both single-file and project mode — consistent, matches how effect sets identify resources today). The future target-provided-resource-set pass must canonicalize aliased resource names before keying the provided-set table, or `Screen` would evade a `Display` gate; cross-referenced in that entry.

- [x] **`std.wasi` module — headless WASM effect surface.** *(2026-06-05)* `runtime/stdlib/wasi.kara`, third entry in `GATED_STDLIB_SOURCES` — rode the `std.web` gating mechanism unchanged. Re-exports `FileSystem`, `Env`, `Network`, `Clock`, `RandomSource`; "re-export" is implemented as same-name redeclaration, which IS identity-preserving because effect sets identify resources by clause string (prelude resource shadowing is sanctioned — `process.kara` on `ProcessTable`). Pinned behaviors (`tests/module_graph.rs` ×3, `tests/cli.rs` ×1): tree splice, import-over-scope-0 resolves cleanly (project + single-file expansion paths), and unimported prelude resources keep working.

- [x] **`host fn` item — parser + resolver + typechecker.** *(2026-06-05)* Implemented end to end with a deliberately thin representation: `host fn` parses into the existing `ExternFunction` AST with the `"host"` **ABI sentinel** — no new item kind. That one choice made most of the entry free: resolver symbol visibility, effect-record registration (`register_extern_function_effects`), typechecker signature registration, AND native codegen (declares externs by signature, never reads `.abi`) all work unchanged; the no-ABI-default rule falls out of the abi match's `_ => {}` arm. Spoof-proofing: `unsafe extern "host" { }` blocks are rejected at parse ("not an ABI — use `host fn`"). Landed pieces: `host` as a **contextual** keyword (deliberate deviation from this entry's "reserve `host`" — a hard keyword broke `create_server(host: String, port: u16)` in the existing suite, and `host` is the most common networking parameter name in the backend programs v1 targets; the `test`-item 2-token-lookahead precedent applies cleanly: only `host` + `fn` at item position dispatches, recorded in syntax.md § 3.16); `parse_host_function` with the dedicated missing-`with` diagnostic (exact tracker wording), generics rejection, and no-body rejection; typechecker `check_host_fn_boundary` (primitives / `Copy`-satisfying / opaque-handle newtypes = single primitive-field struct, `Copy` derive NOT required so handles can carry `Drop`-based release; targeted diagnostics for `ref` / `mut ref` / owned non-`Copy`; raw pointers pass — the unsafe contract is the programmer's, same as `extern "C"`); `extern "C"` paths explicitly exempt. Tests: parser ×7, typechecker ×4, effectchecker ×3.
  - **Surfaced + fixed (pre-existing, all-fns scope):** `parse_effect_list`'s Verb arm iterated `verb.resources` — bare verbs (`blocks`, `suspends`, `panics`) carry an empty resource list, so a bare verb in ANY `with` clause was **silently dropped**: `pub fn f() with blocks { <blocking body> }` was rejected as "performs blocks() but does not declare it", and (worse, silent) callers of a declared-`blocks` signature whose body doesn't itself block — every `#[compiler_builtin]` stdlib stub like `Child.wait`, plus any `host fn` — never inherited the effect. Fixed by registering empty-resource effects for bare verbs, mirroring the extern-ABI-default shape; full suite green after.
  - **Doc fix:** design.md § Host Functions and syntax.md § 3.16 examples showed `fetch_json(url: ref String) -> Result[Bytes, HttpError]` — violating their own restriction list. Replaced with a compliant `(ptr, len) -> RequestHandle` form and a note that the `ref String` ergonomic surface is the library wrapper, not the `host fn`.

- [x] **Parser + resolver: `#[target(...)]` attribute.** *(2026-06-05)* New `src/target.rs` owns the closed set, spec evaluation, and the absence machinery; `CURRENT_TARGET = "native"` is the single source of truth the future `--target` flag swaps. Landed:
  - **Parse-level validation** (`parser/attributes.rs`): closed-set check incl. inside `not(...)` — note `not` is the logical-not KEYWORD, so `not(gpu)` arrives as a unary expr, not a call; empty-args rejection; **mixed positive+negative lists rejected** (spec defines no semantics for the combination — "no general boolean logic" enforced literally); multiple `#[target]` per item rejected with merge guidance. `target` added to `RECOGNIZED_BARE_ATTRIBUTES`.
  - **Absence semantics**: `target::filter_inactive_items` strips gated-away items (bodies included — they may reference target-specific names) before ANY pass runs; wired at both program entries — `cli::Pipeline::resolve` (single-file) and `module::build_program_tree` (project, per-module with tombstones merged onto `ProgramTree.target_tombstones`). Downstream passes (typecheck/effects/interp/codegen) never see inactive items.
  - **Targeted diagnostics**: tombstones (name → rendered spec) answer references at BOTH the unqualified-use site (`error_undefined_name`) and the project import site (`collect_import` unknown-item path, which also skips the binding so later uses get the same targeted message): `'X' is not available on target `native` — it is gated to `#[target(wasm_browser)]``.

- [ ] **WASM SIMD-128 lowering.** Lower `Vector[T, N]` operations to WebAssembly `v128` instructions on `wasm_browser` and `wasm_wasi` targets. `Vector[f32, 4]`, `Vector[i32, 4]`, `Vector[i16, 8]`, `Vector[i8, 16]`, `Vector[f64, 2]` map to single `v128` instructions; wider Kāra vectors split across multiple `v128` ops (tier-2 fallback). Honor `#[require_simd]` on WASM the same way as native — diagnostic fires on scalar fallback. Host environments without SIMD-128 support fall through to scalar, per the portable-by-guarantee rule. Browser-playground perf benchmarks depend on this being a first-class lowering target, not an afterthought. Cross-reference `design.md § Portable SIMD — Vector[T, N]` (auto-fallback rule explicitly names wasm-simd-128 as tier 1) and `brainstorming/archive/v67_simd_strategy.md § 5.1`.

- [ ] **WASM concurrency lowering — sequential default.** On `wasm_browser` and `wasm_wasi` without `--features wasm-threads`, lower `go` / channel / `par` to cooperative sequential scheduling on the main thread. A `go { ... }` spawn enqueues the closure; the scheduler interleaves ready closures and yields to the host event loop on channel-receive blocks so Promise-based host APIs continue to resolve. No Web Workers, no SharedArrayBuffer, no atomics, no COOP/COEP dependency. Verify source-level `go`/channel/`par` semantics match the native lowering — same program behaviour modulo scheduling.

- [ ] **WASM concurrency lowering — `--features wasm-threads` opt-in.** Add `--features wasm-threads` build flag. When active on `wasm_browser`, lower `go`/channel/`par` to Web Worker pool + SharedArrayBuffer + atomics. Pool size defaults to `navigator.hardwareConcurrency` at startup, with a build-manifest override. Runtime graceful degradation: when loaded without SAB available (deployment missed COOP/COEP headers), emit a `console.warn` and fall back to single-worker + main-thread scheduling unless the build manifest explicitly opts out of fallback. Channels transfer ownership across worker boundaries per the source-level semantics — use SAB-backed queue with atomic head/tail.

- [ ] **Scheduler — yield-to-event-loop on WASM channel-receive.** Implement the WASM scheduler contract: `go`-task performing `channel.recv()` on a channel whose pending sender is a host-async operation yields to the browser event loop. Host-async completion (Promise resolution via registered `.then`, `setTimeout` firing, DOM event dispatch) posts to the scheduler's ready queue, which wakes the waiting `go`-task on the next tick. Source-level semantics match native (function appears to block on receive). Verify no user-observable difference from native except scheduling. Applies to both sequential-default lowering and `--features wasm-threads` lowering from Q5 — the yield-to-event-loop behavior is orthogonal to whether the underlying scheduling is single-threaded or multi-worker.

- [ ] **`std.web.time.after(duration) -> Channel[()]` and event-stream wrappers.** Ship the library-layer wrappers that convert callback-driven host APIs into channel producers: `time.after(duration)` (setTimeout), `time.every(duration)` (setInterval), `events.clicks(element)` / `events.keydown(element)` / etc. (DOM events), `time.animation_frames()` (requestAnimationFrame). Each returns a `Channel[T]` with a library-managed sender wired to the underlying `host fn` callback registration. Lifetime: closing the channel deregisters the host listener.

- [ ] **WASM entry-point discovery.** A public function is exported in a WASM build when it is `pub` AND (it is target-tagged `#[target(wasm_browser)]` / `#[target(wasm_wasi)]` matching the build target, OR it is target-agnostic and transitively reachable from a target-tagged entry point). Standard dead-code elimination applies. No separate `#[export]` attribute. For `--bindings browser`, verify exported parameter/return types satisfy the `host fn` boundary restriction (primitives / `Copy` / opaque handles); emit a diagnostic on violation. For `--bindings component`, permit richer WIT-expressible types.

- [ ] **SSR provider-injection pattern — documentation and example.** Ship a worked SSR example in `docs/book/` (or `examples/`) showing the target-agnostic-component + per-target-provider pattern from `design.md § Cross-target Compilation`. The purpose is to keep users away from `#[target(...)]` in component bodies; the attribute is for entry points and irreducible forks only. Example should cover one component that renders to HTML on server and to DOM on client, with both entry points calling the same component function.

- [ ] **GPU routing decision — `#[gpu]` annotation design.** Decide how code is split between CPU and GPU. Recommendation: explicit `#[gpu]` attribute on functions, compiler verifies GPU-safety via effects (no `reads`, `writes`, `sends`, `receives`, `allocates(Heap)`, `panics`, no recursion). Layout groups (`group physics { position, velocity }`) map to GPU buffers. User declares intent, compiler enforces the rules. Design options: (A) fully automatic — too much magic, (B) explicit `#[gpu]` — recommended, (C) layout-driven — could complement B.
