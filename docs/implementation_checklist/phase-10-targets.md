## Phase 10: Additional Targets

- [ ] **v64 P0 — Cross-compile `aws-lc-rs` to all v1-supported targets.** The vendored TLS provider (rustls + aws-lc-rs default crypto) needs to cross-compile cleanly to every v1 target — Linux x86_64 / aarch64, macOS x86_64 / aarch64, Windows x86_64, WASM (browser + WASI), embedded ARM Cortex-M. aws-lc-rs has portable + assembly paths; assembly gives perf but requires per-target build configuration. Catch this during Phase 10 cross-compile work, not six months in. Includes: build-config matrix per target, fallback to portable crypto where assembly is unavailable, WASM-specific path (aws-lc-rs WASM support via `ring` provider fallback may be the right move on browser targets where size matters more than crypto perf), CI cross-compile gate covering all target × profile combinations. See [`design.md § v1 Positioning > backend platform stdlib`](../design.md#v1-positioning--backend-first) for the TLS provider rationale (graduated from v64 2026-05-09).

- [ ] **`std.web` module — browser/host effect resources.** Declare the capability-neutral resource set specified in `design.md § Web / Host Effect Vocabulary`: `Display`, `Storage`, `Console`, `Timer`, `Input`. Each as an ordinary `effect resource` declaration inside `std.web`. Also declare `std.web.net.fetch(url)` reusing the existing `Network` primitive (effect `sends(Network), receives(Network)`) — no browser-specific "Fetch" resource. Wire host-call lowering once Q2 (`host fn`) resolves. Module must be gated (not in prelude); native-only compilations must never see these resource names.

- [ ] **`std.wasi` module — headless WASM effect surface.** Thin v1 module: re-export `FileSystem`, `Env`, `Network`, `Clock`, `RandomSource` from built-in primitives for code that targets wasmtime / Component-Model hosts. Add WASI-specific resources (e.g., precision timers) only if the native primitive coverage proves insufficient. Same module-gating rule as `std.web`.

- [ ] **`host fn` item — parser + resolver + typechecker.** Add `HOST_FUNCTION` grammar to the parser per `syntax.md § 3.16`: `host fn IDENT ( PARAM_LIST ) [ -> TYPE ] EFFECT_LIST ;`. The `with` clause is **required** — emit a dedicated diagnostic if absent ("`host fn` must declare its effects; unlike `extern \"C\"`, no default effect set applies"). Reserve `host` keyword. Resolver: treat `host fn` like `extern "C"` for symbol visibility and effect-record registration. Typechecker: enforce parameter/return-type restriction — permit only primitives, `Copy`-satisfying types, and opaque-handle newtypes (user-defined struct with exactly one primitive-typed field). Reject owned non-`Copy`, `ref T`, `mut ref T`, and generic `host fn` with targeted diagnostics. No body to check.

- [ ] **`host fn` lowering — native target.** On native, lower `host fn foo(...) with ...;` to the same code path as `extern "C" fn foo(...) with ...;`. Effect declarations are trusted (same rule as `extern "C"`). This is the cheapest target and should be implementable first as a sanity baseline.

- [ ] **`host fn` lowering — browser-WASM target.** Emit a WASM `import` entry per `host fn` declaration using a wasm-bindgen-compatible calling convention so existing JS tooling (vite, webpack, esbuild) works without custom loaders. Generate a companion JS glue file mapping `host fn` parameter/return types to/from the JS side: primitives passed by value, opaque handles passed as i32, length-prefixed UTF-8 strings passed as `(ptr, len)` pairs. Wire into the `--target wasm --bindings browser` build output once Q4 resolves.

- [ ] **`host fn` lowering — server-WASM (WASI / Component Model) target.** Until WIT / Component Model runtime support is stable, lower `host fn` on server-WASM to a generated C-ABI call + thin shim layer. When Component Model support is production-ready, swap the lowering to emit WIT-backed Component Model calls. The source-level `host fn` surface must remain stable across this migration.

- [ ] **Effect-checker: target-provided-resource-set parameterization.** Implement the target→provided-resource-set table per `design.md § Cross-target Compilation`. After effect inference, run a target-gate pass per target in the build matrix: reject any function whose transitive effect set references a resource the target does not provide. Diagnostic format follows the `#[gpu]` call-graph style — include the chain from entry point to offending resource use, tagged with the target that produced the error. Target set for v1: `native`, `wasm_browser`, `wasm_wasi`, `gpu`.

- [ ] **Parser + resolver: `#[target(...)]` attribute.** Recognize `#[target(...)]` with the argument grammar per `syntax.md § 8`: comma-separated target names, each optionally wrapped in `not(...)`. No general boolean logic. Validate target names against the closed v1 set (`native`, `wasm_browser`, `wasm_wasi`, `gpu`); unknown names are a parse-level diagnostic. Reject multiple `#[target(...)]` attributes on the same item with a "merge into a single attribute" fix-it. An item with `#[target(...)]` that does not match the current compilation target is treated as absent at resolution time — references to it from active code emit "item not available on target X" diagnostics at the call site.

- [ ] **`karac check` multi-target verification.** When a package's build configuration declares multiple targets, `karac check` runs the type-check + effect-check pipeline once per target, each with the target's provided-resource-set. Tag each diagnostic with the target that produced it. Deduplicate identical diagnostics that appear on every target (they're target-agnostic bugs, not target-specific). Finite cost — the target set is closed at four.

- [ ] **`karac build --bindings` flag.** Add `--bindings browser|component|none` to the WASM target build path. Default `--bindings` is inferred from `--target`: `wasm_browser` → `browser`, `wasm_wasi` → `component`. Omitting `--bindings` on a non-WASM target is ignored. Unknown `--bindings` value is an error with the valid-set listed.

- [ ] **WASM browser artifact emission.** Emit `dist/wasm/<pkg>.wasm` + `<pkg>.js` + `<pkg>.d.ts` when `--bindings browser` is active. JS glue uses wasm-bindgen-compatible calling convention so default vite/webpack/esbuild/rollup loaders work without custom configuration. `.d.ts` declares every exported function's signature on the JS side, including `Result`/`Option` shapes and any exported struct. Package name comes from `kara.toml`.

- [ ] **WASM Component Model artifact emission.** When `--bindings component` is active, call into pinned `wit-bindgen` (version pinned in `kara.toml` toolchain section) to produce either a single `dist/wasm/<pkg>.wasm` with embedded WIT, or — until runtime support is broadly stable — a paired `<pkg>.wasm` core module + `<pkg>.component.wit` descriptor. When the embedded-WIT path becomes the default, emit a one-release deprecation notice for the paired form so downstream tooling can migrate.

- [ ] **WASM raw artifact emission.** When `--bindings none` is active, emit only `dist/wasm/<pkg>.wasm`. No glue, no declarations. For users wrapping Kāra WASM with custom host integration.

- [ ] **WASM SIMD-128 lowering.** Lower `Vector[T, N]` operations to WebAssembly `v128` instructions on `wasm_browser` and `wasm_wasi` targets. `Vector[f32, 4]`, `Vector[i32, 4]`, `Vector[i16, 8]`, `Vector[i8, 16]`, `Vector[f64, 2]` map to single `v128` instructions; wider Kāra vectors split across multiple `v128` ops (tier-2 fallback). Honor `#[require_simd]` on WASM the same way as native — diagnostic fires on scalar fallback. Host environments without SIMD-128 support fall through to scalar, per the portable-by-guarantee rule. Browser-playground perf benchmarks depend on this being a first-class lowering target, not an afterthought. Cross-reference `design.md § Portable SIMD — Vector[T, N]` (auto-fallback rule explicitly names wasm-simd-128 as tier 1) and `brainstorming/archive/v67_simd_strategy.md § 5.1`.

- [ ] **WASM concurrency lowering — sequential default.** On `wasm_browser` and `wasm_wasi` without `--features wasm-threads`, lower `go` / channel / `par` to cooperative sequential scheduling on the main thread. A `go { ... }` spawn enqueues the closure; the scheduler interleaves ready closures and yields to the host event loop on channel-receive blocks so Promise-based host APIs continue to resolve. No Web Workers, no SharedArrayBuffer, no atomics, no COOP/COEP dependency. Verify source-level `go`/channel/`par` semantics match the native lowering — same program behaviour modulo scheduling.

- [ ] **WASM concurrency lowering — `--features wasm-threads` opt-in.** Add `--features wasm-threads` build flag. When active on `wasm_browser`, lower `go`/channel/`par` to Web Worker pool + SharedArrayBuffer + atomics. Pool size defaults to `navigator.hardwareConcurrency` at startup, with a build-manifest override. Runtime graceful degradation: when loaded without SAB available (deployment missed COOP/COEP headers), emit a `console.warn` and fall back to single-worker + main-thread scheduling unless the build manifest explicitly opts out of fallback. Channels transfer ownership across worker boundaries per the source-level semantics — use SAB-backed queue with atomic head/tail.

- [ ] **Scheduler — yield-to-event-loop on WASM channel-receive.** Implement the WASM scheduler contract: `go`-task performing `channel.recv()` on a channel whose pending sender is a host-async operation yields to the browser event loop. Host-async completion (Promise resolution via registered `.then`, `setTimeout` firing, DOM event dispatch) posts to the scheduler's ready queue, which wakes the waiting `go`-task on the next tick. Source-level semantics match native (function appears to block on receive). Verify no user-observable difference from native except scheduling. Applies to both sequential-default lowering and `--features wasm-threads` lowering from Q5 — the yield-to-event-loop behavior is orthogonal to whether the underlying scheduling is single-threaded or multi-worker.

- [ ] **`std.web.time.after(duration) -> Channel[()]` and event-stream wrappers.** Ship the library-layer wrappers that convert callback-driven host APIs into channel producers: `time.after(duration)` (setTimeout), `time.every(duration)` (setInterval), `events.clicks(element)` / `events.keydown(element)` / etc. (DOM events), `time.animation_frames()` (requestAnimationFrame). Each returns a `Channel[T]` with a library-managed sender wired to the underlying `host fn` callback registration. Lifetime: closing the channel deregisters the host listener.

- [ ] **WASM entry-point discovery.** A public function is exported in a WASM build when it is `pub` AND (it is target-tagged `#[target(wasm_browser)]` / `#[target(wasm_wasi)]` matching the build target, OR it is target-agnostic and transitively reachable from a target-tagged entry point). Standard dead-code elimination applies. No separate `#[export]` attribute. For `--bindings browser`, verify exported parameter/return types satisfy the `host fn` boundary restriction (primitives / `Copy` / opaque handles); emit a diagnostic on violation. For `--bindings component`, permit richer WIT-expressible types.

- [ ] **SSR provider-injection pattern — documentation and example.** Ship a worked SSR example in `docs/book/` (or `examples/`) showing the target-agnostic-component + per-target-provider pattern from `design.md § Cross-target Compilation`. The purpose is to keep users away from `#[target(...)]` in component bodies; the attribute is for entry points and irreducible forks only. Example should cover one component that renders to HTML on server and to DOM on client, with both entry points calling the same component function.

- [ ] **GPU routing decision — `#[gpu]` annotation design.** Decide how code is split between CPU and GPU. Recommendation: explicit `#[gpu]` attribute on functions, compiler verifies GPU-safety via effects (no `reads`, `writes`, `sends`, `receives`, `allocates(Heap)`, `panics`, no recursion). Layout groups (`group physics { position, velocity }`) map to GPU buffers. User declares intent, compiler enforces the rules. Design options: (A) fully automatic — too much magic, (B) explicit `#[gpu]` — recommended, (C) layout-driven — could complement B.
