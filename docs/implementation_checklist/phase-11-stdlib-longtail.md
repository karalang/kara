- [ ] Type system: add `f16` (IEEE 754-2008 half-precision) and `bf16` (bfloat16) as primitive types with the same trait surface as `f32`/`f64` (`PartialEq`, `PartialOrd`, arithmetic traits, `Copy`) but NOT `Eq`/`Ord`/`Hash`.
- [ ] Codegen: lower `f16` → LLVM `half`, `bf16` → LLVM `bfloat`. Native instruction emission on capable targets; software promotion to `f32` on others with a `f16_software_emulated` performance lint.
- [ ] Implicit widening: `f16` → `f32`, `bf16` → `f32` (both lossless).
- [ ] Literal suffixes: `1.0f16`, `1.0bf16`.
- [ ] Stdlib: `F16`, `BF16` total-order wrappers (same pattern as `F32`/`F64`).
- [ ] `Tensor[f16, Shape]` and `Tensor[bf16, Shape]` valid once both this and the numerical stdlib ship.

**Scalar float math (`std.math`).** The per-`f64`/`f32` transcendental + utility surface, distinct from the Tensor stack below.

- [x] **Scalar transcendental + rounding math — `sin` / `cos` / `tan` / `atan2` / `exp` / `ln` / `log2` / `pow` / `floor` / `ceil` / `round`.** ✓ DONE (2026-06-22). **Surface decision: value-receiver float methods** (the shipped `x.abs()` / `x.sqrt()` shape), *not* a `std.math` free-fn module — extends the proven `sqrt` path. The eleven methods are classified by a single shared table, `src/float_math.rs` (`classify` → `FloatMathKind::{Unary, Binary}`), keyed by all three backends so the surface can't drift between `karac run` and `karac build`: unary `sin`/`cos`/`tan`/`exp`/`ln`/`log2`/`floor`/`ceil`/`round` (`x.m() -> Self`) and binary `pow`/`atan2` (`x.m(y) -> Self`, one argument of the same float type; a suffix-free *float* literal arg promotes to the receiver type per Q4, an integer literal does not). Both float widths. **Codegen** (`src/codegen/method_call.rs`) lowers most to the overloaded LLVM intrinsic (`llvm.sin`/`llvm.cos`/`llvm.exp`/`llvm.log`/`llvm.log2`/`llvm.pow`/`llvm.floor`/`llvm.ceil`/`llvm.round`); **`tan` and `atan2` are the exceptions** — `llvm.tan`/`llvm.atan2` are LLVM-19+ and absent on the 18.1 pin (confirmed: `Intrinsic::find("llvm.tan")` returns `None`), so they lower to a direct width-correct libm call (`tan`/`tanf`, `atan2`/`atan2f`). **The wasm-libm link story was a non-issue:** these all become libm calls (no single wasm instruction for trig), and the math symbols already live in wasi-libc's `libc.a` — which the `wasm-ld` path already links — so **no archive rebuild, no `--export` flag, no link change** was needed (verified: `karac build --target=wasm_wasi` produces a `.wasm` that runs byte-identical under `wasmtime`). **Interpreter** (`src/interpreter/method_call.rs`) delegates to Rust's `f64::*` (`powf`/`atan2` for the binary forms). Float-only — an integer receiver (incl. existing integer `.pow(u32)`) and user types with same-named methods fall through to normal dispatch. `round` is half-away-from-zero (Rust `f64::round` ≡ `llvm.round` ≡ libm `round`). Tests: `tests/typechecker.rs` (return-`Self` on both widths, binary arg-type strictness + float-literal promotion, arity errors, integer-receiver rejection), `tests/interpreter.rs::test_float_math_transcendental_and_rounding`, `tests/codegen.rs::{e2e_float_math_transcendental, e2e_float_math_f32}` (exact-result inputs → platform-independent assertions matching the interpreter twin; AOT output verified byte-identical to `karac run` and to the wasm build). **Driven by the Plume flow-field dogfood** (`examples/plume/`), which wanted curl-noise / angular fields and had to be built from rational vortices + a noise LIC precisely because no trig existed yet. *(Tensor-level activations `relu`/`sigmoid`/`tanh`/`softmax` and `std.simd.math` polynomial approximations remain their own P1 rows below — this is the scalar surface only.)*

A `ref T` is a shared read borrow — the language forbids writing through it — so the callee performs no stores through the pointer, which is LLVM's `readonly` contract. The one hole is interior mutability (`Atomic[_]`/`Mutex[_]` and a `shared` type's per-field runtime borrow flags mutate through a shared reference, design.md § Part 5), so the attribute is gated on a conservative transitive **Freeze** predicate (`ref_referent_is_freeze`): primitives / `String` / all-Freeze structs / Freeze-element containers (`Vec`/`Option`/`Result`/`Map`/`Tensor`/…) qualify; anything carrying `Atomic`/`Mutex`/`Weak`/a `shared` type — or an unknown/opaque name, an unresolved generic param, a raw pointer, or a `mut ref`/`mut Slice` field — fails *closed* (no attribute).   - [x] **`!alias.scope`/`!noalias` on slice-param loads/stores** (slice-alias slice 4, 2026-07-11). The loop-kernel `restrict` win — a `mut Slice[T]` / `Slice[T]` param's pointer is a *field* of the by-value `{ptr,len}` fat struct, not the parameter, so param-level `noalias` can't reach it; the disjointness is instead lowered onto the element loads/stores as LLVM scoped-alias metadata. A `mut Slice` is an *exclusive* borrow (disjoint from every other slice param, the same guarantee `mut ref`→noalias rests on); a shared `Slice` is disjoint from every exclusive one but may alias another shared slice. Encoding (`src/codegen/slice_alias.rs`, new module): one distinct alias scope per exclusive param; an exclusive param's accesses get `!alias.scope !{Sᵢ}` + `!noalias !{other exclusive Sⱼ}`; a shared param's accesses get `!noalias !{all exclusive Sᵢ}` (no scope of its own). Emitted only when ≥1 exclusive slice param AND ≥2 slice params total (a lone slice has nothing to disambiguate → no metadata). Distinct domain/scope nodes built with the temp+RAUW self-referential recipe (mirrors the `llvm.loop` id in control_flow_bce.rs); attached at the element load in `compile_slice_index` and the store in `compile_slice_index_store`. `build_slice_alias_scopes` runs from BOTH `compile_function` and `compile_mono_function` (after param registration), so generic slice kernels are covered; a bare `mut Slice[T]`/`Slice[T]` with generic `T` is identified structurally. **Shadow guard**: `register_var_from_type_expr` drops a name from the scope map on any (re-)binding, so a local slice shadowing a slice-param name never inherits the param's `!noalias` (fail closed). **Soundness** rests entirely on the borrow checker: a program that aliased an exclusive slice with anything else is unrepresentable in valid Kāra, so the `!noalias` claim cannot be violated by a well-typed program (and the O2 AOT E2E suite exercises the metadata on every existing slice kernel). Tests: `tests/codegen.rs::{slice_alias_mut_and_shared_params, slice_alias_two_mut_params_mutual, slice_alias_single_param_gets_none, slice_alias_in_monomorph}`.   - [x] **`tbaa` type-based alias tags — RETIRED as not-applicable (2026-07-11).** Investigated and deliberately **not shipped**: general C-style TBAA ("accesses of different types never alias") is **unsound for Kāra's memory model**, the same reason rustc emits essentially no type-based TBAA. (1) design.md § 8878 "Tier 0" specifies only `noalias` / scoped-alias — never TBAA — and all of that is now shipped (slices 1–4 above). (2) Kāra adopts **strict-provenance** (design.md § Pointer Provenance) and *permits type punning in `unsafe`*: C `union`s "reinterpret the bytes on every access" (design.md § Unions), `transmute`, and raw-pointer `as` casts all alias one address under two types; there is **no strict-aliasing rule**, so a type-based no-alias claim can be violated by a well-typed program. (3) Even a safe-code-only scalar-TBAA subset hits the tagged-enum-payload hazard (the same offset is read as type A in one arm and written as type B in another — a genuine same-address/different-type pair in safe code). (4) design.md § 8878's explicit **soundness boundary** — "an over-broad [alias fact] is a *silent miscompilation*, never a slowdown … Tier 0 ships only behind a differential-equivalence [fuzzed] guarantee" (citing Rust's multi-year `-Zmutable-noalias` saga) — means any TBAA would first require that fuzzed differential corpus, which does not exist. A future *sound-subset* investigation (invariant-load tagging of genuinely-immutable metadata; punning-aware struct-path TBAA gated on the fuzz corpus) is tracked in [`../deferred.md § Type-Based Alias Analysis (TBAA)`](../deferred.md#type-based-alias-analysis-tbaa). The row's design goal — ownership-derived Tier-0 alias facts — is fully met by the `noalias` + scoped-alias work.
