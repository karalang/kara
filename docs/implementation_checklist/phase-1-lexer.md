## Phase 1: Lexer

- [x] **Lexer: float literal exponent notation.** Extend `number()` in `src/lexer.rs` (lines 339–378) to accept scientific-notation exponents on float literals: `1e10`, `1.5e-3`, `6.022e23`, `2.5E+6`, `1_000e1_000`, and suffixed forms like `2.5E+6f32`. Grammar per `syntax.md § 1.3`: `FLOAT_EXP = ("e" | "E") ["+" | "-"] DIGIT { DIGIT | "_" }`. A token with no decimal point and no exponent stays an integer; any exponent makes it a float. Underscores permitted in both mantissa and exponent digits. Add lexer tests covering each form plus edge cases (`1e`, `1e+`, `1.e10` — the last is a float `1.0e10`). No changes needed downstream — existing `f64`/`f32` handling already covers the values; only token production changes.

- [x] **Lexer: unicode identifier support.** Per design.md § Identifiers — Unicode, non-ASCII identifiers are deferred (P2); the v1 lexer must reject them with a focused diagnostic rather than producing a per-byte cascade of garbage from interpreting UTF-8 continuation bytes as Latin-1. Implemented two related pieces: (1) **identifier path** — `non_ascii_at_lead` recovery on the top-level dispatch (letter codepoint → consume the full would-be identifier, emit one `non-ASCII identifier (containing 'X') is deferred to a future edition` error; non-letter codepoint like `🚀` → single `Unexpected character` error naming the actual codepoint). `identifier()` extends its continue-loop to consume non-ASCII letter codepoints so mid-word non-ASCII (`kāra`) yields one error spanning the whole token. (2) **literal-body paths** — `consume_codepoint` helper replaces `self.advance() as char` in the bodies of `string`, `multi_string`, `char_literal`, and both branches of `interpolated_string` (text + expr), so a multi-byte codepoint inside a string/char literal lands as one `char` rather than a sequence of misinterpreted bytes (`"Kāra"` → `StringLiteral("Kāra")`, not `"K\xc4\x81ra"`-as-five-chars). Column tracking advances by codepoint, not byte, in both recovery and literal paths. Unknown-escape diagnostics also use `consume_codepoint` so the offending char is rendered correctly. Tests in `tests/lexer.rs` cover identifier-start (`αβγ`), mid-word (`kāra`), let-binding recovery, non-letter (`🚀`), string body, char literal (BMP and emoji), multi-string body, and interpolated-string text+expr segments. (From code review CR-31.)

- [x] **Lexer: `b'A'` byte character literal.** Shipped 2026-05-22. Lexer recognizes `b'X'` directly (the `b'b' if peek == b'\''` arm in `src/lexer.rs::scan_token`), handing off to a new `byte_char_literal()` method that accepts printable ASCII, the simple escapes `\n \t \r \0 \\ \' \"`, and `\xHH` (delegating to the pre-existing `parse_hex_byte_escape` helper). `\u{...}` is rejected with the design-specced diagnostic verbatim; multi-byte UTF-8 in the body rejects as `non-ASCII character in byte literal` (the spec's `E_NON_ASCII_IN_BYTE_LITERAL` shape); empty `b''` and multi-byte non-escape bodies (`b'AB'`) also reject. `Token::ByteLiteral(u8)` flows through `ExprKind::ByteLit(u8)` in the AST; typechecker assigns `Type::UInt(UIntSize::U8)`; codegen lowers as `i8`-width constant; interpreter evaluates to `Value::Int(b as i64)` (consistent with how integer-suffixed `u8` values are carried); formatter round-trips `b'A'`/`b'\n'`/`b'\xHH'` shapes. Bare identifier `b` followed by separated `'x'` (whitespace between) continues to lex as two tokens — the `b'` arm is whitespace-sensitive at the lex level. Follow-up slice (not blocking): `b"..."` byte-string form once the design.md:429 vs § Byte and Byte-String Literals divergence is resolved. - [x] **Lexer: raw-identifier escape `r#NAME`.** Spec at `design.md § Raw Identifiers (r#name)` and `syntax.md § 1.3` (v60 item 39). Slices: (1) Lexer dispatch — extend the `b'r'` arm in `src/lexer.rs` to peek at the next byte: `r"` continues into the existing reserved-string-prefix path (`error[E_RESERVED_STRING_PREFIX]`); `r#` followed by an identifier-start character (`[a-zA-Z_]`) enters the new raw-identifier path; `r#` followed by `"` falls into the reserved `r#"..."#` form (currently `error[E_RESERVED_HASH_STRING]`); any other `r`-led sequence falls through to ordinary identifier lexing. (2) Token shape — new `Token::Identifier { name: String, raw: bool, span: Span }` form (or extend the existing identifier token with a `raw: bool` flag). The `name` field stores the bare identifier without the `r#` prefix; `raw = true` records that the source written `r#`. Symbol tables, mangled names, and downstream consumers consume `name`; only the lexer / formatter look at `raw`. (3) Keyword bypass — when `raw = true`, the lexer **skips the keyword check** for `name`. So `r#async` produces `Token::Identifier { name: "async", raw: true }` rather than `Token::ReservedKeyword { name: "async" }`. The token then flows through the parser as a normal identifier. (4) Structural-marker rejection — names that cannot be raw-escaped (`self`, `Self`, `_`, `super`, `crate`, `mod`, `pub`, `priv`, `private`, `mut`, `ref`, `own` per design.md § Raw Identifiers) emit `error[E_RAW_IDENT_NOT_ALLOWED]: 'r#NAME' is not legal; NAME is a structural marker, not a reservable keyword` at lex time. (5) Case-class enforcement — the case class is determined by the identifier portion *after* the `r#` prefix. `r#async` → Value-class (lowercase first letter); `r#Async` → Type-class. The existing case-class enforcement (item 465 below — Phase 5.2) reads the identifier name without seeing the `raw` flag, so this falls out for free. (6) Formatter — `src/formatter.rs` emits `r#` prefix when `raw = true` on an identifier token. The discriminator is preserved across format round-trips. (7) Path segments — every segment of an attribute path / module path / generic-instantiation path may independently carry `r#`; the lexer already produces one `Identifier` token per segment, so the existing parser logic for paths needs no change beyond reading the token's `name` field. Test coverage: positive — `let r#async = 1` parses and binds a variable named `async`; `fn r#try() -> i32 { 0 }` parses and produces a function symbol named `try`; field name `r#move` parses; `obj.r#await` accesses field `await`; module-path segment `std.r#async_io` parses; nested generic `Foo[r#async]` parses. Negative — `r#self` rejected with `E_RAW_IDENT_NOT_ALLOWED`; `r#Self` rejected; `r#_` rejected; `r#mut` rejected; `r#pub` rejected; `r#`-prefix on a digit (`r#1foo`) falls into ordinary lex (rejecting because `r#` not followed by identifier-start = `r` token + `#` token + `1foo` → produces an existing diagnostic, no new error needed). Round-trip — `r#async` written by user, lexed, formatted, lexed again must produce the same token sequence. Phase target: 1 (Lexer); the formatter and round-trip tests fold into the same slice. Cross-reference: design.md § Reserved-for-Future-Use Keywords (refers to this entry as the forward-compatibility lever); design.md § Raw Identifiers; syntax.md §1.1 keyword section's bottom paragraph; syntax.md §1.3 IDENTIFIER + RAW_IDENT_PREFIX grammar.


- [x] **Lexer: f-string interpolation extractor is not string-aware. — FIXED 2026-06-13.** The `{…}` expression extractor in `interpolated_string()` (`src/lexer.rs` ~line 1084) balances **braces only** — it does not track string literals inside the interpolation. **FIX:** the brace loop now carries an `in_lit: Option<u8>` holding the open delimiter (`"` or `'`) while inside a nested string/char literal; braces inside such a literal no longer change `brace_depth`, and a `\X` inside the literal is copied as a verbatim *pair* so `\"`/`\'` don't close it early. `expr_text` stays a byte-for-byte verbatim copy (the B-2026-06-09-1 offset invariant `byte i == expr_start + i` is preserved — regression-tested). A backslash in *expression* position (the genuinely-invalid escaped-quote case) now emits a clear `Token::Error` ("unexpected '\\' in f-string interpolation `{…}`; … use plain quotes") instead of silently passing through as literal text. **Regression tests (landed):** `tests/lexer.rs` — `test_interpolated_string_brace_inside_nested_string` / `_open_brace_inside_nested_string` / `_brace_inside_nested_char` / `_verbatim_offset_preserved_with_nested_string` / `_escaped_quote_is_clear_error`. Verified end-to-end (interp + codegen) on `f"{ "a}b" }"` → `a}b`, `f"{ '}' }"` → `}`. — Original description follows. The `{…}` extractor balanced **braces only** — it did not track string literals inside the interpolation. Two consequences: (1) a brace inside a string literal inside an interpolation (`f"{ m["a}b"] }"`) miscounts depth and truncates the expression early; (2) an **escaped** quote inside the interpolation (`f"{ id(\"hi\") }"`) is copied verbatim into `expr_text`, which then fails to re-parse (a leading `\` is invalid in expression context) and the whole hole is silently emitted as literal text — *wrong output, no error*. **The natural syntax already works:** `f"{ id("hi") }"` (plain nested quotes, no escaping) lexes and evaluates correctly, because with no braces inside the string the brace-only balance happens to land right. **Constraint on the fix:** `expr_text` must stay a *verbatim* byte-for-byte copy of the source slice — the span-rebasing added for B-2026-06-09-1 (the f-string SpanKey-collision fix) relies on `byte i of expr_text == source offset expr_start + i` (see the comment at ~line 1072). So the fix cannot strip the backslashes; it must make the extractor string-aware *only for deciding when to stop counting braces / when a `"` opens-or-closes a nested string*, leaving the copied bytes untouched. The `\"`-in-interpolation case is genuinely invalid input (expression context wants plain `"hi"`); the right behavior there is a **clear diagnostic** ("string literals inside `{…}` use plain quotes — write `{f(\"hi\")}` as `{f(\"hi\")}`→`{f("hi")}`"), not silent literal passthrough. Surfaced by the Tangle dogfooding project (`examples/tangle/interp.kara`) — worked around in the example by binding the lookup to a local first. Low frequency; the plain-quote form is the idiom and works.
