## Phase 1: Lexer

- [x] **Lexer: float literal exponent notation.** Extend `number()` in `src/lexer.rs` (lines 339–378) to accept scientific-notation exponents on float literals: `1e10`, `1.5e-3`, `6.022e23`, `2.5E+6`, `1_000e1_000`, and suffixed forms like `2.5E+6f32`. Grammar per `syntax.md § 1.3`: `FLOAT_EXP = ("e" | "E") ["+" | "-"] DIGIT { DIGIT | "_" }`. A token with no decimal point and no exponent stays an integer; any exponent makes it a float. Underscores permitted in both mantissa and exponent digits. Add lexer tests covering each form plus edge cases (`1e`, `1e+`, `1.e10` — the last is a float `1.0e10`). No changes needed downstream — existing `f64`/`f32` handling already covers the values; only token production changes.

- [x] **Lexer: unicode identifier support.** Per design.md § Identifiers — Unicode, non-ASCII identifiers are deferred (P2); the v1 lexer must reject them with a focused diagnostic rather than producing a per-byte cascade of garbage from interpreting UTF-8 continuation bytes as Latin-1. Implemented two related pieces: (1) **identifier path** — `non_ascii_at_lead` recovery on the top-level dispatch (letter codepoint → consume the full would-be identifier, emit one `non-ASCII identifier (containing 'X') is deferred to a future edition` error; non-letter codepoint like `🚀` → single `Unexpected character` error naming the actual codepoint). `identifier()` extends its continue-loop to consume non-ASCII letter codepoints so mid-word non-ASCII (`kāra`) yields one error spanning the whole token. (2) **literal-body paths** — `consume_codepoint` helper replaces `self.advance() as char` in the bodies of `string`, `multi_string`, `char_literal`, and both branches of `interpolated_string` (text + expr), so a multi-byte codepoint inside a string/char literal lands as one `char` rather than a sequence of misinterpreted bytes (`"Kāra"` → `StringLiteral("Kāra")`, not `"K\xc4\x81ra"`-as-five-chars). Column tracking advances by codepoint, not byte, in both recovery and literal paths. Unknown-escape diagnostics also use `consume_codepoint` so the offending char is rendered correctly. Tests in `tests/lexer.rs` cover identifier-start (`αβγ`), mid-word (`kāra`), let-binding recovery, non-letter (`🚀`), string body, char literal (BMP and emoji), multi-string body, and interpolated-string text+expr segments. (From code review CR-31.)

- [x] **Lexer: raw-identifier escape `r#NAME`.** Spec at `design.md § Raw Identifiers (r#name)` and `syntax.md § 1.3` (v60 item 39). Slices: (1) Lexer dispatch — extend the `b'r'` arm in `src/lexer.rs` to peek at the next byte: `r"` continues into the existing reserved-string-prefix path (`error[E_RESERVED_STRING_PREFIX]`); `r#` followed by an identifier-start character (`[a-zA-Z_]`) enters the new raw-identifier path; `r#` followed by `"` falls into the reserved `r#"..."#` form (currently `error[E_RESERVED_HASH_STRING]`); any other `r`-led sequence falls through to ordinary identifier lexing. (2) Token shape — new `Token::Identifier { name: String, raw: bool, span: Span }` form (or extend the existing identifier token with a `raw: bool` flag). The `name` field stores the bare identifier without the `r#` prefix; `raw = true` records that the source written `r#`. Symbol tables, mangled names, and downstream consumers consume `name`; only the lexer / formatter look at `raw`. (3) Keyword bypass — when `raw = true`, the lexer **skips the keyword check** for `name`. So `r#async` produces `Token::Identifier { name: "async", raw: true }` rather than `Token::ReservedKeyword { name: "async" }`. The token then flows through the parser as a normal identifier. (4) Structural-marker rejection — names that cannot be raw-escaped (`self`, `Self`, `_`, `super`, `crate`, `mod`, `pub`, `priv`, `private`, `mut`, `ref`, `own` per design.md § Raw Identifiers) emit `error[E_RAW_IDENT_NOT_ALLOWED]: 'r#NAME' is not legal; NAME is a structural marker, not a reservable keyword` at lex time. (5) Case-class enforcement — the case class is determined by the identifier portion *after* the `r#` prefix. `r#async` → Value-class (lowercase first letter); `r#Async` → Type-class. The existing case-class enforcement (item 465 below — Phase 5.2) reads the identifier name without seeing the `raw` flag, so this falls out for free. (6) Formatter — `src/formatter.rs` emits `r#` prefix when `raw = true` on an identifier token. The discriminator is preserved across format round-trips. (7) Path segments — every segment of an attribute path / module path / generic-instantiation path may independently carry `r#`; the lexer already produces one `Identifier` token per segment, so the existing parser logic for paths needs no change beyond reading the token's `name` field. Test coverage: positive — `let r#async = 1` parses and binds a variable named `async`; `fn r#try() -> i32 { 0 }` parses and produces a function symbol named `try`; field name `r#move` parses; `obj.r#await` accesses field `await`; module-path segment `std.r#async_io` parses; nested generic `Foo[r#async]` parses. Negative — `r#self` rejected with `E_RAW_IDENT_NOT_ALLOWED`; `r#Self` rejected; `r#_` rejected; `r#mut` rejected; `r#pub` rejected; `r#`-prefix on a digit (`r#1foo`) falls into ordinary lex (rejecting because `r#` not followed by identifier-start = `r` token + `#` token + `1foo` → produces an existing diagnostic, no new error needed). Round-trip — `r#async` written by user, lexed, formatted, lexed again must produce the same token sequence. Phase target: 1 (Lexer); the formatter and round-trip tests fold into the same slice.
