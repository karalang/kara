- [ ] **TypeChecker: refinement type elision pass (v1).** Implement the two-rule elision procedure specified in design.md — Refinement Types, "Compile-time elision procedure (v1)". Ordered upgrade steps:
  1. [ ] **Const-eval hook.** When the typechecker encounters a binding or call-site argument whose expected type is a refined type, first try to reduce the initializer/argument via the existing const-evaluator. If reduction yields a concrete value `v`, proceed to step 2; otherwise skip to step 3.
  2. [ ] **Compile-time predicate evaluation.** Evaluate the refinement predicate against `v` at compile time. On success: emit no runtime check, no error path — the value is admitted. On failure: emit a structured compile error pointing at the initializer with the predicate and the offending value (not a runtime error — this is a deterministic, catchable build-time failure).
  3. [ ] **Type-identity check.** If the static type of the initializer is *exactly* the target refined type (same name, same predicate, same base), emit no runtime check.
  4. [ ] **Reject implicit runtime-value narrowing.** If neither step 2 nor step 3 applies and the target type is a refined type, emit a structured diagnostic: "cannot narrow `T` to `R` implicitly; use `R.try_from(x)?` or `x as R`". Do *not* silently fall back to a runtime check — the design explicitly forbids implicit runtime-value narrowing.
  5. [ ] **Reject implicit cross-refinement.** Same diagnostic category for refined-to-refined implicit coercions, even when the predicates are textually identical. v1 has no implication-based elision; all cross-refinement goes through explicit `try_from`/`as`.
  6. [ ] **Call-site application.** Apply the same four checks at function call argument positions — this is the same code path as binding initialization, not a separate rule. `f(80)` where `f: fn(ValidPort)` succeeds via rule 1; `f(n)` where `n: i32` fails via rule 4.
  7. [ ] **Generic-code guard.** Inside a generic function `fn f[T](v: T)`, never attempt elision against `T` — `T` is opaque. Elision applies only at the *caller's* call/return boundaries against the concrete instantiated type.
  8. [ ] **Diagnostic polish.** When rejecting per rules 4 or 5, suggest the explicit form (`R.try_from(x)?` for recoverable, `x as R` for asserting) and note that flow-sensitive narrowing is not supported so `if x > 0 { f(x) }` will not work.

- [ ] **Distinct types (newtypes).** Implement `distinct type Name = BaseType` with constructor syntax (`Name(val)`), `.raw()` unwrap, no inherited operations unless opted in via `#[derive]`. Interaction with refinement types: `distinct type ValidPort = u16 where self >= 1 && self <= 65535`. `Copy` is not inherited from the base type — must be explicitly derived.

- [ ] **Contracts (`requires`/`ensures`/`invariant`).** Implement contract annotations. `requires`/`ensures` on functions, `invariant` on struct fields. Runtime-checked in debug builds, stripped in release. Declaration ordering: signature → effects → requires → ensures → where → body. See design.md — "Contract evaluation semantics" for the four rules below. Ordered implementation steps:
  1. [ ] **Parse `old(expr)` as a special form inside `ensures` clauses.** Reject `old(...)` occurrences in `requires`, `invariant`, and outside contracts entirely. Reject `old(result)` with a pointed diagnostic.
  2. [ ] **Contract-expression effect-set check (`⊆ {panics}`).** After effect inference on the contract expression, verify the inferred set is a subset of `{panics}`. On violation, emit a compile error pointing at the offending call with message naming the forbidden effect (e.g., `"reads(FileSystem) not permitted in contract expression"`). The `panics` effect is permitted because indexing/division/unwrap are idiomatic in predicates; no other effect is permitted.
  3. [ ] **`old(expr)` desugaring with `Clone` requirement.** For each `old(expr)` occurrence, synthesize a function-entry local that clones `expr` before the body runs. Rewrite the postcondition to read from the local. If the type of `expr` does not implement `Clone`, emit a compile error with message `"\`old(...)\` requires the captured expression to be Clone"`.
  4. [ ] **Consumed-parameter reference check in ensures.** When an `ensures` clause syntactically references a parameter that was consumed (e.g., a bare-`self` receiver or an owned parameter whose value is moved into the return), emit a compile error suggesting `old(self)` or `old(self.field)`. Detection is a scope check at ensures-expression resolution time.
  5. [ ] **Invariant insertion at every `pub` method exit.** For each type with an `invariant` block, insert an invariant check immediately before every return point in every `pub` method (including constructors — pub associated functions that return `Self`). The rule is structural: applies regardless of receiver mode or whether the method syntactically mutates. Private methods do not check. Applies uniformly to `struct`, `shared struct`, and `shared enum`.
  5b. [ ] **`impl invariant` — all-method invariant scope.** Extend the parser, AST, and typechecker to support `impl invariant <expr>` alongside the plain `invariant <expr>` form inside struct bodies. AST change: `StructDef.invariants: Vec<Expr>` → `StructDef.invariants: Vec<Invariant>` where `Invariant { scope: InvariantScope, expr: Expr }` and `InvariantScope` is `Pub` (default — fires at every `pub` method exit, same as today) or `Impl` (fires at every method exit, pub and private). Parser change: when `invariant` is preceded by `impl`, set scope to `Impl`. Checker change: the invariant-insertion pass (sub-item 5 above) checks scope and extends insertion to private method exits when `scope == Impl`. Diagnostic on `impl invariant` inside `shared struct` and `par struct` is identical; private-method invariant check on `ref self` receivers is allowed. Tests: 2 typechecker tests (`impl invariant` accepted, plain `invariant` still accepted) + 2 interpreter/contract tests (plain invariant not checked on private method exit; `impl invariant` checked on private method exit). See design.md § Struct invariants — `impl invariant`.
  6. [ ] **Distinct "contract predicate panicked" fault category.** When a contract predicate panics during evaluation (in debug), abort the program with a distinct fault category `contract predicate panicked at <loc>: <panic message>`, reported separately from `contract violated: <predicate>`. The runtime's panic hook must distinguish between "panic from inside a contract predicate expression" and "panic from predicate returning false". Implemented via a dedicated evaluation frame or a panic-catch wrapper around contract-expression evaluation in debug builds.
  7. [ ] **Test-runner integration.** Under `karac test`, both fault categories (`contract violated` and `contract predicate panicked`) propagate as test failures with distinct messages, so failing tests correctly point at either "checked code is wrong" or "contract is wrong".
  8. [ ] **Release-mode stripping.** In release builds, all contract machinery (predicate evaluation, `old(...)` capture locals, invariant insertion at method exits) is elided entirely — zero runtime cost.
