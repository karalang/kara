- [x] **`Map.entry(k)` + `Entry[K, V]` enum — in-place insert-or-modify.** _(canonical: [phase-8-stdlib-floor.md](phase-8-stdlib-floor.md), search `Map.entry(k)`)_

  - [x] **2. `Entry[K, V]` enum prelude registration** — added to `PRELUDE_TYPES` (with `K, V` arity in `stub_generics`); `Occupied` / `Vacant` added to `PRELUDE_VARIANTS`. The full `EnumInfo` registration in `register_builtin_types` (typechecker) places `Occupied { value: mut ref V }` / `Vacant { key: K, map: mut ref Map[K, V] }` alongside `Option`/`Result`/`IoError`/`VarError`. First stdlib type using `mut ref` on a variant payload — typechecker accepts the borrow-typed fields without changes.
  - [x] **3. `Entry` methods (or_insert / or_insert_with / and_modify)** — new `infer_entry_method` keyed off `Type::Named { name: "Entry", args: [K, V] }`, wired into `infer_method_call`'s named-type dispatch chain. `or_insert(default: V) -> mut ref V` checks the argument against V; `or_insert_with(f: Fn() -> V) -> mut ref V` uses closure-pushdown so `|| Vec.new()` typechecks without an explicit return ascription; `and_modify(f: Fn(mut ref V)) -> Entry[K, V]` returns Entry for chaining. Effect polymorphism on `or_insert_with` / `and_modify` propagates through the existing closure-effect-checking pass (the effect-checker already seeds `Map.entry` as `allocates(Heap)`). 11 typechecker tests cover return-type plumbing, key/default mismatch, closure-pushdown, and arity.
  - [x] **4. Interpreter — `Value::Entry` + dispatch** — `Value::Entry { map_var: Option<String>, key: Box<Value>, slot_idx: Option<usize> }`. The `entry` arm in `eval_method_call` linear-searches `Value::Map`, builds the Entry. `or_insert` / Vacant arm of `or_insert_with` push (key, default) onto the live Map and write back via `env.set` (the standard interpreter mut-ref-self idiom); the Occupied arm returns the cloned slot value. `and_modify` aliases the slot value via `Value::SharedCell` and runs the closure with `[SharedCell]` as the arg so `|v| { v += 1 }` mutates through. `or_insert*` returns the slot value cloned (NOT a true `mut ref V`); the fully-aliased pattern `m.entry(k).or_insert_with(Vec.new).push(row)` is interpreter-incompatible without invasive Map representation changes — codegen handles that case correctly. 7 interpreter tests cover the cardinal cases plus the canonical `and_modify(|v| { v += 1 }).or_insert(1)` chain.
  - [x] **5. Runtime — `karac_map_entry` and `karac_map_lookup_slot`** — two C ABI fns in `runtime/src/map.rs`. The split mirrors the chain-terminal split: `karac_map_entry` (insert-on-vacant) is used by chains terminating in `or_insert` / `or_insert_with` since codegen will write a default through the slot pointer when occupied=false; `karac_map_lookup_slot` (read-only) is used by chains terminating in bare `and_modify` so a missing key never causes an insertion side-effect. Both fns: out-pointer + bool-return shape, resize before probing so the slot pointer is stable for the rest of the call, pointer valid until the next mutating call (`HashMap::entry`'s lifetime contract).
  - [x] **6. Codegen — `m.entry(k)` + chain lowering** — `try_compile_entry_chain` in `compile_method_call` peels `and_modify` wrappers off the receiver and matches `m.entry(k)` at the chain root. One runtime call (`karac_map_entry` or `karac_map_lookup_slot`, terminal-driven) followed by branch blocks for each step, sharing a single slot pointer — exactly one hash per chain. `or_insert(d)` stores the default through the slot on Vacant via standard expression compilation; `or_insert_with(f)` uses the fat-pointer indirect-call path; `and_modify(f)` inlines the closure body directly (binds closure param to a stack alloca initialised from the slot, evaluates the body, stores back through the slot pointer) so `|v| { v += 1 }` mutates the map slot without needing closure-param-as-pointer codegen surgery. Restricted to inline `ExprKind::Closure` — named-fn forms are spec-out-of-scope at v1. Trailing-`and_modify` terminal materialises the Entry struct `{slot_ptr, occupied}` so chained terminals see both halves.
  - [x] **7. Test coverage** — 11 typechecker + 7 interpreter + 7 codegen E2E tests across the work. Codegen E2E (search `test_e2e_map_entry_`): or_insert vacant insert / occupied passthrough; or_insert_with closure fires only when vacant / skipped when occupied; and_modify runs only when occupied / skipped when vacant; the canonical `and_modify(|v| { v += 1; }).or_insert(1)` chain across three calls. The terminal-`.push(row)` form on `or_insert_with(Vec.new).push(row)` is its own piece of work that co-evolves with the per-type Clone codegen bullet (both need a slot-ptr-receiver path on `Vec.push`); the typechecker accepts that chain shape today.
