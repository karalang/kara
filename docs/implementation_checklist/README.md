## Work in Progress (updated 2026-05-04)

Round 10.3 (compound effect polymorphism), the closure-semantics + ownership umbrella (rounds 11.1, 12.5, 12.23–12.27, 12.33–12.40), and the **RC fallback CFG + dominator-tree dataflow** umbrella (rounds 12.7–12.22, 12.38, 12.41–12.42) all closed; current rounds (12.x) advance the closure-calling-through-`ref` track.

- [~] **Closure calling through `ref`.** Interpreter dispatch through `ref` closures — now meaningful because capture-mode is explicit. Repeatable closures (Copy / ref captures) accept `ref` invocation; once-callable closures reject with a clear error. (Phase 4 § Interpreter.)

  - [x] **Round 12.6 (2026-05-02) — repeatable-closure dispatch.** Pinned existing detection infrastructure with explicit tests: closures with `ref |...|` / `mut ref |...|` prefixes are not flagged once-callable and dispatch correctly across multiple invocations through the interpreter. Detection pre-existed in `src/ownership.rs` (`once_callable_closures: HashSet` is populated only when `MoveKind::ClosureCapture` retag fires for a captured owned non-Copy outer value; the K2 enforcement path errors-and-skips-retag for explicit-prefix closures that consume a capture, and never retags at all when the body only reads). 5 ownership tests + 3 interpreter tests covering `ref ||` multi-call, `mut ref ||` multi-call, outer-binding-still-usable-after-call, bare-form regression, in-loop dispatch, and `Vec[Fn() -> i64]` slot dispatch via `vec[i]()`. Commit `895d7bb`.
  - [~] **Once-callable closure rejected when invoked through a `ref` binding.** A `ref Fn(...)` parameter or loop variable yielding `ref Fn(...)` whose underlying closure is once-callable should reject the invocation with a clear error. The current `once_callable_closures: HashSet<String>` (ownership.rs:317) is keyed by binding name in a single function scope and populated only at let-RHS-is-closure sites (ownership.rs:880-893); lifting once-callability to a property that propagates through types and across function / container boundaries is a 5-step build. Sub-rounds below in dependency order.

    - [x] **Round 12.43 (2026-05-04) — Step 1: Type-system carrier for once-callability.** Added `Type::OnceFunction { params, return_type }` as a sibling variant to `Type::Function` in `src/typechecker.rs`. `type_display` now prints `OnceFn(...)` and `OnceFn(...) -> T`. `types_compatible` accepts the `(OnceFunction, OnceFunction)` identity pair and rejects every cross-pair (`Function` ⇄ `OnceFunction` falls through to `_ => false`) — the conservative subtyping baseline for Step 1; later steps may relax `Function ⇒ OnceFunction` widening if needed at parameter slots. Numeric-comparability "false" arms in the six `type_supports_*` queries (`partial_eq` / `eq` / `hash` / `ord` / `display` / `partial_ord`) widened to include `OnceFunction { .. }`. `contains_type_param` / `substitute_type_params` / `resolve_assoc_projections` propagate once-ness through generic substitution. Call-site dispatch in `Call` matches `Type::Function | Type::OnceFunction` so an owned `OnceFn` callee types normally (the actual rejection at slot boundaries lives in Step 3). `src/ownership.rs:2627` closure-param-type-inference path recognizes the new variant via or-pattern. Closure literals still synthesize as `Type::Function` (Step 2 promotes the capture-consume case to `OnceFunction`); `lower_type_expr` for `TypeKind::FnType` still produces `Function` (no surface `OnceFn(...)` annotation yet — defer until Step 2 lands so inference and surface form stay aligned). 9 new unit tests in `once_function_carrier_tests` (src/typechecker.rs) cover display round-trip (default + unit-return + multi-param), identity, cross-pair rejection (both directions), arity mismatch, all six numeric-trait "false" arms, `substitute_type_params` once-ness preservation, and `contains_type_param` traversal.
    - [x] **Round 12.44 (2026-05-04) — Step 2: Closure-expression once-callability inference at construction.** The typechecker now assigns `Type::OnceFunction` to a closure expression iff its body consumes a captured outer non-Copy binding; otherwise `Type::Function`. Detection lives entirely inside `src/typechecker.rs` (no cross-phase rewrite of `expr_types`). New helpers on `TypeChecker`: `flatten_local_scope_snapshot` captures outer-name → type just BEFORE the closure pushes its param scope so closure params and body-local lets are excluded from the capture set; `is_copy_type_during_check` mirrors `ownership::is_copy_type` against the in-progress `env.structs` / `env.enums` / `env.distinct_types`; `closure_consumes_captured_non_copy` runs a Reading/Consuming-mode body walk (`walk_capture_consume` + `walk_capture_consume_block`) mirroring `use_classifier::walk_expr` semantics — owned-arg `Call` slots (decided by `callee_borrow_positions` reading `env.functions`/`env.impls` directly), `StructLiteral` fields, `Return(Some)` / `Break(Some)` / `Question` / `Match` scrutinee with binding arms, and `For` iterables flip into Consuming; `MethodCall` receiver stays Reading (conservative — the typechecker doesn't track `SelfParam` modes, leaving owned-self method-receiver consumes for Step 3 if needed). Shadow-stack tracking handles closure params, body lets, `if let` / `while let` / `for` patterns, match-arm bindings, and nested closures so a body identifier matching a local-scope name isn't mistaken for a capture. Both Closure arms (`infer_expr` synth + `check_expr` expected-type pushdown) call `closure_type_with_capture_inference`, which short-circuits to `Type::Function` for explicit `ref ||` / `mut ref ||` prefixes (matches round 12.6's repeatable-closure rule). New `CaptureWalkMode` enum at module scope. 8 new unit tests in `closure_once_callability_inference_tests` (src/typechecker.rs) covering captures-and-consumes → `OnceFunction`, only-reads-capture → `Function`, capture-free → `Function`, explicit `ref ||` and `mut ref ||` force `Function`, Copy capture → `Function`, closure param shadows outer non-Copy capture → `Function`, body-local `let` shadows outer non-Copy capture → `Function`. Existing test suite (lib + integration + LLVM codegen) green; only pre-existing `pure`-keyword failure remains.
    - [ ] **Step 3 — Reject once-callable in `ref Fn(...)` and `Fn(...)` parameter slots (caller-side).** When a function call's parameter slot type is `Fn(...)` or `ref Fn(...)` and the supplied argument has type `OnceFn(...)`, emit `error[E_ONCE_FN_INTO_FN_SLOT]` at the call site naming the slot, the closure source, and the consumed capture that made it once-callable. Both `Fn` and `ref Fn` slots reject `OnceFn` — the callee in either case may invoke the parameter many times, which violates the once-callable contract. The own-`Fn` slot case also includes a forwarding subtle: passing an `OnceFn` to an own-`Fn` slot moves the closure into the callee, but the callee's signature still says it can call many times. Tests: positive rejection on both slot kinds; negative — repeatable closure accepted in both; cross-call — once-callable passed through one fn into another `Fn` slot also rejects at the inner call site.
    - [ ] **Step 4 — Reject once-callable in container element slots; loop-variable invocation falls out for free.** `Vec[Fn(...)]` insert of an `OnceFn` rejects at the push call site (the slot is `Fn`, not `OnceFn`). For `for f in vec_of_fns { f() }`: once Step 3 lands, the loop body is automatically clean — `Index` returns `ref Fn(...)`, and the call `f()` against a `ref Fn` callee never sees an `OnceFn` value in this path because the type is `Fn`. The active rejection is at the *insert*, not the iteration. The other side is `Vec[OnceFn(...)]`: iteration must consume each element (`for f in vec` over `Vec[OnceFn]` moves), and `f()` then succeeds because each iteration owns its `OnceFn`. Tests: `Vec[Fn]` push of once-callable rejects; `Vec[OnceFn]` push accepts; `for f in vec_once_fns` consumes and the body's `f()` succeeds; ref-iteration `for f in &vec_once_fns` (if the surface allows) yields `ref OnceFn` and `f()` rejects.
    - [ ] **Step 5 — Diagnostic polish + integration sweep.** Error messages name the closure source span, the consumed capture binding, the slot type that rejected, and offer concrete fixes (clone the capture so the closure becomes repeatable, restructure to keep the closure local, change the slot to `OnceFn` if user-controlled). Integration tests cross the propagation paths: parameter passing × method receiver × struct field assignment × generic-function instantiation × loop invocation. Re-run `tests/closures.rs`, `tests/ownership.rs`, and the rc_predicate parity tests as the regression guard.
  - [ ] **Interpreter `mut ref |...|` capture semantics — mutations propagate back across calls.** Today the interpreter snapshots and *clones* captured values into each call's local scope, so a body that mutates a `mut ref` capture observes the mutation only within that one invocation; the outer binding never sees it. Fix likely involves `Rc<RefCell<Value>>` (or equivalent shared-mutability cell) for ref-mode captures so mutations write through to the outer binding. Distinct from the dispatch-correctness slice in 12.6 — that slice verified the closure is *invokable* multiple times; this one is about whether mutations *persist* across invocations.
