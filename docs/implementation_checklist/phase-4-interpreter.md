  - **F1 — Char arms inline, not routed through builtin-Ord-impl dispatch.** Two paths were viable: (i) add char binop arms inline alongside the existing int / float arms, or (ii) route all primitive comparisons through the builtin Ord impl table for symmetry with user-defined `cmp`.   - **Interpreter — method dispatch.** New `"chars"` arm in the value-receiver method-dispatch block (sibling of `"iter"` / `"into_iter"` at `src/interpreter.rs:6024`). For `Value::String(s)`, snapshots `s.chars().map(Value::Char).collect()` into a `Value::Iterator` with `IteratorSource::Eager` — same shape as `iter()` for collections, so adaptor chains (`map`, `filter`) and the `for` step-machine work uniformly without further changes.
  - **Interpreter — for-loop iterable.** New `Value::String(s) => s.chars().map(Value::Char).collect()` arm in the `ExprKind::For` dispatcher (`src/interpreter.rs:2908`-ish), so bare `for c in s` no longer falls through the catch-all that wrapped the whole value in a 1-element Vec.
  - **Tests.** Four interpreter tests (`test_string_chars_for_loop_prints_each`, `test_string_for_loop_iterates_chars`, `test_string_chars_empty_iterates_zero_times`, `test_string_chars_with_map_char_as_key` — the last is the LeetCode #3 idiom guard) and two typechecker tests (`test_string_chars_returns_iterator_char`, `test_string_chars_rejects_arguments`). The Iterator[char] element type is asserted structurally — `for c in "abc".chars() { if c == 'a' { ... } }` typechecks clean only if `c` is inferred as `char`. Full `cargo test` (4 new interp + 2 new typecheck) and `cargo clippy --all --all-targets -- -D warnings` green; `cargo fmt --all -- --check` clean. Closes the kata; sliding-window solution now runs end-to-end via `karac run`.

  *Out of scope (deferred).*
  - **Codegen (`--features llvm`)** — bench-style compiled execution. The LeetCode #3 kata's `bench/` workload needs `karac build` to be usable at N=10⁵+; `karac run` interprets too slowly. Codegen for `String.chars()` requires emitting an iterator-of-char lowering (UTF-8 decode on next, possibly a new `IteratorSource` variant in the runtime). Separate slice when the kata's bench is wanted. Until then, the kata ships with `karac run` only.
  - **`s.char_at(i)` / `s.bytes()` / `s.char_count()`** — same shape (warn-then-panic), but unused by LeetCode #3 and so deferred. The follow-up kata that needs random-access indexing (e.g., the existing `valid_palindrome.kara` which uses `s.char_count()` + `s.char_at(i)`) is the natural-pull trigger.
  - **`stack.pop()` element-type inference on bare `Vec.new()`** — re-running `examples/leetcode/valid_parentheses.kara` after this slice now gets past `chars()` and fails on `let mut stack = Vec.new(); stack.push(c); … stack.pop()` because the typechecker doesn't pin the element type from later `push` sites. Unrelated to String iteration; tracks as a separate inference gap.
