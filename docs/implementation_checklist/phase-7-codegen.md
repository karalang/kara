## Phase 7: LLVM Code Generation

- [ ] **Hot-swap codegen flag — `--enable-hot-swap` (P0 architectural commit).** New codegen feature flag, off by default in v1. When enabled, `extern`-public module symbols are emitted via PLT-style indirection (callers invoke through a function pointer table rather than direct `call @foo`); internal calls stay direct. Without this commit at v1 freeze, retrofitting hot-swap for continuous PGO + shared-object reload (post-v1, see [`deferred.md § Continuous PGO with Shared-Object Hot-Swap`](../deferred.md#continuous-pgo-with-shared-object-hot-swap)) means recompiling every binary — the AOT artifact format effectively becomes versioned. Granularity is **module-level, not function-level** — hot-swap operates by reloading whole modules rather than swapping individual functions; `auth.verify` is hot-swapped by reloading `auth`. Sub-items:

  1. [ ] **Codegen pass.** Add `--enable-hot-swap` to the `karac build` flag set; thread through `codegen.rs` to emit indirection for `extern`-public symbols when set. Internal calls (private + project-internal-default visibility) stay direct.
  2. [ ] **AOT-perf cost benchmark.** Bench the indirection cost on a representative workload before flag ships. Tentative claim "<1% overall" applies amortized over a whole program; worst-case hot-inner-loop sites can be 10–20%. If module-wide cost is unacceptable in benchmarking, fallback to per-symbol opt-in (annotation on individual functions) — but per-symbol contradicts the warehouse-service hot-swap use case if targets are dispersed.
  3. [ ] **Target gating.** Hard-error on `karac build --target=embedded` and `--target=wasm-*` when `--enable-hot-swap` is set — neither target supports the dynamic-symbol-resolution machinery hot-swap requires.
  4. [ ] **Smoke test.** Test that ensures the flag still emits valid LLVM IR + executable binary on a minimal program (one `extern fn`). Catches bit-rot if no consumer exercises the path during v1.x.

- [ ] **Par codegen: cancellation and error propagation.** The MVP `compile_par_block` in `codegen.rs` lowers `par {}` to `karac_par_run` with one OS thread per branch and discards branch return values. The runtime's `karac_par_run` in `runtime/src/lib.rs` runs branches to completion without honoring cancel flags. The language semantics require (design.md § Parallel Failure and Cleanup): (1) fail-fast — first `Err` cancels siblings via an `AtomicBool` cancel flag checked at compiler-inserted effect boundaries; (2) source-order error reporting — the scope returns the earliest-in-source-order error, not the first-completed one; (3) completion-wins-cancellation — a branch that has already passed its last effect-boundary check when cancel fires completes naturally and its `defer`s run. Implementation steps: extend the runtime ABI with a `cancel_flag` parameter to `karac_par_run` and a return slot per branch; thread a result-type union through the ownership/effect pipeline so codegen knows each branch's return type; insert compiler-emitted cancel checks at each `reads`/`writes`/`sends`/`receives` effect-boundary operation. See also Phase 6.2 roadmap entries "Completion wins cancellation" and "Cascading cooperative cancellation."

The first iteration replaced that with `<elem>.const_array(&vec![val; n as usize])` to build a single LLVM constant aggregate — O(1) at IR-construction time, but LLVM's downstream codegen passes crashed on the giant aggregate store at large N (verified SIGSEGV in `write_to_file` at N≥80K for both i64 and bool element types; module verify passed, the crash was during SelectionDAG's expansion of the aggregate store into machine instructions). The shipped fix is a multi-tier approach: (1) **zero-literal let-binding fast path** — `let buf: Array[T, N] = [0; N]` (and `[false; N]`, `[+0.0; N]`, etc.) detected at the let-binding site and lowered to `alloca + llvm.memset.*`, skipping the aggregate-value round-trip entirely. memset is what LLVM would emit for the aggregate store anyway; emitting it directly sidesteps the codegen path that crashes. O(1) at both IR-construction AND codegen time; works at any N. Floats use bit-pattern equality (`f.to_bits() == 0`) so `-0.0` falls through to the general path and the sign bit is preserved. (2) **Non-zero compile-time const, small N** — single `const_array` call, capped at `SAFE_CONST_ARRAY_N=4096` to stay below the LLVM-crash threshold for non-zero aggregates. (3) **Above the cap or for runtime values** — per-element `insertvalue`, capped at `SAFE_INSERT_N=1024`. Beyond that, errors with a typed diagnostic pointing at the zero-init + runtime-fill workaround rather than silently producing pathologically slow IR. The workload-realistic case lands cleanly: sieve at N=2,000,000 (the original benchmark target, previously unbuildable) now compiles in 0.04s and runs at 12.3 ± 0.2 ms vs Rust's 3.6 ± 0.2 ms (3.43× gap, the smallest gap of the four codegen-vs-Rust workloads measured). Four regression tests in `tests/codegen.rs` lock in: zero-init let emits `llvm.memset` and *not* aggregate-store IR (`test_repeat_literal_const_zero_uses_memset`), 64K LUT compiles cleanly with the same memset shape (`test_repeat_literal_large_n_compiles_without_per_element_ir`), small-N const-nonzero emits no `insertvalue` (`test_repeat_literal_const_nonzero_skips_insertvalue`), runtime-value still exercises the per-element fallback (`test_repeat_literal_runtime_value_falls_back_to_insertvalue`). `cargo clippy --all --tests --features llvm -- -D warnings` clean; `cargo test --release --features llvm` passes.

### Phase 7.2: Compiled Stdlib Types + Layout Codegen

Layout-related user-visible rules live in [design.md § Layout Rules](design.md#layout-rules); the items below are codegen/implementation concerns.

- [x] **`Vec[T]` memory layout.** Compile `Vec[T]` to `{ ptr data, i64 len, i64 capacity }` (24 bytes on 64-bit). No SSO/SVO in v1 — move is pointer copy; inline-storage optimization deferred until benchmarks justify the complexity (and the move-semantics break). String and Vec share the same layout shape so buffer-management routines (grow, drop, memcpy-move) are reusable between them.

- [x] **`Vec[T]` growth strategy.** On `push` exceeding capacity, grow to `max(4, cap * 2)` — 2x doubling with a minimum floor of 4 elements on the first growth from empty. Matches Rust `Vec` behavior; avoids a flurry of tiny allocations on small Vecs.

- [x] **`String` memory layout.** Mirror `Vec[u8]`: `{ ptr data, i64 len, i64 capacity }`. String literals use `{ global_ptr, strlen, 0 }` (cap=0 = static, never freed). `String.new()`, `.len()`, `push_str` implemented. Scope cleanup checks cap > 0 before freeing. SSO deferred to Phase 8.

- **`Array[T, N]` codegen.** Emit as LLVM `[N x T]` directly — no struct wrapper, no heap allocation. Indexing is `getelementptr`. `N` is a compile-time constant available at the type level; `len()` is a constant-fold. Value type — stack-allocated, copy on move. Sub-steps:
  - [x] **A. Parser canonicalization on `Array[T, N]`.** Dropped the `[T; N]` Rust-style type branch in `parser.rs`; `Array[T, N]` is the sole syntax in type position, parsed via the path-type + generic-args route.

  - [x] **C. LLVM type mapping for `Array[T, N]`.** `codegen.rs.llvm_array_type` lowers `Array[T, N]` generic args to inkwell `[N x T]`. Wired into `llvm_type_for_type_expr` so parameters, struct fields, and return types all pick it up.

- [x] **SoA physical representation.** One heap allocation per group. - [ ] **SoA field-access translation (hybrid).** `entities[i].field` compiles to a direct group-indexed load/store — no intermediate element materialization. `let e = entities[i]` (whole-element binding) materializes an AoS `Entity` on the stack by copying one field from each group. - [ ] **SoA drop semantics.** On `pop`/`remove`/drop of a SoA-laid-out collection, emit destructor calls for every owned field in every group at the removed index (e.g., freeing an owned `String` stored in the metadata group). - [ ] **Monomorphized collections — runtime restructure.** P1 (post-v1, P0 design-property). Replace today's type-erased C runtime (`runtime/src/map.rs` and any future erased collections) with monomorphizable Kāra/Rust source compiled per user crate. `Map[K, V]`, `Set[T]`, `Vec[T]`, future `BTreeMap[K, V]` emit one specialized implementation per concrete type tuple — direct hash/eq calls, full inlining, no function-pointer indirection, no byte-blob storage. Existing codegen infrastructure (`generic_fns`, `generated_monos`, `mangle_mono_name`, `type_subst` in `src/codegen.rs:338–345`, `12723–12788`, `12931–12970`) extends naturally to runtime collections. `libkarac_runtime.a` shrinks to non-monomorphizable primitives (panic infra, allocator interface, FFI bridges, OS bindings). Empirical justification: indirection microbench 2026-05-06 confirmed 24.9% erasure tax on i64 hash_map (75% of total Karac-vs-std gap); monomorphization moves Map ops from 1.32× → ~1.07× of std::HashMap. Sequencing: (a) close trait-bounds-at-codegen enforcement (entry below — P0 prerequisite), (b) restructure runtime as monomorphizable source, (c) extend codegen to invoke runtime-collection methods through `compile_generic_call`. Estimated effort 4–8 weeks focused. See `design.md § Generics and Monomorphization Strategy`. Bench artifact for re-running the indirection microbench at implementation-start: `bench/indirection_cost/` (standalone Cargo project).

- [ ] **Trait-bounds-at-codegen enforcement.** P0, prerequisite for monomorphized collections. Currently *parsed and validated* at the typechecker (`src/typechecker.rs:4581–4601`, `validate_inline_generic_bounds`) but **not enforced** at monomorphization time — concrete `T` is not checked to satisfy declared trait bounds when a generic is instantiated. Wire enforcement at two sites: (1) the typechecker call-site path, where the concrete type is known and bounds against declared parameters can be resolved against the impl table; (2) the codegen monomorphization request, where cross-module call paths that bypass the typechecker check still see the bound and the impl table. Diagnostics: structured error pointing at the call site, naming the missing bound, suggesting where the impl could be added (matching the existing missing-impl diagnostic shape from `src/typechecker.rs`). Useful as a standalone language-correctness improvement; load-bearing for `Map[K: Hash + Eq, V]` and similar bounded collection types. See `design.md § Trait bounds` (Type Inference subsection) and `design.md § Generics and Monomorphization Strategy`.

- [ ] **Bounds-check elision: LLVM-friendly emission.** P0 v1. Rework bounds-check codegen to emit the check in a form LLVM's range-analysis passes (SCEV, GVN) can prove redundant when the index is provably in-bounds from surrounding loop structure. Concrete shape: `cmp + branch` against the slice's `len`, with the panic block marked `cold` (so it's hoisted out of the hot path) and `llvm.assume` annotations communicating the success-path range to LLVM where appropriate. Constant-index accesses (`xs[3]` where `3 < xs.len()` is statically known) skip the runtime check at codegen time. Validation: re-run sieve / brute_force / coin_change benchmarks after the change — gaps should close to <1.3× of Rust. Estimated effort 2–4 weeks. See `design.md § Bounds-Check Elision and the get_unchecked Escape Hatch`. Foundational rationale: per the O2 → O3 finding (the bench bump produces 0% delta because bounds checks block autovec prerequisite analyses), this is the prerequisite for any LLVM loop optimization to fire on Karac IR.
