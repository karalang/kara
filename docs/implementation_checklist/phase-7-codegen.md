- [ ] **Par codegen: cancellation and error propagation.** The MVP `compile_par_block` in `codegen.rs` lowers `par {}` to `karac_par_run` with one OS thread per branch and discards branch return values. The runtime's `karac_par_run` in `runtime/src/lib.rs` runs branches to completion without honoring cancel flags. The language semantics require (design.md § Parallel Failure and Cleanup): (1) fail-fast — first `Err` cancels siblings via an `AtomicBool` cancel flag checked at compiler-inserted effect boundaries; (2) source-order error reporting — the scope returns the earliest-in-source-order error, not the first-completed one; (3) completion-wins-cancellation — a branch that has already passed its last effect-boundary check when cancel fires completes naturally and its `defer`s run. Implementation steps: extend the runtime ABI with a `cancel_flag` parameter to `karac_par_run` and a return slot per branch; thread a result-type union through the ownership/effect pipeline so codegen knows each branch's return type; insert compiler-emitted cancel checks at each `reads`/`writes`/`sends`/`receives` effect-boundary operation. See also Phase 6.2 roadmap entries "Completion wins cancellation" and "Cascading cooperative cancellation."

- [ ] **Compiler-assisted migration diagnostic: plain `struct` accessed from multiple concurrent tasks.** When the concurrency analysis detects a plain (non-`par`, non-`shared`) struct value whose live range spans a `par {}` fork point and is reachable from more than one branch, emit a **structured error with a machine-applicable fix-diff** instead of a silent type error. The diff must: (1) change `struct Foo { ... }` to `par struct Foo { ... }`; (2) wrap each `mut` field that is accessed from multiple branches in `Mutex[T]`; (3) insert `lock field { ... }` blocks at every write site within `par` regions; (4) add `ref self` receivers on methods that previously took `mut ref self` (since `par struct` disallows `mut ref self`). The migration is always manual — the compiler produces the diff and the programmer reviews it; no auto-apply. Diagnostic code `E_PAR_STRUCT_REQUIRED`; structured JSON output includes a `fix_diff` field with the unified diff. Sequencing: depends on the concurrency analysis (Phase 6) identifying per-binding reachability across branch boundaries. See design.md § Compiler-assisted migration from plain `struct` to `par struct`.

### Phase 7.2: Compiled Stdlib Types + Layout Codegen

Layout-related user-visible rules live in [design.md § Layout Rules](design.md#layout-rules); the items below are codegen/implementation concerns.

- [x] **`Vec[T]` memory layout.** Compile `Vec[T]` to `{ ptr data, i64 len, i64 capacity }` (24 bytes on 64-bit). No SSO/SVO in v1 — move is pointer copy; inline-storage optimization deferred until benchmarks justify the complexity (and the move-semantics break). String and Vec share the same layout shape so buffer-management routines (grow, drop, memcpy-move) are reusable between them.

- [x] **`Vec[T]` growth strategy.** On `push` exceeding capacity, grow to `max(4, cap * 2)` — 2x doubling with a minimum floor of 4 elements on the first growth from empty. Matches Rust `Vec` behavior; avoids a flurry of tiny allocations on small Vecs.

- [x] **`String` memory layout.** Mirror `Vec[u8]`: `{ ptr data, i64 len, i64 capacity }`. String literals use `{ global_ptr, strlen, 0 }` (cap=0 = static, never freed). `String.new()`, `.len()`, `push_str` implemented. Scope cleanup checks cap > 0 before freeing. SSO deferred to Phase 8.

- **`Array[T, N]` codegen.** Emit as LLVM `[N x T]` directly — no struct wrapper, no heap allocation. Indexing is `getelementptr`. `N` is a compile-time constant available at the type level; `len()` is a constant-fold. Value type — stack-allocated, copy on move. Sub-steps:
  - [x] **A. Parser canonicalization on `Array[T, N]`.** Dropped the `[T; N]` Rust-style type branch in `parser.rs`; `Array[T, N]` is the sole syntax in type position, parsed via the path-type + generic-args route.
  - [x] **B. Typechecker const-eval and `Type.Array` lowering.** Added `Array` to the resolver's prelude types; `typechecker.rs.lower_array_type` const-evaluates positive integer-literal sizes and returns `Type.Array { element, size }`. Non-literal const args are rejected. `type_display` shows `Array[T, N]` consistently.
  - [x] **C. LLVM type mapping for `Array[T, N]`.** `codegen.rs.llvm_array_type` lowers `Array[T, N]` generic args to inkwell `[N x T]`. Wired into `llvm_type_for_type_expr` so parameters, struct fields, and return types all pick it up.
  - [x] **D. Array literal construction.** Compile `ExprKind.ArrayLiteral` when inferred type is `Array[T, N]`: `alloca [N x T]`, sequential `getelementptr` + `store` for each element. Reject element-count mismatches at typecheck time.
  - [x] **E. Array indexing with bounds check.** Compile `ExprKind.Index { object, index }` for `Type.Array`: runtime `if idx >= N { panic }` followed by `getelementptr + load`. Constant in-bounds indices skip the runtime check. Mutable indexing emits GEP + store.
  - [x] **F. `for` loop over Array.** Lower `for elem in array { body }` to an integer-counted loop that reuses the indexing path. `len` is a compile-time constant.
  - [x] **G. End-to-end Array example.** `examples/array_basics.kara` exercising construction, indexing, iteration, Array-as-struct-field, and `ref Array[T, N]` parameter; acceptance test that compiles and runs it.

- [ ] **Collection scope for Phase 7.2.** Ship `Array[T, N]`, `Vec[T]`, and `String` codegen. Implement in that order: Array first (simplest — fixed size, no growth), then Vec (adds growth, capacity, iteration), then String (Vec-shaped plus UTF-8 invariant and the minimum f-string formatter). Layout codegen builds on top once Array and Vec are landed. Full API surface (`map`, `filter`, `fold`, `split`, etc.) remains in Phase 8.

- [x] **`Slice[T]` codegen.** Compile `Slice[T]` to `{ ptr data, i64 len }` — 16 bytes on 64-bit. `mut Slice[T]` shares the same physical layout; the mutability is tracked in the type system, not in the runtime representation. Sub-steps:
  - [x] **A. LLVM type mapping.** `llvm_type_for_type_expr` dispatches `Slice` paths and `TypeKind.MutSlice` to `slice_struct_type()` (2-field struct).
  - [x] **B. Coercion at call boundaries.** `coerce_to_slice` synthesizes slice headers from Array (const N), Vec (loaded ptr/len), Slice (pass-through), and ref-Array sources at call sites. Uses `fn_param_slice_elem` to know which parameter slots expect slices.
  - [x] **C. Range-indexing lowering.** `compile_range_slice` emits `{base + start*stride, end - start}` slice headers with runtime `start <= end && end <= src_len` bounds check. Fires at the top of `compile_index` so it works for any use (let-binding, call argument, nested expression).
  - [x] **D. Slice indexing and iteration.** `compile_slice_index` and `compile_for_slice_var` mirror the Vec versions against the 2-field slice struct. `compile_for` dispatches via `slice_elem_types`.
  - [x] **E. `mut Slice[T]` mutable indexing.** `compile_slice_index_store` emits bounds-checked GEP + store through the slice's data pointer; writes propagate to the source collection. Ownership enforcement for uniqueness lives in the typechecker / ownership checker, not here.
  - [x] **F. End-to-end example.** `examples/slice_basics.kara` exercises coercion from Vec and Array, sub-range slicing via `[a..b]`, `.as_slice()` in expression position, and in-place mutation through `mut Slice[T]`. `test_e2e_slice_basics_example` pins the output.

- [x] **SoA physical representation.** One heap allocation per group. Outer Vec becomes `{ ptr_group_0, ptr_group_1, …, ptr_group_N, i64 len, i64 capacity }` — N group pointers plus the shared length/capacity. Growth reallocates every group buffer independently on `push`. One-allocation-per-group matches GPU-buffer intuition and keeps iteration and drop straightforward.

- [ ] **SoA field-access translation (hybrid).** `entities[i].field` compiles to a direct group-indexed load/store — no intermediate element materialization. `let e = entities[i]` (whole-element binding) materializes an AoS `Entity` on the stack by copying one field from each group. Dually, pushing an AoS element into a SoA Vec decomposes the element into group slots. Purely a codegen optimization — the static type of `entities[i]` is `Entity` either way; no language-level signal needed.

- [ ] **SoA drop semantics.** On `pop`/`remove`/drop of a SoA-laid-out collection, emit destructor calls for every owned field in every group at the removed index (e.g., freeing an owned `String` stored in the metadata group). On whole-collection drop, emit the per-group buffer frees after running per-element destructors.

- [x] **Layout validation pass (resolver).** Implement in `src/resolver.rs`. For each `LayoutDef`: resolve `collection_type` to its element `StructDef`; verify every field named in a `group` block exists on the struct; reject fields appearing in more than one group with a compile error; emit the unassigned-fields warning (suppressible by `#[allow(layout_unassigned_fields)]`). Runs before type checking — field names resolve before generic instantiation, and layouts target concrete structs.
