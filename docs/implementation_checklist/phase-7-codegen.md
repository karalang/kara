- [ ] **Par codegen: cancellation and error propagation.** The MVP `compile_par_block` in `codegen.rs` lowers `par {}` to `karac_par_run` with one OS thread per branch and discards branch return values. The runtime's `karac_par_run` in `runtime/src/lib.rs` runs branches to completion without honoring cancel flags. The language semantics require (design.md § Parallel Failure and Cleanup): (1) fail-fast — first `Err` cancels siblings via an `AtomicBool` cancel flag checked at compiler-inserted effect boundaries; (2) source-order error reporting — the scope returns the earliest-in-source-order error, not the first-completed one; (3) completion-wins-cancellation — a branch that has already passed its last effect-boundary check when cancel fires completes naturally and its `defer`s run. Implementation steps: extend the runtime ABI with a `cancel_flag` parameter to `karac_par_run` and a return slot per branch; thread a result-type union through the ownership/effect pipeline so codegen knows each branch's return type; insert compiler-emitted cancel checks at each `reads`/`writes`/`sends`/`receives` effect-boundary operation. See also Phase 6.2 roadmap entries "Completion wins cancellation" and "Cascading cooperative cancellation."

- [x] **Codegen `[v; N]` repeat literal: multi-tier fast path (zero-let memset + capped const-aggregate).** ✓ DONE (2026-05-04). The original implementation emitted N `build_insert_value` calls (compile time linear in N; N=100K hung the build at 30s+, blocking 64K-LUT workloads like `Array[i64, 65536]`). The first iteration replaced that with `<elem>.const_array(&vec![val; n as usize])` to build a single LLVM constant aggregate — O(1) at IR-construction time, but LLVM's downstream codegen passes crashed on the giant aggregate store at large N (verified SIGSEGV in `write_to_file` at N≥80K for both i64 and bool element types; module verify passed, the crash was during SelectionDAG's expansion of the aggregate store into machine instructions). The shipped fix is a multi-tier approach: (1) **zero-literal let-binding fast path** — `let buf: Array[T, N] = [0; N]` (and `[false; N]`, `[+0.0; N]`, etc.) detected at the let-binding site and lowered to `alloca + llvm.memset.*`, skipping the aggregate-value round-trip entirely. memset is what LLVM would emit for the aggregate store anyway; emitting it directly sidesteps the codegen path that crashes. O(1) at both IR-construction AND codegen time; works at any N. Floats use bit-pattern equality (`f.to_bits() == 0`) so `-0.0` falls through to the general path and the sign bit is preserved. (2) **Non-zero compile-time const, small N** — single `const_array` call, capped at `SAFE_CONST_ARRAY_N=4096` to stay below the LLVM-crash threshold for non-zero aggregates. (3) **Above the cap or for runtime values** — per-element `insertvalue`, capped at `SAFE_INSERT_N=1024`. Beyond that, errors with a typed diagnostic pointing at the zero-init + runtime-fill workaround rather than silently producing pathologically slow IR. The workload-realistic case lands cleanly: sieve at N=2,000,000 (the original benchmark target, previously unbuildable) now compiles in 0.04s and runs at 12.3 ± 0.2 ms vs Rust's 3.6 ± 0.2 ms (3.43× gap, the smallest gap of the four codegen-vs-Rust workloads measured). Four regression tests in `tests/codegen.rs` lock in: zero-init let emits `llvm.memset` and *not* aggregate-store IR (`test_repeat_literal_const_zero_uses_memset`), 64K LUT compiles cleanly with the same memset shape (`test_repeat_literal_large_n_compiles_without_per_element_ir`), small-N const-nonzero emits no `insertvalue` (`test_repeat_literal_const_nonzero_skips_insertvalue`), runtime-value still exercises the per-element fallback (`test_repeat_literal_runtime_value_falls_back_to_insertvalue`). `cargo clippy --all --tests --features llvm -- -D warnings` clean; `cargo test --release --features llvm` passes.

### Phase 7.2: Compiled Stdlib Types + Layout Codegen

Layout-related user-visible rules live in [design.md § Layout Rules](design.md#layout-rules); the items below are codegen/implementation concerns.

- [x] **`Vec[T]` memory layout.** Compile `Vec[T]` to `{ ptr data, i64 len, i64 capacity }` (24 bytes on 64-bit). No SSO/SVO in v1 — move is pointer copy; inline-storage optimization deferred until benchmarks justify the complexity (and the move-semantics break). String and Vec share the same layout shape so buffer-management routines (grow, drop, memcpy-move) are reusable between them.

- [x] **`Vec[T]` growth strategy.** On `push` exceeding capacity, grow to `max(4, cap * 2)` — 2x doubling with a minimum floor of 4 elements on the first growth from empty. Matches Rust `Vec` behavior; avoids a flurry of tiny allocations on small Vecs.

- [x] **`String` memory layout.** Mirror `Vec[u8]`: `{ ptr data, i64 len, i64 capacity }`. String literals use `{ global_ptr, strlen, 0 }` (cap=0 = static, never freed). `String.new()`, `.len()`, `push_str` implemented. Scope cleanup checks cap > 0 before freeing. SSO deferred to Phase 8.

- **`Array[T, N]` codegen.** Emit as LLVM `[N x T]` directly — no struct wrapper, no heap allocation. Indexing is `getelementptr`. `N` is a compile-time constant available at the type level; `len()` is a constant-fold. Value type — stack-allocated, copy on move. Sub-steps:
  - [x] **A. Parser canonicalization on `Array[T, N]`.** Dropped the `[T; N]` Rust-style type branch in `parser.rs`; `Array[T, N]` is the sole syntax in type position, parsed via the path-type + generic-args route.
  - [x] **B. Typechecker const-eval and `Type.Array` lowering.** Added `Array` to the resolver's prelude types; `typechecker.rs.lower_array_type` const-evaluates positive integer-literal sizes and returns `Type.Array { element, size }`. Non-literal const args are rejected. `type_display` shows `Array[T, N]` consistently.
  - [x] **C. LLVM type mapping for `Array[T, N]`.** `codegen.rs.llvm_array_type` lowers `Array[T, N]` generic args to inkwell `[N x T]`. Wired into `llvm_type_for_type_expr` so parameters, struct fields, and return types all pick it up.
  - [x] **D. Array literal construction.** Compile `ExprKind.ArrayLiteral` when inferred type is `Array[T, N]`: `alloca [N x T]`, sequential `getelementptr` + `store` for each element. Reject element-count mismatches at typecheck time.
  - [x] **E. Array indexing with bounds check.** Compile `ExprKind.Index { object, index }` for `Type.Array`: runtime `if idx >= N { panic }` followed by `getelementptr + load`. Constant in-bounds indices skip the runtime check. Mutable indexing emits GEP + store.
  - [x] **F. `for` loop over Array.** Lower `for elem in array { body }` to an integer-counted loop that reuses the indexing path. `len` is a compile-time constant.
  - [x] **G. End-to-end Array example.** `examples/array_basics.kara` exercising construction, indexing, iteration, Array-as-struct-field, and `ref Array[T, N]` parameter; acceptance test that compiles and runs it.

- [x] **SoA physical representation.** One heap allocation per group. Growth reallocates every group buffer independently on `push`. One-allocation-per-group matches GPU-buffer intuition and keeps iteration and drop straightforward.

- [ ] **SoA field-access translation (hybrid).** `entities[i].field` compiles to a direct group-indexed load/store — no intermediate element materialization. `let e = entities[i]` (whole-element binding) materializes an AoS `Entity` on the stack by copying one field from each group. - [ ] **SoA drop semantics.** On `pop`/`remove`/drop of a SoA-laid-out collection, emit destructor calls for every owned field in every group at the removed index (e.g., freeing an owned `String` stored in the metadata group).
