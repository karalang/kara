- [x] **Hot-swap codegen flag — `--enable-hot-swap` (P0 architectural commit).** ✓ Landed 2026-05-18 (cumulative across slices 1+2 of phase-7 line 5, validated by [`investigations/hot_swap_indirection_cost.md`](../investigations/hot_swap_indirection_cost.md)). New codegen feature flag, off by default in v1. When enabled, `extern`-public module symbols are emitted via PLT-style indirection (callers invoke through a function pointer table rather than direct `call @foo`); internal calls stay direct. Without this commit at v1 freeze, retrofitting hot-swap for continuous PGO + shared-object reload (post-v1, see [`deferred.md § Continuous PGO with Shared-Object Hot-Swap`](../deferred.md#continuous-pgo-with-shared-object-hot-swap)) means recompiling every binary — the AOT artifact format effectively becomes versioned. Granularity is **module-level, not function-level** — hot-swap operates by reloading whole modules rather than swapping individual functions; `auth.verify` is hot-swapped by reloading `auth`. Sub-items:

  1. [x] **Codegen pass.** ✓ Landed 2026-05-17 (slice 2). `--enable-hot-swap` parses through `parse_build_command`, threads via `Command::Build` / `Command::BuildProject` → `cmd_build` / `cmd_build_project` → `run_multi_file_codegen` → new `compile_to_object_with_hot_swap` entry (with a parallel `compile_to_ir_with_hot_swap` for tests). Codegen state carries `hot_swap_enabled` + `hot_swap_slots` (fn name → slot index) + `hot_swap_fns` (ordered slot/value list). `declare_function` in `src/codegen/functions.rs` registers a slot for every user-defined `pub fn`. Between the decl pass and the body pass, `compile_program` calls `pre_emit_hot_swap_table` to emit `@karac_hotswap_table = [N x ptr] zeroinitializer` so call-site lowering can GEP into it. At call dispatch (`call_dispatch.rs`), pub-fn callees with a slot lower as load-from-table + `build_indirect_call` via the new `build_hot_swap_indirect_call` helper; private callees stay direct. At finalize, `finalize_hot_swap_table` emits an internal-linkage ctor `@__karac_init_hot_swap_table` that stores each declared pub-fn address into its slot, registered in `@llvm.global_ctors` at priority 65535. v1 binaries store direct function addresses so dispatch behavior is unchanged; the indirection exists for post-v1 reload.

Both within the entry's prior "<1% overall" amortized claim; neither approaches the prior "10–20% worst case" upper bound — indirect-call branch prediction (target never varies in v1 binaries) + 8-wide out-of-order issue absorb the load+indirect-call into the existing data-dependency chain. Full method, disassembly, and caveats in [`docs/investigations/hot_swap_indirection_cost.md`](../investigations/hot_swap_indirection_cost.md). **Flag is safe to expose; no fallback to per-symbol opt-in needed.**

- [ ] **Par codegen: cancellation and error propagation.** The language semantics require (design.md § Parallel Failure and Cleanup): (1) fail-fast — first `Err` cancels siblings via an `AtomicBool` cancel flag checked at compiler-inserted effect boundaries; (2) source-order error reporting — the scope returns the earliest-in-source-order error, not the first-completed one; (3) completion-wins-cancellation — a branch that has already passed its last effect-boundary check when cancel fires completes naturally and its `defer`s run.   **Current state (audit 2026-05-18).** Substantial scaffolding is already in place — the gap is narrower than "the runtime doesn't honor cancel flags" from the original framing:

- [ ] **Auto-par reduction recognition: split Σ / Π / min / max accumulators across threads (P1.x).** Sibling to the non-`par` regions entry above. Today's concurrency analyzer correctly identifies a loop-carried dependency through an accumulator as serializing — for a body like `for k in 0..K { sum = sum + f(inputs[k]); }`, `karac build --concurrency-report` prints `<no parallelization opportunities detected>` because every iteration reads-then-writes `sum`. **Recognize the dependency shape:** when the only loop-carried write is `acc = acc <op> expr` (or the methodic form `acc.combine(expr)`) and `<op>` is in the **associative + commutative** allow-list (`+`, `*`, `min`, `max`, `|`, `&`, `^` for ints/floats; `String.push` is associative but NOT commutative, ruled out; `Vec.push` likewise), lower the loop to a fan-out + reduce: each worker processes a contiguous slice of the iteration space into a per-thread partial; a final serial pass combines the partials with the same `<op>`.   *Settled design (2026-05-15).*
  - **F1 — Source-level over LLVM-side.** The original plan in this entry pre-graduation reached for `llvm.assume` annotations + cold-marked panic blocks to let LLVM's SCEV / GVN prove the redundancy. That approach was attempted and rejected — LLVM's JumpThreading didn't split the phi block by predecessor to fold the bounds check on the back-edge path, likely because the panic block's `puts` + `exit` side effects make JumpThreading's cost-benefit conservative.   - **F2 — Soundness via half-check splitting.** The original unsigned `icmp uge idx, len` catches both negative-idx and idx-too-big in one compare; splitting into signed lower + upper halves lets either half stay (preserving safety) while the other elides. Per-half emission also uses the signed predicate family that matches source-level signed loop guards, so when only one half is elided, LLVM's instcombine can still CSE the remaining one with the loop guard's predicate.
  - **F3 — Bare-identifier indices only.** The elision proof requires matching the index expression syntactically; `v[i + 1]` falls through to the full runtime check. Sound (never elides when can't prove), at the cost of some real-world patterns. See deferred follow-ups below.

- [ ] **`Vec.new() + push` loop-idiom recognition → `Vec.filled` lowering.** Deferred. Surfaced 2026-05-15 during kata-88 perf investigation: the pattern `let mut v: Vec[T] = Vec.new(); for/while ... { v.push(expr); ... }` runs **~3× slower than Rust's `vec![T; N].collect()`** when the loop count is large (1M+), because Kara's `Vec.push` doubles capacity from 4 on each growth, producing log-N reallocations + element copies.   **Compiler-side fix.** Loop-idiom recognition pass that detects the canonical `Vec.new + push-in-loop` shape and rewrites it to a single `malloc(n * sizeof) + fill loop` (essentially the same IR `Vec.filled` already emits). - [x] **Type-aware operator dispatch for unsigned integer ops (Div, Mod, Shr, ordered comparisons).** ✓ Done 2026-05-19. Surfaced 2026-05-19 during kata-153 `binary_search` perf investigation. **Bug:** `src/codegen/expr_ops.rs::compile_binop` dispatches every binary operator to its **signed** LLVM op regardless of operand type — `BinOp::Div` → `build_int_signed_div` (line 1076), `BinOp::Mod` → `build_int_signed_rem` (line 1077), `BinOp::Shr` → `build_right_shift(.., true, ..)` (line 1108, arithmetic shift), and `BinOp::Lt/LtEq/Gt/GtEq` → `IntPredicate::SLT/SLE/SGT/SGE` (lines 1086-1101, signed predicates). Operands typed `u8..u64` produce identical LLVM IR to `i8..i64`, so unsigned semantics are silently lowered as signed. This **violates the spec at [`design.md § Numeric Semantics`](../design.md#numeric-semantics)**, which documents arithmetic shift only for signed Shr and (by intent) unsigned semantics for unsigned types.

  **Empirical anchor.** kata-153 `binary_search` measures **rust 1.33× faster than kara** (50.5 vs 67.4 ms at K=2M outer × N=2M inner). Assembly diff between kara/main and rust/main inner binary-search loops shows one extra serialized instruction pair in kara — `subs` (sets flags) + `cinc x, x, lt` (rounding correction) + `asr` (arithmetic shift), vs rust's `sub` + `lsr` (unsigned shift). Source-level fix attempt: swapping `lo`/`hi` from i64 to u64 in a sibling `bench/binary_search_u64.kara` (already cleaned up) produced **identical** runtime to the i64 variant (70.3 vs 70.4 ms, within σ) — confirming the bug is karac codegen, not source type. The 1.32× rust-faster ratio held across both i64 and u64 sources.

  **Why the fix isn't a 10-line patch.** Karac collapses signed/unsigned to the same LLVM int type at `src/codegen/types_lowering.rs:209-213` (`IntSuffix::I8 | IntSuffix::U8 → i8_type()`, etc.) — correct since LLVM has no signed/unsigned type distinction, but it destroys the signedness info before `compile_binop` runs. Compile_binop receives `BasicValueEnum`, which carries no source-level type tag. The fix requires threading typechecker's `Type::Int(_)` vs `Type::UInt(_)` annotation (defined at `src/typechecker/types.rs:17-18`) through to the codegen call-chain, or a side-channel map on the `Codegen` struct (e.g. `expr_signedness: HashMap<NodeId, IsSigned>` populated from the typechecker output).

  **Surface beyond kata-153.** Any unsigned `/`, `%`, `>>`, `<`, `<=`, `>`, `>=` on `u8..u64` pays the same tax. Common patterns affected: hashing (`h % bucket_count`), bit-packing offset math, range bisection (binary search-shape), sliding-window arithmetic. The bug is **production-language correctness-of-codegen**, not a perf-only nit — `(-1i32 as u32) < 1u32` should evaluate true (unsigned ULT) but with the current SLT-only path evaluates false. P0 v1: a production language can't ship with signed-comparison semantics for unsigned types.

  1. Plumb typechecker signedness into codegen.   2. Update the four affected ops in `compile_binop`: signed → unsigned LLVM dispatch when both operands are u8..u64. Keep mixed-signedness as a typechecker error (it already is — verify with a test).
  3. **Correctness tests** (must precede the dispatch change): pin three behaviors that today produce wrong answers — `(-1i32 as u32) < 1u32 == true`, `5u32 / 2u32 == 2u32` (no rounding-toward-zero artifact differences from signed for positive values, but the IR must use `udiv` so LLVM picks `lshr`), `(255u8 as u32) >> 1 == 127u32`.
