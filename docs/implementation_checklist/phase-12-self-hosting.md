# Phase 12: Self-Hosting

> **The v1 pivot.** Rewrite `karac` in KƒÅra. Executes AFTER the Phase 8 floor + Phase 9 (done) + Phase 10 (mostly done), and BEFORE Phase 11 ‚Äî real order **`8 ‚Üí 9 ‚Üí 10 ‚Üí 12 ‚Üí 11`** (numeric order ‚âÝ execution order). Design record + rationale in [`../roadmap.md` ¬ß Phase 12](../roadmap.md#phase-12-self-hosting); the *why* ‚Äî self-hosting is the credibility proof for a solo/AI-built language ‚Äî is the driver. This tracker holds the action items: pre-pivot blockers, port sequencing, the bootstrap procedure, and the triage of what self-hosting does / doesn't exercise.

## Strategy

- **Port, don't redesign.** The Rust `karac` is the spec, the differential oracle (diff Rust-output vs KƒÅra-output on the same inputs), and a near-line-for-line translation source. Most of the port is mechanical.
- **The Rust compiler is now a means, not the product** ‚Äî the bootstrap seed + oracle. Invest in it only to keep it (a) a trustworthy oracle and (b) complete enough (Phase 8 floor) to compile the port. **Stop adding NEW compiler features to Rust after the pivot** ‚Äî they would have to be re-implemented in KƒÅra anyway; the pivot's "write once" savings apply only to *unbuilt* features.
- **The bar is "production dev platform," not "passes the fixpoint."** Phase 11 and residual Phase 10 GPU get built ON the self-hosted compiler, so it must be pleasant to do real feature work in (usable diagnostics, fast iteration, complete coverage).
- **Self-hosting is a bug-finder, but only for the compiler-shaped subset** of the language (enums, pattern matching, recursion, strings, Vec/Map/Set, Option/Result/`?`, FFI). It is near-blind to floats/numerics, the concurrency runtime, networking, and Tensor ‚Äî keep katas/demos as the bug-finder for those (see SEPARATE-TRACK below).

## Pre-pivot blockers (FIX-BEFORE)

These corrupt the stage-1 build or the differential oracle ‚Äî land them before leaning on the port. (Triaged 2026-06-10; full reasoning at the bottom of this file.)

### Cluster 1 ‚Äî memory-safety in collections-of-heap-elements ‚öÝ

The compiler is built on `Vec[(Span, Token)]`, `Map[String, Symbol]`, etc. These shallow-copy heap-containing compound elements ‚Üí use-after-free / double-free. The output is value-correct but ASAN-dirty ‚Äî the worst case for a self-hosted compiler: it appears to work while corrupting memory, then crashes non-deterministically on a large build (like building itself). One root family (trivially-copyable misclassification + non-recursive deep-copy).

- [x] ‚öÝ **`Vec[(i64, String)]` clone UAF ‚Äî DONE 2026-06-10** (`63440878`). Root cause was NOT a shallow clone (the clone fn already deep-clones): pushing an inline-f-string tuple left the f-string accumulator's `FreeVecBuffer` armed, freeing the source String right after the push ‚Üí the Vec element dangled (UAF on read/clone), and the Vec scope-exit drain didn't recurse into the tuple element's String (leak). Fixed by per-element acc suppression at tuple construction + one-level recursive drain into tuple/struct heap fields. push/read, `.clone()`, `.try_clone()`, scope-drop all single-free clean. Regression tests (the durable artifact): `tests/memory_sanitizer.rs::asan_vec_tuple_heap_element_*` + `tests/codegen.rs::test_e2e_vec_tuple_heap_element_push_clone`. Spun-off non-blocker: `bugs.md` B-2026-06-10-8 (RC-fallback let-bound-tuple drop leak ‚Äî a leak, not UAF).
- [ ] ‚öÝ **`Map.insert` / `Set.insert` owned-param UAF** ‚Äî stores the alias while the caller retains the free. ‚Üí [`phase-7-codegen.md`](phase-7-codegen.md) ¬ß *Owned-param retention follow-up: Map.insert / Set.insert*.
- [ ] ‚öÝ **Map/Set deep-copy doesn't recurse heap elements** ‚Äî `Vec[Map[K,V]]` consumed at a retaining site flat-copies the outer buffer and aliases the map handles. ‚Üí [`phase-7-codegen.md`](phase-7-codegen.md) ¬ß *Owned-param retention follow-up: deep copy doesn't recurse through Map/Set elements*.
- [ ] ‚öÝ **`Option[String]` / `Option[Vec[‚Ä¶]]` dropped-without-destructuring leaks its inline payload** ‚Äî the compiler holds `Option[Token]` / `Option[String]` pervasively (lexer/parser return them), and every undestructured drop leaks. ‚Üí [`bugs.md` B-2026-06-10-6](bugs.md) (filed by the concurrent session 2026-06-10).

### Cluster 2 ‚Äî floor prerequisites the port needs (Phase 8)

The port can't be written cleanly without these. The rest of the Phase 8 floor can finish in parallel.

- [x] **FFI surface for the LLVM-C binding** ‚Äî *(minimal proof RUNS 2026-06-11.)* `#[repr(C)]`, String marshaling (`CStr.from_ptr`), raw-pointer params/out-params (`ptr.mut`/`ptr.null_mut`), foreign-library linking (`[link]`), and PascalCase symbol binding (`#[link_name]`). The [LLVM-C FFI spike's minimal proof](../spikes/self-hosting-llvm-c-proof.md) ‚Äî a KƒÅra program that `unsafe extern "C"`-calls ~20 LLVM-C functions to build `i64 main(){ret 42}`, verify it, and emit a Mach-O object ‚Äî **compiles, links `libLLVM-18`, and runs (`./answer; exit=42`)** under the stage-0 Rust `karac`. Five prerequisites landed to get here (see below + the spike). Still unbuilt but **off the proof's success path**: `CStr.to_string` (full-fidelity error-message read ‚Äî the proof uses static strings + libc `exit`), and `#[repr(C)]`/callback-passing (not exercised by this proof; needed when the real port passes structs/callbacks). ‚Üí [`phase-8-stdlib-floor.md`](phase-8-stdlib-floor.md) ¬ß FFI + the [LLVM-C FFI spike](../spikes/self-hosting-llvm-c-ffi.md).
- [ ] **`process.exit(code)` has no codegen lowering** *(build-surfaced 2026-06-11, minor).* `process.exit` works under `karac run` (interpreter) but `karac build` fails `no handler for method 'exit' on variable 'process'`. The LLVM-C proof works around it with libc `exit` via FFI; a real port wanting `process.exit` in compiled binaries needs a codegen dispatcher arm (lower to libc `exit` / a runtime call). Not a self-hosting blocker (libc `exit` is the honest primitive anyway). ‚Üí general codegen gap.
- [x] **`#[link_name]` honored on `unsafe extern` fn imports** *(2026-06-11).* KƒÅra rejects PascalCase extern fn names (Value-class/snake_case rule) and the entire LLVM-C API is PascalCase (`LLVMContextCreate`, ‚Ä¶). `#[link_name("LLVMContextCreate")]` on a snake_case KƒÅra fn now redirects the emitted LLVM symbol, so a legal KƒÅra identifier binds the C symbol. Implementation: `codegen/functions.rs` (`link_name_attr` extractor, the `link_section` string-value pattern); `codegen.rs` (`extern_link_names: HashMap<KƒÅra‚Üísymbol>` field); `codegen/declarations.rs` `declare_one_extern_function` (register the import under the symbol, dedup-reusing a codegen built-in of that name like `strlen` so LLVM doesn't auto-rename to `strlen.1`); `codegen/call_dispatch.rs` (translate KƒÅra name ‚Üí symbol before `get_function`). Verified: `getpid`/`strlen` bind + link + run; negative case (no attribute ‚Üí undefined KƒÅra symbol); E2E `tests/codegen.rs::test_e2e_extern_link_name_binds_foreign_symbol`. The LLVM-C PascalCase bindings are now accepted by the front-end. ‚Üí [LLVM-C FFI proof ¬ß gate](../spikes/self-hosting-llvm-c-proof.md).
- [x] **Codegen "Undefined variable" for a local read inside an `unsafe {}` block in an auto-par branch** *(2026-06-11; root cause was NOT "*mut T FFI return" ‚Äî see below).* The 6-line `malloc`/`free` reproducer (`let m = unsafe { malloc(8) }; unsafe { free(m) }; println("ok")` ‚Üí `Undefined variable 'm'`) turned out to be an **auto-parallelization** bug, not an FFI bug: the auto-par capture-set collector `refs_in_expr` (`codegen/closures.rs`) had no `ExprKind::Unsafe` arm, so it hit the catch-all `_ => {}` and never saw `m` read inside `unsafe { free(m) }` ‚Äî the par-branch env struct omitted `m`, and the branch fn faulted at `load_variable("m")`. (The *analyzer* `collect_expr_reads` already handled `Unsafe`, so it correctly serialized ‚Äî the two collectors were out of sync.) The trailing-statement sensitivity was the auto-par grouping heuristic; `KARAC_AUTO_PAR=0` made it vanish. **Fix:** aligned `refs_in_expr` with the analyzer ‚Äî added `Unsafe`/`Try`/`Par`/`Lock`/`Question`/`Pipe`/`NilCoalesce`/`OptionalChain` arms (strictly-more-capture is always safe). Regression: `tests/par_codegen.rs::test_auto_par_branch_captures_local_read_inside_unsafe_block_e2e` (verified to fail pre-fix). Full codegen (1501) + par_codegen (112) + concurrency (80) + closures (14) green. **General-purpose correctness fix** ‚Äî any auto-par branch reading a local through one of those forms was miscaptured, not just FFI.
- [x] **`*mut T` raw pointers are now `Copy`** *(2026-06-11).* The auto-par fix exposed the next gate: the `LLVMContextCreate`‚Üí`LLVMContextDispose` proof failed ownership ‚Äî `let ctx = ...create(); module_create(name, ctx); context_dispose(ctx)` ‚Üí *"value 'ctx' moved here, used again"*. `*const T` was `Copy` (via `ref`-mode param inference), but `*mut T` was move-only, so a handle passed to a second FFI call fired a use-after-move ‚Äî and every LLVM-C handle is a `*mut` passed to many builder/dispose calls. **Fix:** added a `Type::Pointer { .. } => true` arm to `ownership.rs::is_copy_type` ‚Äî raw pointers are `Copy` regardless of `const`/`mut` (Rust parity; copying a pointer is a bitwise scalar copy that can't violate ownership, and raw pointers carry no `Drop`, so no double-free risk). **This unblocked the full handle chain: `LLVMContextCreate` ‚Üí `LLVMModuleCreateWithNameInContext(name, ctx)` ‚Üí `LLVMContextDispose(ctx)` now builds, links against `libLLVM-18`, and runs** (prints + exit 0) ‚Äî KƒÅra calling real libLLVM end-to-end. Regression: `tests/ownership.rs::test_raw_mut_pointer_is_copy` + `test_raw_const_pointer_is_copy` (the `*mut` one verified to fail pre-fix). Ownership (306) + codegen + memory_sanitizer + par_codegen green. ‚Üí [LLVM-C FFI proof ¬ß gate](../spikes/self-hosting-llvm-c-proof.md).
- [ ] **`CStr.to_string() -> Result[String, Utf8Error]`** *(build-surfaced 2026-06-11).* The outbound half of the `char*` read path (`from_ptr(p).to_string()` in the proof's `read_and_dispose`). The `Utf8Error` type exists (`runtime/stdlib/utf8_error.kara`) but `CStr.to_string()` has no typecheck/codegen lowering, and `String.from_utf8` is interpreter-only (no runtime UTF-8 validator to reuse). The remaining half of the phase-8 *CString owning type + conversions* item ([phase-8:774](phase-8-stdlib-floor.md)). Needed for the proof to **compile** (called only on an error path, but must typecheck + codegen). ‚Üí [LLVM-C FFI proof ¬ß gate](../spikes/self-hosting-llvm-c-proof.md).
- [x] **Native-library link directive (`kara.toml [link]`)** *(2026-06-11)* ‚Äî `karac` had no way to link an external native library; the AOT link line was hardcoded `cc <obj> <runtime.a> -lm -lpthread -ldl`. Now a `[link]` table (`libs = [...]`, `search-paths = [...]`) appends `-L<path>` / `-l<name>` to the native `cc` invocation, so the codegen leg can link `libLLVM-18`. Landed across four layers: `manifest.rs` (`parse_link_table` ‚Üí `Manifest.link_libs` / `link_search_paths`, array-of-non-empty-strings validation, unknown-key soft-warn); `target.rs` (`NativeLinkConfig` + `set_native_link_config` / `native_link_config`, `OnceLock` first-set-wins, the `set_target_cpu_override` pattern); `codegen/driver.rs` `link_executable_impl` (appends `-L` then `-l` after the runtime archive, no-op when unset ‚Üí link line byte-identical to before); `cli.rs` (`apply_native_link_config` wired into both single-file and project build paths, manifest-only ‚Äî no CLI/env tier, since a libdir is a project fact). Tests: 7 manifest unit tests + 2 link E2E (negative: a missing lib's name reaches the linker error ‚Üí proves `-l` injection; positive: `zlibVersion` resolves *only* with `[link] libs=["z"]`, fails without ‚Üí contrast attributes resolution to the directive). Surfaced + designed by the spike's sub-q 2 (Linking). ‚Üí [LLVM-C FFI spike](../spikes/self-hosting-llvm-c-ffi.md) ¬ß Prerequisites.

## Self-hosting-surfaced blockers (FIX-BEFORE the lexer can run)

The lexer's natural shape ‚Äî `struct SpannedToken { token: Token, span: Span }` lexed by
`mut ref self` methods ‚Äî surfaced two pre-existing compiler bugs. Both reproduce on clean
`main`; neither is caused by the floor slice. The lexer (and all of self-hosting: every AST
node nests enums in structs) was blocked until #1 ‚Äî **now fixed**.

- [x] ‚öÝ **#1 (CODEGEN, bootstrap-critical) ‚Äî struct fields of a USER enum collapse to one `i64` ‚Äî DONE 2026-06-10** (worktree `user-enum-struct-field-layout`). Root cause confirmed exactly as diagnosed: `compile_program` ran `declare_structs` **before** `declare_enums`, so a struct field referencing a user enum wasn't in `enum_layouts` yet and `llvm_type_for_name` fell through to `i64` (`types_lowering.rs`), losing the payload word ‚Äî codegen then failed `Undefined variable 'name'` at the match arm (interp was correct). **Fix (the diagnosed two-pass declaration):** split `declare_structs` into (a) `register_struct_metadata` (AST field side tables ‚Äî `struct_field_type_{names,exprs}`, `struct_field_names` ‚Äî no LLVM types), then `declare_enums` (b) whose `payload_word_count_for_type_expr` struct branch now recurses through `struct_field_type_exprs` (AST) instead of `llvm_type_word_count(struct_types)`, so enum sizing no longer needs struct LLVM types, then (c) `build_struct_types` (LLVM types, `enum_layouts` now populated). Same ordering applied to the stdlib declaration path (`declare_stdlib_program`). Verified both directions (enum-in-struct AND struct-in-enum payload), wide multi-word payloads, and a scalar field *after* the enum field (correct read proves exact payload sizing). High blast radius cleared: full `--features llvm` suite + memory_sanitizer green. **Regression tests:** `tests/codegen.rs::test_e2e_user_enum_field_in_struct` (value-correctness) + `tests/memory_sanitizer.rs::asan_user_enum_field_in_struct_heap_payload` (heap String payload, no UAF/double-free). ‚Üí cross-ref [`phase-7-codegen.md`](phase-7-codegen.md).
- [ ] **#2 (INTERP) ‚Äî `mut ref self` method mutations don't persist.** A `mut ref self` method's writes to `self` are lost on return (direct field writes work; method-call writes don't). Repro: `struct C{n:i64} impl C{ fn inc(mut ref self){ self.n = self.n+1 } } let mut c=C{n:0}; c.inc(); // c.n still 0`. Codegen is correct. Pre-existing on `main`. Not on the lexer's critical path (the port runs under AOT, not interp), but it makes `karac run` unsound for any self-mutating method. ‚Üí cross-ref [`phase-4-interpreter.md`](phase-4-interpreter.md).
- [ ] **#3 (CODEGEN, minor) ‚Äî f-string over a match-arm payload binding yields empty/garbage.** `match e { A(name) => f"‚Ä¶{name}‚Ä¶" }` compiles but prints empty (f-string span-collision family, B-2026-06-09-1). Workaround in the port: `push_str` + `to_string` instead of f-strings around match-bound vars. Low priority.

Building `selfhost/src/main.kara` with the AOT path (post-#1) surfaced a **chain** of further blockers, each masking the next (codegen aborts on the first failing body, so #5/#6 only became visible once #4 was fixed). Numbered #4+ in surface order:

- [x] **#4 (CODEGEN, bootstrap-critical) ‚Äî index a `Vec`/`Slice`/`Array` field through `self` (`self.bytes[self.current]`) ‚Äî DONE 2026-06-11** (worktree `selfhost-lexer`, commit `af139057`). Died with "Index operator applied to non-array type". The field-access-rooted index arm resolves the receiver via `lower_field_access_ptr`, whose receiver `match` handles `Identifier`/`Index` but deliberately leaves `SelfValue` at `Ok(None)` ‚Äî that fall-through is load-bearing for the method-receiver path (`self.count.fetch_add(...)` atomic dispatch must NOT take the generic field-receiver path). So `self.field[i]` fell to the generic tail and compiled the field to a `{ptr,len,cap}` VALUE. Fix: normalise a `SelfValue` inner to a synthetic `Identifier("self")` (self is registered under the name "self" in every per-binding registry) **only in `compile_index`'s FieldAccess arm** ‚Äî the shared helper, and thus the atomic/method path, is byte-identical. Regression test `tests/codegen.rs::test_e2e_index_vec_field_through_self`; full `--features llvm` codegen suite green (1496), atomic-field-method tests re-verified.
- [x] **#5 (CODEGEN, bootstrap-critical) ‚Äî String/Vec method on a `self.field` receiver ‚Äî DONE 2026-06-11** (worktree `selfhost-lexer`). `self.src.substring(self.start, self.current)` died with "no handler for method 'substring' on non-identifier receiver". The field-receiver method helper (`try_compile_field_receiver_method`) resolves its receiver via `lower_field_access_ptr`, which leaves `SelfValue` at `Ok(None)` (load-bearing ‚Äî the atomic-on-self path `self.count.fetch_add(...)` must keep falling through to `compile_atomic_method`). So `self.field.method()` fell through to the generic dispatch ‚Üí no handler. Fix mirrors #4: the FieldAccess dispatch arm in `compile_method_call` normalises a `SelfValue` inner to a synthetic `Identifier("self")`, **gated on `!is_atomic_receiver(object)`** so the atomic fall-through is byte-identical. Verified `substring`/`contains`/`len` on `self.src` with field-rooted args (`token_text`); atomic_* suite re-green. Regression test `tests/codegen.rs::test_e2e_string_method_on_self_field`. Cleared the `substring` blocker ‚Üí the lexer now compiles all bodies and reaches module verification, where **#6** (the `Span` collision) is the sole remaining blocker.
- [x] **#6 (CODEGEN, bootstrap-critical) ‚Äî user `struct Span` collided with the always-injected stdlib `std.tracing` `Span` ‚Äî DONE 2026-06-11** (worktree `selfhost-lexer`). The lexer names its span type `struct Span { line, column, offset, length }` ‚Äî the most natural name, used throughout the Rust `karac` (`token.rs`). `runtime/stdlib/tracing.kara` defines `struct Span { name, span_id, parent_id, fields }` and `src/prelude.rs` injects it always (`STDLIB_SOURCES`); codegen's `struct_types` is flat name-keyed, so `declare_stdlib_program(tracing)` overwrote `struct_types["Span"]` with the tracing layout ‚Üí user `Span` literal built against the wrong type ‚Üí `Invalid InsertValueInst operands` at module verification. **Decision (the language designer chose DCE / collision-skip over gating tracing-behind-import):** a new `user_redefines_stdlib_type(user, stdlib)` gates BOTH `declare_stdlib_program` and `compile_stdlib_program` for each real-source stdlib module (tracing, ordering) ‚Äî skip the WHOLE module when the user defines a struct/enum whose name it exports (skipping just the colliding type is unsafe: the module's own bodies reference its types through the shared maps). Surface derived from the module's own `items` (self-maintaining). Safe: a program redefining a module's public type can't use that module anyway, and an unused module's bodies are dead-stripped at link, so no size cost when there's no collision. Verified: the lexer skeleton now BUILDS + RUNS (token output produced); full `--features llvm` codegen suite green (1497, incl. tracing-using tests ‚Äî kept when no collision). Regression test `tests/codegen.rs::test_e2e_user_struct_shadows_stdlib_tracing_span` (user `Span` constructed in a returning method, fields read back). **Clears the build blocker** ‚Üí surfaced #7 (the lexer now runs but mis-lexes; see below). *(Note: the general usage-DCE binary-size win was found largely illusory ‚Äî LTO/`-dead_strip` already removes unused stdlib bodies; #6's real value is the collision fix, which collision-skip delivers completely.)*
- [x] **#7 (CODEGEN, bootstrap-critical) ‚Äî untyped integer literal in an `if`/`else` branch wasn't coerced to the branch's narrow type ‚Äî DONE 2026-06-11** (worktree `selfhost-lexer`). Surfaced the moment #6 let the lexer RUN: it built but mis-lexed (`ab` ‚Üí two 1-char idents, spaces ‚Üí ERROR, `12` ‚Üí `1`/`2`), because the lexer's `fn peek(ref self) -> u8 { if ‚Ä¶ { 0 } else { self.bytes[self.current] } }` always returned `0` ‚Üí every scan loop exited immediately. Root cause in codegen, NOT typecheck: `const_int_for_suffix` lowers a suffixless literal by its suffix only ‚Üí default `i64`, so an `if ‚Ä¶ { 0 } else { <u8> }` had branch values `i64` vs `i8`; `compile_if`'s phi-merge built a phi only when `tv.get_type() == ev.get_type()` and otherwise silently returned a const-`i64 0` **placeholder** ‚Äî so the whole `if` evaluated to 0 (even `if true { 97u8 } else { 0 }` ‚Üí 0, since the `0`'s width poisoned the merge). Minimal repro: `let a: u8 = if false { 0 } else { 97u8 };` ‚Üí 0; `0u8` (typed) / `Vec[i64]` worked. **Pre-existing** (no #4/#5/#6 surface). Fix: new `unify_int_branch_widths` truncates the wider branch to the narrower width before the phi ‚Äî the typechecker has already unified both branches to one type, so the only reachable width mismatch is the default-`i64`-literal-vs-narrower case, and the wider side is therefore a checker-verified-fitting constant (gated on `is_const()`; a typechecker-impossible non-const wide branch is left to the existing placeholder path). Applied to both `compile_if` and `compile_if_let`. Verified: the lexer now lexes `(ab, 12)` correctly (multi-char ident, multi-digit int, whitespace skipped, exact byte-offset spans). Regression test `tests/codegen.rs::test_e2e_if_branch_untyped_int_literal_narrow_width`.
- [ ] ‚öÝ **#8 (AUTO-PAR, SOUNDNESS, bootstrap-critical) ‚Äî a sequentially-dependent scan loop in a `self`-receiver method gets auto-parallelized and races, corrupting the struct's state.** **Surfaced 2026-06-11 by the differential oracle** (`tests/selfhost_lexer.rs`) ‚Äî the tool's very first catch, of a bug that single manual runs masked. The AOT lexer's spans are **NONDETERMINISTIC**: ~13% of runs mis-span a token, its `start` reverting to the *pre-`skip_whitespace`* offset (e.g. `IDENT "   out"` at offset 8 instead of `out` at 11; `INT 12` at offset 4 instead of 5). **It is an auto-par bug, NOT a #4 regression** ‚Äî building with `KARAC_AUTO_PAR=0` is 100% deterministic + correct (200/200; the full oracle program hashes identically across 25 runs), while the auto-par build flakes. Same FAMILY as the just-fixed `4ee83f5b` (auto-par capture-set `refs_in_expr` missing an expression arm) but a DIFFERENT uncovered form ‚Äî `4ee83f5b` does not resolve it. The scan loops (`while is_digit(self.peek()) { self.advance() }`, `skip_whitespace`) carry a loop-dependency (each iter reads+writes `self.pos`), so either (a) the auto-par dependency analysis is wrongly parallelizing a sequentially-dependent loop over a `self`-field, or (b) the capture-set for a `self`-receiver method call / `self.field` index inside the loop body is incomplete, so a par-branch reads stale/uncaptured `self` state. Discriminators: `mut/ref self` methods with NO field index ‚Üí deterministic (`m8`); `ref S`-param fns WITH field index ‚Üí deterministic (`m8d`); `self`-receiver methods WITH `self.bytes[self.pos]` index in a loop ‚Üí nondeterministic (`m8c`/`m8b`) ‚Äî so the uncovered form is the `self`-field-index (or `self`-method-call) read inside an auto-par-eligible loop. NOT the heap-element-Vec cluster (dropping the pushed element's `String` field keeps it nondeterministic). **The oracle builds the lexer with `KARAC_AUTO_PAR=0` and PASSES** (validates the sequential lexer logic vs the Rust seed); drop that build env once #8 lands. Fix where: the auto-par analyzer (`concurrency.rs` dep analysis) must decline a loop with a loop-carried `self`-field read/write, and/or the codegen capture-set (`refs_in_expr`, `codegen/closures.rs`) must cover `self`-field-index / `self`-method-call reads ‚Äî investigate with `--concurrency-report` on the repro to see what it's parallelizing. Needs ASAN + a `--concurrency-report` trace. **Minimal repro** (`m8c`, ~13% corrupt over runs; `KARAC_AUTO_PAR=0` ‚Üí 0% corrupt):
```kara
struct S { bytes: Vec[u8], pos: i64, start: i64 }
impl S {
    fn at_end(ref self) -> bool { self.pos >= self.bytes.len() }
    fn peek(ref self) -> u8 { if self.pos >= self.bytes.len() { 0 } else { self.bytes[self.pos] } }
    fn adv(mut ref self) { self.pos = self.pos + 1; }
    fn skip_ws(mut ref self) { while not self.at_end() { let c = self.peek(); if c == b' ' { self.adv(); } else { break; } } }
    fn next_off(mut ref self) -> i64 { self.skip_ws(); self.start = self.pos; self.start }
}
// drive `next_off` over "  ab  cd" across many runs ‚Üí `start` is intermittently the pre-skip offset
```
The oracle (`tests/selfhost_lexer.rs`) builds with `KARAC_AUTO_PAR=0` and PASSES; acceptance criterion for #8 = drop that build env and it still passes. Sibling auto-par fix that's the same class: `4ee83f5b` (`refs_in_expr` capture-set arms).

## Bootstrap procedure (3-stage)

- [ ] **Stage-1:** Rust `karac` compiles the KƒÅra compiler source ‚Üí stage-1 binary (correct, but ~2√ó slow ‚Äî the codegen IR-quality pass is a Phase 11 item not yet in either compiler).
- [ ] **Stage-2:** stage-1 compiles the same source ‚Üí stage-2.
- [ ] **Stage-3 fixpoint:** stage-2 compiles the same source ‚Üí stage-3; **stage-2 and stage-3 must be byte-identical.** Ship stage-2.
- [ ] **Differential gate in CI:** over a corpus of `.kara` programs, Rust-`karac` output == KƒÅra-`karac` output (the oracle). Lock before declaring self-hosted.

Speed note: the codegen IR-quality pass (Phase 11, compiler-internal) is written in KƒÅra *after* the pivot and recovers the self-hosted compiler's own speed via this same staging ‚Äî see [`../roadmap.md` ¬ß Codegen Optimization](../roadmap.md#codegen-optimization-ir-quality-pass).

**Done when:** `karac build <kara-compiler-src>` produces a binary that compiles KƒÅra programs with output identical to the Rust `karac`, and the 3-stage bootstrap reaches a byte-identical fixpoint. From here the Rust `karac` is frozen as the bootstrap seed and all new compiler work lands in KƒÅra.
