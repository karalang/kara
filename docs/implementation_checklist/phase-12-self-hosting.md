## Strategy

- **Port, don't redesign.** The Rust `karac` is the spec, the differential oracle (diff Rust-output vs Kāra-output on the same inputs), and a near-line-for-line translation source. Most of the port is mechanical.
- **The Rust compiler is now a means, not the product** — the bootstrap seed + oracle. Invest in it only to keep it (a) a trustworthy oracle and (b) complete enough (Phase 8 floor) to compile the port. **Stop adding NEW compiler features to Rust after the pivot** — they would have to be re-implemented in Kāra anyway; the pivot's "write once" savings apply only to *unbuilt* features.
- **The bar is "production dev platform," not "passes the fixpoint."** Phase 11 and residual Phase 10 GPU get built ON the self-hosted compiler, so it must be pleasant to do real feature work in (usable diagnostics, fast iteration, complete coverage).
- **Self-hosting is a bug-finder, but only for the compiler-shaped subset** of the language (enums, pattern matching, recursion, strings, Vec/Map/Set, Option/Result/`?`, FFI). It is near-blind to floats/numerics, the concurrency runtime, networking, and Tensor — keep katas/demos as the bug-finder for those (see SEPARATE-TRACK below).
- **Pick katas to match the *new* surface of the current port step**, not just any hard problem. Each step has a stress profile — lexer → byte/string/escape; **parser → recursive-enum / `Box`-of-self / tree-build-and-drop**; typechecker → generic-instantiation, trait-bound, exhaustiveness. A kata that re-hits an already-proven surface (another array kata before the parser) tests nothing new. The green light to advance a step is a *burst in that profile going quiet* — the same signal the array/string katas already gave. Gate each step's port on **its** bug family being quiet, not on a globally-flat curve; the parser's gate is recursive-heap (`Box`-in-enum) drop/move/clone, since that is the first port artifact to build the AST shape at scale, and a latent Cluster-1 corruption there is a non-deterministic crash in a program too big to minimize — flush it with focused katas *before* the port, never use the parser itself as the fuzzer.
- **Bug triage during self-hosting.** A kata-found bug in an *already-ported* phase is a **two-codebase** bug. Compare both compilers against the kata's **external** ground truth (the known-correct answer) — *not* Rust-vs-Kāra agreement, which is blind to a bug both inherited. Three outcomes: **(a) unported phase** — Rust wrong, the Kāra phase doesn't exist yet → fix Rust only, carry the *fixed* code forward at port time (never port the bug); **(b) shared-logic** — both compilers agree and both are wrong (the original Rust logic was wrong, faithfully ported) → fix **Rust first, then port the fix**, pin a regression on **both** sides; **(c) port-mistake** — Rust correct, Kāra wrong (they disagree) → fix **Kāra only**, never touch the correct oracle. The Rust-vs-Kāra diff catches (c) instantly (that *is* the self-hosting payoff) but is structurally blind to (b) — which is why the kata corpus's external answers stay load-bearing after self-hosting. **Backend/codegen/runtime** bugs (double-frees, sign-extension) are a separate axis: one backend until codegen ports, so fix-once — but extra-urgent, since they miscompile the port's *own source*, not just the kata. Each defect's ledger row records the port-side classification in its `class` field (`unported`/`shared-logic`/`port-mistake`) and the fix SHA(s) in `fix`; the prose + where-the-fix-landed detail lives in the owning phase tracker ([`bug-ledger.md`](../bug-ledger.md) is the standard — the single tracker; the old `bugs.md` scratchpad was retired 2026-06-14).
- **Gate kata/oracle checks on `karac build` (or `karac check`), never on `karac run` alone — `run` is type-lenient** ([`B-2026-06-13-15`](bug-ledger.md), surfaced by kata #67/#415). `karac run` is the deliberate lenient script path (phase-10 run-leniency decision, 2026-06-06, `src/cli.rs` ~L4020): it computes typecheck **after** its fatal-error gate, so *hard* type errors downgrade to `warning[typecheck]` and the program executes anyway, with the interpreter/codegen substituting a **placeholder** (empty `String` / const-0) for the unrepresentable value → **silent wrong output, exit 0**. `karac check` and `karac build` correctly reject the same program (`error[typecheck]`, exit 1, no binary). The hazard was acute for the self-hosting oracle precisely where this kata found it — a rejected `byte as char` in the lexer's number path *would have* run under `run` and emitted garbage, and the dual-oracle agreement check (case (b) above) is **blind** to it because both the placeholder and the bug live on the same `run` lane. **FIXED `b59eb070`** by partitioning the error set: `TypeErrorKind::is_run_fatal()` marks the invalid-cast family (`E_INT_AS_CHAR`, `E_INT_AS_BOOL`, `E_FLOAT_AS_BOOL`, `E_CHAR_AS_NARROW_INT`, the pointer-provenance casts, refinement source mismatch — every `as` the checker rejects, since none has a defined runtime lowering) as fatal even on `run`; `cmd_run` gates on it right after `effectcheck()` and aborts with `error[typecheck]` (the same abort tier as provider-escape / RAII-across-yield). Genuinely soft type errors (mismatch, arity, exhaustiveness) keep downgrading to `warning[typecheck]` and executing — the partition is **intentionally narrow**, pinned both directions by `tests/cli.rs::{run_value_corrupting_cast_aborts, run_noncast_type_error_warns_and_executes}`. Standing guidance unchanged: gate kata oracles on the `karac build` + `karac check` lanes anyway — they remain the belt-and-braces mitigation for any *future* hard-error class not yet in the run-fatal partition.

### Cluster 1 — memory-safety in collections-of-heap-elements ⚠

The compiler is built on `Vec[(Span, Token)]`, `Map[String, Symbol]`, etc. These shallow-copy heap-containing compound elements → use-after-free / double-free. The output is value-correct but ASAN-dirty — the worst case for a self-hosted compiler: it appears to work while corrupting memory, then crashes non-deterministically on a large build (like building itself). One root family (trivially-copyable misclassification + non-recursive deep-copy).

- [x] ⚠ **`Vec[(i64, String)]` clone UAF — DONE 2026-06-10** (`63440878`). Root cause was NOT a shallow clone (the clone fn already deep-clones): pushing an inline-f-string tuple left the f-string accumulator's `FreeVecBuffer` armed, freeing the source String right after the push → the Vec element dangled (UAF on read/clone), and the Vec scope-exit drain didn't recurse into the tuple element's String (leak). Fixed by per-element acc suppression at tuple construction + one-level recursive drain into tuple/struct heap fields. push/read, `.clone()`, `.try_clone()`, scope-drop all single-free clean. Regression tests (the durable artifact): `tests/memory_sanitizer.rs::asan_vec_tuple_heap_element_*` + `tests/codegen.rs::test_e2e_vec_tuple_heap_element_push_clone`. Spun-off non-blocker — FIXED 2026-06-11 (`669db992`): the RC-fallback let-bound-tuple/struct drop leak (was `bugs.md` B-2026-06-10-8) — the box free at rc==0 now recurses into the boxed aggregate's String/Vec fields (`register_rc_fallback_box_drop` + `emit_aggregate_heap_field_frees`); regressions `tests/memory_sanitizer.rs::asan_rc_fallback_{tuple,struct}_heap_field_drop_no_leak` + `…_moved_no_double_free`. Remaining sibling leak (filed `bugs.md` B-2026-06-11-4, not a blocker): an owned tuple/struct passed *by value* to a fn still leaks its String in the callee (uncovered owned-param drop site).

### AST-port wrapping convention (DECIDED 2026-06-14)

The Rust AST recurses through three self-referential node enums — **`ExprKind`** (`Box<Expr>` operands: `Binary.left/right`, `Unary.operand`, `Call.callee`, `MethodCall.object`, `If.condition`, `Match.scrutinee`, `Index.object/index`, …; `Vec<Expr>` / `Vec<CallArg>` / `Vec<MatchArm>` sequences; `Option<Box<Expr>>` optionals), **`TypeKind`** (`Box<TypeExpr>`: `Array.element`, `Optional.inner`, `Ref`/`MutRef`/`MutSlice`/`Weak`, `Function.return_type`; plus the **cross-type** edge `Const(Box<Expr>)` — a type carrying an expr), and **`PatternKind`** (`Box<Pattern>`, `Vec<Pattern>` for `Tuple`/`Or`/slice prefix+suffix). `StmtKind`/`Block` add `Vec<Stmt>` + `final_expr: Option<Box<Expr>>`; `Item` adds `Box<TraitMethod>` etc. The port cannot mirror those shapes verbatim (`E_ENUM_NESTED_ENUM_PAYLOAD` + the infinite-size recursion). Four rules:

1. **Recursive node enums are `shared enum`.** `Expr`, `TypeExpr`, `Pattern` (and any other self-referential node type) are declared `shared enum`. `shared` is RC reference semantics: the direct analog of Rust's `Box`/heap indirection — it breaks the otherwise-infinite enum size, makes every child edge a pointer, AND is one of the v1-sanctioned "layers" for enum-in-enum nesting. A `Box<Expr>` edge becomes a field of type `Expr` (a `shared` handle): cheap to move/clone (RC bump), cheap to drop (RC dec). Cross-type recursion (`TypeKind::Const` → `Expr`, `Function.return_type` → `TypeExpr`) just references the other `shared` node type — same mechanism.
2. **One node-struct per non-trivial variant.** Each `ExprKind`/`TypeKind`/`PatternKind` variant carrying more than a single scalar/`String`/`Vec`/`shared`-handle payload carries exactly ONE per-variant **node struct** — never a direct enum or `Option<enum>` payload, and (to stay uniform) never a multi-field tuple/struct-variant. E.g. `Add(BinaryExpr)` with `struct BinaryExpr { op: BinOp, left: Expr, right: Expr }`. A struct field may *freely* be an enum (`op: BinOp`), a recursive `shared` edge (`left: Expr`), a `Vec[Expr]`, or an `Option[Expr]` — none of which is legal as a *direct* enum-variant payload — so funnelling every operand through a node struct sidesteps `E_ENUM_NESTED_ENUM_PAYLOAD` once, everywhere. It also gives every node a nameable type for the parser's constructors and the AST oracle's renderer. **Trivial variants keep direct payloads** (`Integer(i64, String)`, `Bool(bool)`, `Identifier(String)`, unit `SelfValue`): scalars/`String`/`Vec`/`shared` handles aren't bare enums.
3. **Sequence + optional children.** Sequences → `Vec[Expr]` (Vec of `shared` handles; the `Vec<Expr>`/`Vec<MatchArm>` analog). Optionals → `Option[Expr]` **as a struct field** (a struct field may be `Option<enum>`; a bare enum-variant payload may not — another reason every variant funnels through a node struct). `MatchArm`/`CallArg`/`FieldInit`/`Param`/`Variant` are plain structs holding `shared` node fields. The L4 escape persists for any *non-recursive* enum operand (a numeric suffix, a bare operator enum): store its lexeme `String`, or make it a node-struct field.
4. **Differential-test the rendered AST.** Mirror the lexer oracle: render the parsed tree to a canonical deterministic string (pre-order, fully parenthesized, spans included) in BOTH the Rust seed (over `karac::parse`) and the Kāra port, and diff over a corpus up to `karac`'s own source. The renderer is what turns a recursive-`shared`-tree drop/clone/move bug into an observable text diff instead of a silent miscompile.

**Precedent already in tree:** the shipped lexer `Token` follows Rules 1–3 implicitly — `InterpPart.Expr(InterpExpr)` wraps the enum payload in a struct (R2), `InterpolatedStringLiteral(Vec[InterpPart])` is the collection layer (R3), `Integer(i64, String)` stores the suffix lexeme not the enum (R2/L4 escape), `CStr`/`SpannedToken`/`Span` are plain structs.

**Memory-safety dependency (the gate this convention waits on):** the load-bearing primitive here is the **recursive `shared enum`** — building, cloning, moving, and dropping a deep RC tree of node structs — which is exactly the surface the parallel *recursive-heap gate* (the kata burst) is flushing. This convention is the target design regardless of what the gate finds (bugs get fixed in `karac`, never designed around — the no-workarounds rule), but **do not begin transcribing `ast.rs` until that gate is quiet** (a burst of recursive-tree katas producing no new Cluster-1 corruption). The convention answers *what shape to write*; the gate answers *whether that shape is memory-safe at scale*.

## Bootstrap procedure (3-stage)

- [ ] **Stage-1:** Rust `karac` compiles the Kāra compiler source → stage-1 binary (correct, but ~2× slow — the codegen IR-quality pass is a Phase 11 item not yet in either compiler).
- [ ] **Stage-2:** stage-1 compiles the same source → stage-2.
- [ ] **Stage-3 fixpoint:** stage-2 compiles the same source → stage-3; **stage-2 and stage-3 must be byte-identical.** Ship stage-2.
- [ ] **Differential gate in CI:** over a corpus of `.kara` programs, Rust-`karac` output == Kāra-`karac` output (the oracle). Lock before declaring self-hosted.

Speed note: the codegen IR-quality pass (Phase 11, compiler-internal) is written in Kāra *after* the pivot and recovers the self-hosted compiler's own speed via this same staging — see [`../roadmap.md` § Codegen Optimization](../roadmap.md#codegen-optimization-ir-quality-pass).

**Done when:** `karac build <kara-compiler-src>` produces a binary that compiles Kāra programs with output identical to the Rust `karac`, and the 3-stage bootstrap reaches a byte-identical fixpoint. From here the Rust `karac` is frozen as the bootstrap seed and all new compiler work lands in Kāra.

### Carrying Rust bug-fixes into the Kāra port (the three flows)

During the pivot the Rust `karac` keeps getting bug-fixed — it must stay a *trustworthy* oracle (the "Port, don't redesign" strategy depends on Rust-output being correct). The question is how each fix reaches the Kāra compiler. There is no manual "list of fixes to re-apply" — the mechanism is sequencing + the oracle, in three flows:

- **Flow A — fix lands BEFORE its module is ported (the common case): free.** The port is near-line-for-line translation of the *current* Rust source, so a fix already in Rust is born into the Kāra translation. This is exactly why the Cluster-1 memory-safety bugs are filed **FIX-BEFORE** — fix in Rust first, then port, and the `.kara` module is correct by construction. **Sequence the port behind the fix whenever the choice exists.**
- **Flow B — fix lands AFTER its module is already ported: re-port the diff.** This is the only flow needing active carry-over. The Rust fix is a small diff against a structurally-parallel file; apply the equivalent diff to the corresponding `.kara` module. Because the two files are near-isomorphic, the diff maps almost directly. **Procedure:** when a fix lands in a Rust phase whose Kāra port already exists (check the Port-sequencing checkboxes above), open the Rust commit, translate its diff into the `.kara` peer in the same slice, and let Flow C verify it. Do **not** defer this to "later" — an unported fix is a latent oracle divergence (see [[feedback_surfaced_bugs_tracked_or_fixed]]-style discipline: fix in-slice or file a tracked `[ ]` entry naming the Rust commit + the `.kara` module).
- **Flow C — the oracle is the ledger, not your memory.** Over the corpus (and crucially `karac`'s own source — "the port IS the corpus"), Rust-output must equal Kāra-output. A fix that changed Rust behavior but didn't reach the Kāra port surfaces as a **divergence the differential gate flags mechanically** (oracle bug #8 was caught this way on its first run). You never rely on remembering which fixes to port; the oracle tells you which ones diverge.

**Front-end vs. codegen asymmetry — fixes are captured differently:**

- **Front-end / semantic phases** (lexer → ownership): pure logic, ported line-for-line. Flows A/B carry the fix as *code*; Flow C verifies.
- **Codegen is NOT a line-for-line port** — Rust `codegen.rs` uses `inkwell`; the Kāra codegen uses **LLVM-C via `extern "C"` FFI** (different backend API), so codegen fixes (Cluster-1 UAF/double-free, #B2–#B5, #9 move-suppression, drop-completeness) are *re-implemented*, not transliterated. What transfers is (1) the **invariant** (e.g. "drop recurses into nested aggregate heap fields"; "suppress the source accumulator on move-out") re-encoded against LLVM-C, and (2) the **regression test** — every codegen fix landed with a `tests/codegen.rs` value-correctness E2E + a `tests/memory_sanitizer.rs` ASAN case, and *those `.kara` programs are backend-agnostic*. When codegen moves to Kāra the **same suite re-runs against the new backend and refuses to go green until the invariant is re-implemented** — that is how codegen fixes are "captured" (the test corpus, not a ported diff). This is why the tracker calls the regression tests "the durable artifact."

**Two backstops:** the stage-2==stage-3 **fixpoint** is the final self-consistency proof of the Kāra codegen (a missing codegen fix won't close it); and **freezing the Rust seed** at the fixpoint reverses the flow entirely — from there all work, fixes included, lands only in Kāra and this question dissolves. The bounding rule that keeps the carry-over tractable: **stop adding NEW features to Rust after the pivot** (Strategy bullet 2) — only *bug fixes to existing behavior* flow through A/B/C; new behavior is written once, in Kāra.

## Parser pre-port: recursive-heap gate

> **The gate (Strategy bullet 5).** The parser is the first port artifact to build the AST tree at scale; a latent `shared`/`Box`-in-enum drop/move/clone corruption there is "a non-deterministic crash in a program too big to minimize." Rule: **flush this bug family with focused katas BEFORE writing the parser — never use the parser itself as the fuzzer.** This is that flush, against the AST-port wrapping convention (the [AST-port constraint blockquote](#) above): **`shared enum` for recursive edges (the `Box<Expr>` analog → RC pointer), plain `struct` for tagged-union operands, `Vec` for sequence children.**
>
> ```kara
> shared enum Expr { Num(i64), Add(BinOp), Neg(Unary), Call(CallExpr) }
> struct BinOp  { left: Expr, right: Expr }   // recursive edge through a struct
> struct Unary  { operand: Expr }
> struct CallExpr { callee: Expr, args: Vec[Expr] }   // sequence children
> ```
>
> The prior #9/#14 fixes (B-2026-06-14-11/-12) cleared *linear* heap-element / by-value-aggregate cases; **none exercised a recursive `shared` enum whose payload struct owns children of its own type** — the actual AST shape.
