# Phase 12: Self-Hosting

> **The v1 pivot.** Rewrite `karac` in Kāra. Executes AFTER the Phase 8 floor + Phase 9 (done) + Phase 10 (mostly done), and BEFORE Phase 11 — real order **`8 → 9 → 10 → 12 → 11`** (numeric order ≠ execution order). Design record + rationale in [`../roadmap.md` § Phase 12](../roadmap.md#phase-12-self-hosting); the *why* — self-hosting is the credibility proof for a solo/AI-built language — is the driver. This tracker holds the action items: pre-pivot blockers, port sequencing, the bootstrap procedure, and the triage of what self-hosting does / doesn't exercise.

## Strategy

- **Port, don't redesign.** The Rust `karac` is the spec, the differential oracle (diff Rust-output vs Kāra-output on the same inputs), and a near-line-for-line translation source. Most of the port is mechanical.
- **The Rust compiler is now a means, not the product** — the bootstrap seed + oracle. Invest in it only to keep it (a) a trustworthy oracle and (b) complete enough (Phase 8 floor) to compile the port. **Stop adding NEW compiler features to Rust after the pivot** — they would have to be re-implemented in Kāra anyway; the pivot's "write once" savings apply only to *unbuilt* features.
- **The bar is "production dev platform," not "passes the fixpoint."** Phase 11 and residual Phase 10 GPU get built ON the self-hosted compiler, so it must be pleasant to do real feature work in (usable diagnostics, fast iteration, complete coverage).
- **Self-hosting is a bug-finder, but only for the compiler-shaped subset** of the language (enums, pattern matching, recursion, strings, Vec/Map/Set, Option/Result/`?`, FFI). It is near-blind to floats/numerics, the concurrency runtime, networking, and Tensor — keep katas/demos as the bug-finder for those (see SEPARATE-TRACK below).

## Pre-pivot blockers (FIX-BEFORE)

These corrupt the stage-1 build or the differential oracle — land them before leaning on the port. (Triaged 2026-06-10; full reasoning at the bottom of this file.)

### Cluster 1 — memory-safety in collections-of-heap-elements ⚠

The compiler is built on `Vec[(Span, Token)]`, `Map[String, Symbol]`, etc. These shallow-copy heap-containing compound elements → use-after-free / double-free. The output is value-correct but ASAN-dirty — the worst case for a self-hosted compiler: it appears to work while corrupting memory, then crashes non-deterministically on a large build (like building itself). One root family (trivially-copyable misclassification + non-recursive deep-copy).

- [x] ⚠ **`Vec[(i64, String)]` clone UAF — DONE 2026-06-10** (`63440878`). Root cause was NOT a shallow clone (the clone fn already deep-clones): pushing an inline-f-string tuple left the f-string accumulator's `FreeVecBuffer` armed, freeing the source String right after the push → the Vec element dangled (UAF on read/clone), and the Vec scope-exit drain didn't recurse into the tuple element's String (leak). Fixed by per-element acc suppression at tuple construction + one-level recursive drain into tuple/struct heap fields. push/read, `.clone()`, `.try_clone()`, scope-drop all single-free clean. Regression tests (the durable artifact): `tests/memory_sanitizer.rs::asan_vec_tuple_heap_element_*` + `tests/codegen.rs::test_e2e_vec_tuple_heap_element_push_clone`. Spun-off non-blocker — FIXED 2026-06-11 (`669db992`): the RC-fallback let-bound-tuple/struct drop leak (was `bugs.md` B-2026-06-10-8) — the box free at rc==0 now recurses into the boxed aggregate's String/Vec fields (`register_rc_fallback_box_drop` + `emit_aggregate_heap_field_frees`); regressions `tests/memory_sanitizer.rs::asan_rc_fallback_{tuple,struct}_heap_field_drop_no_leak` + `…_moved_no_double_free`. Remaining sibling leak (filed `bugs.md` B-2026-06-11-4, not a blocker): an owned tuple/struct passed *by value* to a fn still leaks its String in the callee (uncovered owned-param drop site).
- [x] ⚠ **`Map.insert` / `Set.insert` owned-param UAF — DONE 2026-06-11.** Stored the alias while the caller retained the free → double-free at scope exit / UAF on bucket read. Fixed by wiring the canonical `Vec.push` consume-site pair (`suppress_fstr_acc_if_moved_out` + `maybe_defensive_copy_param_arg`) into the `Map.insert` key+value arms (`src/codegen/maps.rs`) and the `Set.insert` element arm (`src/codegen/collections.rs`). Regression artifacts: `tests/codegen.rs::{test_e2e_map_insert_owned_string_param_value, test_e2e_set_insert_owned_string_param}` + `tests/memory_sanitizer.rs::{asan_map_insert_owned_string_param_value, asan_map_insert_owned_string_param_key, asan_set_insert_owned_string_param, asan_map_insert_fstring_value}` (all four ASAN-fail pre-fix). Spun-off non-insert sibling: the `vec[i] = s_param` / `obj.field = s_param` assignment-store path — same family, different consume site (Assign arm) — filed as its own entry in [`phase-7-codegen.md`](phase-7-codegen.md). → [`phase-7-codegen.md`](phase-7-codegen.md) § *Owned-param retention follow-up: Map.insert / Set.insert*.
- [x] ⚠ **`Vec[Map]` / `Vec[Set]` ownership — DONE 2026-06-11.** Filed as "deep-copy doesn't recurse heap elements," but the real bug was bigger: a `Map` moved into a `Vec` was never ownership-transferred — the Vec aliased a handle the origin binding still freed, and the Vec's own drop never freed map elements. A `Vec[Map]` built from a local map and **returned** dangled (use-after-free; AOT `-1` vs interp `777`). Fixed by making the Vec own its map elements: (1) the Vec drop frees each element handle (`track_vec_of_maps_var` + `MapElemDrop` + `emit_free_one_map_handle`), (2) `Vec.push` (+ siblings) transfers ownership by suppressing the source Map's `FreeMapHandle` (the suppression helper fixed to scan all frames), (3) the defensive copy clones each handle. Regression artifacts: `tests/memory_sanitizer.rs::{asan_vec_map_returned_from_helper_no_uaf, asan_vec_map_loop_built_returned_no_uaf, asan_vec_map_push_then_read_same_scope_single_free, asan_vec_map_param_deep_copy_no_double_free, asan_vec_set_param_deep_copy_no_double_free, asan_vec_map_string_value_param_deep_copy_no_double_free}` + `tests/codegen.rs::{test_e2e_vec_map_returned_from_helper, test_e2e_vec_map_param_deep_copy}` (the returned-`Vec[Map]` UAF tests verified to fail pre-fix). Deferred benign-leak remainder (match-arm / ref-rvalue `Vec[Map]` binding sites) filed in phase-7. → [`phase-7-codegen.md`](phase-7-codegen.md) § *`Vec[Map]` / `Vec[Set]` ownership*.
- [x] ⚠ **`Option`/`Result` inline-heap payload undestructured-drop leak — DONE.** Core `Option[String]`/`Option[Vec]` fix landed 2026-06-10 (`9995e88b`, concurrent session); the three open follow-ons landed 2026-06-11: (1) **`Result[T,E]`** inline Ok/Err payloads (the compiler is built on `Result` — `FreeInlineResultPayload`, two-variant overlay), (2) **`Option[Map]`/`Option[Set]`** inline handle payloads (`FreeInlineOptionMapPayload` via `emit_free_one_map_handle`; tag→None move-out suppression), (3) **non-`Call` let-RHS** (`let x = if c { Some(a) } else { None }` — registration broadened past `Call` to provably-fresh `if`/`match`/block tails via `rhs_is_fresh_inline_enum`, which excludes moves/aliases). The Result registration also required `?`-site source suppression (`suppress_question_source_inline_payload`) — `r?` consumes the source, so its inline-payload free is disarmed (else the unwrap binding / returned Err double-frees; pinned by `asan_question_multiword_ok_payload_owned_once`). Regression artifacts: `tests/memory_sanitizer.rs::{asan_result_ok_string_undestructured_freed, asan_result_err_string_undestructured_freed, asan_result_consumed_match_no_double_free, asan_option_map_undestructured_freed, asan_option_if_else_fresh_payload_freed}` + `tests/codegen.rs::{test_e2e_result_inline_payload_drop_paths, test_e2e_option_map_and_noncall_rhs_drop_paths}`. **Bounded remainders (leak, never corruption):** a *match-arm-bound* `Option[Map]` payload still leaks (the separate deferred match-bound-`Map` tracking gap), and `let x = if c { opt_a } else { opt_b }` over EXISTING Option bindings crashes pre-existingly on `main` (move-of-existing, independent of this fix). → [`bugs.md` B-2026-06-10-6](bugs.md) (local triage).

`main`; neither is caused by the floor slice. The lexer (and all of self-hosting: every AST
node nests enums in structs) was blocked until #1 — **now fixed**.

- [x] **#2 (INTERP) — `mut ref self` method mutations don't persist — DONE 2026-06-11** (worktree `interp-mut-ref-self`). A `mut ref self` method's writes to `self` were lost on return: the tree-walk interpreter passed the receiver **by value** (`arg_vals[0] = obj`, `method_call.rs`), bound it to `self` in a fresh scope, ran the body, then popped the scope with **no write-back to the call-site place** — so `c.inc()` was a no-op on `c`. - [x] **#B2 (CODEGEN) — block expression in value position freed its tail heap value — DONE 2026-06-11** (worktree `block-expr-value-heap-return`; was [`bugs.md` B-2026-06-11-2](bugs.md), surfaced probing #3's brace-wrapped variants). C: **number forms** — `0x`/`0b`/`0o` radix (`i64.from_str_radix`), float (fractional + `e`/`E` exponent, via `f64.parse`), `_` separators (`strip_underscores` over `.chars()`), and `i8…u128`/`f32`/`f64` suffixes (consumed for span-fidelity; suffix-kind storage deferred). D: **string / multi-string / char / byte literals** — `"…"` (literal runs via substring, `\n \t \r \\ \"` escapes), `"""…"""` (raw body), `'c'` (ASCII + simple escapes; read via `first_char`), `b'A'` (one ASCII byte or escape → `u8`). String/char values render through a shared `escape_for_render`; the oracle corpus generator escapes `"`/`\` so literal inputs embed verbatim. D-cont: **f-string interpolation** (`f"…{e}…"` → `Vec[InterpPart]` of `Text`/`Expr`, each `Expr` carrying the verbatim source + its ABSOLUTE offset/line/column for span-rebasing — exact-match vs the Rust seed, incl. nested `{…}` and escapes in text) and **c-strings** (`c"…"` → `CStr { bytes: Vec[u8], source_len }`, with `\n \t \r \\ \"` and `\xHH` escapes via `hex_val`/`byte_of`; interior NUL rejected). E: **error-recovery surface** — raw identifiers (`r#NAME`, payload is bare `NAME` / span covers `r#NAME`; structural markers `self`/`mut`/`ref`/… rejected via `is_structural_marker`), reserved single-letter **string prefixes** (`x"…"`, `_"…"`, `r"…"` → Error spanning the whole construct, `\`-escapes consumed; `f"…"`/`c"…"` are the only recognized ones), reserved **`#`-guarded strings** (`#"…"#`, `##"…"##`, … with matched-hash scan; `#[attr]` still lexes as `Pound`), reserved **future keywords** (`f16`/`bf16`/`gen`/`become`/`async`/… in the keyword table), the reserved **`expr_<year>` fragment-specifier namespace** (`is_reserved_fragment_specifier_namespace`: `expr_` + 4 ASCII digits parsing to a year in 2020..=2099 → Error; `r#expr_2026` exempt via raw-ident path), and **single-codepoint non-ASCII recovery** (`non_ascii_at_lead` + `utf8_byte_len` consume the whole UTF-8 codepoint, advancing `current` by byte-length but `column` by one codepoint, → one "unexpected character" Error). Error tokens render as bare `ERROR` (message discarded), so the oracle asserts **span** parity — each error path consumes the byte-identical extent of the Rust seed. Adding the `Token::Error` arm made the oracle's `render_rust` match **exhaustive** over the full token set, so its catch-all `panic!` was removed (a new seed variant now fails to compile until rendered). Oracle corpus ~115 inputs, passes with auto-par ON; fmt + clippy clean. **Still deferred:** `\u{…}` / `\0` escapes (gated on a codepoint→char builder — see #10), multi-byte c-string/char bodies, and the **multi-codepoint non-ASCII *identifier* recovery** (folding a run of Unicode letters into one error — needs a Unicode `char.is_alphabetic` in karac, tracked as **#13**; the shipped single-codepoint path is exact for any isolated non-ASCII byte sequence). Surfaced **#9** (let-bound bare-enum heap-payload double-free) — sidestepped, tracked above (now unblocked). **Next:** `\u{…}` (once #10-family lands), newline/multi-line span coverage, then the **parser stage**. - [ ] **#12 (TYPECHECK, non-blocking) — `return` in expression position is typed `()`, not `!` (never), so it can't share an `if`/`match` value-expression with a real value arm.** Surfaced by slice D: `let ch = if … { '\n' } else { return Error };` fails (`if/else branches have incompatible types: 'char' and '()'`). Rust types `return`/`break`/`continue` as `!`, which coerces to any branch type; Kāra types them `()`. Worked around in `char_literal` / `byte_char_literal` with a `let mut x = <sentinel>;` + if-*statements* (early `return`s in statement position, which IS fine). Pervasive in the parser (recursive descent loves `let x = parse_or_bail()? else { return recover() }`-shaped value-position bailouts), so worth fixing for port ergonomics. → typechecker (never-type coercion).

- [ ] **#10 (CODEGEN, non-blocking) — `char.try_from(u8)` and `n as char` have no codegen handler.** Surfaced by slice C: `b as char` is rejected by the typechecker (`E_INT_AS_CHAR`, correct) which suggests `char.try_from(n)`, but `char.try_from` then fails codegen (`no handler for method 'try_from' on variable 'char'`). The lexer's `strip_underscores` sidesteps it by iterating `.chars()` and `push`-ing each kept `char` (no byte→char conversion). Worth implementing before any port code genuinely needs a byte→`char` (e.g. building a `char` from a scanned byte). → general codegen gap.
- [ ] **#11 (CODEGEN, non-blocking) — builtin parse-family `Option` payload binding loses its concrete type for method dispatch.** `match i64.parse(s) { Some(v) => v.to_string() }` fails codegen (`no handler for method 'to_string' on variable 'v'`); same for `i64.from_str_radix` / `f64.parse`. The VALUE flows correctly (binding `v` into an enum payload / arithmetic works — the lexer relies on exactly this), and **user-constructed** `Option` payloads dispatch fine (`let o: Option[i64] = Some(5); … v.to_string()` → "5"; `Option[String]` `.len()` works). So the gap is narrow: the *builtin* parse functions' `Option[T]` return type isn't propagated to the match-bound payload for method dispatch. Not a bootstrap blocker (parse results are used as values, not receivers), but track for parser ergonomics. → general codegen gap.
- [ ] **#13 (STDLIB, non-blocking) — no Unicode `char.is_alphabetic` / `char.is_numeric` classifier.** Kāra ships only `u8.is_ascii_*` byte predicates; there is no Unicode-aware classification on `char`. Surfaced by slice E's non-ASCII recovery: the Rust seed's `non_ascii_at_lead` / `identifier()` fold a *run* of Unicode letters into one "non-ASCII identifier" error (`cp.is_alphabetic()` gating an ident-continue loop), which the port can't reproduce faithfully. **Shipped instead (exact for the tested subset):** `non_ascii_at_lead` emits a single-codepoint "unexpected character" Error — byte-identical to the seed for any isolated non-ASCII codepoint *not* immediately followed by an identifier-continue char (the only divergent case is a multi-codepoint non-ASCII identifier like `αβγ` or an ASCII ident with a non-ASCII tail like `xβ`, both kept OUT of the corpus). The full recovery (consume-the-run + the in-`identifier()` non-ASCII-continue path) lands once a Unicode `char.is_alphabetic`/`is_numeric` builtin exists. **Not a bootstrap blocker:** the port (and all of `karac`'s own source) uses ASCII identifiers; non-ASCII is purely an error-diagnostic nicety. → stdlib char predicates + lexer `non_ascii_at_lead`/`identifier` follow-on.

## Bootstrap procedure (3-stage)

- [ ] **Stage-1:** Rust `karac` compiles the Kāra compiler source → stage-1 binary (correct, but ~2× slow — the codegen IR-quality pass is a Phase 11 item not yet in either compiler).
- [ ] **Stage-2:** stage-1 compiles the same source → stage-2.
- [ ] **Stage-3 fixpoint:** stage-2 compiles the same source → stage-3; **stage-2 and stage-3 must be byte-identical.** Ship stage-2.
- [ ] **Differential gate in CI:** over a corpus of `.kara` programs, Rust-`karac` output == Kāra-`karac` output (the oracle). Lock before declaring self-hosted.

Speed note: the codegen IR-quality pass (Phase 11, compiler-internal) is written in Kāra *after* the pivot and recovers the self-hosted compiler's own speed via this same staging — see [`../roadmap.md` § Codegen Optimization](../roadmap.md#codegen-optimization-ir-quality-pass).

**Done when:** `karac build <kara-compiler-src>` produces a binary that compiles Kāra programs with output identical to the Rust `karac`, and the 3-stage bootstrap reaches a byte-identical fixpoint. From here the Rust `karac` is frozen as the bootstrap seed and all new compiler work lands in Kāra.
