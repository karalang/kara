# Phase 12: Self-Hosting

> **The v1 pivot.** Rewrite `karac` in Kāra. Executes AFTER the Phase 8 floor + Phase 9 (done) + Phase 10 (mostly done), and BEFORE Phase 11 — real order **`8 → 9 → 10 → 12 → 11`** (numeric order ≠ execution order). Design record + rationale in [`../roadmap.md` § Phase 12](../roadmap.md#phase-12-self-hosting); the *why* — self-hosting is the credibility proof for a solo/AI-built language — is the driver. This tracker holds the action items: pre-pivot blockers, port sequencing, the bootstrap procedure, and the triage of what self-hosting does / doesn't exercise.

## Strategy

- **Port, don't redesign.** The Rust `karac` is the spec, the differential oracle (diff Rust-output vs Kāra-output on the same inputs), and a near-line-for-line translation source. Most of the port is mechanical.
- **The Rust compiler is now a means, not the product** — the bootstrap seed + oracle. Invest in it only to keep it (a) a trustworthy oracle and (b) complete enough (Phase 8 floor) to compile the port. **Stop adding NEW compiler features to Rust after the pivot** — they would have to be re-implemented in Kāra anyway; the pivot's "write once" savings apply only to *unbuilt* features.
- **The bar is "production dev platform," not "passes the fixpoint."** Phase 11 and residual Phase 10 GPU get built ON the self-hosted compiler, so it must be pleasant to do real feature work in (usable diagnostics, fast iteration, complete coverage).
- **Self-hosting is a bug-finder, but only for the compiler-shaped subset** of the language (enums, pattern matching, recursion, strings, Vec/Map/Set, Option/Result/`?`, FFI). It is near-blind to floats/numerics, the concurrency runtime, networking, and Tensor — keep katas/demos as the bug-finder for those (see SEPARATE-TRACK below).

## Pre-pivot blockers (FIX-BEFORE)

These corrupt the stage-1 build or the differential oracle — land them before leaning on the port. (Triaged 2026-06-10; full reasoning at the bottom of this file.)

### Cluster 1 — memory-safety in collections-of-heap-elements ⚠

The compiler is built on `Vec[(Span, Token)]`, `Map[String, Symbol]`, etc. These shallow-copy heap-containing compound elements → use-after-free / double-free. The output is value-correct but ASAN-dirty — the worst case for a self-hosted compiler: it appears to work while corrupting memory, then crashes non-deterministically on a large build (like building itself). One root family (trivially-copyable misclassification + non-recursive deep-copy).

- [x] ⚠ **`Vec[(i64, String)]` clone UAF — DONE 2026-06-10** (`63440878`). Root cause was NOT a shallow clone (the clone fn already deep-clones): pushing an inline-f-string tuple left the f-string accumulator's `FreeVecBuffer` armed, freeing the source String right after the push → the Vec element dangled (UAF on read/clone), and the Vec scope-exit drain didn't recurse into the tuple element's String (leak). Fixed by per-element acc suppression at tuple construction + one-level recursive drain into tuple/struct heap fields. push/read, `.clone()`, `.try_clone()`, scope-drop all single-free clean. Regression tests (the durable artifact): `tests/memory_sanitizer.rs::asan_vec_tuple_heap_element_*` + `tests/codegen.rs::test_e2e_vec_tuple_heap_element_push_clone`. Spun-off non-blocker: `bugs.md` B-2026-06-10-8 (RC-fallback let-bound-tuple drop leak — a leak, not UAF).
- [x] ⚠ **`Map.insert` / `Set.insert` owned-param UAF — DONE 2026-06-11.** Stored the alias while the caller retained the free → double-free at scope exit / UAF on bucket read. Fixed by wiring the canonical `Vec.push` consume-site pair (`suppress_fstr_acc_if_moved_out` + `maybe_defensive_copy_param_arg`) into the `Map.insert` key+value arms (`src/codegen/maps.rs`) and the `Set.insert` element arm (`src/codegen/collections.rs`). Regression artifacts: `tests/codegen.rs::{test_e2e_map_insert_owned_string_param_value, test_e2e_set_insert_owned_string_param}` + `tests/memory_sanitizer.rs::{asan_map_insert_owned_string_param_value, asan_map_insert_owned_string_param_key, asan_set_insert_owned_string_param, asan_map_insert_fstring_value}` (all four ASAN-fail pre-fix). Spun-off non-insert sibling: the `vec[i] = s_param` / `obj.field = s_param` assignment-store path — same family, different consume site (Assign arm) — filed as its own entry in [`phase-7-codegen.md`](phase-7-codegen.md). → [`phase-7-codegen.md`](phase-7-codegen.md) § *Owned-param retention follow-up: Map.insert / Set.insert*.
- [x] ⚠ **`Vec[Map]` / `Vec[Set]` ownership — DONE 2026-06-11.** Filed as "deep-copy doesn't recurse heap elements," but the real bug was bigger: a `Map` moved into a `Vec` was never ownership-transferred — the Vec aliased a handle the origin binding still freed, and the Vec's own drop never freed map elements. A `Vec[Map]` built from a local map and **returned** dangled (use-after-free; AOT `-1` vs interp `777`). Fixed by making the Vec own its map elements: (1) the Vec drop frees each element handle (`track_vec_of_maps_var` + `MapElemDrop` + `emit_free_one_map_handle`), (2) `Vec.push` (+ siblings) transfers ownership by suppressing the source Map's `FreeMapHandle` (the suppression helper fixed to scan all frames), (3) the defensive copy clones each handle. Regression artifacts: `tests/memory_sanitizer.rs::{asan_vec_map_returned_from_helper_no_uaf, asan_vec_map_loop_built_returned_no_uaf, asan_vec_map_push_then_read_same_scope_single_free, asan_vec_map_param_deep_copy_no_double_free, asan_vec_set_param_deep_copy_no_double_free, asan_vec_map_string_value_param_deep_copy_no_double_free}` + `tests/codegen.rs::{test_e2e_vec_map_returned_from_helper, test_e2e_vec_map_param_deep_copy}` (the returned-`Vec[Map]` UAF tests verified to fail pre-fix). Deferred benign-leak remainder (match-arm / ref-rvalue `Vec[Map]` binding sites) filed in phase-7. → [`phase-7-codegen.md`](phase-7-codegen.md) § *`Vec[Map]` / `Vec[Set]` ownership*.
- [ ] ⚠ **`Option[String]` / `Option[Vec[…]]` dropped-without-destructuring leaks its inline payload** — the compiler holds `Option[Token]` / `Option[String]` pervasively (lexer/parser return them), and every undestructured drop leaks. → [`bugs.md` B-2026-06-10-6](bugs.md) (filed by the concurrent session 2026-06-10).

### Cluster 2 — floor prerequisites the port needs (Phase 8)

The port can't be written cleanly without these. The rest of the Phase 8 floor can finish in parallel.

- [x] **FFI surface for the LLVM-C binding** — *(minimal proof RUNS 2026-06-11.)* `#[repr(C)]`, String marshaling (`CStr.from_ptr`), raw-pointer params/out-params (`ptr.mut`/`ptr.null_mut`), foreign-library linking (`[link]`), and PascalCase symbol binding (`#[link_name]`). The [LLVM-C FFI spike's minimal proof](../spikes/self-hosting-llvm-c-proof.md) — a Kāra program that `unsafe extern "C"`-calls ~20 LLVM-C functions to build `i64 main(){ret 42}`, verify it, and emit a Mach-O object — **compiles, links `libLLVM-18`, and runs (`./answer; exit=42`)** under the stage-0 Rust `karac`. Five prerequisites landed to get here (see below + the spike). Off the proof's success path (the proof uses static strings + libc `exit`): `CStr.to_string` (full-fidelity error-message read — **since built**, see the dedicated item below) and `#[repr(C)]`/callback-passing (still unbuilt; not exercised by this proof; needed when the real port passes structs/callbacks). → [`phase-8-stdlib-floor.md`](phase-8-stdlib-floor.md) § FFI + the [LLVM-C FFI spike](../spikes/self-hosting-llvm-c-ffi.md).
- [x] **`process.exit(code)` codegen lowering — DONE 2026-06-11** (worktree `process-exit-codegen`). `process.exit` worked under `karac run` (interpreter handles the dotted callee as a path-call) but `karac build` failed `no handler for method 'exit' on variable 'process'`: the parser hands codegen a **method call** with `process` as an identifier receiver, and `compile_method_call` had no arm for it. **Fix:** a dispatcher arm at the top of `compile_method_call` (`codegen/method_call.rs`) — gated on `method == "exit"`, receiver `Identifier("process")`, and `process` not being a real local (mirrors the ambient-resource guard) — lowers to the libc `exit(i32)` extern already declared in `Codegen::new` (`codegen.rs:2501`), coercing the code to i32 (truncate/sign-extend the compiled value — a default-i64 literal arrives wider), then terminates the block with `unreachable` (the call is `Never`). Verified: `process.exit(42)` builds, flushes prior stdout, and the process exits with status 42. Regression test `tests/codegen.rs::test_e2e_process_exit_sets_status_code` (asserts `status.code() == Some(42)`). Full `--features llvm` codegen suite green. **Noted divergence (not a blocker):** libc `exit` does NOT run pending `defer` blocks, whereas the interpreter's `ExitUnwind` is defer-respecting (`interpreter/exec.rs:33`). This matches Rust `std::process::exit` / Go `os.Exit` convention and is invisible to the self-hosting oracle (AOT-vs-AOT, never runs interp); design.md does not spec exit-defer semantics. If a unified semantics is wanted later, either make codegen unwind defers on exit or make the interpreter skip them — a language-design call, tracked here.
- [x] **`#[link_name]` honored on `unsafe extern` fn imports** *(2026-06-11).* Kāra rejects PascalCase extern fn names (Value-class/snake_case rule) and the entire LLVM-C API is PascalCase (`LLVMContextCreate`, …). `#[link_name("LLVMContextCreate")]` on a snake_case Kāra fn now redirects the emitted LLVM symbol, so a legal Kāra identifier binds the C symbol. Implementation: `codegen/functions.rs` (`link_name_attr` extractor, the `link_section` string-value pattern); `codegen.rs` (`extern_link_names: HashMap<Kāra→symbol>` field); `codegen/declarations.rs` `declare_one_extern_function` (register the import under the symbol, dedup-reusing a codegen built-in of that name like `strlen` so LLVM doesn't auto-rename to `strlen.1`); `codegen/call_dispatch.rs` (translate Kāra name → symbol before `get_function`). Verified: `getpid`/`strlen` bind + link + run; negative case (no attribute → undefined Kāra symbol); E2E `tests/codegen.rs::test_e2e_extern_link_name_binds_foreign_symbol`. The LLVM-C PascalCase bindings are now accepted by the front-end. → [LLVM-C FFI proof § gate](../spikes/self-hosting-llvm-c-proof.md).
- [x] **Codegen "Undefined variable" for a local read inside an `unsafe {}` block in an auto-par branch** *(2026-06-11; root cause was NOT "*mut T FFI return" — see below).* The 6-line `malloc`/`free` reproducer (`let m = unsafe { malloc(8) }; unsafe { free(m) }; println("ok")` → `Undefined variable 'm'`) turned out to be an **auto-parallelization** bug, not an FFI bug: the auto-par capture-set collector `refs_in_expr` (`codegen/closures.rs`) had no `ExprKind::Unsafe` arm, so it hit the catch-all `_ => {}` and never saw `m` read inside `unsafe { free(m) }` — the par-branch env struct omitted `m`, and the branch fn faulted at `load_variable("m")`. (The *analyzer* `collect_expr_reads` already handled `Unsafe`, so it correctly serialized — the two collectors were out of sync.) The trailing-statement sensitivity was the auto-par grouping heuristic; `KARAC_AUTO_PAR=0` made it vanish. **Fix:** aligned `refs_in_expr` with the analyzer — added `Unsafe`/`Try`/`Par`/`Lock`/`Question`/`Pipe`/`NilCoalesce`/`OptionalChain` arms (strictly-more-capture is always safe). Regression: `tests/par_codegen.rs::test_auto_par_branch_captures_local_read_inside_unsafe_block_e2e` (verified to fail pre-fix). Full codegen (1501) + par_codegen (112) + concurrency (80) + closures (14) green. **General-purpose correctness fix** — any auto-par branch reading a local through one of those forms was miscaptured, not just FFI.
- [x] **`*mut T` raw pointers are now `Copy`** *(2026-06-11).* The auto-par fix exposed the next gate: the `LLVMContextCreate`→`LLVMContextDispose` proof failed ownership — `let ctx = ...create(); module_create(name, ctx); context_dispose(ctx)` → *"value 'ctx' moved here, used again"*. `*const T` was `Copy` (via `ref`-mode param inference), but `*mut T` was move-only, so a handle passed to a second FFI call fired a use-after-move — and every LLVM-C handle is a `*mut` passed to many builder/dispose calls. **Fix:** added a `Type::Pointer { .. } => true` arm to `ownership.rs::is_copy_type` — raw pointers are `Copy` regardless of `const`/`mut` (Rust parity; copying a pointer is a bitwise scalar copy that can't violate ownership, and raw pointers carry no `Drop`, so no double-free risk). **This unblocked the full handle chain: `LLVMContextCreate` → `LLVMModuleCreateWithNameInContext(name, ctx)` → `LLVMContextDispose(ctx)` now builds, links against `libLLVM-18`, and runs** (prints + exit 0) — Kāra calling real libLLVM end-to-end. Regression: `tests/ownership.rs::test_raw_mut_pointer_is_copy` + `test_raw_const_pointer_is_copy` (the `*mut` one verified to fail pre-fix). Ownership (306) + codegen + memory_sanitizer + par_codegen green. → [LLVM-C FFI proof § gate](../spikes/self-hosting-llvm-c-proof.md).
- [x] **`CStr.to_string() -> Result[String, Utf8Error]` — DONE 2026-06-11** (worktree `cstr-to-string`). The outbound `char*` read path; the remaining half of the phase-8 *CString owning type + conversions* item ([phase-8:774](phase-8-stdlib-floor.md)). Landed across five layers, structurally a twin of the `env.var -> Result[String, VarError]` lowering: **(1) typecheck** — a `to_string` arm in `infer_cstr_method` (`typechecker/stdlib_seq.rs`) typing it `Result[String, Utf8Error]` (the generic `to_string` intercept at `expr_method_call.rs:1693` only catches Vec/Map/Set, so a CStr receiver correctly falls through to the CStr dispatch that records the `CStr.to_string` callee key). **(2) interpreter** — a `Value::CStr` arm in `method_call.rs` placed *before* the generic Display `to_string` (which would return a bare `String` and mismatch the `Result` type); mirrors the `String.from_utf8` oracle (`std::str::from_utf8` + `error_len()`). **(3) runtime** — new `karac_runtime_cstr_to_string` extern (`runtime/src/lib.rs`, `keep!`-registered) — validates UTF-8, writes a heap String into an out-slot on Ok (`{ptr,len,cap}`, `cap==len`), or the variant discriminant (0=InvalidByte, 1=IncompleteSequence) into a second out-slot on Err; the real UTF-8 validator the `String.from_utf8`-is-interp-only gap lacked. **(4) codegen** — `compile_cstr_to_string` (`codegen/method_call.rs`) + the extern decl in `Codegen::new` + a `Utf8Error` seed in `seed_builtin_enum_layouts` (`codegen/declarations.rs`); codegen owns the enum-tag assignment, *selecting* the `Utf8Error` variant tag from the runtime discriminant (no extra basic block) and building `Result.Ok(String)` / `Result.Err(Utf8Error)` via `build_nonshared_enum_value` + phi (Result's seeded 5-word payload holds the 4-word Utf8Error; the "Result-payload widening" the old `env_build` comment feared already landed for `env.var`). The `Utf8Error` seed is the full 3-variant shape (incl. `Other(String)`, 3-word payload, `VecOrString` drop) so user-side `Other` matching stays sound, though codegen only ever builds the two unit byte-error variants. **Verified** AOT + interp parity on all four arms: ASCII Ok, multi-byte Ok (`héllo` round-trips), `\xff`→InvalidByte, `\xe2\x82`→IncompleteSequence. Regression: `tests/codegen.rs::test_e2e_cstr_to_string_result` + 4 runtime unit tests (`cstr_to_string_tests::*`). Full `--features llvm` suite + clippy + fmt green. **Spun-off gap (not a blocker):** `Array[u8,N].as_ptr()` has no codegen handler (surfaced while building an Err-path test; `c"\x.."` literals reach the Err path without it). → [LLVM-C FFI proof § gate](../spikes/self-hosting-llvm-c-proof.md).
- [x] **Native-library link directive (`kara.toml [link]`)** *(2026-06-11)* — `karac` had no way to link an external native library; the AOT link line was hardcoded `cc <obj> <runtime.a> -lm -lpthread -ldl`. ## Self-hosting-surfaced blockers (FIX-BEFORE the lexer can run)

The lexer's natural shape — `struct SpannedToken { token: Token, span: Span }` lexed by
`mut ref self` methods — surfaced two pre-existing compiler bugs. Both reproduce on clean
`main`; neither is caused by the floor slice. The lexer (and all of self-hosting: every AST
node nests enums in structs) was blocked until #1 — **now fixed**.

- [x] ⚠ **#1 (CODEGEN, bootstrap-critical) — struct fields of a USER enum collapse to one `i64` — DONE 2026-06-10** (worktree `user-enum-struct-field-layout`). Root cause confirmed exactly as diagnosed: `compile_program` ran `declare_structs` **before** `declare_enums`, so a struct field referencing a user enum wasn't in `enum_layouts` yet and `llvm_type_for_name` fell through to `i64` (`types_lowering.rs`), losing the payload word — codegen then failed `Undefined variable 'name'` at the match arm (interp was correct). **Fix (the diagnosed two-pass declaration):** split `declare_structs` into (a) `register_struct_metadata` (AST field side tables — `struct_field_type_{names,exprs}`, `struct_field_names` — no LLVM types), then `declare_enums` (b) whose `payload_word_count_for_type_expr` struct branch now recurses through `struct_field_type_exprs` (AST) instead of `llvm_type_word_count(struct_types)`, so enum sizing no longer needs struct LLVM types, then (c) `build_struct_types` (LLVM types, `enum_layouts` now populated). Same ordering applied to the stdlib declaration path (`declare_stdlib_program`). Verified both directions (enum-in-struct AND struct-in-enum payload), wide multi-word payloads, and a scalar field *after* the enum field (correct read proves exact payload sizing). High blast radius cleared: full `--features llvm` suite + memory_sanitizer green. **Regression tests:** `tests/codegen.rs::test_e2e_user_enum_field_in_struct` (value-correctness) + `tests/memory_sanitizer.rs::asan_user_enum_field_in_struct_heap_payload` (heap String payload, no UAF/double-free). → cross-ref [`phase-7-codegen.md`](phase-7-codegen.md).
- [ ] **#2 (INTERP) — `mut ref self` method mutations don't persist.** A `mut ref self` method's writes to `self` are lost on return (direct field writes work; method-call writes don't). Repro: `struct C{n:i64} impl C{ fn inc(mut ref self){ self.n = self.n+1 } } let mut c=C{n:0}; c.inc(); // c.n still 0`. Codegen is correct. Pre-existing on `main`. Not on the lexer's critical path (the port runs under AOT, not interp), but it makes `karac run` unsound for any self-mutating method. → cross-ref [`phase-4-interpreter.md`](phase-4-interpreter.md).
- [ ] **#3 (CODEGEN, minor) — f-string over a match-arm payload binding yields empty/garbage.** `match e { A(name) => f"…{name}…" }` compiles but prints empty (f-string span-collision family, B-2026-06-09-1). Workaround in the port: `push_str` + `to_string` instead of f-strings around match-bound vars. Low priority.

Building `selfhost/src/main.kara` with the AOT path (post-#1) surfaced a **chain** of further blockers, each masking the next (codegen aborts on the first failing body, so #5/#6 only became visible once #4 was fixed). Numbered #4+ in surface order:

- [x] **#4 (CODEGEN, bootstrap-critical) — index a `Vec`/`Slice`/`Array` field through `self` (`self.bytes[self.current]`) — DONE 2026-06-11** (worktree `selfhost-lexer`, commit `af139057`). Died with "Index operator applied to non-array type". The field-access-rooted index arm resolves the receiver via `lower_field_access_ptr`, whose receiver `match` handles `Identifier`/`Index` but deliberately leaves `SelfValue` at `Ok(None)` — that fall-through is load-bearing for the method-receiver path (`self.count.fetch_add(...)` atomic dispatch must NOT take the generic field-receiver path). So `self.field[i]` fell to the generic tail and compiled the field to a `{ptr,len,cap}` VALUE. Fix: normalise a `SelfValue` inner to a synthetic `Identifier("self")` (self is registered under the name "self" in every per-binding registry) **only in `compile_index`'s FieldAccess arm** — the shared helper, and thus the atomic/method path, is byte-identical. Regression test `tests/codegen.rs::test_e2e_index_vec_field_through_self`; full `--features llvm` codegen suite green (1496), atomic-field-method tests re-verified.
- [x] **#5 (CODEGEN, bootstrap-critical) — String/Vec method on a `self.field` receiver — DONE 2026-06-11** (worktree `selfhost-lexer`). `self.src.substring(self.start, self.current)` died with "no handler for method 'substring' on non-identifier receiver". The field-receiver method helper (`try_compile_field_receiver_method`) resolves its receiver via `lower_field_access_ptr`, which leaves `SelfValue` at `Ok(None)` (load-bearing — the atomic-on-self path `self.count.fetch_add(...)` must keep falling through to `compile_atomic_method`). So `self.field.method()` fell through to the generic dispatch → no handler. Fix mirrors #4: the FieldAccess dispatch arm in `compile_method_call` normalises a `SelfValue` inner to a synthetic `Identifier("self")`, **gated on `!is_atomic_receiver(object)`** so the atomic fall-through is byte-identical. Verified `substring`/`contains`/`len` on `self.src` with field-rooted args (`token_text`); atomic_* suite re-green. Regression test `tests/codegen.rs::test_e2e_string_method_on_self_field`. Cleared the `substring` blocker → the lexer now compiles all bodies and reaches module verification, where **#6** (the `Span` collision) is the sole remaining blocker.
- [x] **#6 (CODEGEN, bootstrap-critical) — user `struct Span` collided with the always-injected stdlib `std.tracing` `Span` — DONE 2026-06-11** (worktree `selfhost-lexer`). The lexer names its span type `struct Span { line, column, offset, length }` — the most natural name, used throughout the Rust `karac` (`token.rs`). `runtime/stdlib/tracing.kara` defines `struct Span { name, span_id, parent_id, fields }` and `src/prelude.rs` injects it always (`STDLIB_SOURCES`); codegen's `struct_types` is flat name-keyed, so `declare_stdlib_program(tracing)` overwrote `struct_types["Span"]` with the tracing layout → user `Span` literal built against the wrong type → `Invalid InsertValueInst operands` at module verification. **Decision (the language designer chose DCE / collision-skip over gating tracing-behind-import):** a new `user_redefines_stdlib_type(user, stdlib)` gates BOTH `declare_stdlib_program` and `compile_stdlib_program` for each real-source stdlib module (tracing, ordering) — skip the WHOLE module when the user defines a struct/enum whose name it exports (skipping just the colliding type is unsafe: the module's own bodies reference its types through the shared maps). Surface derived from the module's own `items` (self-maintaining). Safe: a program redefining a module's public type can't use that module anyway, and an unused module's bodies are dead-stripped at link, so no size cost when there's no collision. Verified: the lexer skeleton now BUILDS + RUNS (token output produced); full `--features llvm` codegen suite green (1497, incl. tracing-using tests — kept when no collision). Regression test `tests/codegen.rs::test_e2e_user_struct_shadows_stdlib_tracing_span` (user `Span` constructed in a returning method, fields read back). **Clears the build blocker** → surfaced #7 (the lexer now runs but mis-lexes; see below). *(Note: the general usage-DCE binary-size win was found largely illusory — LTO/`-dead_strip` already removes unused stdlib bodies; #6's real value is the collision fix, which collision-skip delivers completely.)*
- [x] **#7 (CODEGEN, bootstrap-critical) — untyped integer literal in an `if`/`else` branch wasn't coerced to the branch's narrow type — DONE 2026-06-11** (worktree `selfhost-lexer`). Surfaced the moment #6 let the lexer RUN: it built but mis-lexed (`ab` → two 1-char idents, spaces → ERROR, `12` → `1`/`2`), because the lexer's `fn peek(ref self) -> u8 { if … { 0 } else { self.bytes[self.current] } }` always returned `0` → every scan loop exited immediately. Root cause in codegen, NOT typecheck: `const_int_for_suffix` lowers a suffixless literal by its suffix only → default `i64`, so an `if … { 0 } else { <u8> }` had branch values `i64` vs `i8`; `compile_if`'s phi-merge built a phi only when `tv.get_type() == ev.get_type()` and otherwise silently returned a const-`i64 0` **placeholder** — so the whole `if` evaluated to 0 (even `if true { 97u8 } else { 0 }` → 0, since the `0`'s width poisoned the merge). Minimal repro: `let a: u8 = if false { 0 } else { 97u8 };` → 0; `0u8` (typed) / `Vec[i64]` worked. **Pre-existing** (no #4/#5/#6 surface). Fix: new `unify_int_branch_widths` truncates the wider branch to the narrower width before the phi — the typechecker has already unified both branches to one type, so the only reachable width mismatch is the default-`i64`-literal-vs-narrower case, and the wider side is therefore a checker-verified-fitting constant (gated on `is_const()`; a typechecker-impossible non-const wide branch is left to the existing placeholder path). Applied to both `compile_if` and `compile_if_let`. Verified: the lexer now lexes `(ab, 12)` correctly (multi-char ident, multi-digit int, whitespace skipped, exact byte-offset spans). Regression test `tests/codegen.rs::test_e2e_if_branch_untyped_int_literal_narrow_width`.
- [x] ⚠ **#8 (AUTO-PAR, SOUNDNESS, bootstrap-critical) — statements that touch `self` were seen as independent and auto-parallelized, racing on the struct's state — DONE 2026-06-11** (worktree `selfhost-lexer`). **Surfaced by the differential oracle** (`tests/selfhost_lexer.rs`) — its very first catch, of a bug single manual runs masked: the AOT lexer's spans were **NONDETERMINISTIC** (~13% of runs mis-spanned a token, its `start` reverting to the *pre-`skip_whitespace`* offset, e.g. `IDENT "   out"` at offset 8 instead of `out` at 11). It was auto-par, NOT a #4 regression (`KARAC_AUTO_PAR=0` → 100% deterministic + correct). **Root cause (confirmed via `karac check --concurrency-report`):** the analyzer parallelized `self.skip_ws()` and `self.start = self.pos` ("no data or effect dependencies") because its per-statement read/write collectors **dropped the `self` receiver** — `collect_expr_reads` and `collect_assign_target_defines` (`concurrency.rs`) had no `ExprKind::SelfValue` arm (it sat in a no-op leaf group), so a `self.field` read recorded nothing and a `self.field = …` / mutating `self.method()` recorded no write. Two statements both touching `self` therefore showed empty read/write sets → no dependency edge → grouped + raced. Same FAMILY as `4ee83f5b` (auto-par capture-set missing an arm), different collector. **Fix:** added an `ExprKind::SelfValue` arm to both collectors recording the canonical name `"self"` (matched on both read and write sides), and removed `SelfValue` from `collect_expr_reads`'s no-op leaf group. Now `self`-mutation serializes correctly. Verified: `m8c` 250/250 deterministic with auto-par ON; the full lexer hashes identically across 30 auto-par-ON runs (== the auto-par-OFF hash → deterministic AND correct). The oracle now builds with auto-par ON (the `KARAC_AUTO_PAR=0` workaround removed) and PASSES. Regression tests: `tests/concurrency.rs::test_self_field_writes_serialize` + `::test_mut_ref_self_method_call_serializes_with_self_read` (assert no parallel group for self-touching statements). Gates: fmt, clippy `--all-targets --features llvm`, concurrency (82), par_codegen (113), full codegen, oracle — all green. **Minimal repro** (was ~13% corrupt with auto-par ON pre-fix; now 0%):
```kara
struct S { bytes: Vec[u8], pos: i64, start: i64 }
impl S {
    fn at_end(ref self) -> bool { self.pos >= self.bytes.len() }
    fn peek(ref self) -> u8 { if self.pos >= self.bytes.len() { 0 } else { self.bytes[self.pos] } }
    fn adv(mut ref self) { self.pos = self.pos + 1; }
    fn skip_ws(mut ref self) { while not self.at_end() { let c = self.peek(); if c == b' ' { self.adv(); } else { break; } } }
    fn next_off(mut ref self) -> i64 { self.skip_ws(); self.start = self.pos; self.start }
}
// drive `next_off` over "  ab  cd" across many runs → `start` is intermittently the pre-skip offset
```
The oracle (`tests/selfhost_lexer.rs`) now builds with auto-par ON and PASSES. Sibling auto-par fix, same class (a collector missing an expression arm): `4ee83f5b` (`refs_in_expr` capture-set).

Start at lexer → parser (need only the well-tested core; can begin before the full floor lands). Each stage is differential-tested against the Rust `karac` on the same inputs — including the compiler's own source.

- [~] **Lexer in Kāra** — tokens + spans; diff token stream vs Rust lexer. **IN PROGRESS** (2026-06-10). Scan model decided: **byte-indexed** (index a `Vec[u8]` from `source.bytes()`, mirror `src/lexer.rs` near-line-for-line) — chosen over char-iterator so `(offset,length)` spans reproduce the Rust lexer bit-for-bit (exact differential oracle), the port stays mechanical, and lookahead stays O(1). Skeleton + scaffold live in `selfhost/` (root; NOT `bootstrap/`, which is the staging procedure). Harness pivoted from `karac run` (interp) to **`karac build` (AOT)** — the interpreter can't run the lexer (blocker #2 below) and AOT is the real bootstrap oracle anyway. **Bootstrap-critical blockers #1 FIXED 2026-06-10; #4 (`af139057`) + #5 + #6 (Span collision) + #7 (if-branch literal width) FIXED 2026-06-11.** The skeleton BUILDS, RUNS, and lexes `(ab, 12)` correctly on a clean run via AOT (multi-char idents, multi-digit ints, whitespace skipping, exact byte-offset spans). **Differential oracle BUILT** (`tests/selfhost_lexer.rs`): a shared corpus is lexed by both `karac::tokenize` (Rust seed) and the Kāra lexer (`karac build`-AOT, since interp mishandles `mut ref self`), and the rendered token streams are diffed — and it immediately caught **#8**, a nondeterministic auto-par soundness bug that single runs masked (now FIXED — the analyzer dropped the `self` receiver from its read/write collectors, so self-mutating statements raced). ## Bootstrap procedure (3-stage)

- [ ] **Stage-1:** Rust `karac` compiles the Kāra compiler source → stage-1 binary (correct, but ~2× slow — the codegen IR-quality pass is a Phase 11 item not yet in either compiler).
- [ ] **Stage-2:** stage-1 compiles the same source → stage-2.
- [ ] **Stage-3 fixpoint:** stage-2 compiles the same source → stage-3; **stage-2 and stage-3 must be byte-identical.** Ship stage-2.
- [ ] **Differential gate in CI:** over a corpus of `.kara` programs, Rust-`karac` output == Kāra-`karac` output (the oracle). Lock before declaring self-hosted.

Speed note: the codegen IR-quality pass (Phase 11, compiler-internal) is written in Kāra *after* the pivot and recovers the self-hosted compiler's own speed via this same staging — see [`../roadmap.md` § Codegen Optimization](../roadmap.md#codegen-optimization-ir-quality-pass).

**Done when:** `karac build <kara-compiler-src>` produces a binary that compiles Kāra programs with output identical to the Rust `karac`, and the 3-stage bootstrap reaches a byte-identical fixpoint. From here the Rust `karac` is frozen as the bootstrap seed and all new compiler work lands in Kāra.
