- [ ] **`if let` / `while let` / `let...else` scrutinee temporary scope.** Spec lives at design.md § `if let` and `let...else` (v60 item 29). Sub-steps: (1) NLL/drop-placement pass — verify that scrutinee-expression temporaries (anonymous intermediates produced by the right-hand side of `=` in `if let PAT = SCRUT { … } else { … }`, `while let PAT = SCRUT { … }`, `let PAT = SCRUT else { … };`) have a live range scoped to the matching arm: in the `then` arm of `if let`, the temporaries live through the entire body and drop at arm exit (because pattern-bound names may borrow into them); in the `else` arm of `if let` / `else if let` / `let...else`, the temporaries have already been dropped *before* the arm body begins; in `while let`, per-iteration drop at body exit before the next iteration's scrutinee is evaluated. (2) Code-path verification — verify the code path that lowers `if let` / `let-else` / `while let` to control flow places the scrutinee-temp drops on the *non-matching* edge before the else-arm code (rather than placing them at the join point after both arms). The lowering pattern: evaluate scrutinee → match against pattern → on miss, drop temps then jump to else; on hit, bind names then enter then-arm body, drop temps at body exit. (3) Pattern-binding interaction — pattern-bound names (`value` in `if let Some(value) = ...`) are local bindings of the arm, NOT temporaries; their drop follows the standard `let`-binding rules and is unaffected by this item. The rule covers only the anonymous intermediates of the scrutinee expression. (4) Deferred composition — the future `if let` chains feature (deferred to Phase 5+) will apply the same rule per-link: each link's temporaries drop before the next link is evaluated (on success) or before the else arm runs (on failure). (5) Test plan: lock-in-scrutinee `if let Some(v) = mu.lock().get(k) { ... } else { log(...) }` — assert via instrumented destructors that the MutexGuard's `Drop` fires before the else block runs; `let Ok(config) = parse(mu.lock().as_str()) else { return Err(...) };` — guard drops before the divergent else block; `while let Some(item) = mu.lock().pop_front() { process(item) }` — per-iteration lock release verified by instrumented drop order across iterations; pattern-bound name lifetime in then-arm holds borrows into scrutinee temps without dangling. - [ ] **Panic semantics at the FFI boundary — `extern "C"` auto-abort + `extern "C-unwind"` propagation.** Spec lives at design.md § Panic Semantics at the FFI Boundary (v60 item 27). Sub-steps: (1) Codegen — for every `pub extern "C" fn` defined in Kāra, emit two LLVM symbols: (a) the bare Kāra-callable function (mangled by the standard rules unless `#[unsafe(no_mangle)]` overrides) and (b) a separate exported wrapper symbol with the C-facing name that wraps the call in a `catch_panic`-equivalent landing pad and routes any caught panic to `std.process.abort()`. Kāra-internal calls go through the bare symbol with no overhead; C-side calls go through the trampoline. The trampoline's abort path writes the panic message + source location to stderr before calling `abort()`. (2) Codegen elision — under `panic = "abort"` build profile, the trampoline is a no-op (every panic is already abort); emit only the bare symbol with the C-facing name (or the wrapper as a tail-call that just forwards). (3) Build-profile gate — `extern "C-unwind"` definitions and imports are rejected at parse under `panic = "abort"` profiles with `error[E_EXTERN_C_UNWIND_REQUIRES_UNWIND_PROFILE]`. The same gate already rejects `catch_panic` calls under abort profiles (per item 26's checklist entry); both rejections share the diagnostic surface. (4) Typechecker — `pub extern "C-unwind" fn` definitions whose body's inferred effects include `panics` MUST declare `panics` in the signature; otherwise emit `error[E_EXTERN_C_UNWIND_REQUIRES_PANICS]` with a suggested-fix span adding `with panics` to the signature. `pub extern "C" fn` definitions MAY declare `panics`; the declared effect describes Kāra-side calls (the boundary trampoline is invisible to the effect system). (5) Effect-system integration — no new rules needed beyond (4); the existing inference and propagation machinery handles all four cases (import/export × C/C-unwind) uniformly because the trampoline is a codegen-only artifact. (6) FFI absorption pattern — the design recommends wrapping the body in `catch_panic` to surface errors to C as error codes rather than relying on auto-abort; the auto-abort trampoline is the safety net for panics that escape the user-written catch (e.g., panic during the panic-handler logic). - [ ] **`catch_panic[T]` panic-absorption primitive.** Spec lives at design.md § Catching Panics: `catch_panic[T]` (v60 item 26). Sub-steps: (1) Typechecker — recognise `catch_panic` as a `#[compiler_builtin]` with the special signature `fn catch_panic[T, with E](f: Fn() -> T with E) -> Result[T, PanicInfo] with (E - panics)`. The `(E - panics)` arithmetic is closed-form in the compiler — it is not a user-writable pattern. At every call site, the result type's effect ceiling is the closure's effect set with `panics` removed; if the closure does not have `panics`, `karac lint` flags the call as a likely mistake but the call remains legal (always returns `Ok`). (2) Build-profile gate — under `panic = "abort"` (kernel/embedded/ISR profiles), `catch_panic` calls are rejected at typecheck with `error[E_CATCH_PANIC_REQUIRES_UNWIND]: catch_panic is unavailable under panic = "abort"; remove the call or change the build profile`. The same gate rejects `extern "C-unwind"` declarations under abort builds. (3) Stdlib — `PanicInfo` (with `message`, `location`, `kind`, `thread_name`, `backtrace`), `PanicKind` (`#[non_exhaustive]` enum: `Explicit`, `Unwrap`, `Indexing`, `IntegerOverflow`, `DivisionByZero`, `ContractViolation`, `Foreign`, `DropDuringUnwind`), `SourceLocation` types in the prelude (paired with `catch_panic` and the panic-hook surface). `PanicInfo` is `Send + Clone`. (4) Codegen (Phase 7) — lower `catch_panic(closure)` to LLVM's `invoke` instruction with a landing pad that catches the unwind, captures `PanicInfo` from the runtime panic-payload, and routes to the `Err(...)` arm. The runtime side maintains a thread-local "in-flight panic" tag; `process.exit()` sets the tag to `Exit` rather than `Recoverable`, and the catch-frame inspects the tag and re-raises rather than producing `Err(PanicInfo)` (per the design.md/process.exit interaction rule, originally pinned in deferred.md). (5) Auto-concurrency interaction — `catch_panic` does not extend the panic-domination rule for parallel scopes; a panic absorbed inside a single branch produces `Ok(Err(PanicInfo))` from that branch, the parallel scope sees a successful result, no peer cancellation. The structured-concurrency machinery already does its own internal panic-isolation at task boundaries; `catch_panic` is the user-facing surface of the same primitive. (6) Drop-time interaction — destructor panics during the closure's normal scope-exit produce `Err(PanicInfo { kind: DropDuringUnwind, ... })`; destructor panics *during* an in-flight unwind that `catch_panic` is already absorbing become double-panics and abort the process (no double-panic protection). (7) FFI integration — panics that crossed an `extern "C-unwind"` ABI boundary surface as `PanicKind.Foreign`; the absorption pattern from the FFI section uses `catch_panic` directly. Test plan (when codegen lands): explicit `panic!` inside closure → `Err` with `Explicit`; out-of-bounds index → `Err` with `Indexing`; `process.exit(2)` propagates through (verified via subprocess test); double-panic during catch → process-aborts; `catch_panic` under `panic = "abort"` profile → typecheck error; closure carrying `with reads(Db) writes(Cache) panics` produces a `Result` whose result-type's effect ceiling is `with reads(Db) writes(Cache)` (panics stripped, others passed through). (Phase 7 — depends on LLVM unwinding substrate; typechecker stub work can land in Phase 5.2 to gate the build-profile rejection diagnostics.)

- [x] **Scheduler minimum invariants (language-level, cross-scheduler). Implemented:** The Phase 6 scheduler and any future runtime profile must satisfy five invariants that programs written against language-level rules rely on: (1) **no lost work** — a completed branch's `Ok` result is never silently discarded (it reaches the scope return, a `collect_all` result, or at minimum its `defer`/mutations still run); (2) **cancellation eventually observed** — every branch either completes or observes cancellation in bounded time after a triggering event; (3) **cancellation cascades** into nested regions; (4) **termination** — programs without pure tight loops terminate when all branches finish or cancel; (5) **scheduler deadlock-freedom** — the scheduler itself must not introduce deadlock beyond what user-level locks can create. Backpressure under overload, fairness between unrelated regions, and scheduler-specific cancel optimizations are **not** in this invariant set — they are per-profile concerns deferred to runtime chapters, and programs that depend on them must declare the dependency in their build profile. See design.md § Scheduler-layer concerns and minimum invariants.
