- [ ] **Completion wins cancellation.** Implement the rule that a sibling branch which has already passed its last compiler-inserted effect-boundary check when cancel fires completes naturally and returns `Ok`. Its `defer` blocks must run and its mutations to shared state must remain visible. Real work must never be retroactively converted to `Cancelled`. Whether the `Ok` surfaces at the scope boundary is a separate question governed by the scope-return precedence rules (fail-fast `par` still lets a source-earlier `Err` win; `collect_all` exposes everything). This rule is what makes `collect_all`'s "every branch runs to completion" guarantee load-bearing. See design.md ┬¦ Parallel Failure and Cleanup.

- [ ] **Cascading cooperative cancellation across nested `par` regions.** When an outer scope cancels, the cancel observation must reach branches inside nested parallel regions through the same cooperative effect-boundary mechanism ŌĆö no special cross-scope machinery. The effect-boundary check inside the inner branch observes the outer cancel and triggers the inner region's fail-fast, which propagates to the inner region's siblings. No branch should escape cancellation simply because it is nested. Worst-case latency is the sum of effect-boundary distances along the nesting path; document it.

- [x] **`karac query concurrency`.** Return parallelization decisions with reasoning. Moved from Phase 5 ŌĆö requires concurrency analysis to be implemented first.

- [x] **Parameterized-resource distinctness rules.** Conservative collapse: two parameterized resources `R[a]` and `R[b]` are treated as distinct only when `a` and `b` are distinct string/integer literals at the call site, or are variables provably bound to different values (e.g., different function parameters). When distinctness is ambiguous (computed values, aliases, runtime-determined keys), the compiler conservatively collapses to the same resource and serializes. The rule is intentionally strict: if the programmer cannot predict the outcome without running the compiler, they'll reach for `seq { }` ŌĆö which is the correct escape hatch, not a failure mode. Future extension: `#[distinct]` annotation to assert distinctness when the compiler can't prove it.

- [ ] **`if let` / `while let` / `let...else` scrutinee temporary scope.** Spec lives at design.md ┬¦ `if let` and `let...else` (v60 item 29). Sub-steps: (1) NLL/drop-placement pass ŌĆö verify that scrutinee-expression temporaries (anonymous intermediates produced by the right-hand side of `=` in `if let PAT = SCRUT { ŌĆ” } else { ŌĆ” }`, `while let PAT = SCRUT { ŌĆ” }`, `let PAT = SCRUT else { ŌĆ” };`) have a live range scoped to the matching arm: in the `then` arm of `if let`, the temporaries live through the entire body and drop at arm exit (because pattern-bound names may borrow into them); in the `else` arm of `if let` / `else if let` / `let...else`, the temporaries have already been dropped *before* the arm body begins; in `while let`, per-iteration drop at body exit before the next iteration's scrutinee is evaluated. (2) Code-path verification ŌĆö verify the code path that lowers `if let` / `let-else` / `while let` to control flow places the scrutinee-temp drops on the *non-matching* edge before the else-arm code (rather than placing them at the join point after both arms). The lowering pattern: evaluate scrutinee ŌåÆ match against pattern ŌåÆ on miss, drop temps then jump to else; on hit, bind names then enter then-arm body, drop temps at body exit. (3) Pattern-binding interaction ŌĆö pattern-bound names (`value` in `if let Some(value) = ...`) are local bindings of the arm, NOT temporaries; their drop follows the standard `let`-binding rules and is unaffected by this item. The rule covers only the anonymous intermediates of the scrutinee expression. (4) Deferred composition ŌĆö the future `if let` chains feature (deferred to Phase 5+) will apply the same rule per-link: each link's temporaries drop before the next link is evaluated (on success) or before the else arm runs (on failure). - [ ] **Panic semantics at the FFI boundary ŌĆö `extern "C"` auto-abort + `extern "C-unwind"` propagation.** Spec lives at design.md ┬¦ Panic Semantics at the FFI Boundary (v60 item 27). - [ ] **`catch_panic[T]` panic-absorption primitive.** Spec lives at design.md ┬¦ Catching Panics: `catch_panic[T]` (v60 item 26). Sub-steps: (1) Typechecker ŌĆö recognise `catch_panic` as a `#[compiler_builtin]` with the special signature `fn catch_panic[T, with E](f: Fn() -> T with E) -> Result[T, PanicInfo] with (E - panics)`. The `(E - panics)` arithmetic is closed-form in the compiler ŌĆö it is not a user-writable pattern. At every call site, the result type's effect ceiling is the closure's effect set with `panics` removed; if the closure does not have `panics`, `karac lint` flags the call as a likely mistake but the call remains legal (always returns `Ok`). (2) Build-profile gate ŌĆö under `panic = "abort"` (kernel/embedded/ISR profiles), `catch_panic` calls are rejected at typecheck with `error[E_CATCH_PANIC_REQUIRES_UNWIND]: catch_panic is unavailable under panic = "abort"; remove the call or change the build profile`. The same gate rejects `extern "C-unwind"` declarations under abort builds. (3) Stdlib ŌĆö `PanicInfo` (with `message`, `location`, `kind`, `thread_name`, `backtrace`), `PanicKind` (`#[non_exhaustive]` enum: `Explicit`, `Unwrap`, `Indexing`, `IntegerOverflow`, `DivisionByZero`, `ContractViolation`, `Foreign`, `DropDuringUnwind`), `SourceLocation` types in the prelude (paired with `catch_panic` and the panic-hook surface). `PanicInfo` is `Send + Clone`. (4) Codegen (Phase 7) ŌĆö lower `catch_panic(closure)` to LLVM's `invoke` instruction with a landing pad that catches the unwind, captures `PanicInfo` from the runtime panic-payload, and routes to the `Err(...)` arm. The runtime side maintains a thread-local "in-flight panic" tag; `process.exit()` sets the tag to `Exit` rather than `Recoverable`, and the catch-frame inspects the tag and re-raises rather than producing `Err(PanicInfo)` (per the design.md/process.exit interaction rule, originally pinned in deferred.md). (5) Auto-concurrency interaction ŌĆö `catch_panic` does not extend the panic-domination rule for parallel scopes; a panic absorbed inside a single branch produces `Ok(Err(PanicInfo))` from that branch, the parallel scope sees a successful result, no peer cancellation. The structured-concurrency machinery already does its own internal panic-isolation at task boundaries; `catch_panic` is the user-facing surface of the same primitive. (6) Drop-time interaction ŌĆö destructor panics during the closure's normal scope-exit produce `Err(PanicInfo { kind: DropDuringUnwind, ... })`; destructor panics *during* an in-flight unwind that `catch_panic` is already absorbing become double-panics and abort the process (no double-panic protection). (7) FFI integration ŌĆö panics that crossed an `extern "C-unwind"` ABI boundary surface as `PanicKind.Foreign`; the absorption pattern from the FFI section uses `catch_panic` directly. Test plan (when codegen lands): explicit `panic!` inside closure ŌåÆ `Err` with `Explicit`; out-of-bounds index ŌåÆ `Err` with `Indexing`; `process.exit(2)` propagates through (verified via subprocess test); double-panic during catch ŌåÆ process-aborts; `catch_panic` under `panic = "abort"` profile ŌåÆ typecheck error; closure carrying `with reads(Db) writes(Cache) panics` produces a `Result` whose result-type's effect ceiling is `with reads(Db) writes(Cache)` (panics stripped, others passed through). (Phase 7 ŌĆö depends on LLVM unwinding substrate; typechecker stub work can land in Phase 5.2 to gate the build-profile rejection diagnostics.)

- [x] **Scheduler minimum invariants (language-level, cross-scheduler). Implemented:** The Phase 6 scheduler and any future runtime profile must satisfy five invariants that programs written against language-level rules rely on: (1) **no lost work** ŌĆö a completed branch's `Ok` result is never silently discarded (it reaches the scope return, a `collect_all` result, or at minimum its `defer`/mutations still run); (2) **cancellation eventually observed** ŌĆö every branch either completes or observes cancellation in bounded time after a triggering event; (3) **cancellation cascades** into nested regions; (4) **termination** ŌĆö programs without pure tight loops terminate when all branches finish or cancel; (5) **scheduler deadlock-freedom** ŌĆö the scheduler itself must not introduce deadlock beyond what user-level locks can create. Backpressure under overload, fairness between unrelated regions, and scheduler-specific cancel optimizations are **not** in this invariant set ŌĆö they are per-profile concerns deferred to runtime chapters, and programs that depend on them must declare the dependency in their build profile. See design.md ┬¦ Scheduler-layer concerns and minimum invariants.
